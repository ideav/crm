# Техническое задание: Построение контекстной памяти для ИИ-агента

## 1. Цель системы
Реализовать графовую (древовидную) базу знаний, которая при получении нового запроса позволяет:
1. **Извлекать** релевантную историю (похожие запросы, связанные события, ошибки, контекст проектов).
2. **Обогащать** контекст агента перед генерацией ответа.
3. **Сохранять** новый опыт с учетом его семантической близости к уже существующим узлам.

## 2. Архитектура «Дерева памяти»

Память представляет собой динамический граф, в котором узлы — это события, а ребра — смысловые связи с весами.

### 2.1. Структура узла (Node / Fact)
Каждый узел должен содержать:
- **ID** — уникальный идентификатор записи.
- **Raw Input** — исходный текст запроса/комментария.
- **Cleaned Query** — очищенный текст (без мусора, стоп-слов, магических чисел/ID).
- **Embedding Vector** — векторное представление (для поиска ближайших соседей).
- **Глобальный контекст (Domain)** — категория верхнего уровня (например: `integrations`, `project_X`, `infra`, `hr_portal`).
- **Тематика (Topic)** — тип запроса (например: `bug_fix`, `feature_request`, `complaint`, `question`).
- **Конкретика (Essence)** — 1-3 ключевых слова/сущности (например: `workplace`, `Tokyo`, `SRF`, `layout`).
- **Мета-данные**:
  - `created_at` / `updated_at`
  - `popularity_counter` (счетчик обращений)
  - `decay_factor` (коэффициент устаревания)
  - `importance_weight` (агрегированный вес важности)

### 2.2. Типы связей (Edges)
- `SIMILAR_TO` — семантическая близость (косинусное расстояние между векторами).
- `CAUSED_BY` / `FOLLOWED_BY` — каузальная связь («этот баг вызвал вот этот коммит», «после этого запроса пришла жалоба»).
- `BELONGS_TO` — жесткая связь с проектом/доменом.
- `REFERENCES` — упоминание тех же сущностей (ID, имена, названия).

## 3. Алгоритм обработки входящего запроса (Pipeline)

### Шаг 1: Предобработка и категоризация
На входе сырой текст (например, *«создай рабочее место, как в прошлый раз, только без ошибки SRF»*).
1. **Очистка**: удаление магических чисел и ID (или их маркировка с минимальным весом).
2. **Извлечение контекста**:
   - *Домен*: определяем по ключевым словам или истории пользователя.
   - *Тема*: классифицируем интент.
   - *Суть*: вычленяем 1-2 главных ключевых слова (без «воды»).
3. **Векторизация**: создание эмбеддинга очищенного запроса.

### Шаг 2: Поиск ближайших соседей («Гирлянда истории»)
Прежде чем сохранить новый узел, ищем в базе топ-N (по умолчанию 16) ближайших узлов по косинусному расстоянию векторов.
- **Зона охвата**: N ближайших соседей + их прямые связи первого уровня (каузальные цепочки).
- **Результат**: агент получает «гирлянду» — краткое содержание прошлых похожих запросов, связанных с ними багов и решений.

### Шаг 3: Обновление весов (Reinforcement)
При чтении узлов из базы:
1. Увеличиваем `popularity_counter` у всех затронутых узлов.
2. Усиливаем вес связи (`SIMILAR_TO`), по которой произошел переход.

### Шаг 4: Запись нового узла
1. Создаем новый узел с `popularity_counter = 1`.
2. Создаем связи `SIMILAR_TO` к найденным соседям с весом, равным косинусной близости.
3. Если есть явная временная/логическая связь с одним из соседей (например, это ответ на баг-репорт), добавляем связь `CAUSED_BY` с высоким весом.

## 4. Механизм забывания (Затухание мусора)

Проблема: в базу попадает много «одноразового мусора» (ошибки, опечатки, уникальные имена, которые больше не повторятся).

**Решение — Коэффициент затухания (Decay):**
- Каждый узел имеет `decay_score`, который со временем уменьшается (например, по экспоненте).
- При каждом обращении к узлу `decay_score` поднимается до максимума (омоложение).
- Узлы с `decay_score` ниже порога и `popularity_counter == 1` (обратились всего один раз при создании) исключаются из выдачи топ-N или архивируются.
- Условие удаления: Если связь `SIMILAR_TO` есть, но узел-мусор (опечатка) не имеет усиливающих связей `CAUSED_BY` и к нему нет повторных переходов — он уходит на дно графа.

## 5. Функциональные требования к API базы данных

1. **`search_similar(vector, top_n=16)`**
   Возвращает список узлов с связями 1-го порядка, отсортированных по `score = cosine_similarity * importance_weight`.

2. **`get_garland(node_id)`**
   Рекурсивно собирает цепочку: `запрос -> похожие -> каузальные связи (баги/фиксы)`. Глубина — 2-3 шага.

3. **`upsert_node(vector, metadata)`**
   Добавляет узел и автоматически связывает его с ближайшими соседями.

4. **`touch_node(node_id)`**
   Увеличивает счетчик популярности и сбрасывает decay при обращении.

## 6. План развития (Дополнительные идеи для реализации)
- Веса по проектам: Хранить не только тему, но и ID пользователя/команды, чтобы повышать вес локального контекста.
- Ручная коррекция дерева: Возможность оператору сказать «этот запрос и этот баг не связаны» (удаление ребра с весом 0).
- Суммаризация: При слишком большом количестве узлов в гирлянде автоматически генерировать короткую сводку (Summary), чтобы не перегружать контекст агента.

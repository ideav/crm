# VecMory — отчёт: MCP-память развёрнута + план теста «с памятью / без»

*Дата: 25.06.2026. Дополняет [`summary_vm.md`](summary_vm.md), [`Plan-draft.md`](Plan-draft.md),
[`vecmory_deck.md`](vecmory_deck.md). Фиксирует переход от «прототип проверен» к «инструмент
работает у агента», и даёт воспроизводимую методику измерить пользу памяти.*

---

## 1. Резюме

VecMory-память доведена до **рабочего MCP-инструмента**, который Claude Code (и совместимый
агент через тот же `~/.claude`) видит как набор тулз и зовёт прямо в диалоге. Память лежит в
базе Integram (квартет `id, up, t, val`), эмбеддер **локальный** (CPU, без внешних API),
поиск — косинус по записям. Тезис deck «никакой новой инфраструктуры» подтверждён практикой:
**ни вектор-СУБД, ни GPU, ни отдельного сервиса — память живёт рядом с данными**.

Ключевое инженерное решение: **не писать клиент заново**, а обернуть уже протестированный
Python-бэкенд памяти (эмбеддинг + запись/поиск в Integram) тонким MCP-слоем. Один стек,
один стор, два потребителя.

---

## 2. Что сделано

### Архитектура (3 слоя, переиспользование)

| Слой | Роль |
|---|---|
| **MCP-сервер** (FastMCP, транспорт stdio) | тонкая обёртка: объявляет тулзы, зовёт бэкенд |
| **Бэкенд памяти** (Python, переиспользован) | локальный эмбеддер + клиент Integram (`_m_new`/отчёты), косинус |
| **Хранилище** (Integram `mem`) | таблица записей: текст + вектор (JSON в MEMO-поле) |

- **Эмбеддер:** локальный `fastembed`/ONNX, модель `paraphrase-multilingual-MiniLM-L12-v2`
  (384-dim, мультиязычная, знает русский). Версия эмбеддера зафиксирована и совпадает с той,
  которой уже наполнен корпус, → векторы запроса и хранимые **сопоставимы** (критично для recall).
- **Хранилище:** запись = строка таблицы; текст эмбеддится и нормализуется (L2), вектор пишется
  отдельным полем как JSON. Поиск — brute-force косинус (граф-ANN пока выключен; на текущих
  объёмах brute-force и так мгновенный, граф окупается на 10k+).
- **Регистрация:** MCP добавлен в user-scope конфиг Claude Code (`~/.claude.json`), поэтому
  доступен во всех проектах и любому агенту, читающему тот же `~/.claude`. `claude mcp list`
  показывает сервер как **✔ Connected**.

### Тулзы наружу (агент шлёт текст, не векторы)

| Тулза | Что делает |
|---|---|
| `recall(query, k=5)` | семантический поиск: топ-k похожих записей (грабли/решения/контекст) |
| `remember(text)` | сохранить запись: локальный эмбеддинг → запись в Integram |
| `memory_status()` | состояние: всего записей / в графе / свежих / средняя степень графа |

### Проверки (вживую, не на моках)

- `memory_status` отдаёт реальные счётчики живой базы.
- **Roundtrip `remember → recall`:** записали новую запись, `total` вырос на 1, последующий
  `recall` вернул её **топ-1** с ожидаемым косинусом. Полный цикл «запомнил → нашёл» работает.
- Тулзы зовутся по **настоящему MCP stdio-протоколу** (initialize → list_tools → call_tool),
  а не напрямую из Python, — это и есть тот путь, которым ходит Claude Code.

### Дисциплина

- **Бэкап конфигурации** Claude (`~/.claude.json`) сделан до регистрации сервера; откат — одной
  командой.
- **Секреты** (токен базы) — только из окружения, в репозиторий не попадают.
- Окружение MCP **изолировано** (отдельный venv), не задевает другие сервисы на хосте.

## 3. Текущие границы (честно)

- **Память не инжектится автоматически.** Сейчас агент должен *сам* позвать `recall`. Авто-режим
  («перед каждым запросом подмешать похожее, после ответа сохранить урок») делается хуками
  Claude Code — это небольшая отдельная доработка (см. §4.3).
- **Граф-ANN выключен** — поиск линейный. Для десятков–сотен записей это незаметно; включение
  графа — задача под рост объёма.
- **Единый стор.** Память агента может делить таблицу с другим потребителем той же базы. Если
  начнёт мешать (нерелевантные пересечения) — корпуса разводятся отдельными таблицами без правки
  кода (бэкенд параметризован именем таблицы через окружение).

## 4. План: как протестировать «с памятью» и «без»

Цель — не «работает ли поиск» (это §2), а **даёт ли память измеримую пользу агенту**. Это
ровно тот вопрос, который deck называет главным («память о граблях, а не о пользователе»).
Тестируем на двух уровнях.

### 4.1. Уровень 1 — функциональная корректность (можно уже сейчас)

Базовая проверка, что ретрив сам по себе адекватен.

- **Recall@k на размеченном наборе.** Завести 20–50 пар «запрос → какая запись должна найтись».
  Прогнать `recall`, посчитать долю, где нужная запись попала в топ-k. Цель — фиксировать число
  и следить, чтобы не падало при изменениях.
- **Косинус-санити.** Точное косинусное ранжирование сверить с эталоном `numpy` (как в
  `summary_vm.md` — там совпало до 5-го знака).
- ⚠️ **Опираться на ранг, не на абсолютный порог.** У мультиязычных эмбеддеров высокий базовый
  косинус (даже несвязные тексты ≈ 0.8+). Поэтому берём **top-k**, а не «всё выше X».

### 4.2. Уровень 2 — A/B агента «с памятью vs без» (главная ценность)

**Гипотеза:** агент, у которого есть доступ к памяти граблей, *реже повторяет прошлые ошибки*,
быстрее приходит к решению и требует меньше итераций оператора.

**Подготовка.**
1. **Засеять корпус граблей.** Взять реальные «симптом → причина → фикс» из закрытых задач и из
   ручной базы (`CLAUDE.md`, `docs/kb/`) и записать их через `remember`. Это даёт агенту, что
   вспоминать.
2. **Набор задач T (10–20 шт).** Задачи, чьё корректное решение *опирается* на засеянные грабли
   (например: «меняем X — что проверить, чтобы не сломать Y?», где про «Y ломается» уже есть
   запись). Часть задач — **контрольные** (никакая запись не помогает) — чтобы поймать ложную
   уверенность.

**Прогон (на каждую задачу — две ветки).**
- **With memory:** сессия с подключённым VecMory; агент *обязан* перед ответом сделать
  `recall(задача)` и учесть найденное.
- **Without memory:** свежая сессия **без** тулзы (или с временно снятым сервером). Тот же
  промпт, та же модель, те же настройки.
- Рандомизировать порядок и, по возможности, развести «кто готовил задачи» и «кто оценивает».

**Метрики.**

| Метрика | Как считать |
|---|---|
| Попадание в грабли | агент сам упомянул/применил релевантный известный риск? (да/нет) |
| Повтор ошибки | наступил ли на грабли, которые были в памяти? (да/нет) |
| Итерации до решения | сколько раундов правок до приёмки оператором |
| Время / стоимость | wall-clock и токены на задачу |
| Вердикт оператора | «с памятью лучше / так же / хуже» |
| Ложные срабатывания | на контрольных задачах память не сбила с толку? |

**Что считаем успехом.** На задачах-с-граблями ветка *with memory* статистически чаще попадает
в известный риск и реже его повторяет, при не худшем времени; на контрольных — не деградирует.

### 4.3. Чтобы «с памятью» стало автоматическим (опционально, небольшая доработка)

Чтобы не полагаться на то, что агент сам вспомнит позвать `recall`:

- **Хук `UserPromptSubmit`** → `recall(промпт)` и подмешать топ-k в контекст *до* ответа.
- **Хук `Stop`** → `remember(итог/урок)` после завершения.
- **Тумблер** (env-флаг) для мгновенного вкл/выкл — это и есть рубильник A/B без перекатки кода.

### 4.4. Параллельный, более зрелый трек: «с RAG / без RAG» в прикладной сборке

Помимо памяти-для-агента, тот же бэкенд уже используется для **retrieval-augmented сборки
приложений**: похожие прошлые проекты подмешиваются в промпт строителя. Там A/B оформлен
**тумблером** (`вкл/выкл` одним флагом) и измеряется на реальных заявках по тем же осям
(полнота структуры, корректность, время, стоимость, вердикт оператора). Это даёт второй,
количественный контур доказательства пользы памяти.

### 4.5. Можно ли уже сейчас?

- **Ручной A/B (Уровень 2) — да, прямо сейчас:** тулзы доступны, нужен лишь засеянный корпус и
  набор задач. Это самый быстрый способ получить первые цифры «с/без».
- **Уровень 1 (recall@k) — да,** как только размечен мини-набор.
- **Автоматический режим (хуки)** и **граф-ANN** — небольшие доработки, не блокируют первые
  замеры.
- **Прикладной «с RAG/без RAG»** (§4.4) — управляется тумблером; требует только включения и
  прогона партии заявок.

## 5. Как воспроизвести (без секретов)

Минимальный путь (значения токена/базы — из окружения, не из репозитория):

```bash
# 1. Окружение MCP-сервера
python3 -m venv venv
venv/bin/pip install mcp fastembed aiohttp numpy python-dotenv

# 2. Конфиг бэкенда (.env): имя базы, токен, имя таблицы, поля, модель эмбеддера
#    MEM_DB_NAME=... MEM_TOKEN=...(env, не в git) MEM_TABLE=... MEM_EMBED_MODEL=...

# 3. Регистрация в Claude Code (user-scope → виден во всех проектах)
claude mcp add --scope user vecmory -- /путь/venv/bin/python /путь/server.py
claude mcp list          # vecmory: ... ✔ Connected

# 4. Smoke-тест по stdio-протоколу: initialize → list_tools → memory_status → recall
venv/bin/python test_mcp.py
```

После этого в сессии Claude Code доступны `recall` / `remember` / `memory_status`.

## 6. Вывод

Память VecMory из «доказанного прототипа» стала **инструментом, которым агент пользуется**:
локальный эмбеддер, хранилище в Integram, ноль новой инфраструктуры — ровно как обещает deck.
Следующий и самый ценный шаг — не код, а **цифры**: прогнать A/B «с памятью / без» по методике
§4.2 (ручной режим возможен уже сейчас) и параллельно снять количественный «с RAG / без RAG» в
прикладной сборке (§4.4).

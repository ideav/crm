**Дата:** 2026-06-02
**Источник:** запрос Андрея — отказаться от фиксированных «Типов резки»; полосы (раскладку ножей) считать динамически при планировании и хранить под конкретной «Производственной резкой».
**БД:** боевая ateh (`https://ideav.ru/ateh`). Токен `X-Authorization` (ротируется).

## 0. Видение эпика (контекст; реализуется по подпроектам)

Сейчас раскрой задаётся **фиксированными «Типами резки»** (таблица 1072) с подчинённой
таблицей **«Полоса»** (1073: Ширина мм / Количество / Назначение). Позиция заказа и
Производственная резка **ссылаются** на Тип резки; orders.js подбирает тип по
сырью+ширине (фича C); production-planning генерирует резки, матча позиции к типам.

**Новая модель:** Типов резки нет. У каждой **Производственной резки** своя раскладка —
подчинённые **Полосы**, посчитанные при планировании из **ширин необеспеченных позиций
одного сырья** (бин-пакинг в ширину джамбо) + **добор ходовыми** (отчёт `preferable_widths`,
`position_qty_sum` = «Популярность»). Сырьё и ширина входа резки берутся **авто из
Партии сырья → Вид сырья**. Диспетчер правит раскладку во встроенной в production-planning
форме (на базе cut-calc) и **сохраняет Полосы** под Резкой (Тип резки не сохраняется — его нет).

### Декомпозиция эпика
- **F1** — чистое ядро раскладки (TDD): бин-пакинг ширин позиций в джамбо + добор ходовыми + дата-окно объединения.
- **F2 (этот документ)** — схема + отчёты (live): Полоса→Резка, убрать Тип резки и ссылки, поле «Срок изготовления», правка отчётов.
- **F3** — production-planning: генерация Резка+Полоса+Обеспечение из ядра; встроенный редактор полос (авто сырьё/ширина, панель ходовых, сохранение Полос); CUT_REQ без Типа.
- **F4** — зачистка: orders.js (убрать фичу C + показать «Срок изготовления» неприметно), cut-calc (извлечь ядро редактора/ретайр), cut-map (полосы из Резки), доки.

**Порядок:** начинаем с **F2** (по решению Андрея). Полный F2 выполняется сразу;
живые РМ временно ломаются до деплоя F3/F4 — допустимо, atex в фазе настройки/теста.

### 1.2 Переподчинение Полосы (разрушающее для старого кода)

### 1.3 Удаление ссылок и таблицы Типа резки (разрушающее)

## 2. Порядок выполнения (минимизировать поломку и риск _d_)
1. **A1** (аддитивно, безопасно): «Срок изготовления» на Позицию и Заказ → проверить метаданные.
2. **R1, R3** (отчёты, источники): cut_knives→SUM Полос, убрать cut_type/position_cut_type → прогнать отчёты.
3. **Черновой тест механизма переподчинения** (throwaway таблицы) → зафиксировать рабочий `_d_*`.
4. **B1** (Полоса→Резка) на боевой → проверить метаданные 1078 и Полосы.
5. **C1, C2** (убрать ссылки Тип резки) → проверить.
6. **R2** (дроп cut_types_index).
7. **C3** (дроп таблицы Тип резки 1072) → финальная проверка: метаданные целы, отчёты идут.

После каждого `_d_*` — повторный `?JSON_META`/прогон отчёта; при искажении `val`/metadata —
чинить по [[feedback_integram_schema_d_commands]] (`_d_save/{tableId} val=Имя&t=3`).

## 3. Критерии приёмки F2
- Позиция (1076) и Заказ (1075) имеют реквизит «Срок изготовления» (DATE), проверено метаданными.
- Полоса (1073) — подчинённая **Производственной резки (1078)**, НЕ Типа резки; поля Ширина/Количество/Назначение целы; `object/1078/?JSON_META` показывает Полосу в подчинённых.
- Реквизиты Позиция/Резка «Тип резки» удалены; таблица «Тип резки» (1072) удалена; метаданные всех затронутых таблиц валидны (workplaces резолвят имена без ошибок).
- `report/cut_planning?JSON_KV`: `cut_knives` = сумма Количества полос резки; колонки `cut_type` нет; отчёт исполняется без ошибок.
- `report/positions_list` и `order_pipeline` без `position_cut_type*`; исполняются.
- `cut_types_index` удалён; `preferable_widths` исполняется без изменений.

## 4. Вне F2
- Чистое ядро раскладки (F1), генерация и редактор полос в production-planning (F3),
  удаление фичи C в orders.js + показ «Срока изготовления» на форме + cut-map по своим
  полосам + доки (F4). Никакого кода рабочих мест в F2 — только схема и отчёты на live
  и этот документ.

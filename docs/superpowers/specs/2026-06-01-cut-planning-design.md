# Подзадача B: РМ «Расчёт резки» (Диспетчер) — подбор комбинаций

**Дата:** 2026-06-01
**Источник:** ideav/atex#52 «Планирование резок», задача B («рабочее место расчёта
резки … подбирать комбинации и количества … минимизировать отходы — остаток ≤20мм …
неиспользованную ширину пустить на ходовые позиции … только Диспетчер сохраняет …
дубли комбинаций не допускаются»).
**Эпик:** [[atex_epic52_planning]]. Соседние: A (PR #3062), C (PR #3066).

## Контекст и проблема

Диспетчеру нужно из джамбо заданной ширины подобрать раскрой: набрать заказанную
ширину ролика максимально плотно и **добрать остаток ходовыми ширинами**, чтобы отход
был ≤ допуска (20 мм). Излишек идёт на склад как ходовые позиции впрок. Подобранную
комбинацию Диспетчер сохраняет как «Тип резки» (с полосами); дубли не допускаются.

Сейчас есть РМ «Калькулятор типов резки» (`cut-calc.js`) — ручной ввод полос и расчёт
«Итого ножей»/«Остаток, мм». Авто-подбора комбинаций и запрета дублей нет.

## Решения (согласованы)

1. **Новое РМ** «Расчёт резки» (Диспетчер) — отдельные template/js/css/пункт меню,
   рядом с cut-calc. Переиспользует чистое ядро расчёта из `cut-calc.js`
   (`usedWidth`/`remainder`/`computeSummary`).
2. **Ходовые ширины — серверный отчёт.** Создаём в боевой ateh запрос
   `preferable_widths` (по образцу `preferable_cut_combs`, queryId 8341): над
   «Позиция заказа» — ширина (группировка) + `SUM(Кол-во)` по убыванию, фильтр по
   виду сырья. Рабочее место зовёт его **по имени**: `report/preferable_widths?JSON_KV&FR_position_material_id={matId}`.
3. **Вход подбора:** Вид сырья (ширина входа = `Вид сырья.Ширина, мм`) + целевая
   ширина ролика. Допуск 20 мм (по умолчанию).
4. **Критерий подбора:** минимизировать отход (остаток ≤ допуска); при равном
   отходе — предпочесть более ходовые ширины (бо́льшая `SUM(Кол-во)`).
5. **Сохранение как «Тип резки»** (+ «Полоса»): полосы целевой ширины — назначение
   «Заказ», добор — «Склад». **Запрет дублей**: дубль = тот же `Вид сырья` И тот же
   мультинабор полос `{ширина×количество}` (порядок неважен) среди существующих типов.

## Чистое ядро подбора (тестируемое)

В новом РМ — экспортируемый объект `calc` (или переиспользование `AtexCutCalc.calc`
+ новые функции), покрытый модульными тестами `experiments/atex-cut-planning.test.js`.

`suggestCombination(inputWidth, targetWidth, candidates, tolerance)`:
- `candidates`: `[{ width, freq }]` — ходовые ширины этого сырья, отсортированы по
  убыванию `freq` (из отчёта). Целевая ширина может и не быть в `candidates`.
- Шаг 1: набрать целевую ширину — `nTarget = floor(inputWidth / targetWidth)`
  полос (назначение «Заказ»); `rem = inputWidth − nTarget·targetWidth`.
- Шаг 2: добрать `rem` ходовыми (назначение «Склад») так, чтобы итоговый отход был
  минимален и ≤ `tolerance`. Перебор ограничен (rem < targetWidth, конечный список
  ширин) — точный поиск по минимуму отхода; при равном отходе выбирается вариант с
  бо́льшей суммой `freq` задействованных ширин (более ходовые).
- Возврат: `{ strips:[{width, qty, purpose}], used, remainder, withinTolerance }`,
  где `purpose ∈ {Заказ, Склад}`. Если уложиться в допуск нельзя — вернуть лучший
  (минимальный) отход с `withinTolerance=false` (Диспетчер решает).

Точный алгоритм добора и тай-брейки фиксируются тестами в плане (TDD).

## Дедупликация при сохранении

`combinationSignature(materialId, strips)` → канонический ключ: `materialId` +
отсортированный список `ширина×количество` (числа округляются `round3`). Перед
сохранением — загрузить существующие «Тип резки» этого сырья и их «Полоса»
(`object/{Полоса}?F_U={typeId}`), построить сигнатуры; если совпадение есть —
**блокировать** сохранение и показать, какой тип дублируется. (Чистая функция
сигнатуры — тестируемая.)

## Рабочее место (DOM-слой)

Новое РМ `templates/atex/cut-planning.html` + `download/atex/js/cut-planning.js`
+ `download/atex/css/cut-planning.css` + пункт в `docs/atex_menu.json` + строки в
`update.conf` (деплой atex→ateh), роль Диспетчер. Следовать
`docs/WORKSPACE_DEVELOPMENT_GUIDE.md` (шаблонизатор, версионирование, `_m_*`, XSRF,
модалки вместо alert).
- Поля: Вид сырья (searchable-ref), целевая ширина, допуск (дефолт 20), кнопка
  «Подобрать».
- «Подобрать» → `ensure`-загрузка ходовых (отчёт по сырью) → `suggestCombination`
  → таблица полос (ширина/кол-во/назначение) + сводка (занятая ширина, остаток,
  в допуске?). Полосы редактируемы вручную (как в cut-calc).
- «Сохранить как тип резки» → проверка дубля → `_m_new/{Тип резки}` + полосы
  `_m_new/{Полоса}` (переиспользовать паттерн сохранения из `cut-calc.js`).

## Критерии приёмки / тесты

- Отчёт `preferable_widths` создан в live ateh; `report/preferable_widths?JSON_KV&FR_position_material_id={matId}` отдаёт ширины сырья по убыванию `position_qty_sum`; путь задокументирован.
- `suggestCombination`: пример из ТЗ (вход 880, цель 60 → `60×14 + 40×1`, остаток 0);
  добор несколькими ширинами (`156×5 + 60×1 + 40×1`); тай-брейк min-отход → более
  ходовые; недостижимый допуск → лучший отход + `withinTolerance:false`.
- `combinationSignature`: одинаковые наборы в разном порядке → один ключ; разные
  количества/ширины/сырьё → разные ключи.
- Дедуп блокирует сохранение дубля и пропускает новую комбинацию.

## Деплой

- Клиентский код РМ → atex→ateh через `update.php` (Андрей); новый пункт меню.
- Серверный отчёт `preferable_widths` создаётся в live ateh (агентом по токену) —

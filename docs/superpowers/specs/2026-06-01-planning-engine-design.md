# Подзадача D2: движок упорядочивания очереди резок (чистое ядро)

планирования — чистая, тестируемая функция упорядочивания. Стекается на D1 (PR #3072:

**Где:** ядро `planning` в `download/atex/js/production-planning.js`. Без DOM/сети.
**Эпик:** [[atex_epic52_planning]]. Готово: A #3062, C #3066, B #3069, E #3070, D1 #3072.

## Контекст и решения (согласованы)

Движок упорядочивает резки одного станка за день, минимизируя трудоёмкость переналадок.
Решения: подход — **жадный (nearest-neighbor)** по стоимости переналадки; приоритет
**сырьё > намотка > ножи**; **остаток джамбо учитываем** (через непрерывность партии +
штраф за неудобный остаток). Алгоритм — эвристика; диспетчер правит результат (D3).

D2 — только чистый движок (упорядочивание заданного набора). Наполнение реальными
данными, генерация резок, кнопка «Запланировать», ручная правка — **D3**.

## Дескриптор резки (вход движка)

```
{ id, materialId, winding, knifeCount, knifeWidths:[Number,…], isFoil:Boolean,
  batchId, jumboRemainingM:Number, rollerWidth:Number }
```
- `materialId` — id «Вида сырья»; `winding` — 'IN'|'OUT'|''; `knifeCount` — Итого ножей;
  `knifeWidths` — ширины полос (для расхождения конфигурации ножей); `isFoil` — сырьё Фольга;
  `batchId` — id «Партии сырья» (джамбо); `jumboRemainingM` — остаток метража джамбо после
  этой резки; `rollerWidth` — ширина ролика (для «узкие→широкие»). D3 заполнит из реальных
  данных; D2 не лезет в сеть/схему.

## Функции ядра (экспорт в `planning`)

### Настраиваемые веса приоритета (в коде, значения 10..100)
Веса вынесены в **именованный объект-настройку** в ядре `planning`, чтобы приоритет
менялся правкой чисел (диапазон 10..100; больше число — дороже/нежелательнее эта смена):
```javascript
// Приоритет планирования: правь эти числа (10..100), чтобы изменить важность.
// Больше — тем дороже соответствующая переналадка. По умолчанию:
// сырьё > намотка > партия(джамбо) > остаток > ножи > ширина.
var PLANNING_WEIGHTS = { material: 100, winding: 70, batch: 50, remainder: 40, knife: 25, width: 10 };
// Масштабы нормировки непрерывных компонент (вторичны):
var KNIFE_SCALE = 8;    // при скольких «переставленных ножах» knife-компонента ≈ максимум
var WIDTH_SCALE = 100;  // при каком «сужении, мм» width-компонента ≈ максимум
```
Экспортировать `PLANNING_WEIGHTS` (и масштабы) в `planning`; `changeoverCost`/`orderCuts`
принимают `weights` опционально, по умолчанию — `PLANNING_WEIGHTS`.

### `changeoverCost(prev, next, weights)` → Number
**Взвешенная сумма с нормировкой каждой компоненты в 0..1** (чтобы веса 10..100 честно
выражали приоритет). Чем больше — тем хуже:
```
cost = 0
  + w.material   * (prev.materialId !== next.materialId ? 1 : 0)              // бинарно
  + w.winding    * (norm(prev.winding) !== norm(next.winding) ? 1 : 0)        // бинарно
  + w.batch      * (prev.batchId !== next.batchId ? 1 : 0)                    // бинарно
  + w.remainder  * (prev.batchId !== next.batchId && awkwardRemainder(prev.jumboRemainingM) ? 1 : 0)
  + w.knife      * min(1, (abs(prev.knifeCount - next.knifeCount) + widthSetDistance(prev.knifeWidths, next.knifeWidths)) / KNIFE_SCALE)
  + w.width      * min(1, max(0, prev.rollerWidth - next.rollerWidth) / WIDTH_SCALE)
```
- Модель — **взвешенная сумма** (не строгий лексикографический порядок): относительные
  величины весов задают приоритет, и их можно перенастроить в коде (10..100). При дефолтах
  одиночная смена сырья (100) дороже смены намотки (70) дороже партии/ножей — но это
  настраиваемо.
- `awkwardRemainder(m)` = `0 < m < REMAINDER_OK_M` (по умолчанию 600) — «неудобный» остаток
  (рулон не дорезан до ≈0 и не оставлен крупным >600 м). m≈0 (≤ EPS) и m≥600 → не штрафуется.
- `widthSetDistance(a,b)` = размер симметрической разности мультимножеств ширин (сколько
  ножей надо переставить). Терпимо к числам/строкам/округлению.
- `norm(winding)` — нормализация ('IN'/'OUT'/'' к верхнему регистру).

### `orderCuts(cuts, weights?)` → массив резок в порядке (с полем `sequence` 1..N)
1. Разделить на `rest` (не Фольга) и `foil` (isFoil).
2. Для каждой группы — жадная последовательность:
   - старт: самая «узкая/много-ножевая» — `argmin (rollerWidth, -knifeCount, id)` (реализует
     начало дня с узких/много ножей);
   - далее на каждом шаге `next = argmin changeoverCost(current, candidate)`; при равенстве —
     меньший детерминированный ключ `(rollerWidth, -knifeCount, id)`.
3. Конкатенация `rest ++ foil` (Фольга всегда в конце).
4. Проставить `sequence` = 1..N в полученном порядке; вернуть массив резок (копии с `sequence`).
   Вход не мутировать.

## Критерии приёмки / тесты (TDD, `experiments/atex-production-planning.test.js`)

- `widthSetDistance([60,60,40],[60,40,40])` → 2 (одна 60 и одна 40 различаются по кратности);
  пустые/равные → 0; терпимость к '60' vs 60.
- `awkwardRemainder`: 0→false, 100→true, 600→false, 1200→false, отрицат→false.
- `changeoverCost` (дефолтные веса): одиночная смена сырья (100) дороже одиночной смены
  намотки (70) дороже максимальной смены ножей (25) — проверить на сконструированных парах;
  одинаковые резки → 0.
- **Настраиваемость весов:** с кастомным объектом весов (напр. `{winding:100, material:50, …}`)
  `orderCuts` группирует сперва по намотке, а не по сырью — доказывает, что приоритет
  меняется правкой `PLANNING_WEIGHTS` (значения 10..100). `PLANNING_WEIGHTS` экспортирован.
- `orderCuts`:
  - резки трёх разных сырьёв группируются по сырью (одинаковые подряд — число границ сырья
    минимально);
  - внутри сырья — группировка по намотке;
  - внутри намотки — «узкие→широкие» (rollerWidth не убывает там, где нет других различий);
  - Фольга — строго в конце;
  - резки одного джамбо идут подряд (batch-непрерывность);
  - детерминизм (стабильный результат при равных стоимостях); вход не мутируется;
  - `sequence` проставлен 1..N.

## Вне объёма D2 (это D3)

- Заполнение дескрипторов из реальных данных (тип резки → ножи/ширины; партия → сырьё/остаток;
  позиция → намотка; Фольга по виду сырья; стоп-лист E при назначении на станок).
- Кнопка «Запланировать» (запуск движка по станкам), сохранение «Очередности» в резки.
- Генерация резок из необеспеченных позиций заказов.
- Ручная правка очереди (reorder).

# Issue #3280 — разбиение резок по дням, плановое время старта, колонка «Назначение»

Фоллоу-ап к закрытому issue #3278 / реверченному PR #3279 (#3281). Рабочее место
«Планирование производства» (`download/atex/js/production-planning.js`).

## Контекст реверта

PR #3279 закрывал #3278, но был реверчен. Из 4 частей корректна одна:

| Часть | Статус | Причина |
|---|---|---|
| t1078 = время старта | ❌ | Менялась только база `nextCutMainValue` (08:00); механизм `Math.max(…, last+1)` сохранён → по-прежнему «+1 сек на резку». Настоящее время старта не пишется. |
| Лимит дня | ❌ | Резки, не влезающие в день, выбрасывались целиком. Нужно: обрезать по проходам, остаток продолжить на след. день. |
| «Назначение» (Заказ/Склад) | ❌ | Логика брала `self.supplies[].finishedBatchId`, но отчёт `cut_planning` не отдаёт ссылку Обеспечение→Партия ГП → всё «Склад». CSS-грид `1fr 1fr 1fr 40px` не расширен под 5-ю колонку → «кривая вёрстка». |
| Приоритет по сроку (`dueKey`) | ✅ | Переиспользуем. |

## Требования (решения Андрея)

1. **Разбиение резки между днями.** Резка, упирающаяся в конец рабочего дня
   (08:00–16:30, далее уборка), обрезается по числу влезающих **проходов**; остаток
   проходов продолжается **с 08:00 следующего рабочего дня той же резкой, без
   переналадки** (ножи остаются на станке). «Заглянуть во вчера» = планируя следующий
   день, станок продолжает недоделанную резку.
   - Хранение: **две отдельные записи** «Производственной резки» (N проходов сегодня +
     остаток завтра), с делением Обеспечения и копией Полос.
   - Идемпотентность при повторном «Запланировать»: **без схемного маркера** — продолжения
     распознаются эвристикой (тот же станок + сырьё + набор ножей/полос + соседний рабочий
     день, продолжение стартует ровно в 08:00). Перед планированием такие цепочки сливаются
     в одну логическую резку и пере-разбиваются. Остаточный риск (ручное вмешательство в
     записи) задокументирован в PR.

2. **Плановое время старта в t1078.** В первую колонку «Производственной резки» (DATETIME,
   главное значение) писать **плановое время старта именно этой резки** = начало окна
   тайминга (`startMin − setupMin` из `buildSchedule`) как Unix-штамп. Пишется на этапе
   планирования («записать потом»), не при создании.
   - `.atex-pp-cut-num` показывает то же время, что начало `.atex-pp-cut-time` (ЧЧ:ММ);
     `title` = дата + время до минут.

3. **Колонка «Назначение» в строках Полос (Партия ГП).** «Заказ», если на запись Полосы
   есть ссылка из Обеспечения; иначе «Склад».
   - Источник ссылки: добавить в отчёт `cut_planning` колонку `supply_finished_batch_id`
     (abn_ID реквизита Обеспечение→«Партия ГП»). `rowsToPlanning` уже читает это имя.
   - Вёрстка: `grid-template-columns` строк `.atex-pp-strip-row` → 5 колонок.

## Чистое ядро (тестируемо без DOM/сети)

- `windingMinutes(runMeters, points)` — минуты на 1 проход (уже есть; длительность линейна
  по проходам: `plannedCutDurationMinutes = windingMinutes × runs`).
- **`splitMachineQueue(orderedCuts, opts)`** — вход: упорядоченная очередь одного станка
  (одна логическая последовательность), `opts = {dayStartMin, dayEndMin, leader, times,
  perPassByCut, runsByCut}`. Выход: массив сегментов
  `[{cutId, dayOffset, runs, startMin, setupMin, isContinuation, parentCutId}]`.
  Проход за проходом укладывается в окна по дням; переход через конец дня → продолжение
  на след. день с `setupMin=0` (ножи остаются).
- **`mergeContinuationChains(cuts)`** — эвристическое слияние записей-продолжений обратно в
  логические резки до пере-разбиения (станок+сырьё+ножи+соседний день+старт 08:00).
- **`scheduleStartTimestamp(dayMidnightMs, windowStartMin)`** — минуты расписания → штамп.

## Живая запись (Фаза B, после подтверждения)

- Отчёт `cut_planning`: +колонка `supply_finished_batch_id`.
- `doRun`/`saveSequences`: `_m_set t1078` = штамп старта для каждой резки; создание/удаление
  записей-продолжений; деление Обеспечения по проходам; копия Полос на продолжение.

# Доработка РМ «Приём и ведение заказов» (`orders`)

Рабочее место `templates/atex/orders.html` + `download/atex/js/orders.js`
(роль **Менеджер**). Пять доработок: редактируемые позиции, подбор «Тип резки»
при правке, фильтр по датам + быстрый поиск + фильтр статуса в одной строке,
сортировка по колонкам заказа, перевод чтений на отчёт `report/`.

## Контекст и проблема

Сейчас (`orders.js`, 1384 строки, читает напрямую `object/_m_*`):

- Позиции заказа **только просматриваются**; редактируется лишь статус позиции
  (`_m_set`). Добавить позицию можно через форму.
- Подбор «Тип резки» по виду сырья + ширине (`matchCutTypes` →
  `data-atex-cuttype-allowed`, PR #3066) работает **только в форме добавления**
  позиции; к правке существующих не подключён (правки нет).
- Тулбар: фильтр **только по статусу**. Дат, поиска, сортировки нет.
- Данные грузятся прямыми чтениями таблиц (отчёта нет — помечено «следующий этап»).

Поля (из metadata, резолв по имени):
- **Заказ (1075):** Клиент(1125), Пользователь/Менеджер(1126), **Дата создания(1128, date)**,
  Дата согласования(1130, date), Статус(1131), Лидер(1133), Примечания(1134),
  Позиция заказа(1135, подчинённая).
- **Позиция заказа (1076):** Кол-во, Вид сырья(ref), Тип резки(ref), Ширина мм,
  Длина м, Диаметр втулки(ref), Тип намотки, Статус.

## Решения (согласованы)

1. **Архитектура фильтрации — гибрид.** Даты и статус — серверный фильтр отчёта
   (перезапрос). Быстрый поиск и сортировка — клиентские, по уже загруженным строкам.
2. **Редактирование позиций — inline в ячейках** (поля прямо в строке,
   «Сохранить»/«Отмена») + удаление позиции.
3. **Быстрый поиск — по всему** видимому: № заказа, клиент, поля позиций.
4. **Дата фильтра — «Дата создания»** (1128). С = вчера по умолчанию, По — пусто.
5. **Сортировка — только по колонкам заказа** (не позиций).
6. Запись остаётся прямыми командами `_m_*`; отчёт — только чтение.

## Отчёт `orders_list` (создаётся на бою)

Мастер **Заказ (1075)** → подчинённая **Позиция заказа (1076)**, плоский LEFT JOIN
(как `cut_planning`/`order_pipeline`): один ряд на позицию, заказы без позиций
остаются (пустые `position_*`). Имена колонок — латиница snake_case с префиксом сущности.

| Колонка | Источник | Назначение |
|---|---|---|

| `order_no` | 1075 главное значение | № заказа |
| `order_client` | Клиент (1125, ref) | подпись клиента |
| `order_manager` | Пользователь (1126, ref) | менеджер |
| `order_created` | Дата создания (1128, date) | **фильтр FR_/TO_, сортировка** |
| `order_approved` | Дата согласования (1130, date) | колонка |
| `order_status` | Статус (1131) | **фильтр (точное), колонка** |
| `position_id` | 1076 abn_ID (t104=85) | ключ позиции, цель `_m_set`/`_m_del` |
| `position_qty` | Кол-во | правка |
| `position_raw` | Вид сырья (ref) | правка, вход для `matchCutTypes` |

Запуск: `report/orders_list?JSON_KV&FR_order_created={C}&TO_order_created={По}` (+
фильтр статуса при выбранном; точный механизм exact-фильтра по тексту статуса
валидируем при создании отчёта). id-колонки (`order_id`, `position_id`) — через
функцию `abn_ID` (t104=85), как в `cut_planning`. Грант на объект 22 не нужен (доступ по нижележащим
таблицам; роль Менеджер имеет к ним доступ — проверить при создании, как для `cut_planning`).

Не переиспользуем `order_pipeline` (8303): он уходит вглубь
Обеспечение→Резка→ГП и множит ряды; для РМ нужен ровно 2-уровневый Заказ→Позиция.

## Поток данных (`orders.js`)

1. **Загрузка:** `loadOrders()` → `report/orders_list?JSON_KV` с серверными
   фильтрами (даты + статус). Чистая `rowsToOrders(rows)` разворачивает плоские
   ряды в `[{order, positions:[…]}]` (dedup заказов по `order_id`, позиции из рядов
   с непустым `position_id`) — зеркало `rowsToPlanning`/`rowsToEntities`.
2. **Клиентский слой над загруженными заказами:** `searchOrders(list, query)` (по
   всем полям заказа и его позиций), `sortOrders(list, key, dir)` (по колонке заказа).
   Применяются к результату `rowsToOrders` перед рендером; перезапрос не нужен.
3. **Рендер:** существующий `renderOrders`/`renderPositions` (таблица + разворот
   позиций), расширенный режимом inline-правки.
4. **Запись** (как сейчас, прямые команды; после успеха — `loadOrders()` перезапрос):
   - правка позиции — `_m_set/{position_id}` по изменённым `t{reqId}`;
   - удаление позиции — `_m_del/{position_id}`;
   - добавление позиции — без изменений;
   - смена статусов — без изменений.

## UI

### Тулбар — одна строка (#3)
`[+ Создать заказ]  [Статус ▾]  [С: <дата=вчера>]  [По: <дата>]  [🔍 поиск…]  [⟳ Обновить]`
- Изменение **статуса / С / По** → перезапрос отчёта (сервер).
- Ввод в **поиск** → мгновенная клиентская фильтрация (debounce ~200мс).
- «С» инициализируется вчерашней датой; «По» пусто (без верхней границы).

### Inline-правка позиций (#1 + #2)
- В строке позиции — кнопки **«✎»** (правка) и **«🗑»** (удалить).
- По «✎» ячейки позиции становятся редактируемыми: Кол-во, Вид сырья (ref-select),
  **Тип резки (фильтруемый ref-select)**, Ширина, Длина, Диаметр втулки (ref-select),
  Тип намотки, Статус; кнопки **«Сохранить»/«Отмена»**.
- **#2:** select «Тип резки» в режиме правки фильтруется через существующий
  `matchCutTypes(state.cutTypeIndex, materialId, width)` — показываются только типы,
  совпадающие по виду сырья и ширине текущей позиции (та же механика
  `data-atex-cuttype-allowed`, что в форме добавления). При смене вида/ширины в
  процессе правки список пересчитывается.
- «Сохранить» → `_m_set` по изменённым полям → перезапрос → выход из режима правки.

### Сортировка (#4)
- Клик по заголовку колонки заказа (№, Клиент, Дата создания, Статус, …) →
  `sortOrders` asc/desc, индикатор стрелкой. Клиентская, по загруженным заказам.
  Позиции внутри заказа не сортируются (вне объёма).

## Чистое ядро и тестируемость

Новые чистые функции экспортируются (как уже сделано для `matchCutTypes` и др.),
юнит-тесты в `experiments/atex-orders.test.js` (расширяем существующий):
- `rowsToOrders(rows)` — плоский отчёт → заказы+позиции (dedup, пустые позиции);
- `searchOrders(list, query)` — фильтр по всем полям (заказ + позиции);
- `sortOrders(list, key, dir)` — сортировка заказов по колонке (числа/даты/текст);
- (переиспользуем) `matchCutTypes`.

Крупные новые блоки (inline-edit, загрузка через отчёт) держим компактными; при
росте `orders.js` — выделять связные части, не плодя дублирование.

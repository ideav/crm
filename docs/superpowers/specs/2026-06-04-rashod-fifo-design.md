# Дизайн: «Расход сырья» с FIFO вместо ссылки «Партия сырья» (ideav/crm#3120, группа C)

Статус: дизайн на согласование. Затрагивает схему (бэкап обязателен), отчёты и логику РМ
«Планирование производства» (`templates/atex/production-planning.html` + `js/production-planning.js`).

## Зачем
Сейчас у резки одна ссылка **«Партия сырья»** (`Производственная резка`.req **1159** → `Партия сырья` 1074).
Нужно (issue #3120):
- **п.2** — убрать «Партия сырья» из таблицы резки;
- **п.3** — учитывать сырьё через подчинённую **«Расход сырья»**, где может быть **несколько записей по разным партиям**, подбираемых по **FIFO**;
- **п.4** — подсвечивать карточку резки светло-красным, если **нет подходящей партии** или **не удалось зарезервировать** нужный объём записями «Расход сырья».

## Что есть на боевой схеме (проверено через API)
- **«Расход сырья» = таблица `1079`**, подчинённая резке (req **1172** на `1078`, `type=13 arr_id=1079`, `up=cut`). Поля:
  - `1175` **Израсходовано, м²** (число);
  - `1176` **Партия сырья** (ref → 1074).
  - Записей пока нет (таблица не использовалась).
- **«Партия сырья» 1074**: `1117` Вид сырья (ref→1069), `1119` Дата прихода, `1121` Получено, м², `1123` Остаток, м², `8454` Длина, м, `8456` Остаток, м.
- **«Вид сырья» 1069**: `1084` Полное название, `1086` Ширина, мм (= ширина джамбо), `1088` Длина рулона, м.
- Отчёт `cut_planning` (8384) берёт материал/партию резки **из ссылки 1159**:
  - `cut_material_batch` = `1159:… -> Партия сырья`
  - `cut_material` / `cut_material_id` = `1117:Партия сырья -> Вид сырья`
  - `cut_batch_id` = `1159 … -> Партия сырья` (abn_ID)
  → при удалении 1159 (п.2) эти 4 колонки ломаются и должны переисточняться из «Расход сырья».

## Целевая модель
Источник истины по сырью резки — **записи «Расход сырья» (1079)**, каждая = `{ Партия сырья, Израсходовано м² }`.
- **Вид сырья резки** = вид сырья её партий расхода (все записи одной резки — один вид сырья).
- Целевой **вид сырья** берётся из **обеспечиваемых позиций** резки (`Обеспечение → Позиция → Вид сырья`); все позиции резки должны быть одного вида сырья (иначе резка некорректна — кандидат на подсветку).

## FIFO-резервирование (п.3)
Вход: резка с обеспечениями (позициями) одного вида сырья `M`; требуемый объём `N` м² (см. «Открытый вопрос»).
Алгоритм:
1. Кандидаты — партии 1074 вида `M` с `Остаток, м² > 0`, отсортированные по `Дата прихода` ↑ (FIFO; тай-брейк — меньший id).
2. Идём по партиям, на каждую создаём запись «Расход сырья» (`up=cutId`, `t1176=batchId`, `t1175=min(остаток_партии, остаток_потребности)`), уменьшаем остаток потребности.
3. Пока потребность не закрыта и есть партии — продолжаем. Если партий не хватило → резка **не обеспечена** (подсветка п.4).
4. **Остаток партии**: уменьшать `Остаток, м²` партии на зарезервированное — или считать «свободный остаток» = `Остаток − Σ резервов` динамически. *(Решение ниже — см. «Открытый вопрос 2».)*

**Триггер (решено):** авто при **создании/планировании** резки. На практике — резка обычно создаётся без обеспечений, поэтому резерв строится:
- в `generateCuts` (автогенерация уже создаёт резку+полосы+обеспечения — добавляем шаг резерва);
- при **прогоне планирования** / по изменению обеспечений (когда у резки уже есть позиции).
Повторный резерв = снести прежние записи «Расход сырья» резки и построить заново (идемпотентно).

## Изменения отчётов
- **Новый отчёт `cut_consumption`** (cut→Расход 1079→Партия 1074[→Вид сырья 1069]). Колонки:
  `cut_id` (abn_ID), `cons_batch_id`, `cons_batch_label`, `cons_material_id`, `cons_material`, `cons_m2`.
  Клиент читает его отдельно (как `cut_strips`/`stripAgg`) и сливает в каждую резку:
  `cut.materialId`, `cut.materialName`, `cut.batches=[{id,label,m2}]`, `cut.reservedM2=Σ`.
- **`cut_planning`**: убрать 4 колонки, завязанные на 1159 (`cut_material_batch`, `cut_material`,
  `cut_material_id`, `cut_batch_id`). Материал/партии берём из `cut_consumption`.
  Подпись партии в карточке = склейка меток партий (или «N партий»).

## Фазы реализации

## Решения (подтверждены)
1. **Потребность — в погонных метрах**, кв.м пересчитываются из них справочно.
   `N_пог = max(«Метраж, м») по обеспечениям резки` (параллельный слиттинг: один джамбо
   разматывается на длину прогона = самая длинная позиция; все полосы режутся разом).
   FIFO расходует **погонный остаток** партий (`Остаток, м`). В записи «Расход сырья»
   `Израсходовано, м²` = `зарезервировано_пог × ширина_джамбо(м)` (справочно).
2. **Свободный остаток — на лету.** Резерв ≠ списание: `свободно_пог = Остаток, м − Σ активных
   резервов по партии`. Поле `Остаток, м/м²` партии **не изменяем** (полностью обратимо).

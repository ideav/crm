# База знаний Integram (`docs/kb/`) — дизайн

**Дата:** 2026-06-02
**Зачем:** `docs/integram-app-workflow.md` разросся (2566 строк) и мешает и человеку, и
агенту. Нужна **переносимая база знаний в репозитории** `ideav/crm/docs/kb/`, которую
любой агент на любом сервере читает по теме и **дополняет по ходу** (граблями/рецептами).
Знания сейчас в голове Андрея и в коде/данных проектов — задача перенести их агентам в
доступном, ищущемся виде. Андрей один — агентов много, источник истины должен жить в репо.

## Принципы
- **Человек-сначала → потом агент.** Каждый док начинается коротким человекочитаемым
  введением (что это, когда нужно, «на пальцах»), затем явная граница, после которой —
  точные операции/команды для агента. Человек читает верх, говорит «понял, дальше агент».
- **Один файл = одна тема.** Мелкие фокусные доки вместо монолита.
- **Ищется по ключам.** Индекс-каталог (тема → файл + ключи) + сводный индекс граблей.
- **Дополняемо отовсюду.** Единый шаблон записи граблей + правило «нашёл грабли → допиши».
- **Без дублей.** Сильные существующие доки (integram-reports, StructureRules, MCP, dir_admin)
  не копируем — индекс на них ссылается; из монолита переносим то, чего в них нет.

## Структура `docs/kb/`
- **`README.md`** — каталог KB: таблица «тема → файл → ключи»; **сводный индекс граблей**
  (симптом/ключ → тема/файл); раздел **«Как дополнять»** (шаблон + правило).
- **`00-start.md`** — онбординг (макс. человеко-ориентированный): авторизация
  (токен `X-Authorization` + cookie `idb_{db}` + `_xsrf`; пароль/Bearer не работают),
  модель данных (БД/таблица/реквизит/ссылка/подчинённая), как слать команды (`_m_*`/`_d_*`/`report/`),
  где что лежит. Связь: WORKSPACE_DEVELOPMENT_GUIDE, integram-app-workflow (обзор цикла).
- **`schema.md`** — DDL `_d_*`: создать/изменить таблицу, реквизит, ссылку (`_d_ref`+`_d_req`),
  подчинённую таблицу; **удаление** (реквизит `_d_del_req`, таблица `_d_del`; ref-тип
  бывшего справочника `referenced`; снять гранты/колонки отчётов перед дропом; порядок!);
  маски (DIR_MASK/FILE_MASK), типы колонок. Связь: StructureRules.md.
- **`crud.md`** — DML `_m_*`: создание/правка/удаление записей; **главное значение = `t{tableId}`**
  (не `t3`); `up=` для подчинённых; чтение `object/{tbl}/?JSON_OBJ|JSON_KV`, фильтры `F_U`/`F_I`,
  окна `LIMIT,N`; multi-ref; массовый импорт.
- **`queries.md`** — отчёты `report/`: создание колонок (`_m_new/28`, `t28`/`t100`/`t104`/`t109`),
  форматы `JSON_KV`/`JSON_DATA`/`JSON`, фильтры `FR_`/`TO_` (+ операторы `>`/`<`), SET-запросы,
  колонка-формула как серверный фильтр. **Решение по дублю:** `queries.md` = человеко-вводная
  KB-страница + ссылка на `integram-reports.md` как на полный справочник; уникальные детали
  из монолита переносятся в `integram-reports.md` (а не копируются в kb).
- **`roles.md`** — роли и гранты: объект-разрешения, `t116`/`t136` (READ/WRITE), встроенный
  объект **FILE** (`BT["FILE"]`, гейтит dir_admin), права на таблицы/отчёты, «не выдавать массово».
- **`files.md`** — `dir_admin` файловый API (del[]/mkdir/touch/upload/gf; auth; ?JSON=1) +
  безопасность (traversal, allowlist, no-exec в download/templates). Источник: уже есть раздел
  в integram-app-workflow + memory — перенести/слинковать.
- **`deploy.md`** — `update.php`/`update.conf` (тянет ideav/crm:main, только копирует, не удаляет;
  index.php не в update.conf), PR-флоу (форк **unidel2035**), git **worktree** в общем `/home/hive/crm`,
  ветки. Связь: UPDATE_SCRIPT.md, BRANCH_MAINTENANCE.md.

## Шаблон документа (обязательный)
```markdown
# <Тема>
> Часть базы знаний Integram. Индекс: docs/kb/README.md

## На пальцах (для человека)
5–10 строк: что это, когда нужно, основная идея. Без команд.

> ────────── дальше — справочник для агента ──────────

## Операции (для агента)
Точные вызовы: метод, путь, параметры, пример (curl/JSON), что в ответе.

## Грабли
- **Симптом:** … → **Причина:** … → **Фикс:** … *(дата, PR/issue)*
```

## Каталог + индекс граблей (в README)
- Таблица: `Тема | Файл | Ключи (для поиска при затыке)`.
- Сводный индекс граблей: `Симптом/ключ | Тема | Файл#якорь` — чтобы при ошибке найти решение из README за один взгляд.

## Судьба монолита `integram-app-workflow.md`
Остаётся как **обзор полного цикла + точка входа для человека** (на него много ссылок —
не ломаем). Детальные справочные блоки (метаданные, запросы, схемные команды, dir_admin,
удаление таблиц) **вырезаются в `kb/*`**, на их месте — короткое резюме + ссылка в `kb/`.
Док худеет, перестаёт мешать.

## Дополнение «из разных мест» (масштаб)
Правило (в `kb/README.md` и продублировать в `CLAUDE.md`/workflow):
> Наткнулся на грабли или нашёл рабочий рецепт → допиши в нужный `docs/kb/<тема>.md`
> (секция «Грабли» или «Операции») по шаблону + добавь строку в индекс `kb/README.md`.
> Источник истины — репо; любой агент на любом сервере пополняет одинаково.

## Реализация (фазами, один PR в ideav/crm)
1. `kb/README.md` + `00-start.md` + шаблон (скелет, индекс).
2. `schema.md`, `crud.md` (перенос из монолита).
3. `queries.md` (+согласование с integram-reports.md), `roles.md`.
4. `files.md`, `deploy.md`.
5. Ужать `integram-app-workflow.md` → резюме + ссылки в kb/; правило дополнения в CLAUDE.md.
Каждая фаза — самостоятельный коммит; после каждой KB остаётся связной.

## Критерии приёмки
- `docs/kb/README.md` — рабочий каталог (тема→файл→ключи) + сводный индекс граблей + «как дополнять».
- Темы созданы по шаблону (человек-сначала → агент → грабли); внутренние ссылки рабочие.
- Дублей с существующими доками нет (ссылки, не копии).
- Монолит ужат, ссылки на него не сломаны.
- Правило дополнения зафиксировано (README + CLAUDE.md).

## Вне объёма (на потом)
- Полное слияние/перемещение всех старых доков в kb/ (сейчас — ссылки).
- Перевод/EN-версия. Автоматическая проверка ссылок (линтер).

# Подзадача D1 (фундамент планирования): поля «Тип намотки» + «Очередность»

**Дата:** 2026-06-01
**Источник:** ideav/atex#52 «Планирование резок», задача 3 (алгоритм планирования).
Это **фундамент** для движка планирования D2/D3: данные, без которых движок не на
чём работать.
**Рабочие места:** «Заказы» (`orders.js`), «Планирование производства» (`production-planning.js`).
**Эпик:** [[atex_epic52_planning]]. Соседние готовы: A #3062, C #3066, B #3069, E #3070.

## Контекст и решения (согласованы)

Движок D2/D3 учитывает **тип намотки** (IN/OUT — менять при переналадке нежелательно)
и раскладывает резки в очередь станка через **«Очередность»**. Этих данных в схеме нет.
Решения: объём движка — генерация + упорядочивание (D2/D3); «Тип намотки» живёт на
**Позиции заказа**; начинаем с фундамента D1.

D1 добавляет два поля и их показ — без оптимизации.

## Изменения схемы (live ateh + docs/atex_metadata.json)

| Таблица (live id) | Поле | Тип |
|---|---|---|
| Позиция заказа (1076) | **«Тип намотки»** | строка (type 3), фикс-значения **IN / OUT** |
| Производственная резка (1078) | **«Очередность»** | число (type 13/14) |

«Тип намотки» — строковое поле с фиксированным выбором IN/OUT (как «Статус» в
orders.js: значение свободное, но UI предлагает фиксированный список). Справочник не
заводим (2 значения, лёгкость; engine сравнивает на равенство строки). Поля заводятся
на live через `_d_*` (см. [[feedback_integram_schema_d_commands]]); metadata-doc
обновляется.

## `orders.js` — «Тип намотки» у позиции

- В `POSITION_FIELDS` добавить поле `winding` (`names:['Тип намотки']`), рендерить как
  **select из фиксированного списка `['IN','OUT']`** (переиспользовать механизм
  фикс-enum, как у `status` через `positionStatuses`, либо отдельный select-хелпер).
- Сохранять значение в позицию (`t{reqId 'Тип намотки'}`), показывать в таблице позиций.
- Пустое значение допустимо (намотка не задана).

## `production-planning.js` — «Очередность» в очереди

- В отчёт `cut_planning` (live) добавить колонку **`cut_sequence`** (← реквизит
  «Очередность» резки), чтобы РМ получало её тем же отчётом (паттерн #3068 — читаем
  отчётом, не object/).
- `rowsToPlanning`/`mapCut` читает `cut_sequence` → `cut.sequence` (число; пусто → null).
- В `groupBySlitter` **сортировать резки внутри каждого станка по `sequence`** (по
  возрастанию; пустые — в конец), затем существующий tiebreak (по подписи/номеру).
- Показывать «Очередность» в строке резки очереди (число или «—»).

> Установку/правку «Очередности» (движок + ручной reorder) делает D3. D1 только
> читает, сортирует и показывает.

## Чистое ядро (тестируемое)

- `orders.js`: `normalizeWinding(value)` → `'IN'|'OUT'|''` (терпимо к регистру/пробелам;
  чужое → '') — экспорт в `AtexOrdersTesting`.
- `production-planning.js`: расширить `mapCutRecord` (читает `sequence`) и `groupBySlitter`
  (сорт по sequence) — покрыть в `experiments/atex-production-planning.test.js`:
  очередь сортируется по «Очередность» возрастанию, пустые в конец, tiebreak стабилен.

## Критерии приёмки / тесты

- Поля «Тип намотки» (Позиция заказа) и «Очередность» (Производственная резка) есть на
  live; metadata-doc обновлён.
- orders: в форме позиции есть select «Тип намотки» (IN/OUT), значение сохраняется и
  показывается; `normalizeWinding` корректна.
- planning: резки в очереди станка отсортированы по «Очередность» (пустые в конец),
  значение видно; существующие тесты production-planning зелёные.

## Деплой

- Клиентский код (`orders.js`, `production-planning.js`) → atex→ateh через `update.php` (Андрей).
- Схема: поля на live завожу через `_d_*` (Позиция заказа «Тип намотки» строка; Резка
  «Очередность» число) + колонка `cut_sequence` в отчёт `cut_planning` (8384) —
  агентом по токену. Идемпотентно (проверять наличие перед созданием).

## Вне объёма D1

- **Движок упорядочивания** (стоимость переналадок: ножи кол-во+ширина, не менять
  намотку/сырьё, остаток джамбо ≈0 или >600 м, Фольга в конец, узкие→широкие) — **D2**.
- **Генерация резок из необеспеченных позиций** + кнопка «Запланировать» + ручная
  правка очереди (reorder/сохранение «Очередности») — **D3**.

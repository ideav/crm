# Подзадача D3c: ручная правка очереди (кнопки ↑/↓)

**Дата:** 2026-06-01
**Источник:** ideav/atex#52 («Диспетчер … корректирует автоматически сгенерированный
план»). Замыкает петлю: D3a авто-раскладывает очередь, D3c даёт диспетчеру поправить.
**Где:** `download/atex/js/production-planning.js`. Стек на D3a (PR #3075).
**Эпик:** [[atex_epic52_planning]]. Готово: A,B,C,E,D1 (merged), D2 #3074, D3a #3075.

## Контекст и решения (согласовано)

После авто-планирования (D3a, кнопка «Запланировать») диспетчер должен иметь возможность
вручную переставить резки в очереди станка. Механизм — **кнопки ↑/↓** у каждой резки
(просто, надёжно, без drag-библиотек, ок на планшете/десктопе).

## Чистое ядро (тестируемое)

`moveInQueue(orderedCuts, index, dir)` → массив `[{cutId, sequence}]` изменившихся резок:
- `orderedCuts` — резки одного станка в текущем отображаемом порядке (как в очереди, т.е.
  отсортированы по «Очередности»); `index` — позиция перемещаемой; `dir` ∈ {-1 (вверх), +1 (вниз)}.
- `target = index + dir`; если вне границ `[0, len)` → **no-op** (вернуть `[]`).
- Построить новый порядок: поменять местами `index` и `target`.
- Нормализовать «Очередность» = позиция+1 (1..N) по новому порядку (это корректно и при
  null/пустых прежних «Очередностях»).
- Вернуть только те `{cutId, sequence}`, у кого новое значение отличается от прежнего
  (минимизировать `_m_set`).
- Чистая, детерминированная, вход не мутирует. Экспорт в `planning`.

## DOM-слой (production-planning.js)

В `renderQueue`, в группе станка: каждую карточку резки обернуть в строку
`.atex-pp-cut-row`, рядом с карточкой добавить кнопки **↑** и **↓** (`atex-pp-move`):
- у первой резки станка ↑ задизейблена; у последней ↓ задизейблена;
- по клику: `moveInQueue(g.cuts, i, dir)` → для каждого `{cutId, sequence}` сохранить
  «Очередность» (`_m_set`, reqId по имени — как в D3a `runPlanning`) → `reload()`;
- `setBusy`/`notify` как в D3a; клики игнорируются при `this.busy`.
Карточка-кнопка (выбор резки для привязки обеспечения) не меняется — кнопки ↑/↓ вне её.
Желательно вынести сохранение «Очередности» в общий метод (напр. `saveSequences(pairs)`),
переиспользуемый D3a и D3c (DRY) — необязательно, но приветствуется.

## Критерии приёмки / тесты

- `moveInQueue`: вниз/вверх меняет соседей и нормализует 1..N; на границе (вверх у первой,
  вниз у последней) → `[]`; null/пустые прежние «Очередности» → присваиваются 1..N;
  возвращаются только изменённые; вход не мутируется; детерминизм.
- DOM (чтением/фикстура): ↑/↓ у резок, дизейбл на границах, сохранение «Очередности» и reload.
- Регрессия production-planning зелёная.

## Деплой
Клиентский `production-planning.js` → atex→ateh через `update.php`. Схему/отчёты не меняем.

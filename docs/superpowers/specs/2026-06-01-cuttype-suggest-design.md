# Подзадача C: подбор «Тип резки» при вводе заказа

**Дата:** 2026-06-01
**Источник:** ideav/atex#52 «Планирование резок», задача «при вводе заказа подбирать
и выводить подходящие типы резок, с учётом вида сырья и ширины … ограничивать выбор
теми, где есть нужная ширина».
**Рабочее место:** «Заказы» (`download/atex/js/orders.js`, роль Менеджер).
**Эпик:** [[atex_epic52_planning]] (подзадача A — [[project_ateh_atex]] — уже в PR #3062).

## Контекст и проблема

В форме позиции заказа (`orders.js`) есть поля «Вид сырья», «Тип резки», «Ширина, мм»,
«Ø втулки». Выпадающий список «Тип резки» сейчас показывает **все** типы резки без
учёта выбранного сырья и ширины. Менеджер может выбрать несовместимый тип.

Нужно ограничить список «Тип резки» теми, что подходят выбранному сырью и заказанной
ширине.

## Решения (согласованы)

1. **Жёсткий фильтр.** В списке «Тип резки» показываем только подходящие типы (без
   опции «показать все»).
2. **Критерий матча** — тип резки подходит позиции, если выполнены ОБА условия:
   - `Тип резки.Вид сырья` == «Вид сырья» позиции, И
   - заказанная «Ширина, мм» **точно равна** одной из ширин полос типа
     (`Полоса.Ширина, мм`). Для обычных однополосных типов это и есть «ширина типа».
3. **Поведение при частично заполненных полях:**
   - сырьё не выбрано → показываем все типы (фильтровать не по чему);
   - сырьё выбрано, ширина пустая → показываем типы этого сырья;
   - сырьё + ширина → строгий матч (сырьё И точная ширина).
4. **Сброс несовместимого выбора.** Если ранее выбранный «Тип резки» после смены
   сырья/ширины перестаёт проходить фильтр — выбор очищается (чтобы в позицию не
   сохранился несовместимый тип).
5. Схему БД **не меняем** — используем существующие связи `Тип резки(104).Вид сырья`
   (req 1025) и `Полоса(105).Ширина, мм` (req 1038, подчинена типу резки).

## Ограничение API (определяет стратегию загрузки)

Подчинённая таблица «Полоса» без `F_U` возвращает пусто (только `up=1`) —
`docs/MCP.md` §«Фильтр по родителю F_U». Все полосы одним запросом получить нельзя;
ширины полос читаются по каждому типу: `object/{Полоса}/?JSON_OBJ&F_U={typeId}`.
«Тип резки» — таблица верхнего уровня: `object/104/?JSON_OBJ` отдаёт все типы с их
«Вид сырья» сразу.

## Стратегия загрузки данных

- При открытии формы позиции один раз грузим **типы резки**:
  `object/{Тип резки}/?JSON_OBJ&LIMIT=0,5000` → `[{ id, label(название), materialId }]`
  (`materialId` = `parseRef` колонки «Вид сырья»).
- **Полосы — лениво, по выбранному сырью, с кэшем.** Когда в позиции выбран «Вид
  сырья», для типов этого сырья (отфильтрованных в памяти по `materialId`) грузим их
  полосы: для каждого типа `object/{Полоса}/?JSON_OBJ&F_U={typeId}` (параллельно),
  берём множество ширин `Полоса.Ширина, мм`. Результат кладём в кэш
  `stripWidthsByType[typeId] = [widths]`, чтобы не перегружать при правках.
- Индекс для матча: `{ cutTypeId → { materialId, widths:[...] } }` (widths появляются
  после ленивой загрузки полос соответствующего сырья).

Кол-во типов одного сырья ограничено, поэтому ленивая загрузка по сырью даёт
приемлемое число запросов (не грузим полосы для всех типов сразу).

## Чистое ядро (тестируемое)

`matchCutTypes(index, materialId, width)` → массив id подходящих типов:
- `materialId` пуст → вернуть все id (фильтровать не по чему);
- иначе оставить типы с `index[id].materialId === materialId`;
- если `width` задан (непустой) → дополнительно оставить те, у кого `width` точно
  присутствует в `index[id].widths` (сравнение чисел через терпимый `toNumber`,
  округление как в проекте); если для типа ширины ещё не загружены (`widths`
  отсутствует) — тип в этом случае не проходит width-фильтр.

Экспортируется в чистом объекте-ядре `orders.js` (как `calc`/`core` в других РМ) и
покрывается модульными тестами `experiments/atex-orders-*.test.js`
(`node experiments/...`).

## DOM-слой (контроллер orders.js)

- Опции списка «Тип резки» позиции строятся не из полного `refOptions`, а из
  `matchCutTypes(...)` по текущим `materialId`/`width` позиции.
- Список пересобирается при изменении «Вид сырья» (select) и «Ширина, мм» (input).
  При смене сырья — догрузить полосы типов нового сырья (если не в кэше), затем
  переотрисовать опции.
- Если текущее значение «Тип резки» не входит в новый список подходящих — сбросить
  его (очистить select-значение и модель позиции).
- Следовать существующим паттернам `orders.js` (refSelectHtml, getColumn,
  loadRefOptions, attachEvents) и `docs/WORKSPACE_DEVELOPMENT_GUIDE.md`.

## Критерии приёмки / тесты

- `matchCutTypes`: пустой материал → все; материал без ширины → типы материала;
  материал+ширина → точный матч; ширина не совпадает → исключён; тип с несколькими
  полосами проходит, если одна из полос == ширине; тип без загруженных полос при
  заданной ширине — не проходит.
- Сравнение ширины терпимо к запятой/пробелам, но именно равенство (без интервала/допуска).
- Регрессия: существующие тесты orders проходят; выбор «Тип резки», не прошедший
  фильтр, сбрасывается.

## Деплой

Только клиентский код РМ (`download/atex/js/orders.js`) → деплой atex→ateh через
`update.php` (Андрей). Схема БД не меняется.

## Вне объёма C

- Подбор/создание комбинаций резки с минимизацией отхода — подзадача B.
- Допуски/интервалы ширины («Допустимый интервал» из листа «Резки 2») — не вводим;
  матч строго по равенству, как согласовано.

# Подзадача A: остаток метража Jumbo Roll + брак в метрах с фото

**Дата:** 2026-06-01
**Источник:** ideav/atex#52 «Планирование резок», задача 1, плюс уточнения Андрея
(брак в метрах + фото).
**Рабочие места:** «Приёмка сырья» (`intake.js`), «Пульт слиттера» (`slitter.js`).

## Контекст и проблема

Эпик #52 хочет оптимизировать планирование резок. Один из входов алгоритма —
**остаток метража рулона Jumbo Roll**: при планировании остаток должен быть либо
близок к 0, либо больше 600 м (не оставлять «обрезки», которые некуда деть).

Сейчас в схеме atex остаток партии ведётся только в **м²**
(`Партия сырья.Остаток, м²`), а физический остаток рулона в **погонных метрах**
нигде не хранится. При этом пульт слиттера уже фиксирует «Погонаж факт, м» по
каждой резке и списывает остаток партии в м² через подчинённую таблицу
«Расход сырья». То есть метраж уже вводится — не хватает остатка джамбо в метрах.

Дополнительно оператору нужно фиксировать **брак в метрах** (сейчас только
«Брак, м²») и **прикладывать фото** брака.

## Решения (согласованы)

1. **Остаток джамбо — авто из погонажа.** `Остаток, м = Длина, м − Σ Погонаж факт`.
   Оператор не вводит остаток вручную — он уже вводит погонаж.
2. **1 Jumbo Roll = 1 «Партия сырья».** Длина партии задаётся при приёмке.
3. **Брак — одно агрегатное поле на резке + 1 фото.** Без отдельной таблицы (YAGNI).
4. **«Брак, м²» сохраняется, считается авто** = `Брак, м × Ширина сырья (м)`.
   Оператор вводит метры; м² — производная (совместимость с дашбордами/стоимостью).

## Изменения схемы (atex)

Числовой тип `14` = число (SIGNED), `9` = дата, `3` = ссылка, `10` = FILE.

### `Партия сырья` (таблица 106)
| Реквизит | Тип | Назначение |
|---|---|---|
| **Длина, м** | 14 | Метраж джамбо при приёмке. Дефолт = `Вид сырья.Длина рулона, м`. |
| **Остаток, м** | 14 | Остаток длины. При приёмке = «Длина, м». Списывается погонажом. |

«Остаток, м²» и «Получено, м²» остаются без изменений.

### `Производственная резка` (таблица 110)
| Реквизит | Тип | Назначение |
|---|---|---|
| **Брак, м** | 14 | Брак в метрах. Вводит оператор на пульте. |
| **Фото брака** | 10 (FILE) | Фото брака. Один файл на резку. |

«Брак, м²» (req 1106) остаётся; на пульте становится производным (не вводится вручную).

> Ширина сырья берётся из `Вид сырья.Ширина, мм` (req 1003) через
> `Производственная резка → Партия сырья → Вид сырья`.

## Поток данных

### Приёмка (`intake.js`, роль Кладовщик)
1. При оприходовании партии (= одного джамбо) появляется поле **«Длина, м»**.
   По умолчанию подставляется `Вид сырья.Длина рулона, м` выбранного сырья
   (как сейчас «Получено, м²» инициализирует «Остаток, м²»).
2. **«Остаток, м»** инициализируется значением «Длина, м» (read-only при создании).
3. «Получено, м²» / «Остаток, м²» — как сейчас (можно оставить ручной ввод;
   взаимосвязь с длиной через ширину — вне объёма A, не усложняем).

### Пульт слиттера (`slitter.js`, роль Оператор)
1. Оператор вводит «Погонаж факт, м» — **уже есть**.
2. На сохранении резки остаток партии в метрах уменьшается на **дельту погонажа**
   (новый − ранее сохранённый), не ниже 0 — по аналогии с существующим списанием
   «Остаток, м²» через «Расход сырья» (`applyConsumption`/`restoreConsumption`).
   Партия берётся из ссылки `Производственная резка.Партия сырья`.
3. Поле **«Брак, м»**: оператор вводит метры. «Брак, м²» считается авто
   = `Брак, м × Ширина сырья (м)` и пишется в req 1106.
4. Кнопка **«Фото брака»**: `<input type="file" accept="image/*" capture="environment">`
   (на планшете открывает камеру). Файл отправляется multipart-запросом в
   реквизит FILE: `formData.append('t<reqId Фото брака>', file)` (паттерн платформы,
   issue #1310). Превью/ссылка на загруженное фото показывается в форме.

### Выход для планирования (D)
`Партия сырья.Остаток, м` — поле, которое будет читать алгоритм планирования
(остаток ≈0 или >600 м). В объём A входит только его корректное ведение.

## Чистое ядро и тестируемость

Логика без DOM/сети выносится в экспортируемые объекты (как `core` в `slitter.js`
и `calc` в `intake.js`) и покрывается модульными тестами в `experiments/`
(`node experiments/atex-…test.js`).

**Критерии приёмки / тесты:**
- intake: при оприходовании «Остаток, м» = «Длина, м»; дефолт «Длина, м» = `Вид
  сырья.Длина рулона, м`.
- slitter core: `остаток_м(длина, [погонажи]) = max(0, длина − Σ погонаж)`.
- slitter core: при правке погонажа применяется дельта (увеличение списывает
  больше, уменьшение возвращает), результат не ниже 0.
- брак: «Брак, м²» = `Брак, м × ширина_м` (с учётом округления до 3 знаков, как
  `round3`); метры сохраняются в новый реквизит.
- фото: при наличии выбранного файла формируется multipart с ключом
  `t<reqId>`; без файла запрос не содержит файлового поля.

## Вне объёма A

- Алгоритм планирования очереди (подзадача D) — только потребляет «Остаток, м».
- Связь «Остаток, м²» ↔ «Остаток, м» через ширину (двойной учёт) — не трогаем,
  ведём независимо; синхронизация при необходимости — отдельным решением.
- Несколько записей брака / несколько фото — отложено (YAGNI).

# Подзадача D3a: кнопка «Запланировать» — движок на реальных данных

**Дата:** 2026-06-01
**Источник:** ideav/atex#52, задача 3 («Диспетчер запускает планирование … автоматически
сгенерированный план»). Подключает движок D2 (`orderCuts`) к реальным резкам.
**Где:** `download/atex/js/production-planning.js`. Стек на D2 (PR #3074), D1 #3072 (merged).
**Эпик:** [[atex_epic52_planning]]. Готово: A,B,C,E,D1 (merged), D2 #3074.

## Контекст и решения (согласованы)

Движок `orderCuts` (D2) — чистый, работает на дескрипторах резки. D3a наполняет дескрипторы
реальными данными и даёт диспетчеру кнопку «Запланировать»: разложить очередь по станкам и
сохранить «Очередность». Решения: сигналы движка — **расширить отчёт `cut_planning`**;
в v1 использовать **только `knifeCount`** (массив ширин ножей — позже).

D3a — это запуск движка на существующих резках + сохранение порядка. Генерация резок из
позиций — **D3b**; ручная правка (reorder) — **D3c**.

Резка→Обеспечение→Позиция). Имена — latin snake_case:

| `cut_batch_id` | Резка→Партия сырья | abn_ID |
| `cut_jumbo_remaining` | Резка→Партия сырья→«Остаток, м» | — |
| `cut_knives` | Резка→Тип резки→«Итого ножей» | — |
| `cut_winding` | Резка→Обеспечение→Позиция→«Тип намотки» | — |
| `cut_roller_width` | Резка→Обеспечение→Позиция→«Ширина, мм» | — |

Создание — `_m_new/28 up=8384 t28={reqId} t100={имя}` (+ `_m_set t104=85` для abn_ID-колонок),
по образцу [[feedback_integram_schema_d_commands]] / docs/integram-reports.md. Сделать
агентом по токену, идемпотентно (проверять наличие колонки), проверить запуск отчёта.
> Резка ↔ много обеспечений (LEFT JOIN) → N строк; `winding`/`roller_width` берутся от
> первой строки резки (как и сейчас dedup по `cut_id` в `rowsToPlanning`).

## Сборка дескриптора (в `rowsToPlanning`)

В объект резки добавить поля дескриптора движка:
```
materialId      = cut_material_id
materialName    = cut_material
batchId         = cut_batch_id
jumboRemainingM = Number(cut_jumbo_remaining) || 0
knifeCount      = Number(cut_knives) || 0
knifeWidths     = []                 // v1: пусто
winding         = normWinding(cut_winding)
rollerWidth     = Number(cut_roller_width) || 0
isFoil          = /фольг/i.test(cut_material || '')
```
(Существующие поля `id/number/slitter/cutType/materialBatch/planDate/status/sequence/supplies`
сохраняются.) `normWinding` — из ядра `planning` (D2). Чистая часть (маппинг строки→дескриптор)
тестируется.

## Кнопка «Запланировать»

В панели управления (рядом с «Создать резку») — кнопка **«Запланировать»**. По клику:
1. Сгруппировать текущие резки по станку (`groupBySlitter`); резки «без станка» — пропустить
   (планируем только назначенные на станок).
2. Для каждого станка: `planning.orderCuts(cuts)` → порядок; присвоить `Очередность` = 1..N
   по позиции в результате.
3. Сохранить: для каждой резки `_m_set/{cutId}` поле «Очередность» (reqId резолвится по имени);
   только если значение изменилось (минимизировать запросы).
4. `reload()` → очередь перерисуется уже отсортированной по «Очередности» (D1).
5. Сводка-нотификация: «Запланировано N резок на M станках».
- Веса — дефолтные `PLANNING_WEIGHTS` (правятся в коде; D3a их не настраивает в UI).
- Подтверждение перед перезаписью «Очередности» (модалка), т.к. перезаписывает ручной порядок.

## Чистое ядро / тесты

- `rowsToPlanning`: новые поля дескриптора корректно читаются (materialId/winding/knifeCount/
  jumboRemainingM/rollerWidth/isFoil из cut_material имени); пустые → дефолты; `normWinding`
  применяется. Тест в `experiments/atex-production-planning.test.js`.
- Логику «orderCuts по станкам → массив {cutId, sequence}» вынести в чистую функцию
  (напр. `planQueues(cuts, weights?)` → `[{cutId, slitterId, sequence}]`, группировка по станку
  + orderCuts + нумерация), покрыть тестом (детерминизм, без «станка» пропускается, sequence 1..N
  на станок). DOM-сохранение (`_m_set`) — слой контроллера, проверяется чтением.

## Критерии приёмки / тесты

- Отчёт `cut_planning` отдаёт новые колонки; запуск проверен.
- `rowsToPlanning` строит дескриптор (isFoil по имени сырья, winding нормализован, числа).
- `planQueues`: по станкам orderCuts, sequence 1..N на станок, «без станка» исключён, детерминизм.
- Кнопка «Запланировать» сохраняет «Очередность» и перерисовывает (ручная проверка/фикстура).
- Регрессия production-planning зелёная.

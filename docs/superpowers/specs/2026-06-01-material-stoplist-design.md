# Подзадача E: стоп-лист сырья у станка

**Дата:** 2026-06-01
**Источник:** ideav/atex#52 «Планирование резок», задача E («тип сырья Фольга … не
ставится на 2 и 3 станок — сделать станку списочное поле с мульти-выбором, стоп-лист
типов сырья»).
**Рабочее место:** «Планирование производства» (`production-planning.js`, Диспетчер).
**Эпик:** [[atex_epic52_planning]]. Соседние: A #3062, C #3066, B #3069.

## Контекст и проблема

Некоторые типы сырья нельзя резать на конкретных станках (напр. Фольга загрязняет —
не на станки 2 и 3). Сейчас при планировании резки Диспетчер может назначить любую
партию сырья на любой станок — ограничения нет.

Нужно: у станка («Слиттер») задавать стоп-лист запрещённых видов сырья и **блокировать**
создание резки, если её сырьё в стоп-листе выбранного станка.

## Решения (согласованы)

1. **Стоп-лист — мультиссылка-поле** на «Слиттер» → «Вид сырья» (тип `multi`, значение
   в `JSON_OBJ` приходит как `"id1,id2:знач1,знач2"`). Одно поле, без подчинённой таблицы.
2. **Жёсткая блокировка**: при создании резки с сырьём из стоп-листа выбранного станка
   — не сохранять, показать ошибку с названиями станка и сырья.
3. **Редактирование стоп-листа — штатный IntegramTable** (записи «Слиттер»). Нового РМ
   не делаем; E добавляет поле (схема) + контроль в production-planning.

## Изменение схемы (atex)

### `Слиттер` (таблица 101)
| Реквизит | Тип | Назначение |
|---|---|---|
| **Стоп-лист сырья** | мультиссылка (`multi`) → «Вид сырья» (100) | Запрещённые виды сырья для этого станка. |

Создание поля на live (мультиссылка на справочник «Вид сырья», `multi=1`) — делает
Андрей (схемная операция), как реквизиты подзадачи A. В репозитории — обновить
`docs/atex_metadata.json` (новый реквизит у 101). Поле редактируется штатным
табличным редактором; Фольгу в стоп-лист станков 2/3 заносит Андрей данными.

## Точка контроля (production-planning.js)

В `createCut` (перед POST `_m_new/{Производственная резка}`):
- материал резки = `Вид сырья` партии (`Партия сырья → Вид сырья`) выбранной
  `materialBatchId`;
- стоп-лист станка = распарсенные id «Вид сырья» из поля «Стоп-лист сырья» выбранного
  `slitterId`;
- если материал ∈ стоп-листа → **блокировать**: `notify('Сырьё «X» запрещено на станке
  «Y»', 'error')`, не сохранять.

Данные для контроля:
- **Слиттеры со стоп-листом**: расширить загрузку слиттеров — кроме `{id,label}` читать
  поле «Стоп-лист сырья», распарсить мультиссылку в массив id «Вид сырья»
  (`{ id, label, stopMaterialIds:[…] }`).
- **Партии с сырьём**: расширить загрузку партий — читать «Вид сырья» партии
  (`{ id, label, materialId }`), чтобы по `materialBatchId` получить материал.

## Чистое ядро (тестируемое)

В `production-planning.js` — экспортируемый объект `planning` (он уже есть, см.
`experiments/atex-production-planning.test.js`). Добавить:
- `parseMultiRefIds(raw)` → массив id из значения мультиссылки `"id1,id2:знач1,знач2"`
  (пусто/нет двоеточия → разбор по запятой; пусто → `[]`). Терпимо к пробелам.
- `isMaterialBlocked(stopMaterialIds, materialId)` → `bool`: `materialId` (строкой)
  присутствует в `stopMaterialIds`. Пустой стоп-лист / пустой материал → `false`.

DOM-резолв (батч→материал, слиттер→стоп-лист) — слой контроллера; чистые функции
парсинга и проверки — тестируются модульно.

## Критерии приёмки / тесты

- `parseMultiRefIds('1,2:Фольга,Бумага')` → `['1','2']`; `parseMultiRefIds('')` → `[]`;
  одиночное `'5:Фольга'` → `['5']`; терпимость к пробелам.
- `isMaterialBlocked(['1','2'], '2')` → `true`; `isMaterialBlocked(['1'], '3')` → `false`;
  пустой список/материал → `false`.
- Контроль в `createCut`: резка с сырьём из стоп-листа станка не создаётся (ошибка);
  разрешённое сырьё — создаётся (регрессия существующих тестов production-planning).

## Деплой

- Клиентский код `production-planning.js` → atex→ateh через `update.php` (Андрей).
- Схема: поле «Стоп-лист сырья» (мультиссылка) на «Слиттер» создаёт Андрей на live;
  Фольгу в стоп-лист станков 2/3 заносит данными.

## Вне объёма E

- **Фольга «в конец дня» / сервисные работы** — это порядок очереди резок, подзадача D.
- Авто-учёт стоп-листа при авто-планировании очереди — подзадача D (она читает то же поле).
- Отдельный экран редактирования стоп-листа — не делаем (штатный IntegramTable).

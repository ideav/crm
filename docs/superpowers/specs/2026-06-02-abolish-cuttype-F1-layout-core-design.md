# Эпик «Упразднение Типа резки». Подпроект F1: чистое ядро раскладки резок

**Дата:** 2026-06-02
**Эпик:** [[atex_epic52_planning]]. F2 (схема+отчёты) выполнен на live. Это F1 — чистый алгоритм, без live, TDD.
**Где:** новый модуль `download/atex/js/cut-layout.js` (ES5 UMD, как остальные ядра atex); тесты `experiments/atex-cut-layout.test.js` (assertEqual-стиль, `node`).

## Контекст и решения (из брейншторма эпика)

Типов резки больше нет — раскладку ножей («Полосы») считаем динамически при планировании.
Вход: сырьё, ширина джамбо (= Вид сырья «Ширина, мм»), необеспеченные позиции этого сырья
(ширина + кол-во + «Срок изготовления»), ходовые ширины (отчёт `preferable_widths`:
ширина + популярность `position_qty_sum`). Позиции **разных заказов одного сырья**
объединяются в одну резку, если их «Срок изготовления» попадает в окно (по умолчанию **3 дня**).
Раскладка = бин-пакинг заказанных ширин в ширину джамбо + **добор остатка ходовыми**.

F1 — только ЧИСТОЕ ЯДРО (детерминированное, вход не мутирует, node-тесты). Загрузка данных,
создание Полос/Резок, UI — F3. Кол-во резок под `qty` (ceil) — downstream (F3), не здесь.

## Чистые функции (экспорт в `layout`)

Переиспользуем стиль B (`cut-planning.js`): `toNumber`/`round3`/`bestFill`(DFS добора ходовыми)/
`combinationSignature`. F1 их обобщает на несколько целевых ширин.

1. `dueWindowGroups(positions, windowDays)` → `[[position,…], …]`
   - `positions`: `[{id, width, qty, dueKey}]` одного сырья; `dueKey` — число ГГГГММДД
     (пустой срок → `Infinity`). Сортировать по `dueKey`; жадно набирать кластер, пока
     `pos.dueKey - cluster.minDueKey ≤ windowDays` (в днях — сравнение ГГГГММДД упрощённо
     через перевод в дату-индекс; helper `dayDiff`). Позиции без срока (`Infinity`) →
     отдельный последний кластер. `windowDays` по умолчанию 3. Детерминировано, вход не мутирует.

2. `composeLayout(jumboWidth, demands, preferred, tolerance)` → раскладка одного кластера
   - `demands`: `[{width, qty, positionId}]` (ширины заказанных позиций кластера);
     `preferred`: `[{width, popularity}]` по убыванию популярности (ходовые); `tolerance` —
     допустимый |отход|, мм.
   - Шаги: (a) агрегировать `demands` по ширине (Σ qty), сохранить `positionIds`;
     (b) **базовая укладка** — по одной полосе на каждую отдельную заказанную ширину
     (гарантия: каждая заказанная ширина производится); если даже по одной не влезает в
     `jumboWidth` — лишние ширины в `overflow` (в следующую раскладку);
     (c) **дозаполнение по спросу** — пока в джамбо есть место, добавлять ещё полосы
     заказанных ширин, выбирая ширину с наибольшим неудовлетворённым спросом `qty`
     (детерминированно; при равенстве — бóльшая ширина, затем меньший id), чтобы раскладка
     заполняла джамбо пропорционально спросу (ширина 110 при джамбо 910 → до 8 полос);
     (d) **добор ходовыми** — оставшийся отход добрать `bestFill` по `preferred`
     (мин. отход, затем макс. популярность), полосы `purpose:'Склад'`;
     (e) посчитать `used`/`remainder`/`withinTolerance`.
   - Возврат: `{ strips:[{width, qty, purpose:'Заказ'|'Склад', positionIds:[…]}], used, remainder, withinTolerance, overflow:[demands] }`.
     `purpose:'Заказ'` — полосы под заказанные ширины; `'Склад'` — добор ходовыми.

3. `planLayouts(input)` → оркестратор: `input = { jumboWidth, positions, preferred, options:{windowDays=3, tolerance} }`
   - `dueWindowGroups(positions, windowDays)` → для каждого кластера `composeLayout`; если
     `overflow` непустой — повторный `composeLayout` на overflow (пока не уложится всё или
     не исчерпается прогресс). Возврат: `{ layouts:[{ positionsCovered:[ids], strips, used, remainder, withinTolerance, dueKey }], skipped:[{positionId, reason}] }`.
   - Детерминировано; вход не мутирует. `layout.combinationSignature` — для дедупликации/отладки.

## Критерии приёмки / тесты (node)
- `dueWindowGroups`: позиции в пределах окна → один кластер; за окном → разные; без срока → отдельный кластер; вход не мутируется; детерминизм.
- `composeLayout`: каждая заказанная ширина даёт ≥1 полосу «Заказ»; остаток добран ходовыми («Склад») с мин. отходом; ширины сверх джамбо → `overflow`; `withinTolerance` верен; `used+remainder == jumboWidth`.
- `planLayouts`: объединяет позиции окна в одну раскладку; overflow → дополнительная раскладка; `skipped` для позиций шире джамбо (не лезут вообще); пример: MW308 110 и 70 (джамбо напр. 910) → одна раскладка с полосами 110×k и 70×m + добор.
- Чистота: вход не мутируется; те же входы → тот же выход.

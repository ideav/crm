# Код-ревью #2930 — `js/app.js`: лишний перебор всех токенов

> Issue: https://github.com/ideav/crm/issues/2930

## Замечание ревьюера

> «Зачем каждый раз перебирать все токены? Токены надо смотреть для защиты
> от капчи, если пользователь не залогинен, или при переходе в эту базу.»

## Где это в коде

Перебор всех токенов выполняет функция `hasValidAuthToken(host)` в `js/app.js`
(введена в рамках issue #2906). Она проходит по **всем** cookie `idb_*`
(`CookieUtil.getAllIdb()`) и на каждый выполняет сетевой запрос `validateToken`
(`GET …/xsrf?JSON`), чтобы решить, можно ли пропустить капчу для вернувшегося
пользователя (`_captchaBypass`).

Проблема была в том, что цикл **не останавливался** при нахождении первого
валидного токена: флаг `anyValid` выставлялся в `true`, но обход продолжался,
и сетевой запрос выполнялся для каждого оставшегося cookie — на каждой загрузке
страницы.

```js
// Было: перебор ВСЕХ токенов даже после того, как ответ уже известен
async function hasValidAuthToken(host) {
    const dbNames = CookieUtil.getAllIdb();
    let anyValid = false;
    for (const db of dbNames) {
        const data = await validateToken(host, db);
        if (data && data.user && data.user !== 'guest') {
            anyValid = true;          // ответ уже известен, но цикл идёт дальше
        }
    }
    return anyValid;
}
```

## Что сделано в этом PR

Короткое замыкание (short-circuit): возвращаем `true`, как только найден первый
валидный не-гостевой токен, и не делаем сетевые запросы для остальных cookie.

```js
// Стало: останавливаемся на первом валидном токене
async function hasValidAuthToken(host) {
    const dbNames = CookieUtil.getAllIdb();
    for (const db of dbNames) {
        const data = await validateToken(host, db);
        if (data && data.user && data.user !== 'guest') {
            return true;
        }
    }
    return false;
}
```

Просроченные cookie для ещё не проверенных баз чистятся лениво — при фактическом
обращении/валидации соответствующей базы (та же `validateToken`).

## Когда перебор токенов вообще нужен (контекст замечания)

Замечание ревьюера шире: проверять токены имеет смысл **только** в трёх случаях:

1. **Защита от капчи** — когда капча действительно может быть показана.
2. **Пользователь не залогинен.**
3. **Переход в эту базу** — при переключении/входе в конкретную базу.

То есть, помимо короткого замыкания внутри `hasValidAuthToken`, сам **вызов**
этой проверки стоит выполнять лениво — только при рендере панели авторизации/капчи
для незалогиненного пользователя или при переходе в базу, а не безусловно на
каждой инициализации приложения. Это место (вызов в `App`/инициализации,
`this._captchaBypass = await hasValidAuthToken(...)`) — кандидат на дальнейшую
оптимизацию в том же направлении.

## Эффект

- В типичном случае (у вернувшегося пользователя есть хотя бы один валидный
  токен) вместо N сетевых запросов `xsrf` выполняется ровно один.
- Логика принятия решения о капче не меняется: `_captchaBypass` по-прежнему
  становится `true`, если есть хотя бы один валидный не-гостевой токен.

# Реляционная база вместо доски заметок: где Интеграм практичнее Notion

**Сравнение:** Notion

---

## Контекст

Notion широко используется как рабочее пространство для знаний: wiki-страницы, задачи, заметки, описания процессов. Базы данных в Notion — это коллекции страниц, которые можно представить в виде таблицы, доски, галереи или календаря.

Для учётных задач — хранение операционных данных, отчётность, управление правами доступа на уровне строк — Notion проектировался в меньшей степени.

---

## Конкретный сценарий

Команда ведёт в Notion базу данных клиентов и задач. При попытке расширить до полноценного учёта возникают ограничения:

- типы полей в Notion ограничены: нет разделения «целое число / деньги / дробное». Нет нативного поля «файл» в том смысле, что файл загружается как вложение страницы, а не как поле базы данных;
- вычисляемые поля (rollup, formula) работают в рамках одной базы данных. Сложные агрегации по нескольким связанным базам требуют нескольких шагов;
- права доступа в Notion — на уровне страницы или workspace. Ограничить видимость строк внутри одной базы данных по значению поля (например, «менеджер видит только своих клиентов») штатными средствами нельзя;
- API Notion имеет ограничение около 3 запросов в секунду. Для высокочастотной интеграции это может быть узким местом (данные из публичной документации Notion, актуальность нужно проверять);
- базы данных Notion — это страницы Notion. Каждая запись — отдельная страница с блочным редактором. Это удобно для описательного контента, но избыточно для операционных записей.

---

## Что делает Интеграм иначе

**Строгие типы полей.** Текст, число, дата, дата+время, флаг, файл, ссылка на запись — это отдельные типы с соответствующей валидацией и отображением. Это снижает вероятность ввода «числа» в текстовое поле.

**Отчёты с фильтрами и агрегатами.** Отчёт в Интеграме — это запрос к данным с группировкой, суммированием, фильтрами. Результат можно выгрузить или встроить в дашборд.

**Права на уровне строк.** Менеджер видит только свои записи — это настраивается через маску роли без дополнительного программирования.

**Интерактивный отчёт.** Отчёт в Интеграме может быть редактируемым: пользователь видит нужный срез данных и может вносить изменения прямо в нём.

---

## Ограничения Интеграма в этом сценарии

Интеграм не является инструментом для хранения документов и знаний. Wiki-страницы, описания процессов, документация команды — это область Notion. Если нужно совмещать базу данных и документарную часть в одном месте, Notion удобнее. Интеграм решает задачу операционного учёта.

---

## Вывод

Notion подходит для описания процессов и хранения знаний. Для операционного учёта с разграничением доступа, строгими типами данных и отчётами Интеграм имеет более подходящую архитектуру.

## Раскадровка 1-минутного видео

**0:00–0:10 — Хук**
Экран: Notion с базой данных, в которой нет фильтра по ответственному.
Текст за кадром: «Notion отлично хранит знания. Но учётная система — это не база знаний.»

**0:10–0:35 — Демонстрация**
Экран 1: карточка записи в Интеграме — строгие типы полей: число, дата, файл, ссылка.
Экран 2: отчёт с фильтрами, группировкой и суммами.
Экран 3: рабочее место менеджера — только его записи.

**0:35–0:55 — Сравнение**
Текст за кадром: «Notion — для описания процесса. Интеграм — для данных самого процесса.»

**0:55–1:00 — Следующий шаг**
Текст на экране: ссылка на пример учётной базы в Интеграме.

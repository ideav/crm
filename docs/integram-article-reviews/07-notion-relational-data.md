# Реляционная база вместо доски заметок: где Интеграм практичнее Notion

**Сравнение:** Notion

---

## Контекст

Notion широко используется как рабочее пространство для знаний: wiki-страницы, задачи, заметки, описания процессов. Базы данных в Notion — это коллекции страниц, которые можно представить в виде таблицы, доски, галереи или календаря.

Для учётных задач — хранение операционных данных, отчётность, управление правами доступа на уровне строк — Notion проектировался в меньшей степени.

---

## Конкретный сценарий

Команда из 8 человек ведёт в Notion базу клиентов и сделок: одна база «Клиенты», вторая «Сделки», связь между ними через relation. По мере роста объёма и числа ролей появляются конкретные ограничения.

**Что хочется получить:**

- менеджер видит только своих клиентов и сделки по ним;
- руководитель отдела видит сделки всей команды и сумму по этапам воронки;
- финансовый специалист видит только закрытые сделки и платежи по ним;
- внешняя интеграция (BI-дашборд) забирает срез данных раз в минуту.

**Где это упирается в архитектуру Notion:**

- типы полей. В Notion нет разделения «целое число / денежная сумма / процент» как отдельных типов с валидацией — есть общий тип Number с настройкой формата отображения. Файлы прикрепляются к странице как вложения, а не как отдельные поля базы данных. Это удобно для заметок и неудобно, когда поле «договор» должно быть обязательным и одним;
- вычисляемые поля. Свойства `formula` и `rollup` работают в рамках одной базы данных или прямой связи. Цепочка «сделка → клиент → менеджер → отдел» через несколько relation требует промежуточных rollup-полей и теряет производительность ([Notion Help: Database properties](https://www.notion.com/help/database-properties));
- права доступа. Notion управляет доступом на уровне страницы и workspace. Ограничить видимость строк внутри одной базы по значению поля («менеджер видит только своих клиентов») штатными средствами нельзя — приходится разбивать данные на отдельные базы по менеджерам или использовать teamspaces, что усложняет отчётность;
- лимит API. Документация Notion указывает средний лимит около 3 запросов в секунду на интеграцию ([Notion Developers: Request limits](https://developers.notion.com/reference/request-limits)). Для одной интеграции с минутной синхронизацией этого достаточно, но при росте числа интеграций и объёма обновлений лимит становится узким местом;
- модель данных. Каждая запись в Notion — это страница с блочным редактором. Это удобно для описательного контента и избыточно для операционной записи, у которой 12 структурированных полей и нет смысловой страницы внутри.

> Лимиты Notion и поведение свойств могут меняться. Перед публикацией ссылки проверяются повторно по [docs/INTEGRAM_ARTICLE_SERIES_PLAN.md → раздел «Фактчек перед публикацией»](../INTEGRAM_ARTICLE_SERIES_PLAN.md#фактчек-перед-публикацией).

## Что делает Интеграм иначе

**Строгие типы полей.** Текст, целое число, дробное, дата, дата+время, флаг, файл, ссылка на запись из другой таблицы — это отдельные типы с собственной валидацией и отображением. Поле «договор» — это поле типа «файл», а не вложение в страницу. Поле «сумма сделки» — это число с фиксированной точностью, а не строка с форматированием.

**Связи как часть модели.** Связь сделки с клиентом — это ссылка на запись в таблице клиентов. Цепочки связей («сделка → клиент → ответственный менеджер → отдел») доступны в отчётах напрямую, без промежуточных вычисляемых полей.

**Отчёты с фильтрами и агрегатами.** Отчёт — это запрос к данным с фильтрами, группировкой, суммированием, средним. Результат можно выгрузить, встроить в дашборд или передать через API.

**Права на уровне строк.** Видимость строк настраивается маской роли: «менеджер видит только записи, где поле «ответственный» равно ему». Это настройка, а не отдельная база и не отдельный workspace.

**Интерактивный отчёт.** Отчёт может быть редактируемым: пользователь видит нужный срез и меняет данные в нём, без перехода в исходную таблицу.

**API без посредника.** API доступно для интеграций. Отдельный API-пользователь получает только те права, которые ему нужны.

---

## Ограничения Интеграма в этом сценарии

Интеграм не является инструментом для хранения документации и заметок. Wiki-страницы команды, описания процессов, накопление знаний в свободной форме — это область Notion и подобных инструментов. Если задача — совместить базу данных и документарную часть в одном продукте, Notion удобнее. Интеграм решает задачу операционного учёта с ролями и отчётами.

Блочного редактора с произвольной разметкой внутри записи в Интеграме нет: запись — это набор полей, а не страница.

---

## Вывод

Notion подходит для описания процессов и хранения знаний. Для операционного учёта с разграничением доступа на уровне строк, строгими типами данных, многошаговыми связями и отчётами Интеграм имеет более подходящую архитектуру.

Notion — для описания процесса. Интеграм — для самого процесса и его данных.

# API и JSON-экспорт вместо копипаста: как Интеграм связывает учет с внешними системами

**Сравнение:** Excel, Google Sheets, Notion, ручные выгрузки.

---

## Контекст

Данные редко живут в изоляции. Реестр договоров нужен финансовой системе. Данные о клиентах — маркетинговой платформе. Статус заявки — внешнему сервису уведомлений. Когда инструмент учёта не поддерживает автоматического обмена данными, интеграция строится на ручных выгрузках: экспортировал Excel, отправил почтой, коллега вручную загрузил в другую систему.

---

## Конкретный сценарий: ежедневный отчёт по продажам в Power BI

Торговая компания ведёт реестр сделок: менеджеры заносят сделки и статусы оплаты, руководитель отдела продаж смотрит ежедневный дашборд в Power BI. До перехода на Интеграм процесс выглядел так:

1. в 9:00 менеджер открывает таблицу со сделками и сохраняет CSV;
2. отправляет файл аналитику в мессенджере;
3. аналитик кладёт CSV в общую папку и обновляет источник данных в Power BI;
4. если в течение дня менеджер исправил сумму сделки — дашборд устаревает до завтра.

Дополнительно финансовый отдел раз в неделю просит «выгрузку по закрытым сделкам за период» — это ещё один ручной экспорт, который аналитик готовит для бухгалтерии.

В таком процессе три типовых проблемы: данные в Power BI отстают на сутки, ручные выгрузки занимают время и зависят от конкретного сотрудника, при ошибке в исходной таблице цикл «экспорт → отправка → загрузка» повторяется с нуля.

## Что делает Интеграм иначе

**JSON-экспорт отчёта.** Любой отчёт в Интеграме доступен в формате JSON по URL — достаточно открыть отчёт и добавить к URL параметр запроса, чтобы получить структурированный ответ. Внешняя система запрашивает этот URL и получает актуальные данные без участия человека. См. `screenshots/13-api-json-export/01-report-json.png` — тот же отчёт по сделкам, отданный в формате JSON.

**Серверный API.** Помимо отчётов, Интеграм предоставляет API для операций с данными: чтение записей, создание, обновление. API работает с теми же правами, что и пользовательский интерфейс — внешняя система не получает доступ к данным, которых нет у её пользователя. См. `screenshots/13-api-json-export/02-api-endpoint.png` — пример ответа API на запрос списка сделок.

**API-пользователь с ограниченными правами.** Для внешней системы создаётся отдельный пользователь — у него нет доступа в интерфейс, только токен и права только на чтение нужных таблиц. Если токен потерян, пользователь блокируется без влияния на остальную команду. См. `screenshots/13-api-json-export/03-api-user-rights.png` — карточка API-пользователя «PowerBI Reader» и список его прав.

**Power BI как потребитель данных.** В Power BI настраивается источник «Web» с URL JSON-отчёта Интеграма и токеном API-пользователя. Расписание обновления выставляется средствами Power BI (например, каждые 30 минут). См. `screenshots/13-api-json-export/04-powerbi-connection.png` — подключение источника данных в Power BI Desktop, и `screenshots/13-api-json-export/05-powerbi-dashboard.png` — итоговый дашборд, который обновляется без участия аналитика.

## Ограничения Интеграма в этом сценарии

API Интеграма — это API конкретной платформы, а не универсальный стандарт вроде REST с OpenAPI-спецификацией. Внешняя система или интегратор должны изучить структуру запросов и формат ответа. Для сложных сценариев — двусторонняя синхронизация, событийные вебхуки в реальном времени — возможности зависят от конкретной конфигурации и могут потребовать дополнительной серверной обвязки.

Кроме того, отдача больших отчётов в JSON ограничена производительностью базы и сети: для регулярного обмена миллионами строк правильнее проектировать инкрементальную выгрузку (по дате изменения), а не запрашивать весь отчёт целиком.

---

## Вывод

Ручная выгрузка данных — это операционная нагрузка и источник ошибок. JSON-API Интеграма позволяет внешним системам забирать данные напрямую, без участия сотрудника. Дашборд в Power BI или отчёт в финансовой системе остаются актуальными, а команда не зависит от того, кто и когда отправил очередной CSV.

## Фактчек перед публикацией

- Точный синтаксис JSON-экспорта (имя и значение параметра в URL) перепроверить по актуальной документации Интеграма — в статье используется обобщённая формулировка «параметр запроса», конкретное имя подставляется при подготовке скриншота `01-report-json.png`.
- Утверждение «API работает с теми же правами, что и пользовательский интерфейс» подтвердить демонстрацией: запрос API-пользователя с ограниченными правами не возвращает записи, которых он не должен видеть.
- Не использовать конкретные числа по производительности («миллионы строк», «30 минут») как обещание платформы — это иллюстративные ориентиры, реальные значения зависят от конфигурации и нагрузки.
- Для подключения Power BI описать общий путь («источник Web», «токен в заголовке»); конкретные имена пунктов меню Power BI зафиксировать на скриншоте `04-powerbi-connection.png`, чтобы текст не расходился с версией интерфейса.

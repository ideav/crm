# Без переписывания с нуля после первого демо: где Интеграм практичнее вайб-кодинга

**Сравнение:** вайб-кодинг и быстрые AI-прототипы

---

## Контекст

Быстрый AI-прототип решает задачу первого демо: показать заказчику работающий экран, чтобы подтвердить направление. Проблема — это прототип, не продуктовая система. После того как на первом демо выясняются реальные требования, прототип часто приходится выбрасывать и писать заново.

---

## Конкретный сценарий

Команда использовала ИИ для быстрого прототипа системы учёта оборудования. На демо всё выглядит хорошо. Затем возникают реальные требования:

- нужно импортировать исторические данные из Excel-файла на 10 000 строк;
- разные пользователи видят разные объекты в зависимости от подразделения;
- финансовая система ежедневно запрашивает данные через API;
- к каждой единице оборудования крепятся несколько документов (скан-копии паспортов).

Сгенерированный прототип не предусматривал ни одного из этих требований. Добавить их — значит переписать значительную часть кода. Прогнозируемый результат: второй раунд разработки, не меньший по объёму, чем первый.

---

## Что делает Интеграм иначе

**Импорт данных.** Загрузка CSV или Excel — стандартная функция. Исторические данные переносятся без написания миграционных скриптов.

**Структура таблиц.** Добавление подразделения как справочника и маска по значению для ролей — административная настройка.

**API.** Финансовая система подключается к API Интеграма. Для неё создаётся отдельный пользователь с правами только на чтение нужных данных.

**Файлы.** Поле типа «файл» в карточке оборудования позволяет прикрепить документы к записи. Хранение — в файловой системе Интеграма.

**Рабочее место.** Для нового типа пользователя (по подразделению) создаётся роль с маской и рабочее место в меню — без изменения backend.

Всё это — изменения конфигурации, а не переписывание кода.

---

## Ограничения Интеграма в этом сценарии

Если прототип показал, что задача требует специфической бизнес-логики, которую Интеграм не поддерживает (специализированные вычисления, отраслевые стандарты, сложные воркфлоу с ветвлением), переработка неизбежна и в Интеграме. Платформа не решает все задачи автоматически.

---

## Вывод

Первый прототип важен для проверки направления, а не как финальное решение. Разница между ИИ-прототипом и Интеграмом — в том, что второй при появлении новых требований не требует переписывания с нуля. Типовые расширения — импорт, права, API, файлы, новые роли — это настройка существующей платформы.

# HTML-шаблоны вместо закрытого интерфейса: как Интеграм дает больше гибкости no-code-сервисов

**Сравнение:** Airtable Interfaces, Notion views и другие закрытые конструкторы интерфейсов

---

## Контекст

Большинство no-code-платформ предоставляют визуальный конструктор интерфейсов: набор блоков, которые можно перетащить на страницу. Это удобно для стандартных рабочих мест. Ограничение проявляется, когда нужен нестандартный вид: своя навигация, специфичный порядок полей, интеграция внешнего элемента или нестандартный расчёт прямо на экране.

Airtable Interfaces работает на фиксированном наборе элементов (Grid, Gallery, Timeline, Kanban, Calendar, Form, набор виджетов). Notion views даёт переключение Table / Board / Calendar / Timeline / Gallery / List поверх одной базы. В обоих случаях встроить произвольный HTML-блок, внешнюю JS-библиотеку или собственную вёрстку штатными средствами нельзя — нужно либо встраивание (embed) внешнего URL, либо доработка через API на отдельном хостинге.

---

## Конкретный сценарий

Компании нужно рабочее место для диспетчера: большая карта с объектами (требует внешней JS-библиотеки, например, Leaflet или Яндекс.Карт), рядом — список активных заявок из базы, внизу — счётчики статусов и кнопка «взять в работу», которая меняет статус записи. Конструктор Airtable Interfaces или Notion таких комбинаций не собирает: карта вставляется только через embed URL, а кнопки на запись ограничены типовыми действиями.

Второй сценарий — карточка клиента в фирменном формате: логотип, конкретный порядок реквизитов, печатный лист A4 со страничной разбивкой. Встроенные виды платформы дают только табличный/карточный вид, а не произвольную вёрстку для печати.

## Что делает Интеграм иначе

**HTML-шаблоны как файлы.** Рабочие места и стандартные экраны Интеграма — это HTML-файлы из папки `templates/` (`main.html`, `info.html`, `cards.html`, `dash.html`, `form.html`, `table.html`, `report.html`, `kanban.html`, `funnel.html`, `calendar.html` и другие). Шаблон редактируется в любом текстовом редакторе и подключается как рабочее место для роли.

**Подстановка данных через `{ИмяПоля}`.** В шаблон вставляются плейсхолдеры с именами полей, которые подменяются значениями при отрисовке. Пример из реального шаблона `templates/info.html`:

```html
<tr>
    <td><a href="/{_global_.z}/query/{ЗапросID}">{Запрос}</a></td>
    <td>{Описание}</td>
</tr>
```

Здесь `{ЗапросID}`, `{Запрос}`, `{Описание}` — поля записи, а `{_global_.z}` — глобальная переменная. Имена полей совпадают с именами из структуры базы — без отдельного маппинга.

**Повторяющиеся блоки.** Списки и таблицы собираются разметкой `<!-- Begin:ИмяБлока --> ... <!-- End:ИмяБлока -->`: цикл по строкам отчёта или по записям таблицы оформляется как HTML-фрагмент, который повторяется для каждой строки.

**Подключение отчётов.** Внутри шаблона отображается результат отчёта Интеграма — таблица, агрегаты, динамически подгружаемые строки. Дашборд диспетчера собирается как один HTML-файл, в котором рядом стоят карта, отчёт со списком заявок и отчёт со счётчиками статусов.

**Произвольный JS и CSS.** Так как шаблон — это обычный HTML, в него подключается любая внешняя библиотека через `<script src="…">`: карта, график, печатный движок, drag-and-drop. Никаких ограничений закрытого конструктора.

**Права работают независимо.** Шаблон рисует только те данные, к которым у пользователя есть доступ по его роли. Интерфейс не обходит права — он работает поверх них.

## Ограничения Интеграма в этом сценарии

Для редактирования HTML-шаблонов нужен базовый технический навык: понимание HTML и умение работать с текстовым файлом. Это не drag-and-drop. Нетехнический пользователь с нуля шаблон не соберёт — потребуется один раз настроить его с помощью разработчика или технического специалиста. Дальше шаблон правится правкой текста и перезагрузкой.

Готовых «галерей виджетов» уровня Airtable Interfaces в Интеграме нет: набор стандартных рабочих мест (`cards.html`, `kanban.html`, `funnel.html`, `calendar.html`, `dash.html`) закрывает типовые задачи, но визуальный конструктор «соберите страницу из блоков мышкой» не предусмотрен.

---

## Вывод

HTML-шаблоны — более гибкий, но более трудоёмкий подход по сравнению с визуальным конструктором интерфейсов. Для стандартных рабочих мест Airtable Interfaces и Notion views удобнее, потому что не требуют знания HTML. Когда нужен нестандартный вид, интеграция внешнего компонента, печатная вёрстка или точный контроль над вёрсткой, HTML-шаблоны Интеграма дают нужную степень свободы без выхода в отдельную фронтенд-разработку.

## Раскадровка 1-минутного видео

**0:00–0:08 — Хук**
Экран: Airtable Interfaces или Notion с ограниченным набором блоков; курсор тщетно ищет «вставить произвольный HTML».
Текст за кадром: «Конструктор интерфейсов удобен — пока вам нужен только стандартный набор блоков.»

**0:08–0:20 — Проблема**
Экран: макет диспетчерской с картой и списком заявок.
Текст за кадром: «А если на рабочем месте нужна карта, печатная карточка или нестандартный виджет — конструктор заканчивается.»

**0:20–0:45 — Демонстрация Интеграма**
Экран 1 (0:20–0:28): папка `templates/` с файлами `cards.html`, `dash.html`, `info.html`.
Экран 2 (0:28–0:36): открытый `info.html` с подсветкой блока `{ЗапросID}`, `{Запрос}`, `{Описание}` и пары `<!-- Begin:… --> <!-- End:… -->`.
Экран 3 (0:36–0:45): результирующее рабочее место — карта + список заявок + счётчики, собранные на основе одного HTML-файла.

**0:45–0:55 — Сравнение**
Текст за кадром: «HTML-шаблон — это не конструктор. Это обычный файл, который вы редактируете сами, подключаете нужные JS-библиотеки и собираете рабочее место под конкретную роль.»

**0:55–1:00 — Следующий шаг**
Текст на экране: «Смотрите примеры шаблонов в `templates/` и документацию по `{ИмяПоля}`.»

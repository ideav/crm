## 1. Цель и целевая модель

**Цель планирования** (#3985): разместить задания под обеспечение позиций, уложившись в
**минимально возможное время переналадки**, соблюдая блокирующие правила, и затем искать
оптимизационные манёвры. Способ — жадный разброс + поиск перемещений; подбор по «штрафам» за
смену конфигурации и по «сложности» её изменения.

### 1.1. Слот-модель занятости станков

Целевая структура данных, которой в коде **нет** (сейчас — пер-станочная пересборка дня
`splitMachineQueue`/`planCutOperations`, а не единый массив занятости с вставкой/сдвигом):

- На каждый станок — **массив занятости**: упорядоченные слоты произвольной длительности по
  дням. Слот несёт `id`, дату, время начала/окончания и все параметры резки.
- Слоты сначала заполняются **существующими** заданиями в хронологическом порядке из таблицы
  «Задание в производство» (последовательность уникальна в пределах даты для станка). Запрос
  существующих — от даты «С» рабочего места, **без ограничения вперёд**.
- Новые записи **вставляются** в массив, сдвигая всю последовательность после точки вставки.
- **Отпуска** станка тоже пишутся в слоты; по времени не двигаются, но могут менять
  очередность, если перед ними есть свободное рабочее время в этом дне.
- **Ограничения станка**: стоп-лист сырья (есть в коде) **и лимит ширины джамбо** (логика как
  в поле «Код» у «Вид сырья»: `j<1000` → номинальная ширина < 1000) — **лимита ширины в коде
  нет** (у станка только `stopMaterialIds`, `production-planning.js:239`).

### 1.2. Стоимость вставки слота = «вес» + «качество»

Для нового слота ищется пара соседних слотов (предыдущий/следующий), между которыми вставка
дешевле всего — по сумме **весов** и **качеств** двух переходов (предыдущий→новый,
новый→следующий). Составляющие (веса — из настроек `*_COST_MN`, см. §2):

| # | Фактор | Вес при смене | «Качество» |
|---|---|---|---|
| 1 | Набор ширин полос | уменьшение полос → `KNIVES_CHANGE_COST_MN`; **увеличение → `KNIVES_INCREASE_COST_MN`** | отношение `всего полос / совпавших с начала последовательности` (меньше — лучше). Пример: `110×3,60×5,40×10` → новый `110×3,60×5,30×13` = **3/2** |
| 2 | Сырьё + тип намотки | `MATERIAL_CHANGE_COST_MN` | 0. (а) фольга не последним заданием дня и не перед фольгой → `+FOIL_NOTEND_COST_MN`; (б) это перемещение (не первичная вставка) → `+FOIL_NOTEND_COST_MN` |
| 3 | Лидер + втулка | `LEADER_COST_MN` (2) | 0 |
| 4 | Срок одной из полос **позже** дня размещения | `+DEADLINE_COST_MN` (100) | — |
| 5 | Срок одной из полос **равен** дню размещения | `+EXACT_DEADLINE_COST_MN` (33) | — |
| 6 | Простой: от самой старой даты-окончания на других станках до текущего старта > `MAX_SLOTS_DISTANCE_HR` | `+MAX_DISTANCE_COST_MN` (25) | — |

**Исключения:**
- вес смены при вставке **после слота, перешедшего с прошлого дня** (продолжение на след. день) = 0;
- нет кандидатов с приемлемыми п.1/п.2 (суммарный вес > `KNIVES_CHANGE + MATERIAL_CHANGE`) →
  выбирается станок с **минимальной датой окончания** за период (или вовсе без слотов);
- при разбиении слота между частями можно включать **отпуск**, если простой ≤ `MAX_OUTAGE_PLANNABLE_HR` (48 ч);
- если включённый отпуск > `MAX_OUTAGE_PLANNABLE_HR` — штрафы за смену сырья/ширин **сбрасываются в
  значения по умолчанию** (переналадку выполняют после отпуска), ср. #3898 `DOWNTIME_KEEP_SETUP_MAX_DAYS`.

### 1.3. После вставки — сдвиг и релокация

1. Сохранить стоимость вставки (для анализа стоимости будущих сдвигов).
2. Сдвинуть по времени все слоты после точки вставки, **разрешив слияния/разбиения** слотов с
   одним «ID первой части».
3. Проверить сдвинутые: пересчитать стоимость текущей позиции vs альтернатив (кроме позиций в
   том же дне того же станка). **Перемещать**, если:
   - слот идёт **после фольги** в этом дне (фольга должна «всплыть» в конец дня, если нет
     лучшего предложения в других днях);
   - срок одной из полос стал **позже или равен** дню размещения.

   Перемещать при стоимости перемещения **меньше** текущей и продолжать проверку.

### 1.4. Дискретность и разбиение слота

Резка дискретна — по числу резок из «Задания в производство». Неделимый интервал =
`{длительность 1 резки} + LEADER_COST_MN`. Слот, не влезающий в день, разбивается на 2 слота
(признак единства — «ID первой части»). Варианты разбиения: (1) после наладки ножей;
(2) после смены сырья/намотки; (3) между резками (целое число неделимых интервалов). Встраивать
чужой слот между частями одного задания нельзя; при попадании частей в один рабочий день их
**переразбивают и сливают**.

### 1.5. Обед и два перерыва — прозрачны для планирования (НОВОЕ)

Целевое: планирование **не учитывает** обед (`LUNCH_START`×`LUNCH_DURATION_MN`) и два перерыва
(`FIRST_INTERVAL 10:00`, `SECCOND_INTERVAL 15:00`, каждый `INTERVAL_DURATION_MN` = 10). На Ганте
перерывы рисуются **в разрыв** задания, попавшего на них; время окончания/начала последующих
увеличивается на длительность перерыва. День начинается в `DAY_START_HOUR` и длится
`DAY_DURATION_MN` (450).

> Разрыв: сейчас **обратный** подход — генерация **зашивает** обед в `planStart`, а показ его
> лишь «находит» как зазор (#3816/#3846, `lunchBlocksFromSchedule`). Двух перерывов
> (`FIRST_INTERVAL`/`SECCOND_INTERVAL`) в коде **нет вообще** (grep пуст).

Целевая модель задаёт веса **настройками**, а не константами кода. Суффиксы: `_MN` — минуты,
`_HR` — часы, `_DAY` — дни; `_COST_MN` — веса штрафов для стоимости размещения/перемещения.

## 3. Разрывы аналитики (текущий код vs целевая модель)

Что здесь НЕ разрыв: жёсткие технологические правила (одна втулка/намотка/лидер, стоп-лист,
ширина ≤ джамбо, потолок дня с ограниченным нахлёстом) — соблюдаются. Разрыв — про **отсутствие
единой стоимостной модели, наблюдаемости и глобальной оптимизации**.

### A. Нет единой функции стоимости размещения
Стоимость вставки `вес+качество` (§1.2) нигде не вычисляется. Решения — изолированные
лексикографические ступени:
- очередь минимизирует только сумму минут переналадки (`chainChangeoverCost`, `:4891`);
- выбор станка — цепочка тай-брейков (`chooseSlitterBySetup`, `:5053`);
- баланс — локальный `scoreFrom = [дни, пик, Σкв. минут]` (`:5238`).

Ни одна ступень не уступает другой: алгоритм не примет +1 день опоздания ради экономии крупной
переналадки и наоборот — общей «валюты» нет, `ΔCost` не существует.
→ **Ввести чистую `placementCost(prev, slot, next, settings)` → {вес, качество, byFactor}` по §1.2.**

### B. «Качество» перехода (доля совпавших полос) отсутствует
`changeoverParts` (`:1665`) — бинарные минуты (сменилось/нет). Метрики «отношение всего полос к
совпавшим с начала» (§1.2 п.1) нет. → добавить `stripPrefixQuality(prev, next)`.

### C. `KNIVES_INCREASE` не отделён от `KNIVES_CHANGE`
#3600 схлопывает любую смену ножей/сужение в фикс. 30 (`:1674-1681`). Увеличение числа полос
(дороже — 50) не штрафуется отдельно. → различать уменьшение/увеличение полос.

### D. Фольга — жёсткое правило вместо штрафа
`orderCuts` (`:4994`) безусловно уводит `isFoil` в конец; короткая фольга ≤120 мин
(«[требование]» в источнике истины) не реализована. Целевое — штраф `FOIL_NOTEND_COST_MN` (60),
позволяющий фольге стоять не в конце, если это дешевле. → заменить жёсткое правило на вес.

### E. Балансировка развязана с EDD; штрафов смены станка/дня нет
`rebalanceSlitterLoad` (`:5150`) оптимизирует только минуты/дни (`:5238`); опоздание по сроку в
счёт не входит — перенос ради баланса может задвинуть задание за `dueKey`. Штрафов
`CHANGE_SLITTER_COST_MN`/`CHANGE_DAY_COST_MN` за само перемещение нет. Балансировка включается
только при `spanDays ≥ 2` (`:5284`) — перекос внутри одного дня не выравнивается. Слепая зона
span: «дыра» перед будущим отпуском без работы после него span не увеличивает
(`spanDays`, `:5203-5217`; источник истины, стр. 322).

### F. Простой между станками (`MAX_SLOTS_DISTANCE_HR`) не штрафуется
Фактор 6 (§1.2) — старт слота позже самой старой даты-окончания на других станках более чем на
24 ч — не считается. → добавить `MAX_DISTANCE_COST_MN`.

### G. Slot-occupancy модель со вставкой/сдвигом/релокацией отсутствует
Сейчас — пер-станочная пересборка дня встык (`planCutOperations`→`splitMachineQueue`), а не
единый массив занятости с точкой вставки, сдвигом «всех после» и cost-driven перемещением
сдвинутых (§1.1, §1.3). Существующие задания грузятся отчётом `cut_planning`, а не как
хронологический seed слотов. → архитектурный шаг.

### H. Отпуск в середине окна и лимит ширины джамбо по станку
- Сброс/сохранение заправки при отпуске **внутри окна** планирования не реализован — учитывается
  только отпуск на день базы (источник истины, стр. 616; частично #3898).
- Лимит ширины джамбо станка (`j<1000`) не читается — у станка только `stopMaterialIds` (`:239`).

### I. Scope-фильтр — ручная эвристика с историей регрессий
`headInScope` (`:4317`) — растущий набор ad-hoc-предикатов «Дата план × Срок × Зафиксировано»
(#3660/#3820); подтверждённые регрессии #3820, #3918/#3919, #3916. В слот-модели «зона
перепланирования» задаётся явно (какие слоты подвижны). → специфицировать и покрыть
характеризационными тестами.

### J. Наблюдаемость: качество плана невидимо оператору
В UI из аналитики — только значок минут за день у станка (`«456 мин»`, `:10991`, #3743).
Рассуждения балансировки уходят **только в `console`** (`opts.log`, `:5141`). Нет агрегатов:
Σ переналадки, опоздания, перекос, число дроблений/переполнений. → см. §4 (панель качества).

## 4. Расчёт качества плана (отработка комментария #3985)

Комментарий:
> Добавить расчёт качества: суммарное количество и время переналадок за заданный период плюс
> отдельно — с начала периода до конца всех задач. Посчитай идеальную гипотетическую стоимость
> и количество переналадок: все варианты конфигурации ножей и все варианты сырья, какое время
> нужно чтобы по 1 разу настроить их все независимо, невзирая на сроки и прочее — как если бы
> ножи и сырьё идеально переходили друг в друга. Проверь, что если задание первое — до него
> ничего нет — то время наладки ножей и смены сырья надо заложить.

Определяем чистую функцию `planQuality(slotsByMachine, prevSetupBySlitter, settings, window)`.

### 4.1. Фактические переналадки (actual) — два окна

Считаем по двум окнам: **W1 = [С; По]** и **W2 = [С; дата-время конца ВСЕХ задач]** (включая
слоты, вылезшие за «По»).

```
для каждого станка:
  seq = слоты станка в окне, по времени старта
  prev = prevSetupBySlitter[станок]          // заправка из заданий прошлых дней (prevSetupBeforeWindow)
  для i, slot в seq:
    если i == 0 и prev существует и совпадает со slot → 0 переналадок    // #3853
    иначе:
      parts = (i == 0 и prev нет) ? firstSetupParts(slot)                // §4.3: первое задание
                                  : changeoverParts(prev_или_seq[i-1], slot)
      если parts.knife:  knifeCount++;    knifeMin += (увеличение полос ? KNIVES_INCREASE : KNIVES_CHANGE)
      если parts.material: matCount++;    matMin  += MATERIAL_CHANGE_COST_MN
    prev = slot
итог по окну:
  changeoverCount = knifeCount + matCount
  changeoverMin   = knifeMin + matMin
```

Возвращаем `{ W1:{knifeCount,knifeMin,matCount,matMin,changeoverCount,changeoverMin},
W2:{...} }`. (Лидер `LEADER_COST_MN` — не переналадка, а межрезочный интервал; в качество
переналадок не входит, но может показываться отдельной строкой.)

### 4.2. Идеальная гипотетическая стоимость (нижняя граница)

«Как если бы ножи и сырьё идеально переходили друг в друга» и каждую конфигурацию настраивают
**по 1 разу**, невзирая на сроки и станки:

```
KC = множество РАЗНЫХ конфигураций ножей (по knifeWidthSig) среди всех слотов окна
MC = множество РАЗНЫХ (сырьё + тип намотки) среди всех слотов окна

idealKnifeCount = |KC|;  idealKnifeMin = |KC| * KNIVES_CHANGE_COST_MN   // перфект-переход ⇒ базовая цена
idealMatCount   = |MC|;  idealMatMin   = |MC| * MATERIAL_CHANGE_COST_MN
idealCount = |KC| + |MC|;  idealMin = idealKnifeMin + idealMatMin
```

Это строгая нижняя граница: каждую конфигурацию/сырьё нужно настроить хотя бы раз. **Первая
настройка учтена** (каждая конфигурация посчитана один раз = её первичная наладка), что и
требует п.3 комментария. Идеал — «холодный старт» (пред-заправка игнорируется), поэтому фактика
может оказаться **ниже** идеала, если станок вошёл в окно уже настроенным под нужную
конфигурацию (реальная экономия, не ошибка) — это стоит подсвечивать отдельно.

### 4.3. Первое задание — заложить наладку (проверка п.3)

Если у слота нет предшественника (станок пуст / начало периода) и нет валидной заправки
станка → фактика **обязана** учесть полную наладку ножей + смену сырья
(`firstSetupParts`, `:1752` / `firstSetupCost`, `:1760`), а не считать «до него ничего → 0».
В коде это уже делает `carryPrevSetup`/`firstSetupCost` (используется, напр., в `poolMinutes`,
`:5187`) — метрика качества обязана переиспользовать ту же логику, иначе фактика занижается.

### 4.4. Качество = фактика / идеал

```
qualityCount = changeoverCount / idealCount            // ≥ 1 в норме; 1.0 = идеал
qualityMin   = changeoverMin  / idealMin
excessCount  = changeoverCount - idealCount            // избыточные переналадки (потери на упорядочивании/сроках)
excessMin    = changeoverMin  - idealMin
```

Считается для W1 и W2. Панель качества плана (§J) над очередью:
`Σ переналадки: N (M мин) · идеал: N₀ (M₀ мин) · избыток: +ΔN (+Δмин) · опозданий: L`.

### 4.5. Пример

Слоты станка (по времени), заправка при входе — та же, что у слота 1:
`A[110×3,60×5,40×10 · сырьё X]` → `B[110×3,60×5,30×13 · сырьё X]` → `C[80×6 · сырьё Y]`.

- Фактика: A↔B — ножи меняются (40→30, число полос выросло 10→13 ⇒ `KNIVES_INCREASE` 50),
  сырьё то же (0). B↔C — ножи меняются (`KNIVES_CHANGE` 30, полос меньше), сырьё X→Y (15).
  Слот A vs заправка — совпадает ⇒ 0. Итого: knife 2 (50+30=80), material 1 (15) →
  **3 переналадки, 95 мин**.
- Идеал: KC = {`110×3,60×5,40×10`, `110×3,60×5,30×13`, `80×6`} → |KC|=3; MC = {X, Y} → |MC|=2.
  idealMin = 3×30 + 2×15 = **120 мин**, idealCount = **5**.
- Качество: `qualityCount = 3/5 = 0.6`, `qualityMin = 95/120 ≈ 0.79`. Фактика ниже идеала по
  числу (станок вошёл уже настроенным + сырьё сгруппировано) — план на этом станке близок к
  оптимуму по переналадкам.

(Пример иллюстративный; веса — из настроек §2.)

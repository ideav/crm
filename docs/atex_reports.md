# Отчёты (Запросы) в atex/ateh: аудит, пилот, методичка

Дата: 2026-06-01. БД: ateh (`https://ideav.ru/ateh`).

В Integram **«запрос» (отчёт)** — это объект системной таблицы **22**, который ядро
разворачивает в SQL `SELECT` с автоматическими `LEFT JOIN` по метаданным.
Запуск: `GET /{db}/report/{queryId}?JSON_KV` → плоские сджойненные строки.

## 1. Текущий статус: отчёты не используются

Все рабочие места atex читают данные напрямую через `object/{table}/?JSON_OBJ` и
**джойнят/агрегируют на клиенте**. Слой `report/` упомянут комментарием
«перевод чтений на защищённый слой `report/` — следующий этап» в 10 файлах
(cut-calc, orders, warehouse, slitter, portal, cut-map, intake,
production-planning, sleeve-cutter, dashboards), но нигде не задействован.

## 2. Где отчёты выгодны (из аудита, по убыванию выгоды)

| РМ | Что делает сейчас | Выгода от отчёта |
|----|-------------------|------------------|
| **dashboards.js** | `loadAll` (стр. 461–468) тянет **6 таблиц целиком** + 4 `_count`; `makeRow` (225–274) джойнит Заказ→Позиция→Обеспечение→Резка→ГП в JS | 1 отчёт вместо 6 полных выгрузок и клиентских джойнов |
| **production-planning.js** | `loadPositions`(2000)+`loadCuts`(2000)+`loadSupplies`(5000), джойн на клиенте | отчёт «очередь резок с позициями и обеспечением» |
| **portal.js / orders.js** | Заказ→Позиции | отчёт «заказы с позициями и статусом исполнения» |
| cut-map.js | Резка→Тип резки→Полосы (зависимые загрузки) | кандидат послабее |

Общая выгода: меньше round-trip’ов; серверный join/агрегация (не тянуть тысячи
строк на клиент); защищённый слой (отчёт уважает права роли); нет клиентских
join-багов.

## 3. Пилот: отчёт «Pipeline заказа» (queryId 8303)

Создан в боевой ateh. Цепочка **Заказ → Позиция → Обеспечение (расход) → Резка → Партия ГП**.

Колонки (источник → имя в отчёте):

| t28 (источник) | Имя в отчёте |
|---|---|
| 1075 Заказ (Номер) | Заказ |
| 1131 Заказ.Статус | Статус_заказа |
| 1140 Позиция.Тип резки | Тип_резки |
| 1141 Позиция.Ширина, мм | Ширина_мм |
| 1149 Обеспечение.Метраж, м | Расход_м |
| 1154 Обеспечение.Статус | Статус_обеспечения |
| 1078 Производственная резка (Номер) | Резка |
| 1162 Резка.Статус | Статус_резки |
| 1081 Партия ГП (Дата прихода) | Партия_ГП |

Запуск: `GET /ateh/report/8303?JSON_KV`. Незавершённые стадии = пустые поля (LEFT JOIN).

Пример полностью пройденной строки:

```json
{
  "Заказ": "АТХ-3002-2026-05-31", "Статус_заказа": "Выполнен",
  "Тип_резки": "TT-АТХ-3002-2026-05-31 57x10+40x8", "Ширина_мм": "57.00",
  "Расход_м": "1200.00", "Статус_обеспечения": "Выполнено",
  "Резка": "4", "Статус_резки": "Завершён", "Партия_ГП": "1780235295"
}
```

## 4. Как создать отчёт (проверенная последовательность)

```bash
DB=https://ideav.ru/ateh; TOKEN=<актуальный>
XSRF=$(curl -s -H "X-Authorization: $TOKEN" "$DB/xsrf?JSON=1" | jq -r ._xsrf)

# 1. Запрос (тип 22). up=1 ОБЯЗАТЕЛЕН — иначе «Недопустимые данные: up=0».
curl -s "$DB/_m_new/22?JSON&up=1" --data-urlencode "t22=Имя отчёта" \
  --data-urlencode "token=$TOKEN" --data-urlencode "_xsrf=$XSRF"   # → {"id": <queryId>}

# 2. Колонка (тип 28). up=queryId; t28 = ID таблицы ИЛИ реквизита из метаданных;
#    t100 = имя в отчёте (и внешний alias для FR_/TO_-фильтров).
curl -s "$DB/_m_new/28?JSON&up=<queryId>" --data-urlencode "t28=<fieldId>" \
  --data-urlencode "t100=Имя_колонки" \
  --data-urlencode "token=$TOKEN" --data-urlencode "_xsrf=$XSRF"   # → {"id": <colId>}

# 3. Формат отображения (опц.): _m_set/{colId} t84=DATETIME|DATE|NUMBER|…
```

Ключевое:
- **Порядок колонок = порядок join слева-направо**; первая реальная колонка —
  мастер-таблица. Ядро само строит `LEFT JOIN` по метаданным (подчинение `up`,
  ref-связи). Полный справочник полей колонки (тип 28: `t100`, `t58`, `t84`,
  `t101`, агрегации `t104`, сортировка `t109`, фильтры `t102/t103` …) —
  `docs/integram-app-workflow.md`, раздел «Создание запроса программно».

## 5. Найденные грабли

- **`up=1` обязателен** при создании запроса (тип 22), иначе ошибка `up=0`.
- **`JSON_KV` отдаёт СЫРЫЕ значения** — формат из `t84` виден только в
  платформенном `report/{id}?JSON` (поле `format`). DATETIME-штамп (например
  «Партия ГП») форматировать **на клиенте** через
  `AtexRefSearch.formatDateTime` (`download/atex/js/ref-search.js`).
- ⚠️ **Не форматировать дату формулой `t101=FROM_UNIXTIME([THIS],…)`** — это
  ломает join: отчёт схлопывается в одну строку. Формат — только через `t84` +
  клиентский форматтер.

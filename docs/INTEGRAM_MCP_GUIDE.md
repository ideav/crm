## Основные понятия Интеграма

| Термин Интеграма | Аналог в SQL/традиционных БД |
|---|---|
| **Тип** (Type) | Таблица |
| **Реквизит** (Requisite) | Колонка / поле |
| **Объект** (Object) | Запись / строка |
| **Справочник** (Lookup) | Таблица-справочник (reference table) |
| **Подчинённая таблица** | Дочерняя таблица с FK на родителя |
| **Отчёт** (Report) | Сохранённый запрос / VIEW |

## Типы данных реквизитов

При создании реквизита (колонки) необходимо указать `requisiteTypeId` — числовой код типа данных:

| ID | Название | Описание |
|----|----------|----------|
| `3` | SHORT | Короткий текст (до 255 символов) |
| `8` | CHARS | Текст |
| `2` | LONG | Длинный текст |
| `13` | NUMBER | Числовой (целое или дробное) |
| `14` | SIGNED | Числовой со знаком |
| `4` | DATETIME | Дата и время |
| `9` | DATE | Только дата |
| `7` | BOOL | Булево (флажок) |
| `11` | BOOLEAN | Альтернативное булево |
| `12` | MEMO | Большой текст (memo-поле) |
| `10` | FILE | Вложенный файл |
| **ID таблицы** | REFERENCE | Ссылка на запись другой таблицы |

> **Ключевое правило для ссылок:** Чтобы создать колонку-справочник (выпадающий список, FK), нужно передать **ID целевой таблицы** как `requisiteTypeId`, а не стандартный числовой код.

## DDL-операции

DDL (Data Definition Language) — операции по определению структуры данных: создание, изменение, удаление таблиц и колонок.

### 1. Создать таблицу

**Инструмент:** `integram_create_type`

```json
{
  "name": "Продукты",
  "baseTypeId": 3,
  "unique": false
}
```

> ⚠️ **КРИТИЧНО:** `baseTypeId` **всегда должен быть системным типом** (обычно `3` для стандартных таблиц). Никогда не передавайте ID пользовательской таблицы как `baseTypeId`!

**Или создать таблицу сразу с колонками:**

**Инструмент:** `integram_create_table_with_columns`

```json
{
  "tableName": "Продукты",
  "baseTypeId": 3,
  "columns": [
    { "requisiteTypeId": 3,  "alias": "Артикул" },
    { "requisiteTypeId": 2,  "alias": "Описание" },
    { "requisiteTypeId": 13, "alias": "Цена" },
    { "requisiteTypeId": 7,  "alias": "В наличии" }
  ]
}
```

### 2. Добавить колонку к таблице

**Инструмент:** `integram_add_requisite` + `integram_save_requisite_alias`

```json
// Шаг 1: добавить реквизит
{
  "typeId": 994762,
  "requisiteTypeId": 13
}

// Шаг 2: задать псевдоним (название колонки)
{
  "requisiteId": 994790,
  "alias": "Стоимость"
}
```

**Или добавить несколько колонок сразу:**

**Инструмент:** `integram_add_columns_to_table`

```json
{
  "typeId": 994762,
  "columns": [
    { "requisiteTypeId": 3,  "alias": "Код" },
    { "requisiteTypeId": 13, "alias": "Количество" },
    { "requisiteTypeId": 4,  "alias": "Дата поставки" }
  ]
}
```

### 3. Создать колонку-ссылку (справочник / FK)

**Инструмент:** `integram_add_reference_column`

```json
{
  "typeId": 994762,
  "referenceTableId": 994769,
  "alias": "Категория",
  "multiSelect": false,
  "allowNull": true
}
```

> Параметр `referenceTableId` — это ID таблицы, на которую ссылаемся (справочника).

**Или создать справочник и сразу добавить ссылку:**

**Инструмент:** `integram_create_lookup_with_reference`

```json
{
  "targetTableId": 994762,
  "lookupTableName": "Категории товаров",
  "columnAlias": "Категория",
  "values": ["Электроника", "Одежда", "Продукты питания"],
  "multiSelect": false
}
```

### 4. Изменить свойства реквизита

```json
// Переименовать псевдоним
integram_save_requisite_alias: { "requisiteId": 994790, "alias": "Новое название" }

// Разрешить/запретить NULL
integram_toggle_requisite_null: { "requisiteId": 994790 }

// Включить/отключить мультивыбор
integram_toggle_requisite_multi: { "requisiteId": 994790 }

// Изменить несколько атрибутов сразу
integram_modify_requisite_attributes: {
  "requisiteId": 994790,
  "alias": "Категория",
  "setNull": false,
  "multi": false
}

// Изменить порядок колонки
integram_set_requisite_order: { "requisiteId": 994790 }
```

### 5. Удалить колонку или таблицу

```json
// Удалить реквизит (колонку)
integram_delete_requisite: { "requisiteId": 994790 }

// Удалить тип (таблицу) без данных
integram_delete_type: { "typeId": 994762 }

// Удалить таблицу вместе со всеми данными (каскадное удаление)
integram_delete_table_cascade: { "typeId": 994762, "confirm": true }
```

### 6. Получить структуру таблицы

```json
// Метаданные типа (схема таблицы)
integram_get_type_metadata: { "typeId": 994762 }

// Полная структура (тип + все реквизиты + статистика)
integram_get_table_structure: { "typeId": 994762 }

// Получить схему всей базы данных
integram_get_schema: {}

// Метаданные всех типов
integram_get_all_types_metadata: {}
```

## DML-операции

DML (Data Manipulation Language) — операции по работе с данными: создание, чтение, изменение, удаление записей.

### 1. Создать запись (объект)

**Инструмент:** `integram_create_object`

```json
{
  "typeId": 994762,
  "value": "Смартфон Samsung Galaxy",
  "requisites": {
    "994790": "100500",
    "994791": "29999",
    "994792": "1"
  }
}
```

Где ключи в `requisites` — это ID реквизитов, значения — данные для записи.

> **Для ссылочных полей** передавайте **ID объекта** из таблицы-справочника (строкой):
> ```json
> "requisites": {
>   "994793": "994770"
> }
> ```
> Здесь `994793` — ID реквизита-ссылки, `994770` — ID объекта из справочника.

**Создать запись как дочернюю (иерархия внутри одной таблицы):**

```json
{
  "typeId": 994762,
  "value": "Подкатегория",
  "parentId": 994777
}
```

> ⚠️ `parentId` используется **только для иерархии внутри одной таблицы** (дерево объектов). Для связи между разными таблицами используйте ссылочные реквизиты!

### 2. Обновить запись

**Инструмент:** `integram_set_object_requisites` (рекомендуется)

```json
{
  "objectId": 994777,
  "requisites": {
    "994790": "200600",
    "994791": "34999"
  }
}
```

**Или полное обновление:**

**Инструмент:** `integram_save_object`

```json
{
  "objectId": 994777,
  "typeId": 994762,
  "value": "Смартфон Samsung Galaxy S24",
  "requisites": {
    "994790": "200600"
  }
}
```

### 3. Читать данные

```json
// Список объектов (постраничный)
integram_get_object_list: {
  "typeId": 994762,
  "params": { "pg": 1, "LIMIT": 20 }
}

// Все объекты (с автоматической пагинацией)
integram_get_all_objects: {
  "typeId": 994762,
  "pageSize": 100
}

// Количество записей
integram_get_object_count: { "typeId": 994762 }

// Данные для редактирования объекта
integram_get_object_edit_data: { "objectId": 994777 }

// Список всех таблиц
integram_get_dictionary: {}
```

### 4. Поиск данных

```json
// Поиск на естественном языке (рекомендуется)
integram_natural_query: {
  "question": "Найди все продукты дороже 10000 рублей",
  "targetTable": 994762
}

// SQL-подобный запрос
integram_smart_query: {
  "typeId": 994762,
  "filters": [...]
}
```

### 5. Удалить запись

```json
integram_delete_object: { "objectId": 994777 }
```

### 6. Работа с датами

Интеграм ожидает даты в формате ISO: `YYYY-MM-DDTHH:MM`

```json
"requisites": {
  "994795": "2024-06-15T10:30"
}
```

> ⚠️ **Формат `YYYYMMDD` (без дефисов) будет интерпретирован как Unix timestamp** — даты сохранятся неправильно! Всегда используйте формат с дефисами.

## Подчинённые таблицы vs Справочники

Это ключевое различие, которое необходимо понимать при проектировании структур данных в Интеграме.

### Что такое справочник (Lookup / Reference table)?

**Справочник** — это таблица со списком значений, на которые **ссылается** другая таблица.

**Признаки справочника:**
- Хранит небольшой, относительно стабильный набор значений
- Используется как источник для выпадающего списка (dropdown)
- Запись справочника может быть выбрана из многих других записей
- В редакторе структур отображается **с именем таблицы в скобках** `(Категории)`

**Пример:** Статусы заказов, категории товаров, типы организаций, регионы.

```
Таблица "Заказы"
├── Колонка "Статус" → ссылается на → Таблица "Статусы заказов"
│                                        ├── "Новый"
│                                        ├── "В работе"
│                                        └── "Закрыт"
```

**Как создать:**

```json
// 1. Создать таблицу-справочник
integram_create_lookup_table: {
  "tableName": "Статусы заказов",
  "values": ["Новый", "В работе", "Закрыт"],
  "unique": true
}

// 2. Добавить ссылочную колонку в основную таблицу
integram_add_reference_column: {
  "typeId": ID_ТАБЛИЦЫ_ЗАКАЗОВ,
  "referenceTableId": ID_ТАБЛИЦЫ_СТАТУСОВ,
  "alias": "Статус"
}
```

---

### Что такое подчинённая таблица (Subordinate table)?

**Подчинённая таблица** — это набор записей, принадлежащих конкретной родительской записи.

**Признаки подчинённой таблицы:**
- Записи существуют только в контексте родительской записи
- Без родительской записи они не имеют смысла
- Тесно связаны с одной конкретной родительской записью
- В редакторе структур отображается **без скобок** `Позиции заказа`

**Пример:** Позиции в заказе, телефоны контакта, строки счёта, части детали.

```
Таблица "Заказы"
│    Запись: Заказ №1001
│
└── Таблица "Позиции заказа"      (подчинённая)
         ├── Запись: Смартфон × 2 → 59998 руб.
         ├── Запись: Чехол × 1 → 1500 руб.
         └── Запись: Зарядка × 1 → 2000 руб.
```

**Как создать:**

```json
// 1. Создать подчинённую таблицу (baseTypeId: 3, НЕ ID родительской таблицы!)
integram_create_table_with_columns: {
  "tableName": "Позиции заказа",
  "baseTypeId": 3,
  "parentTableId": ID_ТАБЛИЦЫ_ЗАКАЗОВ,    // ← автоматически добавит ссылку
  "parentColumnAlias": "Заказ",
  "columns": [
    { "requisiteTypeId": 3,  "alias": "Товар" },
    { "requisiteTypeId": 13, "alias": "Количество" },
    { "requisiteTypeId": 13, "alias": "Цена" }
  ]
}
```

**Или вручную:**

```json
// Шаг 1: Создать таблицу с обычным baseTypeId: 3
integram_create_type: {
  "name": "Позиции заказа",
  "baseTypeId": 3
}

// Шаг 2: Добавить ссылочный реквизит на родительскую таблицу
integram_add_requisite: {
  "typeId": ID_ПОДЧИНЁННОЙ_ТАБЛИЦЫ,
  "requisiteTypeId": ID_ТАБЛИЦЫ_ЗАКАЗОВ    // ← ID родительской таблицы!
}

// Шаг 3: Назвать ссылочный реквизит
integram_save_requisite_alias: {
  "requisiteId": ID_НОВОГО_РЕКВИЗИТА,
  "alias": "Заказ"
}
```

### Таблица различий

| Характеристика | Справочник | Подчинённая таблица |
|---|---|---|
| **Цель** | Источник значений для выпадающего списка | Набор строк, принадлежащих родительской записи |
| **Связь** | Одна запись → ссылается на → значение в справочнике | Одна родительская запись → имеет → несколько дочерних записей |
| **Жизненный цикл записей** | Независимые, используются многократно | Существуют только в контексте родителя |
| **Примеры** | Статусы, категории, типы, регионы | Позиции заказа, телефоны контакта, строки счёта |
| **Отображение в редакторе** | `(Категории)` — **с скобками** | `Позиции заказа` — **без скобок** |
| **Кардинальность** | Много записей → одно значение справочника | Одна родительская запись → много дочерних |
| **Создание в MCP** | `integram_create_lookup_table` | `integram_create_table_with_columns` с `parentTableId` |

---

### Визуальная разница в редакторе структур

В редакторе структуры Интеграма:

- **Справочники** (reference/lookup) отображаются **с именем таблицы в скобках**: `(Статусы заказов)` — это говорит о том, что данное поле является ссылкой на запись в таблице-справочнике.

- **Подчинённые таблицы** отображаются **без скобок**: `Позиции заказа` — это говорит о том, что данная таблица является дочерней к текущей.

### Практический пример: сервисное ценообразование

Разберём пример из блога Интеграма:

**Задача:** Описать структуру данных для сервиса по ремонту, где:
- Услуги оказываются для различных моделей устройств
- У каждой услуги есть перечень необходимых запчастей с ценами
- Для каждой запчасти есть список поставщиков с ценами закупки

**Правильная структура:**

```
Справочник: Модели устройств           ← справочник (независимая таблица)
│    ├── iPhone 14
│    ├── Samsung Galaxy S24
│    └── ...

Таблица: Услуги
├── Колонка "Название" (SHORT)
├── Колонка "Модель" → (Модели устройств)  ← ссылка-справочник
│
├── Подчинённая: Цены на запчасти          ← подчинённая таблица
│       ├── Колонка "Услуга" → (Услуги)   ← ссылка на родителя
│       ├── Колонка "Цена" (NUMBER)
│       └── Колонка "Запчасть" → (Запчасти)  ← ссылка-справочник
│
└── Справочник: Запчасти                   ← справочник
        │    ├── Дисплей
        │    └── Аккумулятор
        │
        └── Подчинённая: Цены закупки      ← подчинённая таблица
                ├── Колонка "Запчасть" → (Запчасти)
                ├── Колонка "Цена закупки" (NUMBER)
                └── Колонка "Поставщик" → (Поставщики)  ← ссылка-справочник

Справочник: Поставщики                     ← справочник
    ├── ООО "Запчасти РФ"
    └── ...
```

## Практические сценарии

### Сценарий 1: Создать CRM для работы с клиентами

```json
// 1. Создать справочник статусов клиентов
integram_create_lookup_table: {
  "tableName": "Статусы клиентов",
  "values": ["Потенциальный", "Активный", "VIP", "Неактивный"],
  "unique": true
}

// 2. Создать таблицу клиентов
integram_create_table_with_columns: {
  "tableName": "Клиенты",
  "baseTypeId": 3,
  "columns": [
    { "requisiteTypeId": 3,  "alias": "Название компании" },
    { "requisiteTypeId": 3,  "alias": "ИНН" },
    { "requisiteTypeId": 3,  "alias": "Email" },
    { "requisiteTypeId": 3,  "alias": "Телефон" }
  ]
}

// 3. Добавить ссылку на справочник статусов
integram_add_reference_column: {
  "typeId": ID_ТАБЛИЦЫ_КЛИЕНТОВ,
  "referenceTableId": ID_СТАТУСОВ_КЛИЕНТОВ,
  "alias": "Статус"
}

// 4. Создать подчинённую таблицу контактов
integram_create_table_with_columns: {
  "tableName": "Контакты клиента",
  "baseTypeId": 3,
  "parentTableId": ID_ТАБЛИЦЫ_КЛИЕНТОВ,
  "parentColumnAlias": "Клиент",
  "columns": [
    { "requisiteTypeId": 3, "alias": "ФИО" },
    { "requisiteTypeId": 3, "alias": "Должность" },
    { "requisiteTypeId": 3, "alias": "Телефон" },
    { "requisiteTypeId": 3, "alias": "Email" }
  ]
}
```

### Сценарий 2: Создать запись и связать её со справочником

```json
// Предположим, в справочнике "Статусы клиентов" есть запись "Активный" с ID=100500

// Создать клиента и установить статус
integram_create_object: {
  "typeId": ID_ТАБЛИЦЫ_КЛИЕНТОВ,
  "value": "ООО Ромашка",
  "requisites": {
    "ID_РЕКВИЗИТА_ИНН":    "7712345678",
    "ID_РЕКВИЗИТА_EMAIL":  "info@romashka.ru",
    "ID_РЕКВИЗИТА_СТАТУС": "100500"         // ← ID объекта из справочника!
  }
}
```

### Сценарий 3: Создать подчинённые записи

```json
// Добавить контакт к клиенту
integram_create_object: {
  "typeId": ID_ТАБЛИЦЫ_КОНТАКТОВ,
  "value": "Иванов Иван Иванович",
  "requisites": {
    "ID_РЕКВИЗИТА_КЛИЕНТ":    "ID_ОБЪЕКТА_КЛИЕНТА",
    "ID_РЕКВИЗИТА_ДОЛЖНОСТЬ": "Директор",
    "ID_РЕКВИЗИТА_ТЕЛЕФОН":   "+7-900-123-45-67"
  }
}
```

## Частые ошибки

### Ошибка 1: Использование ID таблицы как `baseTypeId`

❌ **Неправильно:**
```json
integram_create_type: {
  "name": "Позиции заказа",
  "baseTypeId": 994762   // ID таблицы "Заказы" — НЕЛЬЗЯ!
}
// Ошибка: "Неверный базовый тип: 994762"
```

✅ **Правильно:**
```json
// Шаг 1: Создать с baseTypeId: 3
integram_create_type: {
  "name": "Позиции заказа",
  "baseTypeId": 3
}

// Шаг 2: Добавить ссылочный реквизит на родителя
integram_add_requisite: {
  "typeId": ID_ПОДЧИНЁННОЙ_ТАБЛИЦЫ,
  "requisiteTypeId": 994762
}
```

### Ошибка 2: Использование `parentId` для связи между разными таблицами

❌ **Неправильно:**
```json
// parentId используется для иерархии ВНУТРИ одной таблицы
integram_create_object: {
  "typeId": ID_ПОЗИЦИЙ,
  "value": "Товар",
  "parentId": ID_ЗАКАЗА   // НЕ работает для разных таблиц!
}
```

✅ **Правильно:**
```json
// Используйте ссылочный реквизит для связи разных таблиц
integram_create_object: {
  "typeId": ID_ПОЗИЦИЙ,
  "value": "Товар",
  "requisites": {
    "ID_РЕКВИЗИТА_ЗАКАЗ": "ID_ОБЪЕКТА_ЗАКАЗА"   // ссылка на заказ
  }
}
```

---

### Ошибка 3: Передача названия вместо ID в ссылочное поле

❌ **Неправильно:**
```json
"requisites": {
  "ID_РЕКВИЗИТА_СТАТУС": "Активный"   // НЕЛЬЗЯ! Передаётся название, а не ID
}
```

✅ **Правильно:**
```json
"requisites": {
  "ID_РЕКВИЗИТА_СТАТУС": "100500"   // Передаётся ID объекта из справочника
}
```

> При **чтении** ссылочные поля возвращают название (например, `"Активный"`).  
> При **записи** нужно передавать ID объекта (например, `"100500"`).

---

### Ошибка 4: Неправильный формат даты

❌ **Неправильно:**
```json
"requisites": {
  "ID_РЕКВИЗИТА_ДАТА": "20240615"   // Без дефисов → интерпретируется как Unix timestamp!
}
```

✅ **Правильно:**
```json
"requisites": {
  "ID_РЕКВИЗИТА_ДАТА": "2024-06-15T00:00"   // ISO формат с дефисами и временем
}
```

## Аутентификация

Перед использованием любых инструментов необходимо аутентифицироваться:

```json
integram_authenticate: {
  "serverURL": "https://app.integram.io",
  "database": "mydb",
  "login": "user@example.com",
  "password": "mypassword"
}
```

Или использовать существующую сессию:

```json
integram_set_context: {
  "serverURL": "https://app.integram.io",
  "database": "mydb",
  "token": "existing_token",
  "xsrfToken": "existing_xsrf_token"
}
```

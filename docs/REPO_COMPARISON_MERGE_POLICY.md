# Сравнение двух репозиториев и политика объединения

## Задача

Issue [#3377](https://github.com/ideav/crm/issues/3377): «Сравнить 2 репозитория и
выработать политику объединения: слить или выбрать один».

Сейчас параллельно двумя командами развиваются две версии сервиса:

- [`unidel2035/dronedoc2026`](https://github.com/unidel2035/dronedoc2026) — фронт **Vue**,
  бэкенд **Node.js**, БД **PostgreSQL**;
- [`ideav/crm`](https://github.com/ideav/crm) — фронт **HTML**, бэкенд **PHP**, БД **MySQL**.

Цель — объединить усилия двух команд в единый продукт, взяв лучшее из обоих, и сформировать
политику: слить кодовые базы или выбрать одну как основу. Учитываются **полнота реализации**,
**перспективность стека** и **набор функционала**, с количественным разбором метрик.

Документ опирается на проверяемые факты (метрики репозитория, `file:line`, GitHub API) и на
уже имеющиеся в репозитории стратегические оценки — план Vue-переписывания
[`VUE_INTEGRAM_REWRITE_PLAN.md`](VUE_INTEGRAM_REWRITE_PLAN.md) и оценку переноса серверного
ядра на Node/Python+PostgreSQL [`INDEX_PHP_PORT_NODE_PYTHON_POSTGRES.md`](INDEX_PHP_PORT_NODE_PYTHON_POSTGRES.md).

## Оговорка про доступ к данным (честность измерений)

Метрики `ideav/crm` измерены напрямую по рабочей копии репозитория и GitHub API на дату
2026-06-13 — это **проверяемые** числа.

Репозиторий `unidel2035/dronedoc2026` на момент подготовки документа **недоступен** под
учётной записью, выполняющей анализ: GitHub возвращает `404 Not Found` и для API
(`gh api repos/unidel2035/dronedoc2026`), и для клонирования, и для веб-страницы. Это значит,
что репозиторий **приватный** (либо переименован/перенесён). Публичного аналога с таким именем
в поиске GitHub нет; у владельца `unidel2035` 55 публичных репозиториев, среди них нет
`dronedoc2026`. Поэтому **количественные метрики `dronedoc2026` в этом документе не измерены**:
для его колонок приведён шаблон, который надо заполнить после выдачи доступа (см. раздел
[«Что нужно для завершения количественного разбора»](#что-нужно-для-завершения-количественного-разбора)).

Чтобы вывод не зависел от недостающих чисел, рекомендация ниже построена **по критериям с
явными порогами**: для каждой развилки указано, какое значение метрики `dronedoc2026` склоняет
решение в ту или иную сторону. Часть выводов опирается на заявленный стек `dronedoc2026`
(Vue/Node/PostgreSQL) — а это в точности то направление, которое в репозитории уже проработано
в двух упомянутых выше планах, что позволяет рассуждать предметно даже без доступа к коду.

## Что на самом деле представляет каждый репозиторий

Прежде чем сравнивать метрики, важно зафиксировать **природу** проектов — иначе «слить две
кодовые базы» читается буквально и неверно.

### `ideav/crm` — фронтенд и инструментарий над низкокодовой платформой Интеграм

`ideav/crm` — это **не самостоятельное CRUD-приложение «HTML+PHP+MySQL»**, а связка из:

- **серверного движка платформы Интеграм** — `index.php` (один файл, **11 554 строки**,
  EAV/триплет-модель `(id, up, ord, t, val)`, компилятор отчётов, движок прав, OAuth/JWT, SMTP,
  AI-прокси, биллинг; подробный разбор — в
  [`INDEX_PHP_PORT_NODE_PYTHON_POSTGRES.md`](INDEX_PHP_PORT_NODE_PYTHON_POSTGRES.md));
- **фронтенда из HTML-шаблонов рабочих мест** (`templates/*.html`) и крупных plain-JS модулей
  (`js/*.js`, генерируемый компонент `js/integram-table.js`);
- **инструментария**: MCP-сервер (`mcp-server/`, **60 инструментов** Интеграм API для
  AI-ассистентов), скрипты деплоя (`update.php`/`update.conf`), экспорт/импорт.

То есть «PHP/MySQL» здесь — это **интерпретатор низкокодовой платформы**, а не прикладной код
CRM. Прикладная логика рабочих мест живёт в декларативных метаданных Интеграма и в HTML/JS
шаблонах. Это ключевой факт для политики объединения: данные, права, отчёты и DDL/DML — на
стороне платформы, а UI — сменяемый слой.

### `dronedoc2026` — заявленный greenfield-стек Vue + Node.js + PostgreSQL

По формулировке issue это **вторая версия сервиса на современном стеке**: Vue-фронтенд,
Node.js-бэкенд, PostgreSQL. Это совпадает с двумя направлениями, которые `ideav/crm` уже
проанализировал как целевые для модернизации:

- **фронтенд → Vue 3 + Vite + TypeScript + Pinia + PrimeVue** (план
  [`VUE_INTEGRAM_REWRITE_PLAN.md`](VUE_INTEGRAM_REWRITE_PLAN.md));
- **серверное ядро → Node.js/Python + PostgreSQL** (оценка
  [`INDEX_PHP_PORT_NODE_PYTHON_POSTGRES.md`](INDEX_PHP_PORT_NODE_PYTHON_POSTGRES.md), где для
  Python уже есть два рабочих прототипа контроллера на FastAPI + PostgreSQL).

Иначе говоря, `dronedoc2026` по стеку — это материализация уже задокументированного «куда мы
хотим прийти». Главный неизвестный параметр — **насколько он полон** относительно функционала
`ideav/crm`.

## Оценка по трём критериям из issue

### 1. Полнота реализации

Здесь `ideav/crm` имеет **измеримое и крупное преимущество**: 46 готовых рабочих мест,
1 739 влитых PR, 1 564 закрытых issue за ~4,5 месяца, развитый инструментарий и документация.
Это работающий продукт в активной эксплуатации и разработке.

Полнота `dronedoc2026` **не измерена** (нет доступа). Это **главная развилка** всей задачи:
если `dronedoc2026` покрывает лишь часть из 46 рабочих мест и не имеет аналога компилятора
отчётов/движка прав Интеграма — он не может быть выбран «как есть» без потери функционала.

### 2. Перспективность стека

Здесь преимущество у `dronedoc2026` (и шире — у направления Vue/Node/PostgreSQL):

- **Фронт:** Vue 3 + TypeScript даёт компонентность, типобезопасность, тестируемость и
  экосистему — против крупных plain-JS модулей и генерируемого бандла `integram-table.js`.
  Сам `ideav/crm` уже признал это направление целевым в
  [`VUE_INTEGRAM_REWRITE_PLAN.md`](VUE_INTEGRAM_REWRITE_PLAN.md) (Vue 3 + Vite + Pinia +
  PrimeVue, причём PrimeVue выбран как естественный из-за уже используемых PrimeIcons).
- **Бэк:** Node.js — модульность против одного PHP-файла на 11,5 тыс. строк и **единый язык**
  с фронтендом и существующим `mcp-server/` (он уже на Node). В оценке
  [`INDEX_PHP_PORT_NODE_PYTHON_POSTGRES.md`](INDEX_PHP_PORT_NODE_PYTHON_POSTGRES.md) для полного
  «как есть» движка рекомендован именно Node.js + PostgreSQL (моноязычный репозиторий +
  `mcp-server/index.js` как живой референс сетевого контракта).
- **БД:** PostgreSQL строже к типам и богаче возможностями, но переход с MySQL — **главный
  риск** переноса (текстовая EAV-модель против строгой типизации; компилятор отчётов на
  диалекте MySQL). Это не «бесплатное» преимущество — см. тот же документ.

Важно: перспективность стека `dronedoc2026` **совпадает с уже выбранным вектором модернизации**
`ideav/crm`. Это аргумент не «против одного из репозиториев», а «за заимствование стека».

### 3. Набор функционала

`ideav/crm` (измеримо): таблицы/отчёты (компонент IntegramTable: бесконечный скролл, 13+ типов
фильтров, сортировка/группировка, inline-edit, Excel/CSV-экспорт, шеринговые ссылки), канбан,
дашборды с формулами, формы объектов, кабинет, специализированные рабочие места (atex,
sportzania, connector-1c, ProcVac, SmartQ и др.), мультиарендность, права/гранты, OAuth/JWT,
почта, AI-чат-прокси, биллинг, MCP-инструменты, деплой.

Набор функционала `dronedoc2026` **не измерен**. Для корректного сравнения нужна **матрица
покрытия** «рабочее место → есть в `dronedoc2026`? → паритет по функциям?» (шаблон ниже).

## Варианты политики объединения

### Вариант A. Выбрать `ideav/crm` как основу, заимствовать стек `dronedoc2026`

Продуктовой базой остаётся `ideav/crm` (он измеримо полнее и активнее), а перспективный стек
`dronedoc2026` вносится **инкрементально** по уже имеющимся планам:

- фронт мигрируется на Vue **поэкранно** через параллельную Vue-оболочку и feature flag
  (`VUE_INTEGRAM_REWRITE_PLAN.md`), а не «big bang»;
- серверное ядро при необходимости переносится на Node + PostgreSQL **strangler-методом**
  (`INDEX_PHP_PORT_NODE_PYTHON_POSTGRES.md`): новый сервис отвечает по тому же контракту,
  трафик переключается поэндпоинтно, параллельный прогон старого и нового движка — регрессия;
- из `dronedoc2026` берутся **готовые Vue-компоненты/экраны и Node-слой**, если они
  закрывают шаги этих планов лучше, чем писать с нуля.

**Плюсы:** не теряется измеримо большой функционал и история; модернизация идёт без простоя;
совпадает с уже принятым в репозитории вектором; обратимо (rollback по feature flag).
**Минусы:** требует адаптации кода `dronedoc2026` под контракт Интеграма; часть его наработок
может не подойти под EAV-модель и движок отчётов/прав.

### Вариант B. Выбрать `dronedoc2026` как основу

Имеет смысл **только если** измерение покажет, что `dronedoc2026` уже покрывает сопоставимый
набор рабочих мест **и** имеет аналоги компилятора отчётов, движка прав и мультиарендности
Интеграма (а не только CRUD). По двум прототипам Python-контроллера, разобранным в
[`INDEX_PHP_PORT_NODE_PYTHON_POSTGRES.md`](INDEX_PHP_PORT_NODE_PYTHON_POSTGRES.md), самая
сложная часть (конструктор запросов/отчётов, права/гранты) обычно реализована **последней**
или отсутствует — поэтому этот вариант **пока не подтверждён данными** и при текущих
измеримых фактах ведёт к потере функционала `ideav/crm`.

### Вариант C. Полное слияние двух кодовых баз «как есть»

Технически непрактично: paradigмы разные (низкокодовая EAV-платформа на PHP/MySQL против
greenfield-приложения на Vue/Node/PostgreSQL). Прямого «merge» исходников не получится — это
де-факто переписывание под один контракт, то есть всё равно Вариант A или B, но дороже и без
ясной точки отсечения. Не рекомендуется.

## Рекомендация

**Провизорная рекомендация — Вариант A: основа `ideav/crm`, стек заимствуется из
`dronedoc2026` инкрементально.** Обоснование на измеримых фактах:

1. **Полнота сейчас измеримо на стороне `ideav/crm`** (46 рабочих мест, 1 739 PR, 4,5 месяца
   интенсивной разработки, развитый инструментарий и KB). Выбор `dronedoc2026` «как есть»
   при неподтверждённой полноте — это риск потери функционала.
2. **Перспективный стек уже выбран** самим `ideav/crm` (Vue для фронта, Node+PostgreSQL для
   бэка) и описан планами миграции. `dronedoc2026` — материализация этого вектора, поэтому его
   ценность максимальна как **источник готовых Vue/Node-наработок**, а не как замена продукта.
3. **Команды уже частично объединены**: владелец `dronedoc2026` (`unidel2035`) — третий по
   числу коммитов контрибьютор `ideav/crm` (342 коммита). Объединение усилий вокруг одной
   основы организационно реалистично.
4. **Риск миграции MySQL→PostgreSQL** — главный технический риск (см.
   `INDEX_PHP_PORT_NODE_PYTHON_POSTGRES.md`), поэтому переход должен быть поэтапным
   (strangler), а не разовой заменой движка на чужую кодовую базу.

**Условие пересмотра в пользу Варианта B:** если измерение `dronedoc2026` покажет
(a) покрытие ≥ ~80% рабочих мест `ideav/crm` с паритетом функций, **и** (b) рабочие аналоги
компилятора отчётов, движка прав и мультиарендности, **и** (c) сопоставимую активность/тесты —
тогда базой разумнее сделать `dronedoc2026`, а из `ideav/crm` переносить контент/метаданные и
недостающие рабочие места. До получения этих чисел вывод остаётся за Вариантом A.

## Что нужно для завершения количественного разбора

Чтобы заполнить пустые колонки и сделать вывод окончательным:

1. **Выдать доступ** на чтение к `unidel2035/dronedoc2026` учётной записи, выполняющей анализ
   (репозиторий приватный — GitHub возвращает 404). Альтернатива — приложить к issue снимок
   репозитория (`.zip`), как это уже делалось для прототипа `horhe-dvlp/integram` в
   [`INDEX_PHP_PORT_NODE_PYTHON_POSTGRES.md`](INDEX_PHP_PORT_NODE_PYTHON_POSTGRES.md).
2. После доступа — снять метрики `dronedoc2026` по шаблону выше (LOC, коммиты, PR, тесты, CI).
3. Заполнить **матрицу покрытия рабочих мест** (ниже) и пересчитать процент паритета функций.
4. Зафиксировать наличие/отсутствие в `dronedoc2026` критичных подсистем Интеграма:
   компилятор отчётов, права/гранты, мультиарендность, OAuth/JWT, почта, AI-прокси, биллинг,
   файлы, экспорт/импорт.
5. На основании чисел — подтвердить Вариант A или переключиться на Вариант B по порогам выше.

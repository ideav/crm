# Алгоритм планирования производства

_Слиттер · atex · `production-planning.js` + `cut-layout.js`_

Документ описывает полный конвейер формирования производственных заданий: от списка
позиций заказов до раскладки полос, выбора станка, очереди резок и расписания по дням.
Цель документа — объяснить алгоритм человеку: что делает система, какие правила нельзя
нарушать, где есть компромиссы и какие части кода отвечают за каждый шаг.

> **Источник истины по алгоритму планирования (#3717).** Перед правкой логики очереди,
> раскладки, расписания, переналадки или фольги сверяйся с этим документом и обновляй его
> в том же PR. Код: `download/atex/js/production-planning.js`,
> `download/atex/js/cut-layout.js`.

> **Важное изменение #3600.** Любая смена ножей стоит фиксированные **30 минут** на весь
> набор, независимо от числа переставленных ножей. Смена сырья/намотки считается отдельно
> и стоит **15 минут** по умолчанию. Это ФИЗИЧЕСКОЕ время наладки (пишется в «Наладка ножей,
> мин»); оно симметрично — снять или доставить ножи одинаково 30 мин.
>
> **Направление смены ножей при выборе порядка #3996.** Для ВЫБОРА ПОРЯДКА резок (не для
> тайминга) доставить ножи (число полос стало **больше**) дороже, чем снять (ТЗ §8 п.1:
> `KNIVES_INCREASE_COST_MN` = 50 > `KNIVES_CHANGE_COST_MN` = 30). Движок очереди складывает
> порядок по `sequencingCost` = физическая переналадка (`changeoverCost`) + штраф за рост числа
> полос. Поэтому наборы идут **по убыванию числа полос**, и это не тай-брейк, а стоимость: не
> сбивается разницей по сырью/партии и работает на любой длине очереди. Физический тайминг при
> этом прежний (30 мин в обе стороны).

## Доступ к данным: метаданные и отчёты (#3897)

> **Пользователь в любой момент может переставить колонки таблицы, добавить новые и удалить
> существующие. Это НЕ должно ничего ломать.** Любой код, читающий данные Integram, обязан
> быть устойчивым к изменению порядка и состава колонок.

Правила (обязательны для `production-planning.js`, `cut-layout.js` и всех модулей atex,
читающих данные):

1. **Никогда не обращайся к колонкам по позиции в массиве.** В ответе
   `object/{table}?JSON_OBJ` каждая запись приходит как `{ i, u, o, r:[…] }`, где порядок
   значений в `r` повторяет порядок **реквизитов в метаданных** — а он меняется, как только
   пользователь переставит колонки. Резолвь индекс по **имени** реквизита через свежие
   метаданные: `columnIndex(meta, reqName)` / `reqIdByName(meta, …)` (см. `mapCutRecord`).
   Единственное допустимое позиционное обращение — `r[0]` (главное значение записи): оно
   всегда первое по структуре, а не по порядку реквизитов.

2. **Метаданные тяни заново на каждой загрузке** (`loadMetadata` → `GET metadata`). Не зашивай
   в константы id или порядок реквизитов. Имена таблиц/реквизитов держим в `TABLE` / `CUT_REQ`
   / `DOWNTIME_REQ` и резолвим по ним. Пропажу или переименование колонки обрабатываем явной
   диагностикой («в отчёте нет колонки …»), а не падением и не молчаливо неверным значением.

3. **Предпочитай отчёты прямому чтению таблиц.** `report/{name}?JSON_KV` отдаёт данные парами
   «имя колонки → значение» и **вообще не зависит** от порядка и состава колонок таблицы —
   переживает любые перестановки. Основная загрузка очереди и обеспечения идёт отчётом
   `cut_planning?JSON_KV` именно поэтому. Прямое `object/{table}?JSON_OBJ` оставляем там, где
   отчёта нет или записей мало, и всегда декодируем по имени (п. 1). Новое поле, нужное логике,
   **добавляй в отчёт**, а не считывай позиционно из таблицы.

> **Известное исключение.** `loadDaySettings` читает системную таблицу «Настройка» позиционно
> (`r[0]`=ключ, `r[1]`=тип/БД, `r[2]`=значение) — её состав фиксирован системой. При переносе
> на чтение по имени резолвить тип/значение через `columnIndex`, не менять схему вслепую.

> Симптом нарушения правила (#3897): после того как заказчик переставил колонки таблицы
> «Задание в производство» (1078), **позиционная** разборка `r` (в отладочных запросах) стала
> выдавать неверные поля. Сам код страницы читает по имени и не пострадал — правило
> зафиксировано, чтобы так оставалось и впредь, в том числе во всех новых модулях.

## Общий конвейер

Все шаги идут от входных данных до записи плана в базу.

| # | Функция | Что делает |
|---|---|---|
| 1 | `positions_list` | Отчет с позициями заказов: сырье, ширина, количество, срок, втулка, направление намотки, длина прогона. |
| 2 | `rowsToGenPositions` | Нормализует строки отчета: числа → числа, пустые значения → `0`/`''`, считает `dueKey` (`ГГГГММДД`), проставляет `isFoil`. |
| 3 | `groupPositionsByPlanningProfile` | Группирует позиции по профилю: сырье + намотка + длина прогона + лидер + втулка + число втулочных полос (#3812). Разные профили физически не смешиваются в одной резке. |
| 4 | `layoutPositionGroups` | Разбивает профиль по `sleeveId`, чтобы в одной резке была одна втулка. |
| 4а | втулочные полосы 0.5"/110 (#3812, #3872) | Резервирует ширину джамбо под полосы 110 мм и дописывает их в раскрой (`appendCoreStrip`). Узкие риббоны (<55 мм) на втулке 0.5" исключаются. #3872: полосы привязываются к уже заказанным позициям 110 мм того же заказа (`selectCoreStripFillers`), не нашлось — синтетика. |
| 5 | `dueWindowGroups` | Объединяет близкие сроки в окно 3 дня. Срочные позиции обрабатываются раньше бессрочных. |
| 6 | `planLayouts` | Строит раскладки полос: seed «1 заказ = 1 резка», затем выгодные слияния по B&B. |
| 7 | `capStockToHeadroom` | Обрезает добор «впрок» по лимиту максимального запаса. Ширины сверх лимита убираются из раскладки. |
| 8 | `chooseSlitterBySetup` | Выбирает станок: стоп-лист сырья, **лимит ширины джамбо** (`«Код»` станка, #4006), вместимость дня, совместимость заправки, баланс загрузки. |
| 9 | `orderCuts` | Упорядочивает резки станка: фольга в конец, блоки сырья, минимум переналадки, число полос по убыванию (штраф за рост полос, #3996). Срок изготовления в упорядочивании не участвует (#3974). |
| 10 | `buildSchedule` / `planCutOperations` | Раскладывает очередь по сменам и дням от «С», дробит длинные резки, создает/обновляет продолжения. Порядок ВНУТРИ дня собирает жадный упаковщик `splitMachineQueue` (gapFill): следующая резка — нефольга раньше фольги (#3717), затем минимум переналадки от предыдущей, затем БОЛЬШЕ полос раньше (#3999), затем очередь. Число полос по дню убывает (#3130) — тай-брейк не ухудшает суммарную переналадку. |

## Модель размещения: `[С; По]` — окно, а не фильтр (#3974)

`planCutOperations` планирует **всё необеспеченное** — открытые задания (статус ≠ «Завершён»),
за **любые** даты. Диапазон `[С; По]` из фильтра — это **окно размещения**, а не фильтр входа:

- **Вход** — вся непокрытая потребность (`self.cuts` без «Завершён»). Обеспеченные
  («Завершён») задания не трогаем — остаются как есть.
- **База размещения** — «С» (день 0). Ничего не ставим раньше «С».
- **Задания ПРОШЛЫХ дней (раньше «С») из входа исключаются (#4294).** Планировщик кладёт всё от «С»
  вперёд, поэтому задание, уже стоящее на дне < «С», нельзя пере-планировать — иначе оно затянется
  в «С». `🔒`-задание прошлого дня движок и так держит (`fixedDay<0`, ниже), а НЕзафиксированное
  раньше **затягивалось в «С» и садилось в середину дня** (issue #4294). Поэтому на общем пути
  (генерация/«Упорядочить»/удаление/↑↓) `buildSequenceOps` убирает из `planInput` **всю цепочку** с
  незафиксированной головой раньше «С» (`cutsBeforeWindowToKeep`, по дню ГОЛОВЫ — чтобы не осиротить
  продолжение в окне, ср. #4292). Ручной перенос `🗓` (`moveScope`) имеет свой scope — его не трогаем.
  - **Заправка станка к началу окна — из исключённых резок, а не из `prev_cut_setup` (#4300).** Исключив
    резки прошлых дней, теряем контекст: станок к «С» уже несёт наладку вчерашней резки ПЛАНА. Если для
    первой резки окна взять `carryPrevSetup` из `prev_cut_setup` (слепок ПОСЛЕДНЕЙ ФИЗИЧЕСКИ НАЧАТОЙ резки,
    обычно **не** вчерашней резки плана), `splitMachineQueue` зарядит ей переналадку от той СТАРОЙ
    конфигурации, тогда как хранимые колонки (`computeCutSetupUpdates`, по ВСЕЙ группе станка) считают её
    near-zero переналадкой от вчерашней. Окно упаковщика становится длиннее хранимой наладки → **«дыра»
    после первого задания дня** размером `changeover(prev_cut_setup, перваяРезка)` (issue #4300). Фикс:
    `prevSetupFromExcludedCuts` восстанавливает заправку = конфигурация ПОСЛЕДНЕЙ (по `planStart`)
    исключённой резки станка, ею переопределяем `prevSetupBySlitter` — упаковщик и колонки сходятся, дыры нет.
- **Набивка** — дни забиваются плотно от «С» до ёмкости (≈450–460 мин), задания переливаются
  за «По», если не влезают в окно. Верхней границы размещения нет.
- **Держит день только `🔒` «Зафиксировано».** Свободные задания в окне (в т.ч. ручной перенос `🗓`
  без `🔒`) при «Создать» перепаковываются от «С». `🔒`-резка с днём раньше «С» остаётся как
  есть — её не двигаем и не удаляем; незафиксированное задание раньше «С» тоже остаётся на месте
  (исключено из входа, #4294 выше).
- **Срок изготовления** задаёт **порядок размещения по дням** (EDD, ТЗ §14 — самый большой вес):
  чем раньше срок, тем раньше день (`deadlineCostFor = dueDay × DEADLINE_COST_MN` в
  `splitMachineQueue`, #4050/#4059), просроченное — вперёд, задание не уезжает за свой срок пока
  раньше есть ёмкость; постановка **после** срока дополнительно штрафуется в `transitionCost`. Срок
  также красит строку очереди (`dueColorClass`) и группирует позиции в резки при генерации (окно
  срока, Фаза 1).
- **Фольга за своим сроком вытесняет поздне-срочную нефольгу (#4068, ТЗ §12).** Фольга ставится в
  конец дня (`isFoil` — старший ключ выбора, #3717), поэтому рисуется после ВСЕЙ нефольги: крупная
  фольга, чьи дни до срока заняты нефольгой, переливалась ЗА срок (issue #4068: «Фольга горячего
  тиснения» срока 30.06 → 01.07). Фикс — **резервирование хвоста дня** (детерминированный второй
  проход упаковщика, `planMachineSegs`): пробный проход выявляет фольгу с `деньРазмещения > срок`;
  `computeFoilDeadlineReservation` группирует фольгу по дню срока и для дней срока с просроченной
  фольгой резервирует в конце дня суммарные минуты фолег этого срока (`foilReserveByDay`,
  `resFoilDayByCut`), после чего `splitMachineQueue` пакует заново: нефольга (и нерезервная фольга)
  видит ёмкость дня **минус резерв** (`reserveForDay`, вычитается из `availFor`/`effCapacity` для
  каждого нерезервного пика) и поздне-срочная переливается позже, а резервная фольга берёт полный
  зарезервированный хвост своего срока. Вытесняется **только** нефольга с более поздним сроком или
  без срока (гейт по вытесняемым минутам в `computeFoilDeadlineReservation`) — ранне-срочную за её
  срок не двигаем; вытеснять нечем → резерв не ставим (фольга остаётся за сроком, честный конфликт
  ёмкости). Несколько фолег одного срока сходятся в один день. Действует при `deadlineAware`
  (генерация, «Упорядочить», а с #4074 — и перенос `🗓`, который теперь пересобирает по срокам).
- **Срок для плашки и цвета** карточки берётся из позиции активного `positions_list`
  (`genPositions`); если позиция уже вышла из активного `positions_list` (заказ закрыт/выполнен,
  либо задание — заказная сирота дробления, чьё Обеспечение восстановлено по `cut_planning` #4175),
  срок читается из отчёта `cut_planning` (колонка `due_date` → `supply.dueKey`), чтобы плашка
  «срок» на карточке не пропадала (`cutDueKeys(..., includeSupplyFallback=true)`, #4051). Тот же
  фолбэк применяет панель «просрочено» (`countOverdueCuts`).
- **Размещение по дням** (`dueKeyByCut`/`dueDayByCut` в `buildSequenceOps`) при **генерации** и
  **«Упорядочить»** использует только `genPositions` — фолбэк на `cut_planning` туда **не идёт**
  (иначе задания без активной позиции получают срок «сегодня» и конкурируют за забитый день 0,
  давая просрочку уже при первом планировании, #4197). Исключение — **ручной перенос `🗓`** (#4074,
  `moveScope.pinCutIds` в `buildSequenceOps`): на этом пути фолбэк **включается**
  (`cutDueKeys(..., includeSupplyFallback=true)`, #4195), иначе задание-сирота дробления со сроком
  только в `cut_planning` при переносе встаёт по переналадке за своим сроком, а панель тут же
  считает его просроченным.

Прежние механизмы «фильтр входа по `[С; По]`» (`#3660`), «якорь дня по хранимой „Дате план“»
(`#3658`) и «ранний срок — ранний день» (`#3815/#3820/#3826/#3970`) сняты этой моделью.

## Ограничения алгоритма

Бизнес-ограничения задает технология. Технические ограничения сужают перебор, чтобы план был
расчетным и удобным оператору. В таблицах ниже статус означает:

- **[в коде]** — правило реализовано;
- **[требование]** — правило согласовано, но код пока ведет себя по старому базовому правилу;
- **[уточняется]** — требование пока не финализировано.

### Бизнес-ограничения

| Ограничение | Статус | Что ограничивает / зачем |
|---|---|---|
| Одна втулка на резку | [в коде] | Все рулоны одной резки надеваются на одну втулку. Позиции с разным `sleeveId` идут в разные резки. |
| Одно направление намотки | [в коде] | `IN` и `OUT` не смешиваются в одной резке: это физическое ограничение станка. |
| Одна длина прогона | [в коде] | Все полосы одной резки наматываются за один прогон, поэтому длина у всех позиций должна совпадать. **Длина прогона резки = «Длина, м» позиции ЗАКАЗА (`cutRunLength` по `positionLengthMap`), а НЕ «Метраж, м» обеспечения (#4301).** Дробление/склейка искажали метраж обеспечения, `cutRunLength` брал его как длину и записывал обратно в «Длина, м» резки → длина «пересчитывалась» и удваивалась (600→1200→2400). Длину НИКОГДА не выводим из метража; нет позиции (сток/сирота) → сохранённая длина резки как есть. |
| Один лидер на резку | [в коде] | Резка заправляется одним лидером. Позиции с разным лидером идут в разные резки. |
| Ширина полос ≤ ширины джамбо | [в коде] | Позиция шире джамбо не планируется и попадает в `skipped` с причиной «шире джамбо». |
| Отход в пределах допуска | [в коде] | Остаток ширины джамбо должен быть в пределах `tolerance`, иначе алгоритм ищет добор ходовыми ширинами. |
| Стоп-лист сырья по станку | [в коде] | Сырье нельзя назначить на станок, где оно запрещено. Например, фольга может быть разрешена только на части станков через справочник. |
| Лимит ширины джамбо по станку (#4006) | [в коде] | У станка в поле «Код» задано условие ширины (напр. `j<1000` = «не ставить ширину больше метра»). Сырьё, чья **номинальная ширина** не удовлетворяет условию, на этот станок не ставится (`isSlitterWidthBlocked`). Пример: `MWR500L` (номинал 1000) исключён со станков 1 и 4. Пусто/нераспознанный код → без ограничения; условие сверяется как контекст `j` фактической ширины (`actualWidthCodeMatches`). |
| Фольга в конце дня | [в коде] | Фольга загрязняет станок, поэтому длинная фольга должна стоять после обычных резок каждого дня. |
| Короткая фольга ≤ 120 минут может идти не в конец дня | [требование] | Если суммарное время резки фольги не превышает 120 минут, она не успевает сильно загрязнить станок. В текущем коде любая фольга пока все равно считается фольгой и ставится в конец дня. |
| Втулка 0.5" — сетка ширин (#3812) | [в коде] | На втулке 0.5" риббон у́же 55 мм не производится (позиция в `skipped`). Ширина > 70 мм режется по обычному правилу. См. «Втулочные полосы 0.5"» ниже. |
| Втулочные полосы 0.5" / 110 мм (#3812, #3872) | [в коде] | При втулке 0.5" шириной 110 мм в раскрой добавляются полосы 110 мм: ширина 55-57 → 2 полосы, 63-64 → 1 полоса. Полосы занимают ширину джамбо той же резки. #3872: если 110 мм уже заказаны (позиции того же заказа) — полосы обеспечивают эти позиции; иначе синтетика. |

### Технические ограничения

| Ограничение | Статус | Что ограничивает / зачем |
|---|---|---|
| ≤ 3 позиции на резку | [в коде] | `maxPositionsPerCut`. Сокращает число вариантов слияний и не делает резку слишком сложной. |
| ≤ 3 ширины на резку | [в коде] | `maxWidthsPerCut`. Слишком разношерстная раскладка хуже читается и сложнее для оператора. |
| Слияние только при выгоде | [в коде] | `planLayouts` сливает две раскладки только если становится лучше по цели: прогоны ↓, затем скрап ↓, затем отход ↓. |
| Окно срока 3 дня | [в коде] | Позиции с близкими сроками объединяются в один кластер. Бессрочные позиции идут отдельным последним кластером. |
| Сырье — непрерывным блоком | [в коде] | Алгоритм избегает шаблона A → B → A, потому что возврат к A дает лишнюю переналадку. |

## Цена нарушения правил

Планирование — это не один критерий, а компромисс. Часть правил жесткая: план, который их
нарушает, недопустим. Остальные факторы можно считать как штраф и затем подбирать веса на
реальных планах.

| # | Фактор | Тип | Натуральная цена нарушения | Текущая обработка | Стартовый вес |
|---|---|---|---|---|---|
| 1 | Срок изготовления | размещение (EDD) | заказ уходит за свой срок | `deadlineCostFor = dueDay × DEADLINE_COST_MN` в `splitMachineQueue`: раньше срок → раньше день (доминирует над переналадкой); постановка после срока штрафуется в `transitionCost` (#4050/#4059); красит строку (`dueColorClass`) | 100 |
| 2 | Фольга в конце дня | бизнес-правило | загрязнение станка и внеплановая уборка перед обычной резкой | `isFoil` уводит длинную фольгу в конец дня (#3717) | 80 |
| 3 | Смена ножей | мягкий, в минутах | `KNIFE` = 30 минут (тайминг); при выборе порядка рост полос дороже убыли (#3996) | физтайминг фиксирован (#3600/#3666); порядок — через `sequencingCost` с направленным штрафом | 30 / 50 |
| 4 | Смена сырья / намотки / партии | мягкий, в минутах | `MATERIAL_WINDING` = 15 минут | минимизируется через `changeoverCost` | 15 |
| 5 | Равномерная загрузка станков | мягкий | переполненный станок при свободном соседнем | `chooseSlitterBySetup` сначала смотрит `overflow`, затем группировку (#3801/#3830) | 20 |
| 7 | Убывание числа полос | стоимость выбора порядка (#3996) + тай-брейк упаковщика (#3999) | доставить ножи дороже, чем снять — растить число полос по очереди накладно | в `orderCuts` — штраф за рост полос в `sequencingCost` (`KNIVES_INCREASE` − `KNIVES_CHANGE`); в упаковщике дня `splitMachineQueue.selectByConfig` — число полос по убыванию ТАЙ-БРЕЙКОМ ниже переналадки (#3999): реальный порядок дня собирает упаковщик, поэтому убывание нужно и там. Стратегия `FATIGUE` по-прежнему ставит сложные раньше | 50 / 30 |
| 8 | Дробление резки по дням | мягкий | продолжение на следующий день и риск лишнего простоя | `splitMachineQueue`, `setup-only` хвост, переиспользование продолжений | 3 |

**Приоритет по умолчанию сейчас:** фольга в конец → минимум переналадки и группировка
сырья → баланс станков → **число полос по убыванию** (штраф за рост полос в `sequencingCost`,

порядок дня собирает упаковщик, а не `orderCuts`). Срок изготовления в приоритете размещения не участвует — только красит строку (#3974).

## Фаза 1. Подготовка и группировка позиций

### Что происходит

Система получает список позиций заказов из отчета `positions_list`. Каждая позиция — это строка
будущего производственного задания: какое сырье нужно, какой ширины ролик, сколько рулонов,
к какому сроку нарезать, какая втулка, какая намотка и какая длина прогона.

### Шаг 1.1. Нормализация строк (`rowsToGenPositions`)

Каждая строка отчета превращается в объект `genPosition`:

- числовые поля (`width`, `qty`, `length`, `windLength`) приводятся к числам;
- пустые числа становятся `0`, пустые строки остаются `''`;
- дата срока становится числовым ключом `dueKey` (`ГГГГММДД`), отсутствие срока — `Infinity`;
- позиция считается утвержденной, если утвержден заказ или сама позиция;
- флаг `isFoil` ставится по типу сырья, если он содержит «фольг»;
- поля `leader`, `sleeveId`, `orderId` остаются строковыми идентификаторами.

### Шаг 1.2. Профиль планирования (`groupPositionsByPlanningProfile`)

Позиции группируются по ключу совместимости:

```text
сырье + направление намотки + длина прогона + лидер + втулка + число втулочных полос
```

Позиции с разным профилем не попадут в одну резку. Это не оптимизационное пожелание, а
физическая совместимость: другой материал, намотка, длина, лидер или втулка требуют отдельной
резки. **Число втулочных полос** (#3812) — отдельное измерение: позиции с разной потребностью
в полосах 110 мм (0/1/2, см. [Шаг 2.0](#шаг-20-втулочные-полосы-05--110-мм-3812)) идут в разные
резки, потому что у каждой своя зарезервированная под полосы ширина джамбо. Для обычных позиций
(не втулка 0.5"/110) число полос = 0 у всех, поэтому разбиения нет.

### Шаг 1.3. Разбивка по втулке (`layoutPositionGroups`)

Если в группе все равно оказались разные `sleeveId`, она делится на подгруппы. Одна подгруппа —
одна втулка. В каждой резке все рулоны одного диаметра.

### Шаг 1.4. Кластеризация по сроку (`dueWindowGroups`)

Внутри подгруппы позиции сортируются по сроку. Позиции с разницей сроков до 3 дней объединяются
в один кластер и могут быть сведены в одну раскладку. Бессрочные позиции идут отдельным
последним кластером.

## Фаза 2. Раскладка полос (`cut-layout.js`)

### Что происходит

Для каждого кластера решается задача раскладки: какие полосы уложить в ширину джамбо, сколько
прогонов нужно, можно ли объединить несколько заказов в одну резку и чем добить остаток джамбо.

### Шаг 2.0. Втулочные полосы 0.5" / 110 мм (#3812)

Втулка задаётся записью «Диаметр втулки»: реквизит «Дюймы» даёт диаметр (0.5"). Ширина втулки
(57 или 110) берётся из реквизита «Ширина втулки, мм», а если он не заведён — из **названия
записи** («Втулка картонная 0.5" ширина 110 мм» → 110, `parseSleeveWidthFromName`). Поэтому
правило работает на текущих данных без изменения схемы. Обе величины проставляются позициям в
`annotatePositionsCutWidth` (`sleeveInches`, `sleeveWidth`).

Правила для втулки **0.5"** считаются по **номинальной** ширине заказа:

- риббон у́же **55 мм** не производится — позиция исключается из планирования и попадает в
  `skipped` («втулка 0.5″: ширина < 55 мм»);
- при ширине втулки **110 мм** в раскрой добавляются полосы 110 мм:
  - ширина продукта **55-57 мм** → **2 полосы** 110 мм (пример: 55×12 + 110×2);
  - ширина продукта **63-64 мм** → **1 полоса** 110 мм (пример: 64×12 + 110×1);
  - прочие ширины (58-62, 65-70) полос не получают; ширина **> 70 мм** режется по обычному
    правилу (как втулка 1"), полос нет.

Полосы 110 мм **занимают ширину джамбо той же резки**: перед укладкой продукта ширина джамбо
уменьшается на `count × 110` (`effJumbo`), а затем `appendCoreStrip` дописывает полосы в раскрой.
Поэтому в раскрое продуктовых полос становится меньше ровно на занятую полосами 110 мм ширину.
Втулочные полосы помечены `core:true`: они показываются «Партией ГП» наравне с продуктом, но
`capStockToHeadroom` их не урезает, а **число прогонов от них не зависит** (фикс. число полос
задаёт продукт-носитель; ширины «только из `core`-полос» исключены из расчёта прогонов —
`coreOnlyStripWidths` в `plannedRunsForLayout`).

**Использование уже заказанных втулок 110 мм (#3872).** Дополнительные втулки 110 мм могут быть
уже в заказе (иногда их делают, иногда нет). При планировании носитель полос НЕ режет их «впустую»,
а сначала ищет **подходящие позиции заказа 110 мм** — необеспеченные, **того же заказа** (обычно
они идут в рамках одного заказа), того же сырья/намотки/длины джамбо, фактической ширины 110 мм
(`isCoreStripFiller`). Найденные позиции (`selectCoreStripFillers`) **привязываются к полосам**
(`appendCoreStrip(..., fillerPositionIds)`): полоса несёт их `positionIds`, а сами позиции
попадают в `positionsCovered` — резка их **обеспечивает на произведённое** `min(заказ, полос × прогонов)`,
излишек уходит в запас, а недостаток (заказ больше выпуска) игнорируется (в рамках одного заказа
это ошибка менеджера). Число прогонов при этом по-прежнему диктует продукт-носитель (позиции 110 мм
их не увеличивают). **Не нашлось** подходящих позиций — полосы добиваются синтетически, как было в
#3812 (`positionIds: []`, обеспечения нет). Профили-носители планируются **первыми**, чтобы забранные
110-мм позиции не ушли в собственную отдельную резку (`coreFillerClaims`, `orderedProfiles`).

### Шаг 2.1. Seed «1 заказ = 1 резка»

Алгоритм начинает консервативно: один заказ и одна ширина дают отдельную seed-группу. Одинаковая
ширина внутри одного заказа консолидируется, но одинаковые ширины разных заказов не склеиваются
принудительно.

### Шаг 2.2. Базовая укладка (`composeLayout`)

Базовая раскладка работает так:

- ширины сортируются по убыванию и укладываются в джамбо хотя бы по одной полосе;
- ширина, которая не влезает в джамбо, попадает в `overflow`;
- для незапасных ширин выбирается минимальное число прогонов, которое закрывает заказ;
- для запасных ширин допустимо перепроизводство в складской запас;
- остаток джамбо добирается ходовыми ширинами через `bestFill`;
- результат содержит `strips`, `used`, `remainder`, `withinTolerance`, `overflow`.

Ходовые ширины приходят из отчёта `report/preferable_widths` и добираются только по
номенклатурам, целесообразным к хранению (`filterStockableWidths` по «Максимальному
запасу»). Отчёт медленный, поэтому его запрашивают **только когда его данные могут
пригодиться** (#3954): при генерации, построении проспекта и в панели состава он не
дёргается для семейств (сырьё + длина + намотка), которых нет в «Максимальном запасе» —
любой их добор всё равно отфильтровался бы в пустоту. Гейт — `maxStockFamilyStockable`
(ширина не учитывается; таблица не настроена → отчёт грузится как прежде).

### Шаг 2.3. Перебор выгодных слияний (`planLayouts`)

После seed-групп алгоритм пробует сливать пары раскладок. Слияние разрешено только если:

- объединенная раскладка физически влезает;
- закрывает все позиции обеих исходных раскладок;
- не превышает лимиты `maxWidthsPerCut` и `maxPositionsPerCut`;
- улучшает цель лексикографически: **прогоны ↓**, затем **скрап ↓**, затем **отход ↓**.

Невыгодные слияния не проводятся. Это защищает от «красивой» общей резки, которая на деле
увеличивает расход джамбо или перепроизводство.

### Шаг 2.4. Ограничение запаса (`capStockToHeadroom`)

После построения раскладок система проверяет «Максимальный запас». Если ширина превышает лимит
с учетом текущих остатков, лишние рулоны убираются из раскладки. Заказное покрытие при этом не
срезается.

## Фаза 3. Назначение станка и очередь

### Что происходит

Каждой готовой раскладке назначается слиттер. Затем резки каждого станка выстраиваются в
очередь, которая минимизирует переналадки и сохраняет технологические правила.

> **Слой размещения по модели вставки (#4085/#3985) — ПО УМОЛЧАНИЮ.** Основной путь назначения станка
> и порядка — единый перебор **всех точек вставки** (пары соседних заданий на всех станках) по
> минимальному штрафу (`computeSlotPlacement`, модуль `15-slot-placement.js`). Срок и фольга — ЛОКАЛЬНЫЕ
> штрафы (см. `docs/atex_planning_tz.md` §8): фольга оседает в конец дня штрафом `FOIL_NOTEND_COST_MN`
> (нефольга размещается первой, затем фольга уводится штрафом в конец — инвариант #3717 держится через
> стоимость, а не жёсткое правило); срок — штрафом `DEADLINE`/`EXACT_DEADLINE`; балансировка возникает
> из штрафа `MAX_DISTANCE`, а не отдельным шагом. `planCutOperations` берёт станок+порядок из слоя
> (`orderAuthoritative`), `rebalanceSlitterLoad` пропускается.
>
> **Первая резка очереди НАСЛЕДУЕТ заправку станка (#4288).** У точки вставки в САМОЕ НАЧАЛО очереди
> станка (`index 0`, реального предыдущего задания нет) переход `prev → slot` считается не «с нуля»,
> а от ТЕКУЩЕЙ ЗАПРАВКИ станка (`prev_cut_setup` → `prevSetupBySlitter`, синтетический `prev` через
> `carryOverPrevCut`). Так совпадающая по сырью/намотке/ножам резка получает переход **0** и
> притягивается открыть день на «своём» станке, а несовпадающая платит реальную переналадку. Раньше
> слой размещения `prevSetupBySlitter` НЕ получал → старт станка весил 0 для ЛЮБОЙ резки, и первая
> резка дня не подхватывала комбинацию, уже стоящую на станке с прошлого дня (issue #4288: 21.07 на
> станке зафиксирована `1253/OUT`, а 22.07 день открывала `MW411/IN` «с полной настройкой»).
> Упаковщик (`splitMachineQueue carryPrevSetup`, #3853), оценка идеала (`qualityIdeal`, #4029) и
> внутридневная пересортировка (`resequenceWithinDays`, #4139) заправку учитывали и раньше — теперь с
> ними согласован и ВЫБОР порядка/станка. Лидер/втулку синтетического `prev` нейтрализуем (отчёт их не
> несёт, `changeoverParts` для carry их тоже не считает). Заправка передаётся и в проходы релокации
> (`relocatePass`/§12). При отсутствии данных о заправке станка (`prevSetupBySlitter[станок]` пуст)
> поведение прежнее — старт весит 0.
>
> **`FOIL_NOTEND` — СИММЕТРИЧНЫЙ штраф (#4202).** Штраф начисляется на переход `фольга → нефольга`
> В ТОМ ЖЕ дне с ОБЕИХ сторон: и когда оцениваемый слот — фольга, а следом нефольга, и когда слот —
> нефольга сразу ПОСЛЕ фольги. Раньше он был односторонним (только на фольге), поэтому оценка
> ПЕРЕМЕЩЕНИЯ нефольги (её цена «остаться»/«встать» за фольгой) штраф не видела — перемещение считало
> НЕПОЛНЫЙ набор штрафов и могло оставить/поставить резку после фольги. Гейт «тот же день» обязателен:
> нефольга в начале СЛЕДУЮЩЕГО дня после фольги, закрывшей прошлый день, нарушением не является. На
> первичное размещение это не влияет (нефольга кладётся первой — фольги в занятости ещё нет); член
> «включается» на релокации/оценке позиций, где фольга уже стои́т.
>
> **`MAX_DISTANCE` — живой штраф спреда (#4106).** До #4106 предикат `distanceExceededFor` нигде не
> создавался (мёртвый хук) → `MAX_DISTANCE_COST_MN` не начислялся, и выбор станка сваливался в чистое
> совпадение сырья: одинаковое сырьё копилось на одном станке (совпадение → вес 0 всегда бьёт смену
> сырья +15), день не держал 450, другие станки простаивали. `makeDistanceExceeded` (`15-slot-placement.js`)
> считает штраф по ЗАНЯТОСТИ: станок-кандидат, ушедший вперёд более чем на `MAX_SLOTS_DISTANCE_HR` от
> самого рано освобождающегося другого станка, получает `+MAX_DISTANCE_COST_MN` (25 > 15) → минимальный
> штраф уводит слот на простаивающий станок. Мера отрыва = `max(день слота, свободный день станка)`:
> оценка дня предпочитает меньший день, поэтому одинаковое сырьё вставляется в НАЧАЛО дня 0 и
> литеральный «старт слота» штраф бы не поймал — свободный день станка ловит и это (перегруженный
> станок дорог для любой вставки); при дописывании в хвост обе меры совпадают.
>
> **Срок держат РЕАЛЬНЫЕ дни, а не оценка (#4095, ТЗ §12).** Внутри перебора вставки день приземления
> оценивается эвристикой `prefixDayOffset` (порт `packMachine`); `capacityMin` этой оценки — **не
> арбитр срока**, лишь первичный порядок (раньше = сырое окно без обеда → оптимистично → штраф срока
> считался против слишком раннего дня → **просрочка даже с нуля**). АРБИТР срока — **реальные дни**
> `splitMachineQueue`: `planCutOperations` пакует по-настоящему, у кого реальный день ≥ срока —
> `relocatePass` тянет раньше и пере-пакует, монотонно + cap раундов.
>
> **Честная цена срока в релокации (#4098, ТЗ §12).** Штраф срока `DEADLINE` для просрочки выкручен
> в потолок — движок сам не поставит задание за срок, если место в срок дешевле. Дыра была в том,
> что при оценке цены «ОСТАТЬСЯ» день брался из ОЦЕНКИ (`prefixDayOffset`), а не из реальной упаковки:
> задание 0.592 мин со сроком в day0 (issue #4098) оценка видела в day0 (штраф «в притык»
> `EXACT_DEADLINE`), тогда как `splitMachineQueue` переливал его на day1 — реальная просрочка, но
> потолочный `DEADLINE` не начислялся, и «остаться» стоило дёшево → мягкая релокация «строго дешевле»
> не находила выгоды. Фикс: цену «остаться» у слота, который РЕАЛЬНО (по `dayByCut` из
> `splitMachineQueue`) за своим сроком, `relocatePass` считает по РЕАЛЬНОМУ дню (`scorePosition` с
> `ctx.selfRealDayKey`) → штраф `DEADLINE` в потолок → штатный выбор самого дешёвого места сам уводит
> задание в срок (день в срок дешевле опоздания). **Единственное исключение:** не двигаем срочное
> задание НА место за его сроком (оценка `placementDayKey > dueKey`) — штраф всё равно вернёт его
> обратно, это пустой перенос. Осталась просрочка после цикла = честный конфликт ёмкости (в срок
> дешёвого места нет).
>
> **Сходимость релокации: слот двигаем ≤ раза за проход (#4104).** `dayByCut` (реальные дни
> `splitMachineQueue`) — ФИКСИРОВАННЫЙ снимок на весь `relocatePass`: после переезда слота его
> реальный день пересчитывается только СЛЕДУЮЩИМ пере-пакингом (внешний цикл §12). Цену «остаться»
> (`cur`) считаем со штрафом срока по этому реальному дню (#4098), а цену «переехать» (`alt`) — по
> оценке дня-приземления кандидата. Пока слот по снимку РЕАЛЬНО за сроком, `cur` всегда «дорого», а
> найдётся место, что «дешевле» → слот **пинг-понгует между станками до cap** (лог #4104: «переносов
> 2000» = 4 раунда × 500, для заказчика — «планировщик правит время»). Фикс: в одном проходе
> `relocatePass` слот переносится **не более раза** (`movedIds`); дальше его двигает лишь следующий
> раунд §12 — уже на СВЕЖИХ реальных днях (переехав, слот мог перестать быть просроченным → штраф
> уходит → перенос не нужен). Это НЕ теряет улучшений (внешний цикл даёт до `slotRefineRounds`
> попыток на свежих данных), но гарантирует завершение за `≤ раунды × |слоты|` переносов.
>
> **Смежность заказа: держать задания одного заказа вместе (#4194).** Заказ может быть разбит на
> НЕСКОЛЬКО заданий (по дням/ёмкости), а вставка чужого задания разрывает их. При выборе точки вставки
> `scorePosition` добавляет к весу штраф/бонус `ORDER_DIFF_PENALTY_MN` (Настройка, дефолт 10):
> «заказы» задания = множество id заказов обеспечиваемых позиций (`supply.orderNo`, у джамбо их
> несколько), кладутся в слот (`slotFromCut`, из `orderIdsByCut`, собранного в `buildSequenceOps`).
> Правило: если слот делит заказ с соседом (`prev` ИЛИ `next`) — **бонус −ORDER_DIFF_PENALTY_MN**
> (тянуть к своему заказу; вес может стать **отрицательным** — это осознанно даёт преимущество перед
> идентичной конфигурацией ЧУЖОГО заказа, чтобы одинаковые конфиги разных заказов не склеивались);
> иначе, если `prev` и `next` принадлежат общему заказу, а слот — ни одному из их заказов — **штраф
> +ORDER_DIFF_PENALTY_MN** (не разрывать смежный заказ). Штраф идёт в ВЕС, не в `setupWeight` — гейт
> §8.4-фолбэка (по setup) не трогается. Выкл при `ORDER_DIFF_PENALTY_MN=0` или отсутствии заказов у
> слота; действует и при первичной вставке, и при релокации (#4118/§12 переиспользуют `scorePosition`).
> **Доп. проход #4118 — затолкать ВСЁ ЕЩЁ просроченное (`relocateOverdueReal`).** Мягкая релокация
> `relocatePass` оценивает кандидатов ОПТИМИСТИЧНОЙ оценкой дня (`capacityMin`) и может «переносить
> вхолостую», оставив задание просроченным. После §12-цикла отдельный проход берёт всё, что по
> РЕАЛЬНЫМ дням ещё за сроком, и заталкивает в наименее штрафное место (можно на другой станок),
> проверяя КАЖДОГО кандидата РЕАЛЬНОЙ упаковкой (`realPackFn`). Гарантии: двигаем ТОЛЬКО само
> просроченное задание; принимаем перенос лишь если его реальный день СТРОГО меньше (сходимость) и
> НИ ОДНО чужое задание от вставки не станет БОЛЬШЕ просрочено. Осталась просрочка = честный дефицит
> ёмкости.
>
> **Просрочка недопустима даже у ЗАФИКСИРОВАННОГО задания (#4224).** Раньше `relocateOverdueReal`
> ПРОПУСКАЛ фиксированные (🔒) задания (`if (s.fixed) return`) при сборе просроченных — и просроченная
> фиксированная резка/фольга оставалась за сроком, хотя более ранний день имел свободное место (пример
> ateh: фольга срок 06.07 приколота на 07.07, а 06.07 занят лишь на 135 из 450 мин). Теперь проход
> собирает просроченные БЕЗ учёта фикса: фикс, приведший к просрочке, невалиден — пользователь
> приколол задание, ожидая его В СРОК. Оговорки, чтобы не сломать смысл фикса: (а) не просроченное
> фиксированное задание проход НЕ трогает (`overdueDays>0` отсеет); (б) фиксированное просроченное
> рескьюим ТОЛЬКО в пределах СВОЕГО станка (`slot.fixed && tid!==sid` → пропуск чужих станков) —
> станок выбрал пользователь, чиним лишь день; (в) вставка не вправе увести ЧУЖОЙ фикс приёмника на
> более поздний день (`fixedOnTid` в harm-гейте). Сходимость та же (строгое уменьшение реального дня +
> `maxRounds`), зацикливания нет.
>
> **РЕАЛЬНЫЙ день — КАЛЕНДАРНЫЙ, а не логический (#4200).** `splitMachineQueue` пакует по ЛОГИЧЕСКИМ
> дням (`dayOffset` 0,1,2…), а нерабочие дни (выходные/праздники/отпуск) сдвигают ОКНО
> (`windowStartMin`), НЕ трогая `dayOffset`. Панель «просрочено» (`countOverdueCuts` по `planStart`)
> и `dueDayByCut` (`dueDayOffsetFromBase`) считают по КАЛЕНДАРЮ, поэтому арбитр §12/#4118 берёт
> КАЛЕНДАРНЫЙ день приземления `floor(windowStartMin/1440)` (= день `planStart`,
> `scheduleStartTimestamp`), а не `dayOffset`. Иначе задание за выходными числилось «в срок ✓» у
> арбитра, а панель — просроченным (расхождение «§12: просрочек нет» vs «панель: N»).
>
> **РЕАЛЬНЫЙ день — день ЗАВЕРШЕНИЯ намотки (#4290).** У РАЗБИТОГО по дням задания намотка идёт на
> несколько дней; срок выполняется, лишь когда готова ПОСЛЕДНЯЯ часть. `windingDaysFromSegs`
> (`realDaysFrom`/`realPackFn`) берёт МАКС (последний) день среди сегментов С ПРОХОДАМИ — день

> просроченным (issue #4290: «большое задание просрочено — та часть, что не влезла; надо сдвигать
> раньше, пока ВСЕ части не влезут в срок»). Расходилось с панелью, которая считает по `planStart`

> по-прежнему НЕ держит — исключён из намоточных сегментов.

### Шаг 3.1. Выбор станка (`chooseSlitterBySetup`)

Сначала отбрасываются станки, где сырье запрещено стоп-листом **или где номинальная ширина сырья
превышает лимит станка** («Код» вида `j<1000`, #4006 — `isSlitterWidthBlocked` по
`nominalWidthByMaterial`). Среди допустимых станков выбор идет по таким критериям:

1. **Емкость дня.** Если резка переполнит рабочий день одного станка, а на другом влезет,
   выбирается станок, где она влезает (#3830).
2. **Совместимость заправки.** Предпочитается станок с тем же набором ножей или тем же сырьем
   и намоткой (#3666/#3801).
3. **Минимальный прирост переналадки.** Сравнивается стоимость очереди до и после добавления
   резки.
4. **Баланс загрузки.** Если совместимых станков несколько, выбирается менее загруженный.
5. **Детерминированный тай-брейк.** При полном равенстве берется меньший id станка.

Это **жадное** назначение (каждая раскладка — независимо). Оно группирует одно сырьё/ножи на
один станок и может скопить работу на нём на несколько дней, пока соседний станок простаивает.
Поэтому после жадного шага идёт выравнивание (шаг 3.1-bis).

### Шаг 3.1-bis. Выравнивание загрузки станков (`rebalanceSlitterLoad`, #3848)

После жадного назначения и до построения очереди план выравнивается **в памяти, по массиву
раскладок** (никаких запросов в базу — см. фактор #5 весовой матрицы, вес 20). Итеративный
локальный поиск: на каждом шаге берётся **перегруженный** станок (заканчивает на 2-й день и
позже — `spanDays ≥ 2`) и одно его задание переносится на станок, где общий счёт лучше. Счёт
состояния — лексикографически `[макс. дата окончания (span целых дней), разнородность сырья на
станках, макс. ДРОБНАЯ дата окончания, пик минут, сумма квадратов минут]`; сумма **квадратов** (а не
просто сумма) штрафует перекос и выталкивает работу на простаивающий станок (простая сумма росла бы
от лишних настроек и склеивала всё на меньшем числе станков). Член **«разнородность сырья»** (#4077)
описан ниже. Существующие резки держат базовую загрузку своих станков
и не двигаются; запрещённое стоп-листом сырьё на станок не переносится; сырьё шире лимита станка
(«Код», #4006) на такой станок не переносится; на станок в отпуске в день
задания не переносится (#3876).

**Загруженность = ДАТА ОКОНЧАНИЯ последней задачи (#3881).** Не сумма минут, а **дата
окончания** — число календарных дней от базы плана до конца работы. Задания станка **реально
укладываются по дням** (`packMachine`, #3965): резки в порядке `orderCuts`, каждая занимает
`настройка + намотку`, дни ограничены ёмкостью, **нерабочие дни станка пропускаются**
(`opts.machineDayOff`) и считаются ЗАНЯТЫМИ. Поэтому станок с отпуском заканчивает **позже** при
тех же резках — его не грузят на послеотпускное время, пока другие простаивают. Станок в отпуске
**без задач после него** «занят» до конца отпуска: «дату окончания отпуска считать последней датой
задач, если после него задач нет». Без инфо о нерабочих днях (`machineDayOff` не задан) — та же
укладка без пропусков (обратная совместимость, #3848).

**Реальная укладка, а не заниженная оценка (#3965), и настройка КАК В РАСПИСАНИИ (#3968).** Дата
окончания и минуты берутся из **фактической укладки по дням**, а не из формулы `poolMinutes`.
Настройка каждой резки считается **как в `buildSchedule`** — переналадка от **ПРЕДЫДУЩЕЙ** резки
очереди (`changeoverCost(seq[i-1], seq[i])`), а не «с нуля» у каждой. Это честно в обе стороны:

- **Разные конфиги** (разное сырьё/намотка) → каждый переход = полная настройка
  (ножи + сырьё) → станок настроечно-**тяжёлый**. Так модель видит реальный перегруз
  настроечно-разнородного станка, и его хвост за отпуском стекает на свободные (#3965/#3957:
  «опять на первый станок накидали после отпуска» — здесь загрузка не занижается).
- **Одинаковые конфиги** (партия одного сырья) → соседние резки ставят ножи/сырьё **один раз**
  (переход = 0), как в реальной по-дневной укладке (`splitMachineQueue`/`selectByConfig`
  группирует по непрерывности конфигурации). Станок настроечно-**лёгкий** — модель это видит и
  не завышает его загрузку.

Промежуточная версия #3965 считала настройку у **каждой** резки «с нуля» (ножи + сырьё). Это
завышало настроечно-**сгруппированный** станок почти вдвое (репро на данных #3968: Станок 1 —
оценка **1479** при реальных **834 мин**; вся его работа — просроченная партия одного сырья).
Балансировщик считал такой станок загруженным и **не докидывал** на него работу → станок оставался
недогруженным (в #3968 — всего 2 дня работы, потом пусто), а соседние копили перегруз и переливали
за ёмкость. `changeoverCost` даёт 0 для одинаковых конфигов и полную настройку для разных, поэтому
оценка совпадает с реальным расписанием в обоих случаях. (`poolMinutes` — прежняя формула #3848 —
группировала конфиги ГЛОБАЛЬНО, без укладки по дням, поэтому занижала настроечно-тяжёлый станок;
`packMachine` группирует в порядке очереди с реальной по-дневной раскладкой.)

**Оценка минут — по конфигурации без срока; разнородность сырья — отдельным членом счёта
(#3974/#4077).** `packMachine → orderCuts` группирует одинаковые конфиги по стратегии (сырьё/ножи) и
даёт **минуты настройки на станок** без разбиения по срокам. Этого довольно для оценки ЗАГРУЗКИ
(дни/минуты, факторы `maxDays`/пик/квадраты). Но реальная по-дневная укладка (`selectByConfig`)
упорядочивает резки по **сроку** (EDD, #4059) и потому **чередует разные сырья с соседними сроками** —
на станке со многими сырьями фактических смен сырья получается больше, чем в сгруппированной оценке
минут (issue #4077: Станок 1 — **22 факт. смены сырья на 10 разных сырьёв**, +12 избытка). Поэтому в
счёт добавлен член **`matHetero` = Σ по станкам `max(0, число разных сигнатур сырья − 1)`** — «лишние»
сырья сверх первого на каждом станке (`materialSig` = сырьё + намотка, тот же ключ, что у §13-идеала):

- **Ранг — сразу под `maxDays`** (целые дни = ДЕНЬ размещения; срок/финиш святы, #4059: днём
  окончания ради группировки не жертвуем), но **над** дробной датой окончания / пиком / квадратами.
  Дробная дата (#3921) — уточнение внутри того же дня, не срок-критичное (задание со сроком в дне D
  ложится в день D хоть при финише 4.2, хоть 4.8), поэтому уступает однородности.
- **Штрафуется разнородность ВНУТРИ станка, не «сырьё на нескольких станках».** Одно сырьё,
  размазанное по нескольким станкам, даёт 0 (каждый станок однороден) → выравнивание одинакового
  сырья по простаивающим станкам (#3848) и стекание дробного хвоста за отпуском на свободный станок
  (#3921/#3957) не страдают. Член растёт только когда на **уже занятый** станок кладут **ДРУГОЕ** сырьё.
- **Итог:** «косметический» перенос ради ровности, разбивающий сырьевой блок между станками без
  сокращения ЦЕЛОГО дня окончания, теперь отвергается; перенос, реально сокращающий день окончания
  (перегруз / уход за срок), — берётся (инвариант #3848 цел).

**Нерабочий день = выходной/праздник ИЛИ отпуск станка (#3957).** `machineDayOff` контроллера
(`balanceDayOff`) считает день нерабочим, если это **выходной/праздник** (`dayIsWorking`, #3788 —
общий для всех станков) **ИЛИ отпуск** данного станка (`slitterOnVacationDay`, #3876). Оба
обязательны: модель span/endPos должна пропускать **те же** дни, что и реальное расписание
(`calendarBlockedRanges` + `downtimeBlockedRanges`). Если учитывать только отпуск, содержимое
станка, влезающее в рабочие дни **до выходных перед отпуском**, «не доходит» до отпуска — цикл
обрывается на первом свободном дне, станок с отпуском выглядит заканчивающим рано (в логе
`⚖ выравнивание` показывался как «4д» вместо «12д»), перегруз за отпуском не виден, и хвост за
отпуском **не стекает** на свободные станки (баг #3957: «опять на первый станок накидали после
отпуска»). Выходные симметричны для всех станков → относительный баланс не искажают; разницу даёт
именно отпуск. *Ограничение:* содержимое моделируется от дня базы плана; «дыра» (свободные дни)
перед будущим отпуском без работы после него span не увеличивает.

**Дробная дата окончания против «плато» ceil (#3921).** Целый `span` (ceil дней) создаёт плато:
при отпуске станок может держать хвост работы **после отпуска** (напр. на 13.07), пока другие
простаивают с 09.07, а перенос одного задания не меняет целый `span` (1680→1500 мин = те же
4 «дня») — счёт по `span` не улучшается, работа застревает за отпуском. Поэтому вторым ключом счёта
идёт `endPos` — **дробная** дата окончания (та же раскладка по дням с пропуском отпуска, но
последний рабочий день заполнен частично, по минутам). По минутам каждый перенос строго снижает
хвост за отпуском → он **стекает** на свободные станки, пока станок в отпуске не перестанет работать
после отпуска (его `endPos` упирается в конец отпуска). Пустой станок в отпуске «занят» до конца
отпуска — как у `span`. Так реализовано требование «смотреть на возможную дату начала задачи (дату
освобождения от задач/отпуска) — кто освобождается раньше, тому и даём».

Процесс **итерационный**, **прекращается при отсутствии прогресса** (нет хода, строго улучшающего
счёт), и защищён от **цикличных перестановок**: множество посещённых комбинаций (`stateHash` по
`plan.slitterId`) — идентичное назначение не повторяется; плюс ходы только строго улучшающие счёт
⇒ счёт монотонно падает. Каждый шаг журналируется в консоль («панель отладки»): старт-загрузка,
каждый перенос `задание Cn: Cx→Cy`, причина остановки. Активно только при заданной ёмкости дня
(генерация); без неё (тесты/обратная совместимость) переносов нет.

**Производительность (#3871).** Минуты станка = намотка + переналадка очереди + настройка первой
резки. Переналадка очереди требует упорядочивания набора, и на каждую пробу переноса считается по
полному набору станка. Полный перебор стартов жадной цепочки (`greedySequence`) — `O(n³)`, и при
сотне с лишним резок «Создать» висел минутами (≈40 с на перенос). На время выравнивания включаются
два упрощения: стоимость перехода кэшируется по паре `id` (один переход не пересчитывается тысячи
раз), а цепочка строится **от одного старта** (`O(n²)` вместо `O(n³)`). Оценка переналадки здесь
нужна лишь как ориентир баланса — финальную очередь всё равно строит `planCutOperations`, поэтому
точность одностартовой цепочки достаточна. Вне выравнивания планировщик считает переналадку как
прежде (полный перебор стартов).

### Шаг 3.2. Очередность резок (`orderCuts`)

`orderCuts` не мутирует входные данные. На выходе каждая резка получает `sequence = 1..N`.

> **`sequence` — внутренний ординал генерации, а НЕ хранимое поле (#3923).** Он живёт только в
> памяти в пределах одного прохода генерации/пересборки: `orderCuts` (и `planQueues`) выстраивают
> порядок, `splitMachineQueue` по нему присваивает `planStart`, `computeCutSetupUpdates` по нему
> считает переналадку. В базу «Очередность» **не пишется** и при загрузке **не читается** —
> единственный сохранённый носитель порядка резки — её `planStart` (главное значение, `t1078`).
> Порядок исполнения на всех РМ (страница планирования, «Диаграмма Ганта», пульт слиттера)
> определяется **строго по возрастанию `planStart`**. См. инвариант в конце Фазы 4.

> **`orderCuts` — не тот, кто строит план (#4085).** При генерации порядок очереди станка задаёт
> слой размещения: `splitMachineQueue` читает `slotPlan.orderIdxByCut`, а `orderCuts` там — запасной
> путь на случай `SLOT_PLACEMENT=0`. Остальные вызовы `orderCuts` — оценка стоимости
> (`orderedChangeoverCost`, `packMachine`) и `planQueues`/`planStartTimestamps`. Правка порядка в
> `orderCuts`/`greedySequence` на сгенерированный план не влияет (issue #4139).

Текущая стратегия `SETUP`:

1. **Минимум переналадки.** Строится жадная цепочка по `sequencingCost` (блоки одного сырья/ножей —
   подряд). Срок изготовления в порядке НЕ участвует (#3974); фольгу в конец дня уводит штраф
   `FOIL_NOTEND_COST_MN` слоя размещения, а не сортировка (#4085).
2. **Число полос по убыванию.** Не тай-брейк, а направленный штраф: доставить ножи
   (`KNIVES_INCREASE`) дороже, чем снять (`KNIVES_CHANGE`), #3996/#3130.

Отдельная стратегия `FATIGUE` может ставить более сложные резки раньше смены, если важнее
усталость оператора, а не минимум переналадки.

### Стоимость переналадки между соседними резками

Переналадка состоит из независимых операций. Каждая либо применяется, либо нет.

| Операция | Код | Мин. | Когда применяется |
|---|---|---:|---|
| Смена ножей / сужение ролика | `KNIFE` | 30 | Изменился набор ножей или следующий ролик уже предыдущего. Цена фиксированная, не зависит от числа ножей (#3600). |
| Смена сырья / намотки / партии | `MATERIAL_WINDING` | 15 | Сменилось сырье, направление намотки или партия. |
| Смена ножей + смена сырья | оба | 45 | Обе операции произошли одновременно: 30 + 15. |
| Лидер между резками | `BETWEEN_CUTS` | `2 × N` | Заправляется после намотки текущей резки. `N` = `Кол-во план`. |
| Уборка в конце смены | `CLEANUP_SHIFT` | 30 | Резервируется в конце рабочего дня, если в этот день есть резки. |

Первая резка очереди сравнивается с текущей заправкой станка из `prev_cut_setup`. Если
конфигурация совпадает, переналадка не нужна. **Если станок ПУСТ (`prev_cut_setup = []`), первая
резка ставит с нуля И НОЖИ, И СЫРЬЁ** (`firstSetupParts` = `KNIFE` + `MATERIAL_WINDING`, #4296):
сырьё реально заправляется на станок — это наладка, а не «менять не с чего». Раньше первой резке
пустого станка засчитывались ТОЛЬКО ножи (#3669/#4156), и панель показывала «смены сырья: 0», а
избыток был отрицательным («план лучше идеала»), хотя идеал §13 эту заправку сырья учитывал.
Заправку/наладку с нуля учитывают все звенья, влияющие на первую резку: упаковщик/тайминг
(`carryPrevSetup`, #3853), хранимые колонки наладки (`computeCutSetupUpdates` → `setupActivityColumns`),
оценка идеала и факта (`planQuality`, #4029) и — с #4288 — сам **слой размещения** (`scorePosition`:
старт станка считается переходом от заправки, а не «с нуля», см. «Первая резка очереди НАСЛЕДУЕТ
заправку станка» выше). До #4288 слой размещения заправку игнорировал, поэтому мог открыть день
несовпадающей резкой и «не подтянуть» комбинацию прошлого дня.

«Текущая заправка» = ПОСЛЕДНЯЯ задача станка СТРОГО ДО начала окна планирования. Запрашивается
по каждому станку отдельно из отчёта `prev_cut_setup` (report 93371) с фильтрами
`FR_slitter_id={станок}` и `FR_task_start=<{планБаза}` (issue #3737), где `планБаза` — полночь
дня «С» из фильтра. Фильтр `task_start<планБаза` обязателен: после генерации в отчёте уже лежат
задачи самого нового плана (их `task_start` в будущем), и без фильтра «верхняя по `task_start`»
оказалась бы будущей задачей плана → ложная смена сырья у первой резки (окно 51 мин вместо 36 —
расхождение карточки и модалки тайминга, #3862). Заправка обновляется и при загрузке страницы, и
перед каждой генерацией (дату окна могли сменить).

### «Сгенерировать» и «Упорядочить»: две кнопки, один конвейер (#4001)

Обе кнопки прогоняют **ту же** Фазу 3 (выбор станка + баланс) и Фазу 4 (порядок + дни). Различие —
создавать ли резки:

- **«Сгенерировать»** строит резки из потребности (Фазы 1–2), назначает станок
  (`chooseSlitterBySetup`), балансирует загрузку (`rebalanceSlitterLoad`), упорядочивает и
  раскладывает по дням (`orderCuts` → `planCutOperations`). Создаёт/обновляет/удаляет записи резок.
- **«Упорядочить»** (`runOptimizeQueue`) берёт **уже существующие** логические резки (статус ≠
  «Завершён») и переставляет их так, чтобы **сперва минимизировать опоздания** (задания за своим
  сроком), а при равных опозданиях — суммарную переналадку. Считает **в памяти** (без записи в БД)
  два плана-кандидата и меряет их **объектив** по всему открытому горизонту —
  `дни_опоздания × LATE_DAY_WEIGHT + переналадка(мин)` (опоздание — `planLatenessDays`, срок ТЗ §14
  старший критерий; переналадка — `planChangeoverMin` = `planQuality.all.changeoverMin`, порядок
  внутри дня по реальному `planStart`) против текущего плана:
  - **B** — пересборка порядка и по-дневной раскладки на **текущих** станках (`buildSequenceOps`
    → `planCutOperations`; раскладка EDD — срочное на ранние дни, #4059);
  - **A** — переназначение станков (`computeReassignmentPlan` → `computeSlitterReassignment` =
    `chooseSlitterBySetup` + `rebalanceSlitterLoad`, тот же жадный выбор + баланс, что при генерации),
    затем та же пересборка.

  Применяется **лучший** кандидат и **только если он строго лучше** текущего по объективу
  (`chooseOptimizeCandidate`, лексикографически: сперва меньше опозданий, затем меньше переналадки);
  при равенстве кандидатов берётся B (без смены станка); если ни один не строго лучше — план **не
  трогается** («уже оптимальна»). Так «Упорядочить» **не оставляет задание за сроком, когда его можно
  разместить раньше** (#4064), а при равных опозданиях не увеличивает переналадку (#4047); ради
  сокращения опозданий переналадку увеличить может (срок важнее). Резки **не пересоздаются** —
  меняются лишь назначение станка, порядок и `planStart`.

`🔒` «Зафиксировано» держат свой станок (входят в баланс базовой загрузкой `fixedByMachine`, не
переносятся) и свой день (#3792). «Слиттер» (`_m_set`) пишется **только** цепочкам, у которых
станок изменился, — **всем** записям цепочки (голова + продолжения), иначе рвётся
`continuationSignature` (голова и продолжение расходятся по станку).

> **Пишем только изменившиеся значения (#4001).** `applySplitPlan` сравнивает новое значение с
> хранимым и шлёт запрос лишь при расхождении: `planStart` (`_m_save` главного значения) — если
> сменился; «Кол-во резок план» и тайминг сегмента («Длительность»/«Резка и Лидер») — если сменились
> проходы; «Метраж, м»/«Вид сырья»/«ID первой части» — лечатся, только если хранимое пусто или иное
> (#3781/#3795/#3892). Снимок «Вид сырья» берётся **до** `healContinuationMaterials` (лечение в
> памяти иначе затрёт пустоту и запись в БД не уйдёт). Полностью неизменившаяся резка не порождает
> ни одного запроса.
>
> **«Партия сырья» продолжения — от головы цепочки, источник `object/` (#4155).** Отчёт
> `cut_planning` не отдаёт «Партию сырья» (ссылка req 192): его `batch_id` — это «Партия ГП», не
> сырьё, поэтому `rowsToPlanning` кладёт `batchId:''`. Из-за пустоты `applySplitPlan` создавал
> продолжение дробления по дням с **пустой** «Партией сырья» (`materialBatch = parentCut.batchId =
> ''`). Собственную «Партию сырья» резки читаем прямо с записи — `loadCutSequences` берёт её через
> `object/` (`parseRef` ref-колонки → id), `loadPlanning` кладёт в `cut.batchId`. Продолжение берёт
> партию **головы** цепочки (`batchForCutId`, не прямого родителя): продолжение, созданное до фикса,
> хранит пустоту, и она расползлась бы по новым сегментам. Одно сырьё на все сегменты дробления.
>
> **Отбор изменившихся апдейтов держит и смену станка (#4108).** Перед `applySplitPlan`
> `autoSequenceQueue` прогоняет `ops.updates` через `filterChangedUpdates` (перф: не гонять
> заведомо пустые апдейты). Критерий «изменилось» — `planStart` **или** проходы **или**
> **«Слиттер»** (`u.slitterId` в слот-режиме #4085, сравнение как в `applySplitPlan`). Ветку станка
> добавили в #4108: слой размещения переставляет резку на другой станок в **ту же позицию дня**
> (напр. первое задание дня в 08:00 на обоих станках) — `planStart` и проходы совпадают, и апдейт с
> одной лишь сменой «Слиттера» раньше отсеивался. Станок в БД оставался прежним, а очередь второго
> станка пересобиралась **без** этой резки → два задания в одно время на одном станке (дубль-08:00,
> Гант рисует сохранённый `planStart` как есть, #4102).

## Фаза 4. Расписание по дням

### Что происходит

Упорядоченная очередь каждого станка раскладывается по рабочим сменам. Каждая резка получает
плановую дату, время старта, длительность, число проходов и при необходимости продолжения на
следующие дни.

### Окно станка для одной резки

Каждая резка занимает окно из последовательных частей:

```text
[смена ножей] → [смена сырья/намотки] → [намотка/резка] → [лидер]
```

Переналадка идет **до** намотки. Лидер идет **после** намотки, но занимает станок как часть
окна текущей резки.

### Логика расписания (`buildSchedule`)

- Резки пакуются в рабочее окно смены в порядке очереди.
- Обед вставляется как пауза. Если длинная резка пересекает обед, намотка прерывается на обед
  и финиш сдвигается (#3816).
- В конце дня резервируется уборка `CLEANUP_SHIFT`.
- Если резка не влезает в смену, она переносится на следующий рабочий день.
- Фольга сохраняется в конце каждого дня при многодневном перепланировании.
- Первая резка дня считает переналадку относительно последней физической резки предыдущего дня.

### Пример тайминга окна

Первая резка дня — фольга горячего тиснения, намотка `IN`. На станке вчера была другая
конфигурация. Метраж прохода 122 м, 20 проходов, норма `WIND_FOIL_305 = 4 мин`,
лидер `BETWEEN_CUTS = 2 мин` после каждого прохода.

> Фольга считается блоками (#3742): `ceil(метраж / 305) × 4`. Проход 122 м стоит полную
> норму 4 минуты, а не пропорциональные 1.6 минуты.

Модалка тайминга (#4006) складывает намотку и лидер в **одно** время прохода — «Намотка и лидер»
`= 4 + 2 = 6 мин` — и одну строку «Итого резка», уже включающую лидер (отдельной строки «лидер
между резками» больше нет; она путала оператора — сумма прохода читалась не с первого взгляда):

| Время | Операция | Длительность |
|---|---|---:|
| 08:00 | смена ножей / сужение ролика | 30 мин |
| 08:30 | смена сырья / намотки / партии | 15 мин |
| 08:45 | Итого резка (намотка и лидер): `6 × 20` | 120 мин |
| 10:45 | готово | — |

### Максимальный нахлёст за конец смены (#3847, ключи _MN #3992)

> **#3992 / #4017 / #4059.** В «Настройке» ключи захлёста несут суффикс `_MN` (минуты):
> `MAX_OVERWORK_CUTS_MN` (5) и `MAX_OVERWORK_TUNE_MN` (10). `loadDaySettings` читает «Настройку»
> **целиком** (#4059 — фильтра по имени ключа нет; тип строки `r[1]` задаёт приоритет `<db>` >
> `ATEH` > общий) и заносит все пары ключ→значение в `daySettings`. Оттуда `resolveWorkingWindow`
> читает лимит через `resolveOverworkLimits` — сначала имя с `_MN`, затем старое имя без суффикса
> (обратная совместимость). Пустое значение `_MN` не перекрывает заполненный старый ключ.

Задание режется по **проходам** (`splitMachineQueue`), и его можно разрывать по дням на границе
прохода. Сколько проходов влезает в хвост дня, ограничено двумя потолками от **`DAY_END_HOUR`**
(реального конца смены, а не `cutEndMin = DAY_END_HOUR − TOTAL_INTERVALS`):

- **`MAX_OVERWORK_CUTS`** — проход (резку) можно положить с нахлёстом, только если он кончится
  **не позже `DAY_END_HOUR + MAX_OVERWORK_CUTS`**. Длиннее — проход уходит на следующий день
  (отдельным сегментом-продолжением).
- **`MAX_OVERWORK_TUNE`** — настройку (ножи / смену сырья) можно положить с нахлёстом, только если
  она кончится **не позже `DAY_END_HOUR + MAX_OVERWORK_TUNE`**. Не влезает — настройка и проходы
  переносятся на следующий день.

Оба значения — из «Настройки» (мин). Это уточняет #3760/#3821: вместо «один любой нахлёстный
проход» (#3760) или «строго встык до `cutEndMin`» (#3821) — **нахлёст разрешён, но ограничен
минутами** за концом смены. Лимит не задан (пусто) → фича выключена, планировщик пакует до
`cutEndMin` без сверхнормативного нахлёста (поведение #3821). Реализация — чистая `splitMachineQueue`
(обе ветки: базовая для генерации и `gapFill` для автозаполнения/пересборки); потолок считает
хелпер `availFor(day, 'cuts'|'tune')`, лимиты приходят из `resolveWorkingWindow`
(`maxOverworkCutsMin` / `maxOverworkTuneMin`, `dayEndHourMin = endMin`).

**Настройка в хвост — максимум под потолком нахлёста настройки (#3955/#3847/#4144).** Когда в хвост
дня не влезает ни один проход, но есть переналадка, в хвост кладётся **наибольшее подмножество** её
компонентов (ножи 30 / смена сырья 15), укладывающееся под потолок нахлёста настройки
`cutEndMin + MAX_OVERWORK_TUNE` (`availFor(day, 'tune')`). Остаток настройки (`pendingSetup`) и
проходы переносятся на следующий день. Если под потолок не проходит даже минимальный компонент —
в хвост **ничего не кладём**, вся резка уходит на следующий день **одной карточкой**. Заказчик:
«ставим то, что оператор успеет сделать, с допустимым нахлёстом» (#3955). Примеры при наладке
ножи 30 + сырьё 15: потолок 30 → **ножи 30** («сделать что-то одно — настройку ножей», #3858);
потолок 50 → ножи+сырьё 45; потолок 29 → **сырьё 15** (ножи не влезают, #4144); потолок 10 → ничего.
Когда настройка влезает целиком до `cutEndMin` — она уходит в хвост дня N без нахлёста, проходы
начинаются с дня N+1 без повторной настройки (#3635 п.5).

Правило одно на все места — функция `chooseTailSetupSubset`: обе ветки `splitMachineQueue`
(базовая для генерации заданий и `gapFill` для `planCutOperations`/«Упорядочить») и раскладка
хранимых колонок `splitTailSetupAtCeiling`. Нахлёст **проходов** (#3847) не тронут.

**Колонки хвоста пишет упаковщик, а не пересчёт (#4144).** Разложение хвоста по колонкам
«Наладка ножей» / «Сырьё-намотка» рождается вместе с сегментом (`setupKnifeMin`/`setupMaterialMin`)
и доезжает до `computeCutSetupUpdates` картой `plannedTailSetup` с ключом «станок + плановый старт».
Пересчитывать хвост по хранимому `planStart` нельзя: тот прошёл снап к целым минутам
(`snapWindowStartsWholeMinutes`, #4061 — занятость дня копится как `round(наладка) + ceil(намотка)`)
и лежит **позже** упаковочного на накопленный `ceil`. На плане из #4144 хвост 16:04 (остаток окна 6,
влезала смена намотки 15) превращался в 16:07 (остаток 3) → под потолок не проходил ни один компонент,
в колонки писались нули: задание нулевой длительности, а 15 минут всплывали на продолжении, наезжали
на соседнюю карточку и раздували следующий день до 469 при допуске 460. Плана под рукой нет
(«Зафиксировать» по хранимым данным) — уже записанные колонки хвоста не трогаем; нет и их — фолбэк на
`splitTailSetupAtCeiling` от хранимого старта.

> Инвариант: ни один проход не кончается позже `DAY_END_HOUR + MAX_OVERWORK_CUTS`, ни одна
> настройка — позже `DAY_END_HOUR + MAX_OVERWORK_TUNE`. Нахлёст ограничен, переполнения дня
> (баг #3821 «483 при 480») нет — потолок абсолютный.

**Потолок держится на ЦЕЛОЙ (хранимой) занятости, а не на дробной (#4149).** Упаковщик решает, что в
какой день, ДРОБНОЙ намоткой, но в базу/на Гант уходит ЦЕЛАЯ раскладка: старт каждой карточки =
старт предыдущей + `round(наладка) + ceil(«Резка и Лидер»)` (`snapWindowStartsWholeMinutes`, #4061).
Накопленный по резкам дня `ceil` (доли минуты, округлённые вверх у каждой карточки) выносил последнюю
карточку/наладочный хвост за потолок нахлёста: день с дробной суммой ≤ 460 хранился и рисовался как
461–462, последняя карточка кончалась в 16:21–16:22 (жалоба #4149: «Станок 1 Пт 03.07 — 461 мин, при
максимуме нахлёста 10»). Поэтому гейт потолка `availFor(day, kind)` считает занятость дня **в целых
минутах** — `dayWholeOccupied(day)` суммирует `round(наладка) + ceil(намотка)` по уже уложенным
сегментам дня (+ обед) — ровно то, что уйдёт в колонки и сложит бейдж (#4131). Упаковщик роняет лишний
`ceil` на следующий день сам: проходы атомарны, поэтому за потолок уносится ЦЕЛЫМ проходом (день не
добирает до 460 на размер последнего вытесненного прохода — это норма, потолок абсолютный важнее
плотности). Дробный `clock` остаётся источником позиций окон (их всё равно переснапывает #4061) и
времени обеда. Инвариант усилен: сумма ХРАНИМЫХ (целых) минут любого дня ≤ ёмкость + нахлёст, ни одна
карточка не кончается позже `cutEnd + MAX_OVERWORK_TUNE`. Это снимает и прежнюю оговорку «остаток на 2
против 460 — ожидаемое округление»: бейдж больше не превышает допуск.

### Многодневное планирование (`planCutOperations`)

Если резка не помещается в один день, она дробится на сегменты:

- первый сегмент обновляет исходную резку;
- остаток создается как продолжение на следующий день;
- продолжения связаны через `continuationSignature`;
- повторный запуск при неизменной раскладке не создает новые записи и переиспользует
  существующие продолжения;
- `setup-only` сегмент может положить часть настройки в хвост дня, а остаток настройки —
  на следующий день (#3805/#3827).

`planCutOperations` планирует **всё необеспеченное** от «С» (см. «Модель размещения», #3974).
Фильтра входа по `[С; По]` нет: каждое «Создать» перепаковывает всю непокрытую потребность от
«С», поэтому спец-обработки «спил-дня вне окна» (#3918) больше не нужно — окна-фильтра нет, все
дни раскладки наши. Единственное, что держит свой день, — `🔒` «Зафиксировано»: свободные резки
набиваются вокруг фикс-резок, `🔒`-резка раньше «С» остаётся как есть (не двигаем/не удаляем).

**Осиротевшие сегменты настройки (#3924).** `setup-only` запись (0 проходов) — всегда **хвост**
цепочки: настройка в конце дня N, намотка-продолжение с дня N+1. Если её голову-резку удалили или
перенесли прежней пересборкой, а «ID первой части» у сегмента опустел, `mergeContinuationChains`
не подшивает его к цепочке и делает **отдельной логической резкой** с `plannedRuns = 0`. Планировщик
такую резку не раскладывает (проходов нет), поэтому штатный delete-путь (`usedByHead`) её не трогает,
и она оседает отдельной **«настройкой»** в дне; её минуты раздувают бейдж дня за ёмкость (510/495 при
~460). `scheduleFromStored` (#3846) показывает сохранённый план как есть — раньше эти сироты прятал
пересчёт на лету. Поэтому `planCutOperations` удаляет **сироты** — логические резки, у которых сумма
проходов цепочки = 0 (#3924/#3943). Логическая резка с 0 проходов — чистый мусор: ни прохода, ни
обеспечений, ни ручной раскладки (её защищает «Зафиксировано», #3508). Чистим такие всегда — вход
планировщика и так вся непокрытая потребность (#3974), отдельного «окна пересборки» нет. Реальные
резки (проходов > 0) и настоящие setup-хвосты (член цепочки, у чьей головы проходов > 0 → сумма
цепочки > 0) под условие не попадают.

**Заказное задание-сирота дробления — «пустой срок / нет связей» (#4163→#4175).** Задание **выпускает
заказ** — его «Партия ГП» несёт «ID заказа», а проходы дают ровно спрос позиции, — но после дробления по
дням у него **нет ни одного «Обеспечения»**. Отчёт `cut_planning` join'ит `Резка → Обеспечение →
Позиция`, поэтому у такого задания ПУСТЫ `order_id` / `order_no` / `cut_winding` / срок: они приходят из
позиции **через** «Обеспечение», а не из «Партии ГП». По этим полям сироту **не отличить** от склада —
целиться в намотку или отчётный `order_id` (как делали ранние итерации #4163/#4171) значит целиться в
симптом. Признак заказной сироты — **реальный «ID заказа» самой «Партии ГП»** (читается `object/{ПартияГП}
?F_U=резка`): непустой → задание выпускает заказ и обязано быть привязано.

**КОРЕНЬ — реюз резок при дроблении рвёт связь с заказом.** `autoSequenceQueue → applySplitPlan` при
пере-разбиении дней **реюзит существующие резки цепочки** как новые сегменты (`reuseId = chain[k]` в
`planCutOperations`). Реюз идёт `update`-путём, а `update`-путь **не создаёт «Обеспечений»** (их создаёт
только create-путь продолжений, §2b). Свежесгенерированные покрывающие сегменты (со своими «Обеспечениями»)
при этом уходят в `deletes`. Итог: **проходы уезжают на реюзнутую резку, «Обеспечение» — на удалённую**
(журнал ateh, заказ 4059: свежие обеспечения 737+220 обнулены/удалены, проходы 43+44 сели на реюзнутые
`497698/497708` без единого «Обеспечения»). `headSupplyRestore` (#4158) возвращает долю удаляемых
продолжений только **голове** при схлопывании — реюзнутый сегмент без «Обеспечения» он не покрывает.

`reconcileOrphanOrderSupplies` **ВОССТАНАВЛИВАЕТ** инвариант «выпускает заказ ⇔ привязана», а НЕ удаляет:
эта резка — **единственный выпуск** заказа (свежая покрывающая уже удалена), удаление недокрыло бы заказ.
Кандидаты — задания с проходами БЕЗ единого «Обеспечения» (и складские, и заказные-сироты). Их «Партии ГП»
читаются (`object/{ПартияГП}?F_U=резка`); у партии с непустым «ID заказа» создаётся «Обеспечение» на
позицию заказа (`up = позиция` по паре `ID заказа | ширина «Партии ГП»` из `genPositions`; ссылка на
«Партию ГП`, рулоны = спрос «Партии ГП»). Складская «Партия ГП» (без «ID заказа») **законно** без
«Обеспечения» — не трогаем. Идемпотентно: после `reload` у резки есть «Обеспечение» → она уже не кандидат
(повторный прогон не плодит дубли). Не удалось сопоставить позицию (её нет в активном `positions_list`) —
**не фабрикуем** связь, а орём (консоль + тост, ТЗ §14/#4059). Удаления заказных резок нет — данные
о выпуске сохраняются.

**ГЛУБЖЕ — почему «ID заказа» «Партии ГП» оказывался ПУСТ (#4179).** `reconcileOrphanOrderSupplies` чинит
сироту только при **непустом** «ID заказа» её «Партии ГП» (пустой — законный склад, `if(!oid)return`). Но
у части сирот он ПУСТ уже при генерации. Корень — `setup-only` **сегмент дробления** заказного задания.
Задание заказа (напр. 3942: `MW308 154×20 + 110×4` = 4 прохода раскладки `154×5+110×1`) при генерации
дробится по дням на `[setup-only (0 проходов), сегмент проходов (4)]` (`splitMachineQueue`, `segRunsAll=[0,4]`).
Для setup-only сегмента `splitSupplyShares` даёт долю рулонов **0**, и `createFinishedBatches` брал «ID заказа»
из этой доли (`ordersByWidth` строился только по долям с `rolls>0`) → «Партия ГП» setup-only сегмента писалась
с **пустым** «ID заказа» = сирота «нет связей». Дальше `mergeContinuationChains` держит головой цепочки
**самую раннюю** запись (setup-only, день N), а `applySplitPlan` реюзит именно её (SET проходы 0→4) и удаляет
покрывающий сегмент проходов — итог совпадает с §выше, но `reconcile` бессилен (пустой «ID заказа»).
**Фикс #4179 — двумя частями:** (1) `createFinishedBatches`: «ID заказа» «Партии ГП» — по **покрытию
раскладки** (заказы покрытых позиций этой ширины), НЕ по доле рулонов сегмента; число рулонов по-прежнему
по доле сегмента (у setup-only = пусто). Складская полоса (нет покрытой позиции этой ширины) остаётся без
«ID заказа» — запас не сломан. (2) `reconcileOrphanOrderSupplies`: рулоны восстановления = `МАКС(хранимое,
полос × проходов задания)` — у реюзнутого setup-only «Кол-во план» устаревает (`полос×0`, fallback → полос),
иначе связь восстановилась бы заниженной (`5` из `20`) и заказ снова недообеспечен → повторная генерация.

**Когда восстанавливать.** Связь рвёт **сама пересборка** `applySplitPlan`, поэтому
`reconcileOrphanOrderSupplies` вызывается **после её `reload`** (по свежим `self.cuts`/`self.supplies`);
восстановили → `reload` ещё раз. А когда план оптимален (переставлять/дробить нечего), `autoSequenceQueue`
НЕ зовёт `applySplitPlan` — поэтому в ветке «изменений нет» он тоже зовёт `reconcileOrphanOrderSupplies`
(восстановили → `render`), иначе «Упорядочить»/«Сгенерировать» на стабильном плане висящую «нет связей»
не чинит. На «показе» (`renderQueue`) чистку не вешаем: у склада «Обеспечения» нет **штатно**, и читать
его «Партии ГП» на каждой отрисовке — лишняя нагрузка; восстановление привязано к точкам пере-разбиения.

**Удаление задания сносит ВСЮ цепочку дробления (#4292).** Кнопка «🗑» удаляла ТОЛЬКО кликнутую запись
и её «Обеспечения». Но задание, **разбитое по дням**, — это ГОЛОВА + ПРОДОЛЖЕНИЯ (одна логическая резка,
`mergeContinuationChains`), а «Обеспечение»/заказ висят на **голове**. Удалив голову, продолжение теряло
обеспечение и заказ → висело «нет связей» и попадало в **ОТХОДЫ**. Авто-чистка сирот (см. выше) его НЕ
трогает: у продолжения **проходы > 0** (реальная намотка), а `planCutOperations` сносит лишь `setup-only`
мусор (0 проходов); `reconcileOrphanOrderSupplies` тоже бессилен — заказ/обеспечение **удалены** (нечего
восстанавливать). Фикс: `deleteCutTask` берёт ПОЛНУЮ цепочку записей (`chainRecordIdsForCut` →
`mergeContinuationChains`) по кликнутой резке (клик по ЛЮБОЙ части сносит цепочку целиком — это одно
задание), собирает «Обеспечения» ВСЕХ звеньев и удаляет всё: сперва «Обеспечения» (пул), затем записи
цепочки **ПОСЛЕДОВАТЕЛЬНО хвост → голова** (продолжения ссылаются на голову «ID первой части» #3892).
Зафиксированное (🔒) звено цепочки блокирует удаление (снять фиксацию, #3508). Одиночная (не дроблёная)
резка — как раньше, `[cutId]`.

**Лидер setup-only сегмента = 0 (#4021).** «Резка и Лидер» = намотка + `BETWEEN_CUTS × Кол-во план`
(лидер после каждого прохода). У `setup-only` сегмента проходов нет (`Кол-во план = 0`) → нет ни
намотки, ни лидера, и «Резка и Лидер» = 0. `cutLeaderRuns()` при 0 проходов возвращает 1 (фолбэк для
реальной резки с несохранённым «Кол-во план»), поэтому `computeCutSetupUpdates` (#3698) и `buildSchedule`
считают лидер `BETWEEN_CUTS × 1 = 2` даже у наладки — бейдж дня с одной наладкой показывал `45 + 2 = 47`
вместо 45 (`scheduleFromStored` читает «Резка и Лидер» как `durationMin`). Лидер учитываем ТОЛЬКО при
`Кол-во план > 0` (в обоих местах; в `buildSchedule` — гейт по `setupTaskIds`, как у `durationMin`).

**Настройка setup-only хвоста делится по потолку нахлёста, остаток — на продолжении (#4026/#4111).**
`setup-only` хвост дня (`Кол-во план = 0`, #3635 п.5) — это НАЛАДКА следующей резки, начатая в конце дня N;
проходы идут в дне N+1. `splitMachineQueue` кладёт в хвост дня N лишь то подмножество переналадки, что
**влезает до потолка нахлёста настройки** `cutEndMin + MAX_OVERWORK_TUNE`
(`minOverlapTailSetupMinutes` — минимальное подмножество, добивающее день до `cutEndMin`, если оно
кончается ≤ потолка), а остаток настройки и проходы — на день N+1. Хранимые колонки наладки
(`computeCutSetupUpdates`, #3698) обязаны совпадать с этой раскладкой, иначе бейдж/окно дня вылезают за
нахлёст (жалоба #4111: полная наладка 45 на хвосте у `cutEnd` → карточка до 16:46 при потолке 16:20;
день `447 + 45 = 492` вместо `447 + 15`). Раскладку зеркалит чистая `splitTailSetupAtCeiling(tailStartMin,
ножи, сырьё, cutEndMin, MAX_OVERWORK_TUNE)` — тот же перебор подмножеств {ножи?, сырьё?} по возрастанию
суммы + гейт потолка, но по ДВУМ именованным компонентам, чтобы поделить хранимые «Наладка ножей» и
«Сырье/намотка». Что остаётся в дне N зависит от **старта хвоста** (`planStart`), а НЕ от загрузки дня:
у `cutEnd` (16:01, до конца окна 9 мин) — только смена сырья 15 (→16:16), ножи 30 на продолжение; раньше
(15:33, 37 мин) — вся наладка 45 влезает (→16:18) и остаётся в дне N; у самого края (16:08, любой компонент
за 16:20) — в дне N ничего, вся наладка на продолжение. Старт хвоста здесь — ЦЕЛАЯ занятость дня (после
#4149, см. ниже): подмножество выбирается от того же значения, что уйдёт в бейдж, поэтому карточка хвоста
не вылезает за 16:20.

> **#4116 — «задание со всеми нулевыми таймингами».** Есть окно `room` (до `cutEnd`) в 31–34 мин
> (напр. `cutEnd 16:13`, хвост 15:40): ни ножи 30, ни сырьё 15 по отдельности НЕ добивают день до
> `cutEnd`, а вся наладка 45 вылезает за потолок (16:25 > 16:23). Раньше `splitTailSetupAtCeiling`,
> зеркаля генерацию плана (#3939: «минимальное добивающее подмножество или ничего»), возвращала
> `{0,0}` — хвост-задание оставалось ПУСТЫМ (все тайминги 0), а вся наладка, включая настройку ножей
> 30, которая заведомо влезала (16:10 ≤ `cutEnd`), уезжала на следующий день. Но здесь принцип другой,
> чем при генерации: хвост-задание **уже создано** в дне N — оставлять его пустым бессмысленно.
> Поэтому если ни одно подмножество не добивает день под потолком, в дне N держим **НАИБОЛЬШЕЕ
> подмножество, ещё влезающее под потолок** (максимум наладки в дне N — настройку ножей 30), а остаток
> (сырьё 15) — на продолжение. `{0,0}` возвращаем ТОЛЬКО когда даже минимальный компонент за потолком
> (случай «у самого края» выше). Это ЕДИНСТВЕННОЕ отличие хранимых колонок от генерации плана
> (`splitMachineQueue`, где хвост либо создаётся минимально-добивающим, либо резка уходит на завтра
> одной карточкой) — но применяется к УЖЕ существующему хвосту, а не к решению его создавать.

**Куда уносится остаток настройки хвоста.** Продолжение = **последующая резка той же цепочки** — сканируем
очередь снимка вперёд по `chainRoot` (нормализация `firstPartId` через `String(...).trim()`, как #3892;
голова хвоста и продолжение делят корень = `id` головы). Продолжение бывает **не соседним** (между ним и
хвостом встают чужие резки) — поэтому именно скан, а не только `arr[i+1]`. Остаток добираем к нему ТОЛЬКО
если у него **нулевая переналадка входа** (`changeover = 0` — та же конфигурация, ножи хвоста и есть его
заправка), иначе у продолжения СВОЯ наладка от нового сырья и добор дал бы двойной счёт. На **ateh** колонки
`cut_first_part` в отчёте `cut_planning` фактически нет → `firstPartId` пуст у всех резок; тогда продолжение
опознаётся именно по `changeover = 0` (ближайшая последующая резка той же конфигурации), а не по цепочке.
Если продолжения нет (следом другое сырьё) **или** у него своя наладка (чужая конфигурация между) — остаток
унести некуда: вся наладка остаётся на хвосте дня N (нахлёст здесь неизбежен, но настройку терять нельзя).
Резка с проходами (`runsC > 0`) — не хвост, настройку не делит. Загрузка дня N+1 на раскладку хвоста **не
влияет**: даже когда день продолжения полон, хвост дня N режется по потолку нахлёста, а переполнение дня N+1
отражается честно (а не прячется полной настройкой 45 за нахлёстом дня N). Диагностика (`window.PP_TRACE`):
для каждого `setup-only`-хвоста печатается `#4111 хвост <id> старт=<мин> наладка[нож/сыр]=… → в дне N …,
на продолжение <id> …`.

**Хвост настройки и потолок нахлёста при простое (#3934).** `splitMachineQueue` кладёт `setup-only`
сегмент в **хвост дня N** с намеренным нахлёстом за конец смены (#3635 п.5/#3739/#3805). Когда у
станка есть хоть один блок простоя («Отпуск» — в реальном плане их десятки, все далеко),
`applyDowntime` прогонял **все** сегменты через `nextFreeWorkMinute`, а тот применял потолок нахлёста
(`fitEnd = cutEnd + MAX_OVERWORK_CUTS`) к **каждому** — даже к не тронутому простоем. Хвостовая
настройка кончается за потолком → её выталкивало на **начало следующего дня поверх её же
продолжения**: «настройка в начале дня» и рост минут дня (бейдж 495/501). Без простоя
`shiftPlacementsPastDowntime` выходит сразу, и того же не было («то есть, то нет»). Потолок нахлёста
(#3907) применяется **только** к сегменту, реально **сдвинутому простоем** (блоком либо встык-курсором
за предыдущим сдвинутым): не тронутый простоем хвост остаётся в конце дня N, продолжение — на дне N+1.

**Потолок нахлёста НЕ выталкивает setup-only хвост (#4021).** `setup-only` хвост — **намеренный**
нахлёст конца дня (#3635 п.5); потолок `fitEnd` (#3907) существует для **проходов**, чтобы сдвинутая
простоем намотка не тянулась в ночь. Но встык-курсор (#3764) нудживал хвост даже на 1 минуту (предыдущая
резка кончалась чуть позже логического старта хвоста), это давало `cursorMoved=true`, и потолок
**выталкивал сам наладочный хвост** за конец смены. Перед выходными это выбрасывало его за **все**
выходные — на 08:00 понедельника **одинокой наладкой**, а `shiftPlacementsPastDowntime` (#3951) сдвигал
весь дневной объём станка на вторник: понедельник «недогружен, сделана только наладка» (issue #4021 —
Станок 1, «Пн 29.06.2026 (47 мин), заданий 1»; 47 = наладка 45 + фантомный лидер 2, см. выше). Фикс:
`nextFreeWorkMinute` получает `skipCeiling`; `shiftPlacementsPastDowntime` ставит его для сегментов, у
которых `acc.overhangTail(it)` истинно (в `splitMachineQueue` — `s.setupOnly`). Такой хвост потолком не
выталкивается — остаётся в конце своего дня (нахлёст ограничен `MAX_OVERWORK_TUNE` при укладке), а
**блоки простоя он по-прежнему обходит** (проверка блоков в `nextFreeWorkMinute` не тронута). Проходы —
как раньше (потолок #3907 для них в силе).

**Сохранение границ дней при сдвиге (#3951).** `shiftPlacementsPastDowntime` пакует сегменты
встык (#3764: резку, сдвинутую простоем, «догоняют» следующие — без зазоров). Но встык-курсор
паковал **продолжение разбитой по дням резки в хвост дня её первой части**: после длинного «Отпуска»
день-сплит схлопывался в один абсолютный день — бейдж дня вылезал за ёмкость (490 при 460), а
следующий день недобирал (issue #3951, лог: планировщик дал ≤ 460 на каждый день, схлопывание
случилось на сдвиге). Фикс: сегмент, исходно стоявший на **более позднем дне**, чем предыдущий, и
после сдвига оказывается на более позднем дне (не пакуется встык в хвост предыдущего). Сегменты
**одного** исходного дня по-прежнему пакуются встык (заполняют день без зазоров).

**Ёмкость дня учитывает частичный простой (#3978).** `splitMachineQueue` пакует день **логически
от `dayStart` (08:00)**, а `applyDowntime` затем сдвигает готовые сегменты за блоки простоя. Пока
ёмкость дня считалась без учёта простоя, день с **частичным** блоком (напр. утро 08:00–10:00 после
отпуска/выходного — смена начинается в 10:00) переполнялся логически; сдвиг за простой выталкивал
вылезший за конец окна сегмент на следующий день **целиком** (дробить после сдвига нечем), и день
недобирал: ~99–129 мин вместо достижимых ~330, а работа каскадом стекала на следующие дни (issue
#3978). Фикс: минуты простоя **внутри рабочего окна дня** уменьшают его ёмкость (`dayLostToBlock`
в `effCapacity`/`availFor`) — укладчик **дробит** резку по уменьшенной ёмкости и добивает
частично-простойный день, а сдвиг лишь ставит сегменты после блока (перелив за потолок исключён,
#3907). **Полностью** заблокированный день (выходной #3788 / отпуск на всё окно, сумма блока ≥
ёмкости) `dayLostToBlock` **не трогает** — им по-прежнему занимается `applyDowntime` (поведение
раскладки выходных/отпуска прежнее). Баг предсуществующий (воспроизводится идентично до и после
#3974); набивка свободных от «С» (#3974) лишь сделала его виднее — свободные резки сваливаются на
первый день, где стоит утренний простой.

### Внутридневная пересортировка после упаковки (#4139)

Слой размещения вставляет резки **по одной**, выбирая точку вставки по минимуму штрафа, и собранный
день больше не чинит. Жадная вставка оставляет «осиротевшие» резки вдали от их группы: одна и та же
конфигурация попадает в день дважды, разорванная чужим сырьём. На плане 01–08.07 (Станок 1):

```
02.07  MW308/8  → MWR113L/8 → MW308/8            60 мин  →  45 после склейки
07.07  MW308/15 → MR194/15  → MW308/15 → MR194/13 75 мин  →  45
```

Поэтому после реальной упаковки (`packAll`, когда день каждой резки уже назначен) очередь каждого
станка пересортировывается **внутри дней** — `resequenceWithinDays`.

**Почему это безопасно для сроков.** День и состав дня не меняются, переставляются только резки
внутри дня → штраф срока (§8 п.4/5) у каждой резки тот же. Ни одна резка не может уехать на более
поздний день: суммарная наладка дня только уменьшается.

**Цель — `sequencingCost`, а не голые минуты.** Минимум физических минут (`changeoverCost`, смена
ножей 30 в любую сторону) разгоняет **рост** числа полос вопреки #3130: на реальном плане он даёт
те же −120 мин, но 12 «ростов полос» вместо 9. Минимум `sequencingCost` (#3996, рост полос дороже
снятия) даёт те же −120 мин и **6** «ростов». Приёмка **двойная**: цель строго лучше **и** реальные
минуты наладки не выросли — снятие двух «ростов» (−20 каждый) окупает лишнюю смену ножей (+30) по
цели, но оператор в цеху заплатит эти 30 минут.

**Группировка рулона тай-брейком (#4151).** `changeoverParts` берёт 15 мин за смену сырья **или**
намотки **или** партии одинаково, поэтому переход «тот же рулон, другая намотка» (MR194 OUT↔IN) и
«другой рулон» (MR194→MWR233) равны по реальным минутам — и движку всё равно, рвать ли группу одного
рулона чужим материалом (жалоба #4151: «MWR233 влез между MR194 — менять рулон, а потом обратно»; по
минутам это оптимально, но физически лишний перемонтаж). Поэтому **только внутри `resequenceWithinDays`**
`sequencingCost` добавляет КРОШЕЧНЫЙ штраф `ROLL_TIEBREAK_MN` (0.001) за смену **рулона** (материал/
партия, НЕ намотка) — порядка величины ниже любой реальной разницы, поэтому лишь разрешает ничьи. При
равной реальной переналадке DP держит резки одного рулона вместе; двойная приёмка (`newReal ≤ oldReal`)
не даёт разменять перемонтаж на лишнюю смену ножей. Модель стоимости, физтайминг и хранимые колонки
«Наладка ножей»/«Смена сырья» НЕ меняются. Флаг активен только здесь — `greedySequence`/`orderCuts`
(тай-брейк «полосы по убыванию» #3785) не трогаем. На плане 03.07 (Станок 1) очередь
`MR194×6 → MWR233 → MR194(IN) → …` перегруппируется в непрерывный блок MR194 при той же реальной
переналадке; план-wide внутридневные перемонтажи 6 → 4, реальная наладка не растёт (#4150).

**Оптимум сквозной, а не подневный.** Перестановка дня меняет резку, которой день заканчивается, а
значит и переход в следующий день. Подневная жадность даёт локально лучшие дни и суммарно **худшую**
очередь (проверено: 1830 → 1965 мин). Поэтому DP по цепочке дней: для каждого дня считаются точные
минимумы гамильтонова пути (Held-Karp по подмножествам) для всех пар «первая группа → последняя», и
дни сшиваются по стоимости стыка. Резки с одинаковой **подписью конфигурации** (`cutConfigSig`:
мультимножество ширин ножей + ширина ролика + сырьё + намотка + партия) схлопываются в один узел —
переход между ними бесплатен. Дни шире `RESEQ_MAX_NODES = 12` разных конфигураций не переставляются.

**Ограничения (иначе станок пропускается целиком).**

- **Фольга последняя в дне** (#3717) — вшито в перебор: после первой фольги идёт только фольга.
  **Чинится ЖЁСТКО (#4224).** Если ВХОДНОЙ порядок уже нарушает правило (напр. §12-релокация по
  реальным дням впихнула нефольгу ЗА фольгу — реальный случай ateh: фольга срок 10.07 осела в
  середине 01.07), проход применяет foil-last порядок **без гейта по стоимости** (`violatesFoilEnd-
  OfDay(runs)` → сразу `return out`, минуя двойную приёмку `newSeq<oldSeq`), а внешняя приёмка при
  улучшении foil-last (`newFoilBad<oldFoilBad`) допускает даже перенос резки на более поздний день —
  НО никогда за срок (просрочка недопустима #4224). #3717 — жёсткое правило, лишняя наладка — его
  цена; **добавлять** нарушения проход по-прежнему не вправе.
- **День-сплит не рвётся** (#3635 п.5): резка, переползающая на следующий день, остаётся последней.
- **Заправка станка** (#3853): начало очереди считается переходом от реальной заправки, а не с нуля.
- Проверка **по каждому станку отдельно** — очереди станков пакуются независимо, неудача на одном
  не должна отменять выигрыш на остальных.

Рубильник — `INTRA_DAY_RESEQUENCE=0` в «Настройке» (по умолчанию проход включён). Не работает при
`preserveOrder` (ручной порядок оператора) и без слоя размещения.

**Результат на плане 01–08.07:** 1830 → 1710 мин (−120, 6.6%); Станок 1 690 → 600, Станок 2 495 →
465; «ростов полос» 9 → 6.

### Кросс-дневная склейка дубля сырья в блоке ножей (#4184/#4205)

`resequenceWithinDays` (#4139) склеивает дубль одной конфигурации **только внутри одного дня**. Но
жадная вставка слоя размещения роняет резки одной конфигурации и **через границу дня**: одно и то же
сырьё+намотка попадает в один блок ножей дважды — единичное задание в начале блока и группа-дубль в
конце, между ними чужие материалы, а концы лежат в разных днях. На плане (Станок 1, блок ножей
`110*8`) сырьё `MWR200|OUT` стоит единичным заданием в начале и тройкой в конце — слить их в один
блок экономит одну смену сырья (−15 мин).

Поэтому после упаковки очередь каждого станка проходит **кросс-дневной постпроход склейки**. Он
работает **по блокам ножей** (порядок самих блоков ножей неизменен, ТЗ; границу блока каждый вид
кандидата определяет своей подписью — см. ниже) и для каждого блока с ≥3 резками пробует
**кандидатов** нового порядка:

- **глобальные** (`clusterMaterialWithinKnifeBlocks`): кластеризовать **все** конфигурации блока по
  первому (`byFirst`, поздний дубль тянется вверх к раннему) или последнему (`byLast`, ранний дубль
  толкается вниз к позднему) появлению; граница блока — `knifeBlockSig` (ширины ножей + ширина джамбо);
- **хирургические** (`materialIslandMergeCandidates`, #4205): для **каждой** конфигурации блока с ≥2
  островами — стянуть к якорному острову (первому **или** последнему появлению) **только её** резки,
  оставив остальной порядок блока как есть. Границу блока здесь берём по **`knifeWidthSig`** (только
  набор ширин ножей), а не по `knifeBlockSig` (#4214 — см. ниже).

**Приёмка кандидата** (реальная упаковка `packOrderedMachine`): (1) реальные минуты наладки строго
уменьшаются — ушла хотя бы одна смена сырья; (2) **ни одна** резка не уезжает на день позже своего
срока (не плодит и не углубляет просрочку — иначе штраф опоздания перебьёт экономию); (3) не портит
«фольга в конце дня» (#3717). Из прошедших кандидатов берётся дающий минимум реальной наладки;
per-station цикл применяет его и повторяет, пока есть улучшение (монотонно по наладке → сходится).

Критерий приёмки — **реальные минуты наладки** (`runChainCost` = Σ `changeoverCost`, ножи + сырьё),
а НЕ полный взвешенный штраф позиции (`scorePosition`, §8). Мягкие веса §8 — смежность заказа
(`ORDER_DIFF_PENALTY_MN` = 10, #4194) и расстояние (#4106) — в приёмке **не пересчитываются**;
жёстко гейтятся лишь два верхних приоритета (ТЗ §11), которые минуты наладки сами не ловят: **срок**
(п. 2 — просрочку постпроход не создаёт) и **фольга** (п. 3). Это осознанный выбор, согласный с
порядком приоритетов: склейка убирает реальную смену сырья (15 мин), стоящую по §11 выше смежности
заказа (10); проиграть по мягкой смежности она может лишь при разрыве нескольких соседств сразу, что
жёсткого вреда не несёт, а срок и фольга — под гейтом. Поэтому «переналадка строго вниз» здесь —
достаточный критерий, а не упрощение полного штрафа §8.

**Зачем хирургические кандидаты (#4205).** Глобальная перестановка тасует **весь** блок сразу,
поэтому приёмка (проверка срока) отвергает её, если от неё уезжает за срок хоть одно **чужое**
задание блока. В примере `MWR200`: единичное приколото своим сроком (день 0) — `byLast` роняет его
собственный срок; `byFirst` тасует весь блок и роняет срок соседа. Обе глобальные раскладки
отвергаются, и выгодная локальная склейка не случается. Хирургический кандидат двигает **только**
резки `MWR200` к их же острову — минимальное движение проходит приёмку там, где глобальная
перетасовка ломала чужой срок. `shouldRelocate` (§12) тут ни при чём: он трогает лишь
просроченные/после-фольги резки, а единичное `MWR200` в срок и не за фольгой.

**Граница блока по ножам, не по джамбо (#4214).** `knifeBlockSig` = ширины ножей **+ ширина джамбо
(`rollerWidth`)**. Когда острова одного сырья идут с **иной шириной джамбо**, чем разделяющие их
задания (напр. `MWR200` на джамбо 500, соседние `MW308` на 0), визуально единый блок ножей `110*8`
**дробится** по границе джамбо: единичное `MWR200` — прогон длиной 1 (склеивать не с чем), тройка —
свой прогон (уже подряд). Хирургический постпроход, шагающий по `knifeBlockSig`, острова через
границу джамбо **не видит** — дубль остаётся разорван (жалоба #4214: «разрыв по сырью — ножи и сырьё
идентичны, и даже лидер такой же»). Поэтому `materialIslandMergeCandidates` берёт границу блока по
**`knifeWidthSig`** (только ножи): джамбо — измерение сырья, а её сужение — смена ножей (`KNIFE` в
`changeoverCost`), которую **приёмка** и взвесит. Так острова одного сырья через разные джамбо
попадают в один блок и сливаются, а если перенос через границу джамбо дороже экономии на смене сырья —
приёмка (переналадка строго вниз) кандидата отвергнет. `clusterMaterialWithinKnifeBlocks` (глобальный
кандидат) остаётся по `knifeBlockSig`: он тасует **весь** блок, и расширять его границу на разные
джамбо незачем — нужную склейку даёт хирургический кандидат (суперсет).

Проход включён всегда, когда работает слой размещения (`slotPlacement && !preserveOrder`) и известны
сроки (`dueDayByCut`); отдельного рубильника в «Настройке» у него нет (`intraBlockDedup` — только
тестовый оверрайд). Блок с зафиксированной (🔒) резкой не трогаем — у неё якорь дня.

### Пересборка дня после переноса/удаления (#3840)

Перенос задания на другой день/станок (`moveCutToDay`) и удаление одной резки (`runDeleteCutTask`)
изымают резку из середины дня станка. Раньше они переписывали `planStart` только самого
перемещаемого задания, а **сохранённый `planStart` прочих резок дня
оставался прежним** — на месте вынутой резки висел простой. Страница планирования его тогда не
показывала (очередь до #3846 пересчитывала тайминг через `buildSchedule` на лету), а РМ «Диаграмма
Ганта (задания)» рисует **сохранённый** `planStart` — там дыра была видна. После #3846 обе РМ
показывают сохранённый план, поэтому такой простой виден на обоих экранах одинаково.

**Удаление резки и перестановка ↑↓** терминально вызывают `autoSequenceQueue(preserveOrder=true)` —
тот же пересчёт, что после генерации: `splitMachineQueue`/`buildSchedule` с `gapFill` пакуют день
**встык**, дыра схлопывается, `planStart` затронутых дней сохраняется. `preserveOrder` держит ручной
порядок оператора **по сохранённому `planStart`** (#3923), фольгу — в конце дня (#3717),
зафиксированные задания — на своих днях (#3792); пишутся только изменившиеся записи (идемпотентно,
#3427). Удаление/↑↓ смещают резки **раньше** (закрытие дыры) — за срок не уводят.

**Ручная ↑/↓ + «Пересчитать наладку» не оставляет просрочку (#4200).** Ручная перестановка ↑/↓
(`moveCutInDay`) меняет местами сохранённый `planStart` двух соседей и метит станок «грязным» →
кнопка «↻ Пересчитать наладку» (#4189). Пересчёт = `autoSequenceQueue(preserveOrder=true)`: он
сохраняет новый ручной порядок, но раньше паковал дни встык БЕЗ проверки срока — свап мог утопить
задание за срок, и просрочка появлялась молча (issue #4200). Теперь `preserveOrder`-пересчёт тоже
проходит доп. проходом #4118 (`relocateOverdueReal`, см. «Срок держат реальные дни»): подтягивает
ТОЛЬКО просроченные задания в срок (в наименее штрафное место, при нужде на другой станок), порядок
задач в срок сохраняет; каждое перемещение и остаток за сроком пишутся в трассу, панель «просрочено:
N» показывает остаток. Сравнение срока — по КАЛЕНДАРНОМУ дню приземления (#4200), чтобы арбитр и
панель не расходились за выходными.

**Перенос задания на другой день (`moveCutToDay`, #4074)** вставляет задание в чужой день и потому
толкает остальные **позже**. Поэтому он пересобирает план **ПО СРОКАМ** — `autoSequenceQueue(preserveOrder=false)`
(`deadlineAware`, как «Упорядочить»): EDD-размещение (#4050/#4059/#4064) и резерв фольги (#4068) не
дают заданиям уехать за срок. Перенесённое задание **закрепляется** на выбранном пользователем дне
(`moveScope.pinCutIds` → временный `c.fixed`=«замок дня» в `buildSequenceOps`, снимается в `finally`);
остальной план раскладывается по срокам вокруг него (перестановка допустима — важно не нарушить
сроки). Прежде перенос завершался `preserveOrder=true` (`deadlineAware` выкл): паковал всё от «С»
без учёта сроков и уводил задания за срок («перенос с несоблюдением сроков» — появлялись просроченные
задания на других станках и днях, issue #4074).

**Опции формы переноса (#4221).** Форма «Перенести задание на другой день» несёт:

- **Положение в дне** — радио `По весу` (по умолчанию), `В начало дня`, `В конец дня`.
  - `В начало`/`В конец` — задание **приколото** (`moveScope.pinCutIds` → временный `c.fixed`): в слое
    размещения оно неподвижный сосед на плейсхолдер-`planStart` (начало/конец дня).
  - `По весу` — задание **НЕ приколачиваем** (иначе осело бы неподвижным в плейсхолдер), а «замыкаем»
    на выбранный день и станок: `moveScope.weightPositionCutIds` → `dayLockByCut[id]=день-смещение`.
    В `computeSlotPlacement` такое задание кладётся **подвижным** (позиция в дне — по наилучшему весу
    `scorePosition`, полный набор штрафов §8), но с `lockDay`/`lockSlitter`: `placeSlot` рассматривает
    только точки замкового **дня** и **станка** (кандидаты вне замка отбрасывает; `bestAny` — страховка,
    если замковый день переполнен). Задание с замком размещаем **последним** в своём классе (нефольга/
    фольга), чтобы оно встало в лучшую щель между уже разложенными соседями, а не в индекс 0 пустого дня.
    День держит сам замок (эмерджентно из порядка + упаковки), `c.fixed` не нужен.
- **В пределах одного станка** (по умолчанию включена) — `moveScope.withinSlitterIds`: во вход
  планировщика (`planInput`) берём задания **только задействованных переносом станков** — целевого и
  (при смене станка) исходного, — а `slitterIds` сужаем до них же. Прочие станки не тронуты. Чтобы
  перенос **не кидал задания между станками**, при scope > 1 станка каждое задание замыкается на СВОЙ
  станок (`machineLockByCut` → `lockSlitter` в `computeSlotPlacement`, `placeSlot` рассматривает только
  этот станок): миграция запрещена, порядок пересобирается на каждом станке отдельно, а исходный станок
  при смене станка перепаковывается (его дыра от ушедшего задания залечивается). #4225 (доработка
  #4221): прежде пересобирался только целевой станок — исходный оставался с дырой.

**Ручная перестановка (#3923).** Кнопки ↑↓ (`moveCutInDay`) меняют порядок дня **обменом
сохранённого `planStart`** двух соседних резок (`preserveOrder`, порядок оператора сразу
материализуется в `planStart`). Перенос на другой день (`moveCutToDay`) намерение по дню сохраняет
через закрепление перенесённого задания (см. выше), но внутридневную ручную очередь прочих резок
подчиняет срокам.

> Инвариант: сохранённый `planStart` дня станка всегда без внутреннего простоя — любая операция,
> изымающая резку из дня, обязана пересобрать день (а не «оставить на следующую генерацию»).

### Показ из сохранённого плана, без live-пересчёта (#3846)

Раньше очередь на странице планирования пересчитывала тайминг **на каждый рендер** через
`buildSchedule` (gapFill). Это расходилось с сохранённым планом, который рисует РМ «Диаграмма
Ганта»: (1) для первой резки бралась наладка «с нуля» (`firstCutSetup`), а не реальная заправка
станка из `prev_cut_setup`; (2) не учитывался обед (#3342). Итог — минуты и времена двух экранов
не совпадали (жалоба #3846), а 40-минутный обед, который генерация зашивает в `planStart`
послеобеденных резок зазором, на Ганте выглядел необъяснённой «дырой».

Теперь обе РМ — **единый источник правды: сохранённый план**. Очередь строит тайминг
`scheduleFromStored(cuts, baseMidnight)` прямо из записанных полей (главное значение `planStart` =
окно/начало настройки; наладка = ножи + смена сырья; «Резка и Лидер» #3700 = намотка + лидер),
без `buildSchedule`. Окно резки = `setup + (намотка+лидер)`, ровно как ширина бара на Ганте →
минуты и времена **всегда совпадают**.

Обед и перерывы (10:00/15:00) на показе не пересчитывают, а **находят** в уже сохранённом расписании
(`intraDayBreaks` из «Настройки»; `lunchBlocksFromSchedule` / `ganttLunchMarkers`). Их несёт
**несущая карточка** — первая, чьё сохранённое окно накрывает время перерыва (`computeQueueBreakMarkers`
в очереди, `ganttLunchMarkers`/`ganttBreakMarkers` на Ганте). Показ:

- **Очередь РМ «Планирование» (#4075/#4094):** серый значок на несущей карточке (🍴 обед / ⏸ перерыв);
  её окно **удлиняется на длительность перерыва** — конец охватывает работу + перерыв, а число минут
  «· N мин» остаётся **рабочим** (как «(N мин)» в скобках у бара Ганта).
- **Гант (#4052):** бар несущего удлиняется на длительность обеда/перерыва (`extendMinByTask`) — та же
  модель «бар честно охватывает работу + перерыв».
- **Карточка настройки** (setup-only, 0 проходов) пишет своё окно так же, как резка:
  `⚙ Настройка ножей и сырья · 16:15 – 16:45 · 30 мин` (`formatSetupScheduleLine`), с тем же сдвигом
  и удлинением от обеда/перерывов.

**Кто двигает последующие задания дня (#4121).** Перерывов 10:00/15:00 в `planStart` нет никогда —
они всегда сдвигают все задания дня после несущего (`shiftByCut` в очереди, `shiftMinByIndex` на
Ганте, накопительно и в хронологическом порядке). Обед сдвигает **только когда его нет в сохранённых
стартах**:

- **обед-зазор** — генерация оставила дыру ≈ `LUNCH_DURATION` перед заданием, стартующим не раньше
  `LUNCH_START`. Он уже учтён в `planStart`: только удлиняет несущую, никого не двигает (иначе
  двойной учёт);
- **сквозной обед** — такой дыры в дне нет, задания идут цугом через `LUNCH_START`. Станок в обед
  стоит, поэтому обед сдвигает всё после несущей, как перерыв.

Обе РМ различают эти случаи одним правилом по зазорам дня: `lunchBakedIntoStarts` в очереди,
флаг `fallback` маркера `ganttLunchMarkers` на Ганте.

Сквозной обед возникает на дне, где `shiftPlacementsPastDowntime` (см. «Отпуск станка») сдвинул
задания за окно простоя: сдвиг (например, +120 мин за «Отпуск» 08:00–10:00) больше обеденного
зазора, и встык-упаковка его схлопывает. Ёмкость дня обед при этом резервирует (`effCapacity`),
а сохранённые старты его не содержат.

**Несущее обед задание якорится на сохранённый старт (#4041).** Обед находят по **сохранённым**
временам (`ganttLunchMarkers` → `cutTimeRange`), а бар «несущего» рисуют по старту, который анти-
нахлёст (#3887) мог подвинуть **вперёд** встык за предыдущим баром. Из-за этого пролёт бара
(старт-встык → старт послеобеденной резки) оказывался короче реального: напр. в БД старт `11:58`,
работа `65` (наладка 15 + резка 49.998), обед `40` → честный пролёт `11:58→13:43 ≈ 105` мин; сдвиг
до `12:02` давал «`12:02-13:42 (65 мин)`» = 100 мин, обед визуально «усыхал» до 35 (issue #4041).
Фикс: при растяжке несущего его левый край **якорят на сохранённый `planStart`** (снимая только
сдвиг встык #3887, сохраняя сдвиг за перерывы #4007) — тогда пролёт бара честно охватывает работу +
обед. Генерацию это не трогает: сохранённый зазор `конец_работы_несущего → старт_послеобеденной`
уже равен обеду (11:58 + 65 = 13:03; послеобеденная 13:42 ≈ 13:43 → зазор 39–40).

**Перерывы двигают бар и его подпись (#4007/#4066).** Два коротких перерыва (`FIRST_INTERVAL`
10:00 / `SECCOND_INTERVAL` 15:00 по `INTERVAL_DURATION_MN`, ТЗ §5) прозрачны планированию — их нет
в сохранённых `planStart`, — но на Ганте рисуются: несущий перерыв бар (чьё сохранённое окно
накрывает время перерыва) удлиняется на `INTERVAL_DURATION_MN`, а все последующие бары дня
сдвигаются вправо на ту же длительность (`ganttBreakMarkers.shiftMinByIndex`, добавляется к
`leftPx`). Подпись времени бара (`barText`) и его старт (`t.startMs`) считаются от **сдвинутого за
перерывы** старта — как и `leftPx`, — поэтому текст совпадает с позицией. Иначе бар, отрисованный со
сдвигом, показывал дообеденное сохранённое время: станок из **459 рабочих минут** рисуется до `16:39`
(`8:00 + 459 работа + 40 обед + 2×10 перерыв`), а подпись последнего задания стояла на `15:38-16:19`,
и «начало следующего бара после перерыва не сдвигалось» (issue #4066). Рабочие минуты в заголовке
станка «N (Σ мин)» и в подписи бара `(N мин)` перерыв **не** включают (только двигает бар и его время).

**Порядок раскладки — по времени `planStart` (#3920/#3923), окно резки — КАК ЕСТЬ (#4099).**
`scheduleFromStored` раскладывает резки станко-дня **строго по возрастанию сохранённого
`planStart`** и ставит окно каждой резки **ровно на её сохранённый `planStart`, без анти-нахлёста**
(#4099 «рисуем как есть», см. ниже). Это же время — единственный ключ порядка резки во всей системе:
«Очередность» больше не хранится (#3923). До её удаления она могла разойтись с `planStart` после
частичной пересборки дня: «застрявшая» резка с ранней «Очередностью», но поздним
`planStart` (хвостовая настройка на 15:58, тогда как остальные резки дня стоят с 08:00). Анти-
нахлёст forward-only: попав в обработку **первой** (по «Очередности»), такая резка выталкивала за
собой **все** резки дня в овертайм (день лез до 23:15 — issue #3920, «Овертайм/недопланирование»).
С раскладкой по `planStart` резка стоит там, где записана — ровно как бар на Ганте, — и соседей не
двигает. (Сама «застрявшая» резка — след неполной пересборки #3885/#3895; лечится повторной
генерацией/«Упорядочить», которая перепланирует её на своё место.)

**Рисуем КАК ЕСТЬ — без сжатия/растяжки (#4099).** Окно каждой резки ставится по её сохранённому
`planStart` **без сдвига**: ни анти-нахлёста встык (прежний #3885), ни потолка смены, ни переноса
переполнения на следующий день. Когда сохранённые `planStart` тяжёлых резок (с настройкой ножей)
перекрываются — issue #4099: у Станка 1 на 01.07 сохранено 24 задания = 1120 мин при смене ≈450
(след неполной пересборки/устаревших `planStart`) — перекрытие **видно как есть**: несколько резок
стоят на пересекающихся окнах, и сразу ясно, что на день назначено больше работы, чем влезает в
смену. Раньше слой показа план «улучшал» — разносил перекрытия встык (#3885/#3920), обрезал бар по
старту соседа (#3708), сдвигал бары за перерывы (#4007), тянул несущий обед бар (#4052), — и
переполненный день превращался в непрерывный цуг далеко за конец смены (бары после 16:30,
«параллельно» с фикс-метками обеда/уборки), пряча суть. Заказчик (#4099): «нефиг сжимать и
растягивать — рисуй как есть». Обе РМ — страница планирования (`scheduleFromStored`) и «Диаграмма
Ганта» (`cut-gantt.js`, `dedupeBarStarts`) — рисуют одно и то же сохранённое окно; перекрытия,
перерывы и обед показываются накладкой поверх, **не двигая и не растягивая** бар. Убрать перекрытие
можно только повторной генерацией/«Упорядочить» (перепланирует резки по ёмкости дня), а не сжатием
на показе.

> **Инвариант (#3923): `planStart` (главное значение `t1078`) — ЕДИНСТВЕННЫЙ источник порядка
> резки.** Все три РМ — страница планирования, «Диаграмма Ганта», пульт слиттера — упорядочивают
> резки станко-дня **строго по возрастанию `planStart`**. «Очередность» не пишется, не читается и
> не участвует в сортировке нигде (`sequence` остаётся только как in-memory ординал генерации, см.
> Шаг 3.2). Страница планирования и Гант показывают одни и те же сохранённые поля; тайминг на
> показе НЕ пересчитывается (пересчёт — только в генерации и пересборке дня, #3840). Расхождение
> экранов = баг персистенции плана, а не показа.

### Бейдж дня «(N мин)»: округление «Длительности» и допуск +2..+4 над бюджетом (#3635 п.4)

После заголовка станко-дня выводится бейдж **«(N мин)»** — суммарные рабочие минуты станка за день:
`N = Σ (наладка ножей + смена сырья + намотка + лидер)` по всем заданиям дня (`dayMinutesBySched`;
уборка #3155 в сумму не входит — у неё своя строка). Бюджет дня, с которым сверяется бейдж:

```
бюджет = (cutEndMin − dayStartMin) − LUNCH_DURATION + MAX_OVERWORK_TUNE
       ≈ (16:10 − 08:00) − 40 + 10 = 460  (для ateh)
```

**Бейдж может показывать на +2..+4 больше бюджета (напр. 462 при бюджете 460) — это округление, а не
превышение плана.** «Длительность, минут» каждой резки хранится и суммируется как **целое число минут,
округлённое ВВЕРХ** (`ceil`, #3635 п.4). Упаковщик (`splitMachineQueue`) пакует резки по **точным**
(дробным) часам `planStart` и укладывает день в бюджет; бейдж же складывает уже округлённые вверх
минуты каждой резки. На полном дне из 12–19 резок эти округления накапливаются в **+2..+4 мин** над
реальным таймингом упаковщика. Пример: хвост настройки на 16:03 + смена сырья 15 → окно кончается в
16:18 ≤ потолка 16:20 (в бюджете), а бейдж дня показывает 462 = 460 + 2.

Чем этот «+2» **не** является:
- **Не** фантомный лидер `BETWEEN_CUTS`. Лидер начисляется ТОЛЬКО при реальных проходах: у `setup-only`
  сегмента (0 проходов) `cutLeaderRuns = 0`, лишних 2 мин нет (#4021, [[atex-pp-setup-only-no-leader]]).
- **Не** недоделённая настройка хвоста. Хвост setup-only делится по потолку нахлёста настройки: в дне N
  остаётся влезающее подмножество (у `cutEnd` — смена сырья), остаток (ножи) уходит на продолжение (#4026/#4111).

Совсем убрать превышение бейджа — **отдельное решение по политике** (кандидат на следующий шаг), одно из:
1. **Считать бейдж по часам окна** (`finishMin − dayStart − обед`, дробно), а не суммой округлённых
   «Длительностей» — тогда бейдж совпадёт с реальным таймингом упаковщика (в 460).
2. **Паковать с запасом на округление** — уменьшить рабочий бюджет упаковщика на ожидаемую сумму
   `ceil`-округлений (~n_резок × 0.5), чтобы даже округлённый бейдж не вылезал за 460.

До принятия такой политики допуск **бейдж ≤ бюджет + ~4** считается нормой (артефакт отображения),
а не признаком перегруженного дня.

### `planStart` пишет splitMachineQueue; computeCutSetupUpdates — только тайминг (#3853/#3855)

Обе РМ рисуют **окно** резки `setup + (намотка+лидер)` и ставят его по сохранённому
`planStart`. Источник `planStart` — **`splitMachineQueue`** (через `planCutOperations` →
`applySplitPlan`): только он знает дробление по дням, нахлёст настройки в хвост дня (#3805,
#3635 п.5), обед, переполнение, gap-fill. Сохранённые `Наладка ножей` / `Сырьё/намотка` /
`Резка и Лидер` пишет **`computeCutSetupUpdates`** (из `persistCutSetupColumns` после
генерации/пересборки) — он `planStart` НЕ трогает.

Две вещи, ломавшие совпадение Ганта и страницы:

1. **Штамп `planStart` с секундами.** `windowStartMin` дробный → штамп нёс `:SS`; Гант обрезал
   вниз, страница округляла вверх → ±1–2 мин. Фикс — `scheduleStartTimestamp` снапит штамп к
   **целой минуте, округляя ВВЕРХ** (единственная точка перевода `windowStartMin` → Unix-штамп).
2. **Расхождение настройки ПЕРВОЙ резки станка.** `splitMachineQueue` брал «ножи с нуля»
   (`firstCutSetup`), а окно — переналадку от реальной заправки (`prev_cut_setup` → `carryPrevCut`).
   Фикс #3853: генерация считает первую резку переналадкой от заправки (`carryPrevSetup` из
   `prevSetupBySlitter`) — ровно как окно в `computeCutSetupUpdates`.

> **Отменён подход #3855** «пересчитывать `planStart` по сохранённым окнам в
> `computeCutSetupUpdates` (`derivePlanStartTimestamps`)»: он наивно паковал каждый сегмент по
> полному окну `ножи+сырьё+резка` от 08:00, ИГНОРИРУЯ нахлёст настройки в хвост дня (#3805) и
> разрыв по дням (#3635 п.5), и перетирал корректный `planStart` от `splitMachineQueue` →
> setup-only-сегмент клал «и ножи, И сырьё» в один день с нахлёстом. `planStart` считает только
> `splitMachineQueue`; РМ читают сохранённое и **не пересчитывают на лету** (#3846).

> Инвариант: `planStart` пишет `splitMachineQueue` (со снапом к целой минуте вверх);
> `computeCutSetupUpdates` пишет ТОЛЬКО тайминг. Разрыв/перекрытие/±минуты — баг расписания
> (`splitMachineQueue`/окна), а не отдельного пересчёта старта.

## Особые правила

### Короткая фольга ≤ 120 минут

Бизнес-правило согласовано: если суммарное время резки фольги не превышает 120 минут, такую
фольгу можно ставить не в конец дня. Она не успевает сильно загрязнить станок.

Сейчас это **требование, а не код**: `orderCuts` отделяет любую `isFoil` в конец очереди, а
`splitMachineQueue` и дневная сортировка также считают любую фольгу фольгой. Для реализации
нужно ввести признак короткой фольги и трактовать ее как обычную резку:

- в `orderCuts` класть короткую фольгу в `rest`, а не в `foil`;
- в `selectByConfig` считать ее `isFoil = 0`;
- в сортировке `(день, isFoil, planStart)` не уводить ее в хвост;
- длинную фольгу оставлять в конце дня как сейчас.

### Втулка 0.5" и узкие риббоны (#3812, #3872)

Реализовано — полное описание в [Шаг 2.0](#шаг-20-втулочные-полосы-05--110-мм-3812). Кратко:

- риббоны на втулке 0.5" у́же 55 мм не производятся (позиция в `skipped`);
- фактическая ширина 55-57 мм режется как 55 мм;
- при втулке 110 мм для ширин 55-57 мм добавляют две полосы 110 мм, для 63-64 мм — одну;
- ширины свыше 70 мм идут по основному правилу для втулок 1";
- #3872: полосы 110 мм привязываются к уже заказанным позициям 110 мм того же заказа
  (обеспечение на произведённое); не нашлось — полосы синтетические.

### Отпуск станка (#3764, #3876, #3881, #3883)

«Отпуск» — подчинённая станку таблица окон простоя `[{ начало, окончание }]`
(`downtimesBySlitter`). В эти окна станок **не работает** (нет сырья и ножей).

**Частичный отпуск vs весь день (#3883).** Окно отпуска задаётся с ВРЕМЕНЕМ (datetime). Разделяем
два уровня: (а) **расписание** (#3764) сдвигает резки за окно ПО ВРЕМЕНИ — короткое окно (напр.
2 часа 08:00–10:00) лишь задерживает старт дня; (б) пометка «станок недоступен ВЕСЬ день»
(`slitterOnVacationDay` → `slitterDownOnDay`) — для выбора станка/балансировки/переноса/сброса
заправки — даётся ТОЛЬКО если окна отпуска **полностью покрывают рабочее окно резки**
`[startMin; cutEndMin]` этого дня. Частичный отпуск станок на день НЕ блокирует (он работает
остаток дня). Раньше любое пересечение суток считалось «весь день в отпуске», и 2-часовое окно
исключало станок из планирования совсем (Гант пустой во все дни).

- **Расписание (#3764).** Уже построенные размещения сдвигаются по времени за окно отпуска
  (`shiftPlacementsPastDowntime`), чтобы в отпуск ничего не наматывалось.
- **Загруженность по дате окончания (#3881, #3921).** При **выравнивании загрузки** «загруженность»
  станка считается как ДАТА ОКОНЧАНИЯ последней задачи (span целых дней + дробный `endPos` с учётом
  отпуска, #3921 — чтобы хвост за отпуском стекал на свободные станки), а не как сумма минут
  (см. [Шаг 3.1-bis](#шаг-31-bis-выравнивание-загрузки-станков-rebalanceslitterload-3848)).
  Станок с отпуском заканчивает позже — его не грузят на послеотпускное время, пока другие
  простаивают; станок в отпуске без задач после него «занят» до конца отпуска.
- **Не ставить задание на станок в его отпуск (#3876).** При **выборе станка**
  (`chooseSlitterBySetup`) станки, у которых в день резки отпуск, исключаются
  (`unavailableSlitterIds`; все в отпуске → откат, чтобы не потерять резку). При
  **выравнивании загрузки** (`rebalanceSlitterLoad`) задание не переносится на станок,
  у которого в день этого задания отпуск (`opts.slitterDayBlocked`). **Ручной перенос**
  (🗓 «Перенести задание») на станок, у которого в выбранный день отпуск, **блокируется**
  с сообщением. Признак «отпуск на весь день» — `slitterDownOnDay`: закрытые окна (с «Окончанием»)
  ПОЛНОСТЬЮ покрывают рабочее окно `[startMin; cutEndMin]` (#3883); окно без «Окончания» не
  блокирует, как и в расписании.
- **Заправка после отпуска (#3876, смягчено #3898).** Станок на время **длинного** отпуска
  без сырья и ножей: у станка, который на отпуске в день базы плана, заправка (`prev_cut_setup`)
  **обнуляется** (`planningPrevSetupBySlitter` → `longVacationOnDay`), поэтому **первая резка
  после отпуска** считает ПОЛНУЮ настройку — смена сырья + ножи с нуля (а не наследует заправку,
  бывшую до отпуска). Один источник и для плана (`splitMachineQueue`), и для хранимых колонок
  (`computeCutSetupUpdates`), и для модалки тайминга — времена совпадают.
- **Короткий отпуск НЕ сбрасывает заправку (#3898).** Отпуск длиной **не более
  `DOWNTIME_KEEP_SETUP_MAX_DAYS` (= 2) КАЛЕНДАРНЫХ дней** заправку сохраняет — после короткого
  простоя станок остаётся со своим сырьём/ножами, и первая резка не пересчитывает настройку с
  нуля. Длиннее порога → сброс (полная настройка). Длину окна меряет `downtimeSpanDays`
  (`vacationSpanDaysOnDay` — для дня базы); отпуск = одна запись окна `[начало; окончание]`.
  Порог вынесен в константу `DOWNTIME_KEEP_SETUP_MAX_DAYS` (легко менять). Частичный отпуск
  (#3883, не весь рабочий день) и так не считается за простой (`slitterOnVacationDay = false`).
  Сброс заправки в середине очереди (станок уходит в отпуск и возвращается уже внутри окна
  планирования) пока не реализован — учитывается отпуск на день базы плана.

## Целевой алгоритм (#3989): фундамент стоимости и качества

ТЗ на новую вставочную раскладку — [`docs/atex_planning_tz.md`](atex_planning_tz.md)
(человеческим языком, согласовано заказчиком). Первый шаг реализации (#3989, Фаза 1) —
**чистые функции-фундамент** в `production-planning.js`, экспортированы в `planning` и покрыты
`experiments/atex-production-planning-3989.test.js`. Движок раскладки пока прежний (жадный);
функции аддитивны и ещё не подключены к генерации:

- `planWeight(settings, key)` — вес штрафа/лимит из «Настройки» (ATEH) с дефолтами ТЗ §14
  (`KNIVES_CHANGE`/`KNIVES_INCREASE`/`MATERIAL_CHANGE`/`FOIL_NOTEND`/`DEADLINE`/`EXACT_DEADLINE`/…).
  **#4059:** эти значения доезжают и до жадного упаковщика — `makePlanningOptions(strategy, times,
  daySettings)` кладёт ключи «Настройки» плоско в `opts.weights`, откуда их читает `planWeight` в
  `splitMachineQueue`/`orderCuts` (без этого срок/переналадка брали бы только `PLAN_WEIGHT_DEFAULTS`,
  игнорируя, например, `DEADLINE_COST_MN=200` из таблицы).
- `transitionCost(prev, next, ctx)` / `insertionCost(prev, slot, next, …)` — стоимость перехода
  и вставки слота = **вес + «качество»** по ТЗ §8: ножи (смена ↓полос = `KNIVES_CHANGE`,
  ↑полос = `KNIVES_INCREASE`) + `stripPrefixQuality` (доля совпавших с начала ширин), сырьё,

- `planQuality(slots, opts)` — качество плана (ТЗ §13 + комментарий #3985): фактические

  границы** (каждая конфигурация ножей и каждое сырьё — по 1 разу), с учётом первого задания

**Фаза 2 (#3992)** — настройки нового движка: `resolveOverworkLimits` (захлёст по ключам `_MN`
с откатом на старые имена), `resolveDayDurationMin` (`DAY_DURATION_MN`), `intraDayBreaks` (обед
+ два перерыва `FIRST_INTERVAL`/`SECCOND_INTERVAL`, прозрачные для плана).

**Фаза 3** — панель качества плана в очереди (`renderQueue`): показывает над днями строку
«Качество плана · переналадки N (M мин) · ножи · смены сырья · идеал N₀ (M₀) · избыток ±Δ ·
комбинаций» (окно `[С;По]`; горизонт `[С;конец]` — в подсказке). Панель не критична: ошибка
расчёта глушится, очередь рисуется без неё.

- **Факт** (переналадки/ножи/смены сырья) — `storedSetupTotals`: **суммы хранимых колонок наладки
  задания** («Наладка ножей, мин» / «Сырье/намотка, мин», #3698), ровно те, что суммирует отчёт
  **«Комбинации по всем позициям»**, — панель сходится с отчётом по построению (#4156). Настройка
  ножей плоская (#3600), у первого задания станка смены сырья нет (`firstSetupParts`). Нет колонок
  #3698 (`hasStored=false`) — панель **выводит ошибку** (консоль + тост + красная плашка), а НЕ
  молча откатывается на оценку по весам: пустые/отсутствующие колонки наладки — ошибка
  конфигурации/данных (ТЗ §14/#4059), не повод показать неверный факт.
- **Трасса «где именно пусто» (#4188).** `storedSetupTotals` возвращает ещё и `emptyWindow`/`emptyAll` —
  перечень заданий с **незаписанными** колонками наладки (`{ id, dayKey, slitter, plannedRuns }`;
  пусто ≠ хранимый «0» = «наладки нет»). При ошибке (`hasStored=false`) `renderQueue` печатает их ID
  и **контекст операции** (`self._ppOp`), при частичной пустоте (`hasStored=true`, но часть заданий
  пуста → суммы **занижены**) — под тумблером `PP_TRACE`. **Ошибка после «Сгенерировать» ТРАНЗИТНА:**
  `runGenerateCuts` делает `reload()`+`render()` сразу после создания записей, но **до**
  `autoSequenceQueue` — а колонки наладки пишет именно последующий
  `autoSequenceQueue` → `applySplitPlan` → `persistCutSetupColumns`; свежесозданные задания ещё без
  колонок (`операция=runGenerateCuts` в трассе), финальный рендер после пересборки ошибки уже не даёт.
- **Идеал / комбинации / окно** — `planQualityView` (маппит `this.cuts` через `mapCutRecord` в
  слоты `planQuality`). **Избыток** = факт окна − идеал окна.
- `planQuality` (веса штрафов, инкремент полос, смена сырья первого задания) остаётся **объективу
  «Упорядочить»** (`planChangeoverMin` = `planQuality.all.changeoverMin`), а не показу оператору.

## Возможный оптимизатор на 3 дня

Сейчас план строится жадно: раскладка → выбор станка → очередь → расписание. Это быстро и
понятно, но не всегда дает лучший глобальный план. Следующий шаг — улучшающий поиск на
скользящем окне примерно 3 дня.

### Целевая функция

```text
Cost(plan) = Σ(вес_фактора × нарушение_фактора(plan))
```

Жесткие ограничения из раздела «Ограничения алгоритма» фильтруют недопустимые планы. Мягкие
факторы из матрицы «Цена нарушения правил» дают штраф.

Примеры штрафов:

- срок: `max(0, день_резки − dueDay) × W1`;
- фольга: длинная фольга не в конце дня × `W2`;
- переналадки: сумма `changeoverCost(prev, next)`;
- баланс: отклонение загрузки станка от средней или факт `overflow`;
- дробление: число продолжений и setup-only сегментов.

### Соседние ходы

Оптимизатор может пробовать локальные перестановки и считать `ΔCost`:

1. поменять две резки местами внутри дня одного станка;
2. перенести резку на другой день того же станка;
3. перенести резку на другой станок того же дня;
4. поменять резки между станками;
5. сдвинуть длинную фольгу в конец или короткую фольгу раньше.

### Поиск и применение

Базовый вариант — hill-climbing: применять лучший ход с отрицательным `ΔCost`, пока план
улучшается. Если потребуется выходить из локальных минимумов, можно добавить simulated
annealing или tabu search с ограничением по времени/итерациям.

Результат должен быть списком обновлений в формате `planCutOperations`: резка, новый станок,
новый день, новая очередность. Применять можно через превью («минус N минут переналадок,
минус K опозданий») или отдельную кнопку «Оптимизировать на 3 дня».

## Глоссарий

| Термин | Значение |
|---|---|
| Джамбо | Исходный рулон сырья. Ширина джамбо — максимальная суммарная ширина полос одной резки. |
| Прогон (`run`) | Один проход через станок. Число прогонов примерно равно `qty / количество полос данной ширины`. |
| Скрап (`scrap`) | Полосы, нарезанные сверх заказа по незапасным ширинам. Это перепроизводство, которое некуда деть. |
| Отход (`waste`) | Незанятая ширина джамбо × прогоны. Идет в технологический обрезок. |
| Лидер | Лента для заправки слиттера перед следующей резкой. Учитывается после намотки текущей резки. |
| Ходовые ширины | Ширины из «Максимального запаса», которыми можно добирать остаток джамбо в складской запас. |
| `isFoil` | Признак фольги: грязное сырье, требующее постановки в конец дня, кроме планируемого исключения для короткой фольги. |
| `sleeveId` | Диаметр втулки. Все рулоны одной резки должны иметь один `sleeveId`. |
| `dueKey` | Числовой ключ срока изготовления (`ГГГГММДД`). Задаёт порядок размещения по дням (EDD: раньше срок → раньше день; постановка после срока штрафуется — #4050/#4059), красит строку очереди (`dueColorClass`) и кластеризует позиции при генерации (окно срока, Фаза 1). |
| `continuationSignature` | Ключ идентичности резки для сборки цепочки продолжений при многодневном разбиении. |
| `fixed` | Признак «Зафиксировано». Закрепляет день резки и защищает от удаления, но не запрещает перестановку внутри дня. |

# Алгоритм планирования производства (слиттер, atex)

Описание алгоритма страницы **«Планирование производства»**
(`templates/atex/production-planning.html`, логика — `download/atex/js/production-planning.js`).
Документ объясняет: как формируется очередь резок, как считается время переналадок
и как строится расписание по сменам; а также где брать «журнал» и «сводную» по таймингу.

> **Источник истины по алгоритму планирования (#3717).** ВСЕГДА сверяйся с этим
> документом ПЕРЕД правкой логики очереди/расписания/переналадки/фольги в
> `download/atex/js/production-planning.js` и ОБНОВЛЯЙ его в ТОМ ЖЕ PR при изменении
> поведения. Документ и код держим в синхроне.

> Изменение #3600: **любая смена ножей теперь стоит фиксированные 30 минут** «на всё
> вместе», независимо от числа переставленных ножей (раньше #3472 — `KNIFE_MOVE × число
> перестановок`). Смена сырья/намотки считается **отдельно** (своя операция).

## 1. Вход

Очередь — массив резок (`cut`), каждая со свойствами (упрощённо):

| поле | смысл |
|---|---|
| `materialId` | сырьё |
| `winding` (`IN`/`OUT`) | намотка |
| `batchId` | партия сырья |
| `knifeCount`, `knifeWidths[]` | набор ножей (число и ширины полос) |
| `rollerWidth` | ширина ролика |
| `isFoil` | признак «Фольга» (планируется в конец) |
| `jumboRemainingM` | остаток джамбо (для «неудобного остатка») |

## 2. Очередь резок — `orderCuts(cuts, weights)`

1. **Фольга — ВСЕГДА в конец дня (критично, #3717).** (Планируемое исключение — короткая фольга
   с временем резки ≤ 120 мин, §7.1; в коде пока нет.) Резки делятся на `rest` и `foil`;
   очередь = `последовательность(rest)` + `последовательность(foil)`. Фольга всегда после
   всего остального. Это правило обязано соблюдаться не только в `orderCuts` (порядок при
   генерации), но и **в по-дневном разбиении `planCutOperations` (preserveOrder)**: там
   сортировка внутри станка идёт по `(день, isFoil, «Очередность»)`, иначе кросс-дневная
   пересборка/посменная упаковка возвращала фольгу в начало/середину дня (`orderCuts`
   делает её последней лишь по ИСХОДНОМУ дню, а `splitMachineQueue` перепаковывает по
   фактическим дням). Любая правка очереди/расписания обязана сохранять фольгу в конце
   каждого дня.
2. **Срок изготовления — ПЕРВИЧНО (EDD, #3815).** Внутри `rest` и `foil` резки группируются
   по «Сроку изготовления» (`c.dueKey` — самый ранний срок обеспечиваемых позиций, YYYYMMDD) по
   возрастанию (`sequenceByDue`): задание с более ранним сроком идёт раньше, чтобы по-дневная
   раскладка поставила его на более ранний день. Переналадка (`sequenceForStrategy`) применяется
   ВНУТРИ одного срока. Резки без срока (`dueKey` не число → `Infinity`) — в последнюю группу;
   если срока нет ни у одной — одна группа = прежнее поведение. Срок проставляется на логические
   резки в `planCutOperations` из `dueKeyByCut` (считается в `autoSequenceQueue` через `cutDueKeys`).
3. **Порядок внутри одного срока** — `sequenceForStrategy`. Критерии (по убыванию приоритета):
   - **SETUP (#3783): минимизация суммарной переналадки — ПЕРВИЧНО** (`greedySequence`).
     Жадная цепочка argmin `changeoverCost` группирует одно сырьё/набор ножей (переход
     внутри группы дешевле смены сырья/ножей) — поэтому сырьё **не идёт вперемешку**.
   - **Тай-брейк «при прочих равных» — число полос (ножей) по убыванию (#3785)** — заложен
     в `startKey` жадной цепочки (`[-knifeCount, rollerWidth, id]`): при равной стоимости
     перехода раньше идёт резка с бо́льшим числом полос.
   - Прежний враппер `byKnifeCountDesc` (#3568) пересортировывал ВСЮ цепочку по числу ножей
     глобально, разбивая группировку сырья и увеличивая переналадку — **убран из SETUP**.
     «Много ножей раньше» (усталость, #3130) остаётся отдельной стратегией `…_FATIGUE`.
   - Стратегии: `PLANNING_STRATEGY_SETUP` (минимум переналадок + полосы↓ тай-брейк),
     `…_FATIGUE` (сложные резки раньше — `fatigueAwareSequence`).
4. Каждой резке проставляется `sequence = 1..N` (вход не мутируется).

`orderedChangeoverCost(cuts)` — суммарная переналадка для очереди после `orderCuts`
(детерминированный минимум, не зависит от исходного порядка).

**Применение порядка (#3792).** Генерация (`generateCuts`/`runGenerateCuts`) пересобирает
очередь видимого диапазона по правилам — `autoSequenceQueue(SETUP, false)`
(`preserveOrder=false`): при вбросе новых заданий очередь упорядочивается заново (группировка
сырья + полосы↓), а не дописывает новые в конец. Кнопка **«Упорядочить»** (`optimizeQueue`)
делает то же по требованию.

**Ручной порядок переживает генерацию только через флаг «Зафиксировано»** (81530). Фиксация
замораживает **только день**:

- зафиксированное задание **остаётся на своём дне** (`dayAnchorByCut`) — пересборка/`gapFill`
  (#3739) не переносит его на другой день и не тянет в хвост более раннего;
- **не разбивается** по дням — один сегмент на своём дне, нахлёст за конец смены допустим
  (`fixedDay` в `splitMachineQueue`);
- **внутри дня** оптимизатор переставлять может: на своём дне фикс берётся раньше свободных
  (перёд дня), переполнение фикса выталкивает свободные на следующий день.

Фикс без «Даты план» (нечем закрепить день) трактуется как свободная резка. Флаг «Зафиксировано»
также «стена» для ручных ↑↓ и защита от удаления (#3508/#3562) — это поведение прежнее.

## 2a. Выбор станка для новой резки — `chooseSlitterBySetup` (#3801)

При генерации каждая новая резка раскладывается на конкретный станок. Цель — **объединять
одинаковую заправку на одном станке и держать загрузку станков ровной** (не сваливать всё на
один, оставляя другой на неполный день).

Для каждого допустимого станка (не в стоп-листе по сырью) считаем, можно ли к нему
**«прицепиться»** без холодной настройки:

- `sameKnives` — на станке уже режется **тот же набор ширин ножей** (`knifeWidthSig`, #3666);
- `sameMaterial` — на станке уже режется **то же сырьё + та же намотка** (резка пойдёт без
  смены сырья);
- `attach` = `sameKnives` **ИЛИ** `sameMaterial` — есть к чему прицепиться.

Выбор (по порядку критериев):

0. **Ёмкость дня — ПЕРВЫЙ критерий (#3830).** `overflow` = рабочие минуты дня станка с этой
   резкой (переналадки + намотки всех заданий дня) **превышают** рабочую ёмкость дня
   (`cutEnd − start − обед` ≈ 450). Станок, где резка **влезает** в день (`overflow` = 0), всегда
   предпочтительнее переполненного (`overflow` = 1) — даже ценой холодной настройки на другом
   станке. Так фольга (общее сырьё «Фольга …», поэтому вся «цепляется» по сырью к одному станку)
   не копится на одном станке за ёмкость дня (≈514 мин при 450), когда у соседнего станок пуст.
   При **равных** `overflow` действует прежняя группировка/балансировка (пп. 1–2).
   Активно только когда ёмкость задана (генерация); без неё (тесты) `overflow` = 0 для всех.
1. **Есть совместимый станок** (`attach` хотя бы у одного) — выбираем среди совместимых,
   сохраняя группировку: тот же набор ножей (#3666) → то же сырьё (#3801) → минимум прироста
   переналадки `delta` (#3268) → аффинность → **наименее загруженный** из равных → меньший id.
   Так одинаковое сырьё/ножи не разносятся по станкам, но между несколькими одинаково
   совместимыми станками новые резки распределяются ровно.
2. **Прицепиться негде** (ни по ножам, ни по сырью — везде холодная настройка) — выбираем
   **наименее загруженный станок** (балансировка), тай-брейк: `delta` → аффинность → id.

Загрузка `loadBySlitterId` — счётчик резок на станок; в генерации он сидируется текущими
резками и инкрементируется по мере раскладки, поэтому новые задания в рамках одной генерации
тоже учитываются в балансе. Рабочие минуты дня для `overflow` (#3830) считаются по группе
станка того дня: намотка каждого задания (`Резка и Лидер`, иначе `Длительность, минут`) плюс
суммарная переналадка очереди.

## 3. Переналадка `prev → next` — `changeoverParts` / `changeoverCost`

Переналадка раскладывается на независимые операции (минуты). У всех резок переналадка
считается относительно предыдущей — в том числе у **первой резки нового дня**: её `prev`
— это последняя резка предыдущего дня (см. §4). **Первая резка очереди станка (#3688):** её
`prev` — это **текущая заправка станка** из отчёта `prev_cut_setup` (последняя по времени
задача станка: сырьё/намотка/набор ножей), обёрнутая в синтетическую «предыдущую резку»
(`carryOverPrevCut`). Поэтому первая резка дня идёт **без переналадки только если на станке
уже стоит та же конфигурация**; если осталось другое (или ничего) — бронируется настройка
ножей и/или смена сырья. Нет данных `prev_cut_setup` → деградация к `firstCutSetup` (ножи с нуля):

- **смена сырья экономится**, если сырьё и намотка совпадают с заправкой (нет `MATERIAL_WINDING`);
- **смена ножей экономится**, если число ножей и интервалы (ширины полос) совпадают (нет `KNIFE`);

иначе соответствующая операция применяется и её время считается обычным `changeoverCost`.

Порядок операций (#3688): **СНАЧАЛА настройка ножей, ПОТОМ смена сырья** (так на станке).

| операция | когда применяется | время |
|---|---|---|
| `KNIFE` | сменился набор ножей (`knifeMoves > 0`) **ИЛИ** сузился ролик | **`KNIFE` фикс. 30 (#3600)** |
| `MATERIAL_WINDING` | сменилось сырьё **ИЛИ** намотка **ИЛИ** партия | `MATERIAL_WINDING` (15) |

- Обе операции **бинарны** (изменилось / нет) и **складываются**:
  - смена сырья → 15;
  - смена ножей → 30;
  - сырьё + ножи → **45** (15 + 30).
- `changeoverCost(prev, next, times)` = сумма минут компонентов.
- До #3600 нож считался как `KNIFE_MOVE(2) × число переставленных ножей`; теперь число
  ножей на время **не влияет** — любая переналадка ножей = 30.

## 4. Расписание по сменам — `buildSchedule(cuts, opts)`

По упорядоченной очереди раскладывает резки во времени:

- **Окно смены** `shiftStartMin … shiftEndMin` (по умолчанию рабочая смена); резки
  пакуются по порядку, не помещается сегодня → перенос на следующий рабочий день.
- **Лидер между резками** `BETWEEN_CUTS` (2 мин, #3401) × число проходов (`Кол-во план`)
  заправляется **в КОНЦЕ каждой резки** (#3688): в `buildSchedule` он не входит в стартовый
  сетап, а добавляется ПОСЛЕ намотки (`sc.leaderMin`) — следующая резка стартует после
  лидера текущей. (В посегментном `splitMachineQueue` лидер по-прежнему распределён по
  проходам — `perPass + leader` — это согласуется по сумме на резку.)
- **Переналадка** перед резкой = `changeoverCost(prev, next)` (§3) — только ножи + смена
  сырья. На границе дня `prev` — последняя резка предыдущего дня; для первой резки очереди
  `prev` — заправка станка из `prev_cut_setup` (#3688, §3).
- **Плавающий обед** (#3342) и **уборка в конце дня** `CLEANUP_SHIFT` (30 мин, #3155).
  Обед — пауза `LUNCH_DURATION` от `LUNCH_START`; рабочая ёмкость дня = окно резки − обед
  (ATEH: 08:00–16:10 = 490 − 40 = **450 мин**).
- **Резка, пересекающая обед, паузит на обед в ходе намотки (#3816).** Обед вставляется
  паузой не только перед резкой, СТАРТУЮЩЕЙ в/после `LUNCH_START`, но и внутри резки, чья
  намотка стартует ДО обеда и идёт через него: финиш такой резки сдвигается на
  `LUNCH_DURATION` (`durationMin` — минуты работы — не меняется). Иначе длинная резка шла
  «сквозь обед»: день заканчивался раньше (≈16:22 вместо ≈17:00), а сумма за день получалась
  как целое окно без вычета обеда (#3816: **502 мин** при ёмкости 450 — зарезервированные под
  обед 40 мин съедал захлёстный проход #3760).
- **Нет «нахлёстного» прохода: день дробится по ёмкости (#3821).** В `splitMachineQueue`
  (ветка `gapFill`) в хвост дня кладутся ТОЛЬКО проходы, целиком влезающие в рабочую ёмкость;
  остаток (включая бывший «нахлёстный» проход #3739/#3760) переносится на следующий день как
  продолжение (дневное дробление #3280). Раньше клали `fittingG + 1` нахлёстный (первый,
  пересекающий конец смены) → для длинного прохода день вылетал за «смену + один шаг наладки»
  (#3821: **483 мин** при максимуме ~480, видно в Диаграмме Ганта — она суммирует минуты
  заданий станка за день). Если в хвост не влезает ни один проход: есть переналадка — кладём её
  в хвост (setup-only, #3635 п.5/#3805), резку на завтра; нет (та же конфигурация) — ничего не
  кладём, резка целиком на завтра (небольшой простой в хвосте допустим). Нахлёстный проход
  остаётся ТОЛЬКО в вырожденном случае — одиночный проход длиннее целого окна (разбить нельзя).
- **Фольга — в конец каждого дня** (см. §2.1, #3717): и в `orderCuts`, и в по-дневном
  `planCutOperations` (preserveOrder) сортировкой `(день, isFoil, «Очередность»)`.
- **Срок изготовления определяет ДЕНЬ (EDD, #3815).** В `splitMachineQueue` (ветка `gapFill`)
  следующее задание дня выбирается `selectByConfig` по ключу `[isFoil, dueKey, переналадка, idx]`:
  сперва не-фольга, затем **более ранний «Срок изготовления»**, затем непрерывность конфигурации
  (минимум переналадки), затем порядок очереди. Так задание с более ранним сроком занимает более
  ранний день, а переналадка определяет порядок ВНУТРИ одного срока. Симптом до #3815: на дне N
  стояла позиция со сроком N+1, а на N+1 — со сроком N (срок не участвовал в выборе дня).
  - **Затягивание по сроку в окно фильтра (EDD vs scope, #3820).** `planCutOperations`
    перепланирует только цепочки, чья ГОЛОВА в окне «Даты плана» `[scopeFromKey; scopeToKey]`
    (#3660 — не трогать чужие даты). Из-за этого задание со сроком 23, «застрявшее» на «Дате
    план» 24, при фильтре `[23; 23]` не попадало в раскладку, и EDD не мог подтянуть его на 23
    (ослабление якоря до дня срока, #3815, работает только для резок В scope). `headInScope`
    дополнительно берёт в окно **НЕзафиксированную** резку, чей `dueKey ≤ scopeToKey` (срок в
    окне или раньше него), но «Дата план» которой стоит **позже** окна — затягиваем просроченное/
    срочное по сроку задание в окно. Задание со сроком **позже** окна и **зафиксированное** (#3508)
    остаются на своих днях (#3660 для чужих дат сохраняется); без `dueKey` окно не расширяется.
  - **Фольга своего срока не вытесняется будущей нефольгой (EDD vs «фольга в конце дня», #3826).**
    Фольга всегда в конце дня (#3717) → в `selectByConfig` ключ начинается с `isFoil`, поэтому
    ЛЮБАЯ нефольга бралась раньше любой фольги. По-дневная раскладка тянула вперёд будущую (срок
    > дня) нефольгу, та съедала хвост дня, и фольга СВОЕГО срока (срок ≤ дня) переливалась на
    следующий день (симптом: фольга со сроком 23 на 24, хотя в дне 23 было место — она не может
    вытеснить нефольгу ВНУТРИ дня, поэтому ей надо защитить место). Фикс (`splitMachineQueue`,
    `gapFill`): если на день приходится фольга своего срока, хвост дня **резервируется под работу
    своего срока** — сперва нефольга своего срока, затем `bestFittingFoil` (крупнейшая фольга
    своего срока, ЦЕЛИКОМ влезающая в остаток, без дробления настройки — иначе намотка-продолжение
    встала бы в НАЧАЛО следующего дня перед нефольгой, нарушив #3717). Будущую нефольгу тянем
    вперёд только пока фольги на дне ещё НЕТ (тогда фольга уедет на след. день в конец — правило не
    нарушается); фольгу крупнее целого дня дробим (заполняя остаток). `dueDayByCut` (день срока от
    базы) передаётся в `splitMachineQueue` для различения «срок ≤ дня» и «тянем вперёд».
- **Нахлёст настройки setup-only-сегмента ограничен (#3805).** Для сегмента НАСТРОЙКИ
  (#3635 п.5: 0 проходов, «настройка в хвосте дня N → намотка с дня N+1») `buildSchedule`
  кладёт в хвост дня N только **минимальное подмножество компонентов настройки**, дотягивающее
  до конца смены (`minOverlapTailSetupMinutes`, как `splitMachineQueue`), а **остаток** настройки
  переносит на продолжение (день N+1). Иначе настройка пересчитывалась как полная переналадка
  (ножи 30 + сырьё 15 = 45) и копилась целиком в хвост дня N — сумма за день вылетала за
  «смену + один шаг наладки» (например 495 мин при максимуме ~480). Сумма настройки по цепочке
  сохраняется (хвост дня N + продолжение дня N+1 = полная переналадка).
  - **Опознание setup-only не зависит от фильтра дат (#3827).** `setupTaskIdSet` помечает
    сегмент настройки (запись «Задание в производство» с «Кол-во план» = 0) по самому признаку
    «0 проходов», **не требуя**, чтобы намотка-продолжение той же цепочки (slitter|материал|
    намотка|ножи) присутствовала в загруженном наборе. Раньше при УЗКОМ фильтре дат продолжение
    (на след. дне) в набор не попадало → сегмент настройки оставался «одиноким», терял признак и
    в расписании считался обычной задачей с **полной** переналадкой в хвосте дня (#3805 не дробил
    её): сумма минут дня в бейдже прыгала в зависимости от ширины фильтра (483 мин при фильтре
    «23», но 467 мин при «23–30»). 0-проходную резку всегда создаёт только планировщик как разрыв
    настройки (`splitMachineQueue`, `setupOnly`) — других источников нет.

**Окно станка резки = `[ножи][смена сырья][намотка][лидер]`** (#3688): старт намотки =
`startMin`; перед ним — переналадка (`setupMin`, ножи+сырьё); после намотки (`finishMin`) —
лидер (`leaderMin` = `BETWEEN_CUTS × проходов`). «Готово» = `finishMin + leaderMin`.

## 5. Времена операций — `DEFAULT_OP_TIMES`

Дефолты (реальные значения берутся из таблицы времён, передаются как `times`):

```js
{ MATERIAL_WINDING: 15, KNIFE: 30, KNIFE_MOVE: 2, BETWEEN_CUTS: 2, CLEANUP_SHIFT: 30 }
```

После #3600 нож использует **`KNIFE`** (30, фикс.); `KNIFE_MOVE` в стоимости
переналадки больше **не участвует** (оставлен в таблице для совместимости).

## 6. Журнал и сводная по таймингу

- **Журнал перехода** (порезочно): `changeoverParts(prev, next, times)` —
  `[{ code, label, minutes }]` только применившихся операций (СНАЧАЛА «смена ножей /
  сужение ролика», ПОТОМ «смена сырья / намотки / партии» — #3688). Стартовый setup резки —
  `setupBreakdown(prev, next, times)` = только переналадка (ножи + сырьё); лидер (#3688)
  показывается отдельной строкой В КОНЦЕ резки, не в setup.
- **Сводная по очереди:** `orderedChangeoverCost(cuts)` — суммарные минуты переналадок;
  `buildSchedule` даёт по дню/слиттеру итоги (время резки, переналадок, всего, окно
  смены, перенос на следующий день).

### Тайминг резки — порядок окна (#3688)

Модалка «Тайминг окна» показывает операции в порядке станка: **настройка ножей → смена
сырья → намотка (резка) → лидер между резками → готово**. Переналадка (ножи/сырьё, §3)
идёт ПЕРЕД намоткой и только если конфигурация отличается от заправки/предыдущей резки;
лидер (`BETWEEN_CUTS × проходов`) — ПОСЛЕ намотки (в конце резки), «готово» — после лидера.

Пример: первая резка дня — фольга, на станке вчера осталось другое (нужны и ножи, и смена
сырья). Метраж прохода 122 м, 20 проходов, `WIND_FOIL_305 = 4 мин`, лидер 2 × 20 = 40 мин.
**Намотка фольги — БЛОКАМИ (#3742): `ceil(метраж / 305) × 4`, не пропорция.** Короткий проход
(122 м < 305 м) всё равно стоит полную норму — 4 мин (не 1.6); 400 м → 8 мин (начат второй блок):

```
Тайминг резки · Фольга горячего тиснения MB · намотка IN
Метраж прохода: 122 м
Плановых проходов: 20
Намотка 1 прохода: 4 мин
Норма намотки: WIND_FOIL_305=4 мин

Тайминг окна:
08:00 · смена ножей / сужение ролика — 30 мин
08:30 · смена сырья / намотки / партии — 15 мин
08:45 · Итого резка: 4 * 20 = 80 мин
10:05 · лидер между резками — 40 мин
10:45 · готово
```

## 7. Исключения и особые правила

Базовые правила (§2–§4) знают исключения. Статус каждого помечен явно: **[в коде]** —
реализовано, **[требование]** — согласовано как правило, но в коде пока нет (планировщик
ведёт себя по базовому правилу), **[уточняется]** — требования ещё не финализированы.

### 7.1. Короткая фольга может НЕ идти в конец дня **[требование]**

Базовое правило — «фольга всегда в конец дня» (§2.1/§4, #3717): фольга медленная и сильно
загрязняет станок, поэтому её ставят последней, чтобы не чистить станок перед обычными резками.

**Исключение:** если суммарное **время резки фольги ≤ 120 мин**, она *не успевает сильно
загрязнить* станок — такую фольгу разрешено ставить **не в конец дня** (наравне с обычными
резками, по сроку/переналадке). Порог 120 мин — настраиваемый параметр (кандидат в «Настройку»,
рядом с `LUNCH_*`/`TOTAL_INTERVALS`).

> Время резки фольги считается по блочной норме намотки `ceil(метраж / 305) × WIND_FOIL_305`
> (#3742, §6) на все проходы плюс лидер — то же число, что уже выводится в тайминге резки.

**В коде сейчас:** фольга идёт в конец дня ВСЕГДА (§2.1, §4: ключ `selectByConfig` начинается с
`isFoil`; `orderCuts` отделяет `foil` в конец; резерв хвоста дня #3826). Чтобы внести исключение,
понадобится: признак «короткая фольга» (`isFoil && cutCutMinutes ≤ FOIL_INLINE_MAX_MIN`)
**не считать** фольгой для целей упорядочивания — то есть трактовать её как обычную резку и в
`orderCuts` (раздел `rest`), и в `selectByConfig` (ключ `isFoil = 0`), и в по-дневной сортировке
`(день, isFoil, «Очередность»)`. Длинная фольга (> порога) остаётся в конце дня по-прежнему.

### 7.2. Втулка 0.5" и узкие риббоны (#3812) **[уточняется]**

Для втулок 0.5" действует своя размерная сетка и правила постановки дополнительных полос
(см. issue #3812). Кратко из требований:

- риббоны на втулке 0.5" **уже 55 мм не производим**;
- фактическая ширина 55–57 мм → режется как 55 мм; при втулке 110 мм в резку ставят **2 полосы
  110 мм** (напр. `55 мм × 12 + 110 мм × 2`);
- ширина 63–64 мм при втулке 110 мм → **1 полоса 110 мм** (напр. `64 мм × 12 + 110 мм × 1`);
- свыше 70 мм — по основному правилу для втулок 1".

**Делать пока не нужно** — требования уточняются. Раздел — заглушка, чтобы правило не потерялось;
при финализации перенести конкретику в §2/раскладку (`cut-layout`) и снять пометку **[уточняется]**.

## 8. Матрица критичности факторов и цена нарушения

Планирование — это компромисс между несколькими целями. Ниже — все факторы, упоминавшиеся в
тикетах, с типом, ценой нарушения и **настраиваемым весом**. Веса — стартовые ориентиры (шкала
0–100, чем больше — тем дороже нарушать); их предполагается править, чтобы целевая функция
оптимизатора (§9) считала «штраф плана» так, как нужно цеху.

### 8.1. Жёсткие ограничения (нарушать нельзя — фильтруют допустимые расстановки)

| Ограничение | Что значит | Где в коде |
|---|---|---|
| Стоп-лист сырья по станку | резку нельзя ставить на станок, где её сырьё запрещено | `isMaterialBlocked` (§2a) |
| «Зафиксировано» (день) | задание оператора остаётся на своём дне, не дробится | `fixedDay` (§2, #3508/#3562) |
| Ёмкость дня / окно смены | резка+переналадка не выходят за рабочее окно (обед, уборка) | `splitMachineQueue`/`buildSchedule` (§4, #3821/#3342/#3155) |
| Наличие партии сырья (FIFO) | без активной партии с остатком резку не создаём | `pickBatchFIFOForRun` (#3453/#3120) |

Эти условия — не «штраф», а **фильтр**: расстановки, их нарушающие, недопустимы и в целевую
функцию не попадают (либо имеют ∞-штраф).

### 8.2. Мягкие цели (минимизируем взвешенный штраф)

| # | Фактор | Цена нарушения (натуральная) | Текущая обработка | Тикеты | Вес\* |
|---|---|---|---|---|---|
| 1 | **Срок изготовления (EDD)** | срыв срока: задание позже `dueKey` → штраф за каждый день опоздания | первичный ключ очереди и выбора дня | #3815/#3820/#3826 | **100 / день** |
| 2 | **Фольга в конце дня** | загрязнение станка → внеплановая чистка перед обычными резками (искл. §7.1: ≤120 мин) | `isFoil` в начало ключа, резерв хвоста дня | #3717/#3826 | **80** |
| 3 | **Переналадка ножей** | смена набора ножей/сужение ролика = `KNIFE` 30 мин потери | минимизируется (`changeoverCost`, группировка) | #3600/#3666 | **30** (= мин) |
| 4 | **Переналадка сырья/намотки** | смена сырья/намотки/партии = `MATERIAL_WINDING` 15 мин | минимизируется (`changeoverCost`) | #3600 | **15** (= мин) |
| 5 | **Равномерная загрузка станков** | один станок переполнен, другой простаивает | `overflow`/`load` при выборе станка | #3801/#3830 | **20** |
| 6 | **Группировка одного сырья** | то же сырьё/ножи разнесены по станкам/дням → лишние переналадки | `attach`/`sameKnives`/`sameMaterial` | #3666/#3801/#3783 | **25** |
| 7 | **Убывание числа полос («усталость»)** | при прочих равных резка с бо́льшим числом полос идёт раньше (свежий оператор) | тай-брейк очереди / стратегия `…_FATIGUE` | #3785/#3568/#3130 | **10** |
| 8 | **Дробление резки по дням** | длинная резка разрезается на дни → доп. простой/настройка в хвосте | `splitMachineQueue` (#3280), `setup-only` хвост | #3280/#3635/#3805 | **10** |
| 9 | **Лидер между резками** | каждый проход цуга несёт лидер `BETWEEN_CUTS × проходов` | в стоимости прохода | #3401/#3688 | **2** (= мин) |

\* **Вес** — стартовый множитель штрафа на общей шкале. Для факторов в минутах (3, 4, 9)
натуральная цена уже в минутах потери ёмкости; для остальных (срок, фольга, баланс) вес
переводит нарушение в ту же «валюту штрафа». **Эти числа — заготовка под правку.** Калибровать
их удобно через целевую функцию оптимизатора (§9): подобрать так, чтобы на реальных планах
ранжирование вариантов совпадало с решением технолога.

**Приоритет по умолчанию (как сейчас в коде):** срок (1) → фольга-в-конец (2) → минимум
переналадки (3,4,6) → баланс станков (5) → полосы↓ (7). Матрица позволяет менять этот порядок
весами, не переписывая правила.

## 9. Возможный алгоритм оптимизации плана (эскиз, код не меняем)

Сейчас план строится **жадно**: очередь упорядочивается по правилам (§2), затем по-дневная
раскладка (§4) и выбор станка (§2a) разово назначают каждой резке станок/день/позицию. Жадный
проход даёт хороший, но не обязательно лучший план — нет шага, который бы *оглянулся на весь
горизонт и поискал выгодные перестановки*. Ниже — эскиз такого шага (улучшающий поиск на окне
~3 дня). **Это описание, а не реализация.**

### 9.1. Целевая функция (штраф плана)

`Cost(plan) = Σ_по факторам ( вес_фактора × нарушение_фактора(plan) )` — взвешенная сумма из
матрицы §8 по всем станкам/дням окна:

- срок: `Σ max(0, день_резки − dueDay) × W1` (опоздание в днях);
- фольга: `Σ (фольга не в конце дня и время резки > порога §7.1) × W2`;
- переналадки: `Σ changeoverCost(prev, next)` (ножи `W3`, сырьё `W4`) по очереди каждого станка;
- баланс: `Σ_станки |загрузка_станка_дня − средняя| × W5` (или штраф за `overflow`);
- группировка/полосы/дробление/лидер: аналогично §8.

Жёсткие ограничения (§8.1) — как фильтр допустимости: недопустимый план не рассматривается.

### 9.2. Окно и состояние

- **Горизонт** — скользящее окно `[день; день+2]` (3 дня), как и фильтр «Дата плана»; за окно
  не выходим (#3660 — чужие даты не трогаем), зафиксированные задания (#3508) — неподвижные «стены».
- **Состояние** — текущее назначение `резка → (станок, день, позиция в очереди дня)`; берётся из
  готового плана (после §2/§4) как стартовая точка.

### 9.3. Множество ходов (соседи)

Для каждого кандидата считаем `ΔCost` (инкрементально, не пересчитывая весь план):

1. **Переставить две резки** внутри дня одного станка (меняет переналадки/полосы);
2. **Перенести резку на другой день** того же станка (меняет срок/баланс/дробление) — уже есть
   ручной аналог (#3602, «🗓»);
3. **Перенести резку на другой станок** того же дня (меняет баланс/группировку) — ручной аналог
   (#3669, «Слиттер»);
4. **Поменять резки между станками** (своп) — разгрузить переполненный за счёт пустого (#3830);
5. **Сдвинуть фольгу** в конец дня / вынести короткую фольгу вперёд (§7.1).

### 9.4. Поиск

- **Hill-climbing:** на каждом шаге берём ход с наибольшим **отрицательным** `ΔCost` (сильнее всех
  улучшает) и применяем; повторяем, пока есть улучшающие ходы (локальный минимум). Дёшево, но
  застревает в локальных оптимумах.
- **Защита от локальных минимумов** (по необходимости): *simulated annealing* (иногда принимаем
  ухудшающий ход с убывающей вероятностью) либо *tabu search* (запрет недавно отменённых ходов),
  чтобы выходить из «ям». Бюджет — фикс. число итераций / время.
- **Инкрементальность:** `ΔCost` хода затрагивает 1–2 станка/дня → пересчёт локальный, весь план
  не трогаем; за счёт этого окно в 3 дня перебирается быстро.

### 9.5. Выход и применение

Результат — список перестановок `[{ резка, новый станок, новый день, новая «Очередность» }]`,
в формате `planCutOperations` (`updates`). Применять можно:

- **превью** (показать «что улучшится: −N мин переналадки, −K срывов срока») и спросить оператора;
- либо **автоприменение** по кнопке «Оптимизировать на 3 дня», уважая `fixed`/scope (#3660/#3508).

**Когда полезно:** жадная раскладка локально оптимальна по переналадке, но может проигрывать по
сроку/балансу глобально (напр. фольга своего срока на одном станке мешает обычным, а своп со
свободным станком убрал бы и опоздание, и переполнение). Улучшающий поиск ровно эти случаи и ловит.

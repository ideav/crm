# Алгоритм планирования производства

_Слиттер · atex · `production-planning.js` + `cut-layout.js`_

Документ описывает полный конвейер формирования производственных заданий: от списка
позиций заказов до раскладки полос, выбора станка, очереди резок и расписания по дням.
Цель документа — объяснить алгоритм человеку: что делает система, какие правила нельзя
нарушать, где есть компромиссы и какие части кода отвечают за каждый шаг.

> **Источник истины по алгоритму планирования (#3717).** Перед правкой логики очереди,
> раскладки, расписания, переналадки или фольги сверяйся с этим документом и обновляй его
> в том же PR. Код: `download/atex/js/production-planning.js`,
> `download/atex/js/cut-layout.js`.

> **Важное изменение #3600.** Любая смена ножей стоит фиксированные **30 минут** на весь
> набор, независимо от числа переставленных ножей. Смена сырья/намотки считается отдельно
> и стоит **15 минут** по умолчанию.

## Общий конвейер

Все шаги идут от входных данных до записи плана в базу.

| # | Функция | Что делает |
|---|---|---|
| 1 | `positions_list` | Отчет с позициями заказов: сырье, ширина, количество, срок, втулка, направление намотки, длина прогона. |
| 2 | `rowsToGenPositions` | Нормализует строки отчета: числа → числа, пустые значения → `0`/`''`, считает `dueKey` (`ГГГГММДД`), проставляет `isFoil`. |
| 3 | `groupPositionsByPlanningProfile` | Группирует позиции по профилю: сырье + намотка + длина прогона + лидер + втулка. Разные профили физически не смешиваются в одной резке. |
| 4 | `layoutPositionGroups` | Разбивает профиль по `sleeveId`, чтобы в одной резке была одна втулка. |
| 5 | `dueWindowGroups` | Объединяет близкие сроки в окно 3 дня. Срочные позиции обрабатываются раньше бессрочных. |
| 6 | `planLayouts` | Строит раскладки полос: seed «1 заказ = 1 резка», затем выгодные слияния по B&B. |
| 7 | `capStockToHeadroom` | Обрезает добор «впрок» по лимиту максимального запаса. Ширины сверх лимита убираются из раскладки. |
| 8 | `chooseSlitterBySetup` | Выбирает станок: стоп-лист сырья, вместимость дня, совместимость заправки, баланс загрузки. |
| 9 | `orderCuts` | Упорядочивает резки станка: срок, фольга, блоки сырья, минимум переналадки, тай-брейки. |
| 10 | `buildSchedule` / `planCutOperations` | Раскладывает очередь по сменам и дням, дробит длинные резки, создает/обновляет продолжения. |

## Ограничения алгоритма

Бизнес-ограничения задает технология. Технические ограничения сужают перебор, чтобы план был
расчетным и удобным оператору. В таблицах ниже статус означает:

- **[в коде]** — правило реализовано;
- **[требование]** — правило согласовано, но код пока ведет себя по старому базовому правилу;
- **[уточняется]** — требование пока не финализировано.

### Бизнес-ограничения

| Ограничение | Статус | Что ограничивает / зачем |
|---|---|---|
| Одна втулка на резку | [в коде] | Все рулоны одной резки надеваются на одну втулку. Позиции с разным `sleeveId` идут в разные резки. |
| Одно направление намотки | [в коде] | `IN` и `OUT` не смешиваются в одной резке: это физическое ограничение станка. |
| Одна длина прогона | [в коде] | Все полосы одной резки наматываются за один прогон, поэтому длина у всех позиций должна совпадать. |
| Один лидер на резку | [в коде] | Резка заправляется одним лидером. Позиции с разным лидером идут в разные резки. |
| Ширина полос ≤ ширины джамбо | [в коде] | Позиция шире джамбо не планируется и попадает в `skipped` с причиной «шире джамбо». |
| Отход в пределах допуска | [в коде] | Остаток ширины джамбо должен быть в пределах `tolerance`, иначе алгоритм ищет добор ходовыми ширинами. |
| Стоп-лист сырья по станку | [в коде] | Сырье нельзя назначить на станок, где оно запрещено. Например, фольга может быть разрешена только на части станков через справочник. |
| Фольга в конце дня | [в коде] | Фольга загрязняет станок, поэтому длинная фольга должна стоять после обычных резок каждого дня. |
| Короткая фольга ≤ 120 минут может идти не в конец дня | [требование] | Если суммарное время резки фольги не превышает 120 минут, она не успевает сильно загрязнить станок. В текущем коде любая фольга пока все равно считается фольгой и ставится в конец дня. |
| Втулка 0.5" и узкие риббоны (#3812) | [уточняется] | Есть отдельные правила для ширин 55-70 мм и втулки 110 мм. Делать пока не нужно, требования уточняются. |

### Технические ограничения

| Ограничение | Статус | Что ограничивает / зачем |
|---|---|---|
| ≤ 3 позиции на резку | [в коде] | `maxPositionsPerCut`. Сокращает число вариантов слияний и не делает резку слишком сложной. |
| ≤ 3 ширины на резку | [в коде] | `maxWidthsPerCut`. Слишком разношерстная раскладка хуже читается и сложнее для оператора. |
| Слияние только при выгоде | [в коде] | `planLayouts` сливает две раскладки только если становится лучше по цели: прогоны ↓, затем скрап ↓, затем отход ↓. |
| Окно срока 3 дня | [в коде] | Позиции с близкими сроками объединяются в один кластер. Бессрочные позиции идут отдельным последним кластером. |
| Сырье — непрерывным блоком | [в коде] | Алгоритм избегает шаблона A → B → A, потому что возврат к A дает лишнюю переналадку. |
| «Зафиксировано» держит день | [в коде] | Зафиксированная резка остается на своем дне и не дробится. Внутри дня оптимизатор может поставить ее раньше свободных. |
| Scope планирования | [в коде] | `planCutOperations` меняет только резки внутри выбранного диапазона дат и срочные незакрепленные резки, которые нужно подтянуть в окно. |
| Емкость дня / окно смены | [в коде] | День ограничен рабочим окном, обедом и уборкой. При нехватке места резка переносится или дробится на продолжение. |
| Наличие партии сырья | [в коде] | Без активной партии с остатком резка не создается. Партия выбирается по FIFO. |

## Цена нарушения правил

Планирование — это не один критерий, а компромисс. Часть правил жесткая: план, который их
нарушает, недопустим. Остальные факторы можно считать как штраф и затем подбирать веса на
реальных планах.

| # | Фактор | Тип | Натуральная цена нарушения | Текущая обработка | Стартовый вес |
|---|---|---|---|---|---|
| 1 | Срок изготовления (EDD) | мягкий, высокий приоритет | каждый день опоздания после `dueKey` | ранний срок идет раньше в очереди и при выборе дня (#3815/#3820/#3826) | 100 / день |
| 2 | Фольга в конце дня | бизнес-правило | загрязнение станка и внеплановая уборка перед обычной резкой | `isFoil` уводит длинную фольгу в конец дня (#3717/#3826) | 80 |
| 3 | Смена ножей | мягкий, в минутах | `KNIFE` = 30 минут | минимизируется через `changeoverCost`; цена фиксирована (#3600/#3666) | 30 |
| 4 | Смена сырья / намотки / партии | мягкий, в минутах | `MATERIAL_WINDING` = 15 минут | минимизируется через `changeoverCost` | 15 |
| 5 | Равномерная загрузка станков | мягкий | переполненный станок при свободном соседнем | `chooseSlitterBySetup` сначала смотрит `overflow`, затем группировку (#3801/#3830) | 20 |
| 6 | Группировка одного сырья / ножей | мягкий | лишние переходы между материалами и конфигурациями | `attach`, `sameKnives`, `sameMaterial`, жадная цепочка | 25 |
| 7 | Убывание числа полос | тай-брейк / стратегия | сложные резки лучше делать раньше смены | в `SETUP` это тай-брейк при равной переналадке; отдельная стратегия `FATIGUE` ставит сложные раньше | 10 |
| 8 | Дробление резки по дням | мягкий | продолжение на следующий день и риск лишнего простоя | `splitMachineQueue`, `setup-only` хвост, переиспользование продолжений | 3 |
| 9 | Лидер между резками | в минутах | `BETWEEN_CUTS × Кол-во план` | лидер считается в конце резки и занимает окно станка (#3401/#3688) | 2 |

**Приоритет по умолчанию сейчас:** срок → фольга в конец → минимум переналадки и группировка
сырья → баланс станков → число полос как тай-брейк. Весовая матрица нужна для будущего
оптимизатора: веса можно менять без переписывания каждого правила.

## Фаза 1. Подготовка и группировка позиций

### Что происходит

Система получает список позиций заказов из отчета `positions_list`. Каждая позиция — это строка
будущего производственного задания: какое сырье нужно, какой ширины ролик, сколько рулонов,
к какому сроку нарезать, какая втулка, какая намотка и какая длина прогона.

### Шаг 1.1. Нормализация строк (`rowsToGenPositions`)

Каждая строка отчета превращается в объект `genPosition`:

- числовые поля (`width`, `qty`, `length`, `windLength`) приводятся к числам;
- пустые числа становятся `0`, пустые строки остаются `''`;
- дата срока становится числовым ключом `dueKey` (`ГГГГММДД`), отсутствие срока — `Infinity`;
- позиция считается утвержденной, если утвержден заказ или сама позиция;
- флаг `isFoil` ставится по типу сырья, если он содержит «фольг»;
- поля `leader`, `sleeveId`, `orderId` остаются строковыми идентификаторами.

### Шаг 1.2. Профиль планирования (`groupPositionsByPlanningProfile`)

Позиции группируются по ключу совместимости:

```text
сырье + направление намотки + длина прогона + лидер + втулка
```

Позиции с разным профилем не попадут в одну резку. Это не оптимизационное пожелание, а
физическая совместимость: другой материал, намотка, длина, лидер или втулка требуют отдельной
резки.

### Шаг 1.3. Разбивка по втулке (`layoutPositionGroups`)

Если в группе все равно оказались разные `sleeveId`, она делится на подгруппы. Одна подгруппа —
одна втулка. В каждой резке все рулоны одного диаметра.

### Шаг 1.4. Кластеризация по сроку (`dueWindowGroups`)

Внутри подгруппы позиции сортируются по сроку. Позиции с разницей сроков до 3 дней объединяются
в один кластер и могут быть сведены в одну раскладку. Бессрочные позиции идут отдельным
последним кластером.

## Фаза 2. Раскладка полос (`cut-layout.js`)

### Что происходит

Для каждого кластера решается задача раскладки: какие полосы уложить в ширину джамбо, сколько
прогонов нужно, можно ли объединить несколько заказов в одну резку и чем добить остаток джамбо.

### Шаг 2.1. Seed «1 заказ = 1 резка»

Алгоритм начинает консервативно: один заказ и одна ширина дают отдельную seed-группу. Одинаковая
ширина внутри одного заказа консолидируется, но одинаковые ширины разных заказов не склеиваются
принудительно.

### Шаг 2.2. Базовая укладка (`composeLayout`)

Базовая раскладка работает так:

- ширины сортируются по убыванию и укладываются в джамбо хотя бы по одной полосе;
- ширина, которая не влезает в джамбо, попадает в `overflow`;
- для незапасных ширин выбирается минимальное число прогонов, которое закрывает заказ;
- для запасных ширин допустимо перепроизводство в складской запас;
- остаток джамбо добирается ходовыми ширинами через `bestFill`;
- результат содержит `strips`, `used`, `remainder`, `withinTolerance`, `overflow`.

### Шаг 2.3. Перебор выгодных слияний (`planLayouts`)

После seed-групп алгоритм пробует сливать пары раскладок. Слияние разрешено только если:

- объединенная раскладка физически влезает;
- закрывает все позиции обеих исходных раскладок;
- не превышает лимиты `maxWidthsPerCut` и `maxPositionsPerCut`;
- улучшает цель лексикографически: **прогоны ↓**, затем **скрап ↓**, затем **отход ↓**.

Невыгодные слияния не проводятся. Это защищает от «красивой» общей резки, которая на деле
увеличивает расход джамбо или перепроизводство.

### Шаг 2.4. Ограничение запаса (`capStockToHeadroom`)

После построения раскладок система проверяет «Максимальный запас». Если ширина превышает лимит
с учетом текущих остатков, лишние рулоны убираются из раскладки. Заказное покрытие при этом не
срезается.

## Фаза 3. Назначение станка и очередь

### Что происходит

Каждой готовой раскладке назначается слиттер. Затем резки каждого станка выстраиваются в
очередь, которая минимизирует переналадки и сохраняет технологические правила.

### Шаг 3.1. Выбор станка (`chooseSlitterBySetup`)

Сначала отбрасываются станки, где сырье запрещено стоп-листом. Среди допустимых станков выбор
идет по таким критериям:

1. **Емкость дня.** Если резка переполнит рабочий день одного станка, а на другом влезет,
   выбирается станок, где она влезает (#3830).
2. **Совместимость заправки.** Предпочитается станок с тем же набором ножей или тем же сырьем
   и намоткой (#3666/#3801).
3. **Минимальный прирост переналадки.** Сравнивается стоимость очереди до и после добавления
   резки.
4. **Баланс загрузки.** Если совместимых станков несколько, выбирается менее загруженный.
5. **Детерминированный тай-брейк.** При полном равенстве берется меньший id станка.

### Шаг 3.2. Очередность резок (`orderCuts`)

`orderCuts` не мутирует входные данные. На выходе каждая резка получает `sequence = 1..N`.

Текущая стратегия `SETUP`:

1. **Не-фольга раньше фольги.** Фольга идет отдельной группой в конец дня (#3717).
2. **Срок изготовления.** Внутри групп резки с меньшим `dueKey` идут раньше (#3815).
3. **Минимум переналадки.** Внутри одного срока строится жадная цепочка по `changeoverCost`.
4. **Число полос по убыванию при равенстве.** Это тай-брейк, а не глобальная пересортировка:
   он не должен ломать блоки сырья (#3783/#3785).

Отдельная стратегия `FATIGUE` может ставить более сложные резки раньше смены, если важнее
усталость оператора, а не минимум переналадки.

### Стоимость переналадки между соседними резками

Переналадка состоит из независимых операций. Каждая либо применяется, либо нет.

| Операция | Код | Мин. | Когда применяется |
|---|---|---:|---|
| Смена ножей / сужение ролика | `KNIFE` | 30 | Изменился набор ножей или следующий ролик уже предыдущего. Цена фиксированная, не зависит от числа ножей (#3600). |
| Смена сырья / намотки / партии | `MATERIAL_WINDING` | 15 | Сменилось сырье, направление намотки или партия. |
| Смена ножей + смена сырья | оба | 45 | Обе операции произошли одновременно: 30 + 15. |
| Лидер между резками | `BETWEEN_CUTS` | `2 × N` | Заправляется после намотки текущей резки. `N` = `Кол-во план`. |
| Уборка в конце смены | `CLEANUP_SHIFT` | 30 | Резервируется в конце рабочего дня, если в этот день есть резки. |

Первая резка очереди сравнивается с текущей заправкой станка из `prev_cut_setup`. Если
конфигурация совпадает, переналадка не нужна. Если данных о заправке нет, система
консервативно считает настройку ножей с нуля.

## Фаза 4. Расписание по дням

### Что происходит

Упорядоченная очередь каждого станка раскладывается по рабочим сменам. Каждая резка получает
плановую дату, время старта, длительность, число проходов и при необходимости продолжения на
следующие дни.

### Окно станка для одной резки

Каждая резка занимает окно из последовательных частей:

```text
[смена ножей] → [смена сырья/намотки] → [намотка/резка] → [лидер]
```

Переналадка идет **до** намотки. Лидер идет **после** намотки, но занимает станок как часть
окна текущей резки.

### Логика расписания (`buildSchedule`)

- Резки пакуются в рабочее окно смены в порядке очереди.
- Обед вставляется как пауза. Если длинная резка пересекает обед, намотка прерывается на обед
  и финиш сдвигается (#3816).
- В конце дня резервируется уборка `CLEANUP_SHIFT`.
- Если резка не влезает в смену, она переносится на следующий рабочий день.
- Фольга сохраняется в конце каждого дня при многодневном перепланировании.
- Первая резка дня считает переналадку относительно последней физической резки предыдущего дня.

### Пример тайминга окна

Первая резка дня — фольга горячего тиснения, намотка `IN`. На станке вчера была другая
конфигурация. Метраж прохода 122 м, 20 проходов, норма `WIND_FOIL_305 = 4 мин`,
лидер `2 × 20 = 40 мин`.

> Фольга считается блоками (#3742): `ceil(метраж / 305) × 4`. Проход 122 м стоит полную
> норму 4 минуты, а не пропорциональные 1.6 минуты.

| Время | Операция | Длительность |
|---|---|---:|
| 08:00 | смена ножей / сужение ролика | 30 мин |
| 08:30 | смена сырья / намотки / партии | 15 мин |
| 08:45 | намотка: `4 × 20` | 80 мин |
| 10:05 | лидер между резками: `2 × 20` | 40 мин |
| 10:45 | готово | — |

### Многодневное планирование (`planCutOperations`)

Если резка не помещается в один день, она дробится на сегменты:

- первый сегмент обновляет исходную резку;
- остаток создается как продолжение на следующий день;
- продолжения связаны через `continuationSignature`;
- повторный запуск при неизменной раскладке не создает новые записи и переиспользует
  существующие продолжения;
- `setup-only` сегмент может положить часть настройки в хвост дня, а остаток настройки —
  на следующий день (#3805/#3827).

`planCutOperations` уважает scope: при фильтре дат меняются только резки в выбранном окне и
незакрепленные срочные резки, которые нужно подтянуть в это окно. Чужие даты и зафиксированные
дни не переписываются.

### Пересборка дня после переноса/удаления (#3840)

Перенос задания на другой день/станок (`moveCutToDay`) и удаление одной резки (`runDeleteCutTask`)
изымают резку из середины дня станка. Раньше они переписывали `planStart` только самого
перемещаемого задания (и «Очередность» соседей), а **сохранённый `planStart` прочих резок дня
оставался прежним** — на месте вынутой резки висел простой. Страница планирования его не
показывала (очередь пересчитывает тайминг через `buildSchedule` на лету), а РМ «Диаграмма Ганта
(задания)» рисует **сохранённый** `planStart` — там дыра была видна.

Теперь обе операции терминально вызывают `autoSequenceQueue(preserveOrder=true)` — тот же
пересчёт, что после генерации: `splitMachineQueue`/`buildSchedule` с `gapFill` пакуют день
**встык**, дыра схлопывается, `planStart` затронутых дней (источник + цель) сохраняется.
`preserveOrder` держит ручной порядок оператора, фольгу — в конце дня (#3717), зафиксированные
задания — на своих днях (#3792); пишутся только изменившиеся записи (идемпотентно, #3427).

> Инвариант: сохранённый `planStart` дня станка всегда без внутреннего простоя — любая операция,
> изымающая резку из дня, обязана пересобрать день (а не «оставить на следующую генерацию»).

## Особые правила

### Короткая фольга ≤ 120 минут

Бизнес-правило согласовано: если суммарное время резки фольги не превышает 120 минут, такую
фольгу можно ставить не в конец дня. Она не успевает сильно загрязнить станок.

Сейчас это **требование, а не код**: `orderCuts` отделяет любую `isFoil` в конец очереди, а
`splitMachineQueue` и дневная сортировка также считают любую фольгу фольгой. Для реализации
нужно ввести признак короткой фольги и трактовать ее как обычную резку:

- в `orderCuts` класть короткую фольгу в `rest`, а не в `foil`;
- в `selectByConfig` считать ее `isFoil = 0`;
- в сортировке `(день, isFoil, Очередность)` не уводить ее в хвост;
- длинную фольгу оставлять в конце дня как сейчас.

### Втулка 0.5" и узкие риббоны (#3812)

Требование пока уточняется. Предварительно:

- риббоны на втулке 0.5" уже 55 мм не производим;
- фактическая ширина 55-57 мм режется как 55 мм;
- при втулке 110 мм для ширин 55-57 мм добавляют две полосы 110 мм;
- для ширин 63-64 мм при втулке 110 мм добавляют одну полосу 110 мм;
- ширины свыше 70 мм идут по основному правилу для втулок 1".

Делать пока не нужно. Когда правило финализируют, его нужно перенести в раскладку полос и
снять статус **[уточняется]**.

## Возможный оптимизатор на 3 дня

Сейчас план строится жадно: раскладка → выбор станка → очередь → расписание. Это быстро и
понятно, но не всегда дает лучший глобальный план. Следующий шаг — улучшающий поиск на
скользящем окне примерно 3 дня.

### Целевая функция

```text
Cost(plan) = Σ(вес_фактора × нарушение_фактора(plan))
```

Жесткие ограничения из раздела «Ограничения алгоритма» фильтруют недопустимые планы. Мягкие
факторы из матрицы «Цена нарушения правил» дают штраф.

Примеры штрафов:

- срок: `max(0, день_резки − dueDay) × W1`;
- фольга: длинная фольга не в конце дня × `W2`;
- переналадки: сумма `changeoverCost(prev, next)`;
- баланс: отклонение загрузки станка от средней или факт `overflow`;
- дробление: число продолжений и setup-only сегментов.

### Соседние ходы

Оптимизатор может пробовать локальные перестановки и считать `ΔCost`:

1. поменять две резки местами внутри дня одного станка;
2. перенести резку на другой день того же станка;
3. перенести резку на другой станок того же дня;
4. поменять резки между станками;
5. сдвинуть длинную фольгу в конец или короткую фольгу раньше.

### Поиск и применение

Базовый вариант — hill-climbing: применять лучший ход с отрицательным `ΔCost`, пока план
улучшается. Если потребуется выходить из локальных минимумов, можно добавить simulated
annealing или tabu search с ограничением по времени/итерациям.

Результат должен быть списком обновлений в формате `planCutOperations`: резка, новый станок,
новый день, новая очередность. Применять можно через превью («минус N минут переналадок,
минус K опозданий») или отдельную кнопку «Оптимизировать на 3 дня».

## Глоссарий

| Термин | Значение |
|---|---|
| Джамбо | Исходный рулон сырья. Ширина джамбо — максимальная суммарная ширина полос одной резки. |
| Прогон (`run`) | Один проход через станок. Число прогонов примерно равно `qty / количество полос данной ширины`. |
| Скрап (`scrap`) | Полосы, нарезанные сверх заказа по незапасным ширинам. Это перепроизводство, которое некуда деть. |
| Отход (`waste`) | Незанятая ширина джамбо × прогоны. Идет в технологический обрезок. |
| Лидер | Лента для заправки слиттера перед следующей резкой. Учитывается после намотки текущей резки. |
| Ходовые ширины | Ширины из «Максимального запаса», которыми можно добирать остаток джамбо в складской запас. |
| `isFoil` | Признак фольги: грязное сырье, требующее постановки в конец дня, кроме планируемого исключения для короткой фольги. |
| `sleeveId` | Диаметр втулки. Все рулоны одной резки должны иметь один `sleeveId`. |
| `dueKey` | Числовой ключ срока изготовления (`ГГГГММДД`) для сортировки и кластеризации. |
| `continuationSignature` | Ключ идентичности резки для сборки цепочки продолжений при многодневном разбиении. |
| `fixed` | Признак «Зафиксировано». Закрепляет день резки и защищает от удаления, но не запрещает перестановку внутри дня. |

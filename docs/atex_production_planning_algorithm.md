# Алгоритм планирования производства

_Слиттер · atex · `production-planning.js` + `cut-layout.js`_

Документ описывает полный конвейер формирования производственных заданий: от списка
позиций заказов до раскладки полос, выбора станка, очереди резок и расписания по дням.
Цель документа — объяснить алгоритм человеку: что делает система, какие правила нельзя
нарушать, где есть компромиссы и какие части кода отвечают за каждый шаг.

> **Источник истины по алгоритму планирования (#3717).** Перед правкой логики очереди,
> раскладки, расписания, переналадки или фольги сверяйся с этим документом и обновляй его
> в том же PR. Код: `download/atex/js/production-planning.js`,
> `download/atex/js/cut-layout.js`.

> **Важное изменение #3600.** Любая смена ножей стоит фиксированные **30 минут** на весь
> набор, независимо от числа переставленных ножей. Смена сырья/намотки считается отдельно
> и стоит **15 минут** по умолчанию.

## Доступ к данным: метаданные и отчёты (#3897)

> **Пользователь в любой момент может переставить колонки таблицы, добавить новые и удалить
> существующие. Это НЕ должно ничего ломать.** Любой код, читающий данные Integram, обязан
> быть устойчивым к изменению порядка и состава колонок.

Правила (обязательны для `production-planning.js`, `cut-layout.js` и всех модулей atex,
читающих данные):

1. **Никогда не обращайся к колонкам по позиции в массиве.** В ответе
   `object/{table}?JSON_OBJ` каждая запись приходит как `{ i, u, o, r:[…] }`, где порядок
   значений в `r` повторяет порядок **реквизитов в метаданных** — а он меняется, как только
   пользователь переставит колонки. Резолвь индекс по **имени** реквизита через свежие
   метаданные: `columnIndex(meta, reqName)` / `reqIdByName(meta, …)` (см. `mapCutRecord`).
   Единственное допустимое позиционное обращение — `r[0]` (главное значение записи): оно
   всегда первое по структуре, а не по порядку реквизитов.

2. **Метаданные тяни заново на каждой загрузке** (`loadMetadata` → `GET metadata`). Не зашивай
   в константы id или порядок реквизитов. Имена таблиц/реквизитов держим в `TABLE` / `CUT_REQ`
   / `DOWNTIME_REQ` и резолвим по ним. Пропажу или переименование колонки обрабатываем явной
   диагностикой («в отчёте нет колонки …»), а не падением и не молчаливо неверным значением.

3. **Предпочитай отчёты прямому чтению таблиц.** `report/{name}?JSON_KV` отдаёт данные парами
   «имя колонки → значение» и **вообще не зависит** от порядка и состава колонок таблицы —
   переживает любые перестановки. Основная загрузка очереди и обеспечения идёт отчётом
   `cut_planning?JSON_KV` именно поэтому. Прямое `object/{table}?JSON_OBJ` оставляем там, где
   отчёта нет или записей мало, и всегда декодируем по имени (п. 1). Новое поле, нужное логике,
   **добавляй в отчёт**, а не считывай позиционно из таблицы.

> **Известное исключение.** `loadDaySettings` читает системную таблицу «Настройка» позиционно
> (`r[0]`=ключ, `r[1]`=тип/БД, `r[2]`=значение) — её состав фиксирован системой. При переносе
> на чтение по имени резолвить тип/значение через `columnIndex`, не менять схему вслепую.

> Симптом нарушения правила (#3897): после того как заказчик переставил колонки таблицы
> «Задание в производство» (1078), **позиционная** разборка `r` (в отладочных запросах) стала
> выдавать неверные поля. Сам код страницы читает по имени и не пострадал — правило
> зафиксировано, чтобы так оставалось и впредь, в том числе во всех новых модулях.

## Общий конвейер

Все шаги идут от входных данных до записи плана в базу.

| # | Функция | Что делает |
|---|---|---|
| 1 | `positions_list` | Отчет с позициями заказов: сырье, ширина, количество, срок, втулка, направление намотки, длина прогона. |
| 2 | `rowsToGenPositions` | Нормализует строки отчета: числа → числа, пустые значения → `0`/`''`, считает `dueKey` (`ГГГГММДД`), проставляет `isFoil`. |
| 3 | `groupPositionsByPlanningProfile` | Группирует позиции по профилю: сырье + намотка + длина прогона + лидер + втулка + число втулочных полос (#3812). Разные профили физически не смешиваются в одной резке. |
| 4 | `layoutPositionGroups` | Разбивает профиль по `sleeveId`, чтобы в одной резке была одна втулка. |
| 4а | втулочные полосы 0.5"/110 (#3812, #3872) | Резервирует ширину джамбо под полосы 110 мм и дописывает их в раскрой (`appendCoreStrip`). Узкие риббоны (<55 мм) на втулке 0.5" исключаются. #3872: полосы привязываются к уже заказанным позициям 110 мм того же заказа (`selectCoreStripFillers`), не нашлось — синтетика. |
| 5 | `dueWindowGroups` | Объединяет близкие сроки в окно 3 дня. Срочные позиции обрабатываются раньше бессрочных. |
| 6 | `planLayouts` | Строит раскладки полос: seed «1 заказ = 1 резка», затем выгодные слияния по B&B. |
| 7 | `capStockToHeadroom` | Обрезает добор «впрок» по лимиту максимального запаса. Ширины сверх лимита убираются из раскладки. |
| 8 | `chooseSlitterBySetup` | Выбирает станок: стоп-лист сырья, вместимость дня, совместимость заправки, баланс загрузки. |
| 9 | `orderCuts` | Упорядочивает резки станка: срок, фольга, блоки сырья, минимум переналадки, тай-брейки. |
| 10 | `buildSchedule` / `planCutOperations` | Раскладывает очередь по сменам и дням, дробит длинные резки, создает/обновляет продолжения. |

## Ограничения алгоритма

Бизнес-ограничения задает технология. Технические ограничения сужают перебор, чтобы план был
расчетным и удобным оператору. В таблицах ниже статус означает:

- **[в коде]** — правило реализовано;
- **[требование]** — правило согласовано, но код пока ведет себя по старому базовому правилу;
- **[уточняется]** — требование пока не финализировано.

### Бизнес-ограничения

| Ограничение | Статус | Что ограничивает / зачем |
|---|---|---|
| Одна втулка на резку | [в коде] | Все рулоны одной резки надеваются на одну втулку. Позиции с разным `sleeveId` идут в разные резки. |
| Одно направление намотки | [в коде] | `IN` и `OUT` не смешиваются в одной резке: это физическое ограничение станка. |
| Одна длина прогона | [в коде] | Все полосы одной резки наматываются за один прогон, поэтому длина у всех позиций должна совпадать. |
| Один лидер на резку | [в коде] | Резка заправляется одним лидером. Позиции с разным лидером идут в разные резки. |
| Ширина полос ≤ ширины джамбо | [в коде] | Позиция шире джамбо не планируется и попадает в `skipped` с причиной «шире джамбо». |
| Отход в пределах допуска | [в коде] | Остаток ширины джамбо должен быть в пределах `tolerance`, иначе алгоритм ищет добор ходовыми ширинами. |
| Стоп-лист сырья по станку | [в коде] | Сырье нельзя назначить на станок, где оно запрещено. Например, фольга может быть разрешена только на части станков через справочник. |
| Фольга в конце дня | [в коде] | Фольга загрязняет станок, поэтому длинная фольга должна стоять после обычных резок каждого дня. |
| Короткая фольга ≤ 120 минут может идти не в конец дня | [требование] | Если суммарное время резки фольги не превышает 120 минут, она не успевает сильно загрязнить станок. В текущем коде любая фольга пока все равно считается фольгой и ставится в конец дня. |
| Втулка 0.5" — сетка ширин (#3812) | [в коде] | На втулке 0.5" риббон у́же 55 мм не производится (позиция в `skipped`). Ширина > 70 мм режется по обычному правилу. См. «Втулочные полосы 0.5"» ниже. |
| Втулочные полосы 0.5" / 110 мм (#3812, #3872) | [в коде] | При втулке 0.5" шириной 110 мм в раскрой добавляются полосы 110 мм: ширина 55-57 → 2 полосы, 63-64 → 1 полоса. Полосы занимают ширину джамбо той же резки. #3872: если 110 мм уже заказаны (позиции того же заказа) — полосы обеспечивают эти позиции; иначе синтетика. |

### Технические ограничения

| Ограничение | Статус | Что ограничивает / зачем |
|---|---|---|
| ≤ 3 позиции на резку | [в коде] | `maxPositionsPerCut`. Сокращает число вариантов слияний и не делает резку слишком сложной. |
| ≤ 3 ширины на резку | [в коде] | `maxWidthsPerCut`. Слишком разношерстная раскладка хуже читается и сложнее для оператора. |
| Слияние только при выгоде | [в коде] | `planLayouts` сливает две раскладки только если становится лучше по цели: прогоны ↓, затем скрап ↓, затем отход ↓. |
| Окно срока 3 дня | [в коде] | Позиции с близкими сроками объединяются в один кластер. Бессрочные позиции идут отдельным последним кластером. |
| Сырье — непрерывным блоком | [в коде] | Алгоритм избегает шаблона A → B → A, потому что возврат к A дает лишнюю переналадку. |
| «Зафиксировано» держит день | [в коде] | Зафиксированная резка остается на своем дне и не дробится. Внутри дня оптимизатор может поставить ее раньше свободных. |
| Scope планирования | [в коде] | `planCutOperations` меняет только резки внутри выбранного диапазона дат и срочные незакрепленные резки, которые нужно подтянуть в окно. |
| Емкость дня / окно смены | [в коде] | День ограничен рабочим окном, обедом и уборкой. При нехватке места резка переносится или дробится на продолжение. |
| Наличие партии сырья | [в коде] | Без активной партии с остатком резка не создается. Партия выбирается по FIFO. |

## Цена нарушения правил

Планирование — это не один критерий, а компромисс. Часть правил жесткая: план, который их
нарушает, недопустим. Остальные факторы можно считать как штраф и затем подбирать веса на
реальных планах.

| # | Фактор | Тип | Натуральная цена нарушения | Текущая обработка | Стартовый вес |
|---|---|---|---|---|---|
| 1 | Срок изготовления (EDD) | мягкий, высокий приоритет | каждый день опоздания после `dueKey` | ранний срок идет раньше в очереди и при выборе дня (#3815/#3820/#3826) | 100 / день |
| 2 | Фольга в конце дня | бизнес-правило | загрязнение станка и внеплановая уборка перед обычной резкой | `isFoil` уводит длинную фольгу в конец дня (#3717/#3826) | 80 |
| 3 | Смена ножей | мягкий, в минутах | `KNIFE` = 30 минут | минимизируется через `changeoverCost`; цена фиксирована (#3600/#3666) | 30 |
| 4 | Смена сырья / намотки / партии | мягкий, в минутах | `MATERIAL_WINDING` = 15 минут | минимизируется через `changeoverCost` | 15 |
| 5 | Равномерная загрузка станков | мягкий | переполненный станок при свободном соседнем | `chooseSlitterBySetup` сначала смотрит `overflow`, затем группировку (#3801/#3830) | 20 |
| 6 | Группировка одного сырья / ножей | мягкий | лишние переходы между материалами и конфигурациями | `attach`, `sameKnives`, `sameMaterial`, жадная цепочка | 25 |
| 7 | Убывание числа полос | тай-брейк / стратегия | сложные резки лучше делать раньше смены | в `SETUP` это тай-брейк при равной переналадке; отдельная стратегия `FATIGUE` ставит сложные раньше | 10 |
| 8 | Дробление резки по дням | мягкий | продолжение на следующий день и риск лишнего простоя | `splitMachineQueue`, `setup-only` хвост, переиспользование продолжений | 3 |
| 9 | Лидер между резками | в минутах | `BETWEEN_CUTS × Кол-во план` | лидер считается в конце резки и занимает окно станка (#3401/#3688) | 2 |

**Приоритет по умолчанию сейчас:** срок → фольга в конец → минимум переналадки и группировка
сырья → баланс станков → число полос как тай-брейк. Весовая матрица нужна для будущего
оптимизатора: веса можно менять без переписывания каждого правила.

## Фаза 1. Подготовка и группировка позиций

### Что происходит

Система получает список позиций заказов из отчета `positions_list`. Каждая позиция — это строка
будущего производственного задания: какое сырье нужно, какой ширины ролик, сколько рулонов,
к какому сроку нарезать, какая втулка, какая намотка и какая длина прогона.

### Шаг 1.1. Нормализация строк (`rowsToGenPositions`)

Каждая строка отчета превращается в объект `genPosition`:

- числовые поля (`width`, `qty`, `length`, `windLength`) приводятся к числам;
- пустые числа становятся `0`, пустые строки остаются `''`;
- дата срока становится числовым ключом `dueKey` (`ГГГГММДД`), отсутствие срока — `Infinity`;
- позиция считается утвержденной, если утвержден заказ или сама позиция;
- флаг `isFoil` ставится по типу сырья, если он содержит «фольг»;
- поля `leader`, `sleeveId`, `orderId` остаются строковыми идентификаторами.

### Шаг 1.2. Профиль планирования (`groupPositionsByPlanningProfile`)

Позиции группируются по ключу совместимости:

```text
сырье + направление намотки + длина прогона + лидер + втулка + число втулочных полос
```

Позиции с разным профилем не попадут в одну резку. Это не оптимизационное пожелание, а
физическая совместимость: другой материал, намотка, длина, лидер или втулка требуют отдельной
резки. **Число втулочных полос** (#3812) — отдельное измерение: позиции с разной потребностью
в полосах 110 мм (0/1/2, см. [Шаг 2.0](#шаг-20-втулочные-полосы-05--110-мм-3812)) идут в разные
резки, потому что у каждой своя зарезервированная под полосы ширина джамбо. Для обычных позиций
(не втулка 0.5"/110) число полос = 0 у всех, поэтому разбиения нет.

### Шаг 1.3. Разбивка по втулке (`layoutPositionGroups`)

Если в группе все равно оказались разные `sleeveId`, она делится на подгруппы. Одна подгруппа —
одна втулка. В каждой резке все рулоны одного диаметра.

### Шаг 1.4. Кластеризация по сроку (`dueWindowGroups`)

Внутри подгруппы позиции сортируются по сроку. Позиции с разницей сроков до 3 дней объединяются
в один кластер и могут быть сведены в одну раскладку. Бессрочные позиции идут отдельным
последним кластером.

## Фаза 2. Раскладка полос (`cut-layout.js`)

### Что происходит

Для каждого кластера решается задача раскладки: какие полосы уложить в ширину джамбо, сколько
прогонов нужно, можно ли объединить несколько заказов в одну резку и чем добить остаток джамбо.

### Шаг 2.0. Втулочные полосы 0.5" / 110 мм (#3812)

Втулка задаётся записью «Диаметр втулки»: реквизит «Дюймы» даёт диаметр (0.5"). Ширина втулки
(57 или 110) берётся из реквизита «Ширина втулки, мм», а если он не заведён — из **названия
записи** («Втулка картонная 0.5" ширина 110 мм» → 110, `parseSleeveWidthFromName`). Поэтому
правило работает на текущих данных без изменения схемы. Обе величины проставляются позициям в
`annotatePositionsCutWidth` (`sleeveInches`, `sleeveWidth`).

Правила для втулки **0.5"** считаются по **номинальной** ширине заказа:

- риббон у́же **55 мм** не производится — позиция исключается из планирования и попадает в
  `skipped` («втулка 0.5″: ширина < 55 мм»);
- при ширине втулки **110 мм** в раскрой добавляются полосы 110 мм:
  - ширина продукта **55-57 мм** → **2 полосы** 110 мм (пример: 55×12 + 110×2);
  - ширина продукта **63-64 мм** → **1 полоса** 110 мм (пример: 64×12 + 110×1);
  - прочие ширины (58-62, 65-70) полос не получают; ширина **> 70 мм** режется по обычному
    правилу (как втулка 1"), полос нет.

Полосы 110 мм **занимают ширину джамбо той же резки**: перед укладкой продукта ширина джамбо
уменьшается на `count × 110` (`effJumbo`), а затем `appendCoreStrip` дописывает полосы в раскрой.
Поэтому в раскрое продуктовых полос становится меньше ровно на занятую полосами 110 мм ширину.
Втулочные полосы помечены `core:true`: они показываются «Партией ГП» наравне с продуктом, но
`capStockToHeadroom` их не урезает, а **число прогонов от них не зависит** (фикс. число полос
задаёт продукт-носитель; ширины «только из `core`-полос» исключены из расчёта прогонов —
`coreOnlyStripWidths` в `plannedRunsForLayout`).

**Использование уже заказанных втулок 110 мм (#3872).** Дополнительные втулки 110 мм могут быть
уже в заказе (иногда их делают, иногда нет). При планировании носитель полос НЕ режет их «впустую»,
а сначала ищет **подходящие позиции заказа 110 мм** — необеспеченные, **того же заказа** (обычно
они идут в рамках одного заказа), того же сырья/намотки/длины джамбо, фактической ширины 110 мм
(`isCoreStripFiller`). Найденные позиции (`selectCoreStripFillers`) **привязываются к полосам**
(`appendCoreStrip(..., fillerPositionIds)`): полоса несёт их `positionIds`, а сами позиции
попадают в `positionsCovered` — резка их **обеспечивает на произведённое** `min(заказ, полос × прогонов)`,
излишек уходит в запас, а недостаток (заказ больше выпуска) игнорируется (в рамках одного заказа
это ошибка менеджера). Число прогонов при этом по-прежнему диктует продукт-носитель (позиции 110 мм
их не увеличивают). **Не нашлось** подходящих позиций — полосы добиваются синтетически, как было в
#3812 (`positionIds: []`, обеспечения нет). Профили-носители планируются **первыми**, чтобы забранные
110-мм позиции не ушли в собственную отдельную резку (`coreFillerClaims`, `orderedProfiles`).

### Шаг 2.1. Seed «1 заказ = 1 резка»

Алгоритм начинает консервативно: один заказ и одна ширина дают отдельную seed-группу. Одинаковая
ширина внутри одного заказа консолидируется, но одинаковые ширины разных заказов не склеиваются
принудительно.

### Шаг 2.2. Базовая укладка (`composeLayout`)

Базовая раскладка работает так:

- ширины сортируются по убыванию и укладываются в джамбо хотя бы по одной полосе;
- ширина, которая не влезает в джамбо, попадает в `overflow`;
- для незапасных ширин выбирается минимальное число прогонов, которое закрывает заказ;
- для запасных ширин допустимо перепроизводство в складской запас;
- остаток джамбо добирается ходовыми ширинами через `bestFill`;
- результат содержит `strips`, `used`, `remainder`, `withinTolerance`, `overflow`.

### Шаг 2.3. Перебор выгодных слияний (`planLayouts`)

После seed-групп алгоритм пробует сливать пары раскладок. Слияние разрешено только если:

- объединенная раскладка физически влезает;
- закрывает все позиции обеих исходных раскладок;
- не превышает лимиты `maxWidthsPerCut` и `maxPositionsPerCut`;
- улучшает цель лексикографически: **прогоны ↓**, затем **скрап ↓**, затем **отход ↓**.

Невыгодные слияния не проводятся. Это защищает от «красивой» общей резки, которая на деле
увеличивает расход джамбо или перепроизводство.

### Шаг 2.4. Ограничение запаса (`capStockToHeadroom`)

После построения раскладок система проверяет «Максимальный запас». Если ширина превышает лимит
с учетом текущих остатков, лишние рулоны убираются из раскладки. Заказное покрытие при этом не
срезается.

## Фаза 3. Назначение станка и очередь

### Что происходит

Каждой готовой раскладке назначается слиттер. Затем резки каждого станка выстраиваются в
очередь, которая минимизирует переналадки и сохраняет технологические правила.

### Шаг 3.1. Выбор станка (`chooseSlitterBySetup`)

Сначала отбрасываются станки, где сырье запрещено стоп-листом. Среди допустимых станков выбор
идет по таким критериям:

1. **Емкость дня.** Если резка переполнит рабочий день одного станка, а на другом влезет,
   выбирается станок, где она влезает (#3830).
2. **Совместимость заправки.** Предпочитается станок с тем же набором ножей или тем же сырьем
   и намоткой (#3666/#3801).
3. **Минимальный прирост переналадки.** Сравнивается стоимость очереди до и после добавления
   резки.
4. **Баланс загрузки.** Если совместимых станков несколько, выбирается менее загруженный.
5. **Детерминированный тай-брейк.** При полном равенстве берется меньший id станка.

Это **жадное** назначение (каждая раскладка — независимо). Оно группирует одно сырьё/ножи на
один станок и может скопить работу на нём на несколько дней, пока соседний станок простаивает.
Поэтому после жадного шага идёт выравнивание (шаг 3.1-bis).

### Шаг 3.1-bis. Выравнивание загрузки станков (`rebalanceSlitterLoad`, #3848)

После жадного назначения и до построения очереди план выравнивается **в памяти, по массиву
раскладок** (никаких запросов в базу — см. фактор #5 весовой матрицы, вес 20). Итеративный
локальный поиск: на каждом шаге берётся **перегруженный** станок (заканчивает на 2-й день и
позже — `spanDays ≥ 2`) и одно его задание переносится на станок, где общий счёт лучше. Счёт
состояния — лексикографически `[макс. дата окончания (span целых дней), макс. ДРОБНАЯ дата
окончания, пик минут, сумма квадратов минут]`; сумма **квадратов** (а не просто сумма) штрафует
перекос и выталкивает работу на простаивающий станок (простая сумма росла бы от лишних настроек и
склеивала всё на меньшем числе станков). Существующие резки держат базовую загрузку своих станков
и не двигаются; запрещённое стоп-листом сырьё на станок не переносится; на станок в отпуске в день
задания не переносится (#3876).

**Загруженность = ДАТА ОКОНЧАНИЯ последней задачи (#3881).** Не сумма минут, а **span** —
число календарных дней от базы плана до окончания. Содержимое (`ceil(минуты/ёмкость)` рабочих
дней) кладётся по дням, **пропуская дни отпуска** станка (`opts.machineDayOff`), а сами дни
отпуска считаются ЗАНЯТЫМИ. Поэтому станок с отпуском заканчивает **позже** при тех же минутах —
его не грузят на послеотпускное время, пока другие простаивают. Станок в отпуске **без задач после
него** «занят» до конца отпуска (`span` = конец отпуска): «дату окончания отпуска считать
последней датой задач, если после него задач нет». Без инфо об отпусках (`machineDayOff` не задан)
`span == ceil(минуты/ёмкость)` — прежнее поведение (#3848). *Ограничение:* содержимое моделируется
от дня базы плана; «дыра» (свободные дни) перед будущим отпуском без работы после него span не
увеличивает.

**Дробная дата окончания против «плато» ceil (#3921).** Целый `span` (ceil дней) создаёт плато:
при отпуске станок может держать хвост работы **после отпуска** (напр. на 13.07), пока другие
простаивают с 09.07, а перенос одного задания не меняет целый `span` (1680→1500 мин = те же
4 «дня») — счёт по `span` не улучшается, работа застревает за отпуском. Поэтому вторым ключом счёта
идёт `endPos` — **дробная** дата окончания (та же раскладка по дням с пропуском отпуска, но
последний рабочий день заполнен частично, по минутам). По минутам каждый перенос строго снижает
хвост за отпуском → он **стекает** на свободные станки, пока станок в отпуске не перестанет работать
после отпуска (его `endPos` упирается в конец отпуска). Пустой станок в отпуске «занят» до конца
отпуска — как у `span`. Так реализовано требование «смотреть на возможную дату начала задачи (дату
освобождения от задач/отпуска) — кто освобождается раньше, тому и даём».

Процесс **итерационный**, **прекращается при отсутствии прогресса** (нет хода, строго улучшающего
счёт), и защищён от **цикличных перестановок**: множество посещённых комбинаций (`stateHash` по
`plan.slitterId`) — идентичное назначение не повторяется; плюс ходы только строго улучшающие счёт
⇒ счёт монотонно падает. Каждый шаг журналируется в консоль («панель отладки»): старт-загрузка,
каждый перенос `задание Cn: Cx→Cy`, причина остановки. Активно только при заданной ёмкости дня
(генерация); без неё (тесты/обратная совместимость) переносов нет.

**Производительность (#3871).** Минуты станка = намотка + переналадка очереди + настройка первой
резки. Переналадка очереди требует упорядочивания набора, и на каждую пробу переноса считается по
полному набору станка. Полный перебор стартов жадной цепочки (`greedySequence`) — `O(n³)`, и при
сотне с лишним резок «Создать» висел минутами (≈40 с на перенос). На время выравнивания включаются
два упрощения: стоимость перехода кэшируется по паре `id` (один переход не пересчитывается тысячи
раз), а цепочка строится **от одного старта** (`O(n²)` вместо `O(n³)`). Оценка переналадки здесь
нужна лишь как ориентир баланса — финальную очередь всё равно строит `planCutOperations`, поэтому
точность одностартовой цепочки достаточна. Вне выравнивания планировщик считает переналадку как
прежде (полный перебор стартов).

### Шаг 3.2. Очередность резок (`orderCuts`)

`orderCuts` не мутирует входные данные. На выходе каждая резка получает `sequence = 1..N`.

> **`sequence` — внутренний ординал генерации, а НЕ хранимое поле (#3923).** Он живёт только в
> памяти в пределах одного прохода генерации/пересборки: `orderCuts` (и `planQueues`) выстраивают
> порядок, `splitMachineQueue` по нему присваивает `planStart`, `computeCutSetupUpdates` по нему
> считает переналадку. В базу «Очередность» **не пишется** и при загрузке **не читается** —
> единственный сохранённый носитель порядка резки — её `planStart` (главное значение, `t1078`).
> Порядок исполнения на всех РМ (страница планирования, «Диаграмма Ганта», пульт слиттера)
> определяется **строго по возрастанию `planStart`**. См. инвариант в конце Фазы 4.

Текущая стратегия `SETUP`:

1. **Не-фольга раньше фольги.** Фольга идет отдельной группой в конец дня (#3717).
2. **Срок изготовления.** Внутри групп резки с меньшим `dueKey` идут раньше (#3815).
3. **Минимум переналадки.** Внутри одного срока строится жадная цепочка по `changeoverCost`.
4. **Число полос по убыванию при равенстве.** Это тай-брейк, а не глобальная пересортировка:
   он не должен ломать блоки сырья (#3783/#3785).

Отдельная стратегия `FATIGUE` может ставить более сложные резки раньше смены, если важнее
усталость оператора, а не минимум переналадки.

### Стоимость переналадки между соседними резками

Переналадка состоит из независимых операций. Каждая либо применяется, либо нет.

| Операция | Код | Мин. | Когда применяется |
|---|---|---:|---|
| Смена ножей / сужение ролика | `KNIFE` | 30 | Изменился набор ножей или следующий ролик уже предыдущего. Цена фиксированная, не зависит от числа ножей (#3600). |
| Смена сырья / намотки / партии | `MATERIAL_WINDING` | 15 | Сменилось сырье, направление намотки или партия. |
| Смена ножей + смена сырья | оба | 45 | Обе операции произошли одновременно: 30 + 15. |
| Лидер между резками | `BETWEEN_CUTS` | `2 × N` | Заправляется после намотки текущей резки. `N` = `Кол-во план`. |
| Уборка в конце смены | `CLEANUP_SHIFT` | 30 | Резервируется в конце рабочего дня, если в этот день есть резки. |

Первая резка очереди сравнивается с текущей заправкой станка из `prev_cut_setup`. Если
конфигурация совпадает, переналадка не нужна. Если данных о заправке нет, система
консервативно считает настройку ножей с нуля.

«Текущая заправка» = ПОСЛЕДНЯЯ задача станка СТРОГО ДО начала окна планирования. Запрашивается
по каждому станку отдельно из отчёта `prev_cut_setup` (report 93371) с фильтрами
`FR_slitter_id={станок}` и `FR_task_start=<{планБаза}` (issue #3737), где `планБаза` — полночь
дня «С» из фильтра. Фильтр `task_start<планБаза` обязателен: после генерации в отчёте уже лежат
задачи самого нового плана (их `task_start` в будущем), и без фильтра «верхняя по `task_start`»
оказалась бы будущей задачей плана → ложная смена сырья у первой резки (окно 51 мин вместо 36 —
расхождение карточки и модалки тайминга, #3862). Заправка обновляется и при загрузке страницы, и
перед каждой генерацией (дату окна могли сменить).

## Фаза 4. Расписание по дням

### Что происходит

Упорядоченная очередь каждого станка раскладывается по рабочим сменам. Каждая резка получает
плановую дату, время старта, длительность, число проходов и при необходимости продолжения на
следующие дни.

### Окно станка для одной резки

Каждая резка занимает окно из последовательных частей:

```text
[смена ножей] → [смена сырья/намотки] → [намотка/резка] → [лидер]
```

Переналадка идет **до** намотки. Лидер идет **после** намотки, но занимает станок как часть
окна текущей резки.

### Логика расписания (`buildSchedule`)

- Резки пакуются в рабочее окно смены в порядке очереди.
- Обед вставляется как пауза. Если длинная резка пересекает обед, намотка прерывается на обед
  и финиш сдвигается (#3816).
- В конце дня резервируется уборка `CLEANUP_SHIFT`.
- Если резка не влезает в смену, она переносится на следующий рабочий день.
- Фольга сохраняется в конце каждого дня при многодневном перепланировании.
- Первая резка дня считает переналадку относительно последней физической резки предыдущего дня.

### Пример тайминга окна

Первая резка дня — фольга горячего тиснения, намотка `IN`. На станке вчера была другая
конфигурация. Метраж прохода 122 м, 20 проходов, норма `WIND_FOIL_305 = 4 мин`,
лидер `2 × 20 = 40 мин`.

> Фольга считается блоками (#3742): `ceil(метраж / 305) × 4`. Проход 122 м стоит полную
> норму 4 минуты, а не пропорциональные 1.6 минуты.

| Время | Операция | Длительность |
|---|---|---:|
| 08:00 | смена ножей / сужение ролика | 30 мин |
| 08:30 | смена сырья / намотки / партии | 15 мин |
| 08:45 | намотка: `4 × 20` | 80 мин |
| 10:05 | лидер между резками: `2 × 20` | 40 мин |
| 10:45 | готово | — |

### Максимальный нахлёст за конец смены (#3847)

Задание режется по **проходам** (`splitMachineQueue`), и его можно разрывать по дням на границе
прохода. Сколько проходов влезает в хвост дня, ограничено двумя потолками от **`DAY_END_HOUR`**
(реального конца смены, а не `cutEndMin = DAY_END_HOUR − TOTAL_INTERVALS`):

- **`MAX_OVERWORK_CUTS`** — проход (резку) можно положить с нахлёстом, только если он кончится
  **не позже `DAY_END_HOUR + MAX_OVERWORK_CUTS`**. Длиннее — проход уходит на следующий день
  (отдельным сегментом-продолжением).
- **`MAX_OVERWORK_TUNE`** — настройку (ножи / смену сырья) можно положить с нахлёстом, только если
  она кончится **не позже `DAY_END_HOUR + MAX_OVERWORK_TUNE`**. Не влезает — настройка и проходы
  переносятся на следующий день.

Оба значения — из «Настройки» (мин). Это уточняет #3760/#3821: вместо «один любой нахлёстный
проход» (#3760) или «строго встык до `cutEndMin`» (#3821) — **нахлёст разрешён, но ограничен
минутами** за концом смены. Лимит не задан (пусто) → фича выключена, планировщик пакует до
`cutEndMin` без сверхнормативного нахлёста (поведение #3821). Реализация — чистая `splitMachineQueue`
(обе ветки: базовая для генерации и `gapFill` для автозаполнения/пересборки); потолок считает
хелпер `availFor(day, 'cuts'|'tune')`, лимиты приходят из `resolveWorkingWindow`
(`maxOverworkCutsMin` / `maxOverworkTuneMin`, `dayEndHourMin = endMin`).

**Настройка в хвост — без нахлёста (#3939).** В ветке `gapFill` (именно она пишет план —
`planCutOperations(self.cuts, {gapFill: true})`) `setup-only` хвост кладётся, **только если
настройка ЦЕЛИКОМ влезает** до конца рабочего окна (`effCapacity − clock`, без нахлёста). Если не
влезает (напр. атомарные ножи 30 при остатке 5, или ножи+сырьё 45 при остатке 20) — в хвост **ничего
не кладём**, вся резка уходит на следующий день **одной карточкой**. Иначе `setup-only` хвост с
нахлёстом раздувал бейдж дня за ёмкость (заказчик: «оверворк», день 542 при ≈460), а при простое
станка `applyDowntime` выталкивал его на начало следующего дня поверх продолжения («настройка в
начале дня», #3934). Когда настройка влезает целиком — она по-прежнему уходит в хвост дня N, а
проходы начинаются с дня N+1 без повторной настройки (#3635 п.5). Нахлёст **проходов** (#3847)
не тронут; базовая (не-`gapFill`) ветка используется для оценок и на запись плана не влияет.

> Инвариант: ни один проход не кончается позже `DAY_END_HOUR + MAX_OVERWORK_CUTS`, ни одна
> настройка — позже `DAY_END_HOUR + MAX_OVERWORK_TUNE`. Нахлёст ограничен, переполнения дня
> (баг #3821 «483 при 480») нет — потолок абсолютный.

### Многодневное планирование (`planCutOperations`)

Если резка не помещается в один день, она дробится на сегменты:

- первый сегмент обновляет исходную резку;
- остаток создается как продолжение на следующий день;
- продолжения связаны через `continuationSignature`;
- повторный запуск при неизменной раскладке не создает новые записи и переиспользует
  существующие продолжения;
- `setup-only` сегмент может положить часть настройки в хвост дня, а остаток настройки —
  на следующий день (#3805/#3827).

`planCutOperations` уважает scope: при фильтре дат меняются только резки в выбранном окне и
незакрепленные срочные резки, которые нужно подтянуть в это окно. Чужие даты и зафиксированные
дни не переписываются.

**Спил-день продолжения (#3918).** Разбитая по дням цепочка in-scope переливает своё продолжение
на следующий день. Если этот день **вне** scope (узкий фильтр), его собственные резки не
перепланируются и остаются с 08:00 — продолжение садится **поверх** них: день переполняется
(бейдж 571 при ёмкости 450), карточки лезут за смену (17:01…23:31), `scheduleFromStored` каскадит
нахлёст. Поэтому верхняя граница scope **расширяется** до дня самого позднего сегмента, **пока**
на новых днях появляются чужие резки (сами они могут переливаться дальше). Это ровно правило
оператора #3660: «не лезь в другие даты, **если только задание не вылезает из этого дня в
следующий**» — спил-день перепланируем вместе с продолжением, дальние (за зазором) не трогаем.

**Осиротевшие сегменты настройки (#3924).** `setup-only` запись (0 проходов) — всегда **хвост**
цепочки: настройка в конце дня N, намотка-продолжение с дня N+1. Если её голову-резку удалили или
перенесли прежней пересборкой, а «ID первой части» у сегмента опустел, `mergeContinuationChains`
не подшивает его к цепочке и делает **отдельной логической резкой** с `plannedRuns = 0`. Планировщик
такую резку не раскладывает (проходов нет), поэтому штатный delete-путь (`usedByHead`) её не трогает,
и она оседает **«настройкой в начале дня»**; её минуты раздувают бейдж дня за ёмкость (510 при 450).
`scheduleFromStored` (#3846) показывает сохранённый план как есть — раньше эти сироты прятал пересчёт
на лету. Поэтому `planCutOperations` удаляет **сироты** — логические резки, у которых сумма проходов
цепочки = 0, — в пределах scope (реальные резки и настоящие setup-хвосты, у чьей головы проходов > 0,
не затрагиваются; сироты вне окна фильтра не трогаем, #3660).

**Хвост настройки и потолок нахлёста при простое (#3934).** `splitMachineQueue` кладёт `setup-only`
сегмент в **хвост дня N** с намеренным нахлёстом за конец смены (#3635 п.5/#3739/#3805). Когда у
станка есть хоть один блок простоя («Отпуск» — в реальном плане их десятки, все далеко),
`applyDowntime` прогонял **все** сегменты через `nextFreeWorkMinute`, а тот применял потолок нахлёста
(`fitEnd = cutEnd + MAX_OVERWORK_CUTS`) к **каждому** — даже к не тронутому простоем. Хвостовая
настройка кончается за потолком → её выталкивало на **начало следующего дня поверх её же
продолжения**: «настройка в начале дня» и рост минут дня (бейдж 495/501). Без простоя
`shiftPlacementsPastDowntime` выходит сразу, и того же не было («то есть, то нет»). Потолок нахлёста
(#3907) применяется **только** к сегменту, реально **сдвинутому простоем** (блоком либо встык-курсором
за предыдущим сдвинутым): не тронутый простоем хвост остаётся в конце дня N, продолжение — на дне N+1.

### Пересборка дня после переноса/удаления (#3840)

Перенос задания на другой день/станок (`moveCutToDay`) и удаление одной резки (`runDeleteCutTask`)
изымают резку из середины дня станка. Раньше они переписывали `planStart` только самого
перемещаемого задания, а **сохранённый `planStart` прочих резок дня
оставался прежним** — на месте вынутой резки висел простой. Страница планирования его тогда не
показывала (очередь до #3846 пересчитывала тайминг через `buildSchedule` на лету), а РМ «Диаграмма
Ганта (задания)» рисует **сохранённый** `planStart` — там дыра была видна. После #3846 обе РМ
показывают сохранённый план, поэтому такой простой виден на обоих экранах одинаково.

Теперь обе операции терминально вызывают `autoSequenceQueue(preserveOrder=true)` — тот же
пересчёт, что после генерации: `splitMachineQueue`/`buildSchedule` с `gapFill` пакуют день
**встык**, дыра схлопывается, `planStart` затронутых дней (источник + цель) сохраняется.
`preserveOrder` держит ручной порядок оператора **по сохранённому `planStart`** (#3923), фольгу —
в конце дня (#3717), зафиксированные задания — на своих днях (#3792); пишутся только изменившиеся
записи (идемпотентно, #3427).

**Ручная перестановка (#3923).** Кнопки ↑↓ (`moveCutInDay`) меняют порядок дня **обменом
сохранённого `planStart`** двух соседних резок, а перенос в начало/конец дня (`moveCutToDay`)
раскладывает день плейсхолдер-`planStart` в желаемом порядке; в обоих случаях терминальный
`autoSequenceQueue(preserveOrder=true)` переупаковывает день встык. Отдельного хранимого поля
порядка нет — намерение оператора сразу материализуется в `planStart`.

> Инвариант: сохранённый `planStart` дня станка всегда без внутреннего простоя — любая операция,
> изымающая резку из дня, обязана пересобрать день (а не «оставить на следующую генерацию»).

### Показ из сохранённого плана, без live-пересчёта (#3846)

Раньше очередь на странице планирования пересчитывала тайминг **на каждый рендер** через
`buildSchedule` (gapFill). Это расходилось с сохранённым планом, который рисует РМ «Диаграмма
Ганта»: (1) для первой резки бралась наладка «с нуля» (`firstCutSetup`), а не реальная заправка
станка из `prev_cut_setup`; (2) не учитывался обед (#3342). Итог — минуты и времена двух экранов
не совпадали (жалоба #3846), а 40-минутный обед, уже зашитый генерацией в `planStart`
послеобеденных резок, на Ганте выглядел необъяснённой «дырой».

Теперь обе РМ — **единый источник правды: сохранённый план**. Очередь строит тайминг
`scheduleFromStored(cuts, baseMidnight)` прямо из записанных полей (главное значение `planStart` =
окно/начало настройки; наладка = ножи + смена сырья; «Резка и Лидер» #3700 = намотка + лидер),
без `buildSchedule`. Окно резки = `setup + (намотка+лидер)`, ровно как ширина бара на Ганте →
минуты и времена **всегда совпадают**.

Единственное, что «считается» на показе, — **обед**: его не пересчитывают, а **находят** в уже
сохранённом расписании как зазор ≈ `LUNCH_DURATION` между концом окна одной резки и началом
следующей в том же дне (`lunchBlocksFromSchedule` в планировании, `ganttLunchMarkers` на Ганте) и
подписывают блоком «🍽 Обед · N мин» в **обоих** РМ (привязан к началу послеобеденной резки).

**Порядок раскладки — по времени `planStart` (#3920/#3923).** `scheduleFromStored`
устраняет нахлёст резок одного дня встык (#3885), обрабатывая их **строго по возрастанию
сохранённого `planStart`**. Это же время — единственный ключ порядка резки во всей системе:
«Очередность» больше не хранится (#3923). До её удаления она могла разойтись с `planStart` после
scope-ограниченной пересборки (#3660): «застрявшая» резка с ранней «Очередностью», но поздним
`planStart` (хвостовая настройка на 15:58, тогда как остальные резки дня стоят с 08:00). Анти-
нахлёст forward-only: попав в обработку **первой** (по «Очередности»), такая резка выталкивала за
собой **все** резки дня в овертайм (день лез до 23:15 — issue #3920, «Овертайм/недопланирование»).
С раскладкой по `planStart` резка стоит там, где записана — ровно как бар на Ганте, — и соседей не
двигает. (Сама «застрявшая» резка — след неполной пересборки #3885/#3895; лечится повторной
генерацией/«Упорядочить», которая перепланирует её на своё место.)

> **Инвариант (#3923): `planStart` (главное значение `t1078`) — ЕДИНСТВЕННЫЙ источник порядка
> резки.** Все три РМ — страница планирования, «Диаграмма Ганта», пульт слиттера — упорядочивают
> резки станко-дня **строго по возрастанию `planStart`**. «Очередность» не пишется, не читается и
> не участвует в сортировке нигде (`sequence` остаётся только как in-memory ординал генерации, см.
> Шаг 3.2). Страница планирования и Гант показывают одни и те же сохранённые поля; тайминг на
> показе НЕ пересчитывается (пересчёт — только в генерации и пересборке дня, #3840). Расхождение
> экранов = баг персистенции плана, а не показа.

### `planStart` пишет splitMachineQueue; computeCutSetupUpdates — только тайминг (#3853/#3855)

Обе РМ рисуют **окно** резки `setup + (намотка+лидер)` и ставят его по сохранённому
`planStart`. Источник `planStart` — **`splitMachineQueue`** (через `planCutOperations` →
`applySplitPlan`): только он знает дробление по дням, нахлёст настройки в хвост дня (#3805,
#3635 п.5), обед, переполнение, gap-fill. Сохранённые `Наладка ножей` / `Сырьё/намотка` /
`Резка и Лидер` пишет **`computeCutSetupUpdates`** (из `persistCutSetupColumns` после
генерации/пересборки) — он `planStart` НЕ трогает.

Две вещи, ломавшие совпадение Ганта и страницы:

1. **Штамп `planStart` с секундами.** `windowStartMin` дробный → штамп нёс `:SS`; Гант обрезал
   вниз, страница округляла вверх → ±1–2 мин. Фикс — `scheduleStartTimestamp` снапит штамп к
   **целой минуте, округляя ВВЕРХ** (единственная точка перевода `windowStartMin` → Unix-штамп).
2. **Расхождение настройки ПЕРВОЙ резки станка.** `splitMachineQueue` брал «ножи с нуля»
   (`firstCutSetup`), а окно — переналадку от реальной заправки (`prev_cut_setup` → `carryPrevCut`).
   Фикс #3853: генерация считает первую резку переналадкой от заправки (`carryPrevSetup` из
   `prevSetupBySlitter`) — ровно как окно в `computeCutSetupUpdates`.

> **Отменён подход #3855** «пересчитывать `planStart` по сохранённым окнам в
> `computeCutSetupUpdates` (`derivePlanStartTimestamps`)»: он наивно паковал каждый сегмент по
> полному окну `ножи+сырьё+резка` от 08:00, ИГНОРИРУЯ нахлёст настройки в хвост дня (#3805) и
> разрыв по дням (#3635 п.5), и перетирал корректный `planStart` от `splitMachineQueue` →
> setup-only-сегмент клал «и ножи, И сырьё» в один день с нахлёстом. `planStart` считает только
> `splitMachineQueue`; РМ читают сохранённое и **не пересчитывают на лету** (#3846).

> Инвариант: `planStart` пишет `splitMachineQueue` (со снапом к целой минуте вверх);
> `computeCutSetupUpdates` пишет ТОЛЬКО тайминг. Разрыв/перекрытие/±минуты — баг расписания
> (`splitMachineQueue`/окна), а не отдельного пересчёта старта.

## Особые правила

### Короткая фольга ≤ 120 минут

Бизнес-правило согласовано: если суммарное время резки фольги не превышает 120 минут, такую
фольгу можно ставить не в конец дня. Она не успевает сильно загрязнить станок.

Сейчас это **требование, а не код**: `orderCuts` отделяет любую `isFoil` в конец очереди, а
`splitMachineQueue` и дневная сортировка также считают любую фольгу фольгой. Для реализации
нужно ввести признак короткой фольги и трактовать ее как обычную резку:

- в `orderCuts` класть короткую фольгу в `rest`, а не в `foil`;
- в `selectByConfig` считать ее `isFoil = 0`;
- в сортировке `(день, isFoil, planStart)` не уводить ее в хвост;
- длинную фольгу оставлять в конце дня как сейчас.

### Втулка 0.5" и узкие риббоны (#3812, #3872)

Реализовано — полное описание в [Шаг 2.0](#шаг-20-втулочные-полосы-05--110-мм-3812). Кратко:

- риббоны на втулке 0.5" у́же 55 мм не производятся (позиция в `skipped`);
- фактическая ширина 55-57 мм режется как 55 мм;
- при втулке 110 мм для ширин 55-57 мм добавляют две полосы 110 мм, для 63-64 мм — одну;
- ширины свыше 70 мм идут по основному правилу для втулок 1";
- #3872: полосы 110 мм привязываются к уже заказанным позициям 110 мм того же заказа
  (обеспечение на произведённое); не нашлось — полосы синтетические.

### Отпуск станка (#3764, #3876, #3881, #3883)

«Отпуск» — подчинённая станку таблица окон простоя `[{ начало, окончание }]`
(`downtimesBySlitter`). В эти окна станок **не работает** (нет сырья и ножей).

**Частичный отпуск vs весь день (#3883).** Окно отпуска задаётся с ВРЕМЕНЕМ (datetime). Разделяем
два уровня: (а) **расписание** (#3764) сдвигает резки за окно ПО ВРЕМЕНИ — короткое окно (напр.
2 часа 08:00–10:00) лишь задерживает старт дня; (б) пометка «станок недоступен ВЕСЬ день»
(`slitterOnVacationDay` → `slitterDownOnDay`) — для выбора станка/балансировки/переноса/сброса
заправки — даётся ТОЛЬКО если окна отпуска **полностью покрывают рабочее окно резки**
`[startMin; cutEndMin]` этого дня. Частичный отпуск станок на день НЕ блокирует (он работает
остаток дня). Раньше любое пересечение суток считалось «весь день в отпуске», и 2-часовое окно
исключало станок из планирования совсем (Гант пустой во все дни).

- **Расписание (#3764).** Уже построенные размещения сдвигаются по времени за окно отпуска
  (`shiftPlacementsPastDowntime`), чтобы в отпуск ничего не наматывалось.
- **Загруженность по дате окончания (#3881, #3921).** При **выравнивании загрузки** «загруженность»
  станка считается как ДАТА ОКОНЧАНИЯ последней задачи (span целых дней + дробный `endPos` с учётом
  отпуска, #3921 — чтобы хвост за отпуском стекал на свободные станки), а не как сумма минут
  (см. [Шаг 3.1-bis](#шаг-31-bis-выравнивание-загрузки-станков-rebalanceslitterload-3848)).
  Станок с отпуском заканчивает позже — его не грузят на послеотпускное время, пока другие
  простаивают; станок в отпуске без задач после него «занят» до конца отпуска.
- **Не ставить задание на станок в его отпуск (#3876).** При **выборе станка**
  (`chooseSlitterBySetup`) станки, у которых в день резки отпуск, исключаются
  (`unavailableSlitterIds`; все в отпуске → откат, чтобы не потерять резку). При
  **выравнивании загрузки** (`rebalanceSlitterLoad`) задание не переносится на станок,
  у которого в день этого задания отпуск (`opts.slitterDayBlocked`). **Ручной перенос**
  (🗓 «Перенести задание») на станок, у которого в выбранный день отпуск, **блокируется**
  с сообщением. Признак «отпуск на весь день» — `slitterDownOnDay`: закрытые окна (с «Окончанием»)
  ПОЛНОСТЬЮ покрывают рабочее окно `[startMin; cutEndMin]` (#3883); окно без «Окончания» не
  блокирует, как и в расписании.
- **Заправка после отпуска (#3876, смягчено #3898).** Станок на время **длинного** отпуска
  без сырья и ножей: у станка, который на отпуске в день базы плана, заправка (`prev_cut_setup`)
  **обнуляется** (`planningPrevSetupBySlitter` → `longVacationOnDay`), поэтому **первая резка
  после отпуска** считает ПОЛНУЮ настройку — смена сырья + ножи с нуля (а не наследует заправку,
  бывшую до отпуска). Один источник и для плана (`splitMachineQueue`), и для хранимых колонок
  (`computeCutSetupUpdates`), и для модалки тайминга — времена совпадают.
- **Короткий отпуск НЕ сбрасывает заправку (#3898).** Отпуск длиной **не более
  `DOWNTIME_KEEP_SETUP_MAX_DAYS` (= 2) КАЛЕНДАРНЫХ дней** заправку сохраняет — после короткого
  простоя станок остаётся со своим сырьём/ножами, и первая резка не пересчитывает настройку с
  нуля. Длиннее порога → сброс (полная настройка). Длину окна меряет `downtimeSpanDays`
  (`vacationSpanDaysOnDay` — для дня базы); отпуск = одна запись окна `[начало; окончание]`.
  Порог вынесен в константу `DOWNTIME_KEEP_SETUP_MAX_DAYS` (легко менять). Частичный отпуск
  (#3883, не весь рабочий день) и так не считается за простой (`slitterOnVacationDay = false`).
  Сброс заправки в середине очереди (станок уходит в отпуск и возвращается уже внутри окна
  планирования) пока не реализован — учитывается отпуск на день базы плана.

## Возможный оптимизатор на 3 дня

Сейчас план строится жадно: раскладка → выбор станка → очередь → расписание. Это быстро и
понятно, но не всегда дает лучший глобальный план. Следующий шаг — улучшающий поиск на
скользящем окне примерно 3 дня.

### Целевая функция

```text
Cost(plan) = Σ(вес_фактора × нарушение_фактора(plan))
```

Жесткие ограничения из раздела «Ограничения алгоритма» фильтруют недопустимые планы. Мягкие
факторы из матрицы «Цена нарушения правил» дают штраф.

Примеры штрафов:

- срок: `max(0, день_резки − dueDay) × W1`;
- фольга: длинная фольга не в конце дня × `W2`;
- переналадки: сумма `changeoverCost(prev, next)`;
- баланс: отклонение загрузки станка от средней или факт `overflow`;
- дробление: число продолжений и setup-only сегментов.

### Соседние ходы

Оптимизатор может пробовать локальные перестановки и считать `ΔCost`:

1. поменять две резки местами внутри дня одного станка;
2. перенести резку на другой день того же станка;
3. перенести резку на другой станок того же дня;
4. поменять резки между станками;
5. сдвинуть длинную фольгу в конец или короткую фольгу раньше.

### Поиск и применение

Базовый вариант — hill-climbing: применять лучший ход с отрицательным `ΔCost`, пока план
улучшается. Если потребуется выходить из локальных минимумов, можно добавить simulated
annealing или tabu search с ограничением по времени/итерациям.

Результат должен быть списком обновлений в формате `planCutOperations`: резка, новый станок,
новый день, новая очередность. Применять можно через превью («минус N минут переналадок,
минус K опозданий») или отдельную кнопку «Оптимизировать на 3 дня».

## Глоссарий

| Термин | Значение |
|---|---|
| Джамбо | Исходный рулон сырья. Ширина джамбо — максимальная суммарная ширина полос одной резки. |
| Прогон (`run`) | Один проход через станок. Число прогонов примерно равно `qty / количество полос данной ширины`. |
| Скрап (`scrap`) | Полосы, нарезанные сверх заказа по незапасным ширинам. Это перепроизводство, которое некуда деть. |
| Отход (`waste`) | Незанятая ширина джамбо × прогоны. Идет в технологический обрезок. |
| Лидер | Лента для заправки слиттера перед следующей резкой. Учитывается после намотки текущей резки. |
| Ходовые ширины | Ширины из «Максимального запаса», которыми можно добирать остаток джамбо в складской запас. |
| `isFoil` | Признак фольги: грязное сырье, требующее постановки в конец дня, кроме планируемого исключения для короткой фольги. |
| `sleeveId` | Диаметр втулки. Все рулоны одной резки должны иметь один `sleeveId`. |
| `dueKey` | Числовой ключ срока изготовления (`ГГГГММДД`) для сортировки и кластеризации. |
| `continuationSignature` | Ключ идентичности резки для сборки цепочки продолжений при многодневном разбиении. |
| `fixed` | Признак «Зафиксировано». Закрепляет день резки и защищает от удаления, но не запрещает перестановку внутри дня. |

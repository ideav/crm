# Алгоритм планирования производства (слиттер, atex)

Описание алгоритма страницы **«Планирование производства»**
(`templates/atex/production-planning.html`, логика — `download/atex/js/production-planning.js`).
Документ объясняет: как формируется очередь резок, как считается время переналадок
и как строится расписание по сменам; а также где брать «журнал» и «сводную» по таймингу.

> Изменение #3600: **любая смена ножей теперь стоит фиксированные 30 минут** «на всё
> вместе», независимо от числа переставленных ножей (раньше #3472 — `KNIFE_MOVE × число
> перестановок`). Смена сырья/намотки считается **отдельно** (своя операция).

---

## 1. Вход

Очередь — массив резок (`cut`), каждая со свойствами (упрощённо):

| поле | смысл |
|---|---|
| `materialId` | сырьё |
| `winding` (`IN`/`OUT`) | намотка |
| `batchId` | партия сырья |
| `knifeCount`, `knifeWidths[]` | набор ножей (число и ширины полос) |
| `rollerWidth` | ширина ролика |
| `isFoil` | признак «Фольга» (планируется в конец) |
| `jumboRemainingM` | остаток джамбо (для «неудобного остатка») |

## 2. Очередь резок — `orderCuts(cuts, weights)`

1. **Фольга выносится в конец.** Резки делятся на `rest` и `foil`; очередь =
   `последовательность(rest)` + `последовательность(foil)`. Фольга всегда после всего
   остального.
2. **Порядок внутри группы** — `sequenceForStrategy` (жадный seed + ветви-и-границы по
   минимизации суммарной переналадки). Критерии (по убыванию приоритета):
   - **#3568: число ножей по убыванию — ПЕРВИЧНО.** Сложные резки (много ножей) идут
     раньше, даже если это разбивает группировку сырья.
   - **Минимизация переналадки (#3268/#3472→#3600)** — тай-брейк среди резок с равным
     числом ножей. Эмерджентно группирует одно сырьё (переход внутри сырья дешевле).
   - Стратегии: `PLANNING_STRATEGY_SETUP` (минимум переналадок), `…_FATIGUE` (сложные
     раньше).
3. Каждой резке проставляется `sequence = 1..N` (вход не мутируется).

`orderedChangeoverCost(cuts)` — суммарная переналадка для очереди после `orderCuts`
(детерминированный минимум, не зависит от исходного порядка).

## 3. Переналадка `prev → next` — `changeoverParts` / `changeoverCost`

Переналадка раскладывается на независимые операции (минуты). Без переналадки идёт только
**самая первая резка очереди** (`prev` нет → `[]`). У всех остальных резок переналадка
считается относительно предыдущей — в том числе у **первой резки нового дня**: её `prev`
— это последняя резка предыдущего дня (см. §4). Граница смены сама по себе переналадку не
обнуляет: первая резка дня идёт без переналадки **только если последняя резка предыдущего
дня была с той же конфигурацией**:

- **смена сырья экономится**, если сырьё и намотка остались вчерашними (нет `MATERIAL_WINDING`);
- **смена ножей экономится**, если число ножей и интервалы (ширины полос) остались как вчера (нет `KNIFE`);

иначе соответствующая операция применяется и её время считается обычным `changeoverCost`.

| операция | когда применяется | время |
|---|---|---|
| `MATERIAL_WINDING` | сменилось сырьё **ИЛИ** намотка **ИЛИ** партия | `MATERIAL_WINDING` (15) |
| `KNIFE` | сменился набор ножей (`knifeMoves > 0`) **ИЛИ** сузился ролик | **`KNIFE` фикс. 30 (#3600)** |

- Обе операции **бинарны** (изменилось / нет) и **складываются**:
  - смена сырья → 15;
  - смена ножей → 30;
  - сырьё + ножи → **45** (15 + 30).
- `changeoverCost(prev, next, times)` = сумма минут компонентов.
- До #3600 нож считался как `KNIFE_MOVE(2) × число переставленных ножей`; теперь число
  ножей на время **не влияет** — любая переналадка ножей = 30.

## 4. Расписание по сменам — `buildSchedule(cuts, opts)`

По упорядоченной очереди раскладывает резки во времени:

- **Окно смены** `shiftStartMin … shiftEndMin` (по умолчанию рабочая смена); резки
  пакуются по порядку, не помещается сегодня → перенос на следующий рабочий день.
- **Лидер между резками** `BETWEEN_CUTS` (2 мин, #3401) заправляется перед **каждым
  проходом**, поэтому входит во время одного прохода и множится на число проходов
  (`Кол-во план`).
- **Переналадка** перед резкой = `changeoverCost(prev, next)` (§3). На границе дня `prev` —
  последняя резка предыдущего дня, поэтому первая резка дня получает переналадку, если её
  конфигурация (сырьё/намотка, набор ножей) отличается от вчерашней.
- **Плавающий обед** (#3342) и **уборка в конце дня** `CLEANUP_SHIFT` (30 мин, #3155).
- **Фольга — в конец смены** (см. orderCuts).

**Длительность резки считается «от одного прохода».** Сначала — время одного прохода:
`лидер + намотка` = `BETWEEN_CUTS` + намотка прохода. Затем — общая длительность резки:
`(лидер + намотка) × проходов`. Переналадка (§3) добавляется одноразово перед резкой.

## 5. Времена операций — `DEFAULT_OP_TIMES`

Дефолты (реальные значения берутся из таблицы времён, передаются как `times`):

```js
{ MATERIAL_WINDING: 15, KNIFE: 30, KNIFE_MOVE: 2, BETWEEN_CUTS: 2, CLEANUP_SHIFT: 30 }
```

После #3600 нож использует **`KNIFE`** (30, фикс.); `KNIFE_MOVE` в стоимости
переналадки больше **не участвует** (оставлен в таблице для совместимости).

## 6. Журнал и сводная по таймингу

- **Журнал перехода** (порезочно): `changeoverParts(prev, next, times)` —
  `[{ code, label, minutes }]` только применившихся операций («смена сырья / намотки /
  партии», «смена ножей / сужение ролика»). Полный setup резки —
  `setupBreakdown(prev, next, times)` (переналадка + лидеры).
- **Сводная по очереди:** `orderedChangeoverCost(cuts)` — суммарные минуты переналадок;
  `buildSchedule` даёт по дню/слиттеру итоги (время резки, переналадок, всего, окно
  смены, перенос на следующий день).

### Тайминг резки — «от одного прохода»

Реквизит/модалка «Тайминг» считает время одного прохода (`лидер + намотка`) и умножает
на число проходов. Пример: метраж прохода 122 м, 15 проходов, норма намотки
`WIND_300 = 1.2 мин` (намотка прохода 0.488 мин), лидер `BETWEEN_CUTS` = 2 мин:

```
Метраж прохода: 122 м
Плановых проходов: 15
Намотка 1 прохода: 0.488 мин
Норма намотки: WIND_300=1.2 мин
Лидер + намотка: 2.488 мин

Тайминг окна:
09:08 · резка: 2.488 * 15 = 37.32 мин
09:45 · готово
```

Переналадка перед резкой (раздел 3) — если конфигурация отличается от предыдущей резки —
показывается отдельной строкой тайминга окна **перед** строкой «резка».

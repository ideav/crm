# Сессия 2026-05-16 — issue #2689: догрузка и недостающие сущности в export_b3x

## Контекст

Issue [#2689](https://github.com/ideav/crm/issues/2689) — три проблемы скрипта `export_b3x.php` (выгрузка из Битрикс24 в CSV):

1. После полной выгрузки скрипт сразу показывает «ВСЕ ВЫГРУЖЕНО» и не догружает новые лиды/сделки, появившиеся позже.
2. Департаменты и пользователи не выгружаются вовсе — нужны полностью, не инкрементально.
3. Задачи не выгружаются — нужны инкрементально по тому же шаблону, что лиды/сделки.

## Что сделали

Каждый пункт — отдельный PR от свежего `main` (правило: в `ideav/*` каждый коммит = отдельный PR, мерджится мгновенно).

### PR #2690 — догрузка лидов/сделок при повторном запуске
- В `export_b3x.php` новая чистая функция `prepareResumeAfterComplete($state)`: если прошлый прогон помечен `is_complete=true` и `last_lead_id`/`last_deal_id` > 0, снимает complete-флаги, сохраняя `last_*_id`. Существующий фильтр `>ID` подхватывает только то, что появилось после прошлого прогона.
- Удалён ранний `exit` при `is_complete=true` — основная петля теперь сама проходит и выводит итоги/ссылки.
- В шапке добавлена строка `[ДОГРУЗКА]` при срабатывании режима резюма.
- Условие очистки CSV (`$lastLeadId == 0 && empty(...)`) не задето: в режиме догрузки `last_lead_id > 0`, файлы сохраняются.
- Тест `experiments/test-issue-2689-resume.php` — 4 сценария.

### PR #2691 — `export_b3x_departments.php`
Новый отдельный скрипт. `department.get` с пагинацией через `start`/`next`, перезапись `departments.csv` целиком при каждом запуске. Хелперы переиспользуются из `export_b3x.php` через `EXPORT_B3X_SKIP_RUN`. Pagination-guard на 1000 итераций. Тест: пагинация, формат CSV, факт перезаписи, guard.

### PR #2692 — `export_b3x_users.php`
Новый скрипт. `user.get` в два прохода (`FILTER=ACTIVE:true`, `ACTIVE:false`) — чтобы попали и уволенные. Дедупликация по ID. Атомарная перезапись `users.csv` через `.tmp` + `rename`. Тест: пагинация двух фильтров, dedup, формат CSV, факт перезаписи, guard.

### PR #2693 — `export_b3x_tasks.php`
Новый скрипт по шаблону `export_b3x.php`: `tasks.task.list` с фильтром `>=CREATED_DATE` по году + `>ID` для инкремента. Отдельные state-файл (`export_tasks_state.json`) и error counter — не пересекаются с лидами/сделками. Resume-режим аналогично PR #2690.

**Особенности tasks.task.list, учтённые в коде:**
- Массив задач лежит в `result.tasks`, а не плоско в `result`. Парсер делает fallback на плоский формат.
- Ключи задач в lowercase (`id`, `title`, `createdDate`, ...). Поэтому `taskFieldsMap` и `select` в lowercase. Русские названия — только в headers CSV.

Тест: метод/фильтр/order, парсинг обоих форматов ответа, 4 сценария resume, типизация state.

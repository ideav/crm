# Отчёты (Запросы) в Integram: методичка, аудит и пилот atex

Дата: 2026-06-01. Общий документ по отчётам Integram; примеры — на боевой БД
ateh (`https://ideav.ru/ateh`).

В Integram **«запрос» (отчёт)** — это объект системной таблицы **22**, который ядро
разворачивает в SQL `SELECT` с автоматическими `LEFT JOIN` по метаданным.
Запуск: `GET /{db}/report/{queryId|имя}?JSON_KV` → плоские сджойненные строки.

## 1. Текущий статус: отчёты не используются

Все рабочие места atex читают данные напрямую через `object/{table}/?JSON_OBJ` и
**джойнят/агрегируют на клиенте**. Слой `report/` упомянут комментарием
«перевод чтений на защищённый слой `report/` — следующий этап» в 10 файлах
(cut-calc, orders, warehouse, slitter, portal, cut-map, intake,
production-planning, sleeve-cutter, dashboards), но нигде не задействован.

## 2. Где отчёты выгодны (из аудита, по убыванию выгоды)

| РМ | Что делает сейчас | Выгода от отчёта |
|----|-------------------|------------------|
| **dashboards.js** | `loadAll` (стр. 461–468) тянет **6 таблиц целиком** + 4 `_count`; `makeRow` (225–274) джойнит Заказ→Позиция→Обеспечение→Резка→ГП в JS | 1 отчёт вместо 6 полных выгрузок и клиентских джойнов |
| **production-planning.js** | `loadPositions`(2000)+`loadCuts`(2000)+`loadSupplies`(5000), джойн на клиенте | отчёт «очередь резок с позициями и обеспечением» |
| **portal.js / orders.js** | Заказ→Позиции | отчёт «заказы с позициями и статусом исполнения» |
| cut-map.js | Резка→Тип резки→Полосы (зависимые загрузки) | кандидат послабее |

Общая выгода: меньше round-trip’ов; серверный join/агрегация (не тянуть тысячи
строк на клиент); защищённый слой (отчёт уважает права роли); нет клиентских
join-багов.

### Правило: минимизируй серверные запросы

Тяни данные **минимумом отчётов** (`report/`), а агрегации и счётчики считай **на
клиенте**. Фронт-вычисления дёшевы и никого не нагружают; разовая трудоёмкость
кода окупается годами сниженной нагрузки на сервер. Если виджету не хватает поля —
добавь колонку в существующий отчёт, а не заводи лишний запрос. Считай `*_id`
(функция abn_ID) и дедуплицируй строки на клиенте вместо отдельных выгрузок.

Пример: рабочее место «Дашборды» (`download/atex/js/dashboards.js`) заменило ~10
запросов (6 выгрузок `object/` + 4 `_count`) на **2 отчёта** (`order_pipeline`,
`material_stock`); все 5 виджетов агрегируются на клиенте из строк (dedup по `*_id`).

## 3. Соглашение об именах (правило проекта)

Имя колонки служит сразу **ключом в `JSON_KV`** и **именем внешнего фильтра
`FR_/TO_` в URL**, поэтому имена — латиница, `snake_case`, без пробелов.

- **Отчёт (поле 22):** `<область>_<суть>` — напр. `order_pipeline`. Без префикса
  базы (внутри одной БД он у всех одинаков и ничего не различает).
  Имя используется и в URL (`report/{имя}`). Переименовать готовый запрос —
  `_m_save/{queryId}?JSON t22=новое_имя` (именно `_m_save` и `t22` — id типа).
- **Колонка (`t100`):** `<сущность>_<поле>`. Префиксы сущностей:
  `order_`, `position_`, `provision_`, `cut_`, `gp_`. ID-колонка — `<сущность>_id`.
  Единицы измерения суффиксом: `_mm`, `_m`. Примеры: `order_id`, `order_status`,
  `position_width_mm`, `provision_used_m`, `gp_arrived`.
- **Почему латиница/snake_case:** исключает URL-кодирование кириллицы и подмену
  пробелов на `_` (ядро делает это для `t100` с пробелами); префикс сущности
  снимает коллизии имён в join-отчёте.

## 5. Как создать отчёт (проверенная последовательность)

```bash
DB=https://ideav.ru/ateh; TOKEN=<актуальный>
XSRF=$(curl -s -H "X-Authorization: $TOKEN" "$DB/xsrf?JSON=1" | jq -r ._xsrf)

# 1. Запрос (тип 22). up=1 ОБЯЗАТЕЛЕН; имя — slug по соглашению.
curl -s "$DB/_m_new/22?JSON&up=1" --data-urlencode "t22=order_pipeline" \
  --data-urlencode "token=$TOKEN" --data-urlencode "_xsrf=$XSRF"   # → {"id": <queryId>}

# 2. Колонка (тип 28). up=queryId; t28 = ID таблицы ИЛИ реквизита; t100 = имя.
#    Порядок колонок = порядок join слева-направо; первая реальная колонка — мастер-таблица.
curl -s "$DB/_m_new/28?JSON&up=<queryId>" --data-urlencode "t28=<fieldId>" \
  --data-urlencode "t100=order_status" \
  --data-urlencode "token=$TOKEN" --data-urlencode "_xsrf=$XSRF"   # → {"id": <colId>}

# 3. ID сущности — функция abn_ID. «Функция» (t104) — это ССЫЛКА; значение = 85
#    (НЕ строка "abn_ID"). Для ref-колонки abn_ID даёт ID справочного значения.
curl -s "$DB/_m_set/<colId>?JSON" --data-urlencode "t104=85" \
  --data-urlencode "token=$TOKEN" --data-urlencode "_xsrf=$XSRF"

# 4. Формат отображения (опц.): _m_set/<colId> t84=DATETIME|DATE|NUMBER|…
```

**Фильтрация при запуске** (только по колонкам с заданным `t100`):

```
GET /{db}/report/{id}?JSON_KV&FR_{колонка}={значение}     # нижняя граница (FROM)
GET /{db}/report/{id}?JSON_KV&TO_{колонка}={значение}     # верхняя граница (TO)
# напр.: ?JSON_KV&FR_provision_status=Выполнено
#        ?JSON_KV&FR_cut_id=1900&TO_cut_id=2000           # диапазон по числовому ID
```

**⚠️ Открытый интервал — нужны операторы `>` / `<`** (проверено на бою, отчёт
`orders_list` по дате):

- **Обе границы заданы → диапазон без операторов:** `FR_{колонка}={С}&TO_{колонка}={По}`.
- **Только нижняя граница (открыт сверху) → оператор `>`:** `FR_{колонка}=>{С}`.
- **Только верхняя граница (открыт снизу) → оператор `<`:** `TO_{колонка}=<{По}`.

Одиночные `FR_{колонка}={значение}` / `TO_{колонка}={значение}` БЕЗ оператора
трактуются как **точное совпадение** (для дат/чисел открытого интервала это даёт
0 строк; для текстового статуса exact-совпадение — рабочее поведение).

```
# напр.: ?JSON_KV&FR_order_created=>2026-05-01                 # с 1 мая и позже
#        ?JSON_KV&TO_order_created=<2026-06-01                 # до 1 июня
#        ?JSON_KV&FR_order_created=2026-05-01&TO_order_created=2026-05-31  # май (диапазон)
```

ID компактнее символьных значений и удобнее для фильтра — поэтому для отбора по
сущности кладём `<сущность>_id` (abn_ID), а символьные значения — только если
нужны для показа.

Полный справочник полей колонки (тип 28: `t100`, `t58` ALIAS, `t84` формат,
`t101` формула, `t104` функция, агрегации, сортировка `t109`, фильтры `t102/t103`)
— `docs/integram-app-workflow.md`, раздел «Создание запроса программно».

## 6. Найденные грабли

- **`up=1` обязателен** при создании запроса (тип 22), иначе ошибка `up=0`.
- **`abn_ID` — это ссылка на функцию, значение `t104=85`** (не строка «abn_ID»;
  строка молча не сохраняется).
- **`JSON_KV` отдаёт СЫРЫЕ значения** — формат из `t84` виден только в
  платформенном `report/{id}?JSON` (поле `format`). DATETIME-штамп (`gp_arrived`)
  форматировать **на клиенте** через `AtexRefSearch.formatDateTime`.
- ⚠️ **Не форматировать дату формулой `t101=FROM_UNIXTIME([THIS],…)`** — ломает
  join (отчёт схлопывается в одну строку).
- **Переименование отчёта** — имя это главное значение записи (тип 22), поэтому
  меняется через `_m_save/{queryId}?JSON t22=новое_имя` (именно `_m_save`, не
  `_m_set`; параметр `t22` — id типа, не `t{recordId}`; `up=1` не нужен).

## 7. `preferable_widths` (queryId 8421) — ходовые ширины по сырью

Отчёт для РМ «Расчёт резки» (ideav/atex#52, подзадача B): какие ширины ролика и в

типу намотки и длине, `SUM(Кол-во)` по убыванию, внешние фильтры по виду сырья,
типу намотки и длине.

|---|---|---|---|
| `position_width_mm` | 1141 Ширина, мм | — (ключ группировки, для показа) | — |
| `position_width_id` | 1141 Ширина, мм | 85 (abn_ID) | — |
| `position_material_id` | 1138 Вид сырья | 85 (abn_ID) | — (внешний фильтр `FR_`) |
| `wind_dir` | 8463 Тип намотки | — | — (внешний фильтр `FR_`) |
| `wind_length` | 1143 Длина, м | — | — (внешний фильтр `FR_`) |
| `position_qty_sum` | 1137 Кол-во | 73 (SUM) | −1 (по убыванию) |

Имена — latin `snake_case`, поэтому `FR_position_material_id`, `FR_wind_dir` и
`FR_wind_length` в URL не требуют кодирования кириллицы.

Создание (проверенная последовательность, `DB=https://ideav.ru/ateh`):
```bash
XSRF=$(curl -s -H "X-Authorization: $TOKEN" --cookie "idb_ateh=$TOKEN" "$DB/xsrf?JSON=1" | jq -r ._xsrf)
# Запрос (тип 22), up=1
QID=$(curl -s -H "X-Authorization: $TOKEN" --cookie "idb_ateh=$TOKEN" "$DB/_m_new/22?JSON&up=1" \
  --data-urlencode t22=preferable_widths --data-urlencode token=$TOKEN --data-urlencode _xsrf=$XSRF | jq -r '.id // .obj')
# Колонки (тип 28), up=QID: t28=реквизит, t100=имя
#   position_width_mm←1141, position_width_id←1141, position_material_id←1138,
#   wind_dir←8463, wind_length←1143, position_qty_sum←1137
# Функции/сортировка через _m_set: t104=85 (abn_ID) для *_id; t104=73 (SUM) + t109=-1 для position_qty_sum
```
Запуск:
`GET /ateh/report/preferable_widths?JSON_KV&FR_position_material_id={видСырьяID}&FR_wind_dir={IN|OUT}&FR_wind_length={длина}`.
`{видСырьяID}` — ID записи «Вид сырья» (= `position_material_id`/abn_ID из ответа),
`{длина}` — требуемая длина позиции в метрах. Незаданный фильтр расширяет выборку.
Клиент `production-planning.js` дополнительно отбрасывает строки с другим `wind_dir`
или `wind_length`, чтобы не использовать ходовые другого профиля. Создано и проверено
2026-06-01; профильные колонки добавлены для #3219/#3221 (queryId 8421).

## 10. SET-действия (запрос как триггер: «Согласовать», смена статуса)

Отчёт может не только читать, но и **изменять** данные: у колонки задаётся
SET-выражение (поле `t132`), которое применяется ко всем записям под фильтром —
**только при `confirmed=1`**. Без `confirmed` отчёт возвращает превью затрагиваемых
строк.

```
# Превью (что изменится):
GET  /{db}/report/{name}?JSON_KV&FR_{колонка}={id}
# Применить:
POST /{db}/report/{name}?JSON_KV     Body: confirmed=1&FR_{колонка}={id}&_xsrf=...
```

SET-выражение (`t132`): литералы в кавычках (`"Согласован"`), даты — встроенными
значениями `[TODAY]`/`[NOW]` (см. список в §«Формулы»). Создание SET-колонки:
`_m_new/28?up={qid} t28={источник} t100={имя} t132={выражение}`.

**⭐ Запрос выполняется с правами ВЛАДЕЛЬЦА**, поэтому SET-отчёт меняет поле, даже
если оно **READ** для текущей роли. Это штатный приём: поле делают read-only для
роли (чтобы не правили вручную), а изменение идёт **только** через кнопку →
SET-отчёт с `confirmed=1`.

**Понизить отдельное поле до READ** (при WRITE на таблицу): добавить роли грант на
**колонку** (reqId), а не таблицу — запись в таблице 116:
`POST _m_new/116?up={roleId} t116={reqId} t136=53` (53=READ). Уровень таблицы
наследуется на колонки, грант на колонку понижает конкретное поле.

**Пример (ateh, РМ orders, 2026-06-02):** кнопка «Согласовано» вызывает отчёты
`approve_order` (`FR_order_id`) / `approve_position` (`FR_position_id`) — SET
ставит Статус=«Согласован/Согласована» + Дата согласования=`[TODAY]`. Поля Статус и
Дата согласования у роли Менеджер — READ (гранты на reqId в таблице 116), поэтому
менеджер меняет их только кнопкой (SET под правами владельца), но не вручную.

# Рабочие места atex — какая версия актуальная и как их объединить

**Задача:** [#2970](https://github.com/ideav/crm/issues/2970) — «Разобраться, где
наиболее полная и актуальная версия рабочих мест».
**Родительская задача:** [#2903](https://github.com/ideav/crm/issues/2903) —
«Подключи рабочие места по API».
**Связанный документ:** [`docs/atex_workplaces.md`](atex_workplaces.md) —
декомпозиция рабочих мест на тикеты.

---

## 0. TL;DR

Существует **три** набора шаблонов рабочих мест atex, созданных в ходе одного
проекта:

| # | Где лежит | Что это | Последнее изменение |
|---|---|---|---|
| **A** | [`ideav/crm` → `templates/atex/`](https://github.com/ideav/crm/tree/main/templates/atex) | Декомпозиция на 10 рабочих мест, подключённых к Integram API | **2026-05-30** |
| B | [`ideav/atex` → `templates/ateh/`](https://github.com/ideav/atex/tree/main/templates/ateh) | 8 ролевых макетов, копия `atex/` с `{_global_.z}` и id `ateh-*` | 2026-05-29 |
| C | [`ideav/atex` → `templates/atex/`](https://github.com/ideav/atex/tree/main/templates/atex) | 8 ролевых статичных макетов с хардкод-данными, id `atex-*` | 2026-05-28 |

> **Вывод: актуальная версия — A (`ideav/crm/templates/atex/`).** Она самая
> свежая, декомпозирована по тикетам #2903, и единственная **реально работает с
> базой по API** (а не показывает захардкоженные демо-данные). Версии B и C из
> репозитория `ideav/atex` — более ранние статичные макеты, на основе которых
> сделана A; их следует считать **архивными** и не развивать.

Объединять нужно **в A**. Ниже — обоснование, карта соответствия рабочих мест и
конкретный план.

---

## 1. Что представляет собой каждая версия

### 1.1 Версия A — `ideav/crm/templates/atex/` (актуальная)

Создана 2026-05-30 в рамках тикетов #2911–#2920 (дочерние к #2903). Отличия:

- **Декомпозиция по функциям/тикетам, а не по ролям.** 10 шаблонов, каждый —
  отдельное рабочее место из таблицы раздела 2 [`atex_workplaces.md`](atex_workplaces.md):

  | Шаблон | Рабочее место | Роль | Тикет |
  |---|---|---|---|
  | `orders.html` | Приём и ведение заказов | Менеджер | #2911 |
  | `cut-calc.html` | Калькулятор типов резки | Диспетчер | #2912 |
  | `production-planning.html` | Планирование производства | Диспетчер | #2913 |
  | `intake.html` | Приёмка сырья | Кладовщик | #2914 |
  | `slitter.html` | Пульт слиттера | Оператор | #2915 |
  | `sleeve-cutter.html` | Пульт втулкореза | Оператор | #2916 |
  | `cut-map.html` | Карта раскроя | Оператор | #2917 |
  | `warehouse.html` | Склад готовой продукции | Кладовщик | #2918 |
  | `dashboards.html` | Дашборды и отчёты | Руководитель | #2919 |
  | `portal.html` | Клиентский портал | Клиент | #2920 |

- **Подключение к API.** Шаблоны — это тонкая разметка; логика грузится из
  отдельных JS-файлов и реально ходит в Integram (`_m_new`/`_m_set`/`object/…`).
  Реквизиты резолвятся из `metadata/{tableId}` по имени, поэтому код переживает
  пересборку базы (id таблиц — в `data-*` атрибутах корня).
- **Встраивается в `main.html`** штатным механизмом Integram через
  `{_global_.z}` и `{_global_.version}`.
- Администратор отдельного рабочего места **не имеет** — справочники ведутся
  штатным UI Интеграма (см. раздел 1 [`atex_workplaces.md`](atex_workplaces.md)).

### 1.2 Версия C — `ideav/atex/templates/atex/` (исходные макеты)

Создана 2026-05-28 (`Move templates into atex namespace`). 8 **ролевых**
HTML-файлов по 40–67 КБ — самодостаточные макеты (весь CSS/JS встроен в один
файл), **с захардкоженными демо-данными и без вызовов API** (`fetch` в коде нет).
Идентификаторы — `atex-*`. Это «дизайн-прототипы» рабочих мест.

### 1.3 Версия B — `ideav/atex/templates/ateh/` (копия C под базу `/ateh/`)

Создана 2026-05-29 как **прямая копия C** с двумя изменениями (см.
[`templates/ateh/README.md`](https://github.com/ideav/atex/blob/main/templates/ateh/README.md)):

- идентификаторы CSS/DOM переименованы `atex-*` → `ateh-*`;
- хардкод имени базы заменён на `{_global_.z}`.

По собственному README это **всё ещё статичные макеты**: «API-привязка ещё не
сделана». Исключение — `slitter-operator-workplace.html`, который успели
подключить к API (diff к C: +210 строк против ±10 у остальных). README версии B
ценен схемой (`Карта таблиц`, ключи `t{reqId}`) — её стоит сохранить.

> Соотношение B и C: **B новее и чуть полнее C** (один экран подключён к API,
> добавлен README со схемой), но обе версии из `ideav/atex` концептуально
> устарели после рефакторинга в A.

## 2. Почему актуальна именно A

| Критерий | A (`crm/templates/atex`) | B (`atex/ateh`) | C (`atex/atex`) |
|---|---|---|---|
| Дата последнего изменения | **2026-05-30** | 2026-05-29 | 2026-05-28 |
| Работа с базой по API | **да** (все 10) | только slitter | нет |
| Соответствие декомпозиции #2903 | **да** (тикет на место) | нет (по ролям) | нет (по ролям) |
| Резолв реквизитов из `metadata` | **да** | нет | нет |
| Встраивание в `main.html` Integram | **да** | частично | нет |
| Покрытие функций | 10 рабочих мест | 8 ролевых экранов | 8 ролевых экранов |

A — единственная версия, выполняющая цель родительской задачи #2903 (рабочие
места, **работающие с базой по API**). B и C — её предшественники.

## 3. Карта соответствия: ролевые экраны (B/C) → рабочие места A

Версии B/C сгруппированы по ролям, A — по функциям. Один ролевой экран B/C
обычно соответствует нескольким рабочим местам A:

| Ролевой экран B/C | Рабочие места A |
|---|---|
| `manager-workplace` | `orders` |
| `dispatcher-workplace` | `cut-calc` + `production-planning` |
| `slitter-operator-workplace` | `slitter` |
| `tube-cutter-workplace` | `sleeve-cutter` |
| `warehouse-workplace` | `intake` + `warehouse` |
| `director-workplace` | `dashboards` |
| `client-portal` | `portal` |
| `admin-workplace` | — (нет аналога: справочники ведутся штатным UI) |
| — | `cut-map` (отдельного ролевого экрана в B/C нет) |

Все функции B/C покрыты в A. Уникальное в B/C, чего нет в A:

- `admin-workplace` (B/C) — в A намеренно отсутствует (см. §1.1). Переносить не
  нужно, но решение стоит зафиксировать.
- **Схема БД** в [`templates/ateh/README.md`](https://github.com/ideav/atex/blob/main/templates/ateh/README.md)
  (карта таблиц, ключи реквизитов) и спека
  [`2026-05-26-atex-schema-setup-design.md`](https://github.com/ideav/atex/blob/main/docs/superpowers/specs/2026-05-26-atex-schema-setup-design.md)
  — справочный материал, полезный для A.
- **Скриншоты и макет-доки** (`docs/screenshots/*`,
  `docs/superpowers/mockups/*`) — визуальный эталон рабочих мест.

## 4. Рекомендации по объединению (в `ideav/crm/templates/atex/`)

1. **Признать A канонической версией.** Развитие рабочих мест ведём только в
   `ideav/crm/templates/atex/`. Версии B и C в `ideav/atex` — заморозить
   (пометить в их README как архивные со ссылкой на A), чтобы не плодить третью
   точку правды.
2. **Перенести в A только то, чего там нет:**
   - схему БД из `ateh/README.md` — как справочник реквизитов рядом с
     `docs/atex_metadata.json` (или ссылкой из `atex_workplaces.md`);
   - скриншоты/макеты из `ideav/atex/docs/` — как визуальный эталон для ревью.
3. **Не переносить** ролевые HTML B/C целиком: их функциональность уже
   реализована в A в виде API-подключённых рабочих мест; обратный порт статичных
   макетов был бы регрессом.
4. **Зафиксировать отсутствие `admin`-места** в A как осознанное решение.

## 5. Рекомендации по `download/atex/` (скрипты и стили)

Задача #2970 отдельно просит «положить скрипты и стили в
[`download/atex/`](https://github.com/ideav/crm/tree/main/download/atex)».
Все 10 рабочих мест используют единую схему `download/{_global_.z}/…`:

| Схема размещения | CSS | JS | Рабочие места |
|---|---|---|---|
| **`download/{_global_.z}/…`** | `download/atex/css/*.css` | `download/atex/js/*.js` | все 10 |

> Почему `download/{_global_.z}/…`, а не глобальные `/css`,`/js`: каталог
> `download/{db}/` изолирует ассеты конкретной базы (`{_global_.z}` = имя БД,
> напр. `atex`), не засоряет общие каталоги Integram и совпадает с прямой
> просьбой #2970.

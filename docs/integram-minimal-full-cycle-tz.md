# Техническое задание: «Мини-Atex — резка рулонов»

> **Реверс-инжиниринг.** Документ восстанавливает требования к продукту, описанному в
> [`integram-minimal-full-cycle-project.md`](integram-minimal-full-cycle-project.md): что система
> должна делать, из чего состоит и как принимается. Источник описывает уже собранное приложение —
> здесь те же факты изложены как требования «до разработки».

**Основание:** задача [#2986](https://github.com/ideav/crm/issues/2986); проект Atex, упрощённый до
минимального состава полного цикла.
**Платформа:** low-code **Интеграм** (таблицы БД + HTML/JS рабочие места, авторизация по токену).
**Связанные документы:** [`integram-app-workflow.md`](integram-app-workflow.md) (рабочий регламент
сборки), [`atex_metadata.json`](atex_metadata.json) (полная схема Atex — следующий этап).

---

## 1. Назначение и цели

1.1. **Цель.** Автоматизировать сквозной путь обработки заказа на резку термобумажных рулонов:
приём заказа → планирование производственной резки → выполнение → контроль руководителем.

1.2. **Объект автоматизации.** Процесс резки рулонов под заказы клиентов в небольшом производстве.

1.3. **Назначение системы.** Заменить ведение заказов и резок в Excel единым приложением с общей
моделью данных, ролевым доступом и рабочими экранами под каждую роль.

1.4. **Границы (минимальный состав).** В объём входят только таблицы и экраны, достаточные для
одного полного прохода. **Не входят** (выносятся в следующий этап — полный Atex): склад,
калькулятор раскроя, втулкорез, клиентский портал, расширенная аналитика.

1.5. **Исходные данные заказчика.** Заказчик не описывает систему технически, а передаёт предметную
область и два Excel-файла:
- `Справочники.xlsx` — листы `Клиенты` (Клиент, Телефон, Email) и `Сырьё` (Вид сырья, Полное
  название, Ширина мм, Длина рулона м);
- `Заказы.xlsx` — листы `Заказы` (Номер заказа, Клиент, Дата создания, Статус, Менеджер,
  Комментарий) и `Позиции` (Номер заказа, Позиция, Вид сырья, Ширина мм, Длина м, Кол-во, Статус).

---

## 2. Термины

| Термин | Значение |
|---|---|
| **Справочник** | Таблица с повторно используемыми значениями (Клиент, Вид сырья). |
| **Ссылка (ref)** | Поле-указатель на запись другой таблицы. |
| **Подчинённая таблица (arr)** | Таблица, записи которой принадлежат родителю (`up=<id родителя>`). |
| **Реквизит** | Поле таблицы. |
| **Рабочее место (РМ)** | HTML/JS-экран под конкретную роль. |
| **Роль** | Набор прав и пунктов меню для группы пользователей. |

---

## 3. Пользователи (роли)

| Роль | Логин | Назначение |
|---|---|---|
| Менеджер | `manager` | Принимает заказы, добавляет позиции. |
| Диспетчер | `dispatcher` | Планирует производственную резку по позициям. |
| Оператор | `operator` | Выполняет резку, фиксирует факт. |
| Руководитель | `director` | Контролирует статусы, только просмотр. |
| Администратор | — | Системная роль для настройки. |

---

## 4. Требования к модели данных (ТД)

**ТД-1.** Система строится на пяти таблицах: `Клиент`, `Вид сырья`, `Заказ`, `Позиция заказа`,
`Производственная резка`. Названия совпадают с полным Atex для последующего расширения.

**ТД-2. Таблица `Клиент`** (справочник): `Клиент` (SHORT, имя записи), `Телефон` (SHORT),
`Email` (SHORT).

**ТД-3. Таблица `Вид сырья`** (справочник): `Вид сырья` (SHORT, имя), `Полное название` (SHORT/MEMO),
`Ширина, мм` (NUMBER), `Длина рулона, м` (NUMBER).

**ТД-4. Таблица `Заказ`**: `Заказ` (SHORT, номер), `Клиент` (**ref → Клиент**),
`Менеджер` (**ref → Пользователь**), `Дата создания` (DATE), `Статус` (SHORT),
`Комментарий` (MEMO), `Позиция заказа` (**arr → Позиция заказа**, подчинённая таблица).

**ТД-5. Таблица `Позиция заказа`**: `Позиция заказа` (SHORT, имя), `Вид сырья` (**ref → Вид сырья**),
`Ширина, мм` (NUMBER), `Длина, м` (SIGNED), `Кол-во` (NUMBER), `Статус` (SHORT: `Новая`,
`Запланирована`, `Готова`).

**ТД-6. Таблица `Производственная резка`**: `Производственная резка` (NUMBER, автонумер `unique=1`),
`Позиция заказа` (**ref → Позиция заказа**), `Дата план` (DATE), `Оператор` (**ref → Пользователь**),
`Статус` (SHORT: `Ожидает`, `В работе`, `Готово`), `Кол-во факт` (NUMBER), `Комментарий` (MEMO).

**ТД-7. Типы связей (критично).** `Клиент` и `Вид сырья` — **справочники по ссылке (ref)**, общие для
многих записей. `Позиция заказа` — **подчинённая таблица** заказа. `Производственная резка`
ссылается на конкретную **позицию**, а не на заказ целиком.

**ТД-8. Самопроверка схемы.** После создания структуры обязательна проверка `metadata?JSON=1`:
`Заказ.Позиция заказа` имеет `arr_id` и не имеет `ref`; `Заказ.Клиент`, `Позиция заказа.Вид сырья`,
`Производственная резка.Позиция заказа`, `Производственная резка.Оператор` имеют `ref`. Появление
`Клиент`/`Вид сырья` как `arr_id` — **дефект** (справочник ошибочно сделан подчинённым).

## 5. Функциональные требования (ФТ)

### 5.1. Импорт исходных данных
**ФТ-1.** Загрузка данных из Excel в порядке зависимостей: Клиенты → Виды сырья → Заказы →
Позиции (как дочерние записи заказа с `up=<id заказа>`) → (Производственные резки не импортируются,
создаются на экране планирования).
**ФТ-2.** Позиция привязывается к заказу по полю `Номер заказа` (поиск заказа → создание дочерней
записи). Контроль: `АТХ-0001` имеет две позиции на один `Вид сырья`; `АТХ-0002` — одну.

### 5.2. Менеджер — приём заказов
**ФТ-3.** Создание заказа (`_m_new/{Заказ}`), выбор клиента из справочника.
**ФТ-4.** Добавление позиций как подчинённых записей заказа (`up=<id заказа>`); клиент в позиции не
выбирается (берётся из заказа).
**ФТ-5.** Изменение статусов заказа и позиции (`_m_set/{recordId}`).
**ФТ-6.** Список заказов с фильтром по статусу; карточка заказа с табличной частью позиций.

### 5.3. Диспетчер — планирование резки
**ФТ-7.** Просмотр позиций со статусом `Новая`/`Запланирована` (очередь незапланированных).
**ФТ-8.** Создание `Производственная резка` со ссылкой на позицию, назначением даты плана и оператора.
**ФТ-9.** Перевод статуса позиции в `Запланирована`.
**ФТ-10.** Очередь резок по датам и статусам.

### 5.4. Оператор — выполнение
**ФТ-11.** Просмотр резок, назначенных текущему оператору.
**ФТ-12.** Перевод статуса резки `Ожидает → В работе → Готово`.
**ФТ-13.** Ввод `Кол-во факт`.
**ФТ-14.** При завершении резки — перевод статуса позиции заказа в `Готова`.

### 5.5. Руководитель — контроль
**ФТ-15.** Счётчики заказов и резок по статусам (только чтение).
**ФТ-16.** Список готовых позиций за период.
**ФТ-17.** Таблица отклонений: позиции/резки, где `Кол-во факт` < `Кол-во`.

## 6. Требования к ролям, правам и меню

**ФТ-18.** Роли и доступ:

| Роль | Меню | Права |
|---|---|---|
| Менеджер | `orders` | WRITE: `Заказ`, `Позиция заказа`; READ: `Клиент`, `Вид сырья` |
| Диспетчер | `planning` | WRITE: `Производственная резка`, `Позиция заказа`; READ: справочники, заказы |
| Оператор | `operator` | WRITE: `Производственная резка`; READ: позиции, справочники |
| Руководитель | `dashboard` | READ: все таблицы проекта |

**ФТ-19.** Пункты меню создаются как дочерние записи роли в таблице `Меню` (`_m_new/151`,
`up=<roleId>`). Права выдаются через объекты роли (`t117` / подчинённая таблица `Объекты` 116).
**ФТ-20.** На первом этапе допустим прямой доступ к таблицам; при появлении внешних пользователей
(портал) чтение переводится на `report/` с проверкой масок доступа.

## 7. Требования к рабочим экранам

| РМ (файл) | Роль | Назначение |
|---|---|---|
| `templates/mini-atex/orders.html` | Менеджер | Заказы + табличная часть позиций |

**ФТ-21.** Стили и JS выносятся в `/download/mini-atex/css/` и `/download/mini-atex/js/`, когда экран
перерастает прототип.

## 8. Нефункциональные требования (НФТ)

**НФТ-1. Единая модель данных.** Все экраны работают с одной общей схемой; копий таблиц под экран нет.
**НФТ-2. Без хардкода идентификаторов.** Запросы к API получают id таблиц и реквизитов из `metadata`,
а не из захардкоженных значений.
**НФТ-3. Корректность связей.** Соблюдение ТД-7/ТД-8 (ref для справочников, arr для позиций) — условие
работоспособности экранов.
**НФТ-4. Платформа.** Реализация средствами Интеграма (`_d_*` — схема, `_m_*` — данные, `report/` —
выборки); авторизация по токену.
**НФТ-5. Самодостаточность.** Минимальное приложение проходится за один сквозной сценарий без
обращения к другим базам/проектам.

## 9. Порядок выполнения работ

1. **Подтверждение пути.** Согласовать с заказчиком резюме приложения и уточнить то, чего нет в
   файлах: один оператор на любые резки или только свои; нужен ли клиентский доступ; есть ли
   логотип/фирменные цвета. К технической работе переходить только после подтверждения.
2. **Схема.** Создать таблицы и связи (`_d_new` → поля → `_d_ref`+`_d_req` → подчинение позиций),
   перечитать `metadata`, выполнить самопроверку (ТД-8).
3. **Данные.** Импортировать справочники и заказы, позиции — дочерними записями.
4. **Роли/меню/пользователи.** Создать роли, пользователей, пункты меню, выдать права (раздел 6).
5. **Экраны.** Реализовать 4 РМ, проверить открытие каждого пункта меню под своей ролью.
6. **Приёмка.** Пройти сценарий раздела 10.

## 10. Приёмочный сценарий (сквозной)

1. **Менеджер** (`manager`, `orders.html`): создать заказ `АТХ-0003` для `ООО "Ромашка-Термолента"`,
   добавить позицию `ТТ-58 ECO 57x40` (Вид сырья `Jumbo Thermal 910/4000`, Ширина 57, Длина 1200,
   Кол-во 10, Статус `Новая`). Позиция видна после обновления страницы.
2. **Диспетчер** (`dispatcher`, `planning.html`): создать резку для `ТТ-58 ECO 57x40`, оператор
   `operator`, дата `2026-06-01`, статус `Ожидает`; статус позиции → `Запланирована`.
3. **Оператор** (`operator`, `operator.html`): перевести резку `Ожидает → В работе → Готово`,
   записать `Кол-во факт: 10`; статус позиции → `Готова`.
4. **Руководитель** (`director`, `dashboard.html`): видит один готовый заказ, одну готовую резку и
   позицию `ТТ-58 ECO 57x40` в готовых; отклонений нет (факт = план).

## 11. Критерии приёмки

- Один документ читается сверху вниз и даёт весь путь без обращения к другим базам.
- Схема использует правильные типы связей (ТД-7): `Клиент`/`Вид сырья` — ссылка, `Позиция заказа` —
  подчинённая таблица.
- Все экраны работают с единой моделью данных (НФТ-1, НФТ-2).
- Сквозной сценарий (раздел 10) проходит от заказа из Excel до готовой резки на экране руководителя.
- Импорт восстанавливает связи: `АТХ-0001` — две позиции на один вид сырья, `АТХ-0002` — одна.

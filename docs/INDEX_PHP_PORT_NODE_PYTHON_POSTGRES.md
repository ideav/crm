# Оценка переноса `index.php` 1-в-1 на Node.js и Python с БД PostgreSQL

## Задача

Issue [#3141](https://github.com/ideav/crm/issues/3141): «`index.php` — оцени перевод
функционала 1-в-1 на Node.js и Python с БД PostgreSQL. Надо 2 варианта сравнить».

Документ оценивает осуществимость, трудоёмкость и риски **дословного (1-в-1) переноса**
серверного движка `index.php` на два стека — **Node.js + PostgreSQL** и **Python + PostgreSQL** —
и сравнивает варианты с рекомендацией. Это оценка, а не сам перенос.

Документ не описывает фронтенд-переписывание — для него есть отдельный
[`VUE_INTEGRAM_REWRITE_PLAN.md`](VUE_INTEGRAM_REWRITE_PLAN.md). Здесь речь только о
серверном ядре Интеграма (`index.php` + `include/connection.php`).

## Что такое `index.php` сегодня (факты)

`index.php` — это весь серверный движок платформы Интеграм в одном файле.

- Размер: **561 444 байт, 11 376 строк, 144 функции, 205 веток `case` в роутере**
  (`index.php:1`).
- Подключения: `include/connection.php` (соединение с БД, SMTP- и OAuth-конфиг,
  константы). Всё остальное — в `index.php`.
- Точка входа — единый роутер. Имя БД (арендатора) берётся из первого сегмента URL
  (`index.php:50`), действие `$a` диспетчеризуется большим `switch($a)` начиная с
  `index.php:9323`.

### Модель данных — EAV/триплет в одной таблице на арендатора

- Каждая «база данных» арендатора — это **отдельная таблица MySQL** с фиксированными
  колонками `(id, up, ord, t, val)`: `up` — родитель, `ord` — порядок, `t` — тип
  (реквизит/таблица), `val` — значение. Это классический EAV/триплет-стор.
- Новый арендатор создаётся как **копия таблицы-шаблона**:
  `CREATE TABLE $z LIKE $template` + `INSERT INTO $z SELECT * FROM $template`
  (`index.php:631`). Все таблицы арендаторов живут в **одной** схеме MySQL.
- Метаданные (DDL) и данные (DML) хранятся в той же таблице, различаясь типом `t`.
  DDL-команды: `_d_new`, `_d_req`, `_d_ref`, `_d_alias`, `_d_save`, `_d_del`,
  `_d_del_req`, `_d_attrs`, `_d_up`, `_d_ord`, `_d_key`, `_d_multi`, `_d_null`
  (`index.php:10521`–`10768`). DML-команды: `_m_new`, `_m_set`, `_m_save`, `_m_del`,
  `_m_del_batch`, `_m_move`, `_m_up`, `_m_ord`, `_m_id` (`index.php:9600`–`10267`).

### Ядро — компилятор отчётов

- `Compile_Report()` (`index.php:2158`, ~2300 строк до `index.php:4455`) и
  `Get_block_data()` (`index.php:4590`) — генератор SQL, который превращает
  декларативное описание отчёта (хранящееся в той же EAV-таблице) в большой SQL-запрос
  с JOIN-ами реквизитов, агрегатами (`COUNT/SUM/AVG/MIN/MAX`), вычисляемыми колонками
  (`abn_*`), фильтрами (`Construct_WHERE`, `index.php:972`) и форматированием.
- Это самая сложная и самая «MySQL-зависимая» часть: строковые операции, регулярки в SQL
  (`REGEXP`), `GROUP_CONCAT`, `IFNULL`, `RAND`, `NOW`, конкатенация — всё на диалекте MySQL.

### Аутентификация, права, безопасность

- Логин по паролю (`login`, `index.php:726`; хэш `sha512` с солью, `Salt`/`Insert`),
  cookie-токены (`idb_{db}`), XSRF-токены (`xsrf`, `index.php:711`).
- OAuth: **Google** (`G_CLIENT_ID`/`G_CLIENT_PK`) и **Яндекс** (`Y_CLIENT_ID`),
  обработка callback `auth.asp` с защитой от дублей через файловые локи (`index.php:58`).
- **JWT** (`verifyJWT`/`authJWT`, `index.php:9239`/`9270`), Yandex **SmartCaptcha**
  (`verifyCaptcha`, `index.php:527`), SMS-рассылка (`SMS_SADR`), сброс пароля
  (`pwd_reset`, `index.php:8314`).
- Система прав: `Validate_Token` (`index.php:1488`), `Check_Grant` (`index.php:1371`),
  `Grant_1level`, `Check_Types_Grant`, гранты READ/WRITE по объектам и колонкам отчётов
  (`CheckRepColGranted`). Защита от инъекций (`checkInjection`, `index.php:883`).

### Прочая функциональность

- Файлы: загрузка/раздача/удаление, хранение по подкаталогам (`GetSubdir`/`GetFilename`,
  `index.php:852`), `dir_admin`.
- Почта: собственный SMTP-клиент на сырых сокетах (`smtpmail`/`server_parse`,
  `index.php:8200`/`8285`), `fsockopen` к `ssl://smtp.yandex.ru`.
- **AI-чат-прокси**: `handleAiChatRequest` (`index.php:8413`) с провайдерами Gemini,
  OpenAI, GigaChat, DeepSeek, «Интеграм AI» (`index.php:8558`), включая Google ADC/JWT
  для service-account токенов.
- Биллинг: баланс арендатора (тип 285), `updateBilling` (`index.php:9290`),
  списание за AI-запросы.
- Экспорт CSV/JSON, импорт батчем, бэкап/восстановление таблицы арендатора
  (`backup`/`restore`, `index.php:4758`).
- ~80 вызовов cURL (внешние HTTP), 53 операции с cookie, прямые `exec()`
  (копирование каталогов арендатора, `index.php:653`).

### Связанность с MySQL

- **211 вызовов `Exec_sql`** (`index.php:770`) — тонкая обёртка над `mysqli_query`;
  173 `mysqli_fetch_array`. Драйвер — `mysqli` (`include/connection.php:14`),
  charset `utf8mb4`, `sql_mode` задаётся в сессии.
- MySQL-специфика в SQL: `CREATE TABLE … LIKE`, `GROUP_CONCAT` (9), `REGEXP` (3),
  `IFNULL` (5), `RAND` (6), `NOW` (16), идентификаторы без кавычек/в backtick'ах,
  имя таблицы = имя арендатора прямо в строке SQL.

**Вывод по текущему состоянию:** это не CRUD-приложение, а **интерпретатор низкокодовой
платформы**. «1-в-1 перенос» означает переписать интерпретатор метаданных, компилятор
отчётов, движок прав и весь набор интеграций, сохранив тот же сетевой контракт
(URL-схему, формат ответов, cookie/токены), который уже используют фронтенд, MCP-сервер
(`mcp-server/`) и рабочие места (`templates/`).

## Две независимые оси сложности

«Перевод 1-в-1 на Node.js/Python с PostgreSQL» — это **две ортогональные задачи**, и их
важно не смешивать при оценке:

1. **Язык/рантайм**: PHP → Node.js *или* PHP → Python. Влияет на скорость разработки,
   найм, экосистему, совместимость с уже существующим кодом репозитория.
2. **СУБД**: MySQL → PostgreSQL. Эта ось **общая для обоих вариантов** и от выбора языка
   почти не зависит. Большая часть риска «1-в-1» сидит именно здесь.

### Ось СУБД: MySQL → PostgreSQL (одинаково для Node.js и Python)

Места, требующие переписывания при переходе на PostgreSQL, независимо от языка:

| Конструкция MySQL | Где | Замена в PostgreSQL |
|---|---|---|
| `CREATE TABLE $z LIKE tmpl` | `index.php:631` | `CREATE TABLE z (LIKE tmpl INCLUDING ALL)` |
| Имя таблицы = имя арендатора | весь файл | сохранить, но строго экранировать идентификаторы |
| `GROUP_CONCAT(x SEPARATOR s)` | компилятор отчётов | `string_agg(x, s)` |
| `IFNULL(a,b)` | формулы/фильтры | `coalesce(a,b)` |
| `RAND()` | выборки | `random()` |
| `col REGEXP pat` | фильтры | `col ~ pat` (POSIX) |
| Неявные приведения типов (всё в `val` — текст) | EAV-модель | явные `CAST`, иначе ошибки сравнения чисел/дат |
| `mysqli_insert_id()` | `Insert` | `INSERT … RETURNING id` |
| `sql_mode`, backtick-кавычки | `connection.php`, SQL | убрать; двойные кавычки для идентификаторов |
| `mysqli_*` API | 211 мест | драйвер `pg`/`psycopg` + переписанные запросы |

Дополнительно:
- **Мультиарендность через таблицу-на-арендатора** в PostgreSQL воспроизводится, но
  естественнее ложится на **схему-на-арендатора** (`CREATE SCHEMA`) — это
  отклонение от «1-в-1», его надо явно решить.
- PostgreSQL строже к типам: вся EAV-модель хранит значения как текст в `val`, а
  сравнения чисел/дат в `Construct_WHERE` и агрегатах полагаются на «ленивые» приведения
  MySQL. Это **основной источник скрытых дефектов** при портировании и требует
  построчного аудита компилятора отчётов.
- `utf8mb4` → PostgreSQL `UTF8` (тривиально), коллации — проверить сортировки.

Эта ось — самая трудоёмкая и рискованная часть всей задачи, и она **не зависит**
от выбора между Node.js и Python.

## Новые вводные (issue #3143): спецификация и Python-наработка

С момента первой редакции появились два артефакта, которые надо учесть в оценке. Оба
доступны и проверены при подготовке этого документа.

### Спецификация «Контроллер БД» (`Контроллер Интеграм 250630.docx`)

Это формальное ТЗ на **минимально работоспособную версию** серверного контроллера модели
данных Интеграма — то есть на ровно то ядро, о котором идёт речь в этой оценке (EAV-модель,
структура и данные), но **без** компилятора отчётов «в полном объёме», биллинга, AI-прокси,
почты и прочей периферии `index.php`. Ключевые факты из ТЗ:

- **Цель — не дословный порт `index.php`, а реализация по спецификации.** ТЗ прямо
  разрешает менять формат ответа: «Структуру JSON можно менять, исходя из удобства и
  правильности реализации». То есть «1-в-1» переопределяется как «контроллер,
  удовлетворяющий контракту из ТЗ», а сам ТЗ и есть контракт.
- **СУБД — сразу PostgreSQL.** Вся физическая модель и примеры SQL даны на диалекте
  PostgreSQL: `bigserial`, `int8`, `text`, `ILIKE`, `array_agg(... ) FILTER (WHERE ...)`,
  `lower(left(val,127))`, `::int8`, `string_agg`-стиль агрегации. Это **снимает часть
  неопределённости оси MySQL→PostgreSQL**: целевой диалект задан и для самых сложных
  запросов (метаданные, выборка объектов, конструктор запросов) приведён эталонный SQL.
- **Та же EAV-модель, но в 4 колонках**: одна таблица на арендатора `(id, up, t, val)`
  с индексами по `(t, val)` и `(up, t)`. Порядок (`ord`) хранится **внутри `val`**, а не
  отдельной колонкой — это отличается от `index.php`, где колонок пять `(id, up, ord, t, val)`.
- **Маршрутизация совпадает с `index.php`**: `{host}/{db name}/{method}/{id}?{params}` —
  имя арендатора первым сегментом URL.
- **Перечень методов MVP** (контракт, по которому надо сверять любой бэкенд):
  `GET terms`, `GET metadata/{termId}`, `GET objects/{termId}?up=…` и `POST objects/graphql`,
  `GET object/{objectId}`, `POST/PATCH/DELETE terms/{id}`, `POST references/{termId}`,
  `POST/PATCH/DELETE requisites/{id}`, `POST modifiers/{id}`, `POST objects`,
  `PATCH object/{id}`, `DELETE objects/{id|termId}`, `GET/POST queries/{id}`.
- **Конструктор фильтров и конструктор запросов** описаны отдельно (упрощённый сценарий) —
  это аналог `Compile_Report()` из `index.php`, но в редуцированном виде и тоже с эталонным
  PostgreSQL-SQL. Самая сложная часть, как и в `index.php`.
- **Безопасность вынесена в требования**: экранирование всех входных параметров, приведение
  `id/up/t` к числам, проверка имён полей по списку «своих» слов (защита от инъекций),
  токен доступа от UI. Это совпадает с тем, что в `index.php` делает `checkInjection`.

**Важно для оценки:** базовые типы в ТЗ, в `index.php` и в наработке **расходятся** по
конкретным `id` — то есть «канонический» контракт типов ещё не стабилизирован:

| `id` | `index.php` (`index.php:450`) | Спецификация (ТЗ) | Наработка (`additional.py` seed) |
|---|---|---|---|
| 3 | `SHORT` | `CHARS` | `SHORT` |
| 5 | `GRANT` | `TERM` | `TERM` |
| 8 | `CHARS` | `MEDIA` | `CHARS` |
| 14 | `SIGNED` | `FLOAT` | `FLOAT` |
| 16 | `REPORT_COLUMN` | `UUID` | `UUID` |

Это прямое подтверждение тезиса исходной оценки: буквальное «1-в-1» с `index.php`
недостижимо без сверки контракта — даже у ТЗ и наработки нумерация типов своя.

### Python-наработка `ideav/python` (`integram-python`)

В репозитории [`ideav/python`](https://github.com/ideav/python) уже лежит **рабочий
прототип контроллера на Python**, реализующий часть спецификации выше.

**Стек:** FastAPI 0.115, `asyncpg` (пул соединений) + `psycopg2`, Pydantic v2,
`uvicorn`, Authlib/`itsdangerous` (заявлены в зависимостях), PostgreSQL 15, Docker Compose
с `adminer`. Бизнес-логика EAV вынесена в **хранимые функции PL/pgSQL**
(`postgres/terms6.sql`, `postgres/objs2.sql`), а FastAPI-роутеры — тонкий слой над ними.

**Что уже реализовано (с привязкой к коду наработки):**

- Слой соединения и пул (`app/db/connection.py`), точка входа и роутеры
  (`app/main.py`, `app/api/routers/*.py`).
- **Термины**: `GET/POST/PATCH/DELETE terms` (`routers/terms.py`, `crud/terms.py`),
  поверх функций `post_terms`/`patch_terms`/`delete_terms` (`postgres/terms6.sql`).
  SQL списка терминов и метаданных **дословно повторяет эталон из ТЗ**.
- **Объекты**: `GET objects/{termId}`, `GET object/{id}`, `POST objects` (батч),
  `PATCH/DELETE object/{id}`, `GET metadata[/{termId}]` (`routers/objects.py`,
  `crud/objects.py`) поверх `post_objects`/`patch_object`/`delete_object`
  (`postgres/objs2.sql`).
- **Ссылки и реквизиты**: `POST references`, `POST requisites`
  (`routers/additional.py`, `crud/additional.py`) поверх `post_references`/`post_requisites`.
- **Создание арендатора**: `GET {db}/create` создаёт таблицу `(id, up, t, val)`,
  индексы и засев базовых типов/демоданных (`crud/additional.py: create_db`).

**Что ещё не реализовано / неполно (объём оставшейся работы):**

- **Конструктор запросов `queries/{id}` и `objects/graphql`** — не реализованы. Это, как и
  в `index.php`, самая сложная часть (автосвязывание таблиц JOIN-ами); в ТЗ описан лишь
  упрощённый сценарий.
- **Конструктор фильтров** — не реализован (есть только демонстрационный код
  `edit_requisite_order`, возвращающий `"ok"`).
- **`POST modifiers/{id}`, `PATCH/DELETE requisites/{id}`** — закомментированы/заглушены.
- **Аутентификация, токен доступа, права/гранты (ROLE/GRANT/MASK)** — отсутствуют;
  CORS открыт на `*` с `allow_credentials=True`.

**Качество / технический долг (важно для честной оценки трудоёмкости):**

- **SQL-инъекции.** Имя арендатора и значения подставляются в запросы через f-строки
  (`f"... FROM {db_name} ..."`, `f"... id IN ({ids_list})"`), а в PL/pgSQL — через
  `format('… %s …', value)` с ручным `replace('''','''''')`. ТЗ требует экранирования —
  текущая реализация ему **не соответствует** и потребует переработки слоя доступа.
- **Захардкоженный DSN с боевым паролем** прямо в `app/core/config.py` (`DB_URL = …`).
- **Глобальный кэш типов** (`config.TYPES`) — общий на все БД, ломает мультиарендность.
- **Сид базовых типов расходится** и с ТЗ, и с `index.php` (см. таблицу выше).
- Тестов нет; CI/линт не настроены (`pyproject.toml` — только `black`).

**Вывод по наработке:** это **не дословный порт `index.php`, а самостоятельная
реализация спецификации «Контроллер БД» на Python+PostgreSQL**, закрывающая «лёгкую»
половину ТЗ (DDL/DML над EAV, метаданные, CRUD терминов/объектов/ссылок/реквизитов).
Самое сложное (конструкторы запросов и фильтров, права, безопасный слой доступа) ещё
впереди. Тем не менее это **реальный задел именно на оси MySQL→PostgreSQL**, которая в
исходной оценке названа главным риском: ключевые EAV-запросы уже переписаны и работают
на PostgreSQL.



**Стек:** Node.js (LTS ≥ 18), Express/Fastify, драйвер `pg`, JWT-библиотека,
nodemailer (вместо самописного SMTP), axios (уже используется в `mcp-server/`).

**Плюсы**
- В репозитории **уже есть production Node-код**: `mcp-server/` (`axios`, ESM,
  `@modelcontextprotocol/sdk`) и весь фронтенд на JS. Единый язык фронт/бэк/инструментов.
- `mcp-server/index.js` уже знает сетевой контракт Интеграма (URL, токены, форматы) —
  готовый референс «как должен отвечать сервер».
- JSON — родной формат; ответы API (`json_encode`) ложатся 1-в-1.
- Команда репозитория уже пишет на JS — низкий порог входа, нет смены языка.
- Сильный async-I/O для cURL-подобных интеграций (AI-прокси, OAuth, SMTP).

**Минусы / риски**
- Числа: PHP-арифметика и MySQL-числа — иные, чем JS `Number` (double). Денежный
  биллинг и агрегаты отчётов требуют аккуратности (целые/`decimal`-библиотеки).
- Нет прямых эквивалентов части PHP-функций (поведение `preg_*` vs RegExp, `mb_*`,
  форматирование дат) — построчная сверка.
- Синхронный «процедурный» стиль `index.php` (последовательные `Exec_sql` + `die`)
  переписывается в async/await — много мест с ранним `exit`/`die` (`my_die`, `myexit`).

**Трудоёмкость:** высокая. Компилятор отчётов (~2300 строк) + движок прав + DDL/DML
переписываются полностью; интеграции (OAuth/JWT/SMTP/AI) — переиспользуют готовые npm-пакеты.

## Вариант B: Python + PostgreSQL

**Стек:** Python 3.12, FastAPI/Flask, драйвер `psycopg` (3), PyJWT, встроенный `smtplib`,
`httpx`/`requests`.

**Плюсы**
- Самая зрелая экосистема для PostgreSQL (`psycopg3`, SQLAlchemy Core), `smtplib`/`email`
  в стандартной библиотеке — самописный SMTP-клиент (`smtpmail`) заменяется штатным.
- Семантика чисел ближе к PHP/SQL: `decimal.Decimal` из коробки для денег/агрегатов —
  меньше риск в биллинге и отчётах.
- `re` близок к PCRE (`preg_*`), строковые/дата-операции выразительны — перенос формул
  компилятора чуть менее рискован.
- Хорош для последующей AI/data-обработки (AI-прокси, аналитика).

**Плюсы (с учётом issue #3143)**
- **Уже есть рабочая наработка** [`ideav/python`](https://github.com/ideav/python): FastAPI +
  `asyncpg` + PL/pgSQL, закрывающая «лёгкую» половину спецификации «Контроллер БД». Это не
  голый старт — DDL/DML над EAV, метаданные и CRUD уже работают на PostgreSQL.
- Часть оси MySQL→PostgreSQL **уже пройдена**: ключевые EAV-запросы переписаны на
  PostgreSQL-диалект и совпадают с эталоном из ТЗ.

**Минусы / риски**
- Появляется **третий язык** (после PHP-движка и JS-фронтенда) — выше когнитивная нагрузка,
  отдельный CI, образ, зависимости, найм. (Но наработка уже снимает вопрос «писать ли вообще
  на Python».)
- Нет готового внутреннего референса контракта, эквивалентного `mcp-server/` (он на JS) —
  **зато теперь есть внешний контракт: спецификация «Контроллер БД»**, и наработка ему следует.
- **Технический долг наработки** (инъекции через f-строки, захардкоженный DSN, глобальный
  кэш типов, расхождение сид-данных, отсутствие тестов/прав) придётся погасить — это часть
  оценки трудоёмкости, см. [раздел про наработку](#python-наработка-ideavpython-integram-python).

**Трудоёмкость:** высокая, сопоставима с вариантом A по полному движку `index.php`; но для
**минимального контроллера по ТЗ** объём уменьшен существующим прототипом — остаётся
конструктор запросов/фильтров, права и доведение слоя доступа до безопасного состояния.

## Сравнение вариантов

| Критерий | Node.js + PostgreSQL | Python + PostgreSQL |
|---|---|---|
| Совместимость с текущим репозиторием | **Высокая** (JS-фронт, `mcp-server/`) | Низкая (новый язык в этом репо) |
| Существующий задел по серверу | Нет (только `mcp-server/` как референс) | **Есть рабочий прототип** (`ideav/python`) |
| Готовый референс контракта | `mcp-server/index.js` | **Спецификация «Контроллер БД» + прототип** |
| Единый язык фронт+бэк+инструменты | **Да** | Нет (3 языка) |
| Точность денег/агрегатов (биллинг, отчёты) | Требует `decimal`-библиотеки | **`Decimal` из stdlib** |
| Совместимость regex с `preg_*` | RegExp (отличия) | **`re` ближе к PCRE** |
| SMTP | nodemailer (внешний) | **`smtplib` (stdlib)** |
| Драйвер PostgreSQL | `pg` (зрелый) | **`psycopg3` (очень зрелый)** |
| Async-I/O для интеграций | **Отличный** | Хороший (FastAPI/async) |
| Порог входа для команды | **Низкий** | Средний/высокий |
| Эксплуатация (доп. рантайм/CI/образ) | **Без нового рантайма** | Новый рантайм и пайплайн |
| Перенос ядра (компилятор/права/DDL/DML) | Сопоставимо, с нуля | **DDL/DML/метаданные/CRUD уже есть** (прототип) |
| Риск миграции MySQL→PostgreSQL | Одинаков (ось общая) | **Частично пройден** (запросы уже на PostgreSQL) |

## Рекомендация

С учётом новых вводных (issue #3143) рекомендация **зависит от того, какой объём работ
реально ставится** — а это две разные задачи, которые раньше сливались в одну формулу «1-в-1».

### Если цель — минимальный «Контроллер БД» по спецификации → **продолжать Python-наработку**

Спецификация прямо описывает **минимально работоспособную версию** и разрешает менять формат
JSON-ответа — то есть это **не** дословный порт `index.php`, а реализация по контракту ТЗ.
Под эту задачу уже есть рабочий прототип [`ideav/python`](https://github.com/ideav/python),
который закрывает «лёгкую» половину ТЗ на PostgreSQL. В этом сценарии:

1. **Задел перевешивает «третий язык».** Главный минус Python в исходной оценке — «в репо нет
   Python-кода» — больше не актуален: код есть и работает. Переписывать его на Node означало бы
   выбросить готовую часть оси MySQL→PostgreSQL.
2. **Контракт задан внешним ТЗ**, а не только `mcp-server/`. Аргумент «у Node есть живой
   референс контракта» слабеет: для контроллера контракт — это спецификация, и прототип ей следует.
3. **Семантика Python (`Decimal`, `re`) ложится на EAV-модель** (всё в `val` — текст, числовые
   приведения) чуть безопаснее.

Прежде чем продолжать, обязательно **погасить технический долг прототипа** (параметризовать SQL
вместо f-строк, убрать захардкоженный DSN, починить мультиарендный кэш типов, согласовать сид
базовых типов с ТЗ, добавить тесты и права) — иначе он не соответствует требованиям безопасности
самого ТЗ.

### Если цель — полный 1-в-1 движок `index.php` → по-прежнему **Вариант A: Node.js + PostgreSQL**

Спецификация покрывает только ядро модели данных. **Компилятор отчётов в полном объёме, права/
гранты, OAuth/JWT, SMTP, AI-прокси, биллинг, файлы, бэкап/восстановление** — всего этого в ТЗ
нет, и для них доводы исходной оценки сохраняются:

1. `mcp-server/index.js` — живой референс сетевого контракта Интеграма для дословной сверки.
2. Фронтенд и инструментарий уже на JavaScript — Node не добавляет нового языка, рантайма и
   CI-пайплайна; Python добавил бы третий язык.
3. Преимущества Python (`Decimal`, `re`, `smtplib`) реальны, но точечны и закрываются библиотеками.

### Итог

- **Минимальный контроллер по ТЗ** — дешевле всего **довести Python-наработку**: задел реален,
  контракт задан спецификацией, ось MySQL→PostgreSQL уже частично пройдена.
- **Полный движок «как есть»** — выгоднее **Node.js** (моноязычный репозиторий + `mcp-server/`).
- Сценарии **не взаимоисключающие**: можно довести Python-контроллер как самостоятельный сервис
  (DDL/DML/метаданные/CRUD), а тяжёлую периферию `index.php` мигрировать отдельно — это
  естественно ложится на **strangler-миграцию** ниже (новый сервис отвечает по контракту,
  трафик переключается поэндпоинтно). Главное — выбрать **один** целевой язык на компонент и не
  плодить третий стек без явной выгоды.

### Важная оговорка про «1-в-1»

Главный риск любого из вариантов — **не язык, а ось MySQL → PostgreSQL** (см. выше):
строгая типизация PostgreSQL против текстового EAV-хранилища и компилятора отчётов,
полагающегося на неявные приведения MySQL. Поэтому буквальный перенос лучше вести
не «big bang», а как **strangler-миграцию**: новый сервис отвечает по тому же контракту,
трафик переключается поэндпоинтно (сначала простые `_m_*`/`_d_*`, в последнюю очередь —
компилятор отчётов), а параллельный прогон старого и нового движка на одинаковых запросах
служит регрессионным тестом «байт-в-байт».

### Предлагаемая последовательность работ (для выбранного варианта)

1. **Зафиксировать контракт**: спецификация «Контроллер БД» + «золотые» запрос/ответ-пары
   из `mcp-server/`, `templates/` и `docs/kb/` (URL, cookie, JSON, коды) как приёмочные тесты.
2. **Слой БД на PostgreSQL**: схема EAV, создание арендатора (`CREATE TABLE … LIKE` →
   `LIKE … INCLUDING ALL` или схема-на-арендатора), обёртка-аналог `Exec_sql` с **параметризацией**.
3. **DDL/DML** (`_d_*`, `_m_*`) — наименее «диалектные», переносятся первыми. *Для Python это
   уже частично есть в наработке `ideav/python` (после погашения тех. долга — см. выше).*
4. **Аутентификация и права** (`Validate_Token`, `Check_Grant`, OAuth/JWT/Captcha).
5. **Компилятор отчётов / конструктор запросов** (`Compile_Report`/`Get_block_data`;
   в ТЗ — `queries/{id}` и конструктор фильтров) — последним, с пословной сверкой
   генерируемого SQL и параллельным прогоном против MySQL.
6. **Интеграции**: SMTP, AI-прокси, биллинг, файлы — на штатных пакетах выбранного стека
   (npm для Node, stdlib/`httpx` для Python).
7. **Параллельный прогон** старого PHP и нового сервиса, регресс по «золотым» парам,
   поэндпоинтное переключение трафика.

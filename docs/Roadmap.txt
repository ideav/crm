# Общий план реализации проекта в Интеграме

## Введение

Этот документ — пошаговый roadmap для реализации любого бизнес-приложения в конструкторе Интеграм. Он описывает полный цикл: от получения ТЗ до передачи готового решения пользователям.

---

## Этап 0: Получение и анализ ТЗ

### Входные данные
- Описание бизнес-процессов
- Список сущностей (клиенты, заказы, проекты, задачи и т.д.)
- Требования к ролям пользователей
- Примеры отчётов и экранов

### Результат этапа
- Выделены основные сущности (таблицы)
- Определены связи между ними
- Выделены роли пользователей
- Составлен список экранов и отчётов

### Контрольные вопросы
1. Какие объекты есть в системе? (Клиент, Проект, Заказ, Задача, Платёж, Сотрудник...)
2. Как они связаны между собой?
3. Кто работает с системой? (Админ, Менеджер, Клиент...)
4. Какие отчёты нужны?

### Пример для «Перелидоз»
- Сущности: Пользователь, Проект, Клиент, Платёж, Задача, Стратегия...
- Связи: Проект → Пользователь (ответственный), Задача → Проект, Платёж → Проект
- Роли: founder, team
- Отчёты: список проектов, список задач, Health Score

## Этап 1: Создание структуры данных (таблицы и колонки)

### Последовательность действий

#### 1.1. Создание базы данных (если ещё нет)
```
curl --location 'https://integram.io/my/_new_db?JSON=1' --form 'db="название_базы"' --form 'template="ru"'
```

#### 1.2. Создание таблиц
Для каждой сущности создаётся таблица.

**Правила именования:**
- Имя таблицы — в единственном числе, на русском
- Первая колонка — собственный идентификатор объекта (не ссылка!)

**Пример создания таблицы «Проект»:**
```
curl --location 'https://integram.io/mydb/_d_new?JSON=1' --form 't="3"' --form 'val="Проект"'
```

#### 1.3. Добавление колонок в таблицы
Для каждой колонки:
1. Создать термин-колонку (`_d_new`)
2. Добавить его в таблицу (`_d_req/{ID_таблицы}`)
3. Сохранить ID реквизита (понадобится для вставки данных)

**Типы колонок:**
| Код | Тип | Когда использовать |
|-----|-----|---------------------|
| 3 | SHORT | Имена, коды, номера |
| 8 | CHARS | Email, URL |
| 9 | DATE | Даты |
| 13 | NUMBER | Суммы, количества |
| 11 | BOOLEAN | Флаги |
| 12 | MEMO | Длинные тексты |

#### 1.4. Создание ссылок между таблицами
Для связи двух таблиц:
1. Создать ссылочный термин (`_d_ref/{ID_целевой_таблицы}`)
2. Добавить ссылку как колонку (`_d_req/{ID_таблицы}`)

### Результат этапа
- Готовая база данных со всеми таблицами, колонками и связями
- Документ с ID всех таблиц и реквизитов

### Контрольный список
- [ ] Все сущности стали таблицами
- [ ] У каждой таблицы есть первая колонка — идентификатор
- [ ] Все ссылки между таблицами созданы
- [ ] Типы колонок выбраны правильно

## Этап 2: Наполнение тестовыми данными

### Последовательность действий

#### 2.1. Подготовка тестовых данных
Создать Excel/CSV с данными для каждой таблицы.

**Пример для таблицы «Проект»:**
| Название проекта | Сайт | Статус | Бюджет |
|------------------|------|--------|--------|
| Альфа | alfa.ru | active | 1000000 |
| Бета | beta.ru | active | 500000 |

#### 2.2. Добавление записей через API
Используя ID реквизитов (полученные на этапе 1), добавить записи.

```
curl --location 'https://integram.io/mydb/_m_new/{ID_таблицы}?JSON=1' \
--form 'up="1"' \
--form 't{ID_реквизита1}="Альфа"' \
--form 't{ID_реквизита2}="alfa.ru"' \
--form 't{ID_реквизита3}="active"' \
--form 't{ID_реквизита4}="1000000"'
```

#### 2.3. Установка ссылочных значений
Для заполнения ссылочных колонок используется `_m_set`.

```
curl --location 'https://integram.io/mydb/_m_set/{ID_записи}?JSON=1' \
--form 't{ID_реквизита_ссылки}="{ID_целевой_записи}"'
```

### Результат этапа
- База заполнена тестовыми данными (по 3-5 записей в каждую таблицу)
- Можно проверять работу экранов и отчётов

### Контрольный список
- [ ] В каждой таблице есть тестовые записи
- [ ] Ссылки между записями работают
- [ ] Данные реалистичны (похожи на боевые)

## Этап 3: Создание рабочих мест (экранов)

### Что такое «рабочее место» в Интеграме
Рабочее место — это набор экранов (форм, списков, отчётов) для определённой роли пользователя.

### Последовательность действий

#### 3.1. Определение ролей и их прав
- Какие роли есть в системе? (admin, manager, client...)
- К каким таблицам у каждой роли доступ?
- Какие действия разрешены? (чтение, запись, удаление)

#### 3.2. Настройка прав доступа (Роли → Доступ)
В таблице «Доступ» указать:
- Какой объект (таблица)
- Какая роль
- Какие права (чтение, запись, удаление, экспорт)

#### 3.3. Создание форм для ввода/редактирования данных
Для каждой таблицы можно настроить форму:
- Какие поля показывать
- Порядок полей
- Обязательные поля
- Значения по умолчанию

#### 3.4. Создание списков (таблиц) для просмотра данных
- Какие колонки показывать
- Фильтры
- Сортировка

#### 3.5. Создание отчётов (если нужны)
Отчёты создаются через таблицу «Запрос»:
- Указать FROM (откуда берём данные)
- Выбрать колонки
- Добавить WHERE (фильтры)
- Настроить GROUP BY (если нужны итоги)

#### 3.6. Создание панелей (дашбордов)
- Собрать несколько отчётов на одном экране
- Добавить графики и диаграммы

#### 3.7. Настройка меню
- Какие пункты меню видны каждой роли
- Какие отчёты и формы открываются по клику

### Результат этапа
- У каждой роли есть своё рабочее место
- Меню настроено
- Отчёты работают

### Контрольный список
- [ ] Роли созданы и настроены
- [ ] Права доступа настроены
- [ ] Формы для ввода данных работают
- [ ] Списки для просмотра работают
- [ ] Отчёты показывают правильные данные
- [ ] Меню настроено для каждой роли

---

## Этап 4: Тестирование и передача

### Последовательность действий

#### 4.1. Функциональное тестирование
- Проверить все сценарии использования
- Проверить права доступа
- Проверить работу ссылок

#### 4.2. Приёмочное тестирование с пользователем
- Показать заказчику рабочие места
- Внести правки по замечаниям

#### 4.3. Передача в эксплуатацию
- Удалить тестовые данные (или оставить)
- Добавить реальных пользователей
- Передать документацию

### Результат этапа
- Система готова к использованию
- Пользователи обучены

## Сводный roadmap (для новичка)

| Этап | Что делать | Примерное время |
|------|------------|------------------|
| **0** | Прочитать ТЗ, выделить сущности, роли, связи | 1-2 часа |
| **1** | Создать таблицы, колонки, ссылки в Интеграме | 2-4 часа |
| **2** | Подготовить и загрузить тестовые данные | 1-2 часа |
| **3** | Настроить формы, списки, отчёты, права, меню | 4-8 часов |
| **4** | Протестировать, передать заказчику | 2-4 часа |

**Итого:** 1-2 рабочих дня на проект среднего размера (10-15 таблиц)

---

## Памятка: золотые правила проектирования

1. **Первая колонка таблицы** — собственное имя объекта (не ссылка!)
2. **Имена таблиц** — в единственном числе, на русском
3. **Ссылки** — в отдельных колонках, не в первой
4. **Типы колонок** — выбирать осознанно (SHORT для имён, CHARS для длинных строк)
5. **Роли** — продумать до начала создания экранов
6. **Тестовые данные** — загружать сразу после создания структуры

## Пример: как выглядит правильно спроектированная таблица

**Таблица «Стратегия» (правильно):**
| Версия стратегии | Проект (ссылка) | Ссылка на Google Doc | Статус |
|------------------|-----------------|----------------------|--------|
| v1.0 | Альфа | https://... | approved |
| v2.0 | Альфа | https://... | draft |
| v1.0 | Бета | https://... | approved |

**Таблица «Стратегия» (неправильно):**
| Проект (ссылка) | Ссылка на Google Doc | Статус |
|-----------------|----------------------|--------|
| Альфа | https://... | approved |

Почему неправильно? Потому что у стратегии нет собственного идентификатора — непонятно, как ссылаться на конкретную версию стратегии.

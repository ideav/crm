# Разбор: почему скрипт из PR #2900 построен по проекту «orbits», хотя о нём в задаче речи не было

> Ответ на задачу [#2905](https://github.com/ideav/crm/issues/2905).
> Расследование того, как в PR [#2900](https://github.com/ideav/crm/pull/2900) исходными данными оказалась чужая база **orbits**, а не целевой проект **atex**.

## Коротко (TL;DR)

Скрипт `docs/create_db_from_scratch.ps1` сам по себе универсален и к «orbits» не привязан. «orbits» проник в результат **через входной файл метаданных**: к задаче #2899 был приложен `metadata_all (5).json`, и этот файл — не схема atex, а **выгрузка живой базы `orbits`** (`GET /orbits/metadata`): системные таблицы платформы Интеграм + прикладные таблицы CRM-проекта orbits (`Клиент`, `Задача`, `Тег`, `Статус`, `Статус задачи`, `Карта`).

PR #2900 принял приложенный файл буквально как «схему, которую надо создать», и закоммитил его как `docs/metadata_all.json` без сверки с реальным доменом проекта. Название «orbits» нигде не звучало, потому что оно было «зашито» внутри дампа метаданных — ИИ унаследовал чужие таблицы оптом, ни разу не сопоставив их с задачей. Это и есть «неверные исходные данные», позже исправленные в PR #2902 (#2901).

## Цепочка событий

| Шаг | Что произошло | Артефакт |
|-----|---------------|----------|
| #2896 | Анализ связей по приложенным метаданным **ateh** и **atex** | `metadata_all (3).json` = ateh, `metadata_all (4).json` = atex |
| #2899 | Запрос «сделать скрипт наполнения БД с нуля», приложен новый файл | `metadata_all (5).json` |
| #2900 | Скрипт сделан, файл (5) закоммичен как `docs/metadata_all.json` (22 таблицы) | этот PR использует **выгрузку orbits** |
| #2901 / #2902 | Замечено: исходные данные неверны — это сторонняя база, не atex; добавлен `atex_metadata.json` (15 таблиц) | исправление |

Обратите внимание на нумерацию приложений: в #2896 файл `(3)` был ateh, `(4)` — atex. В #2899 приложили `(5)` — и это уже не atex, а orbits. Целевым проектом по контексту всё время был **atex** (файл `(4)`), но в #2899 «нулевой схемой» по ошибке оказался дамп orbits (файл `(5)`).

## Доказательства, что `metadata_all.json` — это выгрузка базы `orbits`

1. **Файл в репозитории идентичен приложенному.** `docs/metadata_all.json` структурно совпадает с `metadata_all (5).json` из #2899 (те же 22 таблицы в том же порядке).

2. **Прикладные таблицы — домен orbits.** Из 22 таблиц 16 — системные таблицы платформы Интеграм (`Пользователь`, `Запрос`, `Колонки запроса`, `Формат`, `Роль`, `FROM`, `Доступ`, `Функция`, `Итог`, `Объекты`, `Форма`, `Панель`, `Поле`, `Кнопка`, `Меню`, `Настройка` — есть в любой базе Интеграм). Оставшиеся 6 — прикладные: `Статус`, `Клиент`, `Тег`, `Карта`, `Задача`, `Статус задачи`. Эти имена дословно встречаются в проекте [`ideav/orbits`](https://github.com/ideav/orbits) (см. `README.md`, `projects.js`, `templates/projects.html`, `DML_RULES.md`: `"val": "Тег"`, `"val": "Клиент"`, `"Статус задачи"`). Имя базы в orbits — `window.db = 'orbits'`.

3. **Совпадает «отпечаток» системной таблицы.** Таблица `Пользователь` (id 18) в `metadata_all.json` имеет реквизиты с id `115` (Роль), `41` (Email), `30` (Телефон), `156` (Дата), `33` (Имя), `39` (Примечание), `38` (Фото), `124` (Activity), `130` (Secret), `20` (Password) — ровно тот же набор id, что приведён в `ideav/orbits/DML_RULES.md`.

4. **В метаданных остались id записей исходной базы.** У `Пользователь.Роль` атрибут `attrs = ":!NULL:164"` — значение по умолчанию, ссылающееся на **запись № 164** конкретной (orbits) базы. Это однозначный признак того, что файл — выгрузка живой, наполненной базы, а не чистая целевая схема (в чистой базе записи 164 нет; PR #2900 её и пропускал). Чистую схему так не пишут — её экспортировали.

## Корневая причина (почему «orbits» не было в контексте)

- **ИИ не выбирал orbits.** Он буквально доверился вложению из #2899 как «схеме для создания» и не задал вопрос: «А что это за таблицы и тот ли это проект?».
- **Дамп замаскировал происхождение.** Большая часть файла — системные таблицы Интеграм, одинаковые во всех базах. На их фоне 6 прикладных таблиц orbits не были опознаны как «чужой проект». Имя базы (`orbits`) в выгрузке `metadata` не присутствует — только структура, — поэтому в диалоге оно и не всплыло.
- **Не было сверки с доменом задачи.** Контекст (см. #2896, #2901) указывал на проект **atex** (производство термопринтерных рулонов: `Заказ`, `Тип резки`, `Производственная резка`…). Достаточно было сопоставить имена таблиц из файла с предметной областью atex, чтобы увидеть несоответствие. Эта сверка не выполнялась — отсюда «неверные исходные данные».

Иначе говоря, ошибка не в движке скрипта, а в **молчаливом наследовании входных данных без валидации источника**: на вход подали экспорт старой базы orbits (вероятно, выгрузку сделали по умолчанию/ошибочно не из той базы), а PR #2900 принял его за целевую схему.

## Что уже исправлено и что стоит сделать впредь

- **Исправлено в #2902:** добавлен `docs/atex_metadata.json` (15 таблиц atex + системная `Пользователь`) по [дизайн-спеке atex](https://github.com/ideav/atex/blob/main/docs/superpowers/specs/2026-05-26-atex-schema-setup-design.md); дефолты скрипта переключены на atex. Прежний `metadata_all.json` оставлен как пример формата (выгрузка сторонней базы).
- **На будущее (профилактика):**
  1. Прежде чем строить базу по приложенному `metadata`, **сверять прикладные таблицы с доменом проекта** — системные таблицы Интеграм отфильтровывать, а оставшиеся имена проверять на принадлежность задаче.
  2. Если в метаданных встречаются **значения по умолчанию с id записей исходной базы** (`:!NULL:<id>`) — это сигнал, что файл является выгрузкой чужой/наполненной базы, а не чистой целевой схемой; источник надо явно подтвердить.
  3. При получении файла-вложения **фиксировать, из какой базы он выгружен** (имя БД нет в `metadata` — его надо запрашивать отдельно), чтобы исключить подмену проекта.

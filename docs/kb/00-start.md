# Старт: доступ и модель данных
> Часть базы знаний Integram. Индекс: [docs/kb/README.md](README.md)

## На пальцах (для человека)

**Интеграм** — low-code платформа (`ideav.ru`): база = набор **таблиц**, у таблицы —
**реквизиты** (колонки). Связи двух видов: **ссылка** на справочник (как foreign key) и
**подчинённая таблица** (дочерние записи одного родителя — «табличная часть»). Поверх БД —
**рабочие места**: HTML/JS-шаблоны в `templates/`, которые ходят в базу по API.

Агент работает по **токену** (а не логин/пароль): пользователь заводит в базе юзера `claude`,
вписывает ему токен (15–30 символов) в поле «Токен» и сообщает агенту адрес базы + токен.
Логин/пароль агенту не подходят — форма входа за капчей; токен капчу обходит.

Всё, что агенту нужно знать дальше — по темам в [индексе](README.md): схема (`_d_*`),
данные (`_m_*`), отчёты (`report/`), роли, файлы, деплой.

## Операции (для агента)

### Авторизация (без `/auth`, без капчи)
Токен (колонка `Token`, `t125`, не хешируется) распознаётся в одном из видов:
```bash
# Заголовок (рекомендуется — транспорт SPA, читается на любом запросе)
curl -s -H "X-Authorization: <token>" "https://ideav.ru/{db}/metadata?JSON=1"
# Cookie (для GET-чтений; нужна для сессионных операций вроде dir_admin)
curl -s -b "idb_{db}=<token>" "https://ideav.ru/{db}/metadata?JSON=1"
# Form-поле token= (для POST _m_*/_d_*)
curl -s -X POST "https://ideav.ru/{db}/_m_new/18?JSON=1" -F "token=<token>" -F "_xsrf=<xsrf>" -F "up=1" -F "t18=manager"
```

### `_xsrf` (обязателен для POST `_m_*`/`_d_*`)
```bash
curl -s -H "X-Authorization: <token>" "https://ideav.ru/{db}/xsrf?JSON"
# → {"_xsrf":"<hex>","token":"<token>","user":"claude","role":"...","id":"<userId>"}
```
Дальше `token` + `_xsrf` подставляются во все POST.

### Предел времени запроса — GET-параметр `TIME`
Любой запрос к `index.php` ограничен по времени: **30 c по умолчанию**, `?TIME=<секунды>` задаёт
свой предел, потолок — **600 c** (10 минут). Значения вне диапазона обрезаются, нечисловое —
игнорируется (30 c). Предел ставится сразу в четырёх местах (блок `# <time-limit-4322>` в
`index.php`):

| Место | Что ограничивает |
|---|---|
| `set_time_limit()` | циклы и обработку данных в PHP |
| `MYSQLI_OPT_READ_TIMEOUT` (`include/connection.php`) | ожидание ответа MySQL: таймер PHP в это время не тикает, потому нужен отдельный |
| `SET SESSION max_statement_time` (MariaDB) / `max_execution_time` (MySQL) | выполнение SELECT на сервере БД — работает и когда PHP уже убит |
| `KILL QUERY` из отдельного соединения | добивает запрос, если PHP завершился прямо на нём (таймаут, фатальная ошибка, обрыв клиента) |

Исчерпание предела — ответ **`504 Gateway Timeout`**: браузеру текст «Запрос прерван: превышен
предел времени N c», API (`?JSON…`) — `[{"error": …}]`. Прерванный SQL пишется в лог базы
(`logs/{db}_log.txt`, метка `[time limit Nc]`). Экспорт/импорт базы (`csv_all`, `backup`,
`restore`) поднимает предел до 300 c сам.

## Грабли
- **Симптом:** `401 [{"error":"No authorization token provided"}]` → **Причина:** токен послан как `Authorization: Bearer` или `?token=` (игнорируются) → **Фикс:** только `X-Authorization` / cookie `idb_{db}` / form-поле `token=` *(issue #3000)*.
- **Симптом:** POST `_m_*`/`_d_*` отвергается → **Причина:** нет `_xsrf` (нужен даже при cookie-сессии) → **Фикс:** получить `/{db}/xsrf?JSON` и слать в теле *(2026-06)*.

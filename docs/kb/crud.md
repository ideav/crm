# Данные (DML) — команды `_m_*` и чтение
> Часть базы знаний Integram. Индекс: [docs/kb/README.md](README.md)

## На пальцах (для человека)
Когда схема (таблицы и колонки) уже готова, в неё кладут сами записи: заказы,
клиентов, позиции. Создание, правка и удаление записей — это семейство команд
`_m_*`, а чтение — GET-запросы к `object/`. Главный фокус, на котором все
спотыкаются: значение первой колонки (имя/номер записи) пишется не в `t3`, а в
`t<id таблицы>` — например для таблицы 1075 это `t1075`. Дочерние записи
(позиции заказа) создаются с указанием родителя `up=<id заказа>`, а ссылочные
поля заполняются id записи из целевого справочника. Подробности структуры — в
[schema.md](schema.md), формат метаданных там же.

## Операции (для агента)

Авторизация — см. [00-start.md](00-start.md). `_m_*` — POST с `token`+`_xsrf`.

### Создать запись
```bash
POST /{db}/_m_new/{tableId}?JSON=1
# Корневая запись:
Fields: up=1, t{tableId}="Название записи", t{colId1}=значение, t{colId2}=значение
# Дочерняя (подчинённая) запись — родитель в up:
Fields: up={parentRecordId}, t{tableId}="Название", t{colId1}=значение
# Ссылочное (ref) поле — значение = id записи целевой таблицы:
Fields: t{refReqId}=579     # 579 — id записи в целевом справочнике
# Полезные флаги: ?full=1 (вернуть полный объект), up=1 (корень)
```
**Главное значение записи = `t{tableId}`, а не `t3`.** Пример: имя заказа в
таблице 1075 → `t1075=...`; логин пользователя (таблица 18) → `t18=...`; имя роли
(таблица 42) → `t42=...`; имя меню (таблица 151) → `t151=...`.

### Изменить запись
```bash
POST /{db}/_m_set/{objId}?JSON=1
Fields: t{reqId}=новое_значение         # любую колонку, включая первую t{tableId}
```

### Удалить запись
```bash
POST /{db}/_m_del/{objId}?JSON=1
# Работает для любых таблиц, включая системные.
# Удаление родителя/пункта меню каскадит на подчинённые записи.
```

### Прочитать записи
```bash
GET /{db}/object/{typeId}/?JSON_KV&LIMIT=100&pg=1   # массив под ключом "object"
GET /{db}/object/Заказ/?JSON_KV                     # по имени таблицы (не зная ID)
GET /{db}/table/Заказ/Заказ №100?JSON_OBJ           # конкретная запись по имени
```
Форматы: `JSON_OBJ` — объекты с полями записи (ссылка приходит как `"id:Значение"`,
мульти — `"id1,id2:знач1,знач2"`); `JSON_KV` — «key-value», удобен для UI.

Фильтры/параметры:
| Параметр | Назначение |
|---|---|
| `F_U={parentId}` | только дочерние записи родителя `up=<parentId>` (подчинённые таблицы, меню роли) |
| `F_I={recordId}` | только запись с этим id (карточка) |
| `F_{reqId}={val}` | фильтр по колонке |
| `_count=` | вернуть только число записей (`?JSON_KV&_count=`) |
| `LIMIT=N` · `pg=N` | страница до N записей · номер страницы (1-based) |

Имя для URL кодировать (`encodeURIComponent`), если есть пробелы/кириллица.

### Массовая загрузка / импорт
Скриптом (bash + curl или PowerShell) повторять `_m_new`. Составной ключ
уникальности соблюдается при импорте. На Windows/PowerShell следить за кодировкой
файла-скрипта (UTF-8), иначе кириллица в `val` ломается — см. монолит
[../integram-app-workflow.md](../integram-app-workflow.md) «Кодировка PowerShell».

## Грабли
- **Симптом:** имя/номер записи не задаётся (создаётся пустое) → **Причина:** писали главное значение в `t3` вместо ключа таблицы → **Фикс:** главное значение = `t{tableId}` (например `t22`, `t1075`, `t18`), а не `t3`. *(issue #597)*
- **Симптом:** `object/{subTableId}` без фильтра возвращает `0` записей / «таблица пустая», хотя данные есть → **Причина:** подчинённая таблица хранит записи как дочерние (`up`=id родителя); без `F_U` сервер отдаёт только уровень `up=1` → **Фикс:** читать по родителям — для каждой записи родителя запрашивать `object/{subTableId}?…&F_U=<parentId>`. Опасно перед удалением колонки. *(monolith §3.5)*
- **Симптом:** при пагинации часть записей дублируется → **Причина:** между страницами `LIMIT`/`pg` возможны дубли → **Фикс:** дедуплицировать по `id` при склейке страниц. *(monolith §3.5)*
- **Симптом:** ссылочное (ref) поле «не сохраняется» → **Причина:** в `t{refReqId}` передали отображаемое имя, а не id записи → **Фикс:** значение ref-поля = id записи целевой таблицы (см. [schema.md](schema.md)). *(monolith §6.4)*

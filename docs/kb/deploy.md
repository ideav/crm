# Деплой: update.php, PR, worktree
> Часть базы знаний Integram. Индекс: [docs/kb/README.md](README.md)

## На пальцах (для человека)
Код приложений Интеграма (шаблоны, JS, CSS) живёт в git-репозитории `ideav/crm`,
а на боевой сервер попадает через скрипт `update.php`: его дёргают по URL, он
тянет нужную ветку с GitHub и раскладывает файлы по серверу согласно маппингам
из `update.conf`. Важно понимать: скрипт только **копирует** новые/обновлённые
файлы и ничего не удаляет — устаревшие файлы остаются, их чистят вручную.
Изменения вносят через pull request (push идёт через форк `unidel2035`). Рабочая
копия `/home/hive/crm` общая на несколько параллельных сессий — чтобы ветки не
перемешались, для своей задачи всегда делают изолированный `git worktree`.

## Операции (для агента)

### Деплой через update.php
```bash
GET https://ideav.ru/update.php?config=update.conf
```
- тянет репозиторий `ideav/crm`, ветку `main` (задано в `update.conf`);
- копирует файлы по маппингам `source : target` из `update.conf` (поддержка `*`);
- **только копирует новые/более свежие файлы — НЕ удаляет** ничего на сервере;
- `index.php` не входит в `update.conf` → деплоится отдельно (вручную/иным путём);
- вывод — HTML-сводка: скопировано / пропущено (актуальные) / ошибки.

Формат `update.conf` (секции `[repository name]`, `repository`/`branch`,
маппинги `путь/* : /target/`) и подробности — [../UPDATE_SCRIPT.md](../UPDATE_SCRIPT.md).

### PR-флоу (через форк unidel2035)
Изменения вносятся pull request'ом. Push в основной репозиторий идёт через форк
`unidel2035`:
```bash
gh auth switch -u unidel2035    # переключиться на аккаунт с правом push
gh auth setup-git
# git push <своя-ветка> в форк, затем PR в ideav/crm
gh auth switch -u gaveron18      # вернуть свой аккаунт после
```

### Изоляция в общей рабочей копии (`git worktree`)
`/home/hive/crm` — общая копия, в ней одновременно идут параллельные сессии.
Под свою задачу создавать отдельный worktree, чтобы не смешивать ветки:
```bash
git worktree add /tmp/crm-<задача> origin/main
# работать в /tmp/crm-<задача>; по завершении git worktree remove
```

Обслуживание/чистка веток репозитория — [../BRANCH_MAINTENANCE.md](../BRANCH_MAINTENANCE.md).

### Таймауты веб-сервера под предел запроса
Запрос ограничен пределом времени `TIME` (30 c по умолчанию, потолок 600 c — см.
[00-start.md](00-start.md)). Чтобы предел отрабатывал сам, а не рвался раньше сервером,
серверные таймауты должны быть **не меньше 600 c**:

| Стек | Параметр |
|---|---|
| php-fpm | `request_terminate_timeout = 600` (пул), `max_execution_time` в php.ini ≥ 600 |
| Apache + proxy_fcgi | `ProxyTimeout 600`, `<Proxy>… timeout=600` |
| Apache + mod_php | ничего не нужно: `Timeout` относится к сетевому вводу-выводу и работающий скрипт не рубит (проверено на `php:8.2-apache`, `Timeout 60` и запросе на 90 c) |
| nginx | `fastcgi_read_timeout 600` |

## Грабли
- **Симптом:** файл удалили из репозитория, а на сервере он остался и продолжает работать → **Причина:** `update.php` только копирует, удаление не делает → **Фикс:** удалять устаревшие файлы вручную через файловый менеджер `dir_admin` (см. [files.md](files.md)). *(monolith / [../UPDATE_SCRIPT.md](../UPDATE_SCRIPT.md))*
- **Симптом:** ветки/коммиты перемешались между несвязанными задачами → **Причина:** несколько параллельных сессий работали в одной общей копии `/home/hive/crm` → **Фикс:** для каждой задачи изолированный `git worktree add /tmp/crm-* origin/main`. *(практика проекта)*
- **Симптом:** правки `index.php` не доехали на сервер после `update.php` → **Причина:** `index.php` не входит в маппинги `update.conf` → **Фикс:** деплоить `index.php` отдельно. *(monolith §4.4 / [../UPDATE_SCRIPT.md](../UPDATE_SCRIPT.md))*

# Роли, права, меню
> Часть базы знаний Integram. Индекс: [docs/kb/README.md](README.md)

## На пальцах (для человека)
В Интеграме доступ строится из трёх вещей: **роль** (что за пользователь),
**права на объекты** (к каким таблицам и отчётам у роли есть доступ и на каком
уровне) и **меню** (какие рабочие места видны). У каждого пользователя одна роль;
роль перечисляет разрешённые объекты с уровнем READ/WRITE/BARRED, а отдельная
**маска** ещё и фильтрует строки (например, менеджер видит только своих клиентов).
Роли, пользователи, меню и запросы — это обычные системные таблицы, и правятся
теми же `_m_*`, что и любые данные. Частая засада: только что созданный
справочник по умолчанию не выдан ни одной роли — выпадающий список будет пустым,
пока не выдашь грант.

## Операции (для агента)

### Системные таблицы
| Таблица | id | Что |
|---|---|---|
| Роль | 42 | `t42`=название, `t135`=описание, `t117`=GRANT «Объекты» (orig 116) |
| Пользователь | 18 | `t18`=логин, `t115`=роль (ref), `t20`=пароль (хешируется), `t33`=ФИО |
| Меню | 151 | `t151`=название, `t153`=адрес, `t391`=иконка; подчинено роли через `up` |
| Запрос | 22 | отчёты (см. [queries.md](queries.md)) |
| Объекты | 116 | гранты роли (подчинены роли, `up=<roleId>`) |

Не задавать руками служебные поля пользователя: `Activity` t124, `Token` t125,
`xsrf` t40, `Retries` t301 — их пишет сервер.

**Супер-пользователь базы.** Помимо создаваемых бизнес-пользователей в каждой базе
есть технический администратор с **логином = имя базы**; сервер заводит его при
создании базы и выдаёт роль Администратор (145). У него **полный доступ ко всей
структуре, таблицам, шаблонам и файлам в обход грантов** (`user == имя_базы ||
user == "admin"`). Его **нельзя пересоздавать, переименовывать или менять ему
роль** (сервер запрещает переименование; аварийное восстановление — `POST
/{db}/restore_admin`). В отчёте о создании проекта его описывают первым пунктом.
Полностью — монолит [../integram-app-workflow.md](../integram-app-workflow.md) §5.1.1.

### Создать роль / меню / пользователя
```bash
POST /{db}/_m_new/42?JSON=1    # up=1, t42=менеджер, t135=описание
POST /{db}/_m_new/151?JSON=1   # up=<roleId>, t151=Приём заказов, t153=order-entry
#   иконка: --form-string 't391=<i class="pi pi-file-edit"></i>'   (HTML, ломается без --form-string)
POST /{db}/_m_new/18?JSON=1    # up=1, t18=manager, t115=<roleId>, t20=<пароль>, t33=ФИО
```
Меню — подчинённая таблица роли: пункты роли читаются `object/151/?F_U=<roleId>`.

### Выдать грант (запись в таблицу Объекты 116)
```bash
POST /{db}/_m_new/116?JSON=1
Fields: up=<roleId>, t116=<объект>, t136=<уровень>, t49=<маска>
```
- `t116` — объект: id таблицы/колонки **или** синтетический id: `0`=Type editor
  (редактор структуры), `1`=All objects (все таблицы), `10`=Files (файловый
  менеджер, см. [files.md](files.md)).
- `t136` — уровень: **52**=BARRED (нет доступа), **53**=READ, **54**=WRITE.
- `t49` — маска фильтра по колонке из `t116`; `t55`=EXPORT, `t56`=DELETE.

Грант на таблицу/отчёт работает одинаково (отчёт — такой же объект). Уровни
наследуются таблица→колонки; отдельной колонке уровень можно понизить (BARRED на
финансовое поле). BARRED-объект не возвращается даже в API/metadata.

Найти гранты роли: `object/116/?JSON_OBJ&F_U={roleId}`; роли с грантом на
таблицу: `object/42/?JSON_OBJ&F_116={tableId}`.

### Маски (`t49`) — фильтрация строк
```bash
t116=18  t49=[USER]      # пользователь видит только себя (по первой колонке)
t116=115 t49=164         # по ref-колонке Роль — только пользователей роли 164 (по id)
t116=<refCol> t49=[USER_ID]   # видит только записи, где он назначен ответственным/владельцем
t116=<col> t49=[ИмяЗапроса]   # маска из запроса (если контекстной переменной не хватает)
```
Контекстные переменные: `[USER]`, `[USER_ID]`, `[ROLE]`, `[ROLE_ID]` и др. Маски
на одну колонку складываются по ИЛИ, на разные колонки — по И.

### Файловый менеджер как объект роли
Синтетический объект `10` (Files) даёт роли доступ к файловому менеджеру
(`dir_admin`). Выдавать его адресно — READ для чтения, WRITE открывает загрузку
файлов, то есть фактически инъекцию кода во фронт приложения; не раздавать
широко. Сами операции с файлами — [files.md](files.md).

### Удалить запись
```bash
POST /{db}/_m_del/{objId}?JSON=1   # любые таблицы; удаление пункта меню каскадит на дочерние
```

Уровни доступа в шаблоне приходят в `{_global_.grants}` (base64-JSON `{typeId:WRITE}`);
в metadata колонка несёт `granted` (WRITE → редактируемая). Полные правила —
монолит [../integram-app-workflow.md](../integram-app-workflow.md) §5.6–5.9.

## Грабли
- **Симптом:** отчёт не виден роли / роль не может открыть report → **Причина:** объект «Запрос» (тип 22) и конкретные отчёты по умолчанию не выданы ни одной роли → **Фикс:** выдать READ-грант (`t116=<reportId> t136=53`) роли в таблицу 116. *(monolith §5.6)*
- **Симптом:** только что созданный справочник даёт пустой выпадающий список, ссылочное значение не отображается → **Причина:** новый объект по умолчанию недоступен ролям → **Фикс:** сразу выдать READ (`t116=<новыйId> t136=53`) всем ролям, чьи рабочие места его используют. *(monolith §5.9.1)*
- **Симптом:** иконка меню `t391` (HTML-тег `<i>`) сохраняется битой/пустой → **Причина:** `curl -F` ломает значение с `<`,`>`,кавычками → **Фикс:** передавать через `--form-string 't391=<i class="pi pi-file-edit"></i>'`. *(monolith §5.2)*
- **Симптом:** ограничение «видит только своих» не срабатывает, видны чужие записи → **Причина:** маски на разные колонки складываются по И, на одну — по ИЛИ; вторая маска расширила видимость → **Фикс:** проверить, что маски заданы на правильные колонки `t116` с нужной логикой. *(monolith §5.9.2)*
- **Симптом:** пытаемся переименовать/пересоздать пользователя с именем базы или снять с него админ-роль → **Причина:** это технический супер-пользователь базы, сервер запрещает его переименование (*«Нельзя изменить имя администратора»*) → **Фикс:** не трогать его; новых заводить отдельно, при поломке чинить через `POST /{db}/restore_admin`. *(monolith §5.1.1)*
- **Симптом:** API-чтение таблицы (`object/{id}/?JSON_OBJ`) у роли БЕЗ READ-гранта возвращает **пустой список без ошибки** (HTTP 200, тело `[]`), а не «нет доступа» → клиент молча деградирует (напр. рабочее место строит план на пустом справочнике, разные браузеры/сессии дают разный результат) → **Причина:** `index.php`, `case "&uni_obj"`: `Check_Grant(...,"READ",FALSE)===FALSE` / `Grant_1level($id)===FALSE` делали `break` (пустой ответ), в отличие от отчётов (`&uni_report` → fatal `Check_Grant`) → **Фикс:** для API (`isApi()`) отдавать `my_die(…, "403 Forbidden")` вместо `break` (внятная ошибка `[{"error":…}]`); HTML-UI оставить мягким (`break`). Клиенту — не глушить пустой/ошибочный ответ критичного справочника, а падать с сообщением. *(#4234, atex production-planning)*

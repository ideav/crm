# Схема (DDL) — команды `_d_*`
> Часть базы знаний Integram. Индекс: [docs/kb/README.md](README.md)

## На пальцах (для человека)
Схема в Интеграме — это сами таблицы и их колонки. Всё описание базы строится из
«типов»: таблица — это тип, у которого первая колонка является именем записи
(номер заказа, название сырья), а остальные колонки — это «реквизиты» (поля).
Между таблицами есть ровно два вида связей: **ссылка** (выбор одной записи из
справочника, как FK) и **подчинённая таблица** (строки, целиком принадлежащие
родителю, — позиции заказа). Перепутать их — самая дорогая ошибка: схему придётся
переделывать. Все правки структуры (создать/переименовать/удалить таблицу или
колонку) делаются командами семейства `_d_*`. Правила проектирования таблиц — в
[../StructureRules.md](../StructureRules.md), различение связей — в монолите
[../integram-app-workflow.md](../integram-app-workflow.md) §2.5.

> ────────── дальше — справочник для агента ──────────

## Операции (для агента)

Авторизация и `_xsrf` — см. [00-start.md](00-start.md). Все `_d_*` — это POST с
полями `token=<token>&_xsrf=<xsrf>`. Ниже опускаю их для краткости.

### Типы колонок (`t=<код>`)
| Код | Тип | Назначение |
|---|---|---|
| 3 | SHORT | короткий текст (названия, имена) |
| 8 | CHARS | длинный текст |
| 9 | DATE | дата `YYYY-MM-DD` |
| 4 | DATETIME | дата+время (в API приходит unix-штамп в секундах!) |
| 11 | BOOLEAN | да/нет |
| 12 | MEMO | большой текст |
| 13 | NUMBER | целое число |
| 14 | SIGNED | число с десятичной точкой |
| 10 | FILE | файл/вложение |
| 6 | PWD | пароль (хешируется, не читается) |
| 5 | GRANT | маска прав (колонка «Объекты роли») |
| 2 | HTML · 7 BUTTON · 15 CALCULATABLE · 16 REPORT_COLUMN · 17 PATH | служебные |

Полный список типов конкретной базы: `GET /{db}/basics`.

### Создать таблицу
```bash
POST /{db}/_d_new?JSON=1      # t=<тип первой колонки>, val="Имя"
# t=3, val="Сырьё"           → { obj: <tableId> }
# t=3, val="Сырьё", unique=1 → справочник (уникальная первая колонка)
# t=13, val="Номер резки", unique=1 → NUMBER-автоинкремент (сам считает 1,2,3…)
```
Первая колонка становится именем записи и адресуется в данных ключом
**`t<tableId>`** (см. [crud.md](crud.md)). Для справочников всегда `unique=1`.

### Добавить обычную колонку (3 шага)
```bash
POST /{db}/_d_new?JSON=1        # t=<typeCode>, val="Дата"      → { obj: <typeId> }
POST /{db}/_d_req/{tableId}?JSON=1   # t=<typeId>               → { id: <reqId>, obj: <tableId> }
POST /{db}/_d_alias/{reqId}?JSON=1   # val="Дата прихода"       # имя колонки в этой таблице
```
`_d_alias` пишет имя в `attrs.alias` реквизита (имя в этой конкретной таблице,
например тип «Дата» → колонка «Дата прихода»). Ключ колонки в данных — `t<reqId>`.

### Создать ссылку (FK) — двухшаговый паттерн
```bash
POST /{db}/_d_ref/{targetTableId}?JSON=1     # → { id: <targetTableId>, obj: <refTypeId> }
POST /{db}/_d_req/{sourceTableId}?JSON=1      # t=<refTypeId> → { id: <reqId>, obj: <sourceTableId> }
POST /{db}/_d_alias/{reqId}?JSON=1            # val="Заказчик"  # псевдоним ссылки (необязательно)
```
В `t` подставляется **ref-тип из `_d_ref`**, а не id целевой таблицы. В записях
ref-поле адресуется `t<reqId>`, значение = id записи целевой таблицы. В metadata
у реквизита появится `ref`=id целевой таблицы и `ref_id`.

### Создать подчинённую таблицу
```bash
# 1) родитель и дочерний тип — обычные _d_new
# 2) добавить дочерний ТИП реквизитом родителя:
POST /{db}/_d_req/{parentTableId}?JSON=1   # t=<childTableId>  (id таблицы прямо в t!)
POST /{db}/_d_alias/{childReqId}?JSON=1    # val="Позиция заказа"
# 3) добавить хотя бы одно поле дочерней таблице (иначе не регистрируется как табличная часть)
# 4) записи дочерней таблицы создаются с up=<parentRecordId> (см. crud.md)
```
Тип связи задаётся тем, что в `t` у `_d_req`: ref-тип → **ссылка** (`ref`);
id дочерней таблицы → **подчинённая таблица** (`arr_id`, без `ref`). В этой форме
реквизит адресуется ключом `t<childTableId>`.

### Переименовать тип/таблицу — `_d_save` (НЕ `_d_alias`)
```bash
POST /{db}/_d_save/{tableId}?JSON   # val="Новое имя", t=3   # переименовать таблицу (t=базовый код первой колонки)
POST /{db}/_d_save/{origId}?JSON    # val="Новое имя", t=<baseTypeCode>  # переименовать тип колонки
```
`origId` = `reqs[i].orig` из metadata (id типа), **НЕ** `reqs[i].id` (id
реквизита). `baseTypeCode` = `reqs[i].type` (3=SHORT, 12=MEMO, 13=NUMBER…).
`_d_alias` ставит имя локально (для одной таблицы/одного реквизита); `_d_save`
меняет сам тип — имя обновится во всех ссылках и в metadata.

### Удалить колонку
```bash
POST /{db}/_d_del_req/{reqId}?JSON=1   # forced=1
```
Безвозвратно стирает значения колонки у всех записей. Сначала **считай и сохрани**
значения (для подчинённой таблицы — по родителям через `F_U`, см. [crud.md](crud.md)).

### Удалить таблицу (тип) — `_d_del/{tableId}` (только если нет ссылок и экземпляров)
Принцип: **сначала сними все референсы, потом сам тип.** Чек-лист зачистки:
1. **Ref-тип бывшего справочника.** Если на таблицу когда-то ссылались, `_d_ref`
   создал особый ref-тип «ссылка на таблицу». Его id — в metadata таблицы как поле
   **`referenced`** (`?JSON_META`). Он висит как «1 экземпляр» и блокирует
   удаление. Снять: `_d_del/{referencedId}` (issue #2746).
2. **Гранты ролей** (таблица 116 «Объекты»): роли с грантом —
   `object/42/?JSON_OBJ&F_116={tableId}`; сами грант-записи —
   `object/116/?JSON_OBJ&F_U={roleId}` (первая колонка вида `{tableId}:Имя`);
   удалить каждую `_m_del/{grantId}`.
3. **Колонки отчётов** (таблица 28): отчёты — `object/22/?JSON_OBJ&F_28={reqId}`;
   колонки — `object/28/?JSON_OBJ&F_U={queryId}`; удалить `_m_del/{colId}`.
4. **Ссылки-реквизиты из других таблиц** — `_d_del_req/{reqId} forced=1`.
5. **Записи (экземпляры)** типа и его подчинённых — `_m_del` (дети каскадятся).
6. `_d_del/{tableId}` — теперь пройдёт. Проверка: `object/{tableId}/?JSON_META` → «не найден».

> Порядок важен: записи → реквизиты → тип. Всегда полный бэкап БД до необратимых операций.

### Миграция «простое поле → справочное» (кратко)
сохранить значения → `_d_new t=3` справочник + наполнить → `_d_ref/{dictId}` →
`_d_del_req` старой колонки → `_d_req t=<refType>` + `_d_alias` → восстановить как
ссылки `_m_set t{newReqId}=<id записи справочника>` → выдать роли READ(53) на
справочник. Типовые кейсы — в [../typical-change-requests.md](../typical-change-requests.md).

### Метаданные (`metadata`) — карта схемы
```bash
GET /{db}/metadata?JSON=1          # вся база (массив объектов-таблиц)
GET /{db}/metadata/{typeId}?JSON=1 # одна таблица
```
Объект таблицы: `id`, `up`(=0 у обычных), `type`(код первой колонки), `val`(имя),
`unique`, `granted`(WRITE → можно писать), `referenced`(на таблицу ссылаются),
`reqs[]`(колонки). Реквизит `reqs[i]`: `id`(ключ `t{id}` в данных/фильтрах),
`val`(имя типа), `type`(базовый код), `orig`(id типа — для `_d_save`),
`ref`+`ref_id`(ссылка), `arr_id`(подчинённая таблица), `attrs`(алиас/дефолт/флаги).
Определение связи: есть `ref`+`ref_id` → ссылка; есть `arr_id` без `ref` →
подчинённая таблица; `orig==="1"` без `ref` → «ссылка на любую запись»; иначе
обычная колонка. Под ограниченной ролью BARRED-объекты в metadata не приходят.

Полные правила проектирования таблиц — [../StructureRules.md](../StructureRules.md).

## Грабли
- **Симптом:** после переименования название таблицы во всём UI показывается как `{"alias":"...","default":"..."}`, `metadata?JSON` битый → **Причина:** вызвали `_d_alias` на id таблицы (типа SHORT/DATETIME) — сервер записал JSON-блоб в `val` таблицы → **Фикс:** чинить `_d_save/{tableId} val=Имя&t=3`; для таблиц переименовывать только через `_d_save`, не `_d_alias`. *(monolith §3.2 шаг 5)*
- **Симптом:** взяли из ответа `_d_req` поле `obj` как id колонки — ключ `t{...}` не работает → **Причина:** `_d_req` возвращает `{obj: tableId, id: reqId}`; id реквизита — в `id`, а `obj` — это id таблицы → **Фикс:** брать `id`. *(monolith §3.2 шаг 6)*
- **Симптом:** «нельзя удалить тип при наличии его экземпляров (1)», хотя записей нет → **Причина:** на таблицу когда-то ссылались, остался ref-тип `referenced` → **Фикс:** найти id в поле `referenced` (`?JSON_META`), `_d_del/{referencedId}`, затем снять гранты (116) и колонки отчётов (28), и только потом `_d_del/{tableId}`. *(issue #2746)*
- **Симптом:** после `_d_del_req` дочерние записи стали недостижимы и тип не удаляется → **Причина:** сняли ссылочный реквизит раньше, чем удалили записи-владельцы → осиротевшие экземпляры → **Фикс:** соблюдать порядок: сначала записи (`_m_del`), потом реквизиты, потом тип. *(monolith §3.7)*
- **Симптом:** повторный `_d_new`/`_d_req` «создал дубль» → **Причина:** на самом деле команды идемпотентны — на существующее имя `_d_new` вернёт существующий id в `obj` с `warnings:"Тип ... уже существует!"`, `_d_req` — существующий reqId → **Фикс:** перевызывать безопасно; брать id из ответа, дубликат не создаётся. *(monolith §3.2.1)*

# Импорт данных батчем — `object/{tableId}?JSON&import=1`

> Часть базы знаний Integram. Индекс: [docs/kb/README.md](README.md)

## На пальцах (для человека)
Массовая загрузка записей в существующую таблицу одним файлом — на порядки быстрее, чем
создавать записи по одной (`_m_new`). Файл загружается полем `bki_file`, движок разбирает
его и вставляет строки. Есть **сокращённый формат** (`plain data`): когда структура таблицы
уже создана, в файле передаются только данные — первая строка `DATA`, дальше по записи на
строку.

> ────────── дальше — справочник для агента ──────────

## Операция

```
POST /{db}/object/{tableId}?JSON&import=1
multipart/form-data:
  bki_file = <файл>
  token    = <token>
  _xsrf    = <xsrf>
```
Заголовок `X-Authorization: <token>` — как для всех вызовов. Нужны права **WRITE+EXPORT**
на тип (или роль admin). Ответ — платформенная форма состояния (HTTP 200 при успехе).

⚠️ **Лимит размера файла — 8 МБ** (`max_size` в `index.php`). Большие наборы слать
**чанками**, каждый чанк — отдельный файл со своей первой строкой `DATA`.

## Сокращённый формат (`plain data`)

Движок включает режим `$plain_data`, если **первая строка файла = `DATA`**
(проверка `substr($buffer,0,4)==="DATA"`; BOM снимается). Структура при этом берётся из уже
существующей таблицы — в файле только значения.

```
DATA
<значение 1-й колонки>;<реквизит 1>;<реквизит 2>;…;
<значение 1-й колонки>;<реквизит 1>;<реквизит 2>;…;
```

Правила:
- **первая строка ровно `DATA`**;
- **по одной записи на строку**; поля разделены `;`;
- **в конце каждой строки — завершающий `;`** (см. грабли ниже);
- порядок полей = `имя записи` (значение первой колонки), затем реквизиты **в порядке
  метаданных таблицы** (как в `metadata`);
- значения — как есть; для JSON-значений запятые внутри безопасны (разделитель — `;`);
- кодировка UTF-8.

### Поведение при повторном импорте
- Если **первая колонка уникальна** (или задан составной ключ уникальности) — запись
  ищется по имени и **обновляется** (upsert), дубль не создаётся.
- Если **не уникальна** — каждая строка **добавляется** новой записью.

## Пример

Таблица «Эмбеддинг» с колонками `vec` (MEMO) и `nbr` (MEMO):

```
DATA
obj0001;[0.12,0.34,0.56];["obj0002","obj0017"];
obj0002;[0.55,0.10,0.20];["obj0001"];
```

```bash
# файл batch.bki в формате выше
curl -s -X POST "https://ideav.ru/{db}/object/{tableId}?JSON&import=1" \
  -H "X-Authorization: <token>" \
  -F "token=<token>" -F "_xsrf=<xsrf>" \
  -F "bki_file=@batch.bki;type=text/plain"
```

```python
import requests
H={"X-Authorization":TOKEN}
X=requests.get(f"{B}/{db}/xsrf?JSON",headers=H).json()["_xsrf"]
content="DATA\n"+"\n".join(f"{name};{f1};{f2};" for name,f1,f2 in rows)+"\n"
requests.post(f"{B}/{db}/object/{tableId}?JSON&import=1",
    files={"bki_file":("batch.bki",content,"text/plain")},
    data={"token":TOKEN,"_xsrf":X}, headers=H)
```

## Грабли
- **Симптом:** последнее поле записи «съедает» начало следующей строки (в значение
  попадает `\n` и текст следующей записи) → **Причина:** нет завершающего `;` — движок
  читает строки, пока число полей не превысит число колонок, и дочитывает следующую →
  **Фикс:** ставить `;` в конце **каждой** строки данных.
- **Симптом:** импорт не включает сокращённый режим / ругается на структуру → **Причина:**
  первая строка не `DATA` → **Фикс:** первая строка файла — ровно `DATA`.
- **Симптом:** `die` «максимальный размер файла» → **Причина:** файл > 8 МБ → **Фикс:**
  бить на чанки (каждый со своей строкой `DATA`); ~10k строк ≈ 3 МБ — безопасно.
- **Симптом:** `die` «нет прав на загрузку объектов этого типа» → **Причина:** нет
  WRITE+EXPORT на тип → **Фикс:** выдать гранты роли или импортировать под admin.
- **Подсказка:** порядок реквизитов в строке должен совпадать с порядком в `metadata`
  таблицы; проверить — `GET /{db}/metadata/{tableId}?JSON=1`.

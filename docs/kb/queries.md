# Запросы / отчёты — `report/`
> Часть базы знаний Integram. Индекс: [docs/kb/README.md](README.md)

## На пальцах (для человека)
Запросы (они же отчёты) — это сохранённые на сервере SQL-подобные выборки.
Их вызывают по имени или числовому id и получают готовые данные — для таблиц,
графиков, воронок, дашбордов. Шаблон обращается к отчёту обычным HTTP-запросом,
руками SQL не пишут. Сам запрос — это запись в системной таблице «Запрос» (тип
22), а его колонки — подчинённая таблица «Колонки запроса» (тип 28); поэтому
создаётся и правится запрос теми же `_m_*`, что и любые данные. Фильтры снаружи
передаются параметрами `FR_`/`TO_`.

**Полный справочник по отчётам** (все форматы, конструктор, как ядро строит SQL,
JOIN, агрегации, дашборды) — [../integram-reports.md](../integram-reports.md).
Здесь — только ключевое.

> ────────── дальше — справочник для агента ──────────

## Операции (для агента)

### Вызвать отчёт
```bash
GET /{db}/report/{id}?JSON_KV          # по числовому id (предпочтительно)
GET /{db}/report/{имяЗапроса}?JSON_KV  # по имени (URL-кодировать кириллицу/пробелы)
```
Форматы: `JSON_KV` — массив объектов (ключи = имена колонок отчёта, для UI);
`JSON_DATA` — только первая строка плоским объектом; `JSON` — полный
платформенный формат с метаданными колонок. **Узнать ТИП каждого поля** (когда
по значению непонятно — дата/штамп/число/ссылка): запросить с `?JSON` и смотреть
`columns[].format` (`DATETIME`/`NUMBER`/`SIGNED`/`SHORT`/`BOOLEAN`…). `JSON_KV`/
`JSON_DATA` типов не отдают. `DATETIME` приходит unix-штампом в секундах.

Узнать id: вызвать по имени и взять заголовок `X-Query-Id` (PR #2723); либо
`GET /{db}/object/22?JSON_OBJ` — список всех отчётов (записи таблицы «Запрос»).
`metadata` отчёты НЕ возвращает.

### Фильтры `FR_` / `TO_`
```bash
report/{id}?JSON_KV&FR_Статус=Новый              # точное совпадение
report/{id}?JSON_KV&FR_СтатусID=9002             # ref-поле по id (суффикс ID, надёжнее)
report/{id}?JSON_KV&FR_SKU=%бумага%              # LIKE
report/{id}?JSON_KV&FR_СтатусID=IN(9002,44895)   # несколько значений
report/{id}?JSON_KV&FR_Date=01.01.2026&TO_Date=31.12.2026   # диапазон дат
```
- `FR_`/`TO_` работают **только** по колонкам, у которых задано «Имя в отчёте»
  (`t100`); без него параметр молча игнорируется.
- Для открытого интервала по дате/числу нужен оператор в значении (`>`/`<`),
  иначе `FR_` понимается как точное совпадение и даст 0 строк.
- `F_{colId}` — то же, что `FR_`, но работает только в `object/` (таблицы), не в `report/`.

### SET-запрос (изменение данных) — два шага
```bash
GET  /{db}/report/{id}?JSON_KV&FR_InviteID=236448      # шаг 1: превью (что изменится)
POST /{db}/report/{id}                                  # шаг 2: confirmed=1&FR_InviteID=236448&_xsrf=...
```
Без `confirmed` — превью затрагиваемых записей; с `confirmed=1` — применяет SET.

### Создать отчёт программно (`_m_*`, без `_d_*`)
```bash
POST /{db}/_m_new/22?JSON               # t22=Мой отчёт                 → { id: queryId }
POST /{db}/_m_new/28?JSON&up={queryId}  # t28={fieldId}, t100=ИмяКолонки  # колонка отчёта
POST /{db}/_m_set/{colId}?JSON          # t104=SUM (агрегация), t109=сортировка
POST /{db}/_m_new/44?JSON&up={queryId}  # t44={tableTypeId}, t265=псевдоним, t266=условие  # JOIN
```
Колонка запроса (тип 28): `t28`=id таблицы/реквизита-источника; `t100`=имя в
отчёте (ключ в `JSON_KV` и внешний alias для `FR_`/`TO_`); `t104`=функция
(`SUM`/`COUNT`/`JSON_ARRAYAGG`…); `t109`=сортировка. JOIN — подчинённая таблица
тип 44. Детали полей и как ядро строит SQL — [../integram-reports.md](../integram-reports.md).

## Грабли
- **Симптом:** `FR_` по дате/числу даёт 0 строк → **Причина:** без оператора `FR_` = точное совпадение; для открытого интервала нужен `>`/`<` → **Фикс:** добавить оператор в значение или использовать пару `FR_`/`TO_` для диапазона. *(monolith «Запросы (report/)»)*
- **Симптом:** внешний фильтр `FR_Колонка=` игнорируется (данные без фильтрации) → **Причина:** у колонки не задано «Имя в отчёте» (`t100`) → **Фикс:** задать `t100` колонке через `_m_set`. *(monolith §«Создание запроса»)*
- **Симптом:** имя колонки `t100` с пробелами не совпадает с ключом в `JSON_KV` / фильтром → **Причина:** ядро заменяет пробелы в `t100` на `_` → **Фикс:** именовать `t100` латиницей в snake_case без пробелов. *(monolith §«единый формат»)*
- **Симптом:** `F_{colId}` не фильтрует отчёт → **Причина:** `F_` работает только в `object/`, не в `report/` → **Фикс:** в `report/` использовать `FR_{alias}`. *(issue #549)*
- **Симптом:** значение колонки отчёта — большое число, непонятно дата это/штамп/номер (напр. `material_batches.batch_no` = `1781038800`) → **Причина:** `JSON_KV`/`JSON_DATA` отдают только значения, без типов; угадывать тип по имени колонки нельзя → **Фикс:** запросить тот же отчёт с форматом **`?JSON`** — он отдаёт `columns[]` с полем `format` (`DATETIME`/`NUMBER`/`SIGNED`/`SHORT`/`BOOLEAN`…). Видно, что `batch_no` это `DATETIME` (unix-штамп в **секундах**) → форматировать как дату `new Date(n*1000)` / `AtexRefSearch.formatDateTime`. *(#3454/#3456, cut-optimizer)*
- **Симптом:** поле позиции заказа (срок/габариты/номер заказа) пропадает в UI после закрытия/выполнения заказа, хотя в БД оно есть → **Причина:** «списочный» отчёт (напр. `positions_list`) возвращает **только активные** позиции; как только позиция выпала из активного списка, брать её поля оттуда нечем → **Фикс:** нужные поля дублировать в **широкий** отчёт, который отдаёт и историю (в atex — `cut_planning`, `report/`-строка на каждое обеспечение): `order_no` (#3624), `position_length` (#3633), `due_date` (#4051); клиент читает их фолбэком, когда позиции нет в активном отчёте. *(#3624/#3633/#4051, atex production-planning)*
- **Симптом:** рабочее место ведёт себя так, будто данных нет (в atex-пульте: «Открыть смену» отрабатывает, а смена остаётся закрытой и кнопки в задании не появляются), при этом записи в таблице ЕСТЬ → **Причина:** `report/` отдаёт **пустой массив со статусом 200**, а не ошибку — так выглядит и правка отчёта (добавление колонки), и невыданный роли грант; клиентский фолбэк «по `catch`» на это не срабатывает, потому что исключения нет → **Фикс:** пустой ответ отчёта не считать доказательством отсутствия данных: перепроверять прямым чтением таблицы и, если там записи есть, ОРАТЬ (`console.error` + тост), а не чинить молча. *(24.07.2026, issue #4359)*
- **Симптом:** отчёт годами отдаёт НЕ ТЕ строки (неверная сортировка в определении), и это никак не проявляется → **Причина:** `LIMIT` из определения отчёта (`REP_LIMIT`, `index.php:3770`) режет запрос клиента через `min(запрошенное, заданное)` — `LIMIT=0,5000` молча схлопывается до лимита отчёта, и при неверном `ORDER BY` приезжает не тот срез; а клиент, который пересортировывает ответ сам (берёт `max(...)` по колонке), к порядку нечувствителен и усечения не замечает → **Фикс:** если от отчёта нужен ЭКСТРЕМУМ (последняя/первая запись), не полагаться ни на порядок, ни на «все строки придут»: фильтровать `FR_` до нужного среза и проверять инвариант (число строк меньше запрошенного лимита, экстремум по нужную сторону границы). Тест, кормящий парсер строками вперемешку, закрепляет безразличие к порядку — сломанное определение отчёта он поймать не может. *(24.07.2026, issue #4371, atex `prev_cut_setup`)*
- **Симптом:** фича молча не работает на ОДНОЙ из баз, в консоли чисто, ошибок нет → **Причина:** нужный отчёт заведён не во всех базах; там, где его нет, сервер отвечает `400` и телом `[{"error":"Запрос не найден"}]`, а клиентский `getJson` не смотрит на `resp.ok` и отдаёт это тело вызывающему **как строки данных** — разбор не находит в «строке» нужных полей, возвращает `null`, и фича тихо выключается → **Фикс:** читающая обёртка обязана ронять промис на `!resp.ok` (как это уже делает `post()` на `{error}` в теле), а необязательные фичи — писать `console.error` вместо пустого `catch`. Клиентский JS один на все базы, поэтому список запросов (`object/22/?JSON_OBJ`) надо сверять по КАЖДОЙ базе, где работает РМ. Тесты, подменяющие `getJson`, до имени отчёта не добираются: нужна проверка САМОГО URL. *(24.07.2026, issue #4371: `next_cut_setup` есть на ateh1 и нет на ateh)*
- **Симптом:** «отчёт возвращает не тот край выборки» при верном фильтре → **Причина:** направление сортировки в определении имеет смысл только вместе с `LIMIT`: ядро отдаёт страницу, и обрезается ДАЛЬНИЙ от сортировки край. Для «последней записи ДО границы» нужна сортировка ↓, для «ближайшей ПОСЛЕ» — ↑; одним запросом оба сценария не закрыть → **Фикс:** два запроса с противоположной `Сорт.` (в atex это `prev_cut_setup` ↓ и `next_cut_setup` ↑). Обрезаться может середина группы (резки), тогда набор приедет неполным и сравнение даст ложное «не совпало». *(24.07.2026, issue #4371)*
- **Симптом:** отчёт отдаёт не тот срез, хотя фильтр задан правильно → **Причина:** у колонок отчёта есть СВОИ «Значение (от)/(до)» — умолчания, зашитые в определение (у `prev_cut_setup`: `task_start` от `[NOW]` до `31/12/2099`, `slitter_id` = конкретный станок). Не передал свой `FR_` — работают они → **Фикс:** передавать `FR_` по каждой колонке, от которой зависит ответ, и не полагаться на «фильтра нет → вернёт всё». *(24.07.2026, issue #4371)*
- **Симптом:** одно и то же рабочее место в РАЗНЫХ браузерах даёт РАЗНЫЙ результат СТАБИЛЬНО: в одном чтение таблицы (`object/{id}/?JSON_OBJ`) приходит ПУСТЫМ, в другом — полным (кэш ни при чём — сервер шлёт `Cache-Control: no-store`) → **Причина:** в браузерах активны РАЗНЫЕ сессии/роли; у одной нет READ-гранта на таблицу, и `case "&uni_obj"` на отказ чтения делал `break` → отдавал ПУСТО без ошибки; клиент молча продолжал с пустым справочником. → **Фикс:** см. [roles.md](roles.md) — сервер: `isApi()`-чтение без гранта → `my_die(…403)`; клиент: не глушить пустой/ошибочный ответ, а падать внятно. *(#4234)*

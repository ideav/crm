# Правила UI/UX (рабочие места ideav/crm, atex)

Накопительный свод UI/UX-правил для рабочих мест. Дополняй при каждом новом
решении или фидбеке от пользователя. Дополняет
[`WORKSPACE_DEVELOPMENT_GUIDE.md`](WORKSPACE_DEVELOPMENT_GUIDE.md) (там — техника,
здесь — поведение интерфейса для пользователя).

## Таблицы и поячейковая правка

- **Верстка не «прыгает» при входе ячейки в режим правки.** Высота строки и
  ширина колонок не меняются между отображением и правкой. Практика:
  - фиксировать высоту редактируемых ячеек (`td.<cell> { height: … }`);
  - НЕ менять `padding` ячейки между отображением и правкой;
  - контрол правки — `box-sizing: border-box; width: 100%`, без `min-width`,
    расширяющего колонку;
  - индикатор активной ячейки — через `box-shadow: inset …` (не меняет геометрию),
    не через рамку/padding, добавляющие размер.
- **Поле ввода вписывается ровно в отведённое место ячейки** — без переполнения по
  ширине и высоте.
- **Выпадающие списки** (ref-select, автокомплит) позиционируются абсолютно
  (`position: absolute`), чтобы не сдвигать строку/таблицу.

*Реализовано в РМ `orders`: `download/atex/css/orders.css` —
`.atex-orders-cell`, `.atex-orders-cell-input`, `.atex-orders-cell .atex-orders-ref-select`
(2026-06-02).*

## Отправка формы с клавиатуры (Ctrl+Enter)

- **Во ВСЕХ формах и модалках `Ctrl+Enter` (на Mac — `Cmd+Enter`), нажатый в любом
  поле ввода, отправляет форму** — пользователю не нужно тянуться мышью к кнопке.
  Работает в текстовых полях, `textarea`, `select` и `contenteditable`.
- Поведение **глобальное и автоматическое** — отдельный обработчик в каждой форме не
  пишется. Логика в `js/form-submit.js` (подключён в `templates/main.html`, грузится на
  каждой странице — ядро и все РМ atex). Порядок поиска цели от поля с фокусом:
  1. ближайшая `<form>` → `form.requestSubmit()` — **настоящие `<form>` покрыты сами**;
  2. ближайшая модалка/панель → клик по главной кнопке.
- **Чтобы не-`<form>` форма (панель/карточка/модалка) поддержала Ctrl+Enter:**
  - модалки ловятся автоматически по `[role="dialog"]`, `.menu-modal`, классам
    `*modal*`/`*overlay*`;
  - прочие панели — пометить контейнер атрибутом **`data-submit-scope`** (пустое
    значение). Внутри scope обработчик кликает первую видимую «главную» кнопку:
    `button[type="submit"]`, `*-btn-primary`, `.btn-primary`, `.menu-modal-btn.save`.
  - если в контейнере несколько кнопок и автоопределение неоднозначно — пометить
    нужную кнопку атрибутом **`data-default-submit`** (имеет приоритет).
- **Не мешает локальным обработчикам:** если поле само обрабатывает `Ctrl+Enter`
  (например, фиксация правки ячейки в `integram-table`) и вызывает `preventDefault`,
  глобальный обработчик не вмешивается. Вне форм/модалок Ctrl+Enter безопасен — ничего
  не делает (без ложных сабмитов).
- **Новая форма/модалка → ничего настраивать не нужно**, если это `<form>` или модалка
  по классу/role. Иначе добавь `data-submit-scope` на контейнер (и при необходимости
  `data-default-submit` на главную кнопку).

*Реализовано глобально: `js/form-submit.js`, подключение в `templates/main.html`;
`data-submit-scope` проставлен в панелях `intake`/`cut-calc`/`cut-planning`/
`production-planning`/`sleeve-cutter`. Тест: `experiments/test-ctrl-enter-submit.js`
(2026-06-02).*

## Модальные формы непрозрачны (фон не просвечивает)

- **Формы создания/редактирования записи и любые модалки обязаны иметь непрозрачный
  фон** — содержимое страницы под формой не должно просвечивать. Прозрачная панель
  сливается с таблицей под ней, форма становится нечитаемой.
- Модалка = два слоя:
  - **бэкдроп** — `position: fixed; inset: 0; background: rgba(0,0,0,.4); z-index: …`
    (затемняет фон, перехватывает клики);
  - **панель формы** — явный непрозрачный фон-поверхность **с фолбэком**
    `var(--input-bg, #fff)` (или `var(--bg-primary, #f8fafc)`), плюс `border-radius`,
    `box-shadow`, `z-index` выше бэкдропа.
- **Главная причина прозрачности:** `var(--token)` к несуществующей переменной **без
  фолбэка** резолвится в пустоту → фон прозрачный. Всегда `var(--input-bg, #fff)`,
  не `var(--surface-card)`.
- Полупрозрачный `rgba`/`transparent` — **только для бэкдропа**, не для панели. Не
  оставляй панель без `background` (унаследует прозрачность).
- Проверяй в светлой и тёмной теме.

*Эталон: `js/integram-table/11-column-settings.js:1048-1053` (бэкдроп `rgba(0,0,0,.32)`
+ панель `background:#fff`), `templates/forms.html:1057-1060` (`var(--card-bg,#fff)`).
Дефект прозрачных форм — issue #3360.*

## Поле фильтра/поиска не теряет фокус при пере-рендере

- **Если ввод в поле перестраивает свой контейнер (живой фильтр/поиск),
  поле обязано сохранять фокус и позицию каретки** — иначе печатать невозможно
  (после каждого символа фокус «выпадает»).
- **Грабли:** `container.innerHTML = ''` (или замена поддерева) удаляет
  сфокусированный `input` → браузер шлёт `blur`. Если восстановление фокуса
  завязано на флаг, который ставит/снимает обработчик `blur`/`focus`, то `blur`
  от самой пересборки гасит флаг **раньше** проверки — фокус не возвращается
  (гонка).
- **Рецепт:** снять состояние в **локальные** переменные **до** очистки DOM, и
  восстанавливать по ним:
  - источник истины «поле было активным» — `document.activeElement === oldInput`
    (а не флаг, который может сбросить `blur` от пересборки);
  - каретку запоминать как `oldInput.selectionStart` и возвращать через
    `setSelectionRange` на **прежнее** место (не в конец строки — иначе правка в
    середине запроса сбивается);
  - на первичном рендере (прежнего поля нет) фокус **не** перехватывать.
- Альтернатива без гонки вовсе — **не пересобирать само поле** на ввод: менять
  только зависимую часть (список/счётчики), оставляя панель фильтров нетронутой.

*Реализовано в РМ `production-planning`: `download/atex/js/production-planning.js`
(`renderQueue`, поле `.atex-pp-search`) — снятие `searchHadFocus`/`searchCaret` до
`box.innerHTML=''` и восстановление после. Дефект потери фокуса — issue #3429
(механизм восстановления через флаг — #3411).*

## Выбор в выпадающем меню передаёт фокус связанному полю

- **Если пункт меню лишь настраивает соседнее поле ввода, после выбора каретка
  обязана оказаться в этом поле.** Меню операторов фильтра, выбор единиц, выбор
  режима — всё это половина действия; вторая половина набирается с клавиатуры, и
  требовать после выбора отдельного прицеливания мышью нельзя.
- **Грабли:** меню закрывается, фокуса нет ни у кого (`document.body`), поле
  выглядит готовым к вводу, но первый же символ уходит в пустоту.
- **Рецепт:**
  - фокус ставить в **каждой** ветке обработчика, в том числе после
    `render()` — перерисовка заменяет узел, и «сфокусированный» до неё элемент
    уже выброшен;
  - `focus({ preventScroll: true })` — иначе браузер подтянет поле во вьюпорт и
    собьёт горизонтальную прокрутку таблицы (issue #2744);
  - каретка **в конец** уже набранного (`setSelectionRange(len, len)`) — оператор
    меняют перед вводом значения, а не посреди слова; на `date`/`number`-полях
    `setSelectionRange` бросает, поэтому только для `type="text"`;
  - у поля может не быть `input` вовсе (кнопка выбора у ссылочной колонки) —
    селектор должен покрывать все формы ячейки, а сохранение фокуса при
    пере-рендере — узнавать их все.

*Реализовано в компоненте `integram-table`: `js/integram-table/10-filter-ui.js`
(`focusFilterControl`, меню `.filter-type-menu`) и `04-render-table.js` (снятие
состояния фокуса, в том числе с `.filter-ref-trigger`). Дефект — issue #4373.*

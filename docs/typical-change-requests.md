# Типовые примеры отработки запросов на изменение

Пользователь формулирует поправку к приложению словами («сделать диаметр втулки
справочным значением»). Этот документ описывает, как такие запросы разбираются на
конкретные действия в базе Integram и в коде рабочего места, и собирает типовые
кейсы с готовой последовательностью шагов.

Рабочее место = таблицы и колонки в базе Integram + HTML/JS-шаблон в репозитории
(`templates/<app>/*.html`, `download/<app>/{js,css}/*`). Поэтому любая поправка
затрагивает две стороны: **схему данных** и **форму**. Их меняют согласованно.

---

## Классификация запросов на изменение

| Вид изменения | Что меняется | Где |
| --- | --- | --- |
| Косметика формы | подписи, порядок полей, тексты, добавление элемента на форму | только шаблон/JS |
| Тип колонки | простое поле ↔ число/дата/булево, простое поле ↔ справочник | схема БД + форма |
| Новая сущность | новая таблица/справочник и её вывод на форму | схема БД + форма |

Порядок разбора любого запроса:

1. **Найти объекты БД, из которых строится форма.** Колонка резолвится по имени
   (alias) из метаданных таблицы. Открыть `metadata/<tableId>?JSON=1` и найти
   нужный реквизит по полю `val`.
2. **Найти место на форме.** В JS рабочего места поля описаны картой
   `key → label → names` (резолв по имени колонки). Найти `key` нужного поля.
3. **Изменить обе стороны согласованно** и прогнать юнит-тест рабочего места.
4. **Проверить доступ ролей.** Если создан новый объект БД (таблица/справочник),
   проверить, какие роли его используют, и выдать им доступ — иначе у роли
   справочник не подгрузится (пустой выпадающий список) или ссылочное значение не
   отобразится. См. раздел [«Доступ ролей к новому объекту»](#доступ-ролей-к-новому-объекту).

## Справочник API Integram

Все запросы — `POST ... ?JSON=1`, с `token` и `_xsrf` в теле
(`docs/MCP.md`). Авторизация по токену: заголовок `X-Authorization: <token>` и
cookie `idb_<db>=<token>`.

| Действие | Endpoint | Тело |
| --- | --- | --- |
| Создать таблицу/тип | `_d_new` | `t=3&val=<имя>` → `{obj: id}` |
| Добавить запись | `_m_new/<tableId>` | `up=1&t<tableId>=<значение>` → `{obj: id}` |
| Добавить дочернюю запись | `_m_new/<subTableId>` | `up=<parentId>&t…=<значение>` |
| Прочитать записи (верхний уровень) | `object/<tableId>?JSON_DATA` | возвращает только записи `up=1` |
| Прочитать дочерние записи | `object/<subTableId>?JSON_DATA&F_U=<parentId>` | записи под конкретным родителем |
| Создать ссылочный тип | `_d_ref/<targetTableId>` | — → `{obj: refTypeId}` |
| Добавить колонку | `_d_req/<tableId>` | `t=<typeId>` → `{id: reqId}` |
| Задать имя колонки | `_d_alias/<reqId>` | `val=<имя>` |
| Удалить колонку | `_d_del_req/<reqId>` | `forced=1` |
| Записать/очистить значение | `_m_set/<objId>` | `t<reqId>=<значение>` (пусто = очистить) |
| Выдать роли доступ к объекту | `_m_new/116` | `up=<roleId>&t116=<objId>&t136=<level>` (`53`=READ, `54`=WRITE; `t49`=маска) → `{obj: id}` |
| Прочитать гранты роли | `object/116?JSON_DATA&F_U=<roleId>` | — → записи `r=[<objId>, "<level>:READ|WRITE", маска, …]` |

Типы колонок (`t`): `3` SHORT, `13` NUMBER, `14` SIGNED, `9` DATE, `4` DATETIME,
`11` BOOLEAN, `12` MEMO.

## Кейс: простое поле → справочное (issue #3049)

**Запрос.** На `/ateh/orders` «Диаметр втулки» сделать справочным значением:
вместо ручного ввода числа — выбор из списка.

**Разбор.** Поле «Диаметр втулки» — это колонка таблицы «Позиция заказа»
(подчинённая к «Заказ»). На форме `download/atex/js/orders.js` она описана в
карте `POSITION_FIELDS` под ключом `sleeve` и выводится как `<input type="number">`.
Рядом уже есть справочные поля «Вид сырья» и «Тип резки» (флаг `ref: true` +
хелпер `refSelectHtml`) — они служат образцом.

### Сторона БД

> **Перед удалением колонки — правильно проверь, есть ли данные.** «Позиция
> заказа» — **подчинённая таблица**: её записи дочерние к заказу (`up=<orderId>`).
> Запрос `object/<subTableId>?JSON_DATA` **без фильтра** возвращает только записи
> уровня `up=1` и для подчинённой таблицы покажет `0`, даже когда позиции есть.
> Поэтому **наличие данных в подчинённой таблице проверяется по родителям**:
> перебрать записи родительской таблицы и для каждой запросить
> `object/<subTableId>?JSON_DATA&F_U=<parentId>`. Удаление колонки **необратимо
> уничтожает её значения** — не удаляй старое поле, пока не убедился, что данные
> перенесены (иначе значения у существующих записей будут потеряны, и поле
> окажется пустым).

Последовательность (шаги очистки/переноса значений (4 и 7) выполняются, только
если по проверке выше в таблице действительно есть записи):

1. **Создать таблицу-справочник** «Диаметр втулки»:
   `_d_new t=3 val=Диаметр втулки` → id справочника.
2. **Заполнить справочник** всеми вариантами, которыми заполняется поле. Источник
   значений — реально используемые диаметры (таблица «Втулкорез»): `20, 25, 40, 76`.
   Для каждого: `_m_new/<dictId> up=1 t<dictId>=<значение>`.
3. **Создать ссылочный тип** на справочник: `_d_ref/<dictId>` → refTypeId.
4. **Считать и сохранить текущие значения старого поля** для всех записей (по
   родителям, см. предупреждение выше) — `<objId> → старое значение`. Это надо
   сделать **до** удаления колонки: удаление уничтожает значения безвозвратно.
5. **Удалить простое поле:** `_d_del_req/<oldReqId> forced=1`.
6. **Добавить ссылочную колонку** в таблицу: `_d_req/<tableId> t=<refTypeId>` →
   новый reqId; задать имя `_d_alias/<newReqId> val=Диаметр втулки`.
7. **Восстановить значения как ссылки.** Для каждой записи из шага 4 сопоставить
   старое значение с записью справочника и записать ссылку:
   `_m_set/<objId> t<newReqId>=<id записи справочника>`. Если в справочнике нет
   значения, которым было заполнено поле, — сначала добавить его в справочник.
8. **Выдать ролям доступ к справочнику** (см. раздел ниже). Справочник «Диаметр
   втулки» (8188) выдан на READ ролям, работающим с позициями: Менеджер (1617),
   Диспетчер (1619), Оператор (1621) — `_m_new/116 up=<roleId> t116=8188 t136=53`.

После: реквизит «Диаметр втулки» в метаданных таблицы имеет `type=3` и `ref=<dictId>`.

### Сторона формы

В `download/atex/js/orders.js`:

1. Пометить поле справочным — в `POSITION_FIELDS` добавить `ref: true` ключу
   `sleeve`.
2. В `renderPositionForm` заменить `<input type="number" data-field="sleeve">` на
   `refSelectHtml(...)`, как у `raw`/`cutType`; вычислить `sleeveCol`/`sleeveOptions`.
3. В `createPosition` читать значение из выпадающего списка
   (`#atex-pos-sleeve-<orderId>`), а не из `data-field`.
4. В инициализации добавить предзагрузку опций: `loadRefOptions(sleeveCol.reqId)`.

Вывод значения в таблице позиций (`pos.values.sleeve`) и отправка в запросе
(`buildCreatePositionRequest`) работают без изменений — ссылочные колонки
обрабатываются тем же кодом, что `raw`/`cutType`.

### Проверка

Юнит-тест рабочего места без DOM/сети прогоняет хелперы на фикстуре
`docs/atex_metadata.json`:

```bash
node experiments/test-issue-2911-atex-orders.js
```

Тест проверяет, что колонка `sleeve` резолвится в реквизит, помечена `ref`, и что
создание позиции кладёт выбранную ссылку в тело запроса.

## Доступ ролей к новому объекту

Новый объект БД (таблица/справочник) **по умолчанию недоступен ролям** — доступ
выдаётся явно. Если справочник, на который ссылается поле формы, не выдан роли:

- выпадающий список у роли **пустой** (опции грузятся из справочника — а роль его не видит);
- в таблицах/списках ссылочное значение **не отображается** (ссылка не резолвится в подпись).

### Где хранится доступ

Права роли на объекты — записи в таблице **116 «Объекты»** (дочерние к роли,
`up=<roleId>`):

| Колонка | reqId | Назначение |
| --- | --- | --- |
| _value | t116 | ID объекта (таблицы/справочника/запроса) |
| Доступ | t136 | Уровень: `53`=READ, `54`=WRITE (ref→таблица 47 «Доступ») |
| Маска | t49 | SQL-условие фильтрации записей (пусто = все) |
| EXPORT / DELETE | t55 / t56 | Флаги выгрузки и удаления |

Эффективные права приходят в шаблон рабочего места как `{_global_.grants}` —
base64-JSON `{ "<typeId>": "WRITE|READ", … }`.

### Как проверить и выдать

1. **Определить роли-потребители.** Прочитать гранты по ролям
   (`object/116?JSON_DATA&F_U=<roleId>`) и найти, кто работает с изменяемой
   таблицей и с её справочниками-аналогами. Для нового справочника нужный уровень —
   как у соседних справочников того же поля (обычно READ).
2. **Выдать доступ** каждой такой роли:
   `_m_new/116 up=<roleId> t116=<dictId> t136=53` (READ).
3. **Проверить:** повторно прочитать `object/116?F_U=<roleId>` — у объекта должен
   появиться уровень `53:READ`.

> Правило: создал новый объект БД, который выводится на форму, — сразу выдай на
> него доступ всем ролям, чьи рабочие места его используют. Это часть отработки
> запроса, а не отдельная задача.

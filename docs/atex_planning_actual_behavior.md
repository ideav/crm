# Как планирование atex работает НА САМОМ ДЕЛЕ (ревью кода)

_Рабочее место «Планирование производства» · `download/atex/js/production-planning/`_

> Документ описывает **фактическое** поведение планировщика в коде `main` (реверс-инжиниринг),
> сопоставляет его с ТЗ (`docs/atex_planning_tz.md`) и фиксирует **корень просрочки** (#4338) и
> целевую «тупую, надёжную» схему. Источник истины по замыслу — ТЗ; здесь — что реально исполняется.
>
> ⚠️ Правятся **модули** `production-planning/*.js`; `production-planning.js` — авто-сборка
> (`build-production-planning.sh`). grep на этих файлах пропускает совпадения — искать через `python3`.

## 1. Точка входа и проходы

`buildSequenceOps` (`20-controller.js`) — единый вход для «Сгенерировать» (`preserveOrder=false`) и
«Упорядочить». Считает `dueDayByCut`/`dueKeyByCut` (срок как индекс дня / YYYYMMDD) и зовёт
`planCutOperations` (`10-planning-engine.js`) с `slotPlacement: true` (по умолчанию `SLOT_PLACEMENT=1`).

Внутри — **несколько проходов подряд**, каждый следующий может переопределить предыдущий.
**Реальный день, на который садится задание, определяет `splitMachineQueue` (реальная упаковка);**
слой §8 выбирает только станок + позицию в очереди, но НЕ день напрямую.

| Проход | Функция | Что делает |
|---|---|---|
| **A. §8 размещение** | `computeSlotPlacement` → `placeSlot`/`scorePosition` (`15-slot-placement.js`) | Жадно, по одному заданию: перебор **всех** станков × **всех** точек вставки (`for idx 0..len`), выбор минимального штрафа. Вставку между заданиями УМЕЕТ. День оценивает **эвристикой** `prefixDayOffset` (плоская ёмкость). |
| **B. §12 релокация** | `relocatePass` (`15-slot-placement.js`) | По РЕАЛЬНЫМ дням (`splitMachineQueue`) двигает просроченные/за-фольгой — только если строго дешевле. Кандидата оценивает тем же **оптимистичным** днём → «двигает впустую». |
| **C. #4118/#4200/#4203 спасение** | `relocateOverdueReal` (`15-slot-placement.js`) | Честно: каждую вставку проверяет реальной пере-упаковкой. Двигает **только само** просроченное; принимает, только если его реальный день строго уменьшился И **никакое другое** срочное не углубило просрочку. |
| **D. внутридневная пересортировка** | `10-planning-engine.js` | Косметика порядка внутри уже зафиксированного дня; день не меняет. |

`#4203 «дозаклад по станкам»` — **диагностика** (append-проба для лога), не попытка размещения.

## 2. Веса штрафов (живые значения ateh1 = ТЗ §14)

Читаются `planWeight(settings, key)`: из «Настройки»/ATEH, иначе дефолт кода. Применяются в
`transitionCost`. ⚠️ **Дефолты кода ≠ таблице ATEH** (напр. DEADLINE код 100 / ATEH 200) — берётся ATEH.

| Штраф | ATEH | Дефолт кода | За что | Тип |
|---|---|---|---|---|
| `DEADLINE_COST_MN` | **200** | 100 | день размещения **позже** срока — доминирует | мягкий, но главный |
| `EXACT_DEADLINE_COST_MN` | 9 | 33 | ровно в день срока | мягкий |
| `FOIL_NOTEND_COST_MN` | 80 | 60 | фольга не в конце дня (симметрично #4202) | мягкий |
| `KNIVES_INCREASE_COST_MN` | 50 | 50 | смена ножей (полос больше) | мягкий |
| `KNIVES_CHANGE_COST_MN` | 30 | 30 | смена ножей (полос меньше) | мягкий |
| `MATERIAL_CHANGE_COST_MN` | 15 | 15 | смена сырья/намотки | мягкий |
| `ORDER_DIFF_PENALTY_MN` | 12 (±10) | 10 | соседство/разрыв заказа (#4194) | мягкий |
| `MAX_DISTANCE_COST_MN` | 10 | 25 | отрыв станка вперёд >24ч — разгоняет загрузку | мягкий |
| `MAX_SLOTS_DISTANCE_HR` | 24 | 24 | порог отрыва | лимит |
| `CHANGE_SLITTER/DAY_COST_MN` | 3 / 3 | 3 / 3 | перенос на др. станок/день (только релокация) | мягкий |
| `LEADER_COST_MN` | 2 | 2 | лидер между резками | мягкий |
| `SLOT_SPLIT_COST_MN` | 2 | 2 | разрыв по дням | **МЁРТВЫЙ** — объявлен, нигде не применяется |

Приоритеты (ТЗ §11): **срок(200) > фольга(80) > переналадка(30–50/15) > загрузка(10) > заказ(10) > похожесть**.

## 3. Жёсткие правила (запреты, не веса — ТЗ §15)

- **Раньше «С» — никогда** (упаковка от дня 0 вперёд; прошлые незафикс. цепочки исключены из входа).
- **Стоп-лист сырья + лимит ширины джамбо** (`feasibleMachineFor`) — жёсткий фильтр в каждом переборе.
- **Отпуск станка** (downtime blocked ranges); длинный отпуск сбрасывает заправку.
- **Выходной/праздник** (Календарь #3788, `machineDayOffFor`) — день пропускается; срок считается по
  **КАЛЕНДАРНОМУ** дню (#4200: `floor(windowStartMin/1440)`), не логическому.
- **Фиксация 🔒** (`fixedDay`) — задание держит свой день; в §8 — неподвижный сосед.
- **Захлёст** (`MAX_OVERWORK_CUTS/TUNE`) — можно перетянуть конец смены на ≤ лимит.
- **Заморозка #4326** (сейчас в main = **Вариант A, пин**): задания дня временно `c.fixed`, день НЕ
  блокируется. ⚠️ Это **временная затычка** — по ТЗ/заказчику должно быть «день пустой, жёстко»
  (см. §6): станет жёстким блоком после починки ядра.

## 4. Исключения / мягкие правила

- **Фольга** — НЕ жёсткое правило, а штраф (80), симметричный, тянет фольгу в конец дня (§12.1).
- **Соседство заказа** (±10), **распределение загрузки** (distance 10, без отдельного балансировщика).
- **Разбиение по дням** (§9) + склейка островов одного сырья (принимается только если строго
  снижает переналадку и не создаёт просрочку).
- **gap-fill** (#3739) — режим внутри `splitMachineQueue`, тянет будущее в дыры дня по **совпадению
  сырья; к сроку СЛЕП**.
- **Обед/перерывы/уборка** — для планирования прозрачны (резерв ёмкости), рисуются только на Ганте.

## 5. КОРЕНЬ ПРОСРОЧКИ (#4338) — почему срочное уезжает за срок при свободной ёмкости

Живой случай: 634529 (сырьё 1253 — влезает на любой станок, работа 190 мин, срок 22.07 = день 1),
день 1 занят на 35% — но поставлен на день 2 (просрочка). Три причины складываются:

1. **Оптимистичная оценка дня.** Основной проход считает день кандидата плоской ёмкостью
   (`prefixDayOffset`/`capacityMin`) — код сам помечает это «ЛИШЬ эвристика … оптимистично». Дозаклад,
   реально садящийся на день 2, **выглядит в срок** → штраф `DEADLINE=200` **не срабатывает** →
   перебор берёт дешёвый дозаклад (совпало сырьё, ≈0) вместо вставки в день 1 (≈ переналадка 45).
2. **Штраф срока — только своему слоту.** `insertionCost` начисляет DEADLINE в переходе prev→slot; у
   перехода slot→next `ctxAfter` **без** `placementDayKey` → сосед, которого вставка оттолкнула за
   срок, платит **ноль**. Дешёвое несрочное задание бесплатно топит срочное. Пере-оценки нет (жадно).
3. **Спасение не вытесняет.** `relocateOverdueReal` перебирает все точки, но принимает вставку только
   если она **не углубляет** просрочку **никакого** другого срочного (`harms==false`). В цеху, забитом
   до 6-го дня, почти любая вставка в день 1 кого-то толкает → отвергается → безвредный ход — только
   дозаклад в хвост (день 6). Само оно **не отодвигает несрочного соседа**, чтобы освободить день 1.

**Итог:** планировщик НЕ делает единственное, что помогает — **вытеснить несрочного соседа под
срочное** (ТЗ §12.1 «вытесняя оттуда поздне-срочную нефольгу» — в прозе есть, в коде нет).

## 6. Целевая схема: тупой, прочный, надёжный планировщик

**Ядро — один проход (ТЗ §8):** для каждого задания (по возрастанию срока) перебрать ВСЕ допустимые
точки вставки на всех станках под жёсткими правилами (замороженный день исключён как цель; стоп-лист/
ширина; «С»; отпуск; выходной), посчитать **суммарный штраф** (§8.2) и поставить в **минимум**.
Две правки к текущему §8, чтобы он стал честным:

1. **День кандидата — РЕАЛЬНЫМ упаковщиком** (`realPackFn` уже есть), не оценкой. Тогда `DEADLINE=200`
   срабатывает честно, и вставка в день 1 сама выигрывает у дозаклада в день 6.
2. **Штраф срока — ВЫТЕСНЯЕМЫМ соседям**: минимизировать **суммарное** опоздание плана, а не только
   своего задания. С весом 200 это автоматически = «срок первым, с вытеснением несрочного».

**Пост-проход (обязателен, ограниченный — иначе на плотном графике не разместить):**
1. проверить просрочки; если есть — **затолкать куда влезет** (с вытеснением несрочного/низкого
   приоритета);
2. оптимизировать очередь после сдвигов (снизить суммарные переналадки).

**Термин остановки (анти-зацикл):** итерируем, пока **суммарные переналадки строго убывают** И
**хэш расстановки не повторялся**; повтор хэша ⇒ цикл ⇒ стоп (+ жёсткий потолок итераций как бэкстоп).

**Ограничение «не добивать день силой»:** если день не набивается до `DAY_DURATION_MN` (450) из-за
🔒-заданий или недопущения просрочки — **оставляем как есть**; день дозаполнится при следующих
необеспеченных заказах.

**Что удаляется/схлопывается** после этого: §12 `relocatePass` (оценочная, «двигает впустую»),
#4118/#4200/#4203-петли и append-фолбэк §8.4 — они существуют только чтобы латать враньё оценки.
Остаётся: **один арбитр** (реальный упаковщик), **одна цель** (сумма штрафов, вкл. DEADLINE каждого
задания), **жёсткие правила как фильтры**, + ограниченный пост-проход по хэшу.

**После починки ядра — заморозка → жёсткий блок** (день пустой): вытеснение раскидает срочное по
другим дням без просрочки.

## 7. Находка: проходы РАСХОДЯТСЯ в подсчёте дня (корень «ложной» просрочки)

Живой прогон (симуляция жёсткого блока дня 0 = 21.07) выявил: **применённый `planStart` и детектор
просрочки считают РАЗНЫЙ день одного задания.**

- **634529** (111 проходов, НЕ разбито по дням — продолжений нет): применённый `planStart` = **день 1
  (22.07)** = его срок ⇒ в срок. А детектор просрочки `realDaysFrom` (отдельная пере-упаковка
  `packAll` через `windingDaysFromSegs`, `10-planning-engine.js:3725`) даёт **день 2** ⇒ «просрочено».
- То же у **634304**: применён день 2 (23.07 = срок), детектор — день 3.

Причина: **updates (что пишется в БД) и overdueResidual (что показывает «просрочено: N») берутся из
РАЗНЫХ упаковок** — применённая раскладка и «честная» пере-упаковка детектора не совпадают, когда
день 0 заблокирован. Поэтому:
1. часть тревоги #4338 — **несогласованность индикатора**, а не гарантированно позднее производство;
2. **пока источник дня не единый, «убрать просрочку» неопределено** — жадный проход будет
   оптимизировать против того дня, которого в применённом плане нет.

**Вывод для редизайна:** первый и обязательный шаг — **ОДИН источник реального дня** (та же упаковка,
что пишет `planStart`), и уже по нему — единый штраф и жадный проход. Это и делает узел «тупым и
надёжным»: одна упаковка → один день → один штраф → один перебор минимума. Разногласие проходов
(applied vs detector) устраняется по построению.

## 8. РЕАЛИЗОВАНО (#4338): `greedyRefine` — единый жадный проход по реальному дню

Пост-проход `greedyRefine` (`15-slot-placement.js`, вызывается в `10-planning-engine.js` после
overdue-rescue), построенный по принципу «тупо, прочно, надёжно»:

- **Один источник дня.** Штраф каждого хода считается по РЕАЛЬНОЙ упаковке (`realPackFn` →
  `windingDaysFromSegs`) — той же, что определяет применённый `planStart`. Разногласие §7 (applied
  vs detector) снято по построению.
- **Единый штраф, худшее — первым.** Задания сортируются по штрафу (срок ×200 ПРОПОРЦИОНАЛЬНО дням
  опоздания доминирует над переналадкой ≤45); худшее двигается в позицию минимального суммарного
  штрафа. Стоимость хода учитывает ОБА затронутых станка ⇒ ход **никогда не увеличивает суммарную
  просрочку** (загон соседа за срок делает delta положительной → отклонён). Просрочка не создаётся
  по построению.
- **Только просроченные.** Внешний цикл ограничен заданиями с `dlPen>0` (их единицы). Переналадку
  прочих оптимизируют слой размещения + внутридневная пересортировка + склейка островов —
  дублировать их дорого и не нужно. Уникальная ценность greedy — **вытеснение просроченного**, чего
  старые проходы не умеют. Нет просрочки ⇒ мгновенный выход (норма-генерация ≈ free).
- **Без зацикливания.** Хэш применённых расстановок (`seen`) отсекает повторы; штраф строго убывает
  (delta < −EPS) ⇒ монотонно и конечно. Хэш-столкновение = пропустить это задание, не стоп.

**Замер (живой ateh1, симуляция жёсткого блока дня):** overdue **2→1** (634304 починен). Остаток
**634529** — честный дефицит РАННЕЙ ёмкости дня 1 (задание ~190 мин должно стартовать до ~13:30,
чтобы намотка закрылась к 16:30; ранние слоты дня 1 заняты другими срочными — вытеснить их = загнать
ИХ за срок, delta>0 → отклонено). Это §13.6 «честный дефицит», а не ошибка. Норма без блока: overdue
0, время как раньше. Полный набор автотестов: 156 pass (падают лишь 6 добаговых, к planner не
относятся).

**Заморозка пока = Вариант A (закрепление, не блок)** — просрочки не создаёт. Жёсткий блок дня (день
пустой) безопасен ТОЛЬКО когда вытеснение гарантированно избегает просрочки; при остатке типа 634529
блок вернул бы просрочку — поэтому переход на блок отложен до полной робастности вытеснения.

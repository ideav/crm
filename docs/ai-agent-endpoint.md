# API ИИ-агента для Integram CRM

Контракт между CRM (`index.php`, маршрут `/{db}/ai/agent`) и внешним сервисом
ИИ-агента. Описывает, что нужно реализовать **на стороне агента**, чтобы он
подключился к асинхронному слою чата (issue #3410).

## Где это встраивается

При заданном `INTEGRAM_AGENT_ENDPOINT` функция `callIntegramAgent()` в `index.php`
делает **POST JSON** на endpoint агента. Клиентский слой (issue #3410) уже умеет:
показывать индикатор «агент думает», опрашивать статус задачи и восстанавливать
результат при повторном открытии панели (в т.ч. из другого браузера). Агенту нужно
реализовать только HTTP-контракт ниже.

Если `INTEGRAM_AGENT_ENDPOINT` пуст — агент не вызывается, CRM возвращает заглушку
«готов к работе, оплата подтверждена».

## 1. Запрос, который CRM присылает агенту

```
POST <INTEGRAM_AGENT_ENDPOINT>
Content-Type: application/json; charset=UTF-8
Accept: application/json
User-Agent: Integram AI Agent
Authorization: Bearer <INTEGRAM_AGENT_TOKEN>      # только если токен задан
```

```json
{
  "db": "acme",                 // имя текущей базы — ВСЯ работа только в ней
  "user": "acme",               // имя пользователя (= владелец базы)
  "message": "текст запроса пользователя",
  "attachments": [
    { "name": "f.csv", "type": "text/csv", "size": 1234, "content": "<base64>" }
  ]
}
```

`attachments` — массив (может быть пустым); `content` — содержимое файла в base64
(лимит 10 МБ на файл, накладывается CRM).

## 2. Ответ агента

### Вариант A — синхронно (если успеваете за ≤ 30–60 c)

Работает сразу, без изменений на стороне CRM. HTTP **200**, тело — любая из форм
(CRM понимает их в этом порядке приоритета):

```json
{ "content": "ответ агента" }
```

Также принимаются:
- OpenAI-стиль: `{"choices":[{"message":{"content":"..."}}]}` или `choices[0].text`;
- Gemini-стиль: `candidates[0].content.parts[0].text`;
- просто **plain-text тело** ответа.

Пустой ответ → CRM возвращает пользователю ошибку 502.

> Достаточно поднять этот endpoint и задать `INTEGRAM_AGENT_ENDPOINT`. **Но** если
> ответ дольше таймаута сервера (30–60 c) — запрос оборвётся, и задача зависнет в
> статусе `processing`. Для длительных задач / очереди используйте вариант B.

### Вариант B — асинхронно (рекомендуется для очереди и ответов «на минуты»)

Агент **не держит** соединение. На `POST` сразу отвечает HTTP **202**:

```json
{ "job_id": "agent-internal-id", "status": "queued" }
```

Готовый результат доставляется одним из двух способов:

**B1 — callback (рекомендуется: проще и надёжнее).**
Когда задача готова, агент сам вызывает webhook CRM (URL и секрет CRM передаст в
запросе — см. §3):

```
POST <callback_url>
X-Agent-Secret: <callback_secret>
```
```json
{ "job_id": "<наш job_id из запроса>", "status": "done", "content": "ответ" }
```
```json
{ "job_id": "<наш job_id>", "status": "error", "error": "текст ошибки" }
```

**B2 — polling (CRM опрашивает агента).**
Агент поднимает статус-эндпоинт:

```
GET <status_url>?job_id=<agent-internal-id>
```
```json
{ "status": "queued|processing|done|error", "content": "...", "error": "..." }
```

## 3. Что добавит CRM для варианта B (отдельный тикет)

`callIntegramAgent()` сейчас синхронный. Для варианта B на стороне CRM будет
добавлено:

- **B1:** передача в запрос полей `callback_url`, `job_id`, `callback_secret`;
  новый маршрут `POST /{db}/ai/agent/callback`, который принимает результат и
  дописывает его в задачу. Серверное хранилище задач (issue #3410) уже готово
  принять результат.
- **B2:** сохранение `agent job_id` в задаче и периодический опрос `status_url`.

Контракт `/{db}/ai/agent` для фронтенда при этом **не меняется** — поллинг статуса
и восстановление результата уже реализованы.

## 4. Ошибки

- При ошибке верните HTTP **4xx/5xx** с телом `{"error":{"message":"..."}}`
  (также принимаются `{"error":"..."}` и `{"message":"..."}`). CRM покажет текст
  пользователю и пометит задачу `error`.
- Успешный ответ всегда — **2xx**.

## 5. Требования к endpoint (валидация в CRM, защита от SSRF)

- Только **абсолютный HTTPS-URL**.
- Запрещены `localhost`, `127.0.0.1`, `::1`, хосты `*.local`, приватные и
  служебные IP — хост должен быть публичным.

## 6. Безопасность (обязательно)

См. `docs/integram-app-workflow.md`. Кратко:

- Работать **только в базе `db`** из запроса; никакого межбазового доступа.
- Не агрегировать и не возвращать секреты (колонки `PWD`, `Token`, `xsrf`),
  не раскрывать внутреннюю инфраструктуру.
- Поле `content` рендерится на клиенте как **обычный текст** (без HTML/markdown) —
  возвращайте читаемый plain-text. Если потребуется markdown — нужно добавить
  его рендеринг на клиенте отдельно.

## 7. Конфигурация на стороне CRM

Значения читаются через `aiConfigValue()` — сначала PHP-константа `define(...)`,
затем переменная окружения `getenv(...)`.

| Имя | Назначение | По умолчанию |
| --- | --- | --- |
| `INTEGRAM_AGENT_ENDPOINT` (или `AI_AGENT_ENDPOINT`) | HTTPS-URL агента | — (пусто = заглушка) |
| `INTEGRAM_AGENT_TOKEN` (или `AI_AGENT_TOKEN`) | Bearer-токен авторизации | — |
| `AI_AGENT_TIMEOUT` (или `INTEGRAM_AGENT_TIMEOUT`) | таймаут вызова, сек | 60 (макс 600) |

> Реальный таймаут ограничен ещё и `max_execution_time` / `request_terminate_timeout`
> / `fastcgi_read_timeout` сервера — учитывайте это для варианта A.

## 8. Быстрый тест варианта A

```bash
curl -sS -X POST https://<agent-host>/path \
  -H 'Content-Type: application/json' \
  -H 'Authorization: Bearer <token>' \
  -d '{"db":"acme","user":"acme","message":"привет","attachments":[]}'
# ожидаем: {"content":"..."} (или plain-text), HTTP 200
```

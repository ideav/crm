# Мини-проект Интеграм от Excel до рабочих экранов

**Задача:** [#2986](https://github.com/ideav/crm/issues/2986)
**Основа:** проект Atex, упрощённый до минимального состава.
**Для кого:** для ИИ-агента, который впервые собирает приложение на Интеграм и
должен увидеть один связный путь от пользовательского Excel до работающих
экранов.

Этот документ не заменяет [`docs/integram-app-workflow.md`](integram-app-workflow.md).
Он показывает тот же процесс на одном маленьком примере, чтобы не собирать
картину из фрагментов `atex`, `ateh`, `sportzania` и отдельных статей.

## Задача документа

Пользователь приносит два Excel-файла: справочники и заказы на резку рулонов.
Нужно собрать рабочее приложение **«Мини-Atex: резка рулонов»**:

1. Менеджер принимает заказ и добавляет позиции.
2. Диспетчер планирует производственную резку по позиции заказа.
3. Оператор отмечает выполнение резки.
4. Руководитель видит статусы заказов и резок.

Минимальный состав сохраняет главные уроки Atex:

- повторяющиеся значения из Excel превращаются в справочники и **ссылки**;
- строки заказа становятся **подчинённой таблицей** заказа;
- производственная операция связывается с конкретной позицией заказа;
- экраны работают с одной схемой данных, а не каждый со своей копией таблиц.

Полный Atex живёт в [`docs/atex_metadata.json`](atex_metadata.json), сценарий его
проверки — в [`docs/atex_full_cycle_test_scenario.md`](atex_full_cycle_test_scenario.md),
актуальные рабочие места — в `templates/atex/`, например
[`templates/atex/orders.html`](../templates/atex/orders.html) и
[`templates/atex/production-planning.html`](../templates/atex/production-planning.html).
Здесь оставлены только таблицы и экраны, достаточные для первого полного прохода.

## Исходный Excel

Пользователь не описывает систему технически. Он даёт предметную область и
файлы. В этом примере вход выглядит так.

### Файл `Справочники.xlsx`, лист `Клиенты`

```csv
Клиент,Телефон,Email
ООО "Ромашка-Термолента",+7 495 010-29-86,orders@romashka.example
ИП Петров,+7 812 010-29-86,petrov@example.test
```

### Файл `Справочники.xlsx`, лист `Сырьё`

```csv
Вид сырья,Полное название,Ширина мм,Длина рулона м
Jumbo Thermal 910/4000,"Термобумага, ширина 910 мм, намотка 4000 м",910,4000
Jumbo Eco 760/3000,"Эко-бумага, ширина 760 мм, намотка 3000 м",760,3000
```

### Файл `Заказы.xlsx`, лист `Заказы`

```csv
Номер заказа,Клиент,Дата создания,Статус,Менеджер,Комментарий
АТХ-0001,ООО "Ромашка-Термолента",2026-05-31,Новый,manager,Срочный тестовый заказ
АТХ-0002,ИП Петров,2026-05-31,Новый,manager,Обычная поставка
```

### Файл `Заказы.xlsx`, лист `Позиции`

```csv
Номер заказа,Позиция,Вид сырья,Ширина мм,Длина м,Кол-во,Статус
АТХ-0001,ТТ-58 ECO 57x40,Jumbo Thermal 910/4000,57,1200,10,Новая
АТХ-0001,ТТ-40 ECO 40x30,Jumbo Thermal 910/4000,40,800,8,Новая
АТХ-0002,ТТ-76 ECO 76x60,Jumbo Eco 760/3000,76,1000,6,Новая
```

### Что агент должен понять из Excel

| Наблюдение | Решение в Интеграме |
|---|---|
| `Клиент` повторяется в заказах | `Заказ.Клиент` — Ссылка на таблицу `Клиент` |
| `Вид сырья` повторяется в позициях | `Позиция заказа.Вид сырья` — Ссылка на таблицу `Вид сырья` |
| У одного заказа несколько строк в листе `Позиции` | `Позиция заказа` — Подчинённая таблица заказа |
| `Номер заказа` связывает листы Excel | При импорте позиции создаются с `up=<id заказа>` |
| Производство выполняет не весь заказ, а отдельную строку | `Производственная резка.Позиция заказа` — Ссылка на позицию заказа |

Если агент видит такие повторы, он не спрашивает пользователя «какие таблицы
создать». Он предлагает готовую структуру и спрашивает только то, чего нет в
файлах: нужны ли клиентские логины, сколько операторов, какие статусы считать
финальными, есть ли логотип и фирменные цвета.

## Схема данных

Схема ниже — минимальная версия Atex. Названия таблиц оставлены такими же, чтобы
после первого прохода было легко перейти к полной схеме из `docs/atex_metadata.json`.

### Таблица `Клиент`

| Поле | Тип | Откуда берём | Примечание |
|---|---|---|---|
| `Клиент` | SHORT (3) | `Клиенты.Клиент` | Первая колонка, имя записи |
| `Телефон` | SHORT (3) | `Клиенты.Телефон` | Обычное поле |
| `Email` | SHORT (3) | `Клиенты.Email` | Обычное поле |

### Таблица `Вид сырья`

| Поле | Тип | Откуда берём | Примечание |
|---|---|---|---|
| `Вид сырья` | SHORT (3) | `Сырьё.Вид сырья` | Первая колонка |
| `Полное название` | SHORT (3) | `Сырьё.Полное название` | Можно сделать MEMO, если описание длинное |
| `Ширина, мм` | NUMBER (13) | `Сырьё.Ширина мм` | Число |
| `Длина рулона, м` | NUMBER (13) | `Сырьё.Длина рулона м` | Число |

### Таблица `Заказ`

| Поле | Тип | Связь | Примечание |
|---|---|---|---|
| `Заказ` | SHORT (3) | - | Номер заказа из Excel |
| `Клиент` | ref -> `Клиент` | Ссылка | Повторяющееся значение из листа заказов |
| `Менеджер` | ref -> `Пользователь` | Ссылка | По логину `manager`, если пользователь уже создан |
| `Дата создания` | DATE (9) | - | `YYYY-MM-DD` |
| `Статус` | SHORT (3) | - | Для мини-проекта можно оставить строкой |
| `Комментарий` | MEMO (12) | - | Свободный текст |
| `Позиция заказа` | arr -> `Позиция заказа` | Подчинённая таблица | Позиции принадлежат заказу |

### Таблица `Позиция заказа`

| Поле | Тип | Связь | Примечание |
|---|---|---|---|
| `Позиция заказа` | SHORT (3) | - | Название позиции из Excel |
| `Вид сырья` | ref -> `Вид сырья` | Ссылка | Один вид сырья используется в разных позициях |
| `Ширина, мм` | NUMBER (13) | - | Ширина готового ролика |
| `Длина, м` | SIGNED (14) | - | Намотка |
| `Кол-во` | NUMBER (13) | - | Количество роликов |
| `Статус` | SHORT (3) | - | `Новая`, `Запланирована`, `Готова` |

### Таблица `Производственная резка`

| Поле | Тип | Связь | Примечание |
|---|---|---|---|
| `Производственная резка` | NUMBER (13) | - | Автонумер, `unique=1` |
| `Позиция заказа` | ref -> `Позиция заказа` | Ссылка | Резка выполняется под конкретную позицию |
| `Дата план` | DATE (9) | - | Плановая дата |
| `Оператор` | ref -> `Пользователь` | Ссылка | Кто выполняет |
| `Статус` | SHORT (3) | - | `Ожидает`, `В работе`, `Готово` |
| `Кол-во факт` | NUMBER (13) | - | Сколько роликов сделано |
| `Комментарий` | MEMO (12) | - | Причины отклонений |

### Самопроверка связей

После создания структуры агент перечитывает `metadata?JSON=1` и проверяет:

```text
Заказ.Позиция заказа: есть arr_id, нет ref
Заказ.Клиент: есть ref на Клиент
Позиция заказа.Вид сырья: есть ref на Вид сырья
Производственная резка.Позиция заказа: есть ref на Позиция заказа
Производственная резка.Оператор: есть ref на Пользователь
```

Если `Клиент` или `Вид сырья` появились как `arr_id`, значит справочник ошибочно
создан подчинённой таблицей. Это ошибка из антипримера `ateh`: рабочие места
потом не смогут выбрать общий справочник.

## Порядок сборки

### 1. Подтвердить пользовательский путь

Перед технической работой агент показывает пользователю короткое резюме:

```markdown
# Приложение: Мини-Atex - резка рулонов

Менеджер создаёт заказ и позиции. Диспетчер планирует резку по позиции.
Оператор отмечает выполнение. Руководитель смотрит статусы.

Из ваших файлов будут таблицы: Клиент, Вид сырья, Заказ, Позиция заказа,
Производственная резка.

Нужно уточнить:
- один оператор может выполнять любые резки или только свои;
- нужен ли клиентский доступ на первом этапе;
- есть ли сайт, логотип или фирменные цвета.
```

К технической реализации переходят только после подтверждения.

### 2. Создать таблицы и связи

Скрипт создания структуры может быть PowerShell или bash. Логика такая же, как в
основном workflow:

```text
1. Получить metadata пустой базы.
2. Создать таблицы: Клиент, Вид сырья, Заказ, Позиция заказа, Производственная резка.
3. Добавить обычные поля.
4. Создать ref-связи через _d_ref + _d_req.
5. Подключить Позиция заказа к Заказ как подчинённую таблицу через _d_req t=<id Позиция заказа>.
6. Повторно прочитать metadata и проверить ref/arr_id.
```

Технические команды и особенности `_d_new`, `_d_req`, `_d_ref`, `_d_alias` описаны
в [`integram-app-workflow.md`](integram-app-workflow.md#32-последовательность-api-вызовов).

### 3. Загрузить данные из Excel

Импорт делается в порядке зависимостей:

```text
1. Клиенты
2. Виды сырья
3. Заказы
4. Позиции заказа с up=<id заказа>
5. Производственные резки можно не импортировать, их создаёт экран планирования
```

Для строк листа `Позиции` агент сначала находит заказ по `Номер заказа`, затем
создаёт дочернюю запись `Позиция заказа` с `up=<id найденного заказа>`.

Минимальный контроль после импорта:

```text
Заказ АТХ-0001 существует.
У заказа АТХ-0001 две подчинённые позиции.
Обе позиции ссылаются на один и тот же Вид сырья Jumbo Thermal 910/4000.
Заказ АТХ-0002 существует и имеет одну позицию.
```

## Роли, права и меню

Для первого прохода нужны четыре роли. Администратор остаётся системной ролью и
используется для настройки.

| Роль | Логин | Меню | Права |
|---|---|---|---|
| Менеджер | `manager` | `orders` | WRITE: `Заказ`, `Позиция заказа`; READ: `Клиент`, `Вид сырья` |
| Диспетчер | `dispatcher` | `planning` | WRITE: `Производственная резка`, `Позиция заказа`; READ: справочники и заказы |
| Оператор | `operator` | `operator` | WRITE: `Производственная резка`; READ: позиции и справочники |
| Руководитель | `director` | `dashboard` | READ: все таблицы мини-проекта |

Пункты меню создаются как дочерние записи роли в таблице `Меню` (`_m_new/151`
с `up=<roleId>`). Права выдаются через `Объекты` роли (`t117`) или через
подчинённую таблицу `Объекты` (116), как описано в
[`integram-app-workflow.md`](integram-app-workflow.md#56-права-роли-на-таблицы-и-запросы).

Для мини-проекта можно начать с прямого доступа к таблицам. Если появляется
клиентский портал или внешние пользователи, чтение нужно переводить на `report/`
и проверять маски доступа.

## Рабочие экраны

Файлы ниже — рекомендуемые имена для первой реализации. Они не обязаны совпадать
с полным Atex, но повторяют его логику в малом составе.

### `templates/mini-atex/orders.html`

Роль: менеджер.

Данные:

- читает `Клиент`, `Вид сырья`, `Заказ`, `Позиция заказа`;
- создаёт заказ через `_m_new/{Заказ}`;
- создаёт позицию через `_m_new/{Позиция заказа}` с `up=<id заказа>`;
- меняет статусы заказа и позиции через `_m_set/{recordId}`.

Экран:

- список заказов с фильтром по статусу;
- кнопка создания заказа;
- карточка заказа с табличной частью позиций;
- в позиции выбор клиента не нужен, клиент уже есть на заказе.

Критерий: менеджер создаёт `АТХ-0003`, добавляет хотя бы одну позицию и видит её
после обновления страницы.

### `templates/mini-atex/planning.html`

Роль: диспетчер.

Данные:

- читает позиции со статусом `Новая` или `Запланирована`;
- создаёт `Производственная резка` со ссылкой на позицию;
- меняет статус позиции на `Запланирована`.

Экран:

- список незапланированных позиций;
- форма планирования: позиция, дата, оператор;
- очередь резок по датам и статусам.

Критерий: диспетчер создаёт резку для позиции `ТТ-58 ECO 57x40`, назначает
оператора `operator` и дату `2026-06-01`.

### `templates/mini-atex/operator.html`

Роль: оператор.

Данные:

- читает резки, назначенные текущему оператору;
- меняет статус `Ожидает` -> `В работе` -> `Готово`;
- записывает `Кол-во факт`;
- при завершении меняет статус позиции заказа на `Готова`.

Экран:

- очередь моих резок;
- крупные кнопки статусов;
- поле фактического количества;
- история текущей резки в компактном виде.

Критерий: оператор завершает резку, указывает `Кол-во факт: 10`, позиция заказа
получает статус `Готова`.

### `templates/mini-atex/dashboard.html`

Роль: руководитель.

Данные:

- читает все таблицы мини-проекта;
- ничего не изменяет.

Экран:

- счётчики заказов по статусам;
- счётчики резок по статусам;
- список готовых позиций за период;
- таблица отклонений, где `Кол-во факт` меньше `Кол-во`.

Критерий: после завершения резки руководитель видит один готовый заказ, одну
готовую резку и позицию `ТТ-58 ECO 57x40`.

## Путь от начала до конца

Ниже полный проход, который должен выполнить агент или тестировщик после сборки.

### 1. Подготовить структуру

1. Создать пустую базу Интеграм и пользователя для агента с токеном.
2. Создать таблицы и связи из раздела [Схема данных](#схема-данных).
3. Проверить `metadata`: справочники идут через `ref`, позиции заказа — через
   `arr_id`.

### 2. Загрузить стартовые данные

1. Импортировать `Клиенты`.
2. Импортировать `Сырьё`.
3. Импортировать `Заказы`.
4. Импортировать `Позиции` как дочерние записи заказов.
5. Проверить, что `АТХ-0001` имеет две позиции.

### 3. Создать пользователей и меню

1. Создать роли `Менеджер`, `Диспетчер`, `Оператор`, `Руководитель`.
2. Создать пользователей `manager`, `dispatcher`, `operator`, `director`.
3. Добавить пункты меню `orders`, `planning`, `operator`, `dashboard`.
4. Выдать права из раздела [Роли, права и меню](#роли-права-и-меню).

### 4. Загрузить рабочие экраны

1. Создать файлы `templates/mini-atex/*.html`.
2. Вынести стили и JS в `/download/mini-atex/css/` и `/download/mini-atex/js/`,
   если экран становится больше простого прототипа.
3. Убедиться, что все запросы к API получают id таблиц и реквизитов из
   `metadata`, а не из хардкода.
4. Проверить открытие каждого пункта меню под нужной ролью.

### 5. Пройти бизнес-сценарий

1. Войти как `manager`.
2. Открыть `templates/mini-atex/orders.html`.
3. Создать заказ `АТХ-0003` для `ООО "Ромашка-Термолента"`.
4. Добавить позицию:

```text
Позиция заказа: ТТ-58 ECO 57x40
Вид сырья: Jumbo Thermal 910/4000
Ширина, мм: 57
Длина, м: 1200
Кол-во: 10
Статус: Новая
```

5. Войти как `dispatcher`.
6. Открыть `templates/mini-atex/planning.html`.
7. Создать резку:

```text
Позиция заказа: ТТ-58 ECO 57x40
Дата план: 2026-06-01
Оператор: operator
Статус: Ожидает
```

8. Войти как `operator`.
9. Открыть `templates/mini-atex/operator.html`.
10. Перевести резку `Ожидает` -> `В работе` -> `Готово`.
11. Записать `Кол-во факт: 10`.
12. Войти как `director`.
13. Открыть `templates/mini-atex/dashboard.html`.
14. Проверить, что заказ, позиция и резка отображаются как готовые.

## Критерии готовности

- Есть один документ, который можно прочитать сверху вниз без перехода в другие
  базы и понять весь путь.
- В документе есть исходные Excel-листы, схема данных, роли, меню, экраны,
  порядок загрузки данных и проверочный сценарий.
- Схема использует правильные типы связей: `Клиент` и `Вид сырья` как Ссылка,
  `Позиция заказа` как Подчинённая таблица заказа.
- Экраны работают с одной общей моделью данных.
- Пользовательский путь проходит от заказа в Excel до готовой резки на экране
  руководителя.
- Полный Atex остаётся следующим шагом: после мини-проекта агент может открыть
  `docs/atex_workplaces.md`, `docs/atex_full_cycle_test_scenario.md` и расширить
  схему до склада, калькулятора резки, втулкореза, портала клиента и дашбордов.

# Заявка клиента: Мини-Atex - резка рулонов

**Задача:** [#3137](https://github.com/ideav/crm/issues/3137)
**Откуда взяли пример:** исходная задача [#2986](https://github.com/ideav/crm/issues/2986),
мини-версия процесса Atex.

Пишу простыми словами, как мы обычно объясняем задачу подрядчику. У нас есть
небольшое производство, где из больших рулонов режут маленькие рулоны под
заказы клиентов. Сейчас всё ведём в Excel и переписках. Хотим сначала сделать
самую простую рабочую версию: принять заказ, запланировать резку, отметить
выполнение и посмотреть результат.

Полный Atex нам пока не нужен. Склад, сложный расчёт раскроя, втулкорез,
клиентский портал и большие дашборды можно оставить на следующий этап. На первом
этапе важно пройти один понятный путь от наших Excel-файлов до рабочих экранов.

## Как у нас сейчас

Клиенты присылают заказы на готовые рулоны. Менеджер заносит заказ в Excel:
номер заказа, клиента, дату, статус, свой логин и комментарий.

В заказе может быть несколько позиций. Например, один клиент заказал рулоны
57x40 и 40x30. Эти позиции лежат в отдельном листе Excel, а с заказом связаны по
номеру заказа.

Сырьё тоже ведём в Excel. Один и тот же большой рулон может использоваться в
разных позициях заказов, поэтому его не надо каждый раз заводить заново. То же
самое с клиентами: клиент должен выбираться из общего списка.

После приёма заказа диспетчер должен запланировать резку не всего заказа сразу,
а конкретной позиции заказа. Оператор потом видит свои резки, переводит их в
работу, завершает и указывает фактическое количество. Руководителю нужен простой
экран, где видно, что уже готово и где есть отклонения.

## Что хотим получить

Нужно приложение **«Мини-Atex: резка рулонов»**.

В нём должны быть четыре рабочих места:

| Кто работает | Что делает |
|---|---|
| Менеджер | Создаёт заказ, выбирает клиента, добавляет позиции заказа |
| Диспетчер | Видит новые позиции и планирует по ним производственные резки |
| Оператор | Видит назначенные ему резки, меняет статус и вводит факт |
| Руководитель | Смотрит статусы заказов, резок, готовые позиции и отклонения |

Главное требование: все экраны должны работать с одними и теми же данными.
Нельзя делать отдельные копии заказов или справочников под каждый экран.

## Файлы, которые мы дадим

На старте у нас есть два Excel-файла.

### `Справочники.xlsx`, лист `Клиенты`

```csv
Клиент,Телефон,Email
ООО "Ромашка-Термолента",+7 495 010-29-86,orders@romashka.example
ИП Петров,+7 812 010-29-86,petrov@example.test
```

### `Справочники.xlsx`, лист `Сырьё`

```csv
Вид сырья,Полное название,Ширина мм,Длина рулона м
Jumbo Thermal 910/4000,"Термобумага, ширина 910 мм, намотка 4000 м",910,4000
Jumbo Eco 760/3000,"Эко-бумага, ширина 760 мм, намотка 3000 м",760,3000
```

### `Заказы.xlsx`, лист `Заказы`

```csv
Номер заказа,Клиент,Дата создания,Статус,Менеджер,Комментарий
АТХ-0001,ООО "Ромашка-Термолента",2026-05-31,Новый,manager,Срочный тестовый заказ
АТХ-0002,ИП Петров,2026-05-31,Новый,manager,Обычная поставка
```

### `Заказы.xlsx`, лист `Позиции`

```csv
Номер заказа,Позиция,Вид сырья,Ширина мм,Длина м,Кол-во,Статус
АТХ-0001,ТТ-58 ECO 57x40,Jumbo Thermal 910/4000,57,1200,10,Новая
АТХ-0001,ТТ-40 ECO 40x30,Jumbo Thermal 910/4000,40,800,8,Новая
АТХ-0002,ТТ-76 ECO 76x60,Jumbo Eco 760/3000,76,1000,6,Новая
```

## Что должно получиться из файлов

Из файла `Справочники.xlsx` нужны общие списки:

- `Клиент`: название клиента, телефон, email;
- `Вид сырья`: короткое название сырья, полное название, ширина, длина рулона.

Из файла `Заказы.xlsx` нужны заказы и строки заказов:

- `Заказ`: номер, клиент, дата, статус, менеджер, комментарий;
- `Позиция заказа`: название позиции, сырьё, ширина, длина, количество, статус.

Позиции должны быть внутри своего заказа. Если в Excel у заказа `АТХ-0001` две
строки в листе `Позиции`, то в приложении у заказа `АТХ-0001` тоже должны быть
две позиции.

Клиенты и виды сырья должны быть общими списками для выбора. Если сырьё
`Jumbo Thermal 910/4000` встречается в двух позициях, это всё равно одна запись
сырья, которую выбрали два раза.

Для производства нужна ещё одна сущность:

- `Производственная резка`: какая позиция заказа режется, плановая дата,
  оператор, статус, фактическое количество и комментарий.

Производственная резка должна быть привязана к конкретной позиции заказа.
Например, диспетчер планирует резку по позиции `ТТ-58 ECO 57x40`, а не просто по
заказу `АТХ-0001` целиком.

## Как должна идти работа

Сначала исполнитель показывает нам короткое резюме будущего приложения и
уточняет то, чего нет в файлах:

- один оператор может выполнять любые резки или только назначенные ему;
- нужен ли клиентский доступ уже на первом этапе;
- есть ли логотип, сайт или фирменные цвета;
- какие статусы считаем финальными.

После подтверждения можно собирать приложение.

### 1. Загрузить справочники

Сначала загрузить клиентов и виды сырья. Они нужны до заказов, потому что заказы
и позиции должны выбирать значения из этих списков.

После загрузки должно быть видно:

```text
Клиент: ООО "Ромашка-Термолента"
Клиент: ИП Петров
Вид сырья: Jumbo Thermal 910/4000
Вид сырья: Jumbo Eco 760/3000
```

### 2. Загрузить заказы и позиции

Потом загрузить заказы. После этого загрузить позиции и разложить их по заказам
по колонке `Номер заказа`.

Проверка после загрузки:

```text
Заказ АТХ-0001 есть.
У заказа АТХ-0001 две позиции.
Обе позиции заказа АТХ-0001 используют сырьё Jumbo Thermal 910/4000.
Заказ АТХ-0002 есть.
У заказа АТХ-0002 одна позиция.
```

### 3. Сделать рабочее место менеджера

Файл экрана: `templates/mini-atex/orders.html`.

Менеджер должен видеть список заказов, фильтр по статусу, карточку заказа и
таблицу позиций внутри заказа.

Что нужно уметь делать:

- создать заказ;
- выбрать клиента из списка клиентов;
- добавить позицию в заказ;
- выбрать сырьё из списка сырья;
- изменить статус заказа или позиции.

Проверка: менеджер создаёт заказ `АТХ-0003`, добавляет в него позицию и видит её
после обновления страницы.

### 4. Сделать рабочее место диспетчера

Файл экрана: `templates/mini-atex/planning.html`.

Диспетчер должен видеть новые и запланированные позиции заказов. По выбранной
позиции он создаёт производственную резку: выбирает дату, оператора и начальный
статус.

Проверка: диспетчер создаёт резку для позиции `ТТ-58 ECO 57x40`, назначает
оператора `operator`, ставит дату `2026-06-01` и статус `Ожидает`.

После планирования позиция заказа должна стать `Запланирована`.

### 5. Сделать рабочее место оператора

Файл экрана: `templates/mini-atex/operator.html`.

Оператор должен видеть очередь своих резок. На экране нужны крупные и понятные
действия:

- начать резку;
- завершить резку;
- указать фактическое количество;
- оставить комментарий, если есть проблема.

Проверка: оператор переводит резку `Ожидает` -> `В работе` -> `Готово`, вводит
`Кол-во факт: 10`. После завершения позиция заказа становится `Готова`.

### 6. Сделать экран руководителя

Файл экрана: `templates/mini-atex/dashboard.html`.

Руководитель ничего не редактирует. Ему нужен обзор:

- сколько заказов в каждом статусе;
- сколько резок в каждом статусе;
- какие позиции готовы за выбранный период;
- где фактическое количество меньше заказанного.

Проверка: после завершения резки руководитель видит готовую резку и готовую
позицию `ТТ-58 ECO 57x40`. Если факт равен плану, отклонений по этой позиции нет.

## Кто будет пользоваться

На первом этапе нужны такие логины:

| Роль | Логин | Доступ |
|---|---|---|
| Менеджер | `manager` | Может создавать и менять заказы и позиции, читает клиентов и сырьё |
| Диспетчер | `dispatcher` | Может создавать резки и менять статус позиции, читает заказы и справочники |
| Оператор | `operator` | Может менять свои резки, читает позиции и справочники |
| Руководитель | `director` | Только смотрит все данные мини-проекта |

Меню должно вести каждого пользователя на свой экран:

```text
manager -> orders
dispatcher -> planning
operator -> operator
director -> dashboard
```

## Полный проверочный сценарий

Этот сценарий нужно пройти после сборки.

1. Войти как `manager`.
2. Открыть `templates/mini-atex/orders.html`.
3. Создать заказ `АТХ-0003` для `ООО "Ромашка-Термолента"`.
4. Добавить позицию:

```text
Позиция заказа: ТТ-58 ECO 57x40
Вид сырья: Jumbo Thermal 910/4000
Ширина, мм: 57
Длина, м: 1200
Кол-во: 10
Статус: Новая
```

5. Войти как `dispatcher`.
6. Открыть `templates/mini-atex/planning.html`.
7. Создать производственную резку:

```text
Позиция заказа: ТТ-58 ECO 57x40
Дата план: 2026-06-01
Оператор: operator
Статус: Ожидает
```

8. Войти как `operator`.
9. Открыть `templates/mini-atex/operator.html`.
10. Перевести резку `Ожидает` -> `В работе` -> `Готово`.
11. Записать `Кол-во факт: 10`.
12. Войти как `director`.
13. Открыть `templates/mini-atex/dashboard.html`.
14. Проверить, что заказ, позиция и резка отображаются как готовые.

## Что считаем готовым

Мини-проект можно принимать, если:

- документ с нашей задачей можно прочитать сверху вниз без знания внутреннего
  устройства Интеграма;
- из Excel загрузились клиенты, сырьё, заказы и позиции;
- позиции заказа не потеряли связь со своим заказом;
- клиенты и сырьё выбираются из общих списков, а не дублируются в каждой строке;
- производственная резка создаётся по конкретной позиции заказа;
- менеджер, диспетчер, оператор и руководитель проходят свой путь на отдельных
  экранах;
- все экраны показывают одни и те же данные;
- проверочный сценарий проходит от заказа `АТХ-0003` до готовой резки.

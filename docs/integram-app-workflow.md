# Полный процесс создания Интеграм-приложения

## 0. Что такое Интеграм-приложение

Интеграм — low-code платформа (ideav.ru). Приложение = база данных (workspace) + таблицы + HTML/JS шаблоны для UI. Данные хранятся в таблицах, интерфейс — кастомные HTML-страницы с JS, которые загружаются в файловое хранилище базы.

Регистрация вашего экземпляра доступна [здесь](https://ideav.ru/start.html#reg).

Данный документ описывает все шаги создания полноценного и самодостаточного приложения, начиная с пустой базы Интеграм:
1. Уточнение требований
2. Создание структуры данных и наполнение её тестовыми данными
3. Создание пользователй, их ролей и пунктов меню
4. Создание запросов на выборку и изменение данных (для защиты слоя данных)
5. Создание рабочих мест, работающих с базой по API
6. Написание и прогон тест-плана для полного пользовательского пути

**Ключевые понятия:**
- **База данных (workspace)** — изолированное пространство с таблицами, файлами, пользователями
- **Таблица (type)** — аналог таблицы SQL, имеет ID, колонки, связи
- **Колонка (requisite/req)** — поле таблицы, имеет ID и алиас (русское имя)
- **Ссылка (ref)** — связь-указатель на одну запись справочника (foreign key); см. [2.5](#25-два-типа-связей-ссылка-справочник-и-подчинённая-таблица)
- **Подчинённая таблица** — набор дочерних записей, принадлежащих одному родителю (особый тип связи Интеграма); см. [2.5](#25-два-типа-связей-ссылка-справочник-и-подчинённая-таблица)
- **Запись (object)** — строка таблицы, имеет ID
- **Запрос (report/query)** — команда Интеграму на выборку и обработку данных
- **Шаблон (template)** — HTML-страница, лежит в репозитории в `templates/{имя базы}/` (если папки нет — создать), рендерится Интеграм

## Этап 0: Подключение к базе данных

Прежде чем начать разработку, агент должен получить доступ к базе Интеграм. Когда агент готов приступить к работе, он задаёт вопрос:

> **"Создай базу данных в Интеграм и пользователя `claude` в ней. Сообщи адрес базы и пароль пользователя."**

Пользователь создаёт базу на [ideav.ru](https://ideav.ru), затем:
1. Заходит в базу → Пользователи
2. Создаёт пользователя с логином `claude` и произвольным паролем
3. Сообщает агенту: адрес базы (например `ideav.ru/myapp/`) и пароль

Агент выполняет авторизацию:
```bash
curl -c cookies.txt -X POST 'https://ideav.ru/{db}/auth?JSON=1' \
  -F 'login=claude' -F 'pwd={пароль}'
# Ответ: {"token":"<hex>","_xsrf":"<hex>","id":"<userId>","msg":""}
```

**Что происходит после авторизации:**

1. Сервер возвращает в теле ответа `token`, `_xsrf` и `id` пользователя.
2. Сервер **автоматически устанавливает cookie `idb_{db}=<token>`** (TTL ~6 месяцев).
   Если запрос сделан с `-c cookies.txt`, токен лежит в cookie-jar — повторно
   проходить авторизацию не нужно.

**Передача сессии в последующих запросах — два пути:**

- **Через cookie** (предпочтительно для GET-чтений): добавлять `-b cookies.txt`
  к curl-вызовам, сервер сам распознаёт сессию по cookie `idb_{db}`.
- **Через форму** (нужно для POST на изменение данных): передавать
  `token=<token>` и `_xsrf=<xsrf>` как form-поля — так делает `api_call` в
  эталонных скриптах ниже. **`_xsrf` обязателен** для POST `_m_*`/`_d_*`,
  даже если сессия установлена через cookie.

После получения токена агент самостоятельно работает с API базы.

## Этап 1: Уточнение требований

**Главный принцип: если что-то непонятно — спроси, не угадывай.** Любое предположение о том, как должен работать процесс или как называться поле, лучше уточнить у заказчика. Угадывание приводит к ошибкам в схеме, которые дорого исправлять на следующих этапах.

### 1.1 Вопросы, которые нужно задать

Перед началом проектирования задай эти вопросы (по одному):

1. **"Опиши бизнес-процесс: кто что делает и в каком порядке?"**
   - Понять роли (менеджер, оператор, руководитель)
   - Понять этапы процесса (заказ → планирование → исполнение → контроль)

2. **"Какие экраны/панели нужны? Кто на каждом работает?"**
   - Перечислить все интерфейсы
   - Привязать каждый к роли

3. **"Какие данные вводятся на каждом экране? Какие справочники нужны?"**
   - Выявить все сущности (таблицы)
   - Выявить все поля

4. **"Как связаны сущности между собой?"**
   - Для каждой связи определить её тип по правилам [2.5](#25-два-типа-связей-ссылка-справочник-и-подчинённая-таблица):
   - Ссылки (ref) — указатели на запись справочника (переиспользуемую многими)
   - Родительские/дочерние (подчинённые) таблицы — наборы записей, принадлежащих одному родителю

### 1.2 Результат этапа

На выходе должны быть:
- Список ролей
- Список экранов с привязкой к ролям
- Список таблиц с полями и связями (черновик схемы)

## Этап 2: Проектирование схемы данных

### 2.1 Правила проектирования таблиц (из StructureRules.md)

1. **Первая колонка любой таблицы — собственный идентификатор объекта** (обычно тип SHORT=3, но может быть число, дата или иной тип)
   - Это не ссылка! Это то, ЧТО такое эта запись
   - Примеры: название сырья, номер заказа, имя слиттера

2. **Имена таблиц — единственное число, на русском**
   - Правильно: "Сырьё", "Заказ", "Слиттер"
   - Неправильно: "Сырьё и материалы", "Orders"

3. **Колонки-ссылки (ref) создаются в таблице-источнике**
   - Если "Тип резки" ссылается на "Сырьё" → ref создаётся в "Тип резки"
   - Не создавай колонку в справочнике для обратной связи (это делает сама платформа)
   - Можно делать ссылку на записи в подчиненной таблице, например, на запись об обеспечении какой-то позиции заказа (позиции подчинены заказу)

4. **Первая колонка таблицы является именем объекта** — отдельное поле "Название", "Имя", "ФИО" не нужно, это дублирование

5. **Можно сделать составной ключ уникальности** — при импорте и создании записей он будет соблюдаться, и он может не включать первую колонку

Файл со схемой всех таблиц, колонок и связей. Это станет основой для скрипта создания.

### 2.5 Два типа связей: ссылка (справочник) и подчинённая таблица

В Интеграме между таблицами бывает **ровно два** типа связей — **ссылка** и
**подчинённая таблица**. Перепутать их — самая частая и самая дорогая ошибка
проектирования: схему приходится переделывать, а рабочие места не подключаются
по API (живой разбор такой ошибки — в конце раздела). **Прежде чем добавить любую
связь, определи её тип по правилам ниже.**

Для человека это различие объяснено в блоге Интеграма —
[«Как отличить подчинённую таблицу от справочника»](https://blog.ideav.ru/posts/kak-otlichit-podchinennuyu-tablicu-ot-spravochnika/).
Ниже то же самое формализовано для LLM, чтобы не путаться.

#### 2.5.1 Определения

- **Ссылка (она же «справочник», FK).** Поле, которое указывает на **одну уже
  существующую запись** другой таблицы. Целевая запись существует независимо и
  **переиспользуется многими** записями-источниками. Это прямой аналог foreign
  key реляционной БД. Пример: у «Заказа» поле «Клиент» указывает на запись
  справочника «Клиент»; один и тот же клиент привязан ко множеству заказов. Также
  ссылка — это «Статус», «Вид сырья», «Станок», «Ответственный».

- **Подчинённая таблица (дочерняя, «табличная часть»).** **Набор записей,
  которые целиком принадлежат одной родительской записи** и создаются специально
  под неё. Дочерние записи не переиспользуются и не имеют смысла вне родителя.
  Пример: «Позиции заказа» — каждая позиция принадлежит ровно одному заказу;
  «контакты сотрудника»; «Объекты» и «пункты Меню» у Роли. Это особый тип связи
  Интеграма: вместо отдельной FK-колонки на родителя платформа даёт наглядную
  навигацию — у родителя сразу видно количество и список дочерних записей.

#### 2.5.2 Правило выбора (чек-лист)

Задай по очереди четыре вопроса. Если хотя бы один уверенно указывает на
«справочник» — это **ссылка**. Подчинённую таблицу выбирают, только когда записи
действительно принадлежат одному родителю и нигде больше не используются.

1. **Тест переиспользования.** Может ли одна и та же связанная запись быть
   привязана к нескольким разным родителям?
   - Да (один статус «Новый» — у сотен заказов; один клиент — у многих заказов) →
     **ссылка**.
   - Нет (эта строка существует только ради одного родителя) → **подчинённая
     таблица**.

2. **Тест жизненного цикла.** Существует ли связанная запись сама по себе,
   независимо от родителя, и ведётся ли она как отдельный справочник?
   - Да (статусы, клиенты, виды сырья, станки заполняются заранее и отдельно) →
     **ссылка**.
   - Нет (запись рождается вместе с родителем и удаляется вместе с ним) →
     **подчинённая таблица**.

3. **Тест ввода в UI.** Как пользователь заполняет связь?
   - Выбирает готовое значение из выпадающего списка → **ссылка**.
   - Добавляет новые строки в табличную часть карточки родителя → **подчинённая
     таблица**.

4. **Тест направления и владения.** «Многие источники → один справочник» (N:1),
   FK лежит в источнике → **ссылка**. «Один родитель → много детей» (1:N), родитель
   «владеет» детьми → **подчинённая таблица**.

#### 2.5.3 Сравнение

| Критерий | Ссылка (справочник) | Подчинённая таблица |
|----------|---------------------|---------------------|
| Что хранит | указатель на одну запись другой таблицы | набор собственных записей родителя |
| Переиспользование цели | да, многими источниками | нет, запись принадлежит одному родителю |
| Кардинальность | N:1 (много источников → один справочник) | 1:N (один родитель → много детей) |
| Где «лежит» связь | колонка-FK в таблице-источнике | дочерний тип добавлен реквизитом родителя; запись создаётся с `up=<родитель>` |
| Аналог в SQL | foreign key | составная часть / детейл-таблица |
| Ввод в UI | выбор из выпадающего списка | добавление строк в табличную часть |
| Примеры | Заказ→Клиент, →Статус; Тип резки→Вид сырья; Резка→Станок | Заказ→Позиции; Роль→Объекты, →Меню; Сотрудник→Контакты |

#### 2.5.4 Как тип связи виден в метаданных (самопроверка)

После создания связи перечитай `GET /{db}/metadata?JSON=1` и посмотри на
реквизит-связь в `reqs[]` родительской таблицы:

- **Ссылка** — у реквизита есть поля **`ref`** (= id целевой таблицы) и `ref_id`.
  Базовый `type` = 3 (показывается значение целевой записи). Целевая таблица в
  своём объекте имеет поле `referenced`. Пример из базы `atex` (`Тип резки`):
  ```json
  {"id":"597","val":"Сырьё","orig":"469","type":"3","ref":"469","ref_id":"543"}
  ```
- **Подчинённая таблица** — у реквизита есть поле **`arr_id`** (= id дочерней
  таблицы) и **нет `ref`**. Пример из `atex` (`Заказ`):
  ```json
  {"id":"1177","val":"Позиция заказа","orig":"507","type":"3","arr_id":"507"}
  ```

> **Главный признак для самопроверки:** `ref`/`ref_id` → ссылка; `arr_id` без
> `ref` → подчинённая таблица. Если связь, которая **по смыслу должна быть
> справочником**, в метаданных имеет `arr_id` вместо `ref` — типы перепутаны,
> структуру надо переделать.

В Редакторе структуры (визуально, для человека) то же различие видно так:
справочник отображается с **именем целевой таблицы в скобках**, подчинённая
таблица — **без скобок**.

#### 2.5.5 Как создаётся каждый тип (API)

- **Ссылку** создают двухшаговым паттерном `_d_ref` + `_d_req` — см.
  [3.2 шаг 6](#32-последовательность-api-вызовов). Результат: реквизит с `ref`.
- **Подчинённую таблицу** создают, добавляя **тип дочерней таблицы** реквизитом
  родителя — `_d_req/{parentTableId} t={childTableId}`, а записи создают с
  `up=<parentRecordId>` — см. [3.4](#34-подчинённые-таблицы). Результат: реквизит
  с `arr_id`.

> **Практический вывод.** Тип связи задаётся тем, что подставлено в `t` при
> `_d_req`: ref-тип (полученный из `_d_ref`) даёт **ссылку** с полем `ref`; id
> дочерней таблицы даёт **подчинённую таблицу** с полем `arr_id`. Поэтому ссылку
> создавай двухшаговым `_d_ref` + `_d_req` (см. [3.2 шаг 6](#32-последовательность-api-вызовов)),
> а подчинённую таблицу — `_d_req t={childTableId}` с последующим `up=` (см.
> [3.4](#34-подчинённые-таблицы)).

#### 2.5.6 Разбор реальных баз

Две базы прошли путь `integram-app-workflow.md` с разным результатом — это
эталонные «как не надо» и «почти правильно».

- **`ateh` — антипример (типы перепутаны).** Почти все справочные связи ошибочно
  созданы как подчинённые таблицы: у `Заказ.Клиент`, `Позиция заказа.Вид сырья`,
  `Позиция заказа.Тип резки`, `Производственная резка.Слиттер / Тип резки /
  Партия сырья` в метаданных стоит `arr_id`, а не `ref`. Следствие: каждый заказ
  «владеет» собственными клиентами вместо ссылки на общий справочник «Клиент»,
  справочники не переиспользуются, рабочие места невозможно подключить по API. По
  смыслу это **ссылки**. Подчинёнными в `ateh` корректно остаются только
  `Заказ.Позиция заказа`, `Позиция заказа.Обеспечение`, `Тип резки.Полоса`.

- **`atex` — почти правильно.** Справочники сделаны ссылками
  (`Тип резки.Сырьё` → `ref=469`), позиции — подчинёнными
  (`Заказ.Позиция заказа` → `arr_id=507`). Единственная ошибка связи:
  `Производственная резка` ссылается на **`Заказ`** (498), а должна ссылаться на
  **`Позиция заказа`** (507) — резка выполняется под конкретную позицию заказа, а
  не под заказ целиком.

## Этап 3: Создание таблиц (скрипт)

### 3.1 Структура скрипта

Скрипт на bash, использующий curl для вызовов API Интеграм. Паттерн: [`create_perelidoz.ps1`](https://github.com/ideav/crm/blob/main/docs/create_perelidoz.ps1).

**Базовый шаблон функции `api_call`:**

```bash
api_call() {
    local endpoint="$1"
    local method="${2:-POST}"
    shift 2
    local url="${BASE_URL}/${DB}/${endpoint}?JSON=1"
    # ... добавляет token и _xsrf ко всем запросам
    curl -s -k -X POST "$url" -F "token=$TOKEN" -F "_xsrf=$XSRF" "$@"
}
```

**Критически важно:** функция `log()` должна писать в stderr (`>&2`), иначе вызовы `api_call` внутри `$()` будут захватывать логи вместе с JSON-ответом.

### 3.2 Последовательность API-вызовов

1. **Авторизация:**
   ```bash
   POST /{db}/auth?JSON=1
   Fields: login, pwd
   Returns: { token, _xsrf, id }
   ```

2. **Создание таблицы:**
   ```bash
   POST /{db}/_d_new?JSON=1
   Fields: t=3, val="Имя таблицы"
   Returns: { obj: tableId }
   ```
   Параметр `t` задаёт **тип первой колонки** таблицы (см. 2.2). Эта первая
   колонка автоматически становится «именем записи» (`_value`) и **доступна в
   API записей по ключу `t<tableId>`** — например, для таблицы id=470 при
   `_m_new`/`_m_set` название записи пишется как `t470=<значение>`. То же
   справедливо для всех таблиц (в т.ч. системных: t18=login для Пользователя,
   t42=name для Роли, t151=name для Меню).

   **Автоинкремент** (только для `t=13` NUMBER): добавь параметр `unique=1`:
   ```bash
   POST /{db}/_d_new?JSON=1
   Fields: t=13, val="Номер резки", unique=1
   Returns: { obj: tableId }
   ```
   Первая колонка получит свойство `unique` и при `_m_new` сама посчитает
   следующий целочисленный номер. Полезно для журнальных таблиц с «Номер 1, 2,
   3…».

3. **Создание типа колонки:**
   ```bash
   POST /{db}/_d_new?JSON=1
   Fields: t=<typeCode>, val="Тип колонки"
   Returns: { obj: typeId }
   ```

4. **Добавление колонки в таблицу:**
   ```bash
   POST /{db}/_d_req/{tableId}?JSON=1
   Fields: t=<typeId>
   Returns: { id: reqId }
   ```

5. **Установка отображаемого имени колонки (alias реквизита):**
   ```bash
   POST /{db}/_d_alias/{reqId}?JSON=1
   Fields: val="Имя колонки"
   ```
   Эта команда пишет имя в `attrs.alias` записи-реквизита. Используется при
   создании новой колонки в таблице, чтобы дать ей отличное от типа имя
   (например, тип называется «Дата», а в этой таблице колонка должна
   называться «Дата прихода»). Тот же `_d_alias` ставит псевдоним для
   ref-колонок (ссылка на «Клиент» в карточке заказа может показываться как
   «Заказчик»).

   > **Не вызывай `_d_alias` на ID таблицы (типа SHORT/DATETIME).** Сервер
   > запишет в `val` таблицы JSON-блоб вида `{"alias":"...","default":"..."}`,
   > и название таблицы во всём UI начнёт показываться как этот блоб. Для
   > переименования самой таблицы или существующего типа колонки используй
   > `_d_save` (см. 3.6).

6. **Создание ссылки (ref) — двухшаговый паттерн.**

   > Используй этот паттерн, только если связь по правилам
   > [2.5](#25-два-типа-связей-ссылка-справочник-и-подчинённая-таблица) — это
   > **ссылка** (указатель на переиспользуемую запись справочника). Если связь —
   > **подчинённая таблица** (набор записей, принадлежащих одному родителю),
   > используй паттерн из [3.4](#34-подчинённые-таблицы), а не этот.

   Шаг 6.1: создать ref-тип, указывающий на целевую таблицу:
   ```bash
   POST /{db}/_d_ref/{targetTableId}?JSON=1
   Fields: token=<token>&_xsrf=<xsrf>
   Returns: { id: <targetTableId>, obj: <refTypeId> }
   ```

   Шаг 6.2: добавить ref-тип как реквизит в таблицу-источник:
   ```bash
   POST /{db}/_d_req/{sourceTableId}?JSON=1
   Fields: token=<token>&_xsrf=<xsrf>&t=<refTypeId>
   Returns: { id: <reqId>, obj: <sourceTableId> }
   ```

   В записях (`_m_set`/`_m_new`) ref-поле адресуется по ключу **`t<reqId>`**, где
   `reqId` — это `id` из ответа шага 6.2, а значение — id записи в целевой таблице.
   В metadata реквизит выходит под тем же ключом `t<reqId>` с полем `ref` (= id
   целевой таблицы). Двухшаговый путь гарантирует, что возвращённый `id` и ключ
   записи совпадают.

   В параметр `t` подставляй **ref-тип** из шага 6.1, а не id целевой таблицы.
   От содержимого `t` зависит тип связи:

   | В `t` подставлено | Что создаётся | Ключ записи / metadata |
   |-------------------|---------------|------------------------|
   | ref-тип (после `_d_ref`) | **ссылка** (FK) | `t<reqId>`, поле `ref` |
   | id дочерней таблицы | **подчинённая таблица** | `t<childTableId>`, поле `arr_id` |

   Форму `_d_req/{source} t=<childTableId>` (id таблицы прямо в `t`) применяй,
   когда по правилам [2.5](#25-два-типа-связей-ссылка-справочник-и-подчинённая-таблица)
   нужна **подчинённая таблица**; полный паттерн с `up=` — в [3.4](#34-подчинённые-таблицы).
   В этой форме реквизит адресуется ключом `t<childTableId>`, а не `t<id-из-ответа>`:
   проверяй ключ записи по `arr_id` в metadata.

### 3.2.1 Идемпотентность `_d_new` и `_d_req`

**`_d_new` дедуплицирует имена типов глобально по базе.** Повторный вызов
`POST _d_new t=3 val=Статус` возвращает существующий `obj` с предупреждением
`"Тип Статус уже существует!"` — никакого дубликата не создаётся. Это безопасно
перевызывать; для проверки «есть ли такой тип» можно использовать как зонд:

```bash
# Сессия 2026-05-29 на /ateh/, попытка создать тип "Статус", который уже есть:
api_call _d_new "t=3" "val=Статус"
# Ответ: {"id":"","obj":"409","warnings":"Тип Статус уже существует!"}
# → глобальный type id для "Статус" — 409, переиспользуем.
```

**`_d_req` идемпотентен per (table, type).** Если тип уже привязан к таблице,
повторный вызов вернёт **существующий reqId** с предупреждением
`"Реквизит ... уже существует!"`. Никакого нового реквизита не создаётся.

**Побочное применение — расшифровка чужой схемы с безымянными колонками.**
Если у таблицы есть реквизиты без `attrs.alias`, и нужно понять, какой из них
какому полю соответствует, можно «зондировать» через `_d_req`:

```bash
# Пример: таблица "Клиент" 415 на /ateh/ имеет 5 безымянных реквизитов
# (t417 SHORT, t418 SHORT, t419 MEMO, t421 SHORT, t424 SHORT MULTI).
# Хочу выяснить, есть ли там Email — тип 31 в этой БД:
api_call _d_req/415 "t=31"
# Ответ: {"id":"418","obj":415,"warnings":"Реквизит 31 уже существует!"}
# → значит t418 на Клиенте 415 — это и есть Email.
```

То же сработает для любого другого типа, имя которого известно (Телефон, Дата
и т.д.). Этот трюк экономит время при работе с legacy-БД, в которой колонки
потеряли алиасы.

### 3.3 Порядок создания таблиц

**Важно:** таблицы создаются в порядке зависимостей:
1. Сначала независимые справочники (Сырьё, Слиттер)
2. Потом зависимые справочники (Тип резки — ссылается на Сырьё)
3. Основные таблицы (Заказ)
4. Дочерние таблицы (Позиция заказа)
5. Таблицы событий (Производственная резка — ссылается на всё)

Ссылки (ref) добавляются после создания ВСЕХ таблиц.

### 3.4 Подчинённые таблицы

> Применяй этот паттерн, только когда связь по правилам
> [2.5](#25-два-типа-связей-ссылка-справочник-и-подчинённая-таблица) — это
> **подчинённая таблица** (набор записей, принадлежащих одному родителю). Если
> связь — **ссылка** на переиспользуемый справочник, используй двухшаговый
> `_d_ref` + `_d_req` из [3.2 шаг 6](#32-последовательность-api-вызовов).

Дочерняя таблица становится подчинённой только после того, как её тип добавлен
реквизитом в родительскую таблицу. Одного `up=<parentRecordId>` при создании
записи недостаточно: без родительского реквизита таблица остаётся отдельной, а
создание дочерней записи может завершиться ошибкой «нет доступа к реквизиту».

Минимальная последовательность:

```bash
# 1. Создать родительскую таблицу
orderTableId=$(api_call "_d_new" "t=3&val=$(urlencode 'Заказ')" | jq -r .obj)

# 2. Создать дочерний тип
positionTableId=$(api_call "_d_new" "t=3&val=$(urlencode 'Позиция заказа')" | jq -r .obj)

# 3. Обязательно добавить дочерний тип как реквизит родителя
positionReqId=$(api_call "_d_req/${orderTableId}" "t=${positionTableId}" | jq -r .id)
api_call "_d_alias/${positionReqId}" "val=$(urlencode 'Позиция заказа')"

# 4. Добавить поля дочерней таблицы
qtyTypeId=$(api_call "_d_new" "t=13&val=$(urlencode 'Количество рулонов')" | jq -r .obj)
qtyReqId=$(api_call "_d_req/${positionTableId}" "t=${qtyTypeId}" | jq -r .id)
api_call "_d_alias/${qtyReqId}" "val=$(urlencode 'Количество рулонов')"

# 5. Создавать записи дочерней таблицы с up=id родительской записи
api_call "_m_new/${positionTableId}?full=1" "up=${orderRecordId}&t${positionTableId}=1&t${qtyReqId}=351"
```

Проверка через `metadata`: у родительской таблицы в `reqs[]` должен быть реквизит,
у которого `orig` равен ID дочернего типа, **и присутствует поле `arr_id`** (= ID
дочернего типа), а поля `ref` нет. Наличие `arr_id` и отсутствие `ref` — признак
именно подчинённой таблицы (а не ссылки; см.
[2.5.4](#254-как-тип-связи-виден-в-метаданных-самопроверка)). Если дочерний тип уже
создан, но не подключён к родителю, ремонт структуры — это `_d_req/{parentTableId}`
с `t={childTableId}`, а не изменение прав роли в `t117`.

### 3.5 Чтение существующих записей и метаданных

После создания структуры (и в любой момент позже) данные читаются GET-запросами;
авторизация — через cookie `idb_{db}` (см. Этап 0) или через `?token=...`.

**Метаданные базы — список всех таблиц и их колонок:**
```bash
GET /{db}/metadata?JSON=1
# Возвращает массив объектов: [{id, val, granted, reqs:[{id, val, orig, type, ref, attrs}, ...]}, ...]
# Здесь val у каждого reqs[i] — имя типа колонки, orig — id типа (нужен для _d_save), attrs может содержать {"alias":"..."}.
```

**Список записей конкретной таблицы:**
```bash
GET /{db}/object/{typeId}/?JSON_KV&LIMIT=100&pg=1
# Записи лежат в ответе под ключом "object" (массив объектов).
# LIMIT — размер страницы (по умолчанию небольшой), pg — номер страницы (1-based).
# Между страницами возможны дубликаты — дедупликация по id рекомендуется при пагинации.
```

**Полезные фильтры/параметры:**

| Параметр | Назначение |
|----------|-----------|
| `F_U={parentId}` | только записи с `up=<parentId>` (например, меню конкретной роли: `object/151/?F_U=145`) |
| `F_I={recordId}` | только запись с `vals.id=<recordId>` (например, карточка записи: `object/{typeId}/?F_I=123`) |
| `_count=`        | вернуть только число записей без выгрузки данных: `?JSON_KV&_count=` |
| `LIMIT=N`        | страница до N записей |
| `pg=N`           | номер страницы (1-based) |

Тип таблицы выбирается сегментом пути `object/{typeId}`. Отдельный параметр
`F_T` для выбора типа в `object/` не нужен: в коде фильтры `F_*` разбираются как
фильтры по колонкам или специальные параметры `F_U`/`F_I`.

**Список доступа (используется в редакторе ролей):**
```bash
GET /{db}/grants?JSON=1
# Возвращает массив {id, name} — отображаемые названия всех таблиц и колонок,
# которые можно раздать роли. Полезно для самопроверки переименований:
# если name содержит JSON-блоб {"alias":...} — где-то сработал _d_alias на таблице.
```

> Перечня всех отчётов (`report/`) через API нет — он не возвращается из
> `metadata` и не индексируется в `grants`. Сохраняй ID отчётов в самом скрипте
> сборки или забирай их из заголовка `X-Query-Id` при первом вызове по имени
> (см. раздел «Запросы (report/) — справочник»).

### 3.6 Переименование существующих типов и таблиц

`_d_alias` ставит **alias на конкретный реквизит** (per-occurrence) — это
полезно при создании новой колонки или для псевдонимов ref-колонок. Чтобы
переименовать **сам тип** (то есть имя, которое отображается везде, где этот
тип используется), нужен `_d_save`.

**Переименование таблицы:**
```bash
POST /{db}/_d_save/{tableId}?JSON
Fields: val="Новое имя", t=3
# t=3 — базовый код типа SHORT (для таблиц с первой колонкой SHORT)
# Для таблиц с первой колонкой DATETIME — t=4, и т.д.
# Меняет val самой таблицы; во всех ref-колонках и в metadata имя обновится.
```

**Переименование типа колонки** (например, «Статус слиттера» → «Статус станка»):
```bash
POST /{db}/_d_save/{origId}?JSON
Fields: val="Новое имя", t=<baseTypeCode>
# ВАЖНО: origId — это reqs[i].orig из metadata (id типа), НЕ reqs[i].id (id реквизита-в-таблице).
# baseTypeCode — это reqs[i].type (3 для SHORT, 12 для MEMO, 13 для NUMBER и т.д.),
# а не сам origId.
# Если вызвать с reqs[i].id вместо .orig — сервер вернёт {"error":"Тип не найден."}.
```

**Когда брать `_d_alias`, а когда `_d_save`:**

| Случай | Команда |
|--------|---------|
| Новая колонка только что добавлена через `_d_req`, хочу дать ей имя | `_d_alias/{reqId}` (как в `create_perelidoz.ps1`) |
| Уже созданной ref-колонке нужен псевдоним в этой конкретной таблице | `_d_alias/{reqId}` |
| Переименовать существующую обычную колонку (затрагивает все её использования) | `_d_save/{origId}` с `val=…&t=baseType` |
| Переименовать таблицу | `_d_save/{tableId}` с `val=…&t=3` (или иной базовый код) |

### 3.7 Результат этапа

Файл `create_tables.sh`, который:
- Можно запустить повторно (идемпотентный — проверяет ошибки)
- Пишет лог в `create_tables.log`
- Выводит сводку всех созданных ID в конце

## Этап 4: Разработка шаблонов (HTML/JS)

### 4.1 Структура шаблона

Каждый шаблон = HTML-файл в `templates/` со встроенными стилями и подключением JS:

```html
<style>
/* Локальные стили */
.my-panel { ... }
</style>

<div class="my-panel">
    <!-- HTML интерфейса -->
</div>

<script>
window.SOME_TYPE_ID = 469;  // ID таблицы

<script src="/download/{_global_.z}/tables.js?1"></script>
<script src="/download/{_global_.z}/my-panel.js?1"></script>
```

**Важно:** `{_global_.z}` — шаблонная переменная Интеграм, заменяется на имя базы данных при рендеринге.

### 4.2 JS-файлы

**api.js** — обёртка над API Интеграм:
```javascript
// createRecord(tableId, fields, parentId)
Api.createRecord(469, { 'Тип сырья': 'Бумага', 'Ширина джамбо, мм': 1020 });

// queryTable(typeId, where, sort, limit)
Api.queryTable(469, { 'Статус': 'Новый' }, '-id', 20);

// updateRecord(objId, fields)
Api.updateRecord(571, { 'Остаток, м': 4500 });
```

**tables.js** — общие функции для работы с таблицами (переиспользуются всеми панелями).

**Панельные JS** — логика конкретного экрана (order-entry.js, cutter-panel.js и т.д.).

### 4.3 Загрузка шаблонов на сервер

Шаблоны загружаются в папку `download` базы данных через веб-интерфейс или API:

```bash
POST /{db}/dir_admin
Fields:
  download=1
  add_path=
  upload=Загрузить
  userfile=@file.js
```

После загрузки файлы доступны по URL: `https://ideav.ru/download/{db}/file.js`

### 4.4 Результат этапа

- Папка `js/` с `api.js`, `tables.js` и панельными JS-файлами
- Папка `templates/` с HTML-шаблонами для каждого экрана
- Все файлы загружены и проверены через `curl https://ideav.ru/download/atex/имя_файла`

## Этап 5: Роли, права, пользователи и меню

### 5.1 Системные таблицы

Интеграм автоматически создаёт три системные таблицы:
- **Роль** (таблица 42, type=3) — название роли, описание, права
- **Пользователь** (таблица 18, type=3) — логин, роль (ref на Роль), пароль, email, телефон
- **Меню** (таблица 151, type=3) — пункты меню ролей (адрес, иконка, параметры)

**ВАЖНО:** Для системных таблиц (type=3) ВСЕ колонки передаются через префикс `t` + ID колонки.

> **Меню связано с ролью через `up`, а не через ref-колонку.** У записи Роли
> НЕТ поля «ref→Меню» — пункты меню создаются как **дочерние записи Роли**
> через `_m_new/151` с `up=<role_id>`. Чтобы получить пункты меню конкретной
> роли, используется `GET /{db}/object/151/?F_U=<role_id>` (фильтр по
> родителю). Подробнее в 5.4 ниже.
>
> Колонки Роли (таблица 42): `t117` (GRANT «Объекты», права на объекты),
> `t135` (MEMO, описание) и колонка Меню. Реквизит «Объекты» в метаданных Роли
> выглядит как `{"id":"117","val":"Объекты","orig":"116"}`: `117` — id колонки,
> `orig` `116` — id подчинённой таблицы «Объекты», в которой хранятся права (по
> одной записи на объект базы, выбираются по `up=<role_id>`). Актуальные id
> сверяются через `GET /{db}/metadata?JSON=1` → `reqs[]` таблицы 42 (поля `id`
> и `orig`).

### 5.2 Паттерн `t{column_id}=value`

Все значения передаются как `t{id_колонки}=value` — это единственный формат для всех таблиц (системных и обычных):

```bash
# Роль (таблица 42): t42=название, t135=описание
POST /{db}/_m_new/42?JSON=1
Fields: up=1, t42=менеджер, t135=Управление заказами и планирование

# Пользователь (таблица 18): t18=логин, t115=role_id, t20=пароль
POST /{db}/_m_new/18?JSON=1
Fields: up=1, t18=manager, t115=632, t20=slitter123

# Меню (таблица 151): t151=название, t153=адрес, t391=иконка
POST /{db}/_m_new/151?JSON=1
Fields: up=<role_id>, t151=Приём заказов, t153=order-entry
Icon: --form-string 't391=<i class="pi pi-file-edit"></i>'
```

### 5.6 Права роли на таблицы и запросы

Роль раздаёт права на таблицы, редактор структуры и файловый менеджер. Также роль определяет **пункты меню** для пользователя (см. 5.4): - ссылки на
рабочие места. **Права на таблицы** определяют, к каким данным роль имеет доступ и на каком уровне.

**Колонка Объекты (`t117`).** У роли (таблица 42) есть колонка **Объекты** —
`t117`, тип `5` (`GRANT`). В неё записываются ID объектов базы — таблиц — каждому со своим уровнем доступа. 

Таблицы идентифицируются по своим ID, но также есть синтетические ID, они дают соответствующие доступы:
1. 0 - Type editor - редактор структуры, просмотр (READ) и команды _d_* (WRITE)
2. 1 - All objects - все таблицы просмотр (READ) и команды _m_* (WRITE)
3. 10 - Files - файловый менеджер, операции с файлам шаблонов и скриптов

ID объекта выбирается из выпадающего списка, который сервер отдаёт
по `GET /{db}/grants` (issue #601; список кэшируется — issue #607). Поэтому `t117`
заполняется не произвольным текстом, а выбором уровня из этого списка — как любое
`GRANT`-поле в интерфейсе.

**Уровни доступа** (источник: [`05-access-rights.md`](integram-article-reviews/05-access-rights.md)):

| Уровень | Что разрешает |
|---------|---------------|
| `WRITE` | создание, редактирование, удаление записей |
| `READ` | только чтение |
| `BARRED` | полный запрет — объект/поле не возвращается даже в API-ответах |

Уровни наследуются от таблицы к её колонкам: задав роли `WRITE` на таблицу,
получаем право писать во все её поля, а отдельным полям при необходимости уровень
понижается (например `BARRED` на финансовые поля). Независимыми флагами
настраиваются `EXPORT` (выгрузка) и `DELETE` (удаление) — можно разрешить
редактирование, но запретить массовую выгрузку или удаление.

**Как права видны в рабочем месте.** Эффективные права роли приходят в шаблон в
переменной `{_global_.grants}` — это base64-JSON вида `{ "<typeId>": "WRITE", … }`:

```js
// templates/main.html:138
grants = JSON.parse(atob('{_global_.grants}'));
if (window.grants['1'] === 'WRITE') { /* есть право на запись */ }
```

В метаданных таблицы (`GET /{db}/metadata/{typeId}`) и в колонках отчёта каждая
колонка несёт поле `granted`: значение `WRITE` — поле редактируемое, иное — только
чтение (issue #1508). Компонент `IntegramTable` сам переключает форму в режим
«только чтение», если у роли нет `WRITE` (см.
[`WORKSPACE_DEVELOPMENT_GUIDE.md`](WORKSPACE_DEVELOPMENT_GUIDE.md) разделы 1, 6).

**Права на запросы (`report/`).** Запрос — такой же объект в списке Объекты роли,
как и таблица. Уровень доступа определяет, может ли роль открыть отчёт. Маска роли
(см. 5.7) применяется к отчётам так же, как к таблицам и API, — отдельно
настраивать фильтрацию в каждом отчёте не нужно.

### 5.7 Маска доступа

Маска — это условие фильтрации на уровне роли, ограничивающее, **какие записи**
видит пользователь (в отличие от уровня доступа, который ограничивает, **какие
объекты и поля**). Маска применяется на стороне сервера ко всем запросам —
таблицам, отчётам и API, — поэтому обойти её через экспорт нельзя (источники:
[`05-access-rights.md`](integram-article-reviews/05-access-rights.md),
`help.integram.io/#rec220708794`).

**Подстановки в маске:**

| Подстановка | Значение |
|-------------|----------|
| `[CURRENT_USER]` | текущий пользователь |
| `[ROLE]` | название роли |
| `[ROLE_ID]` | ID роли |

Типовой сценарий «менеджер видит только своих клиентов» — маска по
полю-ответственному, например `Ответственный = [CURRENT_USER]`. Тогда таблица, любой
отчёт по ней и API вернут менеджеру только его записи.

Частный случай — фильтрация по родителю: поле `up=<userId>` используется как
маска, чтобы пользователь видел только записи, созданные под ним
(`help.integram.io/#rec220708794`). Это удобно, когда записи создаются дочерними к
пользователю (`up=<userId>` при `_m_new`, см. 6.3).

Для гостевого доступа предусмотрен встроенный пользователь `guest` со своей ролью:
публичная форма (заявка с лендинга) работает под гостевым контекстом, заявитель не
видит внутренних таблиц и чужих заявок.

> Маска — это SQL-условие, поэтому слишком сложное выражение замедлит выборку на
> больших таблицах; роль стоит тестировать на реальном объёме данных
> ([`05-access-rights.md`](integram-article-reviews/05-access-rights.md)).

### 5.8 Алгоритм раздачи прав при разработке приложения

При создании нового приложения права раздаются по ролям в таком порядке:

1. **Определить роли** (менеджер, оператор, руководитель) — см. 5.3.
2. **Создать пункты меню** для каждой роли — какие рабочие места ей доступны
   (см. 5.4).
3. **Задать права роли на каждую используемую таблицу** — добавить таблицу в
   Объекты роли (`t117`) с нужным уровнем (`WRITE`/`READ`/`BARRED`), при
   необходимости понизив уровень отдельным полям (см. 5.6).
4. **Задать права роли на каждый используемый запрос** (`report/`) — аналогично,
   запрос попадает в тот же список Объекты с уровнем доступа (см. 5.6).
5. **Настроить маску доступа**, если нужна фильтрация по пользователю (например,
   менеджер видит только свои записи) — см. 5.7.

Принцип разграничения:

| Механизм | Что ограничивает |
|----------|------------------|
| Пункты меню (5.4) | **что видно** в интерфейсе роли |
| Права на объекты `t117` (5.6) | **к каким таблицам и отчётам** есть доступ и на каком уровне |
| Маска доступа (5.7) | **какие строки** внутри доступных объектов видит конкретный пользователь |

## Этап 6: Наполнение тестовыми данными

### 6.1 Скрипт наполнения

Создаётся отдельный bash-скрипт (паттерн: `seed_data.sh`), который:
1. Авторизуется
2. Создаёт записи через `_m_new/{tableId}` с указанием значений полей по алиасам

### 6.2 Принципы создания тестовых данных

1. **Сырьё** — 3-5 видов с разной шириной джамбо и остатками
2. **Слиттеры** — 3-4 слиттера с разными статусами (Свободен, Обслуживание)
3. **Типы резки** — по 1-2 на каждый вид сырья, разные втулки и намотки
4. **Заказы** — 3-5 заказов от разных заказчиков, разные даты и статусы
5. **Позиции заказа** — 2+ позиции на заказ, разное количество рулонов
6. **Производственная резка** — для каждого заказа свои резки на слиттерах, разные статусы (Ожидает, Наладка, В работе, Завершён)

### 6.3 API для создания записей

```bash
# Корневая запись (up=1)
POST /{db}/_m_new/{tableId}?JSON=1
Fields: up=1, t{tableId}="Название", t{colId1}=значение, t{colId2}=значение

# Дочерняя запись (up=parentId)
POST /{db}/_m_new/{tableId}?JSON=1
Fields: up=parentId, t{tableId}="Название", t{colId1}=значение

# Установка ref-поля (значение = ID записи из целевой таблицы)
Fields: t{colId}=579  # где 579 — ID записи в целевой таблице
```

### 6.4 Результат этапа

Файл `seed_data.sh`, который создаёт полный набор тестовых данных для демонстрации всех экранов.

## Запросы (report/) — справочник

Запросы (отчёты) — это сохранённые на сервере SQL-подобные выборки, которые
вызываются по имени или числовому ID и возвращают готовые данные для UI. Из
HTML/JS-шаблона запрос вызывается обычным HTTP-запросом (`fetch`/`XMLHttpRequest`),
без ручного составления SQL. Это основной способ наполнять таблицы, графики,
воронки и дашборды данными.

### Вызов запроса

```
GET /{db}/report/{имяЗапроса}?JSON_KV
GET /{db}/report/{id}?JSON_KV
GET /{db}/report/{id}?JSON_DATA
GET /{db}/report/{id}?JSON
```

- `{db}` — имя базы данных. В шаблонах берётся из переменной (`window.db`,
  `{_global_.z}` и т.п.).
- `{имяЗапроса}` — человекочитаемое имя отчёта (например, `Дэшборд`, `Статусы`).
  Имя с не-ASCII символами или пробелами нужно URL-кодировать через
  `encodeURIComponent` (`report/Все%20статусы%20задач`).
- `{id}` — числовой ID отчёта (например, `report/7701`). Предпочтителен: не
  зависит от переименования и не требует кодирования.

Для UI обычно нужен `JSON_KV`: это массив объектов, ключи которого совпадают с
именами колонок отчёта. `JSON_DATA` возвращает только первую строку тем же
плоским объектом, а `JSON` оставляет полный платформенный формат.

```javascript
// Числовой ID
fetch('/' + db + '/report/8027?JSON_KV')
    .then(function (r) { return r.json(); })
    .then(function (rows) { /* rows = [{...}, {...}] */ });

// По имени (с URL-кодированием)
fetch('/' + db + '/report/' + encodeURIComponent('Все статусы задач') + '?JSON_KV');
```

Примеры реальных вызовов в репозитории:
`templates/sportzania/struct.html` (`report/8027`), `templates/funnel.html`
(`report/7701`), `templates/calendar.html` (`report/5230`, `report/Все статусы задач`).

### Как узнать числовой ID запроса

Числовой ID предпочтительнее имени — не требует URL-кодирования и не ломается при
переименовании. Два способа его получить:

1. **Через заголовок ответа.** Вызов запроса по имени возвращает заголовки
   `X-Query-Id` и `X-Query-Name` (реализовано в PR #2723, `index.php:4352`):
   ```javascript
   fetch('/' + db + '/report/' + encodeURIComponent('Мой отчёт') + '?JSON_KV')
       .then(function(r) {
           var id = r.headers.get('X-Query-Id');   // "7701"
           var name = r.headers.get('X-Query-Name'); // URL-encoded name
           return r.json();
       });
   ```
   После первого вызова по имени сохраните полученный ID и используйте его в
   дальнейших запросах.

2. **Через браузер.** Откройте базу → меню «Запросы» → нажмите на нужный запрос —
   ID будет виден в URL (`/{db}/query/{id}`).

> **Важно:** `GET /{db}/metadata?JSON=1` возвращает только определения таблиц и типов,
> но **не** список запросов. Перечня всех отчётов через API нет.

### Фильтрация через FR_

Фильтры передаются как query-параметры с префиксом `FR_` плюс имя колонки. Их
можно добавлять и собирать динамически (удобно через `URLSearchParams`):

| Шаблон | Назначение | Пример |
|--------|-----------|--------|
| `FR_{Колонка}=значение` | фильтр по одному значению | `report/155675?FR_СтатусID=9002` |
| `FR_{Колонка}=IN(v1,v2)` | фильтр по нескольким значениям | `report/155675?FR_СтатусID=IN(9002,44895)` |
| `FR_{Колонка}ID=ID` | фильтр ref-поля по числовому ID | `report/Дэшборд?FR_dept=IN(2889)` |
| `FR_{Колонка}=%значение%` | частичное совпадение (LIKE) | `FR_SKU=%бумага%` |
| `FR_Date=...&TO_Date=...` | диапазон дат (пара `FR_`/`TO_`) | `?JSON_KV&FR_Date=01.01.2026&TO_Date=31.12.2026` |

> ⚠️ **Условие работы фильтра:** `FR_`/`TO_` применяется **только к колонкам,
> у которых в конструкторе запроса задано «Имя в отчёте»** (report alias). Колонка
> без имени недоступна для фильтрации извне — параметр молча игнорируется, данные
> возвращаются без фильтрации. Это намеренная защита: аналитик сам решает, какие
> поля доступны для управления снаружи.

**Ref-поля.** Для ссылочных колонок предпочтительнее фильтровать по числовому ID
целевой записи, а не по её названию — это надёжнее при совпадающих именах. Для
этого к имени колонки добавляется суффикс `ID`: `FR_{Колонка}ID=<refId>` (см.
issue #2843, `js/xcom-match.js`). Сравните: фильтр по названию статуса
(`FR_Статус=Новый`) против фильтра по ID (`FR_СтатусID=9002`, issue #2187).

**Диапазоны дат.** Колонка-дата фильтруется парой параметров: `FR_{Колонка}` —
нижняя граница, `TO_{Колонка}` — верхняя (`FR_Date=01.01.2026&TO_Date=31.12.2026`,
см. `js/dash.js`, `templates/calendar.html`).

**panelFilter в дашбордах.** В отчёте `report/Дэшборд` каждая панель может нести
поле `panelFilter` (issue #2166, #2352):
- если значение похоже на `имя=значение` (например, `FR_dept=IN(2889)`) — его
  добавляют как параметр к запросу данных панели;
- если значение похоже на `имя:значение` (например, `Тип документаID:158844`) —
  фильтр применяется **локально** при заполнении панели, а не передаётся на
  сервер (используется, когда один отчёт наполняет несколько панелей).

**FR_ vs F_ — два формата фильтра.** Параметры фильтрации могут начинаться на
`FR_`, `F_` или `TO_`. Это не одно и то же:

| Префикс | Где работает | Условия |
|---------|-------------|---------|
| `FR_{alias}=` | `report/` **и** `object/` (таблица) | Для report/: только колонки с «Именем в отчёте» |
| `TO_{alias}=` | `report/` **и** `object/` | Вторая граница диапазона (пара к `FR_`) |
| `F_{colId}=` | только `object/` (таблица) | Альтернативная форма `FR_` для URL-параметров таблиц |

`F_` обрабатывается идентично `FR_` в запросах к `object/{typeId}`, но **не**
работает в `report/`. Используйте `FR_` везде, где не уверены (issue #549,
`js/integram-table/14-url-config.js:258`).

`F_I` не является `F_{colId}`: это специальный параметр `object/` для точечной
загрузки записи по `vals.id`. Компонент таблицы читает `F_I` из URL как
`recordId`, добавляет его к запросу `/object/{typeId}/?JSON_OBJ`, а в строке
фильтров показывает как `@{recordId}` на первой колонке, чтобы пользователь видел
и мог снять фильтр.

### Форматы ответа: JSON, JSON_DATA и JSON_KV

- `?JSON_KV` / `?JSON_KV=1` — «key-value»: массив объектов
  `[{"Колонка": "значение", ...}]`,
  ключи = имена колонок отчёта. Самый удобный формат для UI, используется в
  подавляющем большинстве шаблонов. В ответе также есть заголовки
  `X-Query-Id` и `X-Query-Name`.

  ```json
  [
    { "InviteID": "236448", "Статус": "374", "GuestUser": "Fold" }
  ]
  ```

- `?JSON_DATA` / `?JSON_DATA=1` — один плоский объект по первой строке результата.
  Удобен, когда запрос гарантированно возвращает одну запись настроек или одно
  вычисленное значение.

  ```json
  { "InviteID": "236448", "Статус": "374", "GuestUser": "Fold" }
  ```

- `?JSON` / `?JSON=1` — стандартный JSON Интеграм: структура с метаданными
  колонок и строками данных. Применяется, когда нужны служебные поля платформы,
  а не только «плоский» табличный результат.

Параметры формата и фильтры комбинируются в одной строке запроса:
`report/{id}?JSON_KV&FR_Date=01.01.2026&TO_Date=31.12.2026&FR_dept=IN(2889)`.

### Использование в шаблонах (примеры)

1. **Заполнение таблицы/списка.** Загрузить строки и отрисовать их:

   ```javascript
   fetch('/' + db + '/report/' + reportId + '?JSON_KV')
       .then(function (r) { return r.json(); })
       .then(function (rows) {
           rows.forEach(function (row) { renderRow(row); });
       });
   ```

2. **Воронка продаж** — `templates/funnel.html`, `report/7701?JSON_KV`
   с фильтрами по полям «Вакансия», «Имя», «Месяц», «Тип найма» (issue #2109).

3. **Оргструктура** — `templates/sportzania/struct.html`, `report/8027?JSON_KV`
   (issue #2091).

4. **Дашборд задач** — `templates/sportzania/taskdash.html`, `report/155675?JSON_KV`,
   фильтры по всем полям, по умолчанию — текущий год (issue #2174).

5. **Вызов действия по кнопке (запрос как триггер).** Отчёт может не только
   возвращать данные, но и выполнять действие. По кнопке вызывается отчёт с
   фильтром-параметром, а результат подтверждает выполнение (issue #1158,
   `js/cabinet.js`):

   ```javascript
   // POST с _xsrf; confirmed=1 подтверждает выполнение действия
   var url = '/' + db + '/report/236429/?JSON_KV&confirmed=1&FR_InviteID=' + encodeURIComponent(id);
   // Ответ: [{ "InviteID": "236448", ... }] — непустой InviteID = успех
   ```

6. **Динамическая фильтрация из UI.** Собирать параметры `FR_` по мере выбора
   пользователем фильтров — удобно через `URLSearchParams`:

   ```javascript
   var params = new URLSearchParams('JSON_KV');
   if (statusId) params.set('FR_СтатусID', statusId);
   if (deptId)   params.set('FR_dept', 'IN(' + deptId + ')');
   var url = '/' + db + '/report/' + reportId + '?' + params.toString();
   ```

### Запросы на изменение данных (SET-запросы)

К любому отчёту можно добавить операцию изменения данных — поле SET в конструкторе
задаёт, какие колонки и как обновить у всех записей, попавших под фильтр.
Выполнение двухэтапное:

```
# Шаг 1 — превью: посмотреть, что изменится (обычный GET)
GET /{db}/report/{id}?JSON_KV&FR_InviteID=236448

# Шаг 2 — выполнение: применить изменения
POST /{db}/report/{id}
Body: confirmed=1&FR_InviteID=236448&_xsrf=...
```

- **Без `confirmed`** — сервер возвращает список затрагиваемых записей (превью).
  В UI отображается колонка «Выполнить» со ссылкой-подтверждением.
- **С `confirmed=1`** — сервер применяет SET, возвращает колонку «Выполнено».

Типичные применения:
- массово обновить поле у отфильтрованных записей;
- отозвать приглашение/статус по кнопке (issue #1158);
- удалить дубликаты по расписанию (запрос находит дубли, `?confirmed` удаляет).

```javascript
// Пример: отозвать приглашение по кнопке
var form = new URLSearchParams();
form.append('confirmed', '1');
form.append('FR_InviteID', inviteId);
form.append('_xsrf', xsrf);
fetch('/' + db + '/report/ОтозватьПриглашение', {
    method: 'POST',
    headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
    body: form.toString()
}).then(function(r) { return r.json(); })
  .then(function(rows) { /* rows[0].InviteID != "" → успех */ });
```

### Создание запроса программно

**Запрос — это обычная запись** в системной таблице с типом **22** («Запрос»). У неё есть
**подчинённая таблица «Колонки запроса»** (тип **28**), каждая строка которой описывает одну
колонку — источник поля, фильтр, функцию агрегации, сортировку, SET-операцию и т.п.
Ядро Интеграм читает эти строки и строит из них SQL-запрос SELECT.

Управление осуществляется через обычные команды `_m_*` — никаких схемных операций
(`_d_*`) не требуется. Рабочее место `templates/sql.html` реализует полный цикл работы
с запросами через эти же команды.

```bash
# 1. Создать запрос (запись в таблице Запрос, тип 22)
POST /{db}/_m_new/22?JSON
Body: t22=Мой отчёт&_xsrf=...
# Ответ: {"id": 12345}  ← ID нового запроса

# 2. Добавить колонку (запись в «Колонки запроса», тип 28)
#    up — ID запроса-родителя; t28 — ID таблицы/реквизита из метаданных;
#    t100 — отображаемое имя и внешний alias для FR_/TO_
POST /{db}/_m_new/28?JSON&up=12345&t28={fieldId}&t100=НазваниеКолонки
Body: _xsrf=...

# 3. Настроить колонку (обновить любой атрибут через _m_set)
POST /{db}/_m_set/{columnRecordId}?JSON
Body: t100=ИмяДляФильтра&t107=1&_xsrf=...

# 4. Добавить JOIN-таблицу (запись в подчинённой таблице типа 44)
#    t44 — ID типа таблицы для JOIN; t265 — алиас; t266 — условие JOIN
POST /{db}/_m_new/44?JSON&up=12345&t44={tableTypeId}
Body: t265=псевдоним&t266=условие_join&_xsrf=...
```

**Поля записи-колонки** (тип 28, изменяются через `_m_set/{colRecordId}?JSON`):

| Поле | Назначение | Значения |
|------|-----------|---------|
| `t28` | Источник колонки: ID таблицы или реквизита из метаданных | числовой ID |
| `t100` | «Имя в отчёте»: заголовок колонки и имя для внешних `FR_`/`TO_`-фильтров | произвольная строка; пробелы в URL заменяются на `_` |
| `t58` | `ALIAS`: внутренний alias SQL/JOIN, в том числе для выбора нужной записи `REP_JOIN` | строка без пробелов; это не внешний `FR_`-alias |
| `t84` | Формат отображения | `DATE`, `DATETIME`, `NUMBER`, `BOOLEAN`, `SIGNED`, `PWD` |
| `t101` | Формула / выражение | SQL-выражение, `[THIS]`, ссылки на другие колонки или ключевые слова `ID`, `T`, `ORD`, `UP`, `VAL` |
| `t102` | Фильтр FROM (нижняя граница) | значение фильтра |
| `t103` | Фильтр TO (верхняя граница) | значение фильтра |
| `t104` | Функция обработки | `abn_ID`, SQL-функция или агрегация (`SUM`, `COUNT`, `JSON_ARRAYAGG` и т.п.) |
| `t105` | HAVING FROM | значение HAVING-условия |
| `t106` | HAVING TO | значение HAVING-условия |
| `t107` | Скрыть колонку в отчёте | пустая строка = видима, любое значение = скрыта |
| `t109` | Сортировка | число ≠ 0: знак = направление, абс. значение = приоритет |
| `t65` | Итог по колонке в нижней строке отчёта | `SUM`, `AVG`, `MIN`, `MAX`, `COUNT` |
| `t132` | SET-выражение | SQL-выражение для SET-запросов |

Колонка запроса всегда является отдельной записью типа 28. Значение `t28`
указывает на таблицу или реквизит из метаданных. Если колонка выбирает саму
таблицу, ядро берёт значение записи таблицы; если колонка выбирает реквизит,
ядро сначала определяет родительскую таблицу этого реквизита. Для вычисляемой
колонки источник задают как «Вычисляемое»/`0`, а фактическое выражение кладут в
`t101`.

В одном отчёте можно вывести не больше 61 видимой колонки из-за ограничения
MySQL. Скрытые колонки (`t107`) не попадают в ответ, но могут участвовать в
фильтрах, сортировке, формулах, агрегациях и SET-выражениях.

**Поля самого запроса** (тип 22, изменяются через `_m_set/{queryId}?JSON`):

| Поле | Назначение |
|------|-----------|
| `t22` | Имя запроса |
| `t95` | Интерактивный режим (параметры задаются пользователем) |
| `t134` | Лимит строк |

**Получить колонки запроса**:
```
GET /{db}/object/28/?F_U={queryId}&JSON&LIMIT=1000
```

**Выполнить запрос**:
```
GET /{db}/report/{queryId}?JSON           # полный платформенный JSON
GET /{db}/report/{queryId}?JSON_DATA      # первая строка как объект
GET /{db}/report/{queryId}?JSON_KV&FR_Поле=X # с фильтром по t100
```

«Имя в отчёте» (поле `t100`) у колонки — ключевое условие: только колонки с
заданным `t100` доступны для внешней фильтрации через `FR_`/`TO_` (см. выше
«Условие работы фильтра»). Поле `t58` (`ALIAS`) нужно для внутреннего SQL alias и
ручных записей `FROM`/`REP_JOIN`; внешний фильтр по нему не строится.

### Как ядро строит SQL из колонок запроса

**Вся база — одна физическая таблица** с именем равным slug базы (например `ateh`).
Никаких отдельных SQL-таблиц для «Заказов», «Позиций» и т.п. нет — это EAV-модель:
каждая запись имеет поля `id`, `t` (тип), `up` (родитель), `val` (значение).

Ядро не знает заранее схему SQL. Оно читает строки «Колонок запроса» (type 28),
смотрит метаданные таблиц и реквизитов, а затем строит `LEFT JOIN`-цепочки.
Связи берутся только из метаданных:

- реквизит напрямую подчинён таблице через `up`;
- дочерняя/табличная часть связана с родителем через `up`;
- ссылка хранится как отдельная ref-запись, где `up` указывает на источник, а `t`
  указывает на целевую запись справочника.

Алгоритм идёт слева направо по колонкам отчёта.

1. Первая реальная колонка определяет мастер-таблицу. Если колонка выбрала
   реквизит, мастер-таблица — родительская таблица этого реквизита.
2. Для каждой следующей колонки ядро определяет её таблицу: сама таблица, если
   выбрана первая колонка таблицы, или родительская таблица, если выбран реквизит.
3. Если эта таблица уже присоединена, новый реквизит добавляется к существующему
   join через `req.up = parent.id`; дополнительные ветки не проверяются.
4. Если таблица ещё не присоединена, ядро перебирает уже присоединённые таблицы
   от мастер-таблицы к более поздним и берёт первую подходящую связь.
5. Порядок проверки веток фиксированный и соответствует комментариям в
   `Compile_Report()`:

| Ветка | Когда срабатывает | Что соединяет |
|-------|-------------------|---------------|
| `Reference to us` | Уже присоединённая таблица имеет ref-реквизит на новую таблицу | ref-запись источника с целевой записью новой таблицы |
| `We are an Array` | Новая таблица является дочерней/array-таблицей уже присоединённой | `new.up = joined.id` |
| `We have a Reference` | Новая таблица имеет ref-реквизит на уже присоединённую | ref-запись новой таблицы с записью уже присоединённой |
| `We got an Array` | Уже присоединённая таблица является дочерней для новой | `joined.up = new.id` |

Если ни одна ветка не связала очередную таблицу, ядро делает следующий проход:
возможно, сначала нужно присоединить промежуточную таблицу из более поздней
колонки. После 1000 проходов отчёт завершается ошибкой «Не могу связать колонки
отчета».

Alias обычной таблицы в SQL = `a{typeId}` (тип 42 → `a42`). Ref-записи получают
alias `r{typeId}`. Если в отчёте есть ручная запись `REP_JOIN` (type 44), `t58`
колонки должен совпасть с alias этой записи, иначе ядро не применит ручной JOIN к
данной колонке.

#### Пример: права пользователя

Для отчёта с колонками `Пользователь, Роль, Объекты, Описание` используются такие
связи из метаданных:

- `Пользователь` — таблица 18, это мастер-таблица;
- `Роль` — ref-реквизит пользователя на таблицу 42;
- `Объекты` — array-таблица 116 под ролью 42;
- `Доступ -> Описание` — ref-реквизит array-строки 116 на таблицу 47, затем
  обычный реквизит 120 под таблицей 47.

На таких метаданных ядро строит SQL следующего вида:

```sql
SELECT a18.val as 'Пользователь',a42_val as 'Роль',a116_val as 'Объекты',a120_val as 'Описание'
FROM xcom a18
  LEFT JOIN (SELECT r42.up,a42.val a42_val,a42.id a42_id
  FROM xcom r42,xcom a42 USE INDEX (PRIMARY)
    WHERE a42.id=r42.t  AND a42.t=42
   ) a42 ON a42.up=a18.id
  LEFT JOIN (SELECT a116.up,a116.val a116_val,a116.id a116_id
  FROM xcom a116
    WHERE a116.t=116
    ) a116 ON a116.up=a42_id
  LEFT JOIN (SELECT r47.up,a120.val a120_val
  FROM xcom r47,xcom a47 USE INDEX (PRIMARY)
   LEFT JOIN xcom a120 ON a120.up=a47.id AND a120.t=120
    WHERE a47.id=r47.t  AND a47.t=47
   ) a47 ON a47.up=a116_id
WHERE a18.up!=0 AND length(a18.val)!=0 AND a18.t=18
```

В `JSON_KV` результат выглядит как плоские строки:

```json
[
  {
    "Пользователь": "user",
    "Роль": "user",
    "Объекты": "Пользователь",
    "Описание": "Пользователь"
  },
  {
    "Пользователь": "user",
    "Роль": "user",
    "Объекты": "Пользователь -> Имя",
    "Описание": "Пользователь -> Имя"
  },
  {
    "Пользователь": "user",
    "Роль": "user",
    "Объекты": "Запрос",
    "Описание": "Запрос"
  }
]
```

#### Значения, функции, формулы и SET

По умолчанию колонка выбирает `val`. Для ref-реквизита это значение целевой
записи справочника, а не техническая ref-строка. Если нужен ID, используйте
функцию `Abn_ID`/`abn_ID`: для обычного реквизита она возвращает ID записи
реквизита, а для ref-реквизита — ID целевой записи. Также есть системные
`abn_UP`, `abn_TYP`, `abn_ORD`, `abn_REQ`, `abn_BT`.

Функция `t104` может быть обычной SQL-функцией или агрегацией. В коде как
агрегирующие распознаются `AVG`, `COUNT`, `MAX`, `MIN`, `SUM`, `GROUP_CONCAT`,
`JSON_ARRAYAGG`. Если в старом описании встречается `group_arrayagg`, в текущем
коде ему соответствует `JSON_ARRAYAGG`. Если хотя бы одна видимая колонка
агрегируется, ядро автоматически строит `GROUP BY` по всем видимым
неагрегированным колонкам.

**Поле `t101` (Формула)** вставляется в текст SELECT как SQL-выражение:

```sql
-- t101 = "ROUND(a42.val * 0.20, 2)"  →  ядро подставит:
SELECT ROUND(a42.val * 0.20, 2) as 'НДС'
```

Формула может использовать `[THIS]`, чтобы сослаться на текущее поле, и
`[ИмяКолонки]`, чтобы сослаться на выражение другой колонки. После сборки
полей ядро заменяет такие ссылки в SELECT, WHERE, GROUP BY, HAVING, ORDER BY,
JOIN-условиях и SET-выражениях на реальные SQL-выражения. Защита минимальная:
формулы с ключевыми словами `from`, `select`, `table` блокируются, поэтому
формула должна считаться доверенным SQL, а не пользовательским вводом.

SET-запрос использует поле `t132`. Это SQL-выражение вычисляется на тех же
данных, что и SELECT, и применяется только после подтверждения
`confirmed=1`. Без подтверждения отчёт показывает превью затрагиваемых строк.

В фильтрах, формулах и SET-выражениях можно использовать встроенные значения:
`[TODAY]`, `[NOW]`, `[YESTERDAY]`, `[TOMORROW]`, `[MONTH_AGO]`, `[WEEK_AGO]`,
`[MONTH_PLUS]`, `[USER_ID]`, `[USER]`, `[ROLE_ID]`, `[ROLE]`, `[TSHIFT]`,
`[REMOTE_ADDR]`, `[REMOTE_HOST]`, `[HTTP_USER_AGENT]`, `[HTTP_REFERER]`,
`[HTTP_HOST]`, `[REQUEST_URI]`.

# Инструкция для ИИ-агента по разработке Low-Code приложений на Интеграм

> У ИИ есть знания о миллионах приложений и всех типовых пользовательских путях.
> Задача агента — на основе предметной области сделать готовое приложение,
> избавляя пользователя от необходимости принимать какие-либо решения в ходе
> разработки. 80-90% функционала должно быть готово после первой итерации, а далее
> пользователь запросит то, чего не может получить от программистов и готовых
> коробок. **100% кастомизация лучших практик.**

## Что такое Интеграм-приложение

Интеграм — low-code платформа (ideav.ru). Приложение = база данных (workspace) + таблицы + HTML/JS шаблоны для UI. Данные хранятся в таблицах, интерфейс — кастомные HTML-страницы с JS, которые загружаются в файловое хранилище базы.

Данный документ описывает все шаги создания полноценного и самодостаточного приложения, начиная с пустой базы Интеграм. Работа начинается с двух подготовительных этапов — **инициации** (получение доступа к пустой базе по токену) и **постановки задачи** (анализ предметной области и генерация пользовательского пути), — после которых следуют технические этапы:
1. Консолидация требований
2. Создание структуры данных и наполнение её тестовыми данными
3. Создание пользователей, их ролей и пунктов меню
4. Создание запросов на выборку и изменение данных (для защиты слоя данных)
5. Создание рабочих мест, работающих с базой по API
6. Написание и прогон тест-плана для полного пользовательского пути

Если нужен не справочник по всем деталям, а один короткий сквозной пример,
сначала прочитай [`integram-minimal-full-cycle-project.md`](integram-minimal-full-cycle-project.md):
там мини-Atex проходит путь от Excel пользователя до схемы, ролей, экранов и
проверочного сценария.

**Ключевые понятия:**
- **База данных (workspace)** — изолированное пространство с таблицами, файлами, пользователями
- **Таблица (type)** — аналог таблицы SQL, имеет ID, колонки, связи
- **Колонка (requisite/req)** — поле таблицы, имеет ID и алиас (русское имя)
- **Ссылка (ref)** — связь-указатель на одну запись справочника (foreign key); см. [2.5](#25-два-типа-связей-ссылка-справочник-и-подчинённая-таблица)
- **Подчинённая таблица** — набор дочерних записей, принадлежащих одному родителю (особый тип связи Интеграма); см. [2.5](#25-два-типа-связей-ссылка-справочник-и-подчинённая-таблица)
- **Запись (object)** — строка таблицы, имеет ID
- **Запрос (report/query)** — команда Интеграму на выборку и обработку данных
- **Шаблон (template)** — HTML-страница, лежит в репозитории в `templates/{имя базы}/` (если папки нет — создать), рендерится Интеграм

## Этап инициации

На этом этапе о будущем приложении ещё ничего не известно. Единственная задача — получить доступ к **пустой** базе Интеграм по токену. Регистрация вашего экземпляра доступна [здесь](https://ideav.ru/start.html#reg).

Когда агент готов приступить к работе, он просит пользователя:

> **«Создай пустую базу данных в Интеграм, заведи в ней пользователя `claude`, придумай ему токен из 15–30 символов и впиши этот токен прямо в поле "Токен" пользователя. Затем сообщи мне адрес базы и этот токен.»**

Что делает пользователь:
1. Создаёт пустую базу на [ideav.ru](https://ideav.ru)
2. Заходит в базу → Пользователи
3. Создаёт пользователя с логином `claude`
4. Придумывает токен из 15–30 символов (произвольная строка букв и цифр) и вписывает его **прямо в поле «Токен»** (колонка `Token`, `t125`) этого пользователя
5. Сообщает агенту адрес базы (например `ideav.ru/myapp/`) и значение токена

Агент **работает только по токену**: он никогда не вводит пароль и никогда не вызывает форму входа `/auth`. Это принципиально — форма входа (`POST /{db}/auth`) защищена капчей Yandex SmartCaptcha (проверка в функции `verifyCaptcha`), которую автоматический агент пройти не может. Токен — это и есть сессионный ключ пользователя (колонка `Token`, `t125`); имея его, агент авторизуется напрямую и капчу не встречает (issue #2937, #2906).

> Колонка `Token` (`t125`), в отличие от пароля, сервером **не хешируется** и записывается как есть — поэтому пользователь может вписать в неё придуманное значение прямо через веб-интерфейс. Альтернативно администратор может задать токен через API: `POST /{db}/_m_set/{userId}` с `t125=<секрет>`.

**Как агент использует токен** (без `/auth`, без капчи). Токен распознаётся механизмом валидации сессии в любом из видов:

```bash
# 1. Через HTTP-заголовок X-Authorization (рекомендуется — это транспорт
#    браузерного SPA, и его сервер читает на любом запросе)
curl -s -H "X-Authorization: <token>" "https://ideav.ru/{db}/metadata?JSON=1"

# 2. Через cookie idb_{db} (тоже читается сервером для GET-чтений)
curl -s -b "idb_{db}=<token>" "https://ideav.ru/{db}/metadata?JSON=1"

# 3. Через form-поле token= (для POST _m_*/_d_*)
curl -s -X POST "https://ideav.ru/{db}/_m_new/18?JSON=1" \
  -F "token=<token>" -F "_xsrf=<xsrf>" -F "up=1" -F "t18=manager"
```

> ⚠️ Сервер читает токен **только** из заголовка `X-Authorization`, cookie
> `idb_{db}` или form-поля `token=`. Заголовок `Authorization: Bearer <token>`
> и строка запроса `?token=<token>` сервером **игнорируются** — на такой запрос
> приходит `401 [{"error":"No authorization token provided"}]` (issue #3000,
> подтверждено зондированием живого сервера в atex#44). PowerShell-скрипты в
> `docs/` посылают токен заголовком `X-Authorization`, потому что вручную
> заданный через `-Headers` cookie `Invoke-RestMethod` отбрасывает.

**PowerShell-скрипты из `docs/`.** Токен можно передать параметром `-Token` или
сохранить в переменной окружения:

```powershell
$env:INTEGRAM_TOKEN = "<token>"
[Environment]::SetEnvironmentVariable("INTEGRAM_TOKEN", "<token>", "User")
```

Первый вариант задаёт токен только для текущего процесса PowerShell. Второй
вариант сохраняет пользовательскую переменную для будущих сессий; скрипты также
читают её в уже открытой сессии напрямую через .NET API. Порядок поиска:
`Process, User, Machine`. Для `_xsrf` используется такой же порядок поиска
переменной `INTEGRAM_XSRF`.

**Получение `_xsrf` под токеном.** Для POST-команд изменения данных
(`_m_*`/`_d_*`) нужен `_xsrf`. Имея только токен, его забирают одним вызовом —
`/auth` (и значит капча) для этого не требуется:

```bash
curl -s -H "X-Authorization: <token>" "https://ideav.ru/{db}/xsrf?JSON"
# Ответ: {"_xsrf":"<hex>","token":"<token>","user":"claude","role":"...","id":"<userId>","msg":""}
```

**`_xsrf` обязателен** для POST `_m_*`/`_d_*`, даже если сессия установлена через cookie. Дальше агент подставляет полученные `token` и `_xsrf` в `api_call` и самостоятельно работает с API базы.

## Этап постановки задачи: анализ предметной области и генерация пользовательского пути

На этом этапе ты **не пишешь код**. Ты анализируешь эксели пользователя и на основе лучших практик предлагаешь готовую структуру приложения.

У тебя уже есть знания о миллионах приложений и всех типовых пользовательских путях. Задача агента — на основе предметной области предложить готовые решения по структуре и интерфейсу, избавляя пользователя от необходимости принимать дизайнерские решения, которые пользователь только подтвердит.

### 1. Что нужно от пользователя (и только это)

| Вводная | Пример | Зачем агенту |
|---------|--------|--------------|
| Тематика — одно предложение | «Режу джамбо-рулоны на маленькие ролики для термопринтера» | Определить домен (производство) и типовые паттерны |
| Excel-файлы (или описание колонок) | Файл «Заказы.xlsx» с колонками: Клиент, Дата, Ширина, Намотка, Количество | Вытащить сущности, поля и связи |
| Боль — 2-3 фразы | «Теряю заказы», «Не помню какой рулон уже порезан» | Понять приоритетные экраны |

**Не спрашивай пользователя:**
- «Какие экраны вам нужны?» — ты знаешь типовые экраны для его домена
- «Как назвать кнопку?» — Агент должен предлагать стандартные, интуитивно понятные названия для элементов интерфейса на основе лучших практик
- «Какие роли сделать?» — для производства это менеджер/оператор/руководитель, для услуг — администратор/мастер

**Спроси только то, что нельзя вывести из экселей:**
- «Один оператор на станок или несколько?» (влияет на назначение задач)
- «Нужен ли доступ клиентам для отслеживания заказа?» (гостевая роль)
- «Есть ли уже список клиентов или создаёте на лету?» (справочник vs быстрый ввод)
- «Есть ли сайт, логотип или фирменные цвета компании?» (корпоративный стиль для
  оформления рабочих мест — визуальная составляющая важна для клиента, см.
  [Этап 4.1](#41-визуальная-составляющая-и-корпоративный-стиль-клиента))

### 2. Алгоритм анализа экселей

1. **Распарси эксели.** Каждый файл / лист — кандидат в таблицу. Колонки — кандидаты в поля.
2. **Найди повторы.** Если в разных файлах встречается одинаковое название (например «Клиент» или «Тип сырья») — это потенциальная связь.
3. **Определи домен по тематике.** «Производство», «Услуги», «Торговля», «Расписание» — у каждого есть типовой набор экранов.
4. **Выбери максимально подходящий вариант реализации** и примени типовой пользовательский путь для этого домена.

### 3. Что агент выдаёт пользователю на подтверждение

Создай короткий документ без технических деталей. Пользователь должен понять его за 2 минуты и сказать «да» или поправить.

```markdown
# Приложение: [Название из тематики]

## Что будет делать приложение
[Одно предложение — перефразированная тематика пользователя]

## Кто будет работать
- **[Роль 1]:** [что делает — глаголами]
- **[Роль 2]:** [если нужно]

## Основные экраны

### [Название экрана 1]
Открывается [когда], показывает [что], можно [действия]

### [Название экрана 2]
...

## Что я сделал на основе ваших экселей
- Из файла «[имя]» взял сущности: [перечислить]
- Из колонок сделал поля: [перечислить]
- Связи определил по повторяющимся названиям: [перечислить]

## Что нужно уточнить
- [Только вопросы, ответа на которые нет в экселях]
```

Пользователь дал:

```markdown
## Кто будет работать

## Основные экраны

### Журнал заказов

Показывает все заказы, сгруппированные по статусам: Ожидает → В работе → Готов.

### Пульт оператора

## Что я сделал на основе ваших экселей
- Из «Заказы.xlsx» взял сущность «Заказ» с полями: Клиент, Дата, Сырьё, Ширина рулона, Намотка, Количество
- Из «Склад.xlsx» взял сущность «Сырьё» с полями: Название, Ширина джамбо, Остаток

## Что нужно уточнить
- Один оператор работает на одном станке или может работать на разных?
- Нужно ли клиентам дать доступ смотреть статус их заказов?
```

## Этап 2: Проектирование схемы данных

**Принцип работы с неопределённостью:** Если техническая реализация (метод API, параметр, поведение ядра) неясна, агент должен **запросить уточнение**, а не строить предположения. Это предотвращает ошибки проектирования. Любая полученная в результате уточнения информация должна быть **немедленно зафиксирована** в соответствующем разделе документа для использования в будущем.

**Принцип безопасности данных (бэкап перед необратимым):** Перед ЛЮБЫМИ необратимыми изменениями боевой БД — удаление записей (`_m_del`), дроп таблиц/реквизитов (`_d_*`), массовая чистка данных, переподчинение таблиц — агент **сначала делает полный бэкап базы**, и только потом выполняет операцию. Откатить удаление данных/таблиц иначе нельзя; схемные `_d_*` к тому же способны исказить метаданные. Пропуск бэкапа допустим лишь по явному подтверждению заказчика, что данные не ценны или бэкап уже сделан.

### 2.1 Правила проектирования таблиц (из StructureRules.md)

1. **Первая колонка любой таблицы — собственный идентификатор объекта** (обычно тип SHORT=3, но может быть число, дата или иной тип)
   - Это не ссылка! Это то, ЧТО такое эта запись
   - Примеры: название сырья, номер заказа, имя слиттера
   - **Для таблиц-справочников первая колонка всегда уникальна:** при создании
     справочника командой `_d_new` обязательно передавай `unique=1`

2. **Имена таблиц — единственное число, на русском**
   - Правильно: "Сырьё", "Заказ", "Слиттер"
   - Неправильно: "Сырьё и материалы", "Orders"

3. **Колонки-ссылки (ref) создаются в таблице-источнике**
   - Если "Тип резки" ссылается на "Сырьё" → ref создаётся в "Тип резки"
   - Не создавай колонку в справочнике для обратной связи (это делает сама платформа)
   - Можно делать ссылку на записи в подчиненной таблице, например, на запись об обеспечении какой-то позиции заказа (позиции подчинены заказу)

4. **Первая колонка таблицы является именем объекта** — отдельное поле "Название", "Имя", "ФИО" не нужно, это дублирование

5. **Можно сделать составной ключ уникальности** — при импорте и создании записей он будет соблюдаться, и он может не включать первую колонку

### 2.2 Типы колонок (t=...)

| Код | Тип | Описание |
|-----|-----|----------|
| 3 | SHORT | Короткий текст (названия, имена) |
| 8 | CHARS | Длинный текст |
| 9 | DATE | Дата (YYYY-MM-DD) |
| 4 | DATETIME | Дата и время |
| 11 | BOOLEAN | Да/нет |
| 12 | MEMO | Большой текст (описания, комментарии) |
| 13 | NUMBER | Целое число (цены, количества, метры) |
| 14 | SIGNED | Число с десятичной точкой (метражи м², остатки) |
| 6 | PWD | Пароль — хешируется сервером, не возвращается в чтении |
| 10 | FILE | Файл (вложение, аватар) |
| 5 | GRANT | Маска прав доступа (используется в колонке Объекты роли) |
| 2 | HTML | HTML-разметка (теги не экранируются при выводе) |
| 7 | BUTTON | Действие (кнопка) над записью |
| 15 | CALCULATABLE | Вычисляемая колонка |
| 16 | REPORT_COLUMN | Колонка отчёта |
| 17 | PATH | Путь (служебное) |

> Полный список доступных в конкретной БД — `GET /{db}/basics`
> Пример ответа: `{"3":"SHORT","8":"CHARS","9":"DATE","13":"NUMBER","14":"SIGNED","11":"BOOLEAN","12":"MEMO","4":"DATETIME","10":"FILE","2":"HTML","7":"BUTTON","6":"PWD","5":"GRANT","15":"CALCULATABLE","16":"REPORT_COLUMN","17":"PATH"}`

> ⚠️ **DATETIME (type 4) приходит из API как unix-штамп в секундах** (например
> `1777815000`), а не как готовая строка. На рабочем месте всегда форматируй
> такое значение в человекочитаемый вид (`ДД.ММ.ГГГГ ЧЧ:ММ`) — в atex для этого
> есть общий помощник `AtexRefSearch.formatDateTime(value)` (`ref-search.js`).
> Сырой штамп оставляй только для сортировки/ключей. То же касается ссылок на
> запись, чьё главное значение — DATETIME: ярлык ссылки = тот же штамп.

### 2.3 Документирование схемы

Запиши схему в виде таблицы:

```markdown
### Таблица: Сырьё (RawMaterial)
| Поле | Тип | Описание |
|------|-----|----------|
| Название | SHORT (3) | Первая колонка — тип/название сырья |
| Ширина джамбо, мм | NUMBER (13) | Ширина исходного рулона |
| Остаток, м | NUMBER (13) | Остаток на складе |
| Описание | MEMO (12) | Дополнительная информация |

### Связи:
- Тип резки.Сырьё → Сырьё (ref)
```

### 2.4 Результат этапа

Файл со схемой всех таблиц, колонок и связей. Это станет основой для скрипта создания.

### 2.5 Два типа связей: ссылка (справочник) и подчинённая таблица

В Интеграме между таблицами бывает **ровно два** типа связей — **ссылка** и
**подчинённая таблица**. Перепутать их — самая частая и самая дорогая ошибка
проектирования: схему приходится переделывать, а рабочие места не подключаются
по API (Пример такой ошибки и её последствий приведён в разделе 2.5.6.).
**Прежде чем добавить любую связь, определи её тип по правилам ниже.**

Для человека это различие объяснено в блоге Интеграма —
[«Как отличить подчинённую таблицу от справочника»](https://blog.ideav.ru/posts/kak-otlichit-podchinennuyu-tablicu-ot-spravochnika/).
Ниже то же самое формализовано для LLM, чтобы не путаться.

#### 2.5.1 Определения

- **Ссылка (она же «справочник», FK).** Поле, которое указывает на **одну уже
  существующую запись** другой таблицы. Целевая запись существует независимо и
  **переиспользуется многими** записями-источниками. Это прямой аналог foreign
  key реляционной БД. Пример: у «Заказа» поле «Клиент» указывает на запись
  справочника «Клиент»; один и тот же клиент привязан ко множеству заказов. Также
  ссылка — это «Статус», «Вид сырья», «Станок», «Ответственный».

- **Подчинённая таблица (дочерняя, «табличная часть»).** **Набор записей,
  которые целиком принадлежат одной родительской записи** и создаются специально
  под неё. Дочерние записи не переиспользуются и не имеют смысла вне родителя.
  Пример: «Позиции заказа» — каждая позиция принадлежит ровно одному заказу;
  «контакты сотрудника»; «Объекты» и «пункты Меню» у Роли. Это особый тип связи
  Интеграма: вместо отдельной FK-колонки на родителя платформа даёт наглядную
  навигацию — у родителя сразу видно количество и список дочерних записей.

#### 2.5.2 Правило выбора (чек-лист)

Задай по очереди четыре вопроса. Если хотя бы один уверенно указывает на
«справочник» — это **ссылка**. Подчинённую таблицу выбирают, только когда записи
действительно принадлежат одному родителю и нигде больше не используются.

1. **Тест переиспользования.** Может ли одна и та же связанная запись быть
   привязана к нескольким разным родителям?
   - Да (один статус «Новый» — у сотен заказов; один клиент — у многих заказов) →
     **ссылка**.
   - Нет (эта строка существует только ради одного родителя) → **подчинённая
     таблица**.

2. **Тест жизненного цикла.** Существует ли связанная запись сама по себе,
   независимо от родителя, и ведётся ли она как отдельный справочник?
   - Да (статусы, клиенты, виды сырья, станки заполняются заранее и отдельно) →
     **ссылка**.
   - Нет (запись рождается вместе с родителем и удаляется вместе с ним) →
     **подчинённая таблица**.

3. **Тест ввода в UI.** Как пользователь заполняет связь?
   - Выбирает готовое значение из выпадающего списка → **ссылка**.
   - Добавляет новые строки в табличную часть карточки родителя → **подчинённая
     таблица**.

4. **Тест направления и владения.** «Многие источники → один справочник» (N:1),
   FK лежит в источнике → **ссылка**. «Один родитель → много детей» (1:N), родитель
   «владеет» детьми → **подчинённая таблица**.

#### 2.5.3 Сравнение

| Критерий | Ссылка (справочник) | Подчинённая таблица |
|----------|---------------------|---------------------|
| Что хранит | указатель на одну запись другой таблицы | набор собственных записей родителя |
| Переиспользование цели | да, многими источниками | нет, запись принадлежит одному родителю |
| Кардинальность | N:1 (много источников → один справочник) | 1:N (один родитель → много детей) |
| Где «лежит» связь | колонка-FK в таблице-источнике | дочерний тип добавлен реквизитом родителя; запись создаётся с `up=<родитель>` |
| Аналог в SQL | foreign key | составная часть / детейл-таблица |
| Ввод в UI | выбор из выпадающего списка | добавление строк в табличную часть |
| Примеры | Заказ→Клиент, →Статус; Тип резки→Вид сырья; Резка→Станок | Заказ→Позиции; Роль→Объекты, →Меню; Сотрудник→Контакты |

#### 2.5.4 Как тип связи виден в метаданных (самопроверка)

После создания связи перечитай `GET /{db}/metadata?JSON=1` и посмотри на
реквизит-связь в `reqs[]` родительской таблицы:

- **Ссылка** — у реквизита есть поля **`ref`** (= id целевой таблицы) и `ref_id`.
  Базовый `type` = 3 (показывается значение целевой записи). Целевая таблица в
  своём объекте имеет поле `referenced`. Пример из базы `atex` (`Тип резки`):
  ```json
  {"id":"597","val":"Сырьё","orig":"469","type":"3","ref":"469","ref_id":"543"}
  ```
- **Подчинённая таблица** — у реквизита есть поле **`arr_id`** (= id дочерней
  таблицы) и **нет `ref`**. Пример из `atex` (`Заказ`):
  ```json
  {"id":"1177","val":"Позиция заказа","orig":"507","type":"3","arr_id":"507"}
  ```

> **Главный признак для самопроверки:** `ref`/`ref_id` → ссылка; `arr_id` без
> `ref` → подчинённая таблица. Если связь, которая **по смыслу должна быть
> справочником**, в метаданных имеет `arr_id` вместо `ref` — типы перепутаны,
> структуру надо переделать.

В Редакторе структуры (визуально, для человека) то же различие видно так:
справочник отображается с **именем целевой таблицы в скобках**, подчинённая
таблица — **без скобок**.

> Все поля ответа `metadata` (включая `attrs`, алиасы, значения по умолчанию и
> признак «ссылка на любую запись») разобраны в справочнике
> [«Метаданные (`metadata`)»](#метаданные-metadata--справочник).

#### 2.5.5 Как создаётся каждый тип (API)

- **Ссылку** создают двухшаговым паттерном `_d_ref` + `_d_req` — см.
  [3.2 шаг 6](#32-последовательность-api-вызовов). Результат: реквизит с `ref`.
- **Подчинённую таблицу** создают, добавляя **тип дочерней таблицы** реквизитом
  родителя — `_d_req/{parentTableId} t={childTableId}`, а записи создают с
  `up=<parentRecordId>` — см. [3.4](#34-подчинённые-таблицы). Результат: реквизит
  с `arr_id`.

> **Практический вывод.** Тип связи задаётся тем, что подставлено в `t` при
> `_d_req`: ref-тип (полученный из `_d_ref`) даёт **ссылку** с полем `ref`; id
> дочерней таблицы даёт **подчинённую таблицу** с полем `arr_id`. Поэтому ссылку
> создавай двухшаговым `_d_ref` + `_d_req` (см. [3.2 шаг 6](#32-последовательность-api-вызовов)),
> а подчинённую таблицу — `_d_req t={childTableId}` с последующим `up=` (см.
> [3.4](#34-подчинённые-таблицы)).

#### 2.5.6 Разбор реальных баз

Две базы прошли путь `integram-app-workflow.md` с разным результатом — это
эталонные «как не надо» и «почти правильно».

- **`ateh` — антипример (типы перепутаны).** Почти все справочные связи ошибочно
  созданы как подчинённые таблицы: у `Заказ.Клиент`, `Позиция заказа.Вид сырья`,
  `Позиция заказа.Тип резки`, `Производственная резка.Слиттер / Тип резки /
  Партия сырья` в метаданных стоит `arr_id`, а не `ref`. Следствие: каждый заказ
  «владеет» собственными клиентами вместо ссылки на общий справочник «Клиент»,
  справочники не переиспользуются, рабочие места невозможно подключить по API. По
  смыслу это **ссылки**. Подчинёнными в `ateh` корректно остаются только
  `Заказ.Позиция заказа`, `Позиция заказа.Обеспечение`, `Тип резки.Полоса`.

- **`atex` — почти правильно.** Справочники сделаны ссылками
  (`Тип резки.Сырьё` → `ref=469`), позиции — подчинёнными
  (`Заказ.Позиция заказа` → `arr_id=507`). Единственная ошибка связи:
  `Производственная резка` ссылается на **`Заказ`** (498), а должна ссылаться на
  **`Позиция заказа`** (507) — резка выполняется под конкретную позицию заказа, а
  не под заказ целиком.

## Этап 3: Создание таблиц (скрипт)

### 3.1 Структура скрипта

Скрипт на bash, использующий curl для вызовов API Интеграм. Паттерн: [`create_perelidoz.ps1`](https://github.com/ideav/crm/blob/main/docs/create_perelidoz.ps1).

**Базовый шаблон функции `api_call`:**

```bash
api_call() {
    local endpoint="$1"
    local method="${2:-POST}"
    shift 2
    local url="${BASE_URL}/${DB}/${endpoint}?JSON=1"
    # ... добавляет token и _xsrf ко всем запросам
    curl -s -k -X POST "$url" -F "token=$TOKEN" -F "_xsrf=$XSRF" "$@"
}
```

**Критически важно:** функция `log()` должна писать в stderr (`>&2`), иначе вызовы `api_call` внутри `$()` будут захватывать логи вместе с JSON-ответом.

### 3.2 Последовательность API-вызовов

0. **Запрос метаданных текущей базы:**
   ```bash
   GET /{db}/metadata
   ```
   Эта команда вернет всю структуру таблиц базы, по которой можно ориентироваться: таблицы, связи, значения по умолчанию, алиасы и т.д.

1. **Получение `_xsrf` под уже выданным токеном:**
   ```bash
   GET /{db}/xsrf?JSON=1
   X-Authorization: <token>
   Returns: { token, _xsrf, id, user, role }
   ```
   Агент не вызывает `POST /{db}/auth`: этот путь требует пароль и может упереться
   в SmartCaptcha. Токен выдаётся пользователем на этапе инициации.

2. **Создание таблицы:**
   ```bash
   POST /{db}/_d_new?JSON=1
   Fields: t=3, val="Имя таблицы"
   Returns: { obj: tableId }
   ```
   Параметр `t` задаёт **тип первой колонки** таблицы (см. 2.2). Эта первая
   колонка автоматически становится «именем записи» (`_value`) и **доступна в
   API записей по ключу `t<tableId>`** — например, для таблицы id=470 при
   `_m_new`/`_m_set` название записи пишется как `t470=<значение>`. То же
   справедливо для всех таблиц (в т.ч. системных: t18=login для Пользователя,
   t42=name для Роли, t151=name для Меню).

   **Справочники:** для любой таблицы-справочника (независимой или зависимой)
   обязательно передавай `unique=1` уже при создании через `_d_new`:
   ```bash
   POST /{db}/_d_new?JSON=1
   Fields: t=3, val="Сырьё", unique=1
   Returns: { obj: tableId }
   ```
   Этот параметр включает признак уникальности первой колонки справочника. Без
   него справочник сможет принять дубли видимых значений, а ссылки и импорты
   начнут опираться на неоднозначные записи.

   **Автоинкремент** (только для `t=13` NUMBER): добавь параметр `unique=1`:
   ```bash
   POST /{db}/_d_new?JSON=1
   Fields: t=13, val="Номер резки", unique=1
   Returns: { obj: tableId }
   ```
   Первая колонка получит свойство `unique` и при `_m_new` сама посчитает
   следующий целочисленный номер. Полезно для журнальных таблиц с «Номер 1, 2,
   3…».

3. **Создание типа колонки:**
   ```bash
   POST /{db}/_d_new?JSON=1
   Fields: t=<typeCode>, val="Тип колонки"
   Returns: { obj: typeId }
   ```

4. **Добавление колонки в таблицу:**
   ```bash
   POST /{db}/_d_req/{tableId}?JSON=1
   Fields: t=<typeId>
   Returns: { id: reqId }
   ```

5. **Установка отображаемого имени колонки (alias реквизита):**
   ```bash
   POST /{db}/_d_alias/{reqId}?JSON=1
   Fields: val="Имя колонки"
   ```
   Эта команда пишет имя в `attrs.alias` записи-реквизита. Используется при
   создании новой колонки в таблице, чтобы дать ей отличное от типа имя
   (например, тип называется «Дата», а в этой таблице колонка должна
   называться «Дата прихода»). Тот же `_d_alias` ставит псевдоним для
   ref-колонок (ссылка на «Клиент» в карточке заказа может показываться как
   «Заказчик»).

   > **Не вызывай `_d_alias` на ID таблицы (типа SHORT/DATETIME).** Сервер
   > запишет в `val` таблицы JSON-блоб вида `{"alias":"...","default":"..."}`,
   > и название таблицы во всём UI начнёт показываться как этот блоб. Для
   > переименования самой таблицы или существующего типа колонки используй
   > `_d_save` (см. 3.6).

6. **Создание ссылки (ref) — двухшаговый паттерн.**

   > Используй этот паттерн, только если связь по правилам
   > [2.5](#25-два-типа-связей-ссылка-справочник-и-подчинённая-таблица) — это
   > **ссылка** (указатель на переиспользуемую запись справочника). Если связь —
   > **подчинённая таблица** (набор записей, принадлежащих одному родителю),
   > используй паттерн из [3.4](#34-подчинённые-таблицы), а не этот.

   Шаг 6.1: создать ref-тип, указывающий на целевую таблицу:
   ```bash
   POST /{db}/_d_ref/{targetTableId}?JSON=1
   Fields: token=<token>&_xsrf=<xsrf>
   Returns: { id: <targetTableId>, obj: <refTypeId> }
   ```

   Шаг 6.2: добавить ref-тип как реквизит в таблицу-источник:
   ```bash
   POST /{db}/_d_req/{sourceTableId}?JSON=1
   Fields: token=<token>&_xsrf=<xsrf>&t=<refTypeId>
   Returns: { id: <reqId>, obj: <sourceTableId> }
   ```

   В записях (`_m_set`/`_m_new`) ref-поле адресуется по ключу **`t<reqId>`**, где
   `reqId` — это `id` из ответа шага 6.2, а значение — id записи в целевой таблице.
   В metadata реквизит выходит под тем же ключом `t<reqId>` с полем `ref` (= id
   целевой таблицы). Двухшаговый путь гарантирует, что возвращённый `id` и ключ
   записи совпадают.

   В параметр `t` подставляй **ref-тип** из шага 6.1, а не id целевой таблицы.
   От содержимого `t` зависит тип связи:

   | В `t` подставлено | Что создаётся | Ключ записи / metadata |
   |-------------------|---------------|------------------------|
   | ref-тип (после `_d_ref`) | **ссылка** (FK) | `t<reqId>`, поле `ref` |
   | id дочерней таблицы | **подчинённая таблица** | `t<childTableId>`, поле `arr_id` |

   Форму `_d_req/{source} t=<childTableId>` (id таблицы прямо в `t`) применяй,
   когда по правилам [2.5](#25-два-типа-связей-ссылка-справочник-и-подчинённая-таблица)
   нужна **подчинённая таблица**; полный паттерн с `up=` — в [3.4](#34-подчинённые-таблицы).
   В этой форме реквизит адресуется ключом `t<childTableId>`, а не `t<id-из-ответа>`:
   проверяй ключ записи по `arr_id` в metadata.

### 3.2.1 Идемпотентность `_d_new` и `_d_req`

**`_d_new` дедуплицирует имена типов глобально по базе.** Повторный вызов
`POST _d_new t=3 val=Статус` возвращает существующий `obj` с предупреждением
`"Тип Статус уже существует!"` — никакого дубликата не создаётся. Это безопасно
перевызывать; для проверки «есть ли такой тип» можно использовать как зонд:

```bash
# Сессия 2026-05-29 на /ateh/, попытка создать тип "Статус", который уже есть:
api_call _d_new "t=3" "val=Статус"
# Ответ: {"id":"","obj":"409","warnings":"Тип Статус уже существует!"}
# → глобальный type id для "Статус" — 409, переиспользуем.
```

**`_d_req` идемпотентен per (table, type).** Если тип уже привязан к таблице,
повторный вызов вернёт **существующий reqId** с предупреждением
`"Реквизит ... уже существует!"`. Никакого нового реквизита не создаётся.

### 3.3 Порядок создания таблиц

**Важно:** таблицы создаются в порядке зависимостей:
1. Сначала независимые справочники (Сырьё, Слиттер)
2. Потом зависимые справочники (Тип резки — ссылается на Сырьё)
3. Основные таблицы (Заказ)
4. Дочерние таблицы (Позиция заказа)
5. Таблицы событий (Производственная резка — ссылается на всё)

Ссылки (ref) добавляются после создания ВСЕХ таблиц.

### 3.4 Подчинённые таблицы

> Применяй этот паттерн, только когда связь по правилам
> [2.5](#25-два-типа-связей-ссылка-справочник-и-подчинённая-таблица) — это
> **подчинённая таблица** (набор записей, принадлежащих одному родителю). Если
> связь — **ссылка** на переиспользуемый справочник, используй двухшаговый
> `_d_ref` + `_d_req` из [3.2 шаг 6](#32-последовательность-api-вызовов).

Дочерняя таблица становится подчинённой только после того, как её тип добавлен
реквизитом в родительскую таблицу. Одного `up=<parentRecordId>` при создании
записи недостаточно: без родительского реквизита таблица остаётся отдельной, а
создание дочерней записи может завершиться ошибкой «нет доступа к реквизиту».

Минимальная последовательность:

```bash
# 1. Создать родительскую таблицу
orderTableId=$(api_call "_d_new" "t=3&val=$(urlencode 'Заказ')" | jq -r .obj)

# 2. Создать дочерний тип
positionTableId=$(api_call "_d_new" "t=3&val=$(urlencode 'Позиция заказа')" | jq -r .obj)

# 3. Обязательно добавить дочерний тип как реквизит родителя
positionReqId=$(api_call "_d_req/${orderTableId}" "t=${positionTableId}" | jq -r .id)
api_call "_d_alias/${positionReqId}" "val=$(urlencode 'Позиция заказа')"

# 4. Добавить поля дочерней таблицы
qtyTypeId=$(api_call "_d_new" "t=13&val=$(urlencode 'Количество рулонов')" | jq -r .obj)
qtyReqId=$(api_call "_d_req/${positionTableId}" "t=${qtyTypeId}" | jq -r .id)
api_call "_d_alias/${qtyReqId}" "val=$(urlencode 'Количество рулонов')"

# 5. Создавать записи дочерней таблицы с up=id родительской записи
api_call "_m_new/${positionTableId}?full=1" "up=${orderRecordId}&t${positionTableId}=1&t${qtyReqId}=351"
```

Проверка через `metadata`: у родительской таблицы в `reqs[]` должен быть реквизит,
у которого `orig` равен ID дочернего типа, **и присутствует поле `arr_id`** (= ID
дочернего типа), а поля `ref` нет. Наличие `arr_id` и отсутствие `ref` — признак
именно подчинённой таблицы (а не ссылки; см.
[2.5.4](#254-как-тип-связи-виден-в-метаданных-самопроверка)). Если дочерний тип уже
создан, но не подключён к родителю, ремонт структуры — это `_d_req/{parentTableId}`
с `t={childTableId}`, а не изменение прав роли в `t117`.

### 3.5 Чтение существующих записей и метаданных

После создания структуры (и в любой момент позже) данные читаются GET-запросами;
авторизация — через заголовок `X-Authorization: <token>` или cookie `idb_{db}`
(см. Этап инициации). Строку запроса `?token=...` сервер игнорирует (issue #3000).

**Метаданные базы — список всех таблиц и их колонок:**
```bash
GET /{db}/metadata?JSON=1
# Возвращает массив объектов: [{id, val, granted, reqs:[{id, val, orig, type, ref, attrs}, ...]}, ...]
# Здесь val у каждого reqs[i] — имя типа колонки, orig — id типа (нужен для _d_save), attrs может содержать {"alias":"..."}.
```

> Полную расшифровку каждого поля ответа `metadata` (как отличать ссылку от
> подчинённой таблицы, как читать `attrs`, алиасы и значения по умолчанию) см. в
> справочнике [«Метаданные (`metadata`)»](#метаданные-metadata--справочник).

**Список записей конкретной таблицы:**
```bash
GET /{db}/object/{typeId}/?JSON_KV&LIMIT=100&pg=1
# Записи лежат в ответе под ключом "object" (массив объектов).
# LIMIT — размер страницы (по умолчанию небольшой), pg — номер страницы (1-based).
# Между страницами возможны дубликаты — дедупликация по id рекомендуется при пагинации.
```

**Полезные фильтры/параметры:**

| Параметр | Назначение |
|----------|-----------|
| `F_U={parentId}` | только записи с `up=<parentId>` (например, меню конкретной роли: `object/151/?F_U=145`) |
| `F_I={recordId}` | только запись с `vals.id=<recordId>` (например, карточка записи: `object/{typeId}/?F_I=123`) |
| `_count=`        | вернуть только число записей без выгрузки данных: `?JSON_KV&_count=` |
| `LIMIT=N`        | страница до N записей |
| `pg=N`           | номер страницы (1-based) |

> ⚠️ **Подчинённую таблицу без `F_U` не проверить на наличие данных.** Записи
> подчинённой таблицы — дочерние (`u`/`up` = ID родителя). Запрос
> `object/{subTableId}?…` **без `F_U`** (как и `_count=` без `F_U`) возвращает
> только записи уровня `up=1` и для подчинённой таблицы выдаёт **`0`, даже когда
> данные есть**. Поэтому наличие данных в подчинённой таблице проверяется **по
> родителям**: перебрать записи родителя и для каждой запросить
> `object/{subTableId}?…&F_U=<parentId>`. Этот ноль легко принять за «таблица
> пустая» — что опасно перед удалением колонки (см.
> [3.7](#37-изменение-поля-простое--справочное-и-удаление-колонок)).

Тип таблицы выбирается сегментом пути `object/{typeId}`. Параметра для выбора типа
в `object/` не существует: в коде фильтры `F_*` разбираются как
фильтры по колонкам или специальные параметры `F_U`/`F_I`.

#### Обращение к таблице по имени (вместо ID)

К таблице (и к отдельной записи) можно обращаться **по имени, не зная числового
ID**. Это удобно при создании структур: имя известно заранее из схемы, а ID
выдаёт сервер только после `_d_new`, и его пришлось бы сначала вычитывать из
ответа или из `metadata`. По имени URL читается без этого шага:

```bash
# По числовому ID (как выше)
GET /{db}/object/470/?JSON_KV
GET /{db}/table/470?JSON_OBJ

# То же по имени таблицы — ID знать не нужно
GET /{db}/object/Заказ/?JSON_KV
GET /{db}/table/Заказ?JSON_OBJ
```

Сегмент действия (`object`, `table`) обрабатывается одинаково: если следующий
сегмент пути **не число**, сервер ищет таблицу по её названию запросом
`SELECT id FROM {db} WHERE up=0 AND val='<имя>' ORDER BY id` и дальше
работает с найденным ID. Числовой сегмент используется как ID напрямую.

**Запись внутри таблицы — тоже по имени.** Четвёртый сегмент пути адресует
конкретную запись по значению её первой колонки:

```bash
# Запись «Заказ №100» в таблице «Заказ» без числовых ID
GET /{db}/table/Заказ/Заказ №100?JSON_OBJ
```

**Имя нужно URL-кодировать**, если в нём есть пробелы или не-ASCII символы
(сервер применяет `urldecode`): `table/%D0%97%D0%B0%D0%BA%D0%B0%D0%B7` либо
`encodeURIComponent('Заказ')` из JS.

> ⚠️ **Предупреждение: одноимённые таблицы.** Поиск по имени возвращает **самую старую**
> подходящую запись — в SQL-запросе `ORDER BY id LIMIT 1`, а сравнение
> строк в MySQL по умолчанию регистронезависимо. Поэтому если существуют две
> таблицы с одинаковым именем, но разным базовым типом первой колонки (например,
> `Заказ` как строка SHORT и `заказ` как число NUMBER), Интеграм вернёт ту, у которой id меньше.
> Избегайте таких ситуаций: давайте таблицам уникальные имена
> (см. правила [2.1](#21-правила-проектирования-таблиц-из-structurerulesmd)) либо
> обращайтесь по числовому ID, когда нужна гарантированная однозначность.

То же обращение по имени работает и для запросов (`report/{имяЗапроса}`) — см.
[раздел «Запросы (report/)»](#вызов-запроса).

**Список доступа (используется в редакторе ролей):**
```bash
GET /{db}/grants?JSON=1
# Возвращает массив {id, name} — отображаемые названия всех таблиц и колонок,
# которые можно раздать роли. Полезно для самопроверки переименований:
# если name содержит JSON-блоб {"alias":...} — где-то сработал _d_alias на таблице.
```

> **Перечень всех отчётов (`report/`)** доступен через API командой
> `GET object/22?JSON_OBJ` (или `GET object/Запрос?JSON_OBJ`). «Запрос» (тип 22) —
> это обычная таблица Интеграма, но ядро интерпретирует её особо: умеет собирать
> колонки в запрос и строить полноценный SQL, включая вложенные запросы и рекурсию.
> Команда возвращает все записи этой таблицы — то есть список всех отчётов с их
> ID и именами. При необходимости сохраняй ID отчётов в самом скрипте сборки или
> забирай их из заголовка `X-Query-Id` при первом вызове по имени (см. раздел
> «Запросы (report/) — справочник»).

### 3.6 Переименование существующих типов и таблиц

`_d_alias` ставит **alias на конкретный реквизит** (per-occurrence) — это
полезно при создании новой колонки или для псевдонимов ref-колонок. Чтобы
переименовать **сам тип** (то есть имя, которое отображается везде, где этот
тип используется), нужен `_d_save`.

**Переименование таблицы:**
```bash
POST /{db}/_d_save/{tableId}?JSON
Fields: val="Новое имя", t=3
# t=3 — базовый код типа SHORT (для таблиц с первой колонкой SHORT)
# Для таблиц с первой колонкой DATETIME — t=4, и т.д.
# Меняет val самой таблицы; во всех ref-колонках и в metadata имя обновится.
```

**Переименование типа колонки** (например, «Статус слиттера» → «Статус станка»):
```bash
POST /{db}/_d_save/{origId}?JSON
Fields: val="Новое имя", t=<baseTypeCode>
# ВАЖНО: origId — это reqs[i].orig из metadata (id типа), НЕ reqs[i].id (id реквизита-в-таблице).
# baseTypeCode — это reqs[i].type (3 для SHORT, 12 для MEMO, 13 для NUMBER и т.д.),
# а не сам origId.
# Если вызвать с reqs[i].id вместо .orig — сервер вернёт {"error":"Тип не найден."}.
```

**Когда брать `_d_alias`, а когда `_d_save`:**

| Случай | Команда |
|--------|---------|
| Новая колонка только что добавлена через `_d_req`, хочу дать ей имя | `_d_alias/{reqId}` (как в `create_perelidoz.ps1`) |
| Уже созданной ref-колонке нужен псевдоним в этой конкретной таблице | `_d_alias/{reqId}` |
| Переименовать существующую обычную колонку (затрагивает все её использования) | `_d_save/{origId}` с `val=…&t=baseType` |
| Переименовать таблицу | `_d_save/{tableId}` с `val=…&t=3` (или иной базовый код) |

### 3.7 Изменение поля: простое → справочное и удаление колонок

Поправки к готовому приложению («сделать это поле справочным», «убрать колонку»)
отрабатываются по типовым кейсам в
[`typical-change-requests.md`](typical-change-requests.md). Здесь — опорные правила.

**Удаление колонки:**

```bash
POST /{db}/_d_del_req/{reqId}?JSON=1
Fields: forced=1
```

> ⚠️ **Удаление колонки безвозвратно уничтожает её значения у всех записей.** Перед
> удалением **считай и сохрани значения** для всех записей (для подчинённой таблицы
> — по родителям через `F_U`, см. предупреждение в [3.5](#35-чтение-существующих-записей-и-метаданных)).
> Не полагайся на `object/{tbl}` без `F_U`: для подчинённой таблицы он покажет `0`,
> и колонку удалишь «как из пустой», потеряв данные.

**Удаление таблицы (`_d_del/{tableId}` — только когда на тип НЕТ ссылок и экземпляров):**

Главный принцип: **удаляя термин/тип, СНАЧАЛА сними ВСЕ референсы на него, потом сам тип.**
Сервер блокирует `_d_del/{tableId}` сообщениями «нельзя удалить тип при наличии его
экземпляров» или «используется в ролях/отчётах». Полный чек-лист зачистки перед дропом:

1. **Ref-тип бывшего справочника.** Если таблицу когда-то использовали как справочник
   (на неё ссылались), для неё командой `_d_ref` был создан **особый ref-тип «ссылка на
   таблицу»**. Он остаётся в системе и блокирует удаление (выглядит как «1 экземпляр»).
   Его ID — в метаданных таблицы как **`referenced`** (поле `referenced` в `?JSON_META`).
   Удалить: `_d_del/{referencedId}`. (См. также §«поле `referenced`», issue #2746.)
2. **Гранты ролей.** Если тип/колонка выданы ролям — снять гранты. Роли с грантом на
   таблицу: `object/42/?JSON_OBJ&F_116={tableId}`. Сами грант-записи (таблица 116
   «Объекты», подчинены роли) — `object/116/?JSON_OBJ&F_U={roleId}`, у нужных первая
   колонка `{tableId}:Имя` (напр. `1072:Тип резки`); удалить каждую `_m_del/{grantId}`.
3. **Колонки отчётов.** Если реквизит/таблица используются в запросах — удалить эти
   колонки: отчёты с реквизитом — `object/22/?JSON_OBJ&F_28={reqId}`; колонки запроса —
   `object/28/?JSON_OBJ&F_U={queryId}`; удалить `_m_del/{colId}`.
4. **Ссылки-реквизиты из других таблиц** на этот тип — снять `_d_del_req/{reqId} forced=1`.
5. **Экземпляры (записи)** типа и его подчинённых — удалить (`_m_del`); помнить, что
   подчинённые дети каскадятся с родителем, а входящие ссылки «в использовании» удалить
   нельзя, пока не убран ссылающийся объект.
6. Теперь **`_d_del/{tableId}`** пройдёт. Проверить: `object/{tableId}/?JSON_META` → «не найден».

> ⚠️ Порядок важен: если снять ссылочный реквизит (`_d_del_req`) РАНЬШЕ, чем удалить
> записи-владельцы, дочерние экземпляры этого типа осиротеют (станут недостижимы и
> неудаляемы), и тип будет не удалить. Сначала записи, потом реквизиты, потом тип.
> И всегда — **полный бэкап БД до необратимых операций**.

**Миграция «простое поле → справочное» (поле ввода → выбор из справочника):**

1. **Считать и сохранить** текущие значения старого поля для всех записей (по
   родителям) — это надо сделать **до** удаления.
2. **Создать справочник** (`_d_new t=3 val=…`) и **заполнить** его всеми
   вариантами, которыми было заполнено поле (`_m_new/{dictId} up=1 t{dictId}=…`).
3. **Создать ссылочный тип** на справочник (`_d_ref/{dictId}`).
4. **Удалить старую простую колонку** (`_d_del_req/{oldReqId} forced=1`).
5. **Добавить ссылочную колонку** (`_d_req/{tableId} t={refType}`) и задать ей то же
   имя (`_d_alias`).
6. **Восстановить значения как ссылки**: для каждой записи `_m_set/{objId}
   t{newReqId}=<id записи справочника>`.
7. **Выдать ролям доступ к справочнику** — иначе у роли выпадающий список будет
   пустой и ссылочное значение не отобразится (см.
   [5.9.1](#591-запись-прав-напрямую-в-таблицу-объекты-116); новый объект по
   умолчанию недоступен ролям). Уровень — обычно `READ` (`53`).
8. **Обновить форму рабочего места**: поле рендерить поисковым комбобоксом из
   `_ref_reqs/{newReqId}` (см. [4.3.1](#431-поля-выбора-в-сгенерированных-рабочих-местах)),
   значение брать из выбранной записи справочника.

После: реквизит в `metadata` имеет `ref=<dictId>` вместо простого типа.

## Этап 4: Разработка шаблонов (HTML/JS)

### 4.1 Визуальная составляющая и корпоративный стиль клиента

**Визуальная составляющая приложения важна для клиента не меньше, чем
функциональность.** Рабочие места — это «лицо» приложения, по которому клиент
первым делом судит о качестве работы. Аккуратный, узнаваемый интерфейс повышает
доверие, поэтому визуальное оформление нельзя оставлять «на потом» или делать
случайным.

**Главное правило: если у клиента есть корпоративный стиль (бренд) — применяй
его.** Не выдумывай собственную палитру, когда у компании уже есть фирменная.
Пример постановки такой задачи — [issue #2974](https://github.com/ideav/crm/issues/2974)
(«Применить корпоративный стиль и лого для рабочих мест по образцу с сайта
клиента»): цвета и стилистика рабочих мест должны быть похожи на сайт заказчика.

**Что относится к корпоративному стилю:**
- **логотип** — файл или изображение с сайта клиента;
- **фирменные цвета** — основной, акцентный, цвет фона и текста;
- **шрифты** — если бренд использует особые гарнитуры;
- **общая стилистика** — тон оформления сайта компании, существующих приложений,
  бланков, презентаций.

**Где взять стиль:**
- спроси клиента на [Этапе постановки задачи](#этап-постановки-задачи-анализ-предметной-области-и-генерация-пользовательского-пути)
  — есть ли сайт, логотип, брендбук или фирменные цвета;
- если есть сайт — возьми за образец его цвета, шрифты и логотип (например, со
  страницы «О компании»);
- если фирменного стиля нет — применяй нейтральное аккуратное оформление по
  умолчанию, а не случайный «яркий» дизайн.

**Как применять:**
- выноси фирменные цвета в CSS-переменные (`:root { --brand-color: …; }`), чтобы
  менять палитру в одном месте и переиспользовать на всех экранах;
- **бренд (логотип и цвета) размещай в верхнем меню приложения — шапке шаблона
  `templates/<имя базы>/main.html` (`.navbar`), заменяя им лого и стиль
  Интеграма;** если такого файла ещё нет, сначала скопируй общий
  `templates/main.html`, потом применяй брендинг в копии; не вкладывай отдельную
  бренд-шапку внутрь рабочих мест (иначе над контентом оказываются две шапки —
  верхнее меню Интеграма и продублированная под ним бренд-полоса);
- держи **единый стиль на всех экранах** приложения — одинаковые цвета, отступы,
  кнопки и шрифты, чтобы интерфейс выглядел цельным.

#### 4.1.1 Бренд заменяет верхнее меню (а не вкладывается в рабочие места)

`templates/main.html` — общий каркас-обёртка для баз без собственной шапки (см.
[ASSETS_DEPLOYMENT.md](ASSETS_DEPLOYMENT.md)): в нём верхнее меню (`.navbar`) с
логотипом и левое меню, а рабочее место подставляется в `<main
class="app-content">` на месте маркера `<!-- File: a -->`.

Для брендированной базы верхнее меню настраивается в её копии каркаса:

1. Если файла `templates/<имя базы>/main.html` нет, создай его копированием из
   `templates/main.html`. В продакшене `Get_file("main.html")` сначала ищет
   `templates/custom/<имя базы>/main.html`, а уже потом общий `templates/main.html`;
   в репозитории такая копия хранится как `templates/<имя базы>/main.html` и
   деплоится в `templates/custom/<имя базы>/` через `update.conf`.
2. В `<head>` этой копии подключи фирменный стиль базы после общих стилей
   оболочки:
   ```html
   <link rel="stylesheet" href="/download/{_global_.z}/css/brand.css?0{_global_.version}">
   ```
3. `download/<база>/css/brand.css` переопределяет верхнее меню: фон и цвет
   `.navbar`, подменяет логотип (`.navbar-brand .logo { content: url(<лого>) }`),
   при необходимости скрывает имя базы; палитру берёт из файла палитры бренда.
4. Рабочие места подключают только палитру бренда и свой CSS — их цветовая гамма
   остаётся клиентской, но **без** собственной бренд-шапки.

Реализация для базы `atex`: верхнее меню перекрашено в фирменный тёмно-синий и
показывает логотип «АТЕХ»
([templates/atex/main.html](../templates/atex/main.html),
[download/atex/css/brand.css](../download/atex/css/brand.css)); палитра рабочих
мест — в [download/atex/css/atex-brand.css](../download/atex/css/atex-brand.css);
рабочие места `templates/atex/*.html` не содержат вложенной бренд-шапки.

#### 4.1.2 Контраст брендированной шапки — обязательный механизм проверки

**Зачем это правило.** Когда бренд перекрашивает верхнее меню (`.navbar`) в тёмный
фирменный цвет и делает текст и иконки светлыми, кнопки внутри шапки, у которых есть
**собственная поверхность** из общей светлой темы, остаются светлыми. Светлый значок
на светлой поверхности (или, наоборот, тёмный текст на тёмном фоне) — нечитаем. Так
возник дефект [#3008](https://github.com/ideav/crm/issues/3008): иконка ИИ-чата и
подпись `Admin / Admin` стали белыми на светлом фоне (исправлено в
[#3009](https://github.com/ideav/crm/pull/3009)). Это **повторяющийся класс
дефектов**: точечная правка одной кнопки не мешает тому же дефекту вернуться на
следующей кнопке или в следующем бренде. Поэтому механизм проверки описан здесь и
закреплён автоматическим тестом.

**Главное правило (инвариант).** Если бренд перекрашивает `.navbar`, то для **каждого
интерактивного элемента шапки, имеющего собственную поверхность**, бренд обязан задать
**и фон, и контрастный цвет текста/иконки вместе** — и в покое, и в состоянии
наведения/раскрытия (`:hover`, `.open`). Никогда не перекрашивай текст, не перекрасив
его поверхность, и наоборот. Цвет в одиночку не спасает: общий `:hover` подставит
светлый токен темы (`--card-bg`, `--border-color`, `--bg-secondary`) обратно.

**Инвентарь элементов шапки с собственной поверхностью** (держи список синхронным с
`experiments/brand-navbar-contrast.test.js`):

| Элемент | Поверхность в общей оболочке | Что задаёт бренд |
|---------|------------------------------|------------------|
| `.navbar .ai-chat-toggle` | покой `var(--bg-secondary)`, hover `var(--card-bg)` | фирменный фон + цвет, покой и hover |
| `.navbar .account-info` | hover `var(--border-color)` | фирменный фон + цвет, покой и hover |

Прочие элементы шапки (`.navbar-workspace`, `.account-email`, `.user-menu-arrow`,
`.mobile-sidebar-toggle`, `.account-avatar`) поверхности не несут — им достаточно
задать контрастный цвет (`color: var(--<бренд>-on-navy)`), как уже сделано в
[download/atex/css/brand.css](../download/atex/css/brand.css). При появлении в общей
оболочке новой кнопки шапки с собственным фоном добавь её и в таблицу выше, и в тест.

**Как проверять (два уровня, оба обязательны):**

1. **Автоматический регрессионный тест.** Запусти
   `node experiments/brand-navbar-contrast.test.js`. Тест сам находит все бренды
   (`download/<бренд>/css/brand.css`), отбирает те, что перекрашивают `.navbar`, и
   требует для каждого элемента из инвентаря фон **и** цвет в покое **и** на hover.
   Тест не привязан к конкретному бренду — он защищает и `atex`, и любой будущий
   бренд автоматически. Если добавляешь бренд с тёмной шапкой — этот тест уже его
   проверяет; ничего отдельного писать не нужно.
2. **Визуальная проверка скриншотом.** Перед мерджем сними рабочее место в реальном
   рендере (Playwright или браузер) на **десктопе и мобильном** вьюпорте и в
   **светлой, и в тёмной** теме; убедись, что все подписи и иконки шапки читаются.
   Приложи скриншот «после» к pull request (см. правила о скриншотах в описании PR).

**Чек-лист брендирования шапки (добавь в acceptance-критерии PR):**

- [ ] `.navbar` перекрашен фирменным фоном, лого заменено, имя базы при необходимости скрыто.
- [ ] Каждый элемент из инвентаря выше получил фирменный фон **и** контрастный цвет — в покое и на hover.
- [ ] Текстовые элементы шапки без собственной поверхности получили контрастный `color`.
- [ ] `node experiments/brand-navbar-contrast.test.js` проходит.
- [ ] Снят и приложен к PR скриншот шапки (десктоп + мобильный, светлая + тёмная тема), подписи читаются.

### 4.2 Структура шаблона

Каждый шаблон = HTML-файл в `templates/` со встроенными стилями и подключением JS:

```html
<style>
/* Локальные стили */
.my-panel { ... }
</style>

<div class="my-panel">
    <!-- HTML интерфейса -->
</div>

<script>
window.SOME_TYPE_ID = 469;  // ID таблицы
</script>
<script src="/download/{_global_.z}/api.js?1"></script>
<script src="/download/{_global_.z}/tables.js?1"></script>
<script src="/download/{_global_.z}/my-panel.js?1"></script>
```

**Важно:** `{_global_.z}` — шаблонная переменная Интеграм, заменяется на имя базы данных при рендеринге.

#### 4.2.1 Поддержка светлой и тёмной темы в рабочих местах

Каждое сгенерированное рабочее место обязано опираться на токены темы из
`templates/<имя базы>/main.html` (или общего `templates/main.html`), потому что
оно рендерится внутри этой оболочки и наследует `data-theme` с корневого
элемента. Не закрепляй в CSS рабочих мест светлые поверхности вроде `#fff`,
`#f8fafc`, `#f1f5f9` как финальные значения для таблиц, вложенных панелей,
hover-состояний, форм и сообщений.

Используй общие переменные оболочки:

| Зона интерфейса | Токены темы |
|-----------------|-------------|
| Фон рабочего места | `var(--bg-primary)` |
| Вторичная поверхность, шапки таблиц, hover | `var(--bg-secondary)` |
| Карточки, поля ввода, селекты | `var(--input-bg)`, `var(--input-border)` |
| Основной и вторичный текст | `var(--text-primary)`, `var(--text-secondary)` |
| Границы | `var(--border-color)` или `var(--input-border)` |
| Успех/ошибка | `var(--color-success*)`, `var(--color-error*)` |

Если рабочему месту нужен собственный брендовый или информационный цвет, задай
локальную CSS-переменную и при необходимости отдельное значение для
`[data-theme="dark"]`, но не переопределяй глобальные токены темы. Перед PR
проверяй рабочее место в светлой и тёмной теме; для Atex такой дефект был
зафиксирован в `templates/atex/orders.html`, где автоматически сгенерированные
таблицы позиций оставались светлыми внутри тёмной оболочки (issue #3039).

### 4.3 JS-файлы

**api.js** — обёртка над API Интеграм:
```javascript
// createRecord(tableId, fields, parentId)
Api.createRecord(469, { 'Тип сырья': 'Бумага', 'Ширина джамбо, мм': 1020 });

// queryTable(typeId, where, sort, limit)
Api.queryTable(469, { 'Статус': 'Новый' }, '-id', 20);

// updateRecord(objId, fields)
Api.updateRecord(571, { 'Остаток, м': 4500 });
```

**tables.js** — общие функции для работы с таблицами (переиспользуются всеми панелями).

**Панельные JS** — логика конкретного экрана (order-entry.js, cutter-panel.js и т.д.).

### 4.3.1 Поля выбора в сгенерированных рабочих местах

Любой сгенерированный экран, где пользователь выбирает значение из справочника
или другого списка, который может вырасти больше 10-20 пунктов, должен рендерить
не нативный `<select>`, а **поисковый комбобокс**:

- видимый элемент — текстовое поле с `role="combobox"`, `autocomplete="off"` и
  выпадающим списком вариантов;
- выбранный id хранится в скрытом `<input type="hidden">` с тем же `id`/`name`,
  который используется при отправке формы;
- ввод в поле сразу фильтрует уже загруженные варианты на клиенте;
- для ref-полей поиск дополнительно обращается к серверу:
  `_ref_reqs/{reqId}?JSON&LIMIT=50&q=<текст>`, чтобы находить записи за пределами
  начального кэша;
- короткие фиксированные перечисления (например 3-5 статусов) могут оставаться
  обычным `<select>`, если список не растёт из данных пользователя.

Это правило применяется ко всем новым рабочим местам, а не только к общим формам
`IntegramTable`: длинные списки клиентов, сырья, типов резки, партий, станков и
ответственных должны искаться по видимому значению.

### 4.3.2 Счётчик дочерних записей в списках — из родителя

Когда в списке родителей надо показать число подчинённых записей (например
«позиций в заказе»), бери его **из записи родителя**: подчинённая таблица даёт
родителю ROLLUP-колонку-счётчик (в `metadata` у реквизита есть `arr_id`), её
значение в `r[]` — это количество дочерних записей.

Не считай количество по лениво-загружаемым детям (которые подтягиваются только
при раскрытии строки) — иначе у всех нераскрытых родителей счётчик будет `0`.
Пример: `download/atex/js/orders.js`, колонка `posCount` из реквизита «Позиция
заказа» (см. также `WORKSPACE_DEVELOPMENT_GUIDE.md` §6.1).

### 4.4 Загрузка шаблонов на сервер

Шаблоны загружаются в папку `download` базы данных через веб-интерфейс или API:

```bash

  download=1
  add_path=
  upload=Загрузить
  userfile=@file.js
```

После загрузки файлы доступны по URL: `https://ideav.ru/download/{db}/file.js`

- Справочные и длинные пользовательские списки выбора сделаны как поисковые

## Этап 5: Роли, права, пользователи, меню и запросы

### 5.1 Системные таблицы

В Интеграме изначально есть четыре системные таблицы:
- **Роль** (таблица 42, type=3) — название роли, описание, права
- **Пользователь** (таблица 18, type=3) — логин, роль (ref на Роль), пароль, email, телефон
- **Меню** (таблица 151, type=3) — пункты меню ролей (адрес, иконка, параметры)
- **Запрос** (таблица 22, type=3) — инструкции для выборки данных из таблиц этой базы данных

Интеграм знает как применять описание роли к поведению пользователя и как обрабатывать запросы для выборки данных - это зашито в ядре.
В целом, системные таблицы ничем не отличаются от пользователей технически, но если в них что-то удалить или изменить, то, скорее всего, система станет неработоспособна.
Добавлять колонки в служебные таблицы не запрещается, например, практикуется сопроводить запросы полем описания и галкой — пользовательский это отчёт или администраторский.

> **Меню связано с ролью через `up` - это подчиненная таблица.** У записи Роли
> пункты меню создаются как **дочерние записи Роли**
> через `_m_new/151` с `up=<role_id>`. Чтобы получить пункты меню конкретной
> роли, используется `GET /{db}/object/151/?F_U=<role_id>` (фильтр по
> родителю). Подробнее в 5.4 ниже.
>
> Колонки Роли (таблица 42): `t117` (GRANT «Объекты», права на объекты),
> `t135` (MEMO, описание) и колонка Меню. Реквизит «Объекты» в метаданных Роли
> выглядит как `{"id":"117","val":"Объекты","orig":"116"}`: `117` — id колонки,
> `orig` `116` — id подчинённой таблицы «Объекты», в которой хранятся права (по
> одной записи на объект базы, выбираются по `up=<role_id>`). Актуальные id
> сверяются через `GET /{db}/metadata?JSON=1` → `reqs[]` таблицы 42 (поля `id`
> и `orig`).

### 5.2 Паттерн `t{column_id}=value`

Все значения передаются как `t{id_колонки}=value` — это единственный формат для всех таблиц (системных и обычных):

```bash
# Роль (таблица 42): t42=название, t135=описание
POST /{db}/_m_new/42?JSON=1
Fields: up=1, t42=менеджер, t135=Управление заказами и планирование

# Пользователь (таблица 18): t18=логин, t115=role_id, t20=пароль
POST /{db}/_m_new/18?JSON=1
Fields: up=1, t18=manager, t115=632, t20=slitter123

# Меню (таблица 151): t151=название, t153=адрес, t391=иконка
POST /{db}/_m_new/151?JSON=1
Fields: up=<role_id>, t151=Приём заказов, t153=order-entry
Icon: --form-string 't391=<i class="pi pi-file-edit"></i>'
# ВАЖНО: t391 содержит HTML-тег <i> с угловыми скобками и кавычками —
# использовать --form-string, иначе curl -F ломает значение.
```

### 5.3 Создание ролей

Типичные роли для производства:

### 5.4 Создание пунктов меню

Пункты меню создаются как дочерние записи роли (`up=<role_id>`):


Иконки — CSS-классы PrimeIcons (https://primefaces.org/primeicons/):
- `pi pi-file-edit` — редактирование
- `pi pi-calendar` — календарь/планирование
- `pi pi-chart-bar` — графики/отчёты
- `pi pi-sliders-h` — настройки
- `pi pi-arrow-circle-down` — загрузка
- `pi pi-circle` — метка

### 5.5 Создание пользователей

**Минимальный набор полей** — `t18` (логин), `t115` (роль, mandatory), `t20`
(пароль). Опционально — `t33` (ФИО для отображения):

```bash
POST /{db}/_m_new/18?JSON=1
Fields: up=1, t18=manager, t115=<role_id>, t20=mgr_demo_2026, t33=Менеджер
```

Остальные поля (Email t41, Телефон t30, Примечание t39, Дата t156 и т.д.) —
по необходимости. **Не задавай вручную** служебные `Activity` (t124), `Token`
(t125), `xsrf` (t40), `Retries` (t301) — их заполняет сервер.

Пароль (`t20`) хешируется сервером автоматически. Проверка входа тестового
пользователя делается вручную через веб-форму входа; автоматический агент не
вызывает `/auth` и продолжает работать под выданным ему токеном.

### 5.6 Права роли на таблицы и запросы

Роль раздаёт права на таблицы, редактор структуры и файловый менеджер. Также роль определяет **пункты меню** для пользователя (см. 5.4) — ссылки на рабочие места. **Права на таблицы** определяют, к каким данным роль имеет доступ и на каком уровне.

**Колонка Объекты (`t117`).** У роли (таблица 42) есть колонка **Объекты** —
`t117`, тип `5` (`GRANT`). В неё записываются ID объектов базы — таблиц — каждому со своим уровнем доступа. 

Таблицы идентифицируются по своим ID, но также есть синтетические ID, они дают соответствующие доступы:
1. 0 - Type editor - редактор структуры, просмотр (READ) и команды _d_* (WRITE)
2. 1 - All objects - все таблицы просмотр (READ) и команды _m_* (WRITE)
3. 10 - Files - файловый менеджер, операции с файлам шаблонов и скриптов

ID объекта выбирается из выпадающего списка, который сервер отдаёт
по `GET /{db}/grants` (issue #601; список кэшируется — issue #607). Поэтому `t117`
заполняется не произвольным текстом, а выбором уровня из этого списка — как любое
`GRANT`-поле в интерфейсе.

**Уровни доступа** (источник: [`05-access-rights.md`](integram-article-reviews/05-access-rights.md)):

| Уровень | Что разрешает |
|---------|---------------|
| `WRITE` | создание, редактирование, удаление записей |
| `READ` | только чтение |
| `BARRED` | полный запрет — объект/поле не возвращается даже в API-ответах |

Уровни наследуются от таблицы к её колонкам: задав роли `WRITE` на таблицу,
получаем право писать во все её поля, а отдельным полям при необходимости уровень
понижается (например `BARRED` на финансовые поля). Независимыми флагами
настраиваются `EXPORT` (выгрузка) и `DELETE` (удаление) — можно разрешить
редактирование, но запретить массовую выгрузку или удаление.

**Как права видны в рабочем месте.** Эффективные права роли приходят в шаблон в
переменной `{_global_.grants}` — это base64-JSON вида `{ "<typeId>": "WRITE", … }`:

```js
// templates/main.html:138
grants = JSON.parse(atob('{_global_.grants}'));
if (window.grants['1'] === 'WRITE') { /* есть право на запись */ }
```

В метаданных таблицы (`GET /{db}/metadata/{typeId}`) и в колонках отчёта каждая
колонка несёт поле `granted`: значение `WRITE` — поле редактируемое, иное — только
чтение (issue #1508). Компонент `IntegramTable` сам переключает форму в режим
«только чтение», если у роли нет `WRITE` (см.
[`WORKSPACE_DEVELOPMENT_GUIDE.md`](WORKSPACE_DEVELOPMENT_GUIDE.md) разделы 1, 6).

**Права на запросы (`report/`).** Запрос — такой же объект в списке Объекты роли,
как и таблица. Уровень доступа определяет, может ли роль открыть отчёт. Маска роли
(см. 5.7) применяется к отчётам так же, как к таблицам и API, — отдельно
настраивать фильтрацию в каждом отчёте не нужно.

### 5.7 Маска доступа

Маска — это условие фильтрации на уровне роли, ограничивающее, **какие записи**
видит пользователь (в отличие от уровня доступа, который ограничивает, **какие
объекты и поля**). Маска применяется на стороне сервера ко всем запросам —
таблицам, отчётам и API, — поэтому обойти её через экспорт нельзя (источники:
[`05-access-rights.md`](integram-article-reviews/05-access-rights.md),
`help.integram.io/#rec220708794`).

**Подстановки в маске:**

| Подстановка | Значение |
|-------------|----------|
| `[CURRENT_USER]` | текущий пользователь |
| `[ROLE]` | название роли |
| `[ROLE_ID]` | ID роли |

Типовой сценарий «менеджер видит только своих клиентов» — маска по
полю-ответственному, например `Ответственный = [CURRENT_USER]`. Тогда таблица, любой
отчёт по ней и API вернут менеджеру только его записи.

Частный случай — фильтрация по родителю: поле `up=<userId>` используется как
маска, чтобы пользователь видел только записи, созданные под ним
(`help.integram.io/#rec220708794`). Это удобно, когда записи создаются дочерними к
пользователю (`up=<userId>` при `_m_new`, см. 6.4).

Для гостевого доступа предусмотрен встроенный пользователь `guest` со своей ролью:
публичная форма (заявка с лендинга) работает под гостевым контекстом, заявитель не
видит внутренних таблиц и чужих заявок.

> Маска — это SQL-условие, поэтому слишком сложное выражение замедлит выборку на
> больших таблицах; роль стоит тестировать на реальном объёме данных
> ([`05-access-rights.md`](integram-article-reviews/05-access-rights.md)).

### 5.8 Алгоритм раздачи прав при разработке приложения

При создании нового приложения права раздаются по ролям в таком порядке:

1. **Определить роли** (менеджер, оператор, руководитель) — см. 5.3.
2. **Создать пункты меню** для каждой роли — какие рабочие места ей доступны
   (см. 5.4).
3. **Задать права роли на каждую используемую таблицу** — добавить таблицу в
   Объекты роли (`t117`) с нужным уровнем (`WRITE`/`READ`/`BARRED`), при
   необходимости понизив уровень отдельным полям (см. 5.6).
4. **Задать права роли на каждый используемый запрос** (`report/`) — аналогично,
   запрос попадает в тот же список Объекты с уровнем доступа (см. 5.6).
5. **Настроить маску доступа**, если нужна фильтрация по пользователю (например,
   менеджер видит только свои записи) — см. 5.7.

Принцип разграничения:

| Механизм | Что ограничивает |
|----------|------------------|
| Пункты меню (5.4) | **что видно** в интерфейсе роли |
| Права на объекты `t117` (5.6) | **к каким таблицам и отчётам** есть доступ и на каком уровне |
| Маска доступа (5.7) | **какие строки** внутри доступных объектов видит конкретный пользователь |

### 5.9 Типовые приёмы ограничения доступа

Доступ пользователей ограничивается двумя способами, которые на практике
комбинируют для тонкой настройки:

1. **Ограничение в роли** — указать таблицы и колонки в Объектах роли, задать им
   уровни доступа и маски (см. 5.6–5.7).
2. **Ограничение запросом** — написать запрос, который через контекстные
   переменные определяет текущего пользователя и фильтрует данные (см. раздел
   «Запросы» ниже).

Эти подходы дополняют друг друга. Поле, запрещённое к редактированию на уровне
роли, всё равно можно изменить запросом (например, статус заказа или дату его
подтверждения), потому что **запрос всегда выполняется с правами владельца**.

| Механизм | Что позволяет пользователю |
|----------|----------------------------|
| Ограничения роли | произвольное поведение — навигацию и всё, что не запрещено или заранее не предусмотрено разработчиком |
| Запрос | только фиксированное действие — выборку или конкретные точечные изменения, логика которых жёстко определена |

Простых ограничений в роли достаточно при низких требованиях к безопасности —
это удобно пользователям. Ограничения запросами применяют при строгих
требованиях, большом количестве сложных правил и при работе с недоверенными
внешними пользователями, в том числе с гостевым доступом.

#### 5.9.1 Запись прав напрямую в таблицу Объекты (116)

Колонка **Объекты** роли (`t117`, см. 5.1, 5.6) — это GRANT-поле, под которым
лежит подчинённая таблица **Объекты** (тип `116`): по одной записи на объект базы,
выбираются по `up=<role_id>`. Права можно создавать прямо в ней:

```bash
POST /{db}/_m_new/116?JSON=1
Fields: up=<role_id>, t116=<объект>, t136=<уровень>, t49=<маска>
```

**Колонки таблицы Объекты (116):**

| Колонка | Назначение |
|---------|------------|
| `t116` | первая колонка — Объект: id таблицы, колонки или синтетический id |
| `t136` | ссылка на справочник уровня доступа |
| `t49` | маска для значений фильтра по колонке, указанной в `t116` |
| `t55` | флаг, разрешающий экспорт из таблицы (`EXPORT`) |
| `t56` | флаг, разрешающий массовое удаление из таблицы (`DELETE`) |

**Числовые id уровней доступа** (справочник `t136`) — те же `BARRED`/`READ`/`WRITE`
из 5.6, но при записи напрямую указываются числом:

| id | Уровень | Что разрешает |
|----|---------|---------------|
| `52` | `BARRED` | запрещён просмотр и редактирование |
| `53` | `READ` | разрешён просмотр, нельзя редактировать |
| `54` | `WRITE` | разрешены просмотр и редактирование |

**Синтетические id объектов** (`t116`) дают обобщённый доступ (см. также 5.6):

| id | Объект |
|----|--------|
| `0` | Type editor — редактор структуры |
| `1` | All objects — все таблицы |
| `10` | Files — файловый менеджер |

**Примеры записей в Объекты роли:**

```bash
# А. Разрешить конкретную таблицу (пользователь видит всё разрешённое)
t116=18   # Объект — таблица Пользователь (18)
t136=53   # READ — на чтение

# Б. Разрешить все таблицы синтетическим объектом «Все объекты»
t116=1    # Объект — Все объекты
t136=53   # READ — на чтение

# 1. Скрыть конкретное поле от просмотра — пользователь не увидит колонку
t116=124  # Колонка Пользователь -> Activity
t136=52   # BARRED — колонка не будет видна

# 2. Разрешить колонку для редактирования
t116=33   # Колонка Пользователь -> Имя
t136=54   # WRITE — пользователь может менять имя видимых ему пользователей
```

> ⚠️ **Новый объект по умолчанию недоступен ролям.** Создав новую таблицу или
> справочник, который выводится на форму, **сразу выдай на него доступ всем ролям,
> чьи рабочие места его используют** (`t116=<новыйId> t136=53` READ). Иначе у роли
> выпадающий список из этого справочника будет **пустым**, а ссылочное значение в
> таблицах не отобразится. Какие роли работают с объектом — видно по их грантам на
> родительскую/соседние таблицы: `object/116?JSON_DATA&F_U=<roleId>`.

#### 5.9.2 Типовые маски (`t49`)

Маска задаётся в `t49` и ограничивает видимость по колонке из `t116`; уровень
(`t136`) при этом не указывается. Можно добавить несколько масок для одной
колонки — они складываются по **ИЛИ** (достаточно срабатывания хотя бы одной).
Маски на **разные** колонки складываются по **И**. В маске используются
стандартные фильтры Интеграма (`help.integram.io/#search`) и контекстные
переменные (`help.integram.io/#servicewords`, см. также список в разделе
«Запросы»).

**3.1. По первой колонке — пользователь видит только себя:**

```bash
t116=18      # Колонка Пользователь
t49=[USER]   # Контекстная переменная — имя текущего пользователя
```

В комбинации с примером 2 (WRITE на «Имя») пользователь может заполнить своё имя,
видя при этом только себя, тогда как остальные колонки даны ему на чтение.

**3.2. По колонке со справочным значением — фильтр строится по id:**

```bash
t116=115     # Колонка Пользователь -> Роль
t49=164      # Видны только пользователи с ролью user (id=164)
```

Добавив вторую запись-маску на ту же колонку, расширяем видимость по ролям (по
ИЛИ) — например, до роли lead:

```bash
t116=115     # Колонка Пользователь -> Роль
t49=1212     # ...и пользователи с ролью lead (id=1212)
```

Это ограничение по ролям нельзя использовать вместе с примером 2 (WRITE на «Имя»),
иначе пользователь сможет менять имя у любого видимого ему пользователя.
Комбинация 3.1 и 3.2 всё равно оставит видимым только самого пользователя:
условия масок на **разные** колонки складываются по И.

**3.3. По значению, возвращаемому запросом.** Если есть запрос `MyRole`,
возвращающий id роли текущего пользователя, его можно применить как маску, заключив
имя в квадратные скобки — она сработает для любого пользователя:

```bash
t116=115       # Колонка Пользователь -> Роль
t49=[MyRole]   # Если переменная контекста не подошла, Интеграм ищет запрос с таким именем
```

Колонки запроса `MyRole` (вернёт id роли текущего пользователя):

```json
[
  { "i": 1229, "u": 1228, "o": 1, "r": [ "18:Пользователь", "Пользователь", "", "[USER_ID]", "", "85:abn_ID", "", "", "X", "", "", "", "", "" ] },
  { "i": 1233, "u": 1228, "o": 2, "r": [ "42:Роль", "Роль", "", "", "", "85:abn_ID", "", "", "", "", "", "", "", "" ] }
]
```

**3.4. Напрямую по текущему пользователю, назначенному на запись.** Если у клиента
есть ссылка на пользователя-владельца/ответственного, в роли для этой ref-колонки
задаём маску `[USER_ID]` — пользователь увидит только тех клиентов, где он указан
ответственным или владельцем.

**3.5. Косвенно через запрос.** Видимость заказов можно ограничить через ссылку
заказа на клиента: в маску вставляется запрос, возвращающий id клиента, если его
присоединённые колонки и фильтры совпадут с параметрами текущего пользователя, его
группы и т.д. — в зависимости от сложности логики ограничения.

#### 5.9.3 Ограничение запросами

Запрос ограничивает сильнее роли: он позволяет выполнить только заранее
определённое действие. Непростой пример есть в каждой базе — построение меню
пользователя, заданного в роли. Колонки этого запроса:

```json
[
  { "i": 354, "u": 346, "o": 1, "r": [ "151:Меню", "name", "", "", "", "", "", "", "", "", "", "", "", "" ] },
  { "i": 352, "u": 346, "o": 2, "r": [ "42:Роль", "РольID", "РольID", "", "", "85:abn_ID", "", "", "X", "", "-1", "", "", "" ] },
  { "i": 349, "u": 346, "o": 3, "r": [ "18:Пользователь", "Пользователь", "ПользовательID", "", "", "85:abn_ID", "", "", "X", "", "", "", "", "" ] },
  { "i": 358, "u": 346, "o": 4, "r": [ "151:Меню", "menu_id", "МенюID", "", "", "85:abn_ID", "", "", "", "", "", "", "", "" ] },
  { "i": 356, "u": 346, "o": 5, "r": [ "0:Вычисляемое", "Вычисляемое", "IF(МенюID IN([rec]), ПользовательID IS NULL OR (ПользовательID=[USER_ID] AND LOCATE('_request','_request_.FR_RoleID')) OR '_request_.FR_RoleID'=РольID, (ПользовательID=[USER_ID] AND LOCATE('_request','_request_.FR_RoleID')) OR '_request_.FR_RoleID'=РольID)", "1", "", "", "", "", "X", "", "", "", "", "" ] },
  { "i": 371, "u": 346, "o": 6, "r": [ "151:Меню", "menu_up", "", "", "", "86:abn_UP", "", "", "", "", "", "", "", "" ] },
  { "i": 374, "u": 346, "o": 7, "r": [ "151:Меню", "МенюOrd", "", "", "", "93:abn_ORD", "", "", "X", "", "2", "", "", "" ] },
  { "i": 381, "u": 346, "o": 8, "r": [ "153:Меню -> Адрес", "href", "", "", "", "", "", "", "", "", "", "", "", "" ] },
  { "i": 404, "u": 346, "o": 9, "r": [ "391:Меню -> Иконка", "icon", "", "", "", "", "", "", "", "34:HTML", "", "", "", "" ] }
]
```

Здесь выбираются все пункты меню, к ним джойнятся роли (родительская запись), а к
ролям — пользователи со ссылкой на роль. Затем выбирается меню роли пользователя
или меню другой роли, если в запрос передан параметр выбора другой роли. Фильтр
задаёт формула колонки «Вычисляемое», значение которой по условию «Значение (от)»
= `1` (ИСТИНА):

```
IF(МенюID IN([rec]),
   ПользовательID IS NULL OR (ПользовательID=[USER_ID] AND LOCATE('_request','_request_.FR_RoleID')) OR '_request_.FR_RoleID'=РольID,
   (ПользовательID=[USER_ID] AND LOCATE('_request','_request_.FR_RoleID')) OR '_request_.FR_RoleID'=РольID)
```

Используется `_request_.FR_RoleID` — параметр запроса (`help.integram.io/#request`),
позволяющий выбрать меню другой роли, переданной в запрос.

#### 5.9.4 Хорошая практика

Базовые ограничения дают на уровне роли, а действия в рамках бизнес-процесса
реализуют запросами. Например, в роли — право на чтение всех запросов с префиксом
`user_`, а сами запросы (`user_*`) на выборку и изменение данных пользователь
сможет запускать. Запрет роли абсолютен для прямых действий пользователя, но
запрос администратора может его обойти, так как **запрос всегда выполняется с
правами владельца**.

### 5.10 Удаление записей (если нужно)

```bash
POST /{db}/_m_del/{objId}?JSON=1
# Работает для любых таблиц, включая системные.
# При удалении пункта меню — автоматически удаляются все подчинённые пункты.
```

## Этап 6: Наполнение тестовыми данными

### 6.1 Бэкап перед массовой загрузкой

Перед загрузкой справочников, CSV, начальных остатков или других массовых
данных в уже существующую базу сначала сделай серверный бэкап Интеграма. Это
обычный GET-запрос к действию `backup` той базы, в которую идёт загрузка:

```bash
DB=ateh
TOKEN="<token>"

curl -sS -D "backup-${DB}.headers" -o "backup-${DB}.html" \
  -H "X-Authorization: ${TOKEN}" \
  -b "idb_${DB}=${TOKEN}" \
  "https://ideav.ru/${DB}/backup"
```

После успешного запроса файл появляется в файловом менеджере базы:

```text
https://ideav.ru/{db}/dir_admin/?templates=1&add_path=/backups
```

Для `ateh` это:

```text
https://ideav.ru/ateh/dir_admin/?templates=1&add_path=/backups
```

Файл бэкапа — дважды сжатый SQL-дамп всей базы. Если загрузка испортит данные,
его можно развернуть обратно, заменив базу. Не начинай массовую загрузку, если
`/{db}/backup` вернул ошибку авторизации или текст вроде `У вас нет прав на
выгрузку базы`: нужен токен пользователя с правом выгрузки всей базы или
администраторский доступ.

### 6.2 Скрипт наполнения

Создаётся отдельный bash-скрипт (паттерн: `seed_data.sh`), который:
1. Авторизуется
2. Делает бэкап текущей базы, если наполнение идёт не в пустой одноразовый стенд
3. Создаёт записи через `_m_new/{tableId}` с указанием значений полей по алиасам

### 6.3 Принципы создания тестовых данных

1. **Сырьё** — 3-5 видов с разной шириной джамбо и остатками
2. **Слиттеры** — 3-4 слиттера с разными статусами (Свободен, Обслуживание)
3. **Типы резки** — по 1-2 на каждый вид сырья, разные втулки и намотки
4. **Заказы** — 3-5 заказов от разных заказчиков, разные даты и статусы
5. **Позиции заказа** — 2+ позиции на заказ, разное количество рулонов
6. **Производственная резка** — для каждого заказа свои резки на слиттерах, разные статусы (Ожидает, Наладка, В работе, Завершён)

### 6.4 API для создания записей

```bash
# Корневая запись (up=1)
POST /{db}/_m_new/{tableId}?JSON=1
Fields: up=1, t{tableId}="Название", t{colId1}=значение, t{colId2}=значение

# Дочерняя запись (up=parentId)
POST /{db}/_m_new/{tableId}?JSON=1
Fields: up=parentId, t{tableId}="Название", t{colId1}=значение

# Установка ref-поля (значение = ID записи из целевой таблицы)
Fields: t{colId}=579  # где 579 — ID записи в целевой таблице
```

### 6.5 Результат этапа

Файл `seed_data.sh`, который создаёт полный набор тестовых данных для демонстрации всех экранов.

## Управление файлами сервера через `dir_admin` (по API, без SSH)

РМ `dir_admin` (`templates/dir_admin.html`; обработчик в `index.php`, `case "&dir_admin"`)
— файловый менеджер базы. Им можно **создавать/удалять/загружать/читать файлы на боевом
сервере по API** (js/css/шаблоны конкретной БД) — без SSH/FTP. Управляет двумя папками БД:
- `download` → `download/{db}/` (публичные js/css/шрифты; URL `…/download/{db}/…`);
- `templates` → `templates/custom/{db}/` (шаблоны рабочих мест).

**Авторизация** (проверено на боевой ateh): заголовок `X-Authorization: <token>` **И** cookie
`idb_{db}=<token>` (xsrf привязан к сессии-куке — без неё `check()` → `InvalidToken`); `_xsrf`
из `GET /{db}/xsrf?JSON=1` (под той же кукой); в теле POST — `token` и `_xsrf`. Нужен грант WRITE.

**Команды** — POST на `/{db}/dir_admin/?{folder}=1` (folder=`download`|`templates`), общие поля
`add_path=<подпуть>` (`..` запрещён), `_xsrf`, `token`:
- удалить: `del[]=<имя>` (можно несколько) + `delete=1`;
- каталог: `dir_name=<имя>` + `mkdir=1`; файл: `dir_name=<имя>` + `touch=1` (без расширения → `.html`; опасные расширения блокируются BlackList);
- загрузить: multipart `userfile` + `upload=1` (+`rewrite=1`);
- прочитать/проверить файл: `GET …&gf=<имя>` → содержимое или «Файл не найден» (авторитетная проверка существования).

> ⚠️ Ответ команд — HTTP-редирект (`Location:`), не JSON; ошибки — HTML `die()`. Для API
> добавлен флаг **`?JSON=1`** (см. ниже) → команды отвечают `{"ok":true,"action":…}`.
> ⚠️ HTTP-200 на `download/{db}/X.html` НЕ доказывает существование (для `.html` отдаётся
> SPA-фолбэк); проверяй через `gf` (или по `.js`/`.css` — те дают 404, если файла нет).
> ⚠️ `index.php` **не** в `update.conf` — деплоится отдельно, не через `update.php`.

Пример (удалить `download/ateh/js/cut-calc.js`):
`POST /ateh/dir_admin/?download=1` тело `add_path=/js&del[]=cut-calc.js&delete=1&_xsrf=…&token=…`,
cookie `idb_ateh=…`.

### Безопасность файлового доступа

- **Доступ к файлам — отдельный объект роли.** Авторизация `dir_admin` идёт через
  встроенный объект **`BT["FILE"]`** (`RepoGrant()`): явный грант роли на этот объект →
  `READ`/`WRITE`; иначе владелец БД/`admin` → `WRITE`; остальным → `BARRED`. **Не выдавайте
  FILE-объект массово**: WRITE на `dir_admin` = возможность перезаписать любой js/шаблон БД
  (XSS/дефейс/инъекция кода фронта для всех пользователей этой БД) — это админ-уровень.
- **Path traversal закрыт** (хардненинг): `del[]` и имя загружаемого файла нормализуются
  (`basename`, запрет `..`/`/`/`\`), пишут только внутрь папки БД.
- **Исполнение кода запрещено**: `BlackList` блокирует опасные расширения (php*/phtml/phar/cgi/
  pl/asp/jsp/sh/**htaccess**/htpasswd). Плюс на уровне веб-сервера (`docker/apache/integram.conf`)
  в `download/` и `templates/custom/` — `AllowOverride None` + `Require all denied` на скриптовые
  расширения (загруженный `.php`/`.htaccess` не выполнится). ⚠️ **На боевом сервере (вне Docker)
  тот же запрет нужно прописать в его vhost/панели хостинга отдельно** — docker-конфиг прод не покрывает.
- Удаления логируются (`trace`); перед необратимым — бэкап ([[feedback_integram_backup_before_irreversible]]).

### Кодировка PowerShell-скриптов

Все `docs/*.ps1` с русским текстом должны храниться как UTF-8 with BOM. Windows
PowerShell 5.1 иначе читает UTF-8 без BOM как локальную ANSI-кодировку: кириллица
превращается в mojibake, а часть байтов может стать PowerShell-кавычками и
сломать парсер (`ParserError`, `UnexpectedToken`, отсутствующая закрывающая `"`).

Перед запуском или публикацией нового PowerShell-скрипта проверь кодировку:

```bash
python3 experiments/test-issue-2994-powershell-encoding.py
```

JSON-файлы данных остаются обычным UTF-8; PowerShell-скрипты читают их через
`Get-Content -Encoding UTF8`.

## Метаданные (`metadata`) — справочник

Команда `metadata` — это «карта схемы» базы: по ней агент узнаёт, **какие
таблицы существуют, как они связаны, какие у колонок типы, алиасы, значения по
умолчанию и права**, не заглядывая в сами данные. Это первый запрос, с которого
стоит начинать работу с незнакомой базой, и инструмент самопроверки после
создания структуры (см. [2.5.4](#254-как-тип-связи-виден-в-метаданных-самопроверка)).

Всё, что описано ниже, — это тот же формат, который разбирает компонент
`IntegramTable` (`js/integram-table.js`, исходники в `js/integram-table/`),
когда строит таблицу из ответа сервера. Поэтому трактовка здесь и поведение
рабочих мест гарантированно совпадают.

### Две формы вызова

```bash
# 1. Вся база — массив объектов-таблиц (по одному на таблицу)
GET /{db}/metadata?JSON=1
# IntegramTable грузит её один раз и кэширует как globalMetadata
# (js/integram-table/01-core.js:282 loadGlobalMetadata).

# 2. Одна таблица по её типу — один объект-таблица
GET /{db}/metadata/{typeId}?JSON=1
# fetchMetadata(typeId) (js/integram-table/16-state.js:354): сперва ищет таблицу
# в globalMetadata, и только если не нашёл — делает отдельный запрос.
```

> `metadata` возвращает **только определения таблиц и колонок** — ни записей, ни
> запросов (`report/`). Для записей нужен `object/{typeId}` (см. [3.5](#35-чтение-существующих-записей-и-метаданных)),
> а перечень запросов отдаёт `GET object/22?JSON_OBJ` (записи системной таблицы
> «Запрос», тип 22 — см. раздел «Запросы»).

> **Что не возвращается, то запрещено.** Таблицы и колонки, на которые у роли
> уровень `BARRED`, в `metadata` не попадают совсем (см. [5.6](#56-права-роли-на-таблицы-и-запросы)).
> Поэтому набор объектов в ответе зависит от роли токена, под которым сделан
> запрос: под админским токеном видно всё, под ограниченной ролью — только
> разрешённое.

### Объект таблицы

Каждый элемент верхнего уровня описывает одну таблицу (тип):

| Поле | Значение |
|------|----------|
| `id` | ID таблицы (тип). Им адресуют таблицу в `object/{id}`, `metadata/{id}`, в правах роли (`t117`) и т.д. |
| `up` | Родитель в дереве типов; у обычных таблиц `"0"`. |
| `type` | Базовый код **первой колонки** таблицы (имени записи): `3`=SHORT, `9`=DATE, `4`=DATETIME, `13`=NUMBER и т.д. (см. [2.2](#22-типы-колонок-t)). |
| `val` | Имя таблицы. Компонент берёт его как заголовок (`js/integram-table/01-core.js:950`). |
| `unique` | `"1"`, если первая колонка — автоинкремент / уникальна (создаётся `_d_new … unique=1`, см. [3.2 шаг 2](#32-последовательность-api-вызовов)). |
| `granted` | Уровень доступа роли к таблице: `"WRITE"` — можно писать; иное значение — только чтение. Компонент переводит форму в режим «только чтение», если это не `WRITE` (`isTableWritable`, `js/integram-table/01-core.js:201`, issue #1508). |
| `export` | `"1"`, если роли разрешена выгрузка (`js/integram-table/01-core.js:955`). |
| `delete` | `"1"`, если разрешено массовое удаление по фильтру (`isTableDeletable`, issue #2749). |
| `referenced` | Присутствует, если на эту таблицу **ссылаются** из других таблиц (обратные ссылки). Значение — id ref-типа(ов); сервер не даст удалить такую таблицу, пока ссылки не сняты (`js/integram-table/11-column-settings.js:833`, issue #2746). |
| `reqs[]` | Массив колонок-реквизитов (кроме первой колонки-имени) — см. ниже. |

> Первая колонка (имя записи) **в `reqs[]` не входит**: её описывают поля самой
> таблицы (`type`, `val`, `unique`), а в записях и фильтрах она адресуется ключом
> `t{table.id}` (см. [3.2 шаг 2](#32-последовательность-api-вызовов)). Колонки из
> `reqs[]` адресуются ключом `t{req.id}`.

### Реквизит (колонка) — элемент `reqs[]`

| Поле | Значение |
|------|----------|
| `num` | Порядковый номер колонки (для сортировки в UI). |
| `id` | ID **реквизита в этой таблице**. Именно он — ключ колонки в записях и фильтрах: `t{req.id}` в `_m_new`/`_m_set` и в `FR_{req.id}`/`F_{req.id}` (issue #793). |
| `val` | Имя **типа** колонки; для ссылки — имя целевой таблицы. |
| `type` | Базовый код типа (`3`,`8`,`9`,`13`,…; см. [2.2](#22-типы-колонок-t)). У ссылки это `3` — в ячейке показывается значение целевой записи. |
| `orig` | ID **типа** колонки (не реквизита!). Нужен для переименования типа через `_d_save` (см. [3.6](#36-переименование-существующих-типов-и-таблиц)). У ссылки `orig` = id целевой таблицы. |
| `ref` + `ref_id` | Признак **ссылки** (FK): `ref` = id целевой таблицы, `ref_id` = id ref-типа. Компонент считает колонку ссылкой именно по наличию `ref_id` (`isReference = req.hasOwnProperty('ref_id')`, `js/integram-table/02-format-helpers.js:15`). |
| `arr_id` | Признак **подчинённой таблицы**: id дочернего типа. Поля `ref`/`ref_id` при этом отсутствуют. |
| `attrs` | Строка с алиасом, значением по умолчанию и флагами — разбирается отдельно (см. ниже). |
| `granted` | Уровень доступа роли к колонке, когда у таблицы доступ не сплошной `WRITE` (`js/integram-table/02-format-helpers.js:17`, issue #1508). |
| `unique` | `"1"`, если колонка участвует в ключе уникальности / автоинкременте. |

### Ссылка, подчинённая таблица или обычная колонка

Тип связи определяется тремя полями реквизита — это формализация
[2.5.4](#254-как-тип-связи-виден-в-метаданных-самопроверка), ровно как её читает
`js/integram-table/06-render-cell.js:45`:

| Что в реквизите | Что это | Как трактовать |
|-----------------|---------|----------------|
| есть `ref` и `ref_id` | **ссылка** (FK) | `ref` = id справочника, `ref_id` = id ref-типа; в записях значение = id целевой записи. В `object/…?JSON_OBJ` приходит как `"id:Значение"` (мульти — `"id1,id2:знач1,знач2"`). |
| есть `arr_id`, нет `ref` | **подчинённая таблица** | `arr_id` = id дочернего типа; дочерние записи создаются с `up=<родитель>` (см. [3.4](#34-подчинённые-таблицы)). В ячейке показывается количество детей. |
| `orig === "1"`, нет `ref`/`ref_id` | **ссылка на любую запись** | универсальный указатель на запись любого типа (issue #1794). |
| ничего из перечисленного | **обычная колонка** | примитивный тип из `type` (текст, число, дата, файл…). |

> **Самопроверка структуры.** Если связь, которая по смыслу должна быть
> справочником (например `Заказ.Клиент`), в метаданных имеет `arr_id` вместо
> `ref`/`ref_id` — типы перепутаны, структуру надо чинить (разбор реальных баз —
> [2.5.6](#256-разбор-реальных-баз)).

### `attrs` — алиас, значение по умолчанию и флаги

Поле `attrs` несёт оформление колонки в этой конкретной таблице. Сервер отдаёт
его в одной из двух форм, и обе одинаково разбирает `parseIntegramAttrs`
(`js/integram-table/24-global-functions.js:7`):

- **JSON-форма:** `{"alias":"Заказчик","default":"[USER_ID]","required":true,"multi":false,"key":false}`.
- **Старая «двоеточная» форма:** флаги `:!NULL:`, `:MULTI:`, `:KEY:`,
  `:ALIAS=Имя:`, а **всё, что осталось** после вырезания флагов — это значение по
  умолчанию.

Что извлекается:

| Свойство | JSON-ключ | Двоеточная форма | Смысл |
|----------|-----------|------------------|-------|
| Алиас | `alias` | `:ALIAS=Имя:` | Отображаемое имя колонки в этой таблице. Имеет приоритет над `val`: компонент показывает `attrs.alias \|\| val` (`js/integram-table/02-format-helpers.js:24`). Ставится командой `_d_alias` (см. [3.2 шаг 5](#32-последовательность-api-вызовов)). |
| Обязательность | `required` (а также `notNull`/`not_null`) | `:!NULL:` | Поле нельзя оставить пустым (в форме помечается как required). |
| Множественность | `multi` | `:MULTI:` | Мультиссылка — можно выбрать несколько целевых записей (значение приходит как `"id1,id2:знач1,знач2"`). |
| Ключ | `key` | `:KEY:` | Колонка входит в составной ключ уникальности (см. [2.1](#21-правила-проектирования-таблиц-из-structurerulesmd) п.5). |
| Значение по умолчанию | `default` (или `defaultValue`) | остаток строки после флагов | Подставляется в новую запись (см. ниже). |

**Значения по умолчанию** поддерживают встроенные подстановки, которые
`resolveDefaultValue` (`js/integram-table/18-data-source.js:150`) раскрывает на
момент создания записи:

`[NOW]`, `[TODAY]`, `[YESTERDAY]`, `[TOMORROW]`, `[MONTH_AGO]`, `[WEEK_AGO]`,
`[MONTH_PLUS]`, `[USER]`, `[USER_ID]`, `[ROLE]`, `[ROLE_ID]`, `[HTTP_HOST]`,
`[REQUEST_URI]`.

Неизвестный токен или обычная строка трактуются как литеральное значение по
умолчанию. Кроме того, для колонок типа `DATE`/`DATETIME` без явного `default`
компонент сам подставляет текущую дату/время.

**Примеры `attrs` из реальных баз** (таблица `Пользователь`, тип 18):

```jsonc
// Роль — обязательная ссылка с дефолтом «id роли 164»:
{ "id":"115", "val":"Роль", "orig":"42", "type":"3",
  "ref":"42", "ref_id":"114", "attrs":":!NULL:164" }
//   :!NULL: → required; остаток "164" → значение по умолчанию (id роли).

// Дата — обычная колонка-дата с дефолтом «сегодня»:
{ "id":"156", "val":"Дата", "orig":"155", "type":"9", "attrs":"[TODAY]" }
//   нет флагов; "[TODAY]" → значение по умолчанию (текущая дата).
```

То же в JSON-форме `attrs` выглядело бы как
`{"required":true,"default":"164"}` и `{"default":"[TODAY]"}` соответственно.

### Полный разбор примера

Фрагмент `metadata` для таблицы `Тип резки` (проект `atex`):

```jsonc
{
  "id": "104", "up": "0", "type": "3", "val": "Тип резки", "unique": "0",
  "reqs": [
    { "num":1, "id":"1025", "val":"Вид сырья", "orig":"100", "type":"3",
      "ref":"100", "ref_id":"1026" },
    { "num":2, "id":"1027", "val":"Ширина входа, мм", "orig":"1028", "type":"13" },
    { "num":7, "id":"1037", "val":"Полоса", "orig":"105", "type":"3",
      "arr_id":"105" }
  ]
}
```

Трактовка:
- Таблица `Тип резки`, id `104`, первая колонка-имя — SHORT (`type:3`), не
  автоинкремент (`unique:0`). В записях имя пишется как `t104=…`.
- `t1025` «Вид сырья» — **ссылка** на справочник `Вид сырья` (`ref:100`,
  `ref_id:1026`). В `_m_new`/`_m_set` сюда кладут id записи из таблицы 100; в
  выгрузке `object/` значение приходит как `"100xx:Бумага"`.
- `t1027` «Ширина входа, мм» — **обычная** числовая колонка (`type:13`, NUMBER),
  без ссылок и дефолтов.
- `t1037` «Полоса» — **подчинённая таблица** (`arr_id:105`, поля `ref`/`ref_id`
  отсутствуют): дочерние записи типа `105` создаются с `up=<id записи Тип резки>`,
  а в ячейке показывается их количество.

## Запросы (report/) — справочник

Запросы (отчёты) — это сохранённые на сервере SQL-подобные выборки, которые
вызываются по имени или числовому ID и возвращают готовые данные для UI. Из
HTML/JS-шаблона запрос вызывается обычным HTTP-запросом (`fetch`/`XMLHttpRequest`),
без ручного составления SQL. Это основной способ наполнять таблицы, графики,
воронки и дашборды данными.

### Вызов запроса

```
GET /{db}/report/{имяЗапроса}?JSON_KV
GET /{db}/report/{id}?JSON_KV
GET /{db}/report/{id}?JSON_DATA
GET /{db}/report/{id}?JSON
```

- `{db}` — имя базы данных. В шаблонах берётся из переменной (`window.db`,
  `{_global_.z}` и т.п.).
- `{имяЗапроса}` — человекочитаемое имя отчёта (например, `Дэшборд`, `Статусы`).
  Имя с не-ASCII символами или пробелами нужно URL-кодировать через
  `encodeURIComponent` (`report/Все%20статусы%20задач`). Сервер находит отчёт
  запросом `SELECT id FROM {db} WHERE t=22 AND val='<имя>' ORDER BY id`;
  если отчётов с таким именем несколько, вернётся тот, у кого меньше id — давайте им
  уникальные имена или используйте числовой ID.
- `{id}` — числовой ID отчёта (например, `report/7701`). Предпочтителен: не
  зависит от переименования, не требует кодирования и однозначен при совпадающих
  именах.

Для UI обычно нужен `JSON_KV`: это массив объектов, ключи которого совпадают с
именами колонок отчёта. `JSON_DATA` возвращает только первую строку тем же
плоским объектом, а `JSON` оставляет полный платформенный формат.

```javascript
// Числовой ID
fetch('/' + db + '/report/8027?JSON_KV')
    .then(function (r) { return r.json(); })
    .then(function (rows) { /* rows = [{...}, {...}] */ });

// По имени (с URL-кодированием)
fetch('/' + db + '/report/' + encodeURIComponent('Все статусы задач') + '?JSON_KV');
```

Примеры реальных вызовов в репозитории:
`templates/sportzania/struct.html` (`report/8027`), `templates/funnel.html`
(`report/7701`), `templates/calendar.html` (`report/5230`, `report/Все статусы задач`).

### Как узнать числовой ID запроса

Числовой ID предпочтительнее имени — не требует URL-кодирования и не ломается при
переименовании. Два способа его получить:

1. **Через заголовок ответа.** Вызов запроса по имени возвращает заголовки
   `X-Query-Id` и `X-Query-Name` (реализовано в PR #2723):
   ```javascript
   fetch('/' + db + '/report/' + encodeURIComponent('Мой отчёт') + '?JSON_KV')
       .then(function(r) {
           var id = r.headers.get('X-Query-Id');   // "7701"
           var name = r.headers.get('X-Query-Name'); // URL-encoded name
           return r.json();
       });
   ```
   После первого вызова по имени сохраните полученный ID и используйте его в
   дальнейших запросах.

2. **Через браузер.** Откройте базу → меню «Запросы» → нажмите на нужный запрос —
   ID будет виден в URL (`/{db}/query/{id}`).

> **Важно:** `GET /{db}/metadata?JSON=1` возвращает только определения таблиц и типов,
> но **не** список запросов. Перечень всех отчётов отдаёт `GET /{db}/object/22?JSON_OBJ`
> (или `GET /{db}/object/Запрос?JSON_OBJ`) — это записи системной таблицы «Запрос»
> (тип 22).

### Фильтрация через FR_

Фильтры передаются как query-параметры с префиксом `FR_` плюс имя колонки. Их
можно добавлять и собирать динамически (удобно через `URLSearchParams`):

| Шаблон | Назначение | Пример |
|--------|-----------|--------|
| `FR_{Колонка}=значение` | фильтр по одному значению | `report/155675?FR_СтатусID=9002` |
| `FR_{Колонка}=IN(v1,v2)` | фильтр по нескольким значениям | `report/155675?FR_СтатусID=IN(9002,44895)` |
| `FR_{Колонка}ID=ID` | фильтр ref-поля по числовому ID | `report/Дэшборд?FR_dept=IN(2889)` |
| `FR_{Колонка}=%значение%` | частичное совпадение (LIKE) | `FR_SKU=%бумага%` |
| `FR_Date=...&TO_Date=...` | диапазон дат (пара `FR_`/`TO_`) | `?JSON_KV&FR_Date=01.01.2026&TO_Date=31.12.2026` |

> ⚠️ **Условие работы фильтра:** `FR_`/`TO_` применяется **только к колонкам,
> у которых в конструкторе запроса задано «Имя в отчёте»** (report alias). Колонка
> без имени недоступна для фильтрации извне — параметр молча игнорируется, данные
> возвращаются без фильтрации. Это намеренная защита: аналитик сам решает, какие
> поля доступны для управления снаружи.

**Ref-поля.** Для ссылочных колонок предпочтительнее фильтровать по числовому ID
целевой записи, а не по её названию — это надёжнее при совпадающих именах. Для
этого к имени колонки добавляется суффикс `ID`: `FR_{Колонка}ID=<refId>` (см.
issue #2843, `js/xcom-match.js`). Сравните: фильтр по названию статуса
(`FR_Статус=Новый`) против фильтра по ID (`FR_СтатусID=9002`, issue #2187).

**Диапазоны дат.** Колонка-дата фильтруется парой параметров: `FR_{Колонка}` —
нижняя граница, `TO_{Колонка}` — верхняя (`FR_Date=01.01.2026&TO_Date=31.12.2026`,
см. `js/dash.js`, `templates/calendar.html`).

**panelFilter в дашбордах.** В отчёте `report/Дэшборд` каждая панель может нести
поле `panelFilter` (issue #2166, #2352):
- если значение похоже на `имя=значение` (например, `FR_dept=IN(2889)`) — его
  добавляют как параметр к запросу данных панели;
- если значение похоже на `имя:значение` (например, `Тип документаID:158844`) —
  фильтр применяется **локально** при заполнении панели, а не передаётся на
  сервер (используется, когда один отчёт наполняет несколько панелей).

**FR_ vs F_ — два формата фильтра.** Параметры фильтрации могут начинаться на
`FR_`, `F_` или `TO_`. Это не одно и то же:

| Префикс | Где работает | Условия |
|---------|-------------|---------|
| `FR_{alias}=` | `report/` **и** `object/` (таблица) | Для report/: только колонки с «Именем в отчёте» |
| `TO_{alias}=` | `report/` **и** `object/` | Вторая граница диапазона (пара к `FR_`) |
| `F_{colId}=` | только `object/` (таблица) | Альтернативная форма `FR_` для URL-параметров таблиц |

`F_` обрабатывается идентично `FR_` в запросах к `object/{typeId}`, но **не**
работает в `report/`. Используйте `FR_` везде, где не уверены (issue #549,
`js/integram-table/14-url-config.js:258`).

`F_I` не является `F_{colId}`: это специальный параметр `object/` для точечной
загрузки записи по `vals.id`. Компонент таблицы читает `F_I` из URL как
`recordId`, добавляет его к запросу `/object/{typeId}/?JSON_OBJ`, а в строке
фильтров показывает как `@{recordId}` на первой колонке, чтобы пользователь видел
и мог снять фильтр.

### Форматы ответа: JSON, JSON_DATA и JSON_KV

- `?JSON_KV` / `?JSON_KV=1` — «key-value»: массив объектов
  `[{"Колонка": "значение", ...}]`,
  ключи = имена колонок отчёта. Самый удобный формат для UI, используется в
  подавляющем большинстве шаблонов. В ответе также есть заголовки
  `X-Query-Id` и `X-Query-Name`.

  ```json
  [
    { "InviteID": "236448", "Статус": "374", "GuestUser": "Fold" }
  ]
  ```

- `?JSON_DATA` / `?JSON_DATA=1` — один плоский объект по первой строке результата.
  Удобен, когда запрос гарантированно возвращает одну запись настроек или одно
  вычисленное значение.

  ```json
  { "InviteID": "236448", "Статус": "374", "GuestUser": "Fold" }
  ```

- `?JSON` / `?JSON=1` — стандартный JSON Интеграм: объект с тремя ключами —
  именем отчёта (`header`), описанием колонок (`columns`) и данными по колонкам
  (`data`). Применяется, когда нужны метаданные колонок (имя, тип, формат, признак
  редактируемости, целевая таблица ссылки), а не только «плоский» табличный
  результат `JSON_KV`. Именно этот формат «из коробки» разбирает табличный
  компонент `js/integram-table.js`. Полная раскладка — в подразделе
  [«Стандартный JSON (`?JSON`)»](#стандартный-json-json-колонки-имя-отчёта-и-данные)
  ниже.

Параметры формата и фильтры комбинируются в одной строке запроса:
`report/{id}?JSON_KV&FR_Date=01.01.2026&TO_Date=31.12.2026&FR_dept=IN(2889)`.

### Стандартный JSON (`?JSON`): колонки, имя отчёта и данные

`?JSON` / `?JSON=1` возвращает не «плоский» массив строк (как `JSON_KV`), а
объект-конверт с метаданными. Это формат, который табличный компонент
`js/integram-table.js` принимает по умолчанию (`dataSource: 'report'`,
`apiUrl: '/{db}/report/{id}?JSON'`) и из которого строит таблицу с заголовками,
фильтрами, сортировкой и редактированием ячеек.

```json
{
  "header": "Задачи проекта",
  "columns": [
    { "id": "4284", "type": "3596", "format": "CHARS",  "name": "Задача",   "granted": 1, "ref": 0 },
    { "id": "4285", "type": "9",    "format": "DATE",    "name": "Срок",     "granted": 1, "ref": 0 },
    { "id": "4286", "type": "42",   "format": "REF",     "name": "Ответственный", "granted": 1, "ref": 42 }
  ],
  "data": [
    ["Сверстать макет", "Написать тесты"],
    ["2026-06-01", "2026-06-03"],
    ["Иванов", "Петрова"]
  ]
}
```

**`header` — имя отчёта.** Строка с человекочитаемым названием запроса. Компонент
подставляет её как заголовок таблицы, если `title` не задан явно
(`js/integram-table.js`, `loadDataFromReport` → `if (!this.options.title && json.header)`).

**`columns` — описание колонок.** Массив дескрипторов, по одному на колонку отчёта
в порядке вывода. Поля каждого дескриптора:

| Поле | Назначение |
|------|-----------|
| `id` | ID колонки. Используется в параметрах фильтра (`FR_{id}`), сортировки (`ORDER`) и при сохранении порядка/ширины колонок. |
| `name` | Отображаемое имя колонки — это «Имя в отчёте» (`t100`, см. [«Создание запроса программно»](#создание-запроса-программно)). Колонка без `name` не выводится и недоступна для `FR_`/`TO_`. |
| `type` | ID типа Интеграма (как `orig` в metadata): например `3` — SHORT, `9` — DATE, `42` — тип таблицы-справочника для ссылки. |
| `format` | Имя формата для UI (фильтры/рендер): `SHORT`, `CHARS`, `NUMBER`, `SIGNED`, `DATE`, `DATETIME`, `BOOLEAN`, `MEMO`, `FILE`, `HTML`, `REF`. Соответствие «код типа → формат» см. `mapTypeIdToFormat` в `js/integram-table.js`. |
| `granted` | `1` — колонку можно редактировать прямо в таблице (WRITE), `0` — только чтение. |
| `ref` | Для ссылочных колонок (`format: "REF"`) — ID целевой таблицы-справочника; `0` для обычных колонок. Компонент по нему подгружает выпадающий список значений. |

**`data` — значения по колонкам (column-major).** Это массив колонок: `data[i]` —
массив всех значений `i`-й колонки сверху вниз, а не массив строк. Компонент
транспонирует его в строки (`loadDataFromReport`: `row.push(columnData[colIndex][rowIndex])`),
поэтому длины всех вложенных массивов должны совпадать и равняться числу строк.
В примере выше три колонки по две записи дают две строки таблицы:
`["Сверстать макет", "2026-06-01", "Иванов"]` и
`["Написать тесты", "2026-06-03", "Петрова"]`.

> Не путать с `object/{typeId}/?JSON_OBJ` — там данные приходят построчно как
> массив объектов `{i, u, o, r:[...]}` (ID записи, родитель, порядок, значения).
> Это другой формат; его компонент тоже умеет читать (источник `dataSource: 'table'`),
> но у `report/?JSON` раскладка именно по колонкам.

#### Служебные колонки по имени: `…ID` и `…Стиль`/`…style`

Кроме данных компонент извлекает из имён колонок две служебные роли
(`processColumnVisibility` в `js/integram-table.js`). Они задаются прямо в
конструкторе запроса через «Имя в отчёте» (`t100`):

- **`{Колонка}ID`** — рядом с колонкой `{Колонка}` добавьте колонку с тем же
  именем и суффиксом `ID`, отдающую ID записи. Компонент прячет её из вывода и
  использует как идентификатор записи для inline-редактирования ячейки `{Колонка}`
  и для фильтра ref-полей по ID (`FR_{Колонка}ID=<id>`).
- **`{Колонка}Стиль`** или **`{Колонка}style`** (регистр не важен) — колонка с
  CSS-строкой, которая будет подставлена в атрибут `style` ячеек базовой колонки
  `{Колонка}`. Тоже скрывается из вывода. Например, формула колонки «СуммаСтиль»
  может вернуть `color:red;font-weight:bold` для просроченных строк, и компонент
  раскрасит ячейки «Сумма». Стиль вычисляется обычной колонкой отчёта (`t101`),
  значение применяется как есть (`customStyle = ' style="' + styleValue + '"'`).

### Запросы на изменение данных (SET-запросы)

К любому отчёту можно добавить операцию изменения данных — поле SET в конструкторе
задаёт, какие колонки и как обновить у всех записей, попавших под фильтр.
Выполнение двухэтапное:

```
# Шаг 1 — превью: посмотреть, что изменится (обычный GET)
GET /{db}/report/{id}?JSON_KV&FR_InviteID=236448

# Шаг 2 — выполнение: применить изменения
POST /{db}/report/{id}
Body: confirmed=1&FR_InviteID=236448&_xsrf=...
```

Типичные применения:
- массово обновить поле у отфильтрованных записей;
- отозвать приглашение/статус по кнопке (issue #1158);
- удалить дубликаты по расписанию (запрос находит дубли, `?confirmed` удаляет).

### Создание запроса программно

**Запрос — это обычная запись** в системной таблице с типом **22** («Запрос»). У неё есть
**подчинённая таблица «Колонки запроса»** (тип **28**), каждая строка которой описывает одну
колонку — источник поля, фильтр, функцию агрегации, сортировку, SET-операцию и т.п.
Ядро Интеграм читает эти строки и строит из них SQL-запрос SELECT.

Управление осуществляется через обычные команды `_m_*` — никаких схемных операций
(`_d_*`) не требуется. Рабочее место `templates/sql.html` реализует полный цикл работы
с запросами через эти же команды.

```bash
# 1. Создать запрос (запись в таблице Запрос, тип 22)
POST /{db}/_m_new/22?JSON
Body: t22=Мой отчёт&_xsrf=...
# Ответ: {"id": 12345}  ← ID нового запроса

# 2. Добавить колонку (запись в «Колонки запроса», тип 28)
#    up — ID запроса-родителя; t28 — ID таблицы/реквизита из метаданных;
#    t100 — отображаемое имя и внешний alias для FR_/TO_
POST /{db}/_m_new/28?JSON&up=12345&t28={fieldId}&t100=НазваниеКолонки
Body: _xsrf=...

# 3. Настроить колонку (обновить любой атрибут через _m_set)
POST /{db}/_m_set/{columnRecordId}?JSON
Body: t100=ИмяДляФильтра&t107=1&_xsrf=...

# 4. Добавить JOIN-таблицу (запись в подчинённой таблице типа 44)
#    t44 — ID типа таблицы для JOIN; t265 — алиас; t266 — условие JOIN
POST /{db}/_m_new/44?JSON&up=12345&t44={tableTypeId}
Body: t265=псевдоним&t266=условие_join&_xsrf=...
```

**Поля записи-колонки** (тип 28, изменяются через `_m_set/{colRecordId}?JSON`):

| Поле | Назначение | Значения |
|------|-----------|---------|
| `t28` | Источник колонки: ID таблицы или реквизита из метаданных | числовой ID |
| `t100` | «Имя в отчёте»: заголовок колонки и имя для внешних `FR_`/`TO_`-фильтров | произвольная строка; пробелы в URL заменяются на `_` |
| `t58` | `ALIAS`: внутренний alias SQL/JOIN, в том числе для выбора нужной записи `REP_JOIN` | строка без пробелов; это не внешний `FR_`-alias |
| `t84` | Формат отображения | `DATE`, `DATETIME`, `NUMBER`, `BOOLEAN`, `SIGNED`, `PWD` |
| `t101` | Формула / выражение | SQL-выражение, `[THIS]`, ссылки на другие колонки или ключевые слова `ID`, `T`, `ORD`, `UP`, `VAL` |
| `t102` | Фильтр FROM (нижняя граница) | значение фильтра |
| `t103` | Фильтр TO (верхняя граница) | значение фильтра |
| `t104` | Функция обработки | `abn_ID`, SQL-функция или агрегация (`SUM`, `COUNT`, `JSON_ARRAYAGG` и т.п.) |
| `t105` | HAVING FROM | значение HAVING-условия |
| `t106` | HAVING TO | значение HAVING-условия |
| `t107` | Скрыть колонку в отчёте | пустая строка = видима, любое значение = скрыта |
| `t109` | Сортировка | число ≠ 0: знак = направление, абс. значение = приоритет |
| `t65` | Итог по колонке в нижней строке отчёта | `SUM`, `AVG`, `MIN`, `MAX`, `COUNT` |
| `t132` | SET-выражение | SQL-выражение для SET-запросов |

Колонка запроса всегда является отдельной записью типа 28. Значение `t28`
указывает на таблицу или реквизит из метаданных. Если колонка выбирает саму
таблицу, ядро берёт значение записи таблицы; если колонка выбирает реквизит,
ядро сначала определяет родительскую таблицу этого реквизита. Для вычисляемой
колонки источник задают как «Вычисляемое»/`0`, а фактическое выражение кладут в
`t101`.

В одном отчёте можно вывести не больше 61 видимой колонки из-за ограничения
MySQL. Скрытые колонки (`t107`) не попадают в ответ, но могут участвовать в
фильтрах, сортировке, формулах, агрегациях и SET-выражениях.

#### Соглашение об именах отчётов и колонок (правило проекта)

Имя колонки (`t100`) служит сразу **ключом в `JSON_KV`** и **именем внешнего
фильтра `FR_`/`TO_` в URL**, поэтому именуем латиницей в `snake_case`, без пробелов:

- **Отчёт** (`t22`): `<область>_<суть>`, напр. `order_pipeline` (без префикса
  базы — внутри одной БД он у всех одинаков). Имя
  используется и в URL (`report/{имя}`). Переименовать готовый запрос —
  `_m_save/{queryId} t22=имя` (имя — главное значение записи типа 22; именно
  `_m_save` и `t22`, не `_m_set`; `up=1` не нужен).
- **Колонка** (`t100`): `<сущность>_<поле>` с префиксом сущности
  (`order_`, `position_`, `cut_`, …); ID-колонка — `<сущность>_id` (функция abn_ID);
  единицы суффиксом (`_mm`, `_m`). Напр.: `order_id`, `order_status`,
  `position_width_mm`, `gp_arrived`.

Латиница без пробелов исключает URL-кодирование кириллицы и подмену пробелов на
`_` (ядро делает её для `t100` с пробелами); префикс сущности снимает коллизии
имён в join-отчёте.

**Поля самого запроса** (тип 22, изменяются через `_m_set/{queryId}?JSON`):

| Поле | Назначение |
|------|-----------|
| `t22` | Имя запроса |
| `t95` | Интерактивный режим (параметры задаются пользователем) |
| `t134` | Лимит строк |

**Получить колонки запроса**:
```
GET /{db}/object/28/?F_U={queryId}&JSON&LIMIT=1000
```

**Выполнить запрос**:
```
GET /{db}/report/{queryId}?JSON           # полный платформенный JSON
GET /{db}/report/{queryId}?JSON_DATA      # первая строка как объект
GET /{db}/report/{queryId}?JSON_KV&FR_Поле=X # с фильтром по t100
```

«Имя в отчёте» (поле `t100`) у колонки — ключевое условие: только колонки с
заданным `t100` доступны для внешней фильтрации через `FR_`/`TO_` (см. выше
«Условие работы фильтра»). Поле `t58` (`ALIAS`) нужно для внутреннего SQL alias и
ручных записей `FROM`/`REP_JOIN`; внешний фильтр по нему не строится.

### Как ядро строит SQL из колонок запроса

**Вся база хранится в одной физической IDEAV-таблице** с именем, равным slug
базы (например `ateh`). В ней данные, метаданные и управляющие структуры
записываются строками с полями `id`, `t` (тип), `up` (родитель) и `val`
(значение): `id` даёт каждой записи самостоятельный ключ, а составные индексы по
`id`, `up`, `t` и `val` позволяют быстро находить нужные связи. «Заказы», «Позиции»
и другие бизнес-таблицы существуют как логические типы и связи внутри общей
таблицы, а ядро разворачивает их в привычные широкие выборки через
SQL-объединения.

Ядро не знает заранее схему SQL. Оно читает строки «Колонок запроса» (type 28),
смотрит метаданные таблиц и реквизитов, а затем строит `LEFT JOIN`-цепочки.
Связи берутся только из метаданных:

- реквизит напрямую подчинён таблице через `up`;
- дочерняя/табличная часть связана с родителем через `up`;
- ссылка хранится как отдельная ref-запись, где `up` указывает на источник, а `t`
  указывает на целевую запись справочника.

Алгоритм идёт слева направо по колонкам отчёта.

1. Первая реальная колонка определяет мастер-таблицу. Если колонка выбрала
   реквизит, мастер-таблица — родительская таблица этого реквизита.
2. Для каждой следующей колонки ядро определяет её таблицу: сама таблица, если
   выбрана первая колонка таблицы, или родительская таблица, если выбран реквизит.
3. Если эта таблица уже присоединена, новый реквизит добавляется к существующему
   join через `req.up = parent.id`; дополнительные ветки не проверяются.
4. Если таблица ещё не присоединена, ядро перебирает уже присоединённые таблицы
   от мастер-таблицы к более поздним и берёт первую подходящую связь.
5. Порядок проверки веток фиксированный и соответствует комментариям в
   `Compile_Report()`:

| Ветка | Когда срабатывает | Что соединяет |
|-------|-------------------|---------------|
| `Reference to us` | Уже присоединённая таблица имеет ref-реквизит на новую таблицу | ref-запись источника с целевой записью новой таблицы |
| `We are an Array` | Новая таблица является дочерней/array-таблицей уже присоединённой | `new.up = joined.id` |
| `We have a Reference` | Новая таблица имеет ref-реквизит на уже присоединённую | ref-запись новой таблицы с записью уже присоединённой |
| `We got an Array` | Уже присоединённая таблица является дочерней для новой | `joined.up = new.id` |

Если ни одна ветка не связала очередную таблицу, ядро делает следующий проход:
возможно, сначала нужно присоединить промежуточную таблицу из более поздней
колонки. После 1000 проходов отчёт завершается ошибкой «Не могу связать колонки
отчета».

Alias обычной таблицы в SQL = `a{typeId}` (тип 42 → `a42`). Ref-записи получают
alias `r{typeId}`. Если в отчёте есть ручная запись `REP_JOIN` (type 44), `t58`
колонки должен совпасть с alias этой записи, иначе ядро не применит ручной JOIN к
данной колонке.

#### Пример: права пользователя

Для отчёта с колонками `Пользователь, Роль, Объекты, Описание` используются такие
связи из метаданных:

- `Пользователь` — таблица 18, это мастер-таблица;
- `Роль` — ref-реквизит пользователя на таблицу 42;
- `Объекты` — array-таблица 116 под ролью 42;
- `Доступ -> Описание` — ref-реквизит array-строки 116 на таблицу 47, затем
  обычный реквизит 120 под таблицей 47.

На таких метаданных ядро строит SQL следующего вида:

```sql
SELECT a18.val as 'Пользователь',a42_val as 'Роль',a116_val as 'Объекты',a120_val as 'Описание'
FROM xcom a18
  LEFT JOIN (SELECT r42.up,a42.val a42_val,a42.id a42_id
  FROM xcom r42,xcom a42 USE INDEX (PRIMARY)
    WHERE a42.id=r42.t  AND a42.t=42
   ) a42 ON a42.up=a18.id
  LEFT JOIN (SELECT a116.up,a116.val a116_val,a116.id a116_id
  FROM xcom a116
    WHERE a116.t=116
    ) a116 ON a116.up=a42_id
  LEFT JOIN (SELECT r47.up,a120.val a120_val
  FROM xcom r47,xcom a47 USE INDEX (PRIMARY)
   LEFT JOIN xcom a120 ON a120.up=a47.id AND a120.t=120
    WHERE a47.id=r47.t  AND a47.t=47
   ) a47 ON a47.up=a116_id
WHERE a18.up!=0 AND length(a18.val)!=0 AND a18.t=18
```

В `JSON_KV` результат выглядит как плоские строки:

```json
[
  {
    "Пользователь": "user",
    "Роль": "user",
    "Объекты": "Пользователь",
    "Описание": "Пользователь"
  },
  {
    "Пользователь": "user",
    "Роль": "user",
    "Объекты": "Пользователь -> Имя",
    "Описание": "Пользователь -> Имя"
  },
  {
    "Пользователь": "user",
    "Роль": "user",
    "Объекты": "Запрос",
    "Описание": "Запрос"
  }
]
```

#### Значения, функции, формулы и SET

По умолчанию колонка выбирает `val`. Для ref-реквизита это значение целевой
записи справочника, а не техническая ref-строка. Если нужен ID, используйте
функцию `Abn_ID`/`abn_ID`: для обычного реквизита она возвращает ID записи
реквизита, а для ref-реквизита — ID целевой записи. Также есть системные
`abn_UP`, `abn_TYP`, `abn_ORD`, `abn_REQ`, `abn_BT`.

> «Функция» (`t104`) — **ссылочное поле**: записывается ID функции, а не её имя.
> Для abn_ID это `t104=85` (строка «abn_ID» молча не сохранится). ID компактнее
> символьных значений и удобнее для внешнего фильтра `FR_`/`TO_`.

Функция `t104` может быть обычной SQL-функцией или агрегацией. В коде как
агрегирующие распознаются `AVG`, `COUNT`, `MAX`, `MIN`, `SUM`, `GROUP_CONCAT`,
`JSON_ARRAYAGG`. Если хотя бы одна видимая колонка агрегируется, ядро 
автоматически строит `GROUP BY` по всем видимым неагрегированным колонкам.

**Поле `t101` (Формула)** вставляется в текст SELECT как SQL-выражение:

```sql
-- t101 = "ROUND(a42.val * 0.20, 2)"  →  ядро подставит:
SELECT ROUND(a42.val * 0.20, 2) as 'НДС'
```

Формула может использовать `[THIS]`, чтобы сослаться на текущее поле, и
`[ИмяКолонки]`, чтобы сослаться на выражение другой колонки. После сборки
полей ядро заменяет такие ссылки в SELECT, WHERE, GROUP BY, HAVING, ORDER BY,
JOIN-условиях и SET-выражениях на реальные SQL-выражения. Защита минимальная:
формулы с ключевыми словами `from`, `select`, `table` блокируются, поэтому
формула должна считаться доверенным SQL, а не пользовательским вводом.

SET-запрос использует поле `t132`. Это SQL-выражение вычисляется на тех же
данных, что и SELECT, и применяется только после подтверждения
`confirmed=1`. Без подтверждения отчёт показывает превью затрагиваемых строк.

В фильтрах, формулах и SET-выражениях можно использовать встроенные значения:
`[TODAY]`, `[NOW]`, `[YESTERDAY]`, `[TOMORROW]`, `[MONTH_AGO]`, `[WEEK_AGO]`,
`[MONTH_PLUS]`, `[USER_ID]`, `[USER]`, `[ROLE_ID]`, `[ROLE]`, `[TSHIFT]`,
`[REMOTE_ADDR]`, `[REMOTE_HOST]`, `[HTTP_USER_AGENT]`, `[HTTP_REFERER]`,
`[HTTP_HOST]`, `[REQUEST_URI]`.
